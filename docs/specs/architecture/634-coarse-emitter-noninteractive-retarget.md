## Context

EPIC #596 Phase 2 ships a structured event stream to phones that negotiated the `interactive` capability (#632 emits it; #633 wired the live producer). But #589's **coarse** emitter (`assistantTurnEmitterV2`) still fans its finished-turn `message` envelope to **all** open v2 conns via the capability-agnostic `ActiveConnIDs` (#588). Net effect today: an `interactive` phone receives **both** the coarse `message` and the structured stream ‚Äî redundant double-delivery.

This slice re-targets the coarse emitter to **non-interactive conns only**, via #626's capability-aware `ActiveConns`, filtered to `!Interactive`. The two paths then become **mutually exclusive per conn** ‚Äî the exact complement of #632's `interactive`-only structured filter. This completes the ADR 025 ¬ß Phase 2 v2-additive guarantee: capable phones get the structured stream; everyone else stays on the coarse path, unchanged.

**Descope (2026-06-08 rework).** The original AC demanding a live two-phone e2e in which the `interactive` phone *receives the structured envelopes* is not exercisable today: `fakeclaude` writes only `{}\n` to its session JSONL while the structured producer tails the **real** claude JSONL transcript, so an interactive phone under fakeclaude has no structured stream to receive (and the realclaude suite has no Noise-phone harness; #603 is open). That capstone is split into blocked follow-up **#642**. This slice keeps the production fix and the deterministically-provable half: the coarse re-target, its inverse unit oracle, and a runnable coarse-half two-phone fakeclaude e2e.

## Design

### Production change (the entire production delta)

One file: `cmd/pyry/assistant_turn_v2.go`. Three edits, ~8‚Äì10 LOC net.

1. **Add the import** `"github.com/pyrycode/pyrycode/internal/relay"` (the package is already imported elsewhere in `cmd/pyry` ‚Äî `interactive_turn_v2.go`, `relay.go` ‚Äî so no new module edge).

2. **Re-shape the `v2Broadcaster` interface** to the capability-aware surface ‚Äî it becomes structurally identical to `interactiveBroadcaster`:

   ```go
   type v2Broadcaster interface {
       ActiveConns(ctx context.Context) []relay.ActiveConn
       Push(ctx context.Context, connID string, env protocol.Envelope) error
   }
   ```

   Update the doc comment: drop the "capability-agnostic `ActiveConnIDs`" framing; state that the coarse path now fans only to **non-interactive** conns (the complement of #632's interactive filter), `*relay.V2SessionManager` still satisfies it.

3. **Filter the fan-out loop** in `broadcast` (`:142`). The loop swaps from `connID` strings to `relay.ActiveConn` and `continue`s on the capability:

   ```go
   for _, c := range e.bcast.ActiveConns(ctx) {
       if c.Interactive {
           continue // structured-stream conns get the #632 path, never the coarse message
       }
       e.nextID++
       env := protocol.Envelope{ ID: e.nextID, Type: protocol.TypeMessage, TS: time.Now().UTC(), Payload: payloadJSON }
       if err := e.bcast.Push(ctx, c.ConnID, env); err != nil { /* unchanged: ctx-cancel early-return + DEBUG log, now keyed on c.ConnID */ }
   }
   ```

   Everything else in `broadcast` is frozen: the cursor-empty drop, the `conversations.NewID` rand-failure drop, the payload-marshal drop, the per-conn Push-error tolerance, the `nextID++`-per-conn-per-envelope policy, and the **no-application-output-in-logs** contract (logs carry only `chunk_len`, `conversation_id`, `message_id`, `conn_id`, transport `err`).

**Wiring is untouched.** `cmd/pyry/relay.go:327` passes `mgr` (a `*relay.V2SessionManager`) as `bcast`; the manager already exposes `ActiveConns` (`v2session.go:1654`). The `relay.go:323` comment ("ActiveConnIDs + Push") should be updated to "ActiveConns + Push" for accuracy, but no code there changes.

**`ActiveConnIDs` is left in place.** After this change `cmd/pyry` no longer consumes `relay.(*V2SessionManager).ActiveConnIDs`; the method becomes reachable only from `internal/relay` tests (its doc comment already anticipates this). Removing it is **out of scope** ‚Äî the body freezes everything but the coarse emitter's enumeration. Do not touch it.

### The mutual-exclusivity invariant (AC3)

A single conn never receives both paths. This is **structural**, not enforced by runtime coordination: the coarse emitter skips `c.Interactive` conns; the structured emitter (`interactive_turn_v2.go:221`) skips `!c.Interactive` conns. The two filters are exact complements over the same `ActiveConns` snapshot field. No conn satisfies both, none satisfies neither ‚Äî every open conn is on exactly one path. The unit oracle (below) proves the coarse half; #632's existing matrix proves the structured half.

## Concurrency model

No new goroutines, no new synchronization. The change is a filter inside the existing `broadcast`, which runs serially on the emitter's single `Run` goroutine (drained from the bounded `in` channel fed by `Bridge.SetOutputObserver` on the supervisor's PTY-drain goroutine). `ActiveConns` is the manager's existing snapshot funnel (request/reply over `m.snapshot`, serviced on the manager's dispatch goroutine) ‚Äî already concurrency-safe and already used by the structured emitter.

**Why the coarse-half e2e is drivable (load-bearing).** `WriteUserTurn` (`supervisor.go:205-208`) stamps `s.currentConvID = id` **before** capturing the session and calling `deliverFn`/`WaitReady`. Under fakeclaude, `WaitReady` never reaches idle (no real TUI) and the delivery eventually errors at the 30 s `sendMessageDeliverTimeout`, returning `server.binary_offline` ‚Äî but the cursor was already stamped, synchronously, on the send_message dispatch goroutine. The coarse fan-out runs on the **independent** emitter goroutine and only needs a non-empty cursor. So a send_message (whose ack we never await) stamps the cursor, and a subsequently-triggered fakeclaude stdout chunk fans out the coarse `message`. This is precisely the control-flow the skipped #603 test cannot reach (it blocks at the `TypeAck` wait); the new test never waits for the ack.

## Error handling

Unchanged from #589. A per-conn `Push` error is DEBUG-logged (keyed on `c.ConnID`) and the loop continues ‚Äî one dropped conn must not abort the turn for the others (AC2). A `ctx`-cancel mid-fan-out returns early. Cursor-empty / rand-failure / marshal-failure each drop the chunk with a content-free log line. The capability filter adds no new failure mode ‚Äî a skipped interactive conn is a `continue`, not an error.

## Testing strategy

### Unit ‚Äî inverse oracle (AC1, AC3) + stub migration, in `cmd/pyry/assistant_turn_v2_test.go`

The existing `stubV2Broadcaster` implements the old surface. Migrate it:

- Change `snapshots [][]string` ‚Üí `snapshots [][]relay.ActiveConn`; rename/retype the method `ActiveConnIDs(ctx) []string` ‚Üí `ActiveConns(ctx) []relay.ActiveConn` (the scripted-sequence + steady-state-reuse + copy-out logic is otherwise identical). Add a terse helper, e.g. `nonInteractive(ids ...string) []relay.ActiveConn`, so the six existing call sites (all non-interactive by intent) read `newStubV2Broadcaster(nonInteractive("conn-a", "conn-b"))`. The six existing tests keep their assertions verbatim ‚Äî they now flow through the filter as non-interactive conns, so behavior is unchanged. This is a single-file, mechanical migration (no cross-package cascade).

- **New test ‚Äî the inverse oracle.** Drive a mixed single snapshot `{ConnID:"a", Interactive:true}, {ConnID:"b", Interactive:false}` with a stamped cursor; enqueue one chunk; drain pushes. Assert:
  - exactly **one** push, addressed to `"b"`;
  - **never** a push to `"a"` (the interactive conn);
  - the pushed envelope is a well-formed `message` (Type, `Role:"assistant"`, `ConversationID`, valid-UUID `MessageID`, the chunk text) ‚Äî reuse `msgIDOf` / `drainPushes`.
  This is the deterministic proof of AC1 and (with #632's existing interactive-only matrix) AC3.

### E2E ‚Äî coarse-half two-phone fakeclaude, in a **new file** `internal/e2e/relay_two_phone_coarse_test.go`

`//go:build e2e`, package `e2e`. Reuses package-siblings (`driveHandshakeToOpenDaemon`, `decryptInnerEnvelope`, `waitBinaryHello`, `fakephone.Dial`, `decodePairPayload`, `readPersistedServerID`, `StartRotationWithRelay`, `mustJSON`, `sendNoiseMsg`, `readInnerFrame`, `sendNoiseInit`). **Do not modify `relay_v2_handshake_test.go`** (in-flight `feature/449` overlap) ‚Äî define the new capability-aware helper locally.

New local helper (in this file): a capability-aware handshake driver ‚Äî same body as `driveHandshakeToOpenDaemon` but its early-data hello carries `Capabilities: []string{protocol.CapabilityInteractive}` (build the `HelloClientPayload` inline, mirroring `buildHelloEarly`, with the capabilities field set; pass to `initiator.WriteInit`). The non-interactive phone uses the existing `driveHandshakeToOpenDaemon` unchanged.

Test flow (one daemon, fakeclaude child, fakerelay ‚Äî `StartRotationWithRelay` with `PYRY_MOBILE_V2=1` and `PYRY_FAKE_CLAUDE_ASSISTANT_TRIGGER`):

1. **Pair two devices** (`pyry pair ‚Ä¶ --name=phone-a`, `--name=phone-b`) ‚Üí two tokens, same server static pubkey (the daemon's static key is per-instance). Seed one known conversation row (the cursor will reference it; the `ValidateConversation` gate must pass so the stamp happens). Start the daemon; `waitBinaryHello`.
2. **Both phones handshake to open.** Phone A via the capability-aware driver (`interactive` granted ‚Üí `s.interactive=true`); phone B via the plain driver (no caps ‚Üí `s.interactive=false`). Two distinct conn_ids (`c-1`, `c-2`) ‚Üí two independent V2 sessions; fakerelay routes each natively.
3. **Drive a turn without awaiting the ack.** Seal a `send_message` for the known conversation from the **non-interactive** phone B and send it. **Do not** read its reply (it would be a `server.binary_offline` error ~30 s later; irrelevant). This stamps the supervisor cursor synchronously (¬ß Concurrency model).
4. **Trigger the assistant emit with a re-arm loop** (defeats the stamp-vs-emit race): in a bounded loop (overall deadline ~8 s), (re-)write the assistant trigger file with a known marker text, then attempt to read+decrypt one binary‚Üíphone frame on **phone B** with a short sub-deadline. fakeclaude re-emits the marker on each re-armed trigger (50 ms poll, repeat-firing). Loop-until-marker on B, tolerating non-`message` / prelude frames (mirror the skipped test's drain loop). **Assert phone B receives a coarse `message`** whose payload carries the marker, `Role:"assistant"`, the known `conversation_id`, valid-UUID `message_id`, `InReplyTo == nil` (server-initiated).
5. **Assert phone A receives no coarse `message`.** After B's marker lands, the same `broadcast` loop pass that pushed to B structurally skipped A (complementary filters over one snapshot). Operationalize: drain phone A's inbound for a bounded window (~1 s) and assert **no `TypeMessage` envelope** arrives (decrypt-in-capture-order via `decryptInnerEnvelope`; under fakeclaude the structured producer never fires, so A receives nothing turn-related ‚Äî see Open questions for a stronger barrier option).

This test is **runnable, not skipped** ‚Äî it is the deterministic coarse-half proof the slice ships. The interactive phone *receiving the structured stream* is **not** asserted here (capstone #642).

### Build / vet

`go build -o pyry ./cmd/pyry`; `go test -race ./cmd/pyry/...`; `go test -tags=e2e -run TestTwoPhoneCoarse ./internal/e2e/...`; `go vet ./...`. No application output (PTY bytes / assistant text) on any log line at any level (AC5 ‚Äî inherits #589).

## Open questions

- **Negative-assertion strength (step 5).** The bounded-drain ("A receives no `TypeMessage` in ~1 s") is a smoke confirmation; the *deterministic* mutual-exclusivity proof is the unit oracle + #632's matrix. If a stronger e2e barrier is wanted: after B's marker, send a solicited `list_conversations` from phone A and assert A's first inbound app frame is the `conversations` reply (FIFO on A's conn ‚áí no coarse `message` was queued ahead of it). Recommended only if the bounded-drain proves flaky; default to the simpler drain.
- **Re-arm loop tuning.** Poll/sub-deadline/overall-deadline are tuning knobs, not contract. Start with ~8 s overall, ~300 ms sub-deadline (‚â• fakeclaude's 50 ms poll). The only non-flaky failure is "cursor never stamped," which is a real regression. The developer should confirm a single trigger write (no re-arm) is *insufficient* before adding the loop ‚Äî if the latency ordering (stamp ‚â™ first 50 ms poll) holds reliably in CI, a single write + loop-until-marker on B may suffice; the re-arm is the robustness belt.
- **Driver phone choice.** Step 3 drives from phone B; driving from A or a third phone is equivalent (the cursor is supervisor-global, not per-conn). B is chosen so A's inbound stays free of solicited replies for the clean negative assertion.

## Security review

**Verdict:** PASS ‚Äî no MUST FIX.

Run adversarially against the spec above, assuming it has holes. This slice is the exact **complement** of #632 (structured emitter, reviewed PASS); it completes #607's deferred gated-fan-out review for the **coarse** half. Stated threat focus: *an `interactive`-granted phone must never receive the coarse `message` (double-delivery), a non-interactive phone must keep receiving it exactly as today, and the two paths must be mutually exclusive per conn ‚Äî no conn on both, none on neither.*

**Findings:**

- **[Trust boundaries]** No findings. The load-bearing boundary is *which conns receive the coarse `message`*, and it is a **single explicit gate** ‚Äî `if c.Interactive { continue }` in `broadcast` ‚Äî backed by two different-fabric deterministic nets: `ActiveConns` enumerates only `V2StateOpen` (token-authenticated) sessions (`v2session.go:1710`), and `Push` independently re-gates `V2StateOpen` (#571). The emitter never constructs a `conn_id` from event/PTY data; it iterates the manager's snapshot and uses `c.ConnID`, so hostile claude output can never redirect delivery. The inbound boundary (claude PTY output ‚Üí chunk) carries the operator's own session content to the operator's own authorized devices ‚Äî no privilege crossing. **No new inbound surface:** this is a filter change on an existing outbound fan-out; the send_message that drives the e2e is the unchanged #312/#594 handler, and the capability advertisement is #626's unchanged inbound surface.
- **[Trust boundaries ‚Äî capability flag integrity & fail-safe direction]** No findings. The emitter trusts `ActiveConn.Interactive`, whose integrity is #626's (reviewed PASS): built by iterating the daemon's *authoritative* `supportedV2Capabilities` (spoofing impossible by construction), written only on the token-OK branch, fail-closed `false` by default. The fail-safe direction here is the **inverse** of #632 and is correct: a conn whose flag defaults/fails to `false` still receives the coarse path (the *less* sensitive path ‚Äî finished-turn text already shipping since #589), so a flag-integrity failure degrades to "coarse delivered, structured withheld," never "structured leaked to a non-capable phone" and never "both withheld." This slice consumes the flag correctly (`continue` on `== true`) and adds no path that could set or forge it.
- **[Trust boundaries ‚Äî mutual exclusivity (AC3)]** No findings. "No conn on both paths" is **structural**, not runtime-coordinated: coarse skips `c.Interactive`, structured (`interactive_turn_v2.go:221`) skips `!c.Interactive` ‚Äî exact complements over the same set-once `ActiveConn.Interactive` field of one snapshot. No conn satisfies both; none satisfies neither. The inverse unit oracle is the deterministic guard for the coarse half; #632's matrix guards the structured half. **No TOCTOU on the flag:** `Interactive` is set-once at handshake (#626, never mutated, re-key preserves it), so it cannot flip mid-session; the only live transition is open‚Üíclosed, caught by `Push`'s `V2StateOpen` re-gate.
- **[Trust boundaries ‚Äî per-conversation confinement]** OUT OF SCOPE (named, inherited). The coarse fan-out reaches *every* non-interactive conn (the #589 shape), not a `conversation_id`-subscribed subset. Not a confidentiality leak today: one supervised claude, one `CurrentConversation()` cursor, all paired devices belong to the one operator. Payloads carry `conversation_id` for client-side filtering. True subscription routing is deferred to pyrycode-mobile#336 (the #589 deferral, inherited).
- **[Availability ‚Äî interactive phone has no coarse fallback]** OUT OF SCOPE (availability, not security; named so code-review does not flag the removed fallback as a regression). An `interactive` phone now receives *only* the structured stream; if that stream is broken/empty (e.g. under fakeclaude, or a wedged producer) it gets no coarse fallback. This is the explicit ADR 025 design intent ‚Äî structured-stream liveness is #633/#642's concern ‚Äî and it is an availability/UX property, not a confidentiality/integrity/auth one.
- **[Tokens, secrets, credentials]** No findings ‚Äî N/A. The emitter is downstream of all credential handling; it never sees a token or key. Authentication happened in #626's handshake; AEAD sealing happens inside `Push` (#571) on the manager's goroutine. The emitter hands `Push` a plaintext `protocol.Envelope` and touches no nonce/key.
- **[File operations / Subprocess]** No findings ‚Äî N/A in production. No path handling or file I/O on the production path (`conversations.NewID()` reads `crypto/rand`). The e2e's assistant-trigger file write is test-only code under the visibility-fenced `internal/e2e/internal/fakeclaude`, never compiled into `pyry`.
- **[Cryptographic primitives]** No findings. `message_id` is `conversations.NewID()` = `crypto/rand` UUIDv4 ‚Äî a dedup/ordering key, not a secret or capability. No crypto is *performed* here; the seal + nonce single-writer funnel are `Push`/#571's (reviewed PASS).
- **[Network & I/O]** No findings. No direct socket I/O. `MessagePayload.Text` is claude's own PTY output, unbounded by this slice but capped by the Noise frame limit at `Push` (over-large ‚Üí `Encrypt` fails ‚Üí per-conn `Push` error ‚Üí drop + DEBUG + skip; no crash, no DoS). Fan-out is O(open conns) per chunk, conns bounded by paired devices ‚Äî and this slice **reduces** the coarse fan-out (excludes interactive conns), so it is strictly less amplification than #589, never more.
- **[Error messages, logs, telemetry]** No findings ‚Äî AC5 inherits #589's spine, unchanged. Every log branch carries only `chunk_len`, `conversation_id`, `message_id`, `conn_id` (now `c.ConnID`), and the transport `err`. Application output (the PTY chunk ‚Üí `MessagePayload.Text`) is NEVER logged at any level. The marshal-error branch omits both the payload and `err.Error()` (the #589 lesson: `encoding/json` echoes invalid input bytes into its error string). **Deliberate carve-out, documented so code-review does not flag it:** the per-conn `Push`-error branch logs `err` verbatim, mirroring #589/#632 (reviewed PASS) ‚Äî `Push`'s errors are transport sentinels operating on the already-validated `json.RawMessage` this emitter marshalled, so no app content is reachable through them. No telemetry. The filter adds no new log line.
- **[Concurrency]** No findings. Zero goroutines spawned by this slice (a filter inside the existing serial `broadcast` on the emitter's single `Run` goroutine) ‚Üí no new leak, no new lock, no new shared state. The snapshot-then-`Push` gap is the #588/#589 reviewed pattern; `Interactive` is set-once so no TOCTOU on the flag; `Push`'s `V2StateOpen` re-gate catches a conn that left open in the gap. The new e2e's reliance on the cursor-stamp-before-`WaitReady` ordering is *test* correctness, not a production race ‚Äî it introduces no production concurrency surface.
- **[Threat model alignment]** All relevant `docs/protocol-mobile.md` ¬ß Security model / ADR 025 ¬ß Phase 2 threats addressed: capability-spoofing phone (defeated by #626's authoritative-set construction + the inverse gate + two `V2StateOpen` nets), interactive phone double-delivery (defeated by `c.Interactive` skip ‚Äî the AC1 oracle), non-interactive/old phone (still served by the unchanged coarse minting/marshal/`Push`, now correctly *only* this path), unauthenticated peer (`V2StateOpen` gate √ó2). The non-interactive path is strictly the *less* sensitive of the two and already shipping since #589 ‚Äî this slice **narrows** the coarse fan-out and widens no exposure.
