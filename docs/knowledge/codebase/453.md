# Ticket #453 — `internal/relay`: v2 re-key responder swap on open conn (`handleRekeyInit` + atomic CipherState swap + peer-static continuity check)

Closes the v2 re-key responder loop on the binary side. A `noise_init` arriving for a `conn_id` already in `V2StateOpen` is now routed to a new `handleRekeyInit` that runs IK responder again against the binary's static key, enforces peer-static continuity against the [`s.peerStatic` captured at initial handshake by #452](452.md), and atomically swaps `s.send` / `s.recv` in a single tuple assignment on the manager's single dispatch goroutine. Peer-static mismatch closes the conn at WS 4426 with `reason=rekey_peer_static_mismatch` — the cryptographic implementation of [ADR 024](../decisions/024-noise-ik-mobile-e2e.md)'s "no impersonation succeeds" residual-risk claim on the relay-MITM surface (`docs/protocol-mobile.md` § Threat #3). Sibling slice [#454](454.md) (`rekey_request` control-envelope discriminator) and predecessor [#452](452.md) (peer-static capture) cover the rest of the v2 re-key surface; the binary-initiated emit + 1h timer (#450) and the operator-facing `pyry rekey <conn_id>` verb (#451) remain out of scope. Extracted from the abandoned [#449 super-slice](../../specs/architecture/449-v2-rekey-responder.md) on the orphan `feature/449` branch; design lifted verbatim, same pattern as #452 / #454.

## Implementation

- **One production source file modified** — `internal/relay/v2session.go` (+~150 LOC: top-of-function `switch s.state` in `handleNoiseInit`, new `handleRekeyInit` method, two package-comment additions on `V2Session` and `V2SessionManager`). Tests +~290 LOC net in `internal/relay/v2session_test.go` (three new tests, one removed). Inside the spec's S budget.

- **`handleNoiseInit` top gate becomes a `switch s.state`.** The previous `if s.state != V2StateAwaitingInit` early-reject is replaced by a three-arm switch: `V2StateOpen` routes to `handleRekeyInit` and returns; `V2StateAwaitingInit` falls through to the unchanged initial-handshake body; `default` (i.e. `V2StateHandshakeComplete`; `V2StateClosed` is filtered earlier in `handleFrame`) keeps the existing 4421 reject path with a new `reason="noise_init_in_handshake_complete"` log field. The old `reason="noise_init_after_handshake"` label is dropped — the open-state path no longer rejects.

- **`handleRekeyInit` is a ~70-LOC method with exactly five reject branches.** Same shape as initial handshake (`noise.NewResponder` → `ReadInit` → `WriteResp` → marshal `noise_resp` → emit) with three differences: (1) empty early-data on both directions per spec § Re-key (hello validation and token re-check skipped — they ran at initial handshake); (2) peer-static continuity check between `ReadInit` and `WriteResp`; (3) atomic CipherState swap on success via the single tuple assignment `s.send, s.recv = newSend, newRecv`. Each failure branch (`rekey_responder_init_failed`, `rekey_read_init_failed`, `rekey_peer_static_mismatch`, `rekey_write_resp_failed`, `rekey_marshal_noise_resp`) emits a `v2.handshake.reject.ik_failure` WARN line and calls `closeWith(ctx, s, StatusHandshakeFailure /*4426*/, nil)` — the SAME close code the initial handshake uses for the equivalent failure class. No new close code introduced.

- **Atomic swap is structural, not lock-based.** A single Go tuple assignment on the manager's single dispatch goroutine emits two stores, but no other goroutine reads or writes `s.send` / `s.recv` — the loop IS the lock for the non-goroutine-safe `flynn/noise` `CipherState` (#433's contract pushed to the per-conn-dispatch boundary). Two short comment additions pin this: one on `V2Session` ("the re-key responder atomically swaps `s.send` / `s.recv` … old `*CipherState` pointers are dropped from the struct and reclaimed by GC. No explicit `Wipe()` of the key bytes is exposed — the single-owner-goroutine invariant means no code path reads the old state after the swap, which is the practical zeroisation property."), one on `V2SessionManager` ("a single tuple assignment on this goroutine cannot be observed half-applied by any other code path, so the spec's atomic-switchover requirement is structural.").

- **Peer-static comparison uses `bytes.Equal`, intentionally NOT `subtle.ConstantTimeCompare`.** Both operands are public keys (the live peer's static from `resp.PeerStatic()` and the stored `s.peerStatic`), so timing leakage carries no secret. A one-line code comment names the choice to forestall a "should be constant-time" review nit. The #449 orphan spec and [#452.md](452.md) both pre-emptively documented this; the inline code comment is the final rail.

- **State stays `V2StateOpen` across a successful re-key.** `s.device` and `s.peerStatic` are preserved — `s.device` is the device snapshot captured at initial-handshake token validation (used by handlers via `*dispatch.Conn.Auth()`); `s.peerStatic`'s lifetime contract from #452 explicitly forbids overwrite on re-key (it pins the original peer's identity for the session's lifetime). Logs `v2.rekey.accept` with `conn_id` + `device_name` (operator-actionable; SAME device as initial handshake by construction).

- **Anti-enumeration on the peer-static-mismatch reject line.** The reject log on the mismatch branch deliberately omits `device_name`. The re-key initiator's identity is unknown / hostile; logging the captured-at-initial-handshake device-name on a rejected re-key would create an anti-enumeration signal (an attacker probing `conn_id`s could correlate close-event timing with device-name disclosure). The other four reject branches don't have a `device_name` field at all (the local fresh `resp` doesn't carry one); the discipline is explicit only on the mismatch branch where `s.device.Name` is in scope. Pinned by the test's per-line substring assertion (`device_name` must not appear in the line containing `event=v2.handshake.reject.ik_failure`).

- **AEAD-failure teardown on old-key frames after the swap is INHERITED, not new code.** Once `s.recv` points at the new CipherState, any frame sealed under the old keys fails `s.recv.Decrypt` and lands in the existing [#446 tampered-frame branch](446.md): `closeWith(StatusProtocolMismatch /*4421*/, nil)` → session removal from `m.sessions`. AC #4 is verification-only; the test (`TestV2Session_RekeyResponder_OldKeyFrameAfterSwap_4421`) pins this inheritance against the post-swap state.

- **`s.resp` is dead after re-key but left dangling.** The re-key responder uses a fresh local `Responder` and never touches `s.resp`; the existing `s.resp` still points at the initial-handshake responder. This is acceptable — `s.resp` is only consumed inside the `awaitingInit` arm of `handleNoiseInit`. A follow-up cleanup that moves `s.resp` out of the struct is explicitly out of scope (would touch #445's surface without observable benefit).

- **No new close code, no new envelope shape, no wire-format change.** Every re-key failure mode closes at 4426; the swap success emits a normal `noise_resp` inner frame (same shape, same discriminator constant). The binary↔relay wire is unchanged; the phone↔binary AEAD channel rotates its keys structurally.

- **No new ADR.** [ADR 024](../decisions/024-noise-ik-mobile-e2e.md) covers the v2 re-key protocol; this slice implements the responder half of its § Re-key policy. The orphan spec at `docs/specs/architecture/449-v2-rekey-responder.md` resolved every design question this slice needed.

## Patterns established

- **Top-of-function `switch <state>` over `if state != X` when adding a second valid arm.** `handleNoiseInit`'s previous top-of-function `if s.state != V2StateAwaitingInit { close(4421); return }` was the right shape when `awaitingInit` was the only valid state; introducing a second valid state (`V2StateOpen` → re-key) means the early-reject becomes a state dispatcher. The pattern: when an early-reject gate acquires a second valid arm, refactor the gate into a `switch` rather than chaining a second `if state == X` clause — the switch is the explicit transition table and a future third valid arm lands as a third `case` without restructuring. The `default` arm carries the original reject behaviour with a sharper `reason` field that names the specific dead-end state, not the negation of the valid set.

- **Atomic state swap on a single dispatch goroutine is a single tuple assignment.** The re-key swap is `s.send, s.recv = newSend, newRecv` — one statement, two stores, observable atomically because no other observer exists. No mutex, no `sync/atomic`, no `unsafe.Pointer` swap. The pattern generalises whenever paired-pointer state (here CipherStates; elsewhere e.g. paired buffers, paired connections) needs an "all-or-nothing" rotation on a single owner goroutine: a Go tuple assignment IS the atomic primitive; reaching for a lock or atomic-pointer machinery is a category error when the structural invariant ("one goroutine owns these fields") already holds. Documenting the invariant in the package comment is what makes the structural property load-bearing — a future refactor that adds a second goroutine reader would need to read past that comment.

- **Field-reassignment is the practical zeroisation property when the single-owner-goroutine invariant holds.** The old `*CipherState` pointers are dropped from the struct on the swap; Go's GC reclaims the underlying memory. An explicit `Wipe()` of the key bytes is NOT exposed (would require touching #433's surface — deferred). The single-owner-goroutine invariant means no code path reads the old state after the swap, so the keys are unreachable from the program graph the instant the tuple assignment executes — that IS the practical zeroisation, even though the underlying memory may linger until the next GC cycle. The pattern generalises whenever a per-session secret is held in a struct field owned by a single goroutine and the rotation cadence is low enough that the heap-resident-until-GC window is downstream of the threat model. Document the trade-off ("explicit `Wipe()` deferred; field-reassignment is the practical zeroisation") in the package comment so a reviewer's instinct to add `*newSend = noise.CipherState{}` doesn't fire.

- **Continuity check between authenticated-bytes-available and the commit point.** The peer-static continuity check runs between `Responder.ReadInit` (which makes the new peer's static available via `resp.PeerStatic()`) and `Responder.WriteResp` (which would commit the binary to a response). Reject early; never reply on a mismatched re-key — the wire-side signal is the bare close at 4426; no `noise_resp` from the binary lets an attacker correlate "the binary tried to respond" with the close. The pattern generalises whenever a multi-step protocol exposes an identity claim midway and the protocol response commits the responder: insert the identity check between availability and commit, not after the response. The cost is one extra `bytes.Equal` per re-key; the value is a structural impossibility of "responded then noticed the mismatch."

## Lessons learned

- **The same orphan-spec pattern as [#452](452.md) and [#454](454.md).** This slice was the third carve-out from the abandoned `feature/449` super-slice. The architect spec at `docs/specs/architecture/449-v2-rekey-responder.md` on the orphan branch contained verbatim design notes for the state-machine refactor, the `handleRekeyInit` body shape (including the exact failure-mode → log-reason mapping), and the package-comment additions that this slice ships. Re-deriving any of those decisions risks subtle drift (a different reason-string vocabulary, a different swap idiom, a missing comment paragraph). The orphan-spec lesson recurs: when a super-slice is closed `NOT_PLANNED` and broken into smaller pieces, the abandoned spec is the canonical design document for each carved-out slice — read it first, lift the decisions verbatim, name it explicitly in the new spec's "Files to read first" so the developer doesn't re-derive. The orphan branch is a feature of the carve-out, not noise to route around.

- **`bytes.Equal` for the peer-static comparison was challenged THREE TIMES across the slice family — the inline comment is the final rail.** The constant-time-vs-variable-time question was raised and answered in #449's orphan spec, again in [#452.md](452.md) (which captured the future #453's choice ahead of time), and again in #453's spec itself. Despite all of this, the code-level rail that survives diff review is the one-line code comment next to the `bytes.Equal` call. The lesson: when a "should be constant-time" instinct will recur every time a reviewer reads the line, the documentation that survives is the one that lives WHERE the line lives. Spec-level commentary protects the architect's design intent; an inline comment protects the developer's implementation from a well-meaning "while we're here" refactor. Both are belt; the inline comment is the suspenders, and it's the one that doesn't decay when specs archive.

- **Test the inherited behaviour at the new state boundary, not just the new code.** AC #4 ("old-key frame after swap → 4421") describes inherited behaviour from [#446](446.md)'s tampered-frame branch — no new code is needed. The natural reading is "no test needed either"; the correct reading is "the test pins the inheritance against the new state (post-swap)." The same `closeWith(4421)` primitive that fires today on a tampered frame in `open` now also fires on a stale-but-well-formed frame after a key rotation — but ONLY because `s.recv` points at the new CipherState. A future refactor that broke the swap atomicity, or that introduced a "fallback to old `s.recv` on first decrypt failure" could silently regress without breaking [#446](446.md)'s test (which sets up its own happy path). The lesson: when a new operation creates a new authenticated-state boundary, write a verification-only test against the inherited failure behaviour at that boundary — the test's value is not "does the existing branch work" but "does the existing branch fire WHEN the new operation has changed the state."

- **The "removed test" surface is part of the diff review.** Removing `TestV2Session_NoiseInitAfterOpen_4421` is load-bearing — it pinned exactly the behaviour this slice intentionally changes (second `noise_init` in `open` → 4421 close becomes second `noise_init` in `open` → re-key swap). The replacement coverage is the three new tests above; the removal must be explicit in the commit so a reviewer can verify the new tests cover the AC the removed test was protecting. The pattern: when a test removal is a deliberate "this behaviour is changing", name the replacement coverage in the same commit message and pin the removal in the spec's "Files to read first" so the test isn't quietly re-added by a future contributor with stale context.

## Out of scope (filed elsewhere or deferred)

- **`rekey_request` control-envelope discriminator** — [#454](454.md), sibling slice; different function in the same file; independent of this slice and shipped in parallel.
- **`V2Session.peerStatic` capture at the initial IK handshake** — [#452](452.md), predecessor slice; the field this slice reads. Set as `blockedBy` at split time.
- **Binary-initiated re-key (1-hour timer + emit + 30s reply timeout)** — #450. Uses the same `protocol.TypeRekeyRequest` constant landed in [#454](454.md); the binary becomes the IK initiator for its own scheduled re-key.
- **`pyry rekey <conn_id>` operator verb** — #451. CLI surface for manual re-key.
- **Phone-side re-key triggering policy / phone-side timer** — mobile team's concern. The binary responds to phone-initiated `noise_init` regardless of phone-side cadence.
- **Persistence of new `CipherState`s across binary restarts** — out of scope for v2; ADR 024 § Re-key policy specifies in-memory only. A binary restart drops both old and new keys; the next initial handshake re-establishes.
- **Explicit `Wipe()` method on `internal/noise.CipherState`** — would require touching [#433](433.md)'s surface; deferred. The single-owner-goroutine invariant provides the practical zeroisation property; documented in the package comment.
- **`s.resp` reset to nil after handshake completes** — cleanup that belongs in a `V2Session`-shape refactor, not in a re-key slice. Tracked as an open question in `features/v2-session-manager.md`.
- **Per-conn fan-out for handler dispatch** — head-of-line blocking during re-key (~100µs X25519 derivation + AEAD setup + one outbound send) stalls all conns on the single dispatch goroutine, same posture as initial handshake. Priority follow-up before production cutover; same root cause as the [#446](446.md) follow-up, NOT addressed in this slice.
- **Re-key rate-limiting** — defer until observed; same posture as initial-handshake rate-limiting (none). Per-binary single-dispatch-goroutine fan-in naturally caps total re-key rate across all conns.
- **Production wiring of `V2SessionManager` into `cmd/pyry/relay.go`** — daemon path still runs the v1 dispatcher; cutover gated on [#436](436.md)'s pre-flight check.
