# Ticket #454 — `internal/relay` + `internal/protocol`: v2 `rekey_request` control-envelope discriminator

Intercept AEAD-decrypted envelopes whose `type` is `rekey_request` at the v2 dispatch boundary so they are recognised as Mobile Protocol v2 control envelopes and short-circuited away from the application handler chain. The binary is always the IK responder per [ADR 024](../decisions/024-noise-ik-mobile-e2e.md); a `rekey_request` from the phone takes no transport action — the phone re-keys by sending `noise_init` directly, not by signalling. The design is logs-only: INFO for the closed set of `payload.reason` values from `docs/protocol-mobile.md` § Re-key (`scheduled`, `manual`, `compromise`), WARN for anything else (forward-compat for a future mobile reason the binary doesn't yet recognise). Split from the abandoned #449 super-slice; independent of [#452](452.md) (peer-static capture) and [#453](453.md) (re-key responder swap) — different functions in the same file, ships in parallel.

## Implementation

- **Three production source files modified** — `internal/protocol/codes.go` (+23 LOC: one constant + doc-comment block), `internal/protocol/envelope.go` (+7 LOC: doc-comment above `v1TypeSet`, no code change), `internal/relay/v2session.go` (+55 LOC: five-line probe-and-switch at the top of `dispatchAppFrame` plus the new `handleRekeyRequest` method). Tests +48 LOC partition assertion in `internal/protocol/compat_test.go` and +244 LOC in `internal/relay/v2session_test.go`. Total within the spec's XS budget.

- **`protocol.TypeRekeyRequest = "rekey_request"`** lives in a new "Mobile Protocol v2 control-envelope types" block at the **bottom** of `internal/protocol/codes.go`, after the v1 `Push` group. Co-located with the v1 `Type*` constants (project convention — a single file is right while there's only one v2 control type) but rhetorically partitioned by the group-level doc-comment that explicitly names "MUST NOT appear in `v1TypeSet`" with a one-line rationale: a leak would route the envelope to `dispatch.Route`'s handler chain in violation of AC #2. The per-constant doc-comment cross-references both `internal/protocol/envelope.go`'s `v1TypeSet` and `internal/protocol/compat_test.go`'s partition test, so a future contributor adding a third v2-control type lands at both halves of the asymmetry by following the breadcrumbs.

- **Three-place pin on the asymmetry, different fabric for belt-and-suspenders.** The pipeline-wide principle "belt-and-suspenders means different fabric" is honoured by spreading the v1/v2 partition guard across three structurally-independent enforcement layers: (1) the constant's doc-comment ("MUST NOT add to `v1TypeSet`" — instructive, advisory); (2) a new doc-comment **above** `v1TypeSet` ("v2-control types like `TypeRekeyRequest` MUST NOT be added here" — instructive, advisory, on the gate itself); (3) `TestTypeConstants_V1V2Partition` in `compat_test.go` — a deterministic partition assertion that runs in CI on every push (every `Type*` constant must be in `v1TypeSet` **or** in the new test-local `v2OnlyTypes` allowlist, and never in both). The first two are stochastic-rule rails for code review; the third is a deterministic test rail. Either advisory comment alone catches the mistake at review-time; the test catches it at CI-time regardless of review. Three rails, different mechanisms.

- **`v2OnlyTypes` allowlist literal lives in the test, not in production.** A `map[string]bool{TypeRekeyRequest: true}` in `compat_test.go` is the cheapest place to assert the partition because it requires no exported symbol from `internal/protocol`. Production callers cannot accidentally pick it up; the only consumer is the partition test itself. The previously-existing `TestV1TypeSet_CoversAllExportedTypeConstants` is unchanged — it still asserts the v1 set covers all v1 application types — the new test runs alongside it and adds the partition-disjointness check.

- **Probe-and-switch lives at the top of `dispatchAppFrame`, before the per-frame channel and `*dispatch.Conn` allocation.** Five lines: `json.Unmarshal(plaintext, &probeEnv)`, error-and-type guard, call `m.handleRekeyRequest(ctx, s, probeEnv)`, `return`. Decode failures **deliberately fall through** to the existing `dispatch.Route` path so [#446](446.md)'s `protocol.malformed` reply remains the canonical malformed-envelope behaviour — the probe does not consume that path. The probe is single-decision and inline rather than a helper method: hoisting it would cost one new method and three lines of plumbing for a five-line block whose readability is already maximal.

- **The probe is a re-decode.** `dispatch.Route` decodes the same plaintext a second time. Architect explicitly chose the re-decode over refactoring `Route` to accept a pre-decoded `Envelope` — the alternative would touch the v1 dispatcher and its tests for no measurable benefit at this slice's traffic levels. The per-frame cost is one small JSON parse (~µs), well below the per-frame AEAD cost (~100µs).

- **`handleRekeyRequest` is logs-only.** Decodes `env.Payload` as `struct{ Reason string }`, switches on the decoded `Reason`, logs at INFO for `scheduled`/`manual`/`compromise` and at WARN for anything else (including empty, including the JSON-decode-failure case). Three structured fields per line: `event="v2.rekey.request.received"`, `conn_id=s.connID`, `reason=<value>`. **No `m.send`, no `m.closeWith`, no `s.state` mutation, no outbound frame, no `*dispatch.Conn` allocation.** The method takes `ctx` in the signature for parity with `dispatchAppFrame` / `handleNoiseMsg` but is unused — accepted via the `_ context.Context` blank-identifier idiom (matches `handleNoiseInit`'s style where `ctx` only forwards to `closeWith`, and the linter has no opinion).

- **Inner-payload JSON decode failure is tolerated as empty reason; no `protocol.malformed` reply.** The envelope itself took no transport action either way — emitting a sealed error reply for a broken control payload would be a surprising behaviour change. The default arm of the switch (empty `Reason`) handles the JSON-failure case structurally. In practice, decoding `struct{Reason string}` against any well-formed JSON object succeeds with an empty `Reason` when the field is absent or non-string; only top-level non-object JSON would fail outright.

- **Concurrency unchanged.** Both the probe block and `handleRekeyRequest` run on the manager's single dispatch goroutine — the same goroutine that owns `s.send` / `s.recv` and the `m.sessions` map. No new mutex, no new goroutine, no new channel. The existing "Run is the only goroutine the manager owns" invariant is preserved without modification.

- **Forward-compat for unknown reasons is deliberate.** Mobile may add a `reason` value before the binary catches up; WARN-and-tolerate is the correct posture for "future protocol value the binary doesn't recognise yet." Rejecting unknown reasons (by sealing a `protocol.malformed` or closing the conn) would force every future protocol bump to be a binary-side change in lockstep — the spec is explicit that unknown reasons must not break the channel.

- **No new ADR.** [ADR 024](../decisions/024-noise-ik-mobile-e2e.md) covers the v2 protocol shape including the explicit `rekey_request` envelope; this slice ships the responder-side discriminator for it. The orphan spec at `docs/specs/architecture/449-v2-rekey-responder.md` on the dead `feature/449` branch documented the design decisions verbatim and was lifted into the new spec, same pattern as #452.

  - `TestV2Session_OpenState_RekeyRequest_Intercepted` — paired-device handshake to open → AEAD-sealed `{type: "rekey_request", payload: {reason: "scheduled"}}` → application handler chain is never invoked (pinned with an `atomic.Bool` flag on a registered stub handler, mirroring `TestV2Session_OpenState_TamperedNoiseMsg_4421`'s shape) → session stays in `V2StateOpen` → no outbound close envelope.
  - `TestV2Session_OpenState_RekeyRequest_UnknownReasonTolerated` — AEAD-sealed `{reason: "lunar-eclipse"}` → buffer-logger captures `level=WARN`, `event=v2.rekey.request.received`, `reason=lunar-eclipse` → no close, no outbound frame, session stays open.
  - `TestV2Session_OpenState_RekeyRequest_RecognisedReasons` — table-driven over `{scheduled, manual, compromise}`; each subtest asserts `level=INFO`, the correct `reason` field, no close, no outbound frame, state stays open. Each subtest constructs its own manager (single-shot cost is small).
  - `syncLogBuffer` is a mutex-wrapped `bytes.Buffer` (the manager's dispatch goroutine writes; the test goroutine reads). `waitForLogContains` polls the buffer for a substring against a deadline — the synchronisation knob for frames that don't emit an outbound envelope.

## Patterns established

- **Architectural-asymmetry partition pinned in three structurally-different places.** When a load-bearing absence ("constant X exists; X is NOT in set Y") must survive future contributions, three rails carry it: an instructive doc-comment on the constant itself ("do not add me to Y"), an instructive doc-comment on the gate it must stay out of ("v2 types like X must stay out"), and a deterministic test ("every constant must be in Y OR in an explicit allowlist; never both"). The first two catch the mistake at code-review time; the third catches it in CI. The advisory pair could be deleted without breaking the test, and the test could be removed without changing observable behaviour — but together they constitute the belt-and-suspenders pattern from the pipeline-wide principles, with the test being deterministic code and the two doc-comments being stochastic-rule rails for review. Generalise to every "constant X must NOT be a member of set Y" architectural boundary.

- **Test-local allowlist over a production-exported partition.** The `v2OnlyTypes` literal lives in `compat_test.go`, not in `envelope.go`. Production callers cannot accidentally import it; the only consumer is the partition test. The pattern: when a partition exists for structural enforcement (drift detection) rather than runtime dispatch, define the allowlist test-side. The alternative (exporting `V2OnlyTypeSet` from production) bloats the package's API surface and invites callers to use the v2 partition for dispatch logic — but v2 dispatch lives in `internal/relay/v2session.go` and switches on individual constants, not on set membership. The test-local home keeps the architectural separation crisp.

- **Probe-and-switch over refactor-the-callee.** When a new boundary check needs to run before an existing dispatch function (`dispatch.Route`) on shared input bytes, the architect explicitly chose a small re-decode (`json.Unmarshal` into a probe `Envelope`) over refactoring `Route` to accept a pre-decoded `Envelope`. The cost (one extra small JSON parse per frame) is well below the per-frame AEAD cost; the value is that `internal/dispatch` remains v1-pure and the v2 boundary check lives entirely inside the v2 manager. Generalise: when a caller-side discriminator can be cheaply prepended without restructuring the callee's signature, prefer the re-decode — the callee stays focused on its own concern and the caller carries the new architectural concept without coupling the two packages.

- **Logs-only handler for informational control envelopes.** `handleRekeyRequest` emits one structured log line and returns. No state change, no outbound frame, no close. The pattern generalises to any control-envelope class where the binary takes no transport action and the operator-visible value is solely the protocol-event observation. The natural alternative shapes — emit an `ack`, emit a sealed reply, mutate per-session state — all introduce wire surface for no behavioural value, and each invites a future "while we're here, let's also..." extension that breaks the informational contract. The pattern: when a control envelope is purely informational, the handler signature does **not** return an `Envelope` or anything else (Go zero return is the structural pin that the call site emits nothing).

- **Forward-compat WARN over reject for unknown enum values.** The closed set `{scheduled, manual, compromise}` could have been enforced by rejecting unknown reasons (sealing a `protocol.malformed` reply or closing the conn). Instead the binary logs at WARN and tolerates. The trade-off: rejecting would catch protocol drift loudly at the cost of forcing every future mobile-side `reason` addition to be a binary-side change in lockstep — exactly the coupling Mobile Protocol v2 was designed to avoid. WARN-and-tolerate is the correct posture for "future protocol value the binary doesn't recognise yet"; the WARN line is itself the drift signal for operators who watch the log channel. Generalise: when a closed enum is part of a wire-protocol contract that may evolve, the receiver's default arm tolerates with WARN rather than rejecting — the spec's forward-compat clause must be honoured by the receiver, not just the sender.

## Lessons learned

- **The same orphan-spec pattern as #452.** This slice was carved out of the abandoned `feature/449` super-slice, which closed `NOT_PLANNED` after the PO decided the rework was too coarse. The architect spec at `docs/specs/architecture/449-v2-rekey-responder.md` lives on the unmerged `feature/449` branch and contains verbatim design notes for the constant placement, doc-comment cross-references, and probe-and-switch shape that this slice ships. Re-deriving any of those decisions from scratch risks a subtle drift (e.g. placing the constant in a new file, or pushing the discriminator into `dispatch.Route` instead of `dispatchAppFrame`). Same lesson as [#452](452.md): the orphan spec is the canonical design document for the carved-out slice — lift the decisions verbatim, name the orphan spec in "Files to read first" so the developer doesn't re-derive.

- **Architect's "is it worth refactoring `Route`?" question has a structural answer in the package layering, not the LOC budget.** The natural temptation: "the re-decode is wasteful; refactor `Route` to accept a pre-decoded `Envelope` and the v2 manager passes its already-decoded `probeEnv`." Architect chose against it because the refactor would force `internal/dispatch.Route`'s signature to accept v2-knowledge (a pre-decoded `Envelope`) for one caller (the v2 manager); the v1 caller (`Dispatcher.handleOne`) would pay the same shape change for no value. The deeper rule: the cost of a refactor isn't just the moved lines; it's the architectural concept that leaks from the new caller into the existing callee's signature. When the leaked concept is one layer (v2) up the stack from the callee's home (v1 dispatcher), the leak is a regression. Generalise: weigh refactors not by LOC saved but by whose abstractions move into whose package.

- **The `_ context.Context` blank-identifier idiom is the canonical "kept for signature parity, unused locally" shape.** A reviewer might flag "why does `handleRekeyRequest` take `ctx` but never use it?" The answer in `handleNoiseInit` is the precedent — `ctx` is forwarded to `closeWith` but never used directly. `handleRekeyRequest` doesn't forward; the parity is purely with the dispatch-goroutine signature convention. The `_ context.Context` blank-identifier idiom makes the intent explicit at the call signature and doesn't trip `unused-parameter` linting (which Go doesn't have on parameters anyway). Generalise: when a method's signature is shaped by uniformity-with-siblings rather than per-method need, use `_ T` rather than `t T` so a future reader knows the parameter is structurally required, not accidentally dropped.

## Out of scope (filed elsewhere or deferred)

- **The re-key responder swap path itself** — `handleRekeyInit`, atomic `CipherState` swap, peer-static continuity check (reading the `V2Session.peerStatic` captured by [#452](452.md)), AEAD-mismatch teardown verification. **[#453](453.md)**, sibling slice; different function in the same file, ships in parallel.
- **The binary's own emit of `rekey_request`** — #450. The outbound shape reuses the `protocol.TypeRekeyRequest` constant landed here.
- **Re-key timer + scheduler** — #435 super-ticket. #453 and #454 are the concrete slices; the timer that emits scheduled `rekey_request` is its own future slice.
- **Per-`reason` severity escalation.** `compromise` does NOT log at a higher severity than INFO in this slice — out of scope per the ticket. If operator-visible "phone-side keys compromised" alerting becomes a requirement, a future slice could lift `compromise` to ERROR or WARN.
- **Rate-limiting `rekey_request` receipt.** A flood of `rekey_request` envelopes burns one re-decode + one log call per frame; well within the existing per-frame cost envelope. No rate limit needed today.
- **A generalised "v2-control discriminator table"** in `dispatchAppFrame` — refactoring the inline probe-and-switch into a table-driven dispatcher of v2 control types. Today there's exactly one v2 control type (`TypeRekeyRequest`); the table is premature. When a third v2 control type lands (the second is the binary's own outbound `rekey_request` per #450, which reuses the same constant — same type, different direction), the discriminator graduates to a table.
- **`compat_test.go`'s third assertion** — naming each `v2OnlyTypes` member as having a paired probe-and-switch consumer in `dispatchAppFrame`. Same logic: deferred until the discriminator generalises.
