# Ticket #509 — `agentrun/jsonl/tail` delegates path resolution + file-appear wait to tuidriver

First spine migration of `internal/agentrun/jsonl/tail/watcher.go` onto the tui-driver library surface. Replaces the local `~/.claude/projects/<encoded>/<sid>.jsonl` join with [`tuidriver.SessionJSONLPath`](https://github.com/pyrycode/tui-driver/blob/main/pkg/tuidriver/jsonl.go) and the hand-rolled `openWithRetry` / `probeRetryDelays = {0, 50ms, 200ms}` race-absorption preamble with [`tuidriver.WaitForSessionJSONL`](https://github.com/pyrycode/tui-driver/blob/main/pkg/tuidriver/jsonl.go) (polls at `DefaultPollInterval = 50ms`). The fsnotify-based WRITE pump stays verbatim — a later ticket ([#512](https://github.com/pyrycode/pyrycode/issues/512)) replaces the per-line read loop with `tuidriver.TailJSONL` and deletes the package. Net diff: production file shrinks ~30 LOC (deletes `probeRetryDelays`, `openWithRetry`, the `io/fs` + `time` imports, the in-loop `file==nil + CREATE` branch); test file adds two new cases (`TestWatcher_WaitTimeout`, `TestWatcher_SpecialCharWorkdir`). Sibling [#508](https://github.com/pyrycode/pyrycode/issues/508) (inline `tuidriver.EncodeCwd`) and follow-up [#512](https://github.com/pyrycode/pyrycode/issues/512) (replace the read loop) form the rest of the migration plan; this ticket is the first of three and intentionally surgical.

## Implementation

- **Step zero — `go.mod` bump.** `go get github.com/pyrycode/tui-driver@latest && go mod tidy` moves the pin from `v0.0.0-20260519122208-b09fe70e60a7` (2026-05-19) to `v0.0.0-20260523181457-c2dcd1e49992` (2026-05-23). The new version postdates tui-driver #58 (commit `a2edf4f`, 2026-05-22) which is what exports `SessionJSONLPath` + `WaitForSessionJSONL`. AC #1 satisfied. Nothing else in the migration compiles until this lands, so step zero is non-optional.

- **`tail.New` (lines 89–139 → ~109 LOC).** The local join `filepath.Join(home, ".claude", "projects", encoded)` is replaced with `expectedPath := tuidriver.SessionJSONLPath(home, resolved, cfg.SessionID)` + `dir := filepath.Dir(expectedPath)`. The `MkdirAll(dir, 0o700)` and `fsnotify.NewWatcher` + `fsw.Add(dir)` calls stay verbatim. The `expectedPath` field of `Watcher` is set from `tuidriver.SessionJSONLPath`'s return value rather than a separate join, so the test seam and the production path agree by construction.

- **`agentrun.ResolveWorkdir` retained — load-bearing.** `tuidriver.SessionJSONLPath` is a pure join + `tuidriver.EncodeCwd` internally; it does NOT resolve symlinks. Claude's on-disk encoder *does* resolve before encoding (empirically pinned by `TestEncodeProjectDir_DarwinRealpath` and `docs/lessons.md` § "fsnotify reports as-watched, kernel probes report canonicalised"). On macOS, a `t.TempDir()` workdir under `/var/folders/...` must encode as `-private-var-folders-...`, not `-var-folders-...`. Calling `agentrun.ResolveWorkdir(cfg.Workdir)` before passing to `tuidriver.SessionJSONLPath` keeps the macOS-realpath invariant. The code-review further pinned a second reason: `tuidriver.EncodeCwd` internally calls `canonicalisePath` but silently encodes the input as-passed on resolution failure (no error returned); the explicit `ResolveWorkdir` call upstream preserves the pre-existing "error on missing workdir" contract — a typo'd workdir surfaces `tail: resolve workdir: <fs.ErrNotExist>` from `New` rather than silently watching a wrong directory. The spec's stated reason (only the symlink invariant) is incomplete but the design choice is correct.

- **`tail.Watcher.Run` (lines 146–203 → ~150 LOC after edit).** Three pre-existing code blocks collapse into one up-front wait + a slimmer event loop:
  1. Lines 154–165 (initial `os.Stat` to absorb the pre-existing-file race + the `fs.ErrNotExist` branch falling through to the event loop) → deleted.
  2. Lines 179–186 (the in-loop CREATE branch that called `openAndDrain` when `file==nil`) → deleted.
  3. Lines 187–195 (the WRITE pump) → preserved verbatim.

  The new shape is `if err := tuidriver.WaitForSessionJSONL(ctx, w.expectedPath); err != nil { return err }` → `openAndDrain` once → event loop runs `drain` on `Write|Create`. `fsnotify.Add(dir)` is still called in `New`, so any WRITE events that race the wait are buffered in `fsw.Events` (4096-deep default channel) and consumed by the event loop after the initial drain.

- **Event-loop CREATE half kept defensively.** The single remaining event branch is `if ev.Op.Has(fsnotify.Write) || ev.Op.Has(fsnotify.Create) { drain() }`. The `Create` half is a belt against "CREATE arrives after `WaitForSessionJSONL` returned" (e.g. test wrote the file synchronously and the queued CREATE event lands after the loop starts). Calling `drain()` on a reader already at EOF is a no-op and costs nothing — cheaper than reasoning about which event sequences must be filtered.

- **`openAndDrain` simplified — bounded retry deleted.** Pre-#509 signature `openAndDrain(ctx)` ran `openWithRetry(ctx, w.expectedPath)` walking `probeRetryDelays`; on `fs.ErrNotExist` exhaustion it returned `(false, nil)` and logged at Warn. Post-#509 signature is `openAndDrain()` (no `ctx` — wait already covered cancellation), body is `f, err := os.Open(w.expectedPath); if err != nil { return false, fmt.Errorf("tail: open %s: %w", w.expectedPath, err) }`. The `fs.ErrNotExist`-after-CREATE Warn-log path is gone because `WaitForSessionJSONL` returning nil guarantees the file exists right now; a TOCTOU window between the wait returning and `os.Open` succeeding is one-shot and theoretical (claude is append-only during a turn, no delete-then-recreate). If `os.Open` errors, surface it wrapped and bail.

- **`openWithRetry` (lines 270–294) and `probeRetryDelays` (lines 27–31) deleted entirely.** ~40 LOC removed. The `io/fs` and `time` imports go too (no remaining consumers in the file).

- **`Config.Workdir` doc-comment reworded.** Dropped the `agentrun.EncodeProjectDir` reference: "The watcher resolves Workdir and encodes it the same way claude does to locate the per-session JSONL under `~/.claude/projects/`." Pre-#509 named the helper by name; post-#509 the encoding lives in tuidriver and the doc-comment matches.

- **`Watcher.file` / `Watcher.reader` doc-comment NOT updated — stale.** Code review NIT #1: the field comment at `watcher.go:69-72` still says "populated lazily, either by the initial stat in Run or by the first CREATE event." The first-CREATE-event arm no longer exists (population is now a one-shot in `openAndDrain` after `WaitForSessionJSONL` returns). Carried as a follow-up rather than fixed in this PR's diff — the reviewer flagged it as a NIT and landing was approved without it. Listed under § Out of scope below for the next contributor touching the file.

- **Test seam — `expectedEncodedDir` helper.** Pre-#509 signature `expectedEncodedDir(t, home, workdir)` computed `filepath.Join(home, ".claude", "projects", agentrun.EncodeProjectDir(workdir))`. Post-#509 signature is `expectedEncodedDir(t, home, workdir, sessionID)` with body `filepath.Dir(tuidriver.SessionJSONLPath(home, resolved, sessionID))` — mirrors the production code path exactly so the test and the watcher agree on the directory. All four call sites updated (`TestNew_RealpathEncoding`, `TestWatcher_LateCreate`, `TestWatcher_ExistingFile`, `TestWatcher_FixtureIntegration`); the latter's inlined `agentrun.EncodeProjectDir` + `filepath.Join` was folded into the helper too, so the test file now has exactly one place that knows how to compose the encoded dir — mirroring the production code's single source of truth.

- **`TestWatcher_WaitTimeout` (new).** Verifies AC #4's "tuidriver wait-error path (timeout)". Creates a workdir + home but never writes the JSONL; constructs the watcher manually (not via `startWatcher`, which uses `context.Background`); calls `Run(ctx)` with `context.WithTimeout(context.Background(), 150ms)`; asserts `errors.Is(err, context.DeadlineExceeded)` matches through `WaitForSessionJSONL`'s `fmt.Errorf("... %w", context.Cause(ctx))` wrap and that `Run` returns within 500ms of the deadline. The 150ms deadline gives 3× headroom over the 50ms poll interval — fast but robust on a loaded CI runner.

- **`TestWatcher_SpecialCharWorkdir` (new).** Verifies AC #4's "path correctness across special-char cwds (spot-check at the seam)". Does NOT exhaustively re-test the byte rule — tuidriver's own `cwd_test.go` owns that table; this is the single differential probe the AC asks for at the seam.

## Patterns established

- **Migrate from hand-rolled race absorption to a library wait that returns through `%w`-wrapped `context.Cause` — `errors.Is` matchers survive the wrap.** Pre-#509 the watcher absorbed the CREATE-before-ready race with `openWithRetry` walking `probeRetryDelays = {0, 50ms, 200ms}` and `ctx.Done` checks between sleeps; the cancellation path returned the bare `ctx.Err()`. Post-#509 `tuidriver.WaitForSessionJSONL` polls at `DefaultPollInterval = 50ms` and on cancellation returns `fmt.Errorf("session jsonl %s did not appear: %w", path, context.Cause(ctx))`. The caller's `errors.Is(err, context.Canceled)` / `errors.Is(err, context.DeadlineExceeded)` assertions continue to work because `%w` makes the wrap transparent. Generalises whenever delegating a polling preamble to a library: as long as the library wraps the underlying sentinel with `%w` (or returns it directly), `errors.Is` callers downstream don't need code changes — but the test that pins the matching behaviour is what proves it. `TestWatcher_ContextCancellation` is that pin here; it stays green across the migration without edits, which is the load-bearing signal that the contract held.

- **When delegating to a "pure" library helper that internally calls a resolver, keep your own explicit resolver call upstream if (a) your resolver fails closed and the library's fails open, OR (b) the library doesn't perform the resolution at all.** `tuidriver.SessionJSONLPath` is documented as a pure join — it calls `tuidriver.EncodeCwd` but not `EvalSymlinks`. `tuidriver.EncodeCwd` internally calls `canonicalisePath` but silently encodes the input as-passed on resolution failure (no error). Pyrycode's `agentrun.ResolveWorkdir` fails closed on missing workdirs. Keeping the explicit `agentrun.ResolveWorkdir(cfg.Workdir)` call before `tuidriver.SessionJSONLPath` preserves both invariants: the macOS realpath rule (so the encoded dir matches what claude writes to) AND the fail-closed-on-typo error contract. Two reasons to keep a call upstream of a delegation — only one was in the spec; the second emerged in code review and is the more durable one. Generalises whenever a downstream library helper trades silent fallback for ergonomics: audit *both* the semantic invariant and the failure surface before deleting the upstream caller.

- **Defensive `fsnotify.Op.Has(Create)` survives even when the explicit CREATE branch is removed.** The pre-#509 event loop had two branches: `file==nil + Create → openAndDrain` and `file!=nil + (Write|Create) → drain`. Post-#509 only the second branch remains. The `Create` half of the `Op.Has` check is no longer reachable in the normal flow (the file is open by the time the loop runs), but keeping it is cheap — a queued CREATE event that lands after `WaitForSessionJSONL` returned (e.g. test wrote the file synchronously) triggers `drain()` on a reader already at EOF, which is a no-op. The alternative (filter CREATE out and only match Write) would require reasoning about which event sequences are possible; defaulting to "call `drain()` whenever fsnotify reports something" trades one syscall for zero invariants to maintain. Generalises whenever collapsing two event-loop branches into one: if the surviving handler is idempotent on the now-unreachable input, keep the broader match — the noise cost is zero and the reasoning cost saved is real.

## Lessons learned

- **The spec named only one reason to keep `agentrun.ResolveWorkdir`; code review found the second, more durable, reason.** The architect spec said "we still call `ResolveWorkdir` because `tuidriver.SessionJSONLPath` does NOT resolve symlinks (pure join)." That's true, and it pins the macOS realpath invariant. But code review pointed out that `tuidriver.EncodeCwd` internally calls `canonicalisePath` *and silently encodes the input as-passed on resolution failure (no error returned)*. So the spec's stated reason is technically incomplete: even if `SessionJSONLPath` did resolve, dropping `ResolveWorkdir` upstream would turn pyrycode's pre-existing "error on missing workdir" contract into a silent "watch a wrong directory" failure. Recording this because the pattern recurs: when a spec justifies retaining an upstream caller with one semantic invariant, audit whether there's *also* an error-contract invariant — those tend to be the more durable reasons. The codebase doc for #509 captures both (see § Patterns established above) so the next contributor pondering "can we delete `ResolveWorkdir` now?" has the full picture, not just the spec's half.

- **AC #5 was not fully exercised in this run — `make e2e-realclaude` skipped, justifiably.** The PR body notes `make check` (vet + staticcheck + test -race) was green but `make e2e-realclaude` was skipped because no `ANTHROPIC_API_KEY` / `CLAUDE_CODE_OAUTH_TOKEN` was present in the environment. Code review accepted this on the grounds that "the watcher path here is the same one ptyrunner depends on, so a regression would surface there immediately" — but it's worth recording that the byte-equivalence gate (#506) only fires when credentials are present, and a contributor without them is relying on CI / sibling integration to catch failures. Generalises: AC bullets that gate on credentialed integration tests should be paired with a hermetic unit signal whenever feasible — `TestWatcher_WaitTimeout` and `TestWatcher_SpecialCharWorkdir` are exactly that hermetic pair for the AC #4 asks; AC #5's e2e gate doesn't have one in this slice, and that's worth flagging when the next migration step ([#512](https://github.com/pyrycode/pyrycode/issues/512)) lands.

- **The two-line ticket-body footnote about sibling [#508](https://github.com/pyrycode/pyrycode/issues/508) was load-bearing for the implementation order.** PO's "Sibling #508 overlaps in code — both touch the `EncodeProjectDir` call site at `watcher.go:115`" was the right framing: #509 deletes that exact call site, so the second to land sees an already-removed line regardless of order. No coordination required, no merge conflict by construction. Recording this as a confirmation of the planning shape — when two tickets overlap at a single deletion point, the natural answer is "land in any order, the second is a no-op" rather than scheduling them sequentially. The architect's spec hedged on this only insofar as "if #508 lands first and removes `ResolveWorkdir` too, inline `filepath.Abs` + `filepath.EvalSymlinks` here" — that defensive footnote went unused because #508 was still open at #509's pickup time.
