# Ticket #684 — per-session spawn workdir on the Pool (`CreateIn` / `GetOrCreateIn`)

The pool-level primitive for EPIC #672's per-conversation-workdir work (split from #681). Until #684 every session the pool spawned ran in the single shared `tpl.WorkDir` (the daemon's trusted bootstrap workdir): `buildSession` hard-wired `supervisor.Config{ WorkDir: tpl.WorkDir }` and neither `Pool.Create` nor `Pool.GetOrCreate` accepted a per-session directory. #684 adds the surface — `CreateIn` / `GetOrCreateIn` siblings that carry an explicit per-session spawn workdir, threaded through the shared `buildSession` seam into `supervisor.Config.WorkDir`. An empty workdir falls back to `tpl.WorkDir`, so `Create` / `GetOrCreate` become thin delegators with byte-identical behaviour for every existing caller. This slice supplies **no** non-default caller — the pool treats the path as **opaque** (no validation, canonicalisation, or trust handling); the trusted-directory consumer is #685.

See the evergreen feature doc: [features/sessions-package.md § Per-session spawn workdir: `CreateIn` / `GetOrCreateIn` (#684)](../features/sessions-package.md#per-session-spawn-workdir-createin--getorcreatein-684).

## Implementation

2 production `.go` files + 1 new test file; +237/-5. No new `Session` field, no registry-schema change, no new exported types or sentinels.

- **`CreateIn(ctx, label, spawnDir)` / `Create` delegator** (`internal/sessions/pool.go`). `Create`'s mint → build → persist → supervise → activate body moved verbatim into `CreateIn`, with its one call site `buildSession(id, label)` → `buildSession(id, label, spawnDir)`. `Create` shrank to `return p.CreateIn(ctx, label, "")`. No create/persist/supervise logic is duplicated.
- **`GetOrCreateIn(ctx, id, label, spawnDir)` / `GetOrCreate` delegator** (`internal/sessions/get_or_create.go`). Same move-the-body treatment; `GetOrCreate` delegates with `""`. `spawnDir` is applied **only on the create path** — on the take path (session already registered) it is ignored, the existing session keeps its own workdir, mirroring how the take path already drops the caller's `label`.
- **`buildSession` spawn-seam conditional** (`internal/sessions/pool.go:974`). The single behavioural change: `workDir := tpl.WorkDir; if spawnDir != "" { workDir = spawnDir }`, then `supervisor.Config{ WorkDir: workDir, ... }`. `buildSession`'s only two callers are `CreateIn` and `GetOrCreateIn` (both updated); the other `buildSession` mentions in the tree are comments.
- **Survives respawn with no new state.** The per-session workdir lives only in `supervisor.Config`, which the supervisor reads as `cmd.Dir` on **every** (re)spawn (`spawn.go:40-41`, `supervisor.go:638-639`). A custom spawn dir therefore survives child crash-respawns automatically — no `Session` field, no registry persistence needed.
- **Tests** (`internal/sessions/pool_spawndir_test.go`, new). A **cwd-recorder fake claude** observes the child's working directory without adding a production accessor: the template runs `/bin/sh -c 'pwd > "cwd-$2.txt"; exec sleep 3600' --`, and because `buildSession` appends `--session-id <uuid>` the uuid lands at `$2`, so each child writes a per-uuid marker (`cwd-<uuid>.txt`) **into its own cwd**. The marker's *location* (existence-check, not content-compare — sidesteps the macOS `/tmp`→`/private/tmp` symlink rewrite) is the proof of the spawn directory; the per-uuid name distinguishes a session's marker from the bootstrap's `cwd-.txt` in a shared dir. Cases (all `t.Parallel()`): `CreateIn` explicit-dir (AC-1), plain `Create` default-leg in the template workdir (AC-2 + AC-4), `GetOrCreateIn` create-path, and `GetOrCreateIn` take-path-ignores-spawnDir (negative — no marker appears in the second dir).

## Patterns established

- **Optional plumbing → `XxxIn` sibling method, not functional options.** Three mechanisms were on the table (variadic functional option, extra `Create` parameter, threaded field). The codebase has **zero** functional-options precedent (`grep "Option func(" internal/ cmd/` → empty) but **does** have the `XxxIn` idiom (`StartIn(t, home, ...)`, `internal/e2e/harness.go:201` — "the same operation, but in a given directory"). `CreateIn(ctx, label, spawnDir)` = "create, but spawn in `spawnDir`" is the exact analog. Sibling methods keep the public `Create` / `GetOrCreate` signatures byte-identical, so the whole AC-3 call-site set (`sessionMinter` `cmd/pyry/main.go:667`, the `sessions.new` verb `control/server.go:542`, `GetOrCreate` `control/server.go:797`, the `create_conversation` interface caller, all tests) compiles and behaves unchanged with zero churn. Introducing an options framework for a single optional string would have violated "Simplicity first."
- **Expose the capability on the shared seam, on both public entry points.** Threading `spawnDir` through `buildSession` (the seam shared by `Create` and `GetOrCreate`) means both public entry points get the capability, so the consumer slice (#685) picks whichever it needs without reopening this seam. The marginal cost is one extra ~3-line delegator.
- **Observe a child's cwd behaviourally, not via a production accessor.** The cwd-recorder fake claude proves the spawn directory by the *location* of a per-uuid marker file the child writes via a relative path — no `Session.WorkDir()` getter added just for the test.

## Lessons learned

- **A per-session workdir needs no new `Session` field or registry column because `supervisor.Config.WorkDir` is already retained and re-read on every respawn.** Looking for "where does the workdir get persisted so it survives a restart" is the wrong question for this slice — the supervisor keeps it for the child's whole lifetime (including crash-respawns), and no session-reload path reconstructs live supervisors via `buildSession` today (`grep` for load/reload/recover on `*Pool` → none). A custom spawn dir is a spawn-time input only; surviving a **daemon process** restart (vs a child respawn) would be its own ticket and is out of scope.
- **An opaque-path primitive is deliberately not `security-sensitive`.** No untrusted input reaches this slice and its only caller after it still passes the default. An inaccessible directory surfaces at spawn time via the supervisor's existing chdir-failure → backoff path, not via a pool-side `os.Stat`. The trust / canonicalisation / `$HOME`-containment work — and the label — live in the consumer #685.

## Out of scope (deferred)

- **Validation / canonicalisation / trust / `$HOME`-containment** of the supplied directory — #685 (which carries `security-sensitive`).
- **Wiring an actual non-default caller** — #685 (spawn in a validated, trusted per-conversation `Cwd`) and #686 (the per-`Cwd` turn bridge).
- **Registry persistence of the per-session workdir** (survival across a daemon *process* restart). Spawn-time input only today; a future slice if needed.
