# Ticket #594 — `WriteUserTurn` delivers via `Session.DeliverPrompt` (ready-gate + commit-confirm + recovery)

Rewrites `Supervisor.WriteUserTurn` (`internal/supervisor/supervisor.go`) from fire-and-forget
`Session.AttachInput` to a reliable delivery: ready-gate (`WaitReady`) → commit-confirm + corrupted-
paste recovery (`DeliverPrompt`) → return `nil` **only** when claude confirms the turn committed. The
two former silent-success cases — no live session, and an attached-but-not-ready claude — become
**loud failures**, so the relay `send_message` handler reports a retryable failure to the phone
instead of acking a turn that never reached claude. This is task **T3** of Phase 5 / Phase 1 in
[ADR 025](../decisions/025-mobile-remote-head-interactive-session.md), building directly on
[#593]'s supervisor-owned [tui-driver](https://github.com/pyrycode/tui-driver) `Session` (which
deliberately deferred this upgrade) and tui-driver `v1.2.0`'s `DeliverPrompt` / `WaitReady`.

`security-sensitive`: this is the delivery path for untrusted, phone-originated turn payloads
(`send_message` handler → `WriteUserTurn` → live claude). Spec security review: **PASS** (no MUST FIX).

## Implementation

### The rewrite (`WriteUserTurn`)

- **Signature gains a `ctx`:** `WriteUserTurn(ctx context.Context, id string, payload []byte) error`.
  First two steps unchanged — `ValidateConversation` runs first (result propagated verbatim, cursor
  **not** stamped on refusal), then `currentConvID = id` is stamped under `convMu` **before** any
  delivery (so the cursor reflects the attempt even on the loud-failure path).
- **Capture-then-unlock:** captures `sess := s.sess` under `sessMu`, **releases the lock**, then
  delivers on the captured pointer. The old code held `sessMu` across the fast `AttachInput` write;
  `WaitReady + DeliverPrompt` can run for seconds (idle-wait + up to 3 × 3 s commit retries), and
  holding `sessMu` that long would block `runOnce`'s teardown `setSession(nil)`.
- **No silent drop:** `sess == nil` → `fmt.Errorf("supervisor: write user turn: %w", ErrNoLiveSession)`
  (was `return nil`). Otherwise `s.deliverFn(ctx, sess, payload)`; any error wraps with the stable
  prefix. Success → `nil`.

### The `deliverFn` seam (testability under the substrate seal)

`WriteUserTurn` does not call `WaitReady`/`DeliverPrompt` directly. It calls through an unexported
field set once in `New` and overridden only in tests (the same injection seam as `helperEnv`):

```go
deliverFn func(ctx context.Context, sess *tuidriver.Session, payload []byte) error
```

Immutable post-`New`, so `WriteUserTurn` reads it lock-free. The production value is
`deliverViaSession`:

```
WaitReady(ctx)            → err: return fmt.Errorf("wait ready: %w", err)
DeliverPrompt(ctx, DeliverOpts{Prompt: string(payload), Logger: s.log})
                          → err: return err            // already prefixed by tui-driver
!DeliverResult.Committed  → return ErrTurnNotCommitted
                          → nil
```

The seam exists because the happy path and recovery branches are driven by claude's *screen* (idle
prompt, thinking spinner, `[Pasted text]` chip) — literals that live inside tui-driver and **must not
enter pyrycode** (`cmd/substrate-guard`, AC #5). A pyrycode test cannot fake a claude screen without
smuggling a screen literal past the guard; the seam fakes one level *above* the screen-parsing layer,
returning an already-classified `error`/success, so every branch is unit-testable with zero screen
literals.

### Two new exported sentinels

- `ErrNoLiveSession` — no session registered at delivery time (the former silent-drop case).
- `ErrTurnNotCommitted` — `DeliverResult.Committed == false` after the bounded recovery.

`WaitReady`/`DeliverPrompt` errors are wrapped, not replaced, so `errors.Is(err,
context.DeadlineExceeded)` / `context.Canceled` stay matchable through the chain.

### `JSONLPath: ""` (evidence-based)

`DeliverOpts.JSONLPath` (an optional file-appearance commit signal) is left empty. The relay-driven
bootstrap session spawns with `--continue`, **not** `claude --session-id <uuid>` (pool.go defers
`--session-id` to "Phase 1.1+"), so the supervisor holds no stable claude session UUID to build
`SessionJSONLPath(home, cwd, sessionID)` from — and claude rotates its on-disk UUID on `/clear`
anyway. With the `WaitReady` idle gate in front, the spinner commit signal is sufficient; the JSONL
file is redundant. (ptyrunner *can* use `JSONLPath` precisely because it spawns with a pinned
`cfg.SessionID`.) This also avoids the path-traversal/TOCTOU surface `SessionJSONLPath` would add.

### Call-site cascade (ctx threading) — 3 production sites

1. `supervisor.WriteUserTurn` — method rewrite (above).
2. `sessions.Session.WriteUserTurn(ctx, conversationID, payload)` — delegates `ctx` to the supervisor.
3. `handlers.TurnWriter` interface + the `send_message.go` call site.

`cmd/pyry/assistant_turn.go`'s `cursorReader` interface only declares `CurrentConversation()` — **not**
a ctx-threading site.

### Handler: loud failure → retryable wire reply (`send_message.go`)

Two changes beyond ctx threading:

- **Bound the delivery:** `deliverCtx, cancelDeliver := context.WithTimeout(ctx,
  sendMessageDeliverTimeout)` (new const, `30 * time.Second`, matching `sendMessageActivateTimeout`).
  `WaitReady` blocks while claude is busy, so an unbounded ctx would hang the per-conn goroutine on a
  long claude turn.
- **Map failure → `server.binary_offline` (retryable):** `dispatch.Route` does **not** synthesise a
  reply on a bare handler error (it logs WARN, leaves the conn up), so `return err` is *silent* to the
  phone. The error switch becomes: `nil` → ack; `ErrConversationNotFound` → `conversation.not_found`
  (non-retryable); `context.Canceled && ctx.Err() != nil` → propagate (parent conn closing, no doomed
  reply); **`default`** → `replyError(…, CodeServerBinaryOffline, msgServerBinaryOffline, true)` (was
  `return err`). A deliver-timeout is `context.DeadlineExceeded` (not `Canceled`), so it correctly
  lands in `default`. The handler matches only error values it **already imports** (`context`,
  `conversations`); the supervisor's new sentinels fall into `default`, keeping `handlers/` free of the
  `internal/supervisor` import.

## Patterns established

- **Seam one level above the screen, not at it.** When a behaviour is driven by claude's screen but
  pyrycode must stay screen-literal-free (substrate seal), inject a fake *above* the screen-parsing
  layer that returns classified errors/success (`deliverFn`), not a fake claude screen. This is the
  CODING-STYLE "interface only when there's a testing need" justification, satisfied concretely.
- **Capture-then-unlock for slow work on a per-iteration handle.** Capture the `sess` pointer under
  `sessMu`, release, then run the multi-second `WaitReady + DeliverPrompt` on the captured pointer —
  the same "capture the channel reference under `sessMu`, await it unlocked" shape `WaitForPTY` uses.
  Holding a leaf lock across a seconds-long operation would block teardown.
- **Teardown safety replaces lock-held exclusion.** Releasing `sessMu` before delivery means a
  concurrent `setSession(nil)+Close()` can race the captured pointer. Safe because tui-driver `Session`
  methods are teardown-safe by contract (`AttachInput`/`writeRaw` return the closed-file error, **no
  panic**; `Snapshot` reads a frozen buffer) — the same property the #593 `Resize` EBADF defusal
  established. A mid-delivery `Close` lands in the PTY-error path → loud failure, never a crash or
  false ack. This **relaxes** (without breaking) #593's `setSession(nil)`-before-`Close()` ordering.
- **Loud-failure mapping is the consumer's job.** The supervisor returns Go sentinels; the handler maps
  every transient delivery failure to one retryable wire code (`server.binary_offline`) at the call
  site via `errors.Is` — consistent with the repo's
  [refusal-to-wire-code convention](../../PROJECT-MEMORY.md). No new protocol constant was minted.

## Lessons learned

- **Changing a delivery primitive's success contract silently invalidates fakeclaude-backed e2e
  tests.** Lifted from the PR's Lessons-learned: moving from silent-ack to commit-confirm broke every
  `//go:build e2e` `send_message` test that asserts a `TypeAck` against `fakeclaude` — `fakeclaude`
  renders no claude TUI (no idle prompt / spinner), so a `WaitReady` idle-gate can never confirm
  against it. Five such tests were `t.Skip`-ped referencing follow-up [#603]. These run in **no**
  automated gate (`make check` / CI exclude `-tags e2e`), so the breakage is invisible to the green
  gate — worth checking the `-tags e2e` suite whenever a supervised-write contract changes.

## Testing

Test-first (RED → GREEN); stdlib `testing`, table-driven, `-race`. The `deliverFn` seam makes the
supervisor-side branches deterministic without a live claude or any screen literal.

- **The two former silent-success cases (AC #4 anchors):**
  - *No live session* → `NoPTYDrops` **inverted** to `NoSessionFailsLoud`: no session registered →
    `errors.Is(err, ErrNoLiveSession)` + the `"supervisor: write user turn:"` prefix; cursor *was*
    still stamped (stamping precedes the session check).
  - *Attached but not ready* → `NotReadyFailsLoud`: register `&tuidriver.Session{}`, override
    `deliverFn` to return `wait ready: context.DeadlineExceeded` → `errors.Is(err,
    context.DeadlineExceeded)`.
- **Seam-driven branch coverage:** `CommittedReturnsNil` (`deliverFn` → nil ⇒ nil),
  `NotCommittedFailsLoud` (`ErrTurnNotCommitted`, wrapped), `DeliverErrorFailsLoud` (plain write error,
  wrapped + `errors.Is`).
- **Real-machinery not-ready (repurposed `HappyPath`):** `RealDeliverNotReadyFailsLoud` runs the
  *production* `deliverViaSession` (no override) against a real spawned fake child that never renders
  claude-idle; a 600 ms `deliverCtx` expires in `WaitReady` → loud failure. Proves the real ready-gate
  is wired. (The old happy-path assertion — payload reaches the child's stdin via `AttachInput` — is no
  longer how delivery works; a generic fake child never reaches claude-idle, so the real path correctly
  fails.)
- **Regression updates (AC #3):** `CursorReadBack`, `UnknownIDDoesNotMutateCursor`, `NilValidatorSkips`
  flip only their no-session **return-value** assertion (now `ErrNoLiveSession`); their load-bearing
  cursor assertions (stamp-before-check, non-mutation on refusal) are unchanged.
- **Handler tests:** `stubTurnWriter.WriteUserTurn` gains `ctx`; `WrappedError_PassesThroughNoWireReply`
  **inverted** → asserts a `default`-class delivery error now produces a `server.binary_offline`
  retryable envelope (no false ack). `Success_EmitsAck`, `ConversationNotFound`, `ActivateTimeout`,
  `MalformedPayload`, `HandlerCtxCanceled` verified still passing on the new signature.
- **Gate:** `make check` green (vet + `-race` + staticcheck + `cmd/substrate-guard` — no claude-screen
  literal entered pyrycode; the seam returns classified errors, all screen knowledge stays in
  tui-driver).

## Out of scope (deferred)

- **[#603] — teach `fakeclaude` the idle/thinking glyphs** (+ substrate-guard allowlist, mirroring the
  `ptyrunner/helper_test.go` exemption) *or* migrate the five skipped e2e `send_message` tests to the
  real-claude suite. An architect-owned seal decision.
- **Real-claude integration** of the genuine `WaitReady → DeliverPrompt → Committed` path
  (`internal/e2e/realclaude`, build-tagged, not in `make check`) — the e2e harness's territory.
- **Busy-claude queue semantics.** `WaitReady` waits for idle, so a turn sent mid-turn blocks until
  idle or `sendMessageDeliverTimeout` elapses (→ retryable). Queue-behind-current-turn is a separate
  ticket. `sendMessageDeliverTimeout = 30s` is a tuning knob, not a contract.
- **Finer-grained wire codes.** All loud failures map to `server.binary_offline`; distinguishing "claude
  busy" from "session torn down" needs a neutral sentinel package (to keep `handlers/` import-free) and
  a new protocol code — deferred until an observed need.
- **Multi-writer arbitration** on a shared operator-attached head (a spinner from operator activity
  between `WaitReady` returning idle and `DeliverPrompt`'s poll could read as our commit) — inherent to
  screen-based commit detection on a shared head; out of scope for the silent-drop fix.
