# Ticket #538 ŌĆö `agent-run`: `--permission-mode dontAsk` in ptyrunner `buildArgs`

The ptyrunner production path (cutover in [#470](https://github.com/pyrycode/pyrycode/issues/470)) has been passing `--permission-mode default` on every claude spawn. Per the [claude CLI reference](https://code.claude.com/docs/en/cli-reference) ŌĆö *"The `--permission-mode` flag ŌĆ” overrides any `defaultMode` settings found in configuration files"* ŌĆö argv `default` silently shadowed the deny-default `defaultMode: "dontAsk"` that [`settings.WriteSettings`](../features/agentrun-settings-subpackage.md) writes (the JSON-side literal was itself only corrected from `"deny"` ŌåÆ `"dontAsk"` in [#487](487.md)). Argv enum `default` is "prompt the user before each tool" ŌĆö not a sentinel for "consult settings file." So every interactive-TUI spawn since #470 has been deny-default in name only; out-of-allow-list tool calls landed in prompt-mode, which silently drops under headless `claude -p` because no operator is there to answer. The per-agent allow list the dispatcher claimed to enforce was advisory. Production-impact half of [#537](https://github.com/pyrycode/pyrycode/issues/537)'s two-bug pair; the test-credibility half (selfcheck exhibit on `dontAsk`'s read-only-Bash carveout) is deferred to a sibling ticket and depends on this landing first so its real-claude PASS verification means anything.

## Implementation

- **One literal in `buildArgs`.** `internal/agentrun/ptyrunner/runner.go` `buildArgs` (line 451) now writes `--permission-mode dontAsk`. Argv shape is otherwise byte-identical ŌĆö same six pairs (`--session-id`, `--settings`, `--permission-mode`, `--append-system-prompt-file`, `--model`, `--effort`), same forbidden-flag set (`--input-format`, `--output-format`, `--verbose`, `--dangerously-skip-permissions`, `--max-turns`, `--allowed-tools`), same ordering. `TestBuildArgs` in `runner_test.go` flips the matching expected literal; the forbidden-flag loop and length pin remain unchanged. No new test added ŌĆö the existing position-by-position fixture already pins the argv shape; a "dontAsk-specifically" test would duplicate it.

- **Three stale-comment fixups (mechanical, doc-hygiene only).** After the production change the string `"--permission-mode default"` was wrong in three call-site descriptions: (1) `internal/agentrun/selfcheck/selfcheck.go` package doc (`... \`--settings <path> --permission-mode default\` argv pair ...`); (2) `cmd/pyry/agent_run_selfcheck.go` operator-facing FAIL message (`passed via --settings <path> --permission-mode default`); (3) `cmd/pyry/agent_run_test.go` explanatory comment on `TestBuildStreamRunnerClaudeArgs_Shape` describing the PTY argv contract by name. Each is a single-token swap to `dontAsk`. No behaviour change; the streamrunner test itself still bans `--permission-mode` outright and is unaffected.

- **Field-specific argv override (verified empirically in parent [#537](https://github.com/pyrycode/pyrycode/issues/537)).** `--permission-mode dontAsk` overrides settings `defaultMode` only ŌĆö it does **not** overwrite `permissions.allow`. Per-agent allow lists written by [`settings.WriteSettings`](../features/agentrun-settings-subpackage.md) are preserved across the change. The no-touch list (preserved deliberately): `permissions.allow` handling, the `buildStreamRunnerClaudeArgs` argv path in `cmd/pyry/agent_run.go` (separate enforcement model ŌĆö `--dangerously-skip-permissions` + `--allowed-tools`, explicitly bans `--permission-mode`), the selfcheck collaborator surface, and the `internal/e2e/realclaude/permission_protocol_spike_test.go` `permissionMode` knob (manually-driven empirical probe, meant to be edited at probe time).

- **Evergreen feature docs synced.** `docs/knowledge/features/ptyrunner-package.md`'s argv shape block and `docs/knowledge/features/agentrun-selfcheck-package.md`'s package summary + argv-shape block + FAIL-message exhibit all flipped from `default` to `dontAsk`. The selfcheck doc's contract paragraph now cross-links to this ticket file with a one-sentence note on what the prior argv `default` did (shadowed the settings deny-default; silent drop under headless `-p`) ŌĆö same shape as the existing `#487` callout for the JSON-side `deny`ŌåÆ`dontAsk` fix.

## Patterns established

- **Belt-and-suspenders on different fabric ŌĆö both substrates now deterministic.** CLAUDE.md's "Belt-and-Suspenders Means Different Fabric" pipeline principle is now satisfied for the per-agent deny-default: **suspender** is the settings JSON field `permissions.defaultMode: "dontAsk"` (written by [`settings.WriteSettings`](../features/agentrun-settings-subpackage.md), corrected in [#487](487.md)), **belt** is the argv `--permission-mode dontAsk` (written by `buildArgs`, corrected in this ticket). The substrates are distinct: a JSON parse failure in claude (documented `-p` failure mode) leaves argv enforcing; an argv-parser anomaly leaves settings JSON enforcing. Both layers say the same thing; either alone deny-defaults. Neither is stochastic ŌĆö neither is an LLM-agent rule paired with a code-level rule ŌĆö so "different fabric" is satisfied. **Generalises: when a security-relevant configuration value is expressible in two distinct deterministic substrates (argv string + JSON field, env var + config file, headers + URL params), prefer asserting the same value in both rather than picking one. The double-assertion costs ~1 line per substrate and survives a silent parse failure in either layer; the single-substrate alternative is one validation regression away from collapsing to a no-op.**

- **Per-tool/CLI flag overrides settings-file values of the same name ŌĆö the same shape as [#487](487.md)'s lesson, this time on the argv side.** Sibling [#487](487.md) caught a settings-file enum value (`"deny"`) that claude silently rejected and fell back to `"default"` ŌĆö the lesson was "read the docs for enum values." This ticket catches the dual shape: the argv flag value (`default`) was a valid argv enum but documented to **override** the settings-file field of the same name. Both bugs land at the same operator boundary (per-agent deny-default not enforced); both are detectable only by reading the CLI reference precisely. **Generalises: when a tool exposes the same knob via both argv flag and settings file, default to assuming argv shadows settings unless the docs say otherwise ŌĆö and pin the assumption in a test rather than relying on either layer alone. The `TestBuildArgs` position-by-position fixture is the right shape for this: a single-token change to the spec literal is a one-line PR, but the test catches an accidental drift on a future refactor.**

- **Comment/string-literal hygiene as a first-class spec deliverable.** The architect spec explicitly enumerated three stale-comment sites (selfcheck package doc, operator FAIL message, test comment) alongside the production literal and the test fixture ŌĆö five touch sites total for a "one-line fix." None of the three doc-hygiene swaps changes behaviour; each prevents a future reader (operator chasing a FAIL diagnostic, contributor reading the package doc, code-reviewer scanning a test comment) from being misled into thinking the code asserts something it doesn't. **Generalises: when changing a load-bearing constant or argv literal, grep the repo for the prior token and update every doc/comment/operator-string site in the same PR. Splitting them across multiple PRs leaves a window where operator-facing strings lie about what the code does ŌĆö exactly the failure mode this ticket itself was an instance of (post-[#470](https://github.com/pyrycode/pyrycode/issues/470) the package doc and FAIL message kept saying `default` because nobody traced the argv override).**

## Lessons learned

- **Argv flags override same-name settings-file fields by default ŌĆö this is documented but easy to miss when refactoring.** The bug was introduced in [#470](https://github.com/pyrycode/pyrycode/issues/470)'s ptyrunner production cutover, slipped past code review, lived through [#487](487.md)'s settings-file `dontAsk` fix on the JSON side, and was only caught when [#526](526.md)'s selfcheck AllowedTools gate surfaced an adjacent symptom. The reason it was easy to miss: argv `default` is a valid enum value (it does something ŌĆö "prompt the user before each tool"), and under headless `claude -p` the "prompt" silently drops, so a passing FAIL-mode selfcheck looked like proof of enforcement. **Generalises: when a CLI exposes the same knob via both argv and a settings file, treat the argv flag as the authoritative source unless the docs explicitly say "consult settings if flag omitted." If the operator intent is "use the settings file's value," the safe shapes are (a) omit the flag entirely (only safe if the CLI's default-when-omitted is "consult settings"), or (b) write the same value to both substrates ŌĆö never assume argv defers to settings without checking. Sibling-lesson reference: [#487](487.md) for the dual shape on the JSON side (the literal value was `"deny"` by guess and silently fell back to `"default"`; the docs disambiguated to `"dontAsk"`).**

- **A silent-no-op security regression is harder to catch than a loud failure ŌĆö and headless `claude -p` is structurally biased toward silent.** The production behaviour pre-fix was: out-of-allow-list tool calls land in prompt-mode, prompt-mode requires an operator to answer, headless `-p` has no operator, the request just drops. The dispatcher's claim "per-agent allow list is enforced" was technically true (the allow list was on disk, claude was reading it) but practically advisory (any out-of-list call silently no-op'd instead of being denied). A future model that auto-confirms permission prompts, or a behavioural shift in prompt-mode's silent-drop default, would have flipped this from "silent no-op" to "silent allow" ŌĆö the same code, the same settings file, the same argv. **Generalises: when a security control depends on a runtime behaviour that varies by execution mode (interactive vs headless, attended vs unattended, TTY vs no-TTY), explicitly assert the control at the layer that doesn't depend on the runtime ŌĆö here, argv `dontAsk` is "deny" regardless of whether anyone is watching, while argv `default` is "prompt ŌåÆ maybe deny ŌåÆ maybe drop." Defaults that read "secure under interactive use" are not secure under unattended use, and pyry's production surface is unattended by construction.**
