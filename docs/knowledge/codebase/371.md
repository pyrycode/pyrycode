# Ticket #371 ‚Äî fakerelay `WaitBinary` closes the upgrade‚Üíregistration race

`TestForceCloseBinary` flaked under CI's `-race` detector. Fix is a fixture-only synchronization helper; no production code touched.

The race lives in test-fixture API expectations: in `handleBinary` (`internal/e2e/internal/fakerelay/fakerelay.go:231-260`) `websocket.Accept` returns to the client (unblocking the test's `websocket.Dial`) **before** the server-side handler acquires `s.mu` and inserts `s.binaries[serverID] = bc`. `ForceCloseBinary` considers a binary "live" iff `s.binaries[serverID]` is present at the moment of lookup, so a test calling `ForceCloseBinary` immediately after `dialBinary` could land in that window and observe a not-yet-registered binary. The sibling e2e test `TestRelay_1011` already side-steps this by polling `LastBinaryHello` (`internal/e2e/relay_test.go:145-158`), but unit tests that raw-dial without sending a hello had nothing analogous to poll on.

See [features/fakerelay-harness.md](../features/fakerelay-harness.md) for the harness contract.

## Implementation

- **`internal/e2e/internal/fakerelay/fakerelay.go`** ‚Äî adds `func (s *Server) WaitBinary(ctx context.Context, serverID string) bool`, placed adjacent to `LastBinaryHello`. Checks `s.binaries[serverID]` under `s.mu` once on entry (fast path, returns `true` with no sleep if already registered), then polls every 2 ms via `time.NewTicker` until the entry appears or `ctx.Done()` fires. Returns `true` on registration, `false` on ctx expiry. Ticker (not `time.Sleep`) so cancellation is responsive; 2 ms cadence is two-to-three orders of magnitude headroom over the microsecond-scale race window ‚Äî happy path hits on the first or second tick.
- **`internal/e2e/internal/fakerelay/fakerelay_test.go`** ‚Äî `TestForceCloseBinary` now calls `s.WaitBinary(ctx, "alpha")` between `dialBinary` + `t.Cleanup(bin.Close)` and `ForceCloseBinary("alpha")`; the ghost-case assertion at the end is unchanged. New `TestWaitBinary` with two `t.Run` subtests under `t.Parallel()`: happy-path dials a binary and asserts `WaitBinary` returns `true`; timeout-path uses a 10 ms `context.WithTimeout` against a never-dialed server-id and asserts `false`.

## Deliberately not changed

- **`ForceCloseBinary` semantics.** Its fast-return `false` for unknown server-ids is the contract the ghost-case branch at `fakerelay_test.go` pins. Wrapping `ForceCloseBinary` in an internal wait would either slow the ghost case by the wait deadline or require two variants.
- **`handleBinary`'s Accept-then-register order.** Restructuring to pre-Accept registration would impose new ordering constraints on phone routing (the dispatcher at `fakerelay.go:536` would need to gate on a not-yet-existing "ready" channel to avoid forwarding to a nil `bc.conn`). Out of proportion to a fixture-only flake.
- **Production code.** The fakerelay package is test-only (`internal/e2e/internal/`-fenced); the WS wire protocol and production relay are not implicated.

## Patterns established

- **When a fixture probe reads server-internal bookkeeping, pair it with a `Wait*` helper for callers without an existing positive signal.** The harness already had `LastBinaryHello` for tests that send a hello; `WaitBinary` is the parallel for raw-dial unit tests. Generalises: any future fixture-only probe added alongside `ForceCloseBinary` / `LastBinaryHello` should ship with its synchronization helper in the same slice if the underlying state crosses an upgrade‚Üíregister boundary. Don't expect every caller to invent its own poll loop.
- **Polling helpers return `bool`, not `error`, when the only failure mode is ctx expiry.** Matches `LastBinaryHello`'s style; the caller already owns the ctx and can produce its own message. Avoids forcing callers through an `errors.Is(err, context.DeadlineExceeded)` check for a single deterministic failure mode.
- **2 ms tick on a `time.Ticker`, not `time.Sleep`, for in-process fixture polls.** Sleep can't be cancelled mid-flight; a ticker `select`s alongside `ctx.Done()` so a stuck test fails on the caller's deadline instead of the ticker cadence. Pick the cadence two-to-three orders of magnitude tighter than the worst observed jitter for the operation you're waiting on (microseconds here ‚Üí 2 ms tick), not the e2e cadence used for daemon-dial-out polls (`relay_test.go` uses 20 ms).
- **Fix race bugs at the layer where the bug actually is.** This race was in the fixture's API surface, not in the WebSocket protocol. The chosen layer was option (b) from the ticket's AC menu: synchronize the test on a positive signal. Options (a) ‚Äî making `ForceCloseBinary` block internally ‚Äî and (c) ‚Äî a `time.Sleep` grace period in the test ‚Äî were rejected for documented reasons (semantic regression for the ghost case; sleep-cannot-be-cancelled). The reference incident `d1aa67f` legitimately landed at the test-threshold layer; the diagnostic playbook (read test, read impl, decide layer) is what matters, not the layer the precedent happened to land on.

## Lessons learned

- **`websocket.Accept` returning is not "the conn is ready for server-side probes."** The 101 Switching Protocols response is written before the server-side handler finishes its post-Accept bookkeeping. Any test that asserts on server-side maps after a successful `Dial` is racing the handler unless something synchronizes the two. Documented here so the next test that adds a fixture probe doesn't have to rediscover it.
- **CI's main-merge `-race` run is the canonical surface for these flakes, not the PR check.** #371's failure pattern (PR PASS at 19:11:08 ‚Üí main-merge FAIL at 19:11:19 on the same code) is the second instance after `internal/agentrun/budget`'s `d1aa67f`. A green PR check is not evidence of race-freedom; reproducing locally requires `go test -race -count=N` until N is large enough to surface the window.
