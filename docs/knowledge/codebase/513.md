# Ticket #513 — `ptyrunner.Run` consumes `tuidriver.Session.Events` unified stream

Capstone of the [#509](509.md) → [#510](510.md) → [#511](511.md) → [#512](512.md) migration onto the 2026-05-23 tui-driver pin. `ptyrunner.Run`'s post-`WritePrompt` body collapses from a `tuidriver.TailJSONL` channel drain (set up in #512) onto a single `for ev := range ch` switch dispatch over `sess.Events(runCtx, jsonlPath, 0)`. The unified stream interleaves PTY-axis transitions (`EventKindPtyModalShown` / `EventKindPtyMcpFailureShown` / `EventKindPtyNetworkFailureShown` and their `*Hidden` pairs, plus `EventKindPtyIdle` / `EventKindPtyThinking`) with the JSONL-axis events (`EventKindJsonlEntry`, `EventKindJsonlEndOfTurn`) in arrival order. Operator-visible win: **mid-run modals and banners are now caught continuously** at tui-driver's 50 ms `DefaultPollInterval` cadence; previously the post-idle one-shot `HasTrustModal` / `HasMcpFailureBanner` / `HasNetworkFailure` checks at `runner.go:294-305` ran exactly once between `WaitUntil(idle)` and `WritePrompt` and missed anything that surfaced afterwards. Net diff: +123 / −24 across `runner.go` + the two helper/test files; no new exported symbols, no `go.mod` bump, no new imports. The pre-`WritePrompt` phase (Spawn → `WaitUntil(idle)` → one-shot detector trio → `WritePrompt` → `WaitForSessionJSONL`) is preserved verbatim because `Session.Events` calls `TailJSONL` synchronously and that fails with `os.ErrNotExist` until claude creates the per-session JSONL file (which it only does after `WritePrompt`).

## Implementation

- **Single-loop switch dispatch replaces the inline drain.** The pre-#513 body at `runner.go:393-407` was `entries, err := tuidriver.TailJSONL(runCtx, jsonlPath, 0)` + `for entry := range entries { Emit; OnEvent; if IsEndTurn { OnEndOfTurn; break } }`. The post-#513 body at `runner.go:393-425` is `ch, err := sess.Events(runCtx, jsonlPath, 0)` + a labelled `loop:` `for ev := range ch` with a five-arm `switch ev.Kind`: `EventKindPtyModalShown` (Trust class → `ErrTrustModalDetected`, other classes ignored), `EventKindPtyMcpFailureShown` → `ErrMcpFailureBanner`, `EventKindPtyNetworkFailureShown` → `ErrNetworkFailure`, `EventKindJsonlEntry` → `emitter.Emit(ev.Entry)` (capture into `emitErr` on first error) + `counter.OnEvent(ev.Entry)`, `EventKindJsonlEndOfTurn` → `counter.OnEndOfTurn()` + `break loop`. `*Hidden` / `EventKindPtyIdle` / `EventKindPtyThinking` / `EventKindUnknown` fall through the switch with no action (no `default` arm — the spec preferred explicit no-op over a `default: continue` because the switch is exhaustive by intent, not by syntax). The `break loop` label is needed because a bare `break` exits the `switch`, not the `for`; both Go's spec and the project's stdlib-only test suite make this pattern the idiomatic single-loop dispatch shape.

- **Error wrap renamed `tail` → `events`.** The synchronous open/seek failure from the new `sess.Events(...)` call wraps as `fmt.Errorf("ptyrunner: events: %w", err)`; the pre-#513 `ptyrunner: tail: %w` wrap (which named `TailJSONL`'s open/seek failures introduced in #512) is reused at the same call site but renamed to reflect the new entry point. `isCtxErr(runCtx, err)` collapse-to-nil for ctx-cancel during open is preserved verbatim. The `Run` doc-comment's error-contract list at `runner.go:178-205` renames the line in lockstep, and adds a new line under `ErrTrustModalDetected / ErrMcpFailureBanner / ErrNetworkFailure` that names the mid-run detection cadence ("tui-driver's `DefaultPollInterval` (50 ms)"). All four tui-driver call sites in this package now share the same `isCtxErr` + `<wrap-name>: %w` shape: `Spawn`, `WaitUntil`, `WaitForSessionJSONL`, `Session.Events`.

- **Watchdog goroutine stays separate per spec § Decision 1.** The architect explicitly chose "keep" over "fold" for the `runWatchdog` goroutine because its `*tuidriver.Tracker` carries its own `PTYQuietLimit` / `SpinnerFreezeLimit` state and its own configurable tick (default 1 s, tests use 50 ms) — folding would require either a manual `time.Ticker` arm in the `select` (duplicating what `tuidriver.RunWatchdog` already does internally) or a watchdog-event-channel adapter (extra plumbing for no win). Keeping it separate preserves the entire cleanup-LIFO chain verbatim: `cancel() → wg.Wait() → counter.Stop() → emitter.Close() → sess.Close()`. The `wg.Add(1)` / `go runWatchdog(...)` / `defer wg.Wait()` / the second `defer cancel()` block at `runner.go:378-385` is untouched.

- **Cleanup-LIFO doc-comment updated in two places.** `runner.go:215-221` retitles the second-step comment from "`cancel()` signals the watchdog goroutine (and the `tuidriver.TailJSONL` goroutine) to exit" to "`cancel()` signals the watchdog goroutine (and the `tuidriver.Session.Events` merge goroutine) to exit". The five-step LIFO sentence and the "Re-ordering these defers silently breaks the invariant" reminder are byte-identical. `runner.go:171-205` rewrites the package-level paragraph: "drains the per-session JSONL via `tuidriver.TailJSONL` inline" becomes "drains the per-session JSONL AND polls PTY-state transitions via `tuidriver.Session.Events`". The `Stdout` writer contract, the operator-shutdown collapse semantics, and the `emitErr` capture-then-prioritise pattern are unchanged.

- **`emitErr` capture-then-prioritise pattern preserved verbatim.** The three-source return-site at `runner.go:426-432` is byte-identical: `if runCtx.Err() != nil { return nil }; if emitErr != nil { return fmt.Errorf("ptyrunner: emit: %w", emitErr) }; return nil`. The new sentinel-error returns (`ErrTrustModalDetected` / `ErrMcpFailureBanner` / `ErrNetworkFailure`) inside the switch short-circuit out before the return-site is reached, so they take strict priority over both `runCtx.Err()` and `emitErr` — matches the pre-`WritePrompt` one-shot detector's behaviour (which also `return`s the sentinel without consulting `ctx.Err()`). The four-source restructure (ctx-cancel → emit-error → channel-close-no-EOT → clean-EOT) deferred by [#512](512.md)'s NIT is **still deferred** — the AC explicitly preserved the three-source pattern, and #513 honoured that scope cut. The channel-close-without-EOT silent-nil edge case carries forward unchanged.

- **Three new helper-test modes shaped after `jsonl` mode.** `helper_test.go` adds `mid_trust` / `mid_mcp_failure` / `mid_network_failure` to the `runHelper` switch. All three: (a) write `idleGlyph + " "` at startup (no anchor yet, so the parent's pre-`WritePrompt` one-shot detectors all return false and `WaitUntil(idle)` returns), (b) drain stdin to `io.Discard` and send on `stdinSeen` once `WritePrompt` lands, (c) in a goroutine waiting on `stdinSeen`, create the per-session JSONL file empty (so the parent's `WaitForSessionJSONL` resolves and `Session.Events` opens — this is the load-bearing addition; the JSONL file was never created in the pre-`WritePrompt`-only modes), then `fmt.Fprint(os.Stdout, anchor)` + `os.Stdout.Sync()` where `anchor` is the mode-specific string: `"Quicksafetycheck" + idleGlyph + " "` (Trust), `"1 MCP server failed " + idleGlyph + " "` (MCP), `"FailedToOpenSocket " + idleGlyph + " "` (Network). The merge loop's 50 ms ticker observes the rising-edge transition on the next poll and emits the corresponding `*Shown` event. The 20 s `time.After` timeout on the `stdinSeen` wait mirrors the `jsonl` mode and acts as a safety belt against a parent-side hang. All three modes share the existing SIGTERM handler and 30 s ceiling — no helper-side restructure beyond the new switch arm and the gated write goroutine.

- **`TestRun_MidRun_ModalAndBannerDetection` table-drives the three cases.** Sub-tests `"trust modal mid-run"` / `"mcp failure banner mid-run"` / `"network failure mid-run"` each: build a `helperRunCfg` with the corresponding mode and empty `jsonlBody`, run `Run` with a 10 s context timeout, assert `err != nil`, `errors.Is(err, <sentinel>)`, and a substring on the operator-readable message (`"#469's MarkWorkdirTrusted"` / `"MCP failure banner"` / `"FailedToOpenSocket"` — same anchors the pre-`WritePrompt` `TestRun_TrustModalDetected` / `TestRun_McpFailureDetected` / `TestRun_NetworkFailureDetected` tests pin). Both the outer `t.Parallel()` and the inner subtest `t.Parallel()` are set so the three run concurrently. Table-driven where the pre-`WritePrompt` siblings are three top-level functions because the table's scaffolding (`helperRunCfg` mode + assertion shape) is uniform — spec § Test plan explicitly allowed either shape; the table reads cleaner in-place. The pre-`WritePrompt` triplet is left as three top-level functions; the new test is a fourth sibling, not a merge of all six.

- **`logger.Warn` calls on detection mirror the pre-`WritePrompt` one-shot detector exactly.** Each mid-run sentinel return is preceded by a single `logger.Warn("ptyrunner: <surface> detected")` matching the wording at `runner.go:298/302/306`. The spec § Decision 3 cautioned against per-event log volume on JSONL events (the discipline pinned in the package-level doc comment), but the Warn calls fire once on rare-event detection, not per entry — the discipline holds, and the parity with the pre-`WritePrompt` detector is the right symmetry for operators correlating logs.

- **All eight pre-existing tests pass unmodified.** `TestRun_HappyPath_EmitsAndEndOfTurn` continues to work because `mergeEvents` emits `EventKindJsonlEntry` (which the runner re-emits via `emitter.Emit`) before `EventKindJsonlEndOfTurn` (which calls `counter.OnEndOfTurn` and breaks the loop); the init envelope is written by `streamjson.New` before the Events loop opens, so the stdout sequence is byte-identical. `TestRun_CtxCancelDuringStream` continues to work because the channel closes when `runCtx` cancels, the `for` loop exits naturally, and the return-site collapses to nil. `TestRun_EmitErrorPropagation` continues to work because the `failingWriter` error is captured into `emitErr` inside the `EventKindJsonlEntry` arm; the loop continues to `EventKindJsonlEndOfTurn`, calls `OnEndOfTurn`, breaks, and the return-site surfaces `ptyrunner: emit: %w`. `TestRun_BudgetHitBeforeEndOfTurn` continues to work because `counter.OnEvent` runs inside the `EventKindJsonlEntry` arm and fires `Terminate` at the budget-th entry before the EOT event arrives (the post-#512 boundary semantic — budget wins over completion — is preserved). `TestRun_WatchdogFires` continues to work because the watchdog goroutine and its `SetExitReason(ExitReasonError)` + `cancel()` are untouched. `TestRun_TrustModalDetected` / `TestRun_McpFailureDetected` / `TestRun_NetworkFailureDetected` (pre-`WritePrompt` one-shot) continue to work because the helper modes `trust` / `mcp_failure` / `network_failure` write the anchor at startup (not gated on `stdinSeen`), so the pre-`WritePrompt` detector trio fires before the new switch ever opens. `TestRun_CtxCancelDuringSpawn`, `TestBuildArgs`, `TestRun_MissingRequiredFields` are unaffected.

- **No `go.mod` bump.** Same 2026-05-23 tui-driver pin as [#509](509.md) / [#510](510.md) / [#511](511.md) / [#512](512.md): `v0.0.0-20260523181457-c2dcd1e49992`. ## Patterns established

- **A labelled `break loop` is the right shape for a single `for ev := range ch` switch-dispatch where one switch arm needs to exit the loop.** The naive shape (`for { ev, ok := <-ch; if !ok { break }; switch { case ...: counter.OnEndOfTurn(); break } }`) hits the Go-spec gotcha that bare `break` inside `switch` exits the switch, not the for. The `for ev := range ch { switch ev.Kind { ... case EOT: counter.OnEndOfTurn(); break loop } }` shape is one line shorter (no `ok`-channel-close branch), one indent shallower, and reads as a single dispatch. The `loop:` label is unusual in pyry's codebase (zero pre-existing instances in `internal/`) but is the canonical Go idiom for this exact pattern and matches the spec's illustrative shape verbatim. Generalises: **when a `for range`-over-channel loop has exactly one terminating switch arm, prefer `break <label>` over `goto done` or over restructuring to a select-with-ctx-arm**. The `goto done` shape in the spec's illustrative code was an architect-flag-of-the-pattern, not a literal-shape directive — `break loop` reads cleaner.

- **Mid-run failure detection is a strict expansion of when each sentinel fires, not a contract change.** The three sentinel errors (`ErrTrustModalDetected` / `ErrMcpFailureBanner` / `ErrNetworkFailure`) and their operator-readable messages are unchanged. Pre-#513 they fired only at the post-idle one-shot check; post-#513 they additionally fire at any rising-edge transition mid-stream. The blast-radius check (`callers of ptyrunner.Run`) confirmed no consumer needs updating: `cmd/pyry/agent_run.go::runAgentRunPty` propagates the error to its dispatcher exit-code mapping, `internal/agentrun/selfcheck/selfcheck.go::Run` propagates the error to its self-check result, `internal/e2e/realclaude/ptyrunner_byte_equivalence_test.go` asserts on the error chain. None of the three care WHEN the sentinel fires — they care WHICH sentinel fires. Generalises: **when extending the trigger surface for an existing sentinel, audit consumers for "do they branch on the trigger context, or only on the sentinel identity?" — if only the latter, the trigger expansion is wire-compatible with no consumer migration.** Same shape as [#512](512.md)'s `OnEvent` signature migration from `jsonl.Event` to `tuidriver.JSONLEntry`: a strict expansion to a wider type was wire-compatible because the consumer (`budget.Counter`) only read two fields and both renamed cleanly.

- **`Session.Events`'s synchronous open/seek error contract is a faithful mirror of `TailJSONL`'s, so the wrap rename is one-line.** Both `tuidriver.WaitForSessionJSONL` and `tuidriver.TailJSONL` wrap cancellation as `fmt.Errorf("…: %w", context.Cause(ctx))` and surface non-ctx open/seek failures unwrapped. `Session.Events` calls `TailJSONL` internally with the same `ctx`, so its synchronous error has the same shape. Renaming the wrap from `ptyrunner: tail: %w` to `ptyrunner: events: %w` is a pure documentation upgrade — the underlying error class (synchronous open/seek from an internal `TailJSONL` call) is structurally identical. Generalises: **when a library function wraps another library function from the same package, the consumer's wrap-name should track the public entry point, not the internal call** — the entry point is what the consumer's stack trace and log messages will read.

- **A pre-existing helper-mode pattern (here: `jsonl`'s "wait for stdin first byte, then act" shape) is the right substrate for new helper modes that need similar synchronisation.** The three new modes (`mid_trust` / `mid_mcp_failure` / `mid_network_failure`) all needed: (a) initial idle bytes so `WaitUntil(idle)` passes, (b) a wait on the parent's `WritePrompt` landing before writing the mid-run anchor, (c) JSONL file creation so the parent's `WaitForSessionJSONL` resolves. The pre-existing `jsonl` mode already had (a) and (b); the spec's per-mode-table identified (c) as the only new piece. Three additional switch arms + one shared `if mode == "mid_trust" || mode == "mid_mcp_failure" || mode == "mid_network_failure"` block in `runHelper` covered the entire delta. The stdin-drain goroutine and the `stdinSeen` channel send (mode-agnostic at lines 87-104) were untouched. Generalises: **when adding test modes that share synchronisation semantics with an existing mode, reuse the existing mode's coordination primitives rather than building new ones** — the alternative (a parallel `stdinSeen2` channel, a parallel drain goroutine) would have doubled the helper's mode-routing complexity for no win.

## Lessons learned

- **The code-review NIT about the package-level doc-comment drift was a leading indicator that future migration tickets should expand the spec's "doc-comment updates" delta beyond `Run`'s own paragraph.** Code review flagged that `runner.go:22` (a package-level paragraph describing logging discipline) still referred to "`tuidriver.TailJSONL channel drain`" and `runner.go:206` (the `ptyrunner: emit: %w` bullet inside `Run`'s doc) still referred to "captured during the inline drain". Both are technically correct (`TailJSONL` is still in the call graph indirectly, the drain is still inline), just stale-flavoured. The spec only named `runner.go:171-205` and `runner.go:207-219` for synchronisation; both got updated correctly. The miss was at the package-doc level. PASS with the NIT noted — the in-place fix is cosmetic and the next contributor touching the file will pick it up at zero cost. Generalises: **when a spec names specific line ranges for doc-comment synchronisation, the architect should also `grep -n <old-term>` across the file to enumerate the broader sweep**. The cost is one `grep`; the alternative is per-ticket NITs that accumulate as low-grade documentation drift. Same shape as the [#509](509.md) NIT about a stale `Watcher.file` doc-comment — both deferred for the same reason, both representing the same generalisable architect-side check.

- **The helper-side "create the JSONL file empty before writing the stdout anchor" ordering is load-bearing in the new test modes; reversing it produces a race that's invisible at low loop-counts.** The three new helper modes do `os.MkdirAll` → `os.OpenFile(O_CREATE|O_APPEND)` → `_ = f.Close()` → `fmt.Fprint(os.Stdout, anchor)` → `os.Stdout.Sync()`. The order is deliberate: the parent's `WaitForSessionJSONL` is a stat-poll loop that resolves once the file exists, AFTER which `Session.Events` opens the unified stream, AFTER which the merge loop's PTY ticker starts polling. If the helper wrote the anchor to stdout BEFORE creating the JSONL file, the anchor bytes would land in the PTY buffer during a window when no consumer is polling the PTY axis — and depending on the timing of `WaitForSessionJSONL`'s next poll, the merge loop might open AFTER the anchor's rising edge has already been absorbed into the steady-state PTY buffer (no rising-edge transition → no `*Shown` event → test hangs to the 10 s deadline). The current order forces the parent's pipeline to be fully wired before the anchor lands. `go test -race -count=5` confirms no flake at this shape. Generalises: **when a test-helper coordinates with a parent's asynchronous pipeline, the helper's writes should follow the parent's pipeline-readiness signals, not race them**. The 10 s test timeout would catch a regression here, but the timeout-cost is the test budget; deterministic ordering is cheaper.

- **Three new helper modes sharing identical startup bytes is a cosmetic compression that's deliberately not taken in this PR.** Code review flagged that the `case "jsonl":` and `case "mid_trust", "mid_mcp_failure", "mid_network_failure":` arms at `helper_test.go:86-89` both write `idleGlyph + " "` — they could collapse into a single `case "jsonl", "mid_trust", "mid_mcp_failure", "mid_network_failure":` arm. The implementer kept them separate. Rationale: the `jsonl` mode and the three `mid_*` modes are semantically different (the `jsonl` mode drives a JSONL body written entirely by the helper; the `mid_*` modes drive an empty JSONL plus a stdout anchor) — their startup bytes happen to be identical today but the future-add shape is more likely "the `mid_*` modes get a unique startup string" than "all four modes stay collapsed". The cosmetic compression would have to be undone the first time the modes diverge. PASS with the NIT noted; the in-place fix is one-line and reversible. Generalises: **switch-arm consolidation is a refactor with non-zero ongoing cost — only take it when the arms are semantically identical, not when they happen to share a payload today**.

## Out of scope (follow-ups)

- **The four-source `emitErr` restructure** (deferred again — see [#512](512.md)'s same Open Questions § entry). The channel-close-without-EOT silent-nil case (the [#512](512.md) NIT carried forward) remains the natural home for a follow-up that combines (a) the four-source prioritisation (ctx-cancel → emit-error → channel-close-no-EOT → clean-EOT), (b) a new `ptyrunner: events closed without end_turn` wrap, and (c) a test that drives a channel-close without ctx-cancel (e.g. tui-driver-side failure injection). The AC for #513 explicitly preserved the three-source pattern; the restructure stays a separate ticket.
- **Non-Trust modal classes ignored.** The switch ignores `EventKindPtyModalShown` with `ev.Modal` of `Permission` / `MCP` / `Agents` / `SlashPicker` / `ModelSelect` / `AskUserQuestion` / `PermissionsConfig`. If a future failure mode surfaces one of them as a mid-run wedge, the right sentinel is unclear (the existing three sentinels are class-specific; `ErrPermissionPromptDetected` etc. would need to be added). Defer to evidence — no observed failure mode for any of these classes in agent-run today, and the deny-default settings file from [#469] already suppresses permission prompts.
- **PTY-only Events stream before JSONL exists.** The AC's optional carve-out (architect-may-explore-but-not-required) was declined in spec § Decision 2. If a future ticket needs continuous pre-`WritePrompt` PTY monitoring (e.g. fail-fast on a modal class that appears during claude's startup splash), tui-driver would need a `Session.PtyEvents(ctx) <-chan Event` variant. Out of scope here.
- **Cosmetic helper-mode switch-arm consolidation** (the `jsonl` arm + the three `mid_*` arms share `idleGlyph + " "`). Deferred per the third lesson learned above; the in-place fix is one-line and reversible.
- **Package-level doc-comment freshening** (`runner.go:22` and `runner.go:206` reference "`tuidriver.TailJSONL channel drain`" / "inline drain"). Deferred per the first lesson learned above; the in-place fix is two small edits and the next contributor touching the file picks it up at zero cost.
