# Ticket #450 — `internal/relay`: v2 re-key initiator on binary side (1-hour timer + `rekey_request` emit + 30s reply timeout)

Closes the v2 re-key initiator loop on the binary. Each `V2Session` that reaches `V2StateOpen` arms a 1-hour scheduled timer; on fire the manager AEAD-seals an envelope `{type: "rekey_request", payload: {reason: "scheduled"}}` under `s.send`, wraps it as a `noise_msg` inner frame, forwards via `m.send`, and starts a 30 s reply window. If the phone's fresh `noise_init` lands in [`handleRekeyInit` (#453)](453.md) before the window expires, the slice's new `rekeyComplete()` hook clears the awaiting-reply state and re-arms the 1-hour timer from the swap moment; otherwise the conn closes at WS 4426 (`StatusHandshakeFailure`) with a `noise.rekey_failed` log line. The wire emit consumes the `protocol.TypeRekeyRequest` constant landed in [#454](454.md). This is the last v2 re-key piece on the binary side; the operator-facing `pyry rekey <conn_id>` verb is [#451](https://github.com/pyrycode/pyrycode/issues/451). Split from #435 super-ticket; the abandoned `feature/449` super-slice contained the verbatim design notes for the wake-channel shape, the `rekeyComplete` seam, the timer-stop sequence in `closeWith`, and the package-var test-substitution idiom — same orphan-spec pattern as [#452](452.md) / [#453](453.md) / [#454](454.md).

## Implementation

- **One production source file modified** — `internal/relay/v2session.go` (+283 LOC: two package-var constants, the `wakeKind`/`wakeSignal`/`wakeBufferSize` block, three new `V2Session` fields, the manager `wake` channel, `Run`'s `runCtx` derivation + third select arm, `handleWake`, `armRekeyTimer`/`armRekeyReplyTimer`, `emitRekeyRequest`, the `rekeyComplete` method on `*V2Session`, one-line arm in `handleNoiseInit`, one-line `rekeyComplete` call in `handleRekeyInit`, and the timer-stop block in `closeWith`). Tests +325 LOC in `internal/relay/v2session_test.go` (three new tests under a `// --- re-key initiator tests (#450) ---` header plus a `waitForOutboundCount` helper). Single production file modified, no new exported symbols.

- **Two test-substitutable package-var constants land at the top of the file.** `var rekeyInterval = 1 * time.Hour` (scheduled cadence) and `var rekeyReplyTimeout = 30 * time.Second` (reply window). Lowercase / package-var posture mirrors `connection.go:43`'s `handshakeTimeout`; tests substitute via the same `prev := <var>; <var> = …; t.Cleanup(func() { <var> = prev })` idiom. Not part of any public API.

- **`time.AfterFunc` callbacks land on a fresh runtime goroutine — but the dispatch goroutine still owns `s.send` / `s.recv`.** The single-owner-goroutine invariant from #433/#445 is preserved by routing every timer fire through a per-manager wake channel: the callback pushes a `wakeSignal{s, kind}` onto `m.wake` and returns; the manager's `Run` loop pops the signal in a new third select arm and calls `emitRekeyRequest` or `closeWith` itself. No timer callback ever touches `s.send`, `s.recv`, `s.state`, or `s.awaitingRekeyReply` directly.

- **`Run` derives a cancel-on-exit `runCtx` so timer callbacks can unblock cleanly.** `runCtx, cancelRun := context.WithCancel(ctx); defer cancelRun()` replaces the bare `ctx` everywhere downstream (`handleFrame`, `handleWake`, both arming helpers, transitively `closeWith` / `handleNoiseInit` / `handleRekeyInit`). The return-on-cancel arm still returns `ctx.Err()` so the observable behaviour is unchanged ("parent-cancelled returns `context.Canceled`; Frames-closed returns nil"). Callback closures select on `(m.wake <- wakeSignal{…}, <-ctx.Done())` so a fired-but-undelivered wake on a shutting-down Run completes via the ctx branch and the goroutine exits — pinned by the goroutine-leak test.

- **`wakeBufferSize = 16` on the wake channel.** The 1-hour cadence makes concurrent fires across sessions vanishingly rare; cap=1 would also be correct, but 16 absorbs the worst case (every session timing out simultaneously while Run is busy in a slow handler) without forcing the timer callback to block. Buffer sizing is named in a constant for one-line tuneability.

- **Three new `V2Session` fields, all owned by the dispatch goroutine.** `rekeyTimer *time.Timer` (the 1-hour scheduled timer, nil before initial open and nil after `closeWith`); `rekeyReplyTimer *time.Timer` (the 30 s reply timer, nil unless `awaitingRekeyReply` is true); `awaitingRekeyReply bool` (the canonical "are we awaiting a fresh `noise_init` from the phone" predicate). The bool is load-bearing AND distinct from `rekeyReplyTimer != nil`: `time.Timer.Stop()` returns false if the timer already fired and the callback may already be in flight with a wake delivery; the bool is the stable predicate the `wakeRekeyReplyTimeout` arm checks to ignore a stale wake after `rekeyComplete` already won.

- **`handleWake` is the route table.** Two arms: `wakeRekeyEmit` → `emitRekeyRequest`; `wakeRekeyReplyTimeout` → log `noise.rekey_failed` + `closeWith(ctx, s, StatusHandshakeFailure, nil)`. Top-of-function `if w.s.state != V2StateOpen { return }` drops stale wakes silently — a wake arriving on a session that has already transitioned to `V2StateClosed` (e.g. via AEAD-failure teardown from [#446](446.md) between the timer fire and the wake delivery) is dropped without a log call. The state read is race-free because the same dispatch goroutine that would have mutated `s.state` is the goroutine doing the read. The `wakeRekeyReplyTimeout` arm has a second guard (`if !w.s.awaitingRekeyReply { return }`) covering the swap-completed-before-timeout-fired race.

- **`emitRekeyRequest` mirrors the existing `sealError` AEAD-seal template (line 818) and `marshalInnerFrameV2` (line 852).** Build `Envelope{ID: 1, Type: protocol.TypeRekeyRequest, TS: time.Now().UTC(), Payload: {"reason": "scheduled"}}`, `json.Marshal`, `s.send.Encrypt`, `marshalInnerFrameV2(TypeNoiseMsg, ciphertext)`, `m.send`, then set `s.awaitingRekeyReply = true` and arm `s.rekeyReplyTimer = m.armRekeyReplyTimer(ctx, s)`. Envelope `ID` is fixed at 1: there is no `rekey_ack` envelope to correlate by `InReplyTo` — the next successful AEAD round-trip under the new keys is the implicit ack per `docs/protocol-mobile.md` § Re-key. `reason` is always `"scheduled"` in this slice; `"manual"` and `"compromise"` are the `pyry rekey <conn_id>` verb's (#451) future surface.

- **Emit-side seal/marshal failures Warn-and-drop, do NOT close the conn.** Same posture as `sealError` (line 498-502): an internal AEAD-seal failure is not a wire-protocol failure and should not tear down a working session. Three distinct WARN events (`v2.rekey.emit.seal_failed`, `v2.rekey.emit.marshal_failed`, `v2.rekey.emit.skipped_already_awaiting`) carry `event` + `conn_id` only — no flynn-noise error text, no envelope bytes, no AEAD ciphertext. The next 1-hour cadence will attempt another emit. A wake arriving while `awaitingRekeyReply` is already true (which should not happen — `rekeyTimer` is one-shot and only re-armed by `rekeyComplete`, which clears the bool first) skips the re-emit with the `skipped_already_awaiting` WARN line; surfacing it as Warn means operators see it if a future refactor introduces a re-emit bug.

- **`rekeyComplete(m, ctx)` is a method on `*V2Session`.** Clears `awaitingRekeyReply`, stops + nils `rekeyReplyTimer`, stops `rekeyTimer`, and assigns `s.rekeyTimer = m.armRekeyTimer(ctx, s)`. The `(m, ctx)` argument shape carries the dependencies needed to re-arm: `m` for the wake channel, `ctx` (the Run-derived `runCtx`) for the `AfterFunc` callback's escape arm. Runs on the manager's single dispatch goroutine; inherits the no-mutex / no-atomic invariant from the rest of the package. The method's doc-comment is the seam-naming document — explicitly names it as "the bridge that brings responder-side swap completion back to initiator-side cadence."

- **Unconditional `s.rekeyComplete(m, ctx)` call at the success tail of `handleRekeyInit`.** Exactly one line, immediately after the existing `m.send(...)` at `internal/relay/v2session.go:632` (now :816 after the slice's other additions). Unconditional on the swap success path: a spontaneous phone-initiated re-key that the binary did not request still re-bases the 1-hour cadence (any successful swap is a fresh-key moment), and clearing an unset `awaitingRekeyReply` is a no-op; stopping a nil `rekeyReplyTimer` is a no-op. This addresses the one cross-function edit AC #2 names.

- **Initial-arm site is one line at the success tail of `handleNoiseInit`** (immediately after `s.state = V2StateOpen` at the original line 518): `s.rekeyTimer = m.armRekeyTimer(ctx, s)`. The only place `V2StateOpen` is set initially — re-key responder swaps stay in `V2StateOpen` and re-arm via `rekeyComplete` instead.

- **`closeWith` stops both per-session timers before the existing `delete(m.sessions, s.connID)`.** `Stop()` is safe on a fired timer (returns false, no-op); the cleanup is idempotent. The callback's `ctx.Done` arm remains the load-bearing teardown (Run-derived `runCtx` cancels on Run exit, which unblocks any pending callback goroutine); these `Stop()` calls free the runtime timer-heap entries and ensure no pending callback blocks indefinitely on a `m.wake` send.

- **Wake delivery is blocking-send-plus-ctx-escape, not non-blocking-drop.** A dropped `wakeRekeyEmit` would cost the 1-hour cadence one beat and never re-arm (the next re-arm comes only via `rekeyComplete`); a dropped `wakeRekeyReplyTimeout` would park the session in `awaitingRekeyReply=true` forever. With `wakeBufferSize=16` and rare fires, send-blocking is vanishingly rare; if it does happen, the callback waits microseconds for Run to drain one wake and exits cleanly.

- **No new exported symbols, no wire-format change, no new ADR.** [ADR 024](../decisions/024-noise-ik-mobile-e2e.md) § Re-key policy covers the design; this slice ships the initiator half of its responder-and-initiator symmetric requirement. The orphan spec at `docs/specs/architecture/449-v2-rekey-responder.md` resolved every design question this slice needed.

## Patterns established

- **Timer fires → wake channel → dispatch goroutine, never timer fires → session state.** When a stdlib primitive (here `time.AfterFunc`) spawns a fresh goroutine that needs to mutate state owned by a single goroutine elsewhere, the right shape is a buffered channel of typed wake signals + a new select arm on the owner goroutine's loop. The arming helper builds the callback closure inline so it captures the right ctx; the callback selects on `(channel <- signal, ctx.Done())` so cancellation unblocks it cleanly; the owner-side handler does an O(1) state-check guard (here `state != V2StateOpen`) to drop stale wakes silently. The pattern generalises whenever you want runtime-managed timers without giving up the single-owner-goroutine invariant that's load-bearing for non-goroutine-safe primitives (here flynn/noise `CipherState`; elsewhere e.g. a `sql.Conn`, a memory-mapped region, any cgo handle).

- **`runCtx, cancelRun := context.WithCancel(ctx); defer cancelRun()` is the seam for "callback goroutines must unblock when Run exits."** The bare `ctx` from `Run(ctx context.Context)` parameter cancels only on parent-side cancellation; a `Frames`-channel-close exit returns nil from Run without cancelling `ctx`, which leaves any pending timer callback blocked on its wake send forever. Deriving `runCtx` and `defer cancelRun()` guarantees that whatever caused Run to exit also cancels the context the callbacks select on, with no observable change to the parent — the return value is still `ctx.Err()`, not `runCtx.Err()`. The pattern generalises whenever a goroutine spawns "fire-and-forget" callback goroutines that select on a context: derive a cancel-on-exit context inside the spawner and pass that down, never the parent.

- **Bool + pointer is the right shape when `Stop()` can race the fire.** `time.Timer.Stop()` returns false if the timer already fired and the callback may already be in flight. When a "did this complete?" predicate needs to be observable from a code path that races the fire (here `handleWake`'s `wakeRekeyReplyTimeout` arm checking whether `rekeyComplete` already won), a companion bool that the winning code path clears first is the stable signal. The timer pointer is the concrete machinery; the bool is the state. Generalise: when you need both "stop this timer if possible" and "tell me whether the goal of this timer was reached", carry both — pointer-non-nil-ness alone races the goal predicate.

- **Send-blocking with ctx-escape over non-blocking-drop when the dropped event is load-bearing.** Wake channels for state machines should not drop fires — a dropped wake means a state-machine arm never fires and may park the session forever. Use `select { case ch <- signal: ; case <-ctx.Done(): }` with a small-but-real channel buffer; tune the buffer so blocking-send is vanishingly rare; let the ctx-escape arm be the teardown path. Generalise: drop-on-full is right for telemetry, wake signals demand at-least-once delivery within a session's lifetime.

- **Doc-comment the seam name explicitly when a function bridges two subsystems.** `rekeyComplete`'s doc-comment opens with "rekeyComplete is the seam that bridges responder-side swap completion back to initiator-side cadence." Future maintainers reading the responder-side `handleRekeyInit` see the one-line call and want to know "what does this do, and why is it called from here?"; the doc-comment is the answer one click away. The pattern generalises whenever a small method exists primarily to be called from a structurally distant code path — write the seam-naming sentence at the top of the doc-comment, not as a passing remark in the middle.

## Lessons learned

- **The same orphan-spec pattern as [#452](452.md) / [#453](453.md) / [#454](454.md).** This is the fourth carve-out from the abandoned `feature/449` super-slice. The architect spec at `docs/specs/architecture/449-v2-rekey-responder.md` on the unmerged `feature/449` branch contained verbatim design notes for the wake-channel shape (a per-manager channel with a typed signal struct, not per-session channels), the `rekeyComplete` seam (method on `*V2Session` taking `(m, ctx)`), the timer-stop sequence in `closeWith` (before the existing `delete`), and the package-var test-substitution idiom. Re-deriving these decisions from scratch risks subtle drift (e.g. picking per-session channels and burning allocation on every session, or picking a single bool field instead of bool+pointer and missing the swap-complete-vs-timeout race). The recurring orphan-spec lesson: when a super-slice is closed `NOT_PLANNED` and broken into smaller pieces, the abandoned spec is the canonical design document for each carved-out slice — lift the decisions verbatim, name the orphan spec in the new spec's "Files to read first" so the developer doesn't re-derive.

- **AC's "test scenarios" count is not always the test count.** AC #5 named four test scenarios; the slice ships three tests because bullets 1 + 2 collapse into a single joint test (`TestV2Session_RekeyInitiator_Emit_ReArmViaResponder`). The rationale: AC #5 bullet 1's "test calls `s.rekeyComplete()` directly" would have required a test-goroutine method call on the session, violating the single-owner-goroutine invariant. Testing the re-arm behaviour via the natural production caller (`handleRekeyInit`) is structurally stronger — the only production caller of `rekeyComplete` IS `handleRekeyInit`. The lesson: when an AC sketches a test via a direct-call shortcut that would violate the package's structural invariants, prefer the natural-caller path even if it merges multiple AC bullets — the merged test is the structurally honest shape; the AC bullet count is a coverage guide, not a literal test count.

- **`time.AfterFunc` spawns the callback goroutine only when the timer fires.** This is structurally important for the goroutine-leak test's `close_via_manager_exit` phase: an armed-but-not-yet-fired timer has no goroutine attached to it, so the runtime baseline check returns to baseline immediately on `Stop()` without needing the cleanup loop to run. Only the `close_via_reply_timeout` phase actually exercises the callback-vs-Run-exit race. A future reader who finds `runtime.Gosched(); runtime.GC(); time.Sleep(20ms)` in both phases might wonder if the first phase needs it — it doesn't, strictly, but the symmetric shape is defensible (the brief handshake window in the first phase COULD see a fire under heavy CPU load if rekeyInterval were smaller). The lesson: timer-callback lifecycle is firing-time spawn, not arming-time spawn — the runtime-leak signature differs between "armed and stopped" and "fired and racing teardown."

- **Test substitution of package vars is not parallel-safe.** Each of the three new tests is explicitly **not** `t.Parallel()` because they mutate package-level `rekeyInterval` / `rekeyReplyTimeout`, which the dispatch goroutines of any other parallel test would read at session-open or emit time. The `t.Cleanup(func() { rekeyInterval = prev })` restore ordering is correct WITHIN a test but does not isolate against concurrent reads from sibling parallel tests. The lesson: package-var test substitution and `t.Parallel()` are structurally incompatible when the substituted var is read by a goroutine spawned during the test; declare the no-parallel posture explicitly in a comment so a future contributor doesn't add `t.Parallel()` and introduce a flaky race.

- **Anti-leakage of error text from log lines is per-line, not whole-buffer.** The reply-timeout test's "no `err=` field" check operates on the specific log line containing `event=noise.rekey_failed`, not on the whole log buffer — the `v2.handshake.accept` line from the initial handshake legitimately carries no `err=` either, but a different test setup might surface unrelated `err=` fields elsewhere in the buffer. The per-line scan-find-then-substring-check pattern is the same shape #453 used for the `device_name` anti-enumeration check. The lesson: log-field anti-leakage assertions are inherently per-line; a global `strings.Contains(buf, "err=") == false` check false-positives on legitimately-error-carrying sibling lines and false-negatives on the wrong line carrying the leaked field.

## Out of scope (filed elsewhere or deferred)

- **The re-key responder swap path** — `handleRekeyInit`, atomic `CipherState` swap, peer-static continuity check. [#453](453.md), already merged; this slice only adds the one-line `s.rekeyComplete(m, ctx)` call at its success tail.
- **Receive-side classification of incoming `rekey_request` envelopes from the phone** — [#454](454.md), already merged. The `protocol.TypeRekeyRequest` constant this slice consumes was landed there.
- **`s.peerStatic` capture at initial handshake** — [#452](452.md), already merged. Inert in this slice.
- **`pyry rekey <conn_id>` operator verb** — #451. CLI surface for manual re-key; will emit a `rekey_request` envelope with `payload.reason == "manual"` or `"compromise"` instead of `"scheduled"`.
- **`reason: "compromise"` emit path** — only `"scheduled"` is emitted by this slice. The other two values land in #451.
- **Phone-side scheduling policy** — mobile team's concern. The binary tolerates whatever cadence the phone uses for its own re-keys (the post-swap `rekeyComplete` re-bases the 1-hour cadence from the last fresh-key moment, so a phone that re-keys more often naturally suppresses redundant binary-initiated emits).
- **Re-key rate-limiting** — the 1-hour cadence + the `skipped_already_awaiting` defensive branch naturally cap binary-initiated re-keys at 1/hour per session; no rate-limit needed for the timer-driven path. The operator-driven `pyry rekey` verb (#451) will own its own rate-limit policy if needed.
- **`pyry rekey <conn_id>` re-key rate-limiting** — defer until observed; same posture as initial-handshake rate-limiting (none).
- **Per-conn fan-out for handler dispatch** — head-of-line blocking during re-key (~100µs X25519 derivation + AEAD setup + one outbound send) stalls all conns on the single dispatch goroutine, same posture as initial handshake and the inbound re-key in [#453](453.md). Priority follow-up before production cutover.
- **Production wiring of `V2SessionManager` into `cmd/pyry/relay.go`** — daemon path still runs the v1 dispatcher. Cutover gated on [#436](436.md)'s pre-flight check.
- **Persistence of `rekeyTimer` / `rekeyReplyTimer` / `awaitingRekeyReply` state across binary restarts** — out of scope for v2; ADR 024 § Re-key policy specifies in-memory only. A binary restart drops both old and new keys plus all timer state; the next initial handshake re-establishes.
