# Ticket #274 — `internal/protocol`: v1 conversations-write payload structs

Phase 3 Track C — fourth slice of the #256 per-type payload catalog. Four pure DTOs that slot into `Envelope.Payload (json.RawMessage)` once the future dispatcher reads `Envelope.Type`: `CreateConversationPayload` (phone → binary, all three fields optional), `ConversationCreatedPayload` (binary → phone, in-reply-to a create), `PromoteConversationPayload` (phone → binary, all three fields required), `ConversationUpdatedPayload` (binary → phone, broadcast on the server-id). Shapes are fixed verbatim by `docs/protocol-mobile.md` § Message types → `create_conversation` / `conversation_created` / `promote_conversation` / `conversation_updated`; the example payload in each subsection is the golden-file fixture. Sibling slices to #275 (`RegisterPushTokenPayload`), #272 (messaging + backfill), and #273 (conversations-read).

## Implementation

- **One production file, one test file, four fixtures, stdlib-only.** `internal/protocol/conversations_write.go` (four struct declarations + doc comments, single `"time"` import) + `internal/protocol/conversations_write_test.go` (one test function per type, no table-driven) + four fixtures under `internal/protocol/testdata/` (`create_conversation.json`, `conversation_created.json`, `promote_conversation.json`, `conversation_updated.json`). The `TypeCreateConversation` / `TypeConversationCreated` / `TypePromoteConversation` / `TypeConversationUpdated` constants the tests consume already exist in `codes.go` from #255 — no constants added.
- **Four types in one file, by spec-grouping precedent.** The four write-side types form a single create/promote/update lifecycle for one entity. Splitting across files would scatter related shapes; same one-file-per-spec-grouping rationale used in #272's `messaging.go` and #273's `conversations_read.go`.
- **`*T` WITHOUT `omitempty` for spec-optional fields whose example wire shows `null`** — `CreateConversationPayload.{IsPromoted, Name, Cwd}`, `ConversationCreatedPayload.Name`, `ConversationUpdatedPayload.Name`. The contract is documented once in detail on `CreateConversationPayload` and referenced from the other two; this is the third slice to adopt the discipline (sibling: #272 `BackfillSincePayload.ConversationID`, #273 `ConversationSummary.Name`). `omitempty` on a nil pointer would drop the key entirely and break byte-equivalent round-trip with the spec example. A pointer-to-zero (e.g. `*bool` to `false`) round-trips as the scalar `false` (matching the spec example's `"is_promoted": false`), and a nil pointer round-trips as literal `null` — both branches load-bearing and pinned by the round-trip tests.
- **`CreateConversationPayload` has THREE optional pointers — the maximal optionality case in the catalog so far.** The spec example shows all three fields as `null` on the wire (`is_promoted: false`, `name: null`, `cwd: null`). The test pins all three branches: `IsPromoted` non-nil and points to `false` (proves the pointer-to-zero shape — wire `false` survives, doesn't collapse to nil); `Name` and `Cwd` nil (proves the `*string` + no-`omitempty` shape — wire `null` survives, doesn't collapse to empty string).
- **Field declaration order matches the spec example's wire-key order.** `encoding/json` emits fields in struct-declaration order, and `canonical()` (json.Compact) does not reorder keys — any mismatch breaks byte-equivalent round-trip. Per-type order: `CreateConversationPayload{IsPromoted, Name, Cwd}`; `ConversationCreatedPayload{ID, IsPromoted, Cwd, Name, LastUsedAt}` (note `Name` lands AFTER `Cwd` in this one — matches spec example); `PromoteConversationPayload{ConversationID, Name, Cwd}`; `ConversationUpdatedPayload{ID, IsPromoted, Name, Cwd, LastUsedAt}` (note `Name` lands BEFORE `Cwd` in this one — also matches spec example). The two minor reorderings between `_created` and `_updated` are deliberate; a future reorderer must update each fixture in lockstep.
- **`time.Time` round-trip needs no custom marshaller.** Spec uses `"2026-05-08T10:34:01Z"` / `"2026-05-08T10:34:30Z"` — RFC 3339, no fractional seconds. `time.Time.MarshalJSON` uses `RFC3339Nano`, which omits a fractional component when none is present, so a parsed `Z`-suffixed timestamp re-marshals byte-identically. Padding fixtures with `.000Z` would break the round-trip. Same discipline as `Envelope.TS`, `BackfillSincePayload.SinceTS`, `ConversationSummary.{LastMessageTS, LastUsedAt}`.
- **Envelope-level key order in fixtures follows the `Envelope` struct, not the spec example.** Go's marshaller emits `id, type, ts, payload, in_reply_to` (struct-declaration order, with `omitempty` dropping zero `in_reply_to` / `payload_encrypted`). The spec example for `conversation_created` puts `in_reply_to` between `ts` and `payload`; the fixture puts it AFTER `payload` because the round-trip target is the Go-marshaller output, not the spec's literal envelope-level key order. Sibling fixtures (`envelope_full.json`, `conversations.json`) already established this convention. Inside `payload`, however, the per-type struct's declaration order matches the spec example exactly — so payload-internal ordering does match the spec verbatim.
- **`conversation_created.json` carries `in_reply_to: 4`; the test pins it.** First (and only) fixture in this slice with an in-reply-to relationship: `create_conversation` rides `id: 4`, the reply rides `id: 260` with `in_reply_to: 4`. Asserting `env.InReplyTo != nil && *env.InReplyTo == 4` is the load-bearing check that proves the reply-to envelope omitempty discipline survives marshal alongside an array-free payload. The other three fixtures have no `in_reply_to`.
- **Pure DTOs: no methods, no constructors, no `Validate()`.** Identical posture to #275, #272, #273. Required-field validation, ID resolution, name uniqueness checks (`internal/conversations.Registry.Promote`'s domain) — all dispatcher / registry concerns at second-pass decode time. The wire types are decode targets, not behaviour.
- **Test reuses `canonical(t, b)` and `readFixture(t, name)` from `envelope_test.go`.** Same package, no import, no duplication. Per-type test shape mirrors `TestRegisterPushTokenPayload_RoundTrip` and the conversations-read sibling: unmarshal full envelope → assert `Type == Type<X>` (uses the constant) → second-pass unmarshal `Envelope.Payload` → spot-assert required scalars + nil-vs-pointer branches for optional fields → re-marshal envelope → `bytes.Equal(canonical(out), canonical(raw))`. Four flat test functions, no table-driven — each fixture is distinct enough that function-per-type reads cleaner.
- **No `IsV1Compatible` test added.** The four `Type*` constants are already in `v1TypeSet` (envelope.go); `compat_test.go` covers the set.

## Out of scope (deferred)

- The other 7 v1 payload structs (`hello` / `hello_ack` / `error` / `ack` for the #271 handshake-and-control slice; the remaining catalog ticket numbers). Each lands in its own ticket with its own file in this package.
- The `create_conversation` / `promote_conversation` dispatch handlers — second-pass decode against the per-type struct, register-or-lookup in `internal/conversations.Registry`, emit `conversation_created` or `conversation_updated`, broadcast on the server-id, route the reply via `in_reply_to`. Future ticket; registry-to-payload mapping is not this package's concern.
- `conversation_updated` broadcast fan-out — relay-side / dispatcher-side concern; this slice only defines the wire shape the dispatcher emits.
- Validation that `IsPromoted == true` implies `Name != nil` on `ConversationCreatedPayload` / `ConversationUpdatedPayload` — semantic invariant the registry enforces (`Registry.Promote` already gates on non-empty name + uniqueness), not a wire shape concern.
- Custom `json.Marshaler` / `json.Unmarshaler` for the optional pointer fields — the default struct-tag-driven implementation correctly round-trips `null` ↔ nil-pointer with `*T` + no `omitempty`.
- Server-side defaulting behaviour for `CreateConversationPayload`'s `null` fields (binary picks a fresh UUID, generates a name, defaults `cwd` to the workspace root or similar) — registry / dispatcher concern, not a wire-type concern.
