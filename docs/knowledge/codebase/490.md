# Ticket #490 — `WithWorktreeAuthenticated` accepts `CLAUDE_CODE_OAUTH_TOKEN`

Single-helper contract change in [`internal/e2e/realclaude/fixtures.go`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures.go): `WithWorktreeAuthenticated(t)` now reads both `ANTHROPIC_API_KEY` and `CLAUDE_CODE_OAUTH_TOKEN` from the outer test environment and proceeds if either is set; it skips only when both are unset/empty. Closes the silent-skip masquerade flagged in [`codebase/487.md`'s post-mortem](487.md) and named in [`codebase/482.md`](482.md) — `ptyrunner_byte_equivalence_test.go` (#482) and the other 6 opt-in tests never ran on a Max-only Mac (the operator's validation environment) because the credential lives in macOS Keychain (service `Claude Code-credentials`) rather than as `ANTHROPIC_API_KEY`. `claude` accepts the extracted access token via `CLAUDE_CODE_OAUTH_TOKEN`; the helper now re-pins whichever variable(s) the operator exported so the subprocess inherits a real credential while `WithWorktree`'s `HOME` pin (the per-test JSONL namespace isolation that makes parallel realclaude tests safe) stays intact. No signature change; zero edits to the 12 existing call sites.

## Implementation

- **`internal/e2e/realclaude/fixtures.go:39-78` rewrites `WithWorktreeAuthenticated` against a "either-or" contract.** Reads both `apiKey := os.Getenv("ANTHROPIC_API_KEY")` and `oauthToken := os.Getenv("CLAUDE_CODE_OAUTH_TOKEN")`. If `apiKey == "" && oauthToken == ""`, the helper `t.Skipf`s with a single message that names BOTH variables AND embeds the macOS Keychain extraction recipe verbatim: `` security find-generic-password -s 'Claude Code-credentials' -w | jq -r '.claudeAiOauth.accessToken' ``. Otherwise it delegates to `WithWorktree(t)` for the `HOME` pin + `t.TempDir()` workdir, then `t.Setenv`s only the env var(s) that were non-empty. Function signature stays `WithWorktreeAuthenticated(t *testing.T) string`; the 12 existing call sites across the 7 opt-in tests are untouched.

- **The "do NOT `t.Setenv` an absent variable to empty-string" rule is load-bearing.** `os.Getenv` returns `""` for both unset and set-empty, but `os.LookupEnv` distinguishes them, and downstream tooling (claude itself today, future fixture-level env sanitisation in `RunPyryAgentRun` tomorrow) may treat a set-empty value differently from absent. Step 4 of the spec mandates preserving the original outer-env shape; the implementation guards each `t.Setenv` call with an `if apiKey != ""` / `if oauthToken != ""` so a variable that was absent in the outer environment stays absent inside the test. `TestWithWorktreeAuthenticated_OAuthTokenOnly_RepinsAndPreservesAbsentApiKey` (see Tests below) pins this contract structurally with `os.LookupEnv` against the API-key slot.

- **Skip message: a single `t.Skipf` call with Go string concatenation across source lines** ([`fixtures.go:60-66`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures.go)). Required literal substrings asserted by the new test: `ANTHROPIC_API_KEY`, `CLAUDE_CODE_OAUTH_TOKEN`, `` security find-generic-password -s 'Claude Code-credentials' -w ``, `` jq -r '.claudeAiOauth.accessToken' ``. The recipe points operators at the Keychain extraction command + the export target — operator-actionable diagnostic in one paragraph, no link-out.

- **Why "set both if both are present" rather than picking one.** When the operator happens to have both variables set (uncommon — mixed-account Mac), the helper re-pins both. `claude`'s precedence between the two is its own concern; the helper's only job is to preserve whatever the operator chose to export, against future fixture-level env sanitisation. Reference shape in the ticket body matches this exactly. No precedence-flattening logic in the helper.

- **Doc comment rewritten in lockstep ([`fixtures.go:39-54`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures.go)).** Names both variables, points at the Max-only Mac context + Keychain extraction recipe, calls out the "preserve original outer-env shape" rule explicitly so a future maintainer doesn't "simplify" the per-variable `if` guards into an unconditional `t.Setenv` for empty values.

- **Plain `WithWorktree(t)` is unchanged.** The `HOME` pin is non-negotiable: per-test JSONL namespace isolation under `~/.claude/projects/<encoded-cwd>/<sid>.jsonl` prevents cross-test JSONL collision when several tests run in parallel against the same workdir-encoding scheme. Lifting `HOME` isolation would re-introduce that race. The fix is env-var-shaped, not `HOME`-shaped.

- **Two new tests in [`internal/e2e/realclaude/fixtures_test.go:104-177`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures_test.go)** (~74 LoC combined). `TestWithWorktreeAuthenticated_RealAssistant` is unchanged and continues to cover the API-key happy path end-to-end against the live API when `ANTHROPIC_API_KEY` is set; no duplication.

  - **`TestWithWorktreeAuthenticated_SkipsAndNamesBothEnvVarsWhenNeitherSet`** uses the outer/inner subprocess re-exec pattern, identical in shape to [`TestRunPyryAgentRun_Timeout`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures_test.go) (the documented precedent in the spec, `fixtures_test.go:319-342`). Required because `t.Skipf` ends the calling goroutine via `runtime.Goexit()` with no in-process return value — asserting on the skip-message text needs the test binary to re-exec itself with `-test.run=^TestName$ -test.v` and grep the captured output. Sentinel env var is `PYRY_REALCLAUDE_AUTH_SKIP_INNER=1` (distinct from `GO_TEST_HELPER_PROCESS=1` so `TestMain` correctly falls through to `m.Run()` rather than `runFakePyry()`). Outer branch asserts: process exited zero (skip is success, not failure), output contains `--- SKIP: TestName` (confirms skip path, not unrelated failure), output contains each of the four required substrings (both env-var names, the `security find-generic-password` recipe, the `jq` extraction). Inner branch clears both env vars via `t.Setenv("", "")` and calls `WithWorktreeAuthenticated(t)`; a `t.Fatalf` after the call is unreachable on the skip path.

  - **`TestWithWorktreeAuthenticated_OAuthTokenOnly_RepinsAndPreservesAbsentApiKey`** drives the OAuth-only branch in-process — no subprocess, no network call. Synthetic token literal `"test-oauth-token-not-real"`; claude is never invoked. Verifies (a) the helper proceeds past the skip gate when only the OAuth token is set, (b) the token survives `WithWorktree`'s `t.Setenv("HOME", ...)` cleanup (i.e. is still `os.Getenv`-readable after the helper returns), (c) the helper does NOT `t.Setenv` the absent `ANTHROPIC_API_KEY` — asserted with `os.LookupEnv` returning `present == false`. This is the structural counterpart to the live-API test for the OAuth path; no live-API counterpart is feasible since it'd require a Max-only Mac in the test environment.

  - **The `t.Setenv("ANTHROPIC_API_KEY", "")` + `os.Unsetenv("ANTHROPIC_API_KEY")` two-step in the OAuth test is deliberate.** The `t.Setenv` registers the testing-framework cleanup hook (so the original env is restored on test exit even if the test panics); the subsequent `os.Unsetenv` actually removes the variable so `os.LookupEnv` reports `present=false` rather than `present=true && value==""`. Validates the "preserve original outer-env shape" contract from the spec literally.

## Patterns established

- **Either-or auth-fixture contract over single-credential gate.** Test fixtures that gate on "is the operator authenticated to the upstream service" should treat the credential set as the union of all accepted mechanisms, not the historically-first one. The helper's job is to ask "can a credential reach the subprocess?", not "is THIS specific credential present?". Concretely: when the upstream (`claude` here) accepts multiple env-var shapes, the fixture reads all of them and proceeds if any is set. Generalises to any future test fixture gated on an external-service credential where the service supports more than one auth mechanism.

- **Preserve the unset-vs-set-empty distinction in fixture env-pins.** When re-pinning env vars via `t.Setenv` for subprocess inheritance, guard each `t.Setenv` with an `if value != ""` check so an absent variable stays absent. `os.Getenv` flattens the distinction to `""`, but `os.LookupEnv` and some downstream consumers don't — and "absent" is a load-bearing signal for tools that have multiple credential modes. The temptation to "just `t.Setenv` everything" is the wrong shortcut: it materialises set-empty values that the outer environment didn't have, and any downstream code that switches on `os.LookupEnv` reads a different shape than the operator exported.

- **Diagnostic skip messages name the extraction recipe, not just the variable.** When an operator-facing skip message names a credential variable that requires a non-trivial extraction step (Keychain on macOS, `vault read` in a CI lane, etc.), the message should embed the exact shell command that produces the value. `t.Skipf` is the only signal the operator sees; a generic "set `CLAUDE_CODE_OAUTH_TOKEN` to a Max-plan token" leaves them grepping docs. The verbatim `security find-generic-password -s 'Claude Code-credentials' -w | jq -r '.claudeAiOauth.accessToken'` recipe in the skip message is operator-actionable in one paragraph.

- **Subprocess-re-exec test pattern for `t.Skipf` message-shape assertions.** `t.Skipf` calls `runtime.Goexit()` with no in-process return value, so asserting on the skip-message text requires the outer/inner subprocess pattern — the test binary re-execs itself with `-test.run=^TestName$ -test.v` under a sentinel env var, captures combined output, asserts `--- SKIP: TestName` and the required substrings. Sentinel must be distinct from `GO_TEST_HELPER_PROCESS=1` (which `TestMain` already routes into `runFakePyry()`); `PYRY_REALCLAUDE_AUTH_SKIP_INNER=1` here. Mirrors `TestRunPyryAgentRun_Timeout` at `fixtures_test.go:319-342`. Reusable shape for any future test that needs to verify a skip-message contract.

- **Validate the "preserve outer-env shape" contract with `os.LookupEnv`, not `os.Getenv`.** A test that asserts an absent variable stays absent must use `os.LookupEnv` (returns `(value, present bool)`) — `os.Getenv` cannot distinguish unset from set-empty and would silently pass a regression where the helper `t.Setenv`s `""` for an absent variable. Pair with `t.Setenv(var, "") + os.Unsetenv(var)` to register the cleanup hook AND actually remove the variable for the assertion.

## Lessons learned

- **The fixture-wide auth-skip cleanup named as a follow-up in [`codebase/487.md`](487.md) was the right next step.** #487's post-mortem identified that `ptyrunner_byte_equivalence_test.go` (#482) silently skipped on Max-only environments because `WithWorktreeAuthenticated` only recognised `ANTHROPIC_API_KEY`. The follow-up was correctly scoped as "extend the helper" rather than "lift the `HOME` pin" (which would have reopened the JSONL collision race) or "copy `~/.claude/` into the tmp HOME" (which doesn't authenticate per the ticket body's empirical check — Keychain is the only credential source on macOS). The env-var path is the only correct fix while `HOME` stays pinned.

- **Test fixtures gated on a single credential type are silent-skip landmines under heterogeneous auth modes.** Every test that called `WithWorktreeAuthenticated` was structurally invisible to the Max-only validation environment for the entire ptyrunner cutover (#470) and the byte-equivalence smoke (#482) — including the test designed to detect the literal regression that #487 fixed. The lesson is not just "fix #490" but "audit every fixture that gates on a credential": if the gate doesn't recognise all credential shapes the upstream accepts, the gate masquerades as a pass-through under the unrecognised shape. The other realclaude tests stay correct under this fix because they all funnel through this one helper; future credential mechanisms that `claude` adds (a third env var, a token file path, an OS-keyring lookup) should land here too.

- **Code-review NITs on stale comments in untouched files are correctly out-of-scope for the originating PR.** The two NIT findings (`ptyrunner_byte_equivalence_test.go:182`, `mcp_smoke_test.go:21`) describe comments that now slightly mis-state the helper's contract. The right move was the reviewer's call: name them but don't block. Adding them to #490's scope would have inflated an XS production change into a multi-file test-comment sweep; the follow-up belongs as a separate ticket so the architectural change (this helper) lands cleanly and the comment-sweep is independent. Generalises: comment-only fixes in files outside the spec's "files-to-edit" set should be follow-ups, not in-scope churn.

- **The `t.Setenv("X", "") + os.Unsetenv("X")` two-step is the canonical "register cleanup, then actually unset" idiom under Go's `testing` framework.** `t.Setenv` registers the cleanup hook against the value-as-of-call (so the original is restored on test exit), but it also sets the variable to whatever value you passed; if you want the variable to be ACTUALLY unset for the test body while still benefiting from the framework cleanup, you need both calls. `os.Unsetenv` alone wouldn't restore the original on test exit. The OAuth-only test uses this idiom explicitly to validate the `os.LookupEnv(...).present == false` contract.

## Out of scope (follow-ups)

- **Stale-comment sweep in [`ptyrunner_byte_equivalence_test.go:182`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/ptyrunner_byte_equivalence_test.go) and [`mcp_smoke_test.go:21`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/mcp_smoke_test.go).** Both narrowly mis-describe the helper now that it accepts two env vars. Code-review surfaced both as NITs; not blocking, but worth a one-line fix in a follow-up ticket so future readers don't get the wrong contract from the comment.
- **Token-refresh handling inside the test process.** `claude` rotates the access token via `refreshToken` automatically during interactive use; the suite is short (~9 min) relative to the access-token TTL, so token expiry mid-suite is not an observed failure mode. Defer until observed.
- **CI Keychain access.** GitHub Actions runners can't reach the developer's Keychain; CI still uses `ANTHROPIC_API_KEY` injection. The secret-injection bug in `self-check-daily` ([#406](https://github.com/pyrycode/pyrycode/issues/406)) is its own track.
- **Making `CLAUDE_CODE_OAUTH_TOKEN` the default credential for everything `pyry agent-run` does on the operator's Mac.** The dispatcher's `pyry agent-run` runs in the operator's real shell (Keychain accessible normally); the env-var workaround is test-only.
