# Ticket #512 ŌĆö `ptyrunner.Run` consumes `tuidriver.TailJSONL` inline; delete `internal/agentrun/jsonl/tail/`

Sweep ticket that closes the three-step JSONL-tail migration started by [#509](509.md) and [#511](511.md). `ptyrunner.Run` now composes the session JSONL path via `tuidriver.SessionJSONLPath`, waits for it via `tuidriver.WaitForSessionJSONL`, then drains the channel returned by `tuidriver.TailJSONL(ctx, path, 0)` inline in its own goroutine. The `OnEvent` / `OnEndOfTurn` callback shape, the `tail.Watcher.Run` goroutine, and the throwaway `eventToEntry` adapter ([#511](511.md)) all collapse into a straight-line `for entry := range entries { ŌĆ” }` loop. `internal/agentrun/jsonl/tail/` is deleted entirely (~700 LOC removed). `budget.Counter.OnEvent` migrates from `jsonl.Event` to `tuidriver.JSONLEntry` and loses its end-of-turn early-return ŌĆö classification is the caller's job, and at the MaxTurns boundary the budget now wins (`max_turns`, not `completion`). `internal/agentrun/jsonl/` itself stays (two non-tail consumers, [`selfcheck`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/selfcheck/selfcheck.go) and [`e2e/realclaude/fixtures.go`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures.go), still parse to EOF via `jsonl.NewReader`); a future ticket migrates them and deletes the package. Net diff: +131 / ŌłÆ875.

## Implementation

- **Inline channel drain replaces the four-part watcher wiring block.** Pre-#512 `runner.go:360-399` constructed `tail.New(tail.Config{ŌĆ”OnEvent, OnEndOfTurn, ŌĆ”})` and blocked on `watcher.Run(runCtx)`. Post-#512 the body is a straight-line sequence: resolve `home` (cfg.HomeDir or `os.UserHomeDir()` fallback) ŌåÆ `jsonlPath := tuidriver.SessionJSONLPath(home, cfg.WorkDir, cfg.SessionID)` ŌåÆ `tuidriver.WaitForSessionJSONL(runCtx, jsonlPath)` (collapses to nil on ctx-cancel via `isCtxErr`, wrapped as `ptyrunner: wait jsonl: %w` otherwise) ŌåÆ `entries, err := tuidriver.TailJSONL(runCtx, jsonlPath, 0)` (same collapse / `ptyrunner: tail: %w` wrap) ŌåÆ `for entry := range entries { emitter.Emit(entry); counter.OnEvent(entry); if tuidriver.IsEndTurn(entry) { counter.OnEndOfTurn(); break } }` ŌåÆ post-loop `runCtx.Err() != nil` collapses cancellation to nil ŌåÆ `emitErr != nil` surfaces as `ptyrunner: emit: %w`. The `emitErr` capture-then-prioritise pattern from `runner.go:360-394` survives verbatim (`#513` capstone restructures it). Three new error wrap names introduced (`home dir`, `wait jsonl`, `tail`); the pre-#512 `tail: %w` wrap (which named `tail.New` / `Watcher.Run` failures) is repurposed for `TailJSONL`'s synchronous open/seek failure.

- **End-of-turn ordering preserved.** Inside the loop body the ordering is `Emit ŌåÆ OnEvent ŌåÆ IsEndTurn check ŌåÆ OnEndOfTurn`. This matches the pre-#512 `tail.Watcher.drain` shape (`watcher.go:219-222` called `OnEvent` before `OnEndOfTurn`). The `break` exits on the first end-of-turn entry, identical to the watcher's "stop draining" behaviour. Use `startOffset = 0` for `TailJSONL` ŌĆö ptyrunner does not resume mid-file.

- **`eventToEntry` deleted** (`runner.go:420-460`, ~30 LOC). The adapter existed solely to wrap `jsonl.Event` into `tuidriver.JSONLEntry` for `emitter.Emit` while the watcher still produced `jsonl.Event`. With `TailJSONL` delivering `JSONLEntry` natively, the adapter has no caller. The `internal/agentrun/jsonl` and `internal/agentrun/jsonl/tail` imports drop with it.

- **Cleanup-LIFO unchanged; `wg` tracks only the watchdog now.** The five-step LIFO (`cancel() ŌåÆ wg.Wait() ŌåÆ counter.Stop() ŌåÆ emitter.Close() ŌåÆ sess.Close()`) is preserved verbatim. The only structural change: `wg.Add(1)` only fires for the watchdog goroutine; `TailJSONL`'s internal goroutine is bookkept by tui-driver itself (the channel closes when `runCtx` cancels). The doc-comment block at `runner.go:181-216` is rewritten to reflect the new wrap names AND the `wg` scope change. The two `defer cancel()` lines (one before emitter.New, one after the watchdog goroutine spawn) stay in their current positions ŌĆö the LIFO semantic depends on the second `cancel()` firing first.

- **MkdirAll defence dropped from the production path.** The deleted `tail.New` called `os.MkdirAll(dir, 0o700)` before `fsnotify.Add(dir)` as belt-and-suspenders against the encoded project dir not existing on first run. `tuidriver.WaitForSessionJSONL` polls via `os.Stat` and treats a missing parent dir identically to a missing file (both surface as `os.IsNotExist`, both get retried). In production claude itself creates `~/.claude/projects/<encoded>/` on its first JSONL write ŌĆö the wait loop converges. Per the project's [Evidence-Based Fix Selection](../../../CLAUDE.md) principle, dropping the defence is correct: no observed failure mode where `WaitForSessionJSONL` blocked indefinitely on a missing parent dir. The test helper (`helper_test.go:130`) still owns parent-dir creation in jsonl test mode ŌĆö the helper writes the JSONL file synchronously after `WritePrompt` lands, and no other actor creates the dir there.

- **`budget.Counter.OnEvent` signature + semantic change.** Signature goes from `OnEvent(ev jsonl.Event)` to `OnEvent(entry tuidriver.JSONLEntry)`. Two reads remap: `ev.Kind != "assistant"` ŌåÆ `entry.Type != "assistant"`; the `if ev.EndOfTurn { return }` early-return branch at the old `budget.go:109-114` is **deleted**. The `c.mu` discipline and the budget-check arithmetic are untouched. The package's `jsonl` import drops; `tuidriver` takes its place. Package doc + `Counter` struct doc rewritten to reference "the agent-run caller's goroutine that drains the `tuidriver.TailJSONL` channel" in place of "the `tail.Watcher.Run` goroutine".

- **Budget-boundary semantic change committed: `max_turns` wins over `completion`.** Pre-#512, at the exact `MaxTurns`-th assistant entry which is also end-of-turn, `OnEvent`'s EOT branch short-circuited before the budget check fired; `OnEndOfTurn` then set `reason = ReasonCompletion`. Post-#512 the EOT branch is gone, so the budget check fires synchronously inside `OnEvent` (count = MaxTurns), sets `c.fired = true` + `c.reason = ReasonMaxTurns`, arms the grace timer, and calls `cfg.Terminate()`. The caller's subsequent `counter.OnEndOfTurn()` (ptyrunner invokes it because `IsEndTurn(entry)` is true on the same entry) is a no-op because the first-terminal-wins guard at `budget.go:145-147` only writes when `c.reason == ""`. The SIGTERM races claude's natural exit; both converge on "claude exits". The streamjson trailer reflects `ExitReasonMaxTurns` (driven by ptyrunner's Terminate callback into `emitter.SetExitReason`), and `Stop()` in the cleanup LIFO cancels the grace timer before SIGKILL fires. Deliberate change per the ticket's AC ("classification is the caller's job"); the pinning test `TestOnEvent_BudgetBoundaryEndOfTurnIsCompletion` is deleted along with the branch it pinned. A one-paragraph doc-comment on the new `OnEvent` body documents the boundary case explicitly so a future reader understands the deliberate shift.

- **`budget_test.go` literal migration.** The `assistantEvent(endOfTurn bool) jsonl.Event` helper becomes `assistantEntry() tuidriver.JSONLEntry` returning `tuidriver.JSONLEntry{Type: "assistant"}` (the `endOfTurn` parameter is dropped ŌĆö the entry no longer carries an EOT field for `OnEvent` to read). Every `jsonl.Event{Kind: ŌĆ”}` literal across ~14 sites becomes `tuidriver.JSONLEntry{Type: ŌĆ”}`. The non-assistant kinds loop (`user`, `tool_use`, `tool_result`, `system`, `attachment`, `""`) stays as a string slice. `TestOnEndOfTurn_ReasonCompletion`'s second event drops its `(true)` argument (the helper signature shrunk); the `Reason == ReasonCompletion`-after-`OnEndOfTurn` assertion is unchanged because the test runs at half-budget, not at the boundary. `TestOnEvent_SIGTERMFiresExactlyAtBudget` already covers the new boundary behaviour, so deleting `TestOnEvent_BudgetBoundaryEndOfTurnIsCompletion` does not reduce coverage.

- **`ptyrunner` test suite needed zero structural changes.** Every `runner_test.go` case (`TestRun_HappyPath_EmitsAndEndOfTurn`, `TestRun_CtxCancelDuringStream`, `TestRun_EmitErrorPropagation`, `TestRun_BudgetHitBeforeEndOfTurn`, `TestRun_WatchdogFires`, the four short-circuit-before-drain modal cases, `TestBuildArgs`, `TestRun_MissingRequiredFields`) exercises the full inline path through a real PTY + a real JSONL file on disk. The fixtures (`happyPathBody`, `noEotBody`, `helperRunCfg`) drive the new inline drain through the same filesystem path the deleted watcher used. `helper_test.go:124-130`'s doc comment was reworded to reference `tuidriver.WaitForSessionJSONL`'s stat-poll loop in place of `tail.New`'s `fsnotify.Add`; the `os.MkdirAll(filepath.Dir(path), 0o700)` on line 130 stays because the helper still owns parent-dir creation in jsonl test mode.

- **`internal/agentrun/jsonl/tail/{watcher,watcher_test}.go` deleted in full** (~700 LOC, two files). Behaviours the deleted code owned (file-appearance wait, partial-line reassembly, fsnotify event handling, EOT detection, ctx-cancel collapse) are now owned by `tuidriver.TailJSONL` and `tuidriver.WaitForSessionJSONL`, both covered by tui-driver's own test suite. The pyry-side ptyrunner tests provide integration coverage end-to-end.

## Patterns established

- **When a sibling subpackage exists solely to bridge two layers and both layers are migrating to a library equivalent, delete the bridge wholesale in the same ticket that pivots the consumer.** `internal/agentrun/jsonl/tail/` existed to wrap `internal/agentrun/jsonl`'s `Reader` (line parsing) with an `fsnotify`-based file-appearance + WRITE-event loop. Once `tuidriver.WaitForSessionJSONL` covered the appearance race ([#509](509.md)) and `tuidriver.TailJSONL` covered the read loop (this ticket), the bridge has no purpose. The temptation was to keep `tail.Watcher` as a thin shim around `TailJSONL` for "interface stability" ŌĆö that would preserve the `OnEvent` / `OnEndOfTurn` callback shape but add a layer of indirection that hides the actual library call from the consumer. Deleting it forces ptyrunner to know it depends on `tuidriver.TailJSONL` directly, which is the right truth: the callback shape was specific to the pre-library architecture, and after migration the inline `for range` loop is more legible than the callback fan-out. Generalises: **a bridge subpackage that wraps a library call with a callback abstraction is dead weight when the callback shape is exactly one consumer's preference; collapse the callback into a `for range` in the consumer**. Compare with `internal/sessions/rotation/` ŌĆö also an fsnotify wrapper, also a candidate for tui-driver delegation, but it has multiple call shapes and a different lifecycle, so the package shape carries its weight; #512's `tail/` did not.

- **A deliberate semantic change at a deleted-branch boundary needs a doc-comment on the surviving code, not just a deleted test.** The pre-#512 `OnEvent` had an early-return branch for `ev.EndOfTurn` that pinned `completion` over `max_turns` at the boundary. Deleting the branch is mechanically simple. Deleting the test (`TestOnEvent_BudgetBoundaryEndOfTurnIsCompletion`) that pinned the old behaviour is also mechanically simple. What's NOT obvious to a future reader: that the deletion was a *deliberate semantic flip*, not an accident. The fix is a one-paragraph comment on the new `OnEvent` body that names the boundary case explicitly ("at the boundary the budget wins because OnEvent grabs the reason first under c.mu, and the post-call OnEndOfTurn cannot overwrite a non-empty reason"). Without it, a future contributor reading the test suite might re-add the branch as "the obviously missing edge case" ŌĆö the same pattern as [#510](510.md)'s `ok && IsThinking` AND that was deleted *because* the library tightened its contract, except in the opposite direction here: a branch was deleted to *loosen* the consumer's reason classification. Generalises: **when a deliberate behaviour change is small (a deleted branch + a deleted test), the saved cognitive load belongs in a comment on the surviving code, not in a commit message or PR body ŌĆö the comment is what a future contributor sees first.**

- **A library wrap of `context.Cause(ctx)` through `%w` makes `errors.Is(err, context.Canceled)` work transparently; `isCtxErr` is the local helper that also handles bare-`ctx.Err()` fallthrough.** `tuidriver.WaitForSessionJSONL` and `TailJSONL` both wrap cancellation as `fmt.Errorf("ŌĆ”: %w", context.Cause(ctx))` when ctx fires, mirroring `tuidriver.WaitUntil` (the pre-existing seam from `Spawn` / `WaitUntil`). Ptyrunner's `isCtxErr(ctx, err)` helper (`runner.go:467-475`) checks both `errors.Is(err, context.Canceled|DeadlineExceeded)` AND a bare `ctx.Err() != nil` ŌĆö the double-check is defensive: `context.Cause` may return a wrapped cause that `errors.Is` doesn't reach, but `ctx.Err()` is always the bare sentinel. This pattern is now reused across four library call sites (`Spawn`, `WaitUntil`, `WaitForSessionJSONL`, `TailJSONL`) ŌĆö all four collapse-to-nil via the same helper, all four consistently surface non-ctx errors via `fmt.Errorf("ptyrunner: <stage>: %w", err)`. Generalises whenever delegating to a library that wraps `context.Cause`: **keep a single `isCtxErr(ctx, err)` helper and a single collapse-to-nil idiom at the call site**; the alternative (per-call-site `errors.Is` chains) drifts on the third or fourth call site.

- **An open-question section that survives spec ŌåÆ impl is a load-bearing decision marker, not just a TODO.** The architect spec's open question ŌĆö "Should `Run` MkdirAll the encoded project dir defensively?" ŌĆö was resolved in the spec itself with an evidence-based reasoning chain (no observed failure mode + test helper covers test mode + production has 100% historical reliability of claude creating the dir before writing). The decision propagated cleanly into impl (the MkdirAll is dropped) and into this codebase doc (one bullet documenting the rationale). Without the open-question framing, the spec might have papered over the question with a "and MkdirAll defensively, just in case" line; without the explicit "no observed failure" framing, a future contributor might re-add the MkdirAll on a hunch. Generalises: **a spec's open-question section is where Evidence-Based Fix Selection gets exercised in writing; resolved questions belong in the impl notes, not buried in commit messages**, so the next contributor reading the codebase doc sees both the decision and the reasoning.

## Lessons learned

- **A code-review NIT about silent error-loss can be deferred to a sibling restructure ticket rather than fixed in-place when (a) observability isn't fully lost and (b) the deeper fix is already filed.** Code review flagged that `tuidriver.TailJSONL`'s channel can close due to an unrecoverable read error (not ctx-cancel, not EOT), and the inline drain treats it as clean termination ŌĆö the `for range` exits, `runCtx.Err()` is nil, `emitErr` is nil, `Run` returns nil. Pre-#512 the watcher's `Run` returned a non-ctx error that `Run` surfaced as `ptyrunner: tail: %w`. The mitigating factors stacked up: the deferred `emitter.Close()` still writes `ExitReasonError` in the trailer (because `endOfTurnSeen` is false), tuidriver's docs call the failure "rare on an append-only file", and [#513](https://github.com/pyrycode/pyrycode/issues/513) is queued specifically to restructure the `emitErr` capture-then-prioritise pattern (the natural place to add a "channel-closed-without-EOT and ctx-not-cancelled" branch). PASS with the NIT noted. Generalises: **a NIT is the right call when the in-place fix would partially reinvent a sibling ticket's restructure**; absorbing it into the sibling keeps the diff scoped and the restructure cleaner. Compare with [#509](509.md)'s NIT about the stale `Watcher.file` doc-comment ŌĆö also deferred for the same reason: the next contributor touching the file picks it up at zero cost.

- **The "out of scope" footnote in the spec is what kept the diff at S size.** PO and architect both explicitly named `internal/agentrun/jsonl/` deletion AND the `emitErr` capture-then-prioritise restructure as out-of-scope. The spec audit found two non-tail consumers (`selfcheck` and `realclaude/fixtures`) that still need `jsonl.NewReader` over an `io.Reader`-to-EOF; migrating them needs either a new tui-driver export (`ParseEntry(line []byte) (JSONLEntry, bool)`) or an inlined small parser, neither of which fits in an S-sized ticket. Without the upfront audit + explicit scope cut, the ticket would have grown to a sweeping "delete the jsonl package + restructure the emitErr pattern + migrate selfcheck + migrate realclaude fixtures" multi-cycle change. Generalises: **before pivoting a package's primary consumer to a library equivalent, audit *all* consumers up-front and explicitly defer the non-target consumers if they need a different library shape**; the audit cost is one `grep` and one paragraph in the spec, and it's load-bearing for scope discipline.

Options: (a) export `tuidriver.ParseEntry(line []byte) (JSONLEntry, bool)` upstream and have both consumers `bufio.Scanner` + `ParseEntry`; (b) inline a ~25 LOC parser in each consumer (copy of `streamjson_test.go`'s `lineToEntry`); (c) ship a new `tuidriver.ReadJSONL(r io.Reader) <-chan JSONLEntry` library helper. Option (a) is the cleanest ŌĆö single source of truth, drops the duplication in `streamjson_test.go:lineToEntry` too. Filed-time decision belongs in the next ticket's spec.
- **[#513](https://github.com/pyrycode/pyrycode/issues/513) ŌĆö restructure `ptyrunner.Run`'s `emitErr` capture-then-prioritise pattern AND surface tuidriver-channel-close-without-EOT errors.** The code-review NIT (see ┬¦ Lessons learned) is the natural #513 surface: when the channel closes without EOT and ctx is not cancelled, return a new error wrap (`ptyrunner: tail closed without end_turn`) rather than nil. The restructure is the right home because the four error sources (emit failure, channel-close-no-EOT, ctx-cancel, EOT-clean-exit) need a coherent prioritisation, and the current capture-then-prioritise pattern was designed for two sources, not four.
