# Ticket #261 — E2E failure paths for `pyry update` (fetch / verify / broken binary)

Companion to #260: three new e2e tests in `cmd/pyry/update_e2e_test.go` (`(darwin || linux) && e2e_update`) pin the failure shapes of the self-update flow against the in-process fake release server — fetch HTTP 500, checksum mismatch, and broken-new-binary-after-AtomicReplace. The broken-binary case is an *asserts-the-current-design* test: per `docs/knowledge/features/pyry-update-command.md` + `docs/specs/architecture/187-update-atomic-replace.md`, `pyry update` has no rollback — once `AtomicReplace` swaps the new bytes in, the operator must intervene. Test-only change (+333 LOC `update_e2e_test.go`, +19 LOC `internal/brokenpyry/main.go`); zero production diff.

## Implementation

- **Three sibling tests share `installPreUpdateDaemonE2E(t)`.** The first ~30 lines of every failure-path test repeat #260's pre-update install + spawn + inode/PID capture. Extracted to a returns-`*preUpdateState` helper that bundles `targetPath`, `home`, `socket`, `inodeBefore`, `pidBefore`, `cmd1`, `done1`, `stdout1`, `stderr1`. The helper deliberately does NOT register `cmd1` cleanup — the broken-binary test's `runRestart` closure stops cmd1 mid-test, and registering cleanup inside the helper would force a `cmd1Stopped` flag visible to the closure, coupling the helper to test-local control flow. Fetch-/verify-failure tests register `stopDaemonE2E(s.cmd1, ...)` in `t.Cleanup` themselves (one line); broken-binary gates on the closure-local `cmd1Stopped`. `TestUpdate_HappyPath_E2E` stays append-only (does NOT switch to this helper) — append-only kept the spec from churning #260 a second time.
- **`TestUpdate_FetchFailure_E2E` uses a one-off `newFetchFailReleaseServer`, not `newFakeReleaseServer`.** Growing `newFakeReleaseServer` a failure-injection knob (e.g. `assetStatus int`) for a single caller would contaminate its signature against four happy-path callers in `update_test.go`. A parallel ~20-line constructor that serves `/repos/.../releases/latest` (200 with `tag_name`) but returns 500 on the asset download URL is the principled alternative — exercises `o.fetcher.FetchAsset(ctx, tarballURL)`'s error path at `update.go:162-165`, which surfaces as `update: download tarball: …`. The checksums URL is wired but never hit (asset download fails first).
- **`TestUpdate_VerifyFailure_E2E` reuses `newFakeReleaseServer` directly with a bogus checksums body.** `newFakeReleaseServer` accepts the checksums body as a parameter — no new helper required. The body is `fmt.Sprintf("%s  %s\n", strings.Repeat("0", 64), asset)` (64-zero digest line keyed to the genuine asset name); tarball downloads successfully and `update.VerifySHA256` returns `ErrChecksumMismatch`, wrapped as `update: verify checksum: …`. AC asserts non-nil error only — no substring pin on the message.
- **`TestUpdate_BrokenNewBinary_E2E` reuses `runRestart` from happy-path verbatim.** The kill-and-respawn closure stops daemon 1, spawns daemon 2 from `s.targetPath` (now holding the broken-pyry bytes via `AtomicReplace`), and waits for the socket. The broken binary writes `BROKEN_PYRY_TOKEN` to stderr then `os.Exit(1)` before opening any socket → `spawnDaemonE2E`'s wait-goroutine closes `done2` ~immediately → `waitForSocketE2E`'s `<-doneCh` short-circuit returns `"daemon exited before ready"` → the closure propagates that → `doUpdate` wraps at `update.go:202` as `update: binary replaced to <v>, but daemon restart failed: <inner>`. The test asserts both substrings (`"binary replaced to "` + `"daemon restart failed"`, version-agnostic — same shape as `TestUpdate_RestartFailure` at `cmd/pyry/update_test.go:438-460`), inode-changed, on-disk bytes equal `brokenBytes`, success line absent, and stderr2 contains `BROKEN_PYRY_TOKEN` (diagnostic guard — localizes "broken helper ran" from "some other binary exited early" cleanly).
- **`runRestart` is a `t.Fatalf` sentinel for fetch- and verify-failure.** Both paths return errors from `doUpdate` BEFORE the AtomicReplace step, so `runRestart` is never invoked. Registering a fatal-on-call closure (mirroring `TestUpdate_Success`'s shape at `cmd/pyry/update_test.go:113-116`) is cleaner than wiring the kill-and-respawn closure and letting it sit unused — and it catches a regression that would silently fire `runRestart` on a path it shouldn't reach.
- **Shared assertion helpers** (`assertBinaryUnchangedE2E`, `assertDaemonAliveE2E`, `assertNoStragglersE2E`, `assertNoSuccessLineE2E`) keep each test body ~30 lines. `assertDaemonAliveE2E` does both the PID-unchanged check the AC names (one-liner; trivially true within the test's process model — `cmd1.Process.Pid` is captured at `Start()` and never mutates — but documentation that the invariant holds end-to-end), the cheap `os.FindProcess` + `Signal(syscall.Signal(0))` localizer (catches "process died unrelated to update" cleanly), AND the structural `pyry status` round-trip against the original socket. `assertNoStragglersE2E` reads `<home>/bin/` and asserts exactly one entry named `pyry` — no `.pyry.*.tmp` files; trivially true on the fetch-failure path (AtomicReplace never runs) but exists to catch regressions where a future change leaks files before the rename. Skipped on the verify-failure path (structurally identical to fetch-failure — exits before AtomicReplace) and the broken-binary path (AtomicReplace succeeded; its defer-remove only fires before rename — the inode + on-disk-bytes assertions already pin post-replace state).
- **`internal/brokenpyry/main.go` is the broken-pyry stand-in** — 19 LOC, `package main`, writes `BROKEN_PYRY_TOKEN: broken pyry stand-in exiting non-zero` to stderr and `os.Exit(1)` on every invocation. No flag parsing, no signal handling, no `-pyry-socket=` consumer — the binary exits before any of that would run, which IS the failure mode under test. Location is `internal/brokenpyry/` (not `cmd/pyry/internal/brokenpyry/`): a `package main` directory cannot be imported by other Go code, so the visibility tightening that `cmd/pyry/internal/` would provide doesn't apply (the path is just a build-target string for `go build`). Flat `internal/<X>/main.go` matches the existing repo shape (`internal/e2e/`, `internal/install/`, `internal/update/`).
- **`buildBrokenPyryBinE2E(t) string` mirrors `buildPyryBinE2E` exactly** — `go build -o <tmp>/brokenpyry github.com/pyrycode/pyrycode/internal/brokenpyry` with a `PYRY_E2E_BROKEN_BIN` short-circuit for CI prebuild. Same shape and CI contract; one-line target swap + env-var rename. Honoured-env-var pattern is the lessons.md-approved alternative to a `--release-base-url` flag or `PYRY_RELEASE_BASE_URL` env var: build-time tunable for CI, never touches production behaviour, never alters `doUpdate`'s control flow.
- **No new ADR.** All design choices anchor in existing precedent: helper-reuse follows `update_test.go`'s pattern; `runRestart`-as-sentinel mirrors `TestUpdate_Success`; the no-rollback contract is already documented in `pyry-update-command.md` § "Out of scope" and ADR 187; the `internal/brokenpyry/` placement follows the existing `internal/e2e/internal/fakeclaude/` precedent (test-only `package main` helper under `internal/`).
- **No `--release-url` CLI flag.** The ticket body and lessons.md ("Reject hidden env vars added 'just for the test'") explicitly forbid adding a production CLI surface for test-only wiring. `doUpdate` is driven directly from the same-package test, same seam #260 established.

## Lessons learned

- **`runRestart`-as-sentinel beats running-but-ignored on paths that shouldn't reach it.** Two of the three failure tests exit `doUpdate` before AtomicReplace, so `runRestart` is structurally unreachable. The temptation is to wire the same kill-and-respawn closure the broken-binary test uses "for consistency" — but if a regression DOES make `doUpdate` reach `runRestart` on the fetch-/verify-failure path, the closure happily kills daemon 1 and the test surfaces a misleading "daemon 1 dead" failure several assertions later. A `t.Fatalf` sentinel fails loud at the wrong-path-reached moment, with a clear message, exactly where the bug is.
- **Diagnostic stderr-token assertions cost ~3 lines and save 30 minutes of debugging.** The broken-binary test asserts `stderr2` contains `BROKEN_PYRY_TOKEN`. Without it, a future regression where (say) the spawn target gets wired to `/bin/false` instead of the broken-pyry binary would still produce a `daemon exited before ready` → `daemon restart failed` chain — the structural error contract assertion passes, but the test isn't actually exercising the broken-pyry path it claims to. The token assertion localizes "broken helper ran" from "some other binary exited early" the moment the regression lands.
- **Per-ticket `<N>.md` files now have a second consumer.** This is the second slice in the e2e-update series (after #260) — having `260.md` as a self-contained reference for the seam design + helper inventory let this ticket re-anchor on "see #260" instead of re-explaining the package-main-test-seam decision. The convention's payoff compounds as adjacent slices land.

## Out of scope, by design

- **Rollback / `.bak` artefact behaviour.** The broken-binary test asserts the *currently designed* contract (no rollback, error surfaced). If a future ticket introduces `.bak` rollback, the assertion changes — that ticket owns the new assertions.
- **Refactoring `TestUpdate_HappyPath_E2E` to use `installPreUpdateDaemonE2E`.** Append-only; #260 just landed, no regression risk by leaving it.
- **Cross-architecture / cross-OS scenarios.** Build tag enforces darwin || linux.
- **Permission-denied `AtomicReplace`** — not in this ticket's AC; structurally would be a sibling test but the project doesn't have a documented operator scenario for it yet.
- **Promoting `installPreUpdateDaemonE2E` to a shared helper outside this file.** Per the ticket body this is the last failure-path ticket in the series — three callers and a known-finite TODO list isn't a strong promotion signal. If a fourth e2e_update test materializes, that ticket promotes.
