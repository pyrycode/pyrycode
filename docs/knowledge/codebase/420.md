# Ticket #420 — `e2e/realclaude`: permission-denied user-facing signal assertion

Extension to `TestRealClaude_AllowedToolsEnforcement` ([#365](365.md)) that, after the gate-held loop completes, asserts the run produced *some* operator-visible signal — either an assistant-text refusal keyword or a structured `result` envelope on stdout carrying `permission_denials` or `is_error=true`. Closes the regression-alarm gap left by [#365](365.md): the original gate-held assertion would happily pass a future regression where the gate silently no-ops on a denied call (no assistant turn, no diagnostic), leaving the operator staring at a hung-looking session. This second co-located assertion makes that failure mode loud.

The contract guarded is "user gets *some* interpretable signal on denial", not "user gets *this specific* signal". The disjunctive shape is load-bearing: a future claude that swaps text ↔ structured channels still satisfies the test; only a silent no-op fails. Companion to [#418](418.md) (captured-trace null-findings regression on the spike argv) — different argv, no overlap.

## Implementation

- **One-file extension to `internal/e2e/realclaude/allowed_tools_enforcement_test.go`.** +111 LoC, no new file, no fixture-helper changes. Architect chose extension over sibling test because the new assertion needs the exact same argv, model, prompt, and `RunResult` as the existing test — a sibling test would re-run the same haiku call (~$0.01) for no behavioural difference, and the two assertions guard one contract against one model call. The test name `TestRealClaude_AllowedToolsEnforcement` stays accurate — the new assertion is still about enforcement behavior, just on a different axis.
- **Phases 5–6 added after the existing gate-held loop.** The four pre-existing phases (worktree → run → run-level asserts → JSONL walk for forbidden Bash `tool_use`) are unchanged byte-for-byte. The new code runs only when the gate held (the loop continues to `t.Fatalf` on a positive hit, so phases 5–6 are unreachable on a gate-regression).
- **`denialKeywords []string` — file-private fixed set.** Five lowercase substrings: `"cannot"`, `"can't"`, `"unable"`, `"not allowed"`, `"permission"`. Set kept narrow on purpose — broad markers like `"available"` or `"access"` appear in non-refusal contexts and would false-positive. The disjunctive design (text OR structured) absorbs the case where a future haiku declines with outside-the-set wording AS LONG AS it still emits a structured signal; if both channels silently drift the keyword set expands by one or two markers in a follow-up, not by weakening the assertion.
- **`assistantTextRefusalHit(events []JSONLEntry) (bool, int)` — file-private, signature-minimal.** Walks `events`, filters `e.Kind == "assistant"`, decodes `e.Raw` into a narrow struct exposing `message.content[].{type,text}`, and for every `type=="text"` block runs `strings.Contains(strings.ToLower(text), kw)` against each keyword; first hit short-circuits. JSON unmarshal failures skip the line silently — mirrors `bashInvokedInRaw`'s selfcheck.go:283 discipline so a single malformed line can't turn a PASS into an inconclusive. Returns the assistant-entry count alongside the hit so the failure message can quote the search width.
- **`structuredDenialHit(stdout []byte) (bool, int)` — file-private.** `bufio.NewScanner(bytes.NewReader(stdout))` over the captured stream-json stdout, default 64 KiB buffer (result lines are well under that — confirmed against the spike fixture set; no buffer-cap workaround needed). Per line: decode into a narrow `struct{Type string; IsError bool; PermissionDenials []json.RawMessage}`, hit condition is `env.Type == "result" && (len(env.PermissionDenials) > 0 || env.IsError)`. Non-JSON and non-`result` lines are skipped silently. Returns the line count alongside the hit.
- **Disjunctive combine at the call site.** `if !textHit && !structHit { t.Fatalf(...) }` — either alone is sufficient; both miss is the failure mode being guarded. Failure message names `denialKeywords` (the brackets-list echo so a future maintainer sees what the detector accepted), assistant-entry count, stdout-line count, and the resolved JSONL path; **never** echoes raw assistant-text bytes, raw stdout bytes, or raw envelope bytes (selfcheck `selfcheck.go:217-219` discipline applied even though the prompt is synthetic).
- **`is_error` fallback alongside `permission_denials`.** `permission_denials` is the canonical channel today (confirmed in `internal/e2e/realclaude/testdata/permission_protocol_v2.1.143_*.json` and `internal/agentrun/streamjson/testdata/captured_run.jsonl`); the `IsError` branch is the forward-compat tolerance for a future claude release that routes denial through `is_error=true` plus a subtype change without touching `permission_denials`. The detector deliberately stops there — accepting `subtype != "success"` would let unrelated failure modes (max-turns, timeout, API rejection) masquerade as denial signals, muddling the contract.

## Patterns established

- **Co-locate two assertions on one contract when they need the same argv + same model call.** Architect's call: sibling-test vs. extension is a cost+coupling tradeoff. When the two assertions guard one contract end-to-end (same prompt, same gate, same `RunResult`) and one would be a verbatim re-run of the other's setup, extension wins — one model call per CI run, contract visible in one place, no asymmetric drift risk where a future maintainer fixes a prompt for one test and forgets the other exists. The pre-existing four-phase shape stays unchanged and the new assertion appends as phases 5–6; the test name still describes both axes accurately. Generalises to future regression-guard pairs that share an argv shape.
- **Disjunctive signal-detection over channel-pinning when guarding "operator gets *some* signal".** When the contract is "user perceives the denial somehow", pinning a specific channel (assistant text, structured envelope, stderr keyword, exit code) overfits — a future upstream that swaps channels would break the test for a non-regression. Two file-local detectors with `OR` at the call site is the right shape: each detector is small, single-purpose, and stays valid as long as its own channel is one of the acceptable surfaces; the combine point is where the contract is encoded. Generalises to any "interpretable signal" assertion across heterogeneous output streams.
- **Narrow keyword set, not exhaustive vocabulary.** The text-channel detector's keyword set is the ticket's example list verbatim (five entries); the temptation to widen ("declined", "refused", "lack", "access") is the wrong move because broader markers false-positive in non-refusal contexts. The disjunctive design tolerates a keyword miss as long as the structured channel fires — so the right keyword discipline is "narrow and high-signal", with widening as a follow-up driven by observed flakes, not by speculation.
- **`(bool, int)` return on signature-minimal detectors when the count is load-bearing for diagnostics.** Both new detectors return the entries scanned alongside the hit so the failure message can quote the search width (`across N assistant entries`, `across M lines`). The count is cheap to track, makes the failure message self-diagnosing without forcing a future maintainer to instrument and re-run, and stays out of the way on a green pass (Go's blank identifier handles the discard at the lone call site — although here both values are consumed). Skipping the count would have shortened the function by one variable; preserving it is the right cost/value trade for a once-per-CI assertion. Generalises to any test detector whose failure mode is "nothing matched anywhere".

## Lessons learned

- **The contract is "some signal", not "this specific signal" — designing the test that way takes deliberate restraint.** First instinct on writing a "did the operator see the denial" test is to lock on whatever channel claude uses today (`permission_denials` non-empty, or the literal text the model produced on the spec-author's local rehearsal run). That overfits. The disjunctive shape is harder to write (two detectors, two narrow parses, an `OR` at the combine) and harder to read at a glance than `assert len(permission_denials) > 0`, but it is the only shape that survives an upstream channel swap without flapping on a non-regression. Worth re-reading the ticket's "Test does NOT pin which of the two channels carries the signal" AC line before reviewing this kind of test in the future.
- **Don't include both `permission_denials` AND `subtype != "success"` in the structured detector.** Tempted to widen the structured-channel hit condition to `env.Type == "result" && env.Subtype != "success"` as a generic "the model didn't end cleanly" tripwire. Wrong — that would conflate denial with unrelated failure modes (max-turns hit, timeout, API rejection). The detector stays scoped to `permission_denials` (canonical today) + `is_error` (forward-compat for a denial-subtype rename). If a future release introduces a third denial-shaped envelope field, that's when the detector grows, not pre-emptively.
- **`bufio.Scanner` default buffer is fine for stream-json `result` lines.** Briefly considered raising the buffer cap on the structured-channel scanner (the spike test uses 1 MiB to handle large model output). Confirmed against the committed permission-protocol fixtures and the captured-run testdata that `result` envelopes are well under the 64 KiB default — they carry usage stats, denial lists, and metadata, not assistant text bodies. Defensive raise was unnecessary; if a future claude version starts inlining long content in the `result` envelope, the failure mode is a `bufio.Scanner` truncation hit and the right fix at that point is to raise the cap with a cite to the new shape.

## Out of scope (deferred)

- **Pinning specific assistant-text wording.** The keyword set is the ticket's example list verbatim and is intentionally not exhaustive. If a future haiku revision uses outside-the-set language (e.g. `"I lack access to Bash"`) AND simultaneously emits no structured denial signal, the fix is to expand the keyword set in a follow-up — not to weaken the assertion to "any non-empty assistant text".
- **Pinning a specific structured-envelope subtype shape.** Detector accepts `permission_denials` non-empty OR `is_error=true`. Widening to accept `subtype != "success"` would conflate denial with max-turns / timeout / API-rejection failure modes; widening to accept a specific future subtype (`error_permission`) is the wrong shape because it overfits, and the `is_error` fallback already covers that branch generically.
- **A separate fixture-walk regression on the agent-run argv.** [#418](418.md) covers the captured-trace direct-`claude` argv; this test covers live `pyry agent-run --allowed-tools` behavior. Different argv → different test.
- **Boot-time selfcheck argv coverage.** [#336](https://github.com/pyrycode/pyrycode/issues/336)'s domain (`--settings` / `defaultMode=deny`); different argv, different test.
- **Asserting "the model picked Read instead" or "the assistant turn was non-empty in some other sense".** The contract is `text-channel-hit || structured-channel-hit`, nothing more.
