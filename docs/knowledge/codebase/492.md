## Patterns established

- **Operator-facing front door above the developer-facing back door.** When a single doc has to serve both audiences — an operator who wants the workflow and a developer who wants fixture internals — the operator section goes on top, before the implementation deep-dive, with a forward link to the fixture paragraph for readers who need the internals. The fixture paragraph stays as-is (skip messages, `t.Setenv` semantics, `HOME`-pinning rationale); the new operator section names what env var to set and how to obtain it, then defers. This split avoids two failure modes the alternative arrangements have: (a) inlining operator-facing prose into the fixture paragraph buries the workflow several screens into a code-internals discussion (the pre-#492 state, which is what forced the #489 recon to happen at all); (b) duplicating the extraction recipe into both sections creates a drift surface where one of the two copies will eventually lag a credential-format change. The applied pattern uses a single source of truth (the operator section's fenced shell block) and a one-line cross-reference from the fixture paragraph when the fixture paragraph needs to mention the recipe.

- **Negative controls are documentation, not anti-features.** The three "what does NOT work" bullets — `~/.claude.json` copy, full `~/.claude/` copy, lifting `$HOME`-pinning — exist purely so the next operator doesn't repeat the recon that produced them. They occupy four lines of doc but save several hours of someone independently re-running experiments that already ran on 2026-05-20. The convention: when ticket recon eliminates approaches that look plausible but don't work, name them by their concrete failure mode (`apiKeySource:"none"`, "Not logged in") rather than hand-waving ("we tried other things and they didn't work"). Concrete failure modes let the next reader identify whether they're hitting the same dead end or a different one.

- **Token-rotation traps deserve two patterns, not a recommendation.** The doc shows Pattern A and Pattern B as a tradeoff matrix — source-once at shell startup (fast, but goes stale in long-lived shells) vs. just-in-time shell function (always fresh, but pays Keychain access per call and re-triggers the prompt if **Always Allow** wasn't chosen). It explicitly does NOT recommend one over the other. Reason: the right choice depends on operator workflow (one terminal open for days vs. ephemeral shells) and Keychain prompt preference, both of which the doc can't infer. Generalises: when a workflow has a genuine trade-off matrix, document the matrix; reserve recommendations for cases where one pattern dominates the other on the dimensions an operator can be expected to care about.

- **Don't expand `## Related` for in-bullet cross-references.** The new section's two outbound ticket links (#489 inside the negative-controls intro, #406 inside the out-of-scope CI bullet) are contextual cross-references — they justify a specific claim inside a specific bullet, not standalone "see also" entries. Adding them to the doc's bottom-of-file `## Related` block would inflate that block with links whose context is already pinned where they're cited. The convention: `## Related` is for top-level "you might also want to read this" entries; inline ticket numbers stay inline.

## Lessons learned

- **The split sequence (#489 → #490 → #491 → #492) is the right shape when a recon produces both code changes AND doc-worthy operator knowledge.** #489 was the umbrella ticket whose recon produced four artifacts: the fixture-helper change ([#490](490.md)), the consumer-side flip ([#491](491.md)), the operator-auth doc section (this ticket), and the closing post-mortem on #487. Folding all four into a single ticket would have stranded the doc section behind the code changes' review cycle, and the code changes' review attention would have spilled into reviewing prose that didn't share its blast radius. Splitting let each ticket get the right kind of review (code review for #490/#491, prose review for #492) and let the doc explicitly cite the just-landed code as the contract it describes — which is exactly what the architect's "depends on the sibling fixture and test-guard children landing first" constraint encoded. Generalises: when an investigative recon ticket produces N artifacts that span code AND documentation, prefer one ticket per artifact with an explicit dependency chain over a single multi-deliverable ticket; the dependency-chain shape gives prose tickets a contract to describe and code tickets a free review surface.

- **PR body's optional "Lessons learned" section was correctly omitted by the developer.** Per the documentation agent's contract, the developer flags non-obvious surprises in the PR body's "Lessons learned" so they can be lifted into the codebase doc. PR [#495](https://github.com/pyrycode/pyrycode/pull/495) has no such section. The work itself was mechanical: the spec dictated structure, the AC dictated content, the negative controls came from #489's body verbatim, the extraction recipe came from the existing L98 fixture paragraph verbatim, and code review was PASS with one cosmetic NIT that didn't warrant a follow-up. No genuine surprise to lift. Same pattern as [#491](491.md) — not every ticket produces a lesson, and fabricating one to fill the section is anti-signal for future readers.

- **Code review's "below" / "above" prose-orientation NIT is genuinely a no-fix.** The review flagged that the negative-controls bullet references "the `WithWorktreeAuthenticated` paragraph below" — and from the operator-reading-top-down perspective the fixture paragraph IS below (it's at L98, the cross-reference is at L83). The phrase reads naturally because operators read top-to-bottom; rewriting it to "the `WithWorktreeAuthenticated` paragraph at L98" would couple the prose to line numbers that drift on every subsequent edit. Generalises: directional prose ("above" / "below") that's correct under the reader's natural traversal order is preferable to line-number anchors that drift; only switch to numbered anchors when the directional reference is genuinely ambiguous (e.g. the doc has multiple `WithWorktreeAuthenticated` mentions and "below" could mean two different ones).

## Out of scope (follow-ups)

- **In-suite token refresh.** Fixture does not re-read Keychain mid-run; a token extracted at process start that expires mid-run surfaces as a subprocess auth failure. Documented as out-of-scope in the new section; no follow-up ticket filed because the failure mode hasn't been observed in practice (per the "Evidence-Based Fix Selection" principle in `agents/documentation/CLAUDE.md`).
- **CI Keychain access.** GitHub Actions runners have no macOS Keychain; the Max-OAuth path is operator-machine only. Tracked separately under [#406](https://github.com/pyrycode/pyrycode/issues/406) (CI-secret-injection bug). The realclaude suite's CI absence is intentional — per the existing "CI cadence" section, the suite runs locally during the code-review phase of every dispatched ticket via the pipeline.
- **Defaulting `pyry agent-run` to OAuth on the operator's Mac.** Not needed: the dispatcher runs in the operator's real shell where Keychain works normally, and the pre-existing `claude` installation handles auth without pyry's help.
- **Auto-rotation pattern.** Neither Pattern A nor Pattern B observes Keychain for the next refresh and re-exports automatically. A `launchd`-watched-file or `fswatch` shim could do this, but the per-shell-startup + per-command patterns cover the common cases without introducing a background process. If operators report stale-token surprises in practice, a third pattern can be added; until then, two patterns is the right cardinality.
