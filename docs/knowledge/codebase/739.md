## Implementation

Two surgical additions, no new types, no new files (`internal/conversations/registry.go`, `internal/sessions/transition.go`, ~45 production LOC):

- **`conversations.Registry.RebindSession(oldID, newID string) bool`** â€” the write primitive, mirroring the `Update`/`Delete` idiom: scan + mutate as a **single `r.mu` critical section** (no find-then-update TOCTOU window), first-match-and-stop, **no implicit `Save`** (caller owns persistence). On a hit it sets `CurrentSessionID = newID` and `SessionHistory = append(SessionHistory, oldID)`; on a miss it returns `false` with no mutation. A leading `oldID == ""` short-circuit returns `false` **before scanning** â€” an unbound conversation carries `CurrentSessionID == ""`, and a stray empty-id call must never sweep the first unbound row into a rebind.

- **`Pool.rebindConversation(oldID, newID SessionID)` + a `ReasonClear` branch in `notifyTransition`** (`transition.go:50`). `notifyTransition` is the off-lock chokepoint **both** transition reasons pass through; the new branch rebinds **before** the observer fan-out:

  ```go
  func (p *Pool) notifyTransition(t SessionTransition) {
      if t.Reason == ReasonClear {
          p.rebindConversation(t.PreviousID, t.NewID)
      }
      if p.transitionObserver != nil {
          p.transitionObserver(t)
      }
  }
  ```

  `rebindConversation` is a no-op when no registry is wired (`p.convReg == nil`, the case for every existing `transition_test.go` pool) or when `RebindSession` misses (AC#4 â€” `Save` is skipped so the file mtime stays stable). On a successful rebind it persists `conversations.json` via the registry's atomic `Save`; a `Save` error is logged at `Warn` (`event=rebind_conversation.persist_failed`, only session-id routing fields, never conversation content) and swallowed â€” the in-memory rebind is already applied and usable, so durability is best-effort, matching `create_conversation`'s eager persist and `RotateID`'s non-fatal save.

- **Production wiring is live** (verified by code review): `onRotate` is the rotation watcher's `OnRotate` callback (`pool.go:826`); `convReg`/`convRegistryPath` are non-nil in production (`main.go:676,716` â†’ `Pool.New` at `pool.go:418`), so AC#1/#3/#4 fire for real. The `Save`-error `Warn` dereferences `p.log`, which `New` always sets (`slog.Default()` fallback at `pool.go:305`) â€” panic-safe.

## Patterns established

- **Drive transition side-effects from `notifyTransition`, not a second observer.** The transition signal has a **single** observer slot (`SetTransitionObserver`, "set once"), already owned by the `session_transition_v2.go` producer (`:207`, installed before `Pool.Run`). A second observer is impossible. Co-locating the rebind at the common off-lock chokepoint â€” rebind *then* observer â€” makes the #741 ordering **structural**: the in-memory rebind (and its `Save`) complete synchronously before the observer hands the signal to its buffered channel, so #741 always resolves sessionâ†’conversation against the *current* binding.

- **Eviction binding-neutrality is structural, not a runtime guard.** `ReasonEviction` fires with `PreviousID = s.id` and an **empty `NewID`** (`session.go:291`): the session keeps its id, stays in the pool map, and re-activates later under the *same* id, so `CurrentSessionID` stays valid with no write. Because `runActive` returns only `ReasonEviction`/`""` (never `ReasonClear`), an eviction never enters the rebind branch â€” the no-corruption guarantee (AC#2) holds by control flow, not by inspecting `NewID`. The naive failure it avoids: applying "set `CurrentSessionID = NewID`, append `PreviousID`" to an eviction would clear the binding (breaking the `send_message` respawn guard at `main.go:923`) and append a colliding duplicate of the current id.

- **New lock edge, no cycle:** pool transition path â†’ `conversations.Registry.mu`, taken with **no** `Pool.mu`/`Session.lcMu` held (`onRotate` releases `Pool.mu` inside `RotateID` before calling `notifyTransition`). `RebindSession`'s body never calls back into the registry, so the `Update`-callback re-entrancy hazard does not apply. Concurrent rotation-`Save` vs sweep-`Save` on the same path is the existing atomic temp+fsync+rename posture â€” the later rename wins with an equally-consistent snapshot.

- **`SessionHistory` is written in production for the first time.** Through #738 the field was documented but never mutated by any production caller. The append runs **under `r.mu`** and only ever writes at index â‰Ą the old length, so the registry's shallow Save-snapshot stays race-free without a deep copy (see [conversations-registry.md](../features/conversations-registry.md) Â§ *Save concurrency*); `TestPool_OnRotate_RebindRaceConcurrentSave` exercises it under `-race`.

- **Write side only.** This ticket owns the *maintenance* (write); #741's reverse *read* lookup (session id â†’ conversation id) is a separate registry scan. Per PROJECT-MEMORY's "Resist over-DRY on duplicated registry primitives," no shared scan helper was extracted â€” the duplication is tolerated until a third consumer forces it.

## Security (security-sensitive â€” verdict PASS)

The binding maintained here is the attribution a downstream mobile-facing consumer (#741) uses to route a session-boundary event to a conversation, so a mis-write is a cross-conversation leak that **originates here**. The rebind is driven **entirely by server-internal state** (the daemon's own rotation watcher and pool-managed session UUIDs) â€” no untrusted phone input flows in. Mis-attribution is prevented by deterministic byte-exact first-match (each session id binds exactly one conversation, set once at creation) plus the `oldID == ""` guard; the eviction-corruption threat is prevented by the `ReasonClear` branch. `Save` mutates exactly the matched row and persists the whole consistent snapshot atomically â€” no torn write. Logging carries only non-secret session-id routing fields.

One **non-blocking** code-review precision note on the spec prose: the `oldID == ""` guard does **not** defend against a mis-routed *eviction* (an eviction carries a non-empty `PreviousID`); the sole real eviction defense is the `ReasonClear` branch. The guard genuinely defends a *different* catastrophe â€” a stray empty-`oldID` call matching the first unbound conversation. The implementation is correct as written; only the spec's "belt-and-suspenders layer 2 against eviction" framing over-attributes.

## Lessons learned

- **Eviction vs rotation are different binding events.** A "maintain binding across rotation AND eviction" framing conflates two reasons of one fix: only `/clear` rotation changes the session id (binding goes stale â†’ rebind); eviction keeps the id (re-activates the same id) â†’ binding stays valid â†’ **no** rebind, and a naive rebind on the empty-`NewID` eviction signal would actively corrupt the binding. The rebind must branch on reason.
- **The empty-id guard and the reason-branch defend different threats.** Don't conflate them: the `ReasonClear` branch is the eviction defense; the `oldID == ""` guard defends the unrelated stray-empty-call-sweeps-unbound-row case. (Code-review precision note above.)
- **The repo is gofmt-dirty at HEAD under local Go 1.26.** A pre-existing struct-field-alignment block (`registry_test.go` ~L578) flags under local Go 1.26.2 due to skew vs CI's pinned Go â€” not introduced by this PR; left untouched to avoid spraying spurious diffs that fight CI. See [[pyrycode-gofmt-dirty-at-head-go1.26]].
- **Go grouped `const` blocks don't inherit a type per-line.** The first test-const-group revision typed only the first constant; the untyped remainder tripped `SA9004` (`staticcheck`). Fix: drop the lone first-constant type annotation so each group is uniformly untyped â€” the UUID literals convert implicitly to `ConversationID`/`SessionID` at their use sites (test-only, no behavior change; commit `60bd531`).
