# Ticket #430 — Mobile wire protocol v2 (Noise_IK E2E encryption) spec

Architect-deliverable PR against [`docs/protocol-mobile.md`](../../protocol-mobile.md): rewrites the v1 draft as v2, layering end-to-end encryption (Noise_IK over X25519/ChaChaPoly/BLAKE2s) over the existing relay topology. No code changes — this ticket lands the spec the implementation children (#431, #432, #434, #435, #436) build against. The locked-decisions inputs (six items, decided 2026-05-16) and four architect-level open questions (Go library, Kotlin library, re-key envelope shape, `payload_encrypted` flag fate) are resolved inline in the spec.

## Implementation

- **`docs/protocol-mobile.md` rewritten v1 → v2.** Single-file PR; v1 draft preserved in git history only (no `protocol-mobile-v1.md` sibling — hard cutover means no parallel maintenance burden). Six locked decisions encoded as spec sections; four architect questions resolved inline with explicit decision text plus rationale.
- **Cipher suite: `Noise_IK_25519_ChaChaPoly_BLAKE2s`.** Curve25519 + ChaCha20-Poly1305 + BLAKE2s. Standards-grade choice with production precedent (Tailscale's control protocol uses the same suite via its own implementation at `tailscale.com/control/noise`, not via `flynn/noise`; WireGuard transport uses a closely-related IKpsk2 construction). Full rationale and alternatives in [ADR 024](../decisions/024-noise-ik-mobile-e2e.md).
- **AEAD scope: whole inner frame, not just `payload`.** Outer routing envelope (`conn_id`, `frame`, optional `close_code`) stays plaintext for the relay; everything inside `frame` (post-handshake) is AEAD-sealed. Resolves architect-question #4: the v1-reserved `payload_encrypted` flag becomes structurally redundant in v2 (every transport frame is a `noise_msg`) and is therefore **removed**, not renamed.
- **Inner-frame discriminator (`type`):** three new values — `noise_init` (phone → binary, IK message 1, plaintext-at-outer-layer), `noise_resp` (binary → phone, IK message 2, plaintext-at-outer-layer), `noise_msg` (both, AEAD-sealed application traffic). Application message types (`send_message`, `message`, `ack`, etc.) are unchanged in shape but now live inside `noise_msg.data` after base64-decoded ChaChaPoly decryption.
- **Re-key: 1-hour timer + explicit `rekey_request` envelope.** Both sides start the timer at handshake completion; on fire, the phone (always the IK initiator) sends a fresh `noise_init`. The re-key `noise_init`'s early-data payload is **empty** (mirrors WireGuard/Tailscale — the handshake itself is the signal; no marker envelope needed). Atomic switchover: first frame either side sends after the new handshake uses the new keys; old CipherStates zeroised. Resolves architect-question #3 (no `rekey_marker` envelope was added; #435's `awaitingRekeyInit` substate is enough).
- **Static-key storage shapes specified.** Binary side: `~/.pyry/<daemon-name>/static_key.json`, mode `0600` enforced at startup with loud-failure on mismatch (mirrors `pyrycode-relay/internal/relay/tls.go:16-55`'s `0700` check); parent dir mode `0700`; `<daemon-name>` allowlist-canonicalised against path traversal (`lowercase alphanumerics + - + _`, no `..`, no `/`); read path opens with `O_NOFOLLOW` where supported. File format is JSON `{version, algorithm, private_key, public_key, created_at}` with raw 32-byte base64-encoded keys. Mobile side: Android Keystore alias `pyrycode.device_static.<server-id>`, hardware-backed where available; iOS Keychain equivalent with `kSecAttrAccessibleAfterFirstUnlock`. Ephemeral handshake keys process-memory only (zeroised by Noise's `CipherState` lifecycle).
- **Pairing QR payload grows one field.** `server_static_pubkey` (base64-encoded raw 32-byte X25519 public key) added alongside existing `{server, relay, token}`. Pairing UI, verb, and ergonomics unchanged. Resolves the "no v1-pair upgrade path" locked decision: a v1 QR (without `server_static_pubkey`) cannot complete a v2 handshake, which is why the pre-flight gate (`pyry pair list` empty before release flag flips, filed as #436) is load-bearing.
- **Pre-flight `pyry pair list` empty check (#436).** New CI-gated check on the release workflow. Non-empty `pyry pair list` at v2 release time means existing paired devices will fail on first v2 connect with `4426` and have no recovery path other than per-device revoke + re-pair. The pre-flight makes this loud at release time, not silent at first-user-connect.
- **Hard cutover, no version negotiation.** Endpoints move from `/v1/server`, `/v1/client` to `/v2/server`, `/v2/client`. `protocol_versions` array retained shape-compatibly in `hello` as `["v2"]`; v2 implementations do not negotiate down. The v1 `x-pyrycode-token` header is **removed** from the WS upgrade — the device-token now travels inside the AEAD-sealed `hello` early-data, so the relay never sees it in plaintext. The routing-envelope `token` field is correspondingly deprecated (relay MUST NOT set it; binary MUST ignore if present — defensive).
- **Token-validation gating spelled out.** Per-conn-id state machine (specified in #434) distinguishes a `handshakeComplete` substate (CipherStates exist, token not yet validated) from `open` (validated). The application handler chain in `internal/relay/handlers/` MUST NOT be reachable from `handshakeComplete`. Pre-validation `noise_msg` frames after handshake but before token-OK return an AEAD-sealed `auth.invalid_token` error envelope then a `4401` close.
- **AEAD associated-data: empty.** `EncryptWithAd(nil, ...)` / `DecryptWithAd(nil, ...)`. Outer routing envelope is NOT bound to the AEAD because (a) per-handshake key derivation isolates session-to-session, and (b) per-session nonce-counter discipline isolates frames within a session. Binding `conn_id` would force a re-key on every relay-side routing change with no security benefit. MUST-NOT-deviate clause in spec § Transport.
- **WS close code `4426`: handshake failure.** New in v2. Used when an AEAD channel doesn't yet exist (no shared key under which to send an `error` envelope), so the failure surfaces at WS close instead. Mirror-image case: phone presenting wrong `server_static_pubkey` and binary presenting wrong static key both close with `4426`. The latter is the relay-impersonation case — Noise_IK is designed to detect it.
- **Application-envelope size cap: 65519 bytes** (Noise's 65535-byte transport-message limit minus 16-byte AEAD tag). Supersedes v1's 1 MiB `message.too_long` cap. Attachments require envelope-level chunking, out of scope for v2.
- **Go library: `github.com/flynn/noise`** (resolves architect-question #1). Pinned in #431. Maintained; first-class `HandshakeIK` support via `noise.HandshakeIK` constant; early-data payloads supported via `WriteMessage(out, payload)` on every handshake message; ships and asserts against the Noise spec test vectors in CI; pure Go, no cgo, dependency-light. No viable Go alternative (`perlin-network/noise` is a P2P framework, not a Noise Protocol implementation despite the name).
- **Kotlin library: three options surfaced, mobile team picks** (resolves architect-question #2). Recommendation: **`rweather/noise-java`** (Java, JCE primitives, longest-running JVM option). Alternatives: JNI wrapper to `noise-c` (smallest APK, native-speed, NDK complexity); `sander/noise-kotlin` (pure Kotlin, KMP, **not recommended** — explicitly unaudited and ships no crypto). The mobile team commits when they pick up the mobile-side ticket cluster (filed by them, in `pyrycode/pyrycode-mobile`, not here).
- **Implementation children filed in this repo only** (per ticket's scope constraint): #431 (Go Noise library wiring + per-binary static-keypair generation/load with `0600` enforcement), #432 (QR-payload extension to include `server_static_pubkey`), #434 (AEAD wrapping of inner envelope on send/receive paths + per-conn-id state machine + handshake), #435 (re-key timer + `rekey_request` envelope handler + `awaitingRekeyInit` substate), #436 (pre-flight `pyry pair list`-empty check + release-flag gate). Five tickets total; all `size:xs` or `size:s`. **Zero tickets opened in `pyrycode/pyrycode-mobile` or `pyrycode/pyrycode-relay`** — those teams file their own once the spec lands.
- **Self-review security-audit pass recorded inline.** Spec § "Security review" lists every SHOULD FIX → FIXED finding from the architect's own adversarial pass per `agents/architect/security-review.md`. Notable findings folded into the spec body: empty-AD AEAD documented explicitly with MUST-NOT-deviate clause (round-2 code-review feedback addressed in `cc77c3b`); base64 alphabet pinned to `base64.StdEncoding`; per-frame 65535-byte decoded cap named explicitly; static-key fingerprint truncation standardised on BLAKE2s(pubkey)[:8] (64-bit, defeats 2³² preimage search); `O_NOFOLLOW` + parent-dir mode `0700` + `<daemon-name>` allowlist canonicalisation on the static-key read path. Round-3 code-review FAIL on `noise_init` re-key payload (line 212 had an undefined `rekey_marker` TBD) resolved in `7dcab1a` by committing Option A (empty early-data on re-key — WireGuard/Tailscale precedent).

## Patterns established

- **Architect spec PRs land directly against the protocol doc, not via a separate `specs/architecture/<N>-*.md` sibling.** For pure spec deliverables (no code), the protocol/architecture doc IS the spec; a sibling `specs/` file would just duplicate. The codebase note + ADR pair (this file + ADR 024) absorbs the discoverability function that `specs/architecture/` provides for code-implementing tickets. See `docs/specs/architecture/` for the contrast — those exist because the codebase has implementation work alongside the spec; #430 has none, so the spec is single-source-of-truth.
- **Hard-cutover protocol versioning is viable when `pyry pair list` is empty.** The pre-flight check (#436) is the lightweight enabler that lets v2 skip soft-fallback machinery entirely. The lesson generalises: a protocol-version cutover is "free" iff there is a deterministic on-disk indicator that no live wire is using the old version, AND that indicator is gateable on the release workflow. For v3 (whenever it ships), the equivalent gate would need a v2-aware tool ("`pyry session list --protocol=v2`" or similar).
- **Self-review security audit recorded in the spec body, not in a separate `security-review.md`.** The spec doc is small enough (~470 lines) and security-sensitive enough that a separate file would risk drift between "what the spec says" and "what the review checked." Inlining the findings as `SHOULD FIX → FIXED` markers under § "Security review" keeps the review pinned to the spec it reviewed.
- **The four architect-level questions all get answered in the spec, not in a sibling.** "Surface viable options for the mobile team to pick from" still belongs in the spec body — the spec is what the mobile team reads first. The architect's job is to remove ambiguity from the spec; offloading "library choice" to a separate doc means a developer reads the spec, sees "TBD — see appendix," and now has two docs to reconcile.

## Lessons learned

- **The `rekey_marker` TBD in the spec was a third-round code-review FAIL, caught only on PR #437 round 3.** First two review rounds passed; the round-3 reviewer noticed line 212 still said "empty or contains a rekey_marker envelope (TBD — see implementation tickets)" — an undefined envelope type, no child ticket scoped to resolve it, contradicting the parent AC#2 (spec must be implementable end-to-end, no TBD remaining). Resolution was a 1-line commit (`7dcab1a`) committing Option A (empty early-data, WireGuard precedent). Lesson: a spec's "no TBDs" AC needs a grep-for-TBD-or-architect's-call pass, not just a "looks done" read. The reviewer's mechanical pass caught what three rounds of substantive review missed.
- **The empty-AEAD-AD decision was inferable from the security-review entry but not stated in the spec body until round-2 code-review feedback.** First-round spec landed (`3df30c4`) with the choice present but only in § "Security review"; the design rationale (per-handshake key isolation, per-session nonce counter) wasn't in § "Transport" where an implementer reading the spec top-to-bottom would find it. Round-2 review fix (`cc77c3b`) added the explicit § "Transport" callout with a MUST-NOT-deviate clause. Lesson: a security-review entry is a *check*, not a *spec*. The spec body must state the choice and justify it where implementers will look; the security-review entry should reference the spec body, not be the primary record.
- **The Tailscale/flynn-noise attribution had to be corrected.** First-round draft (`3df30c4`) claimed Tailscale's production deployment of `Noise_IK_25519_ChaChaPoly_BLAKE2s` as parity evidence for `flynn/noise`. Tailscale ships its own Noise implementation (`tailscale.com/control/noise`), not `flynn/noise`. Round-2 fix (`cc77c3b`) reworded as "cipher-suite parity, not library-level parity." Lesson: precedent claims need to be precise about *what* is shared. The cipher suite is well-trodden; that's the load-bearing claim. The library is a separate evaluation and rests on its own published API + test-vector CI assertions, not on someone else's production deployment.
- **The four architect-level questions were framed as "research and document"; #1 (Go library) was actually a no-controversy single-option pick.** The architect spent token budget framing alternatives that had no viable competitor (`perlin-network/noise` is a P2P framework, not a Noise implementation despite the name). Lesson: when an architect-level question has one viable answer and the rest are non-options, the spec should say "no viable alternative" plainly, not pretend there's a trade-off to weigh. Saves reader time and signals confidence.
- **The `payload_encrypted` flag was dropped, not renamed.** Architect-question #4 framed the choice as "rename or document the naming drift." The actual answer turned out to be "neither — the flag is structurally redundant in v2 because every transport frame is a `noise_msg`, so removing it is cleaner than renaming." Lesson: when a v1 flag was a *placeholder* for a future feature and the future feature makes the flag redundant, deleting the flag is structurally better than renaming. v1 had no shipped install base so deletion costs nothing; renaming would have preserved a vestigial field forever.

## Out of scope (deferred)

- **Attachments.** v2 is the encryption layer only. First attachment release rides on top in v2's first feature release; will need envelope-level chunking to stay under the 65519-byte cap.
- **Voice memos.** Deferred at the attachment level.
- **Multi-device key sharing.** Each paired phone has its own Noise session + device-static keypair. No cross-device key sync.
- **Per-message-counter rotation.** Noise's 2⁶⁴ counter is not a practical limit; revisit if real-world abuse case appears (v3 territory).
- **Push notification payload encryption.** Out-of-band channel; APNs/FCM payloads remain plaintext.
- **v1↔v2 fallback / migration tooling.** Explicitly removed by locked decision #4. Hard cutover gated by #436 pre-flight check.
- **Permission scoping.** Mobile-originated messages still execute with the same authority as the desktop; tiered scopes are a v3 concern.
- **Static-key rotation verb (`pyry rotate-static-key`).** Deferred to v3. v2-era recovery from suspected static-key compromise is per-device revoke + manual `static_key.json` regeneration + manual re-pair.
- **Hardware-backed static-key storage on the binary side** (TPM, macOS Secure Enclave). v2 ships file-mode `0600` + no logging; hardware backing deferred to v3.
- **Filing tickets in `pyrycode-mobile` or `pyrycode-relay`.** Out of scope per ticket's filing-scope constraint. Mobile and relay teams file their own once this spec lands.
