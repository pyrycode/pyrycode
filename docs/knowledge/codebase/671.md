# Ticket #671 — Turn bridge cold-start tails the session JSONL from offset 0 so the live reply streams (#421 final blocker)

The live mobile turn bridge dropped the streamed assistant reply: after #670's
workspace pre-trust fix, pyrycode-mobile#421's rung-3 e2e passed connect → list →
create → **send** (the user turn commits and renders) but stalled on the reply.
The daemon logged `turnbridge: dropping unrepresentable event kind=9` and nothing
streamed. The ticket handed a two-candidate fork — a cross-repo `tuidriver.AssistantText`
extractor gap, or an in-repo tail-offset race — and the architect **resolved it to
the in-repo offset (candidate 2)**: the dropped `kind=9` lines were *not* the reply
(they were the ordinary non-representable lines a turn also writes — the echoed
`type:"user"` prompt, claude init metadata, the #668 inert transcript-growth line);
the reply itself was **never read** because the tail started past it. The fix makes
`resolveLatestSessionJSONL` return `startOffset = 0` for a cold-start session so the
in-flight reply is no longer skipped. ~10 lines of logic, one production file +
a debug diagnostic, in-repo only — **no tui-driver change, no `go.mod` bump**.

## Implementation

All behavior change is in `cmd/pyry/interactive_turn_stream_v2.go`; a read-only debug
diagnostic was added in `internal/turnbridge/producer.go`. `NewSessionSubscriber` and
`mapEntry` are untouched.

- **`resolveLatestSessionJSONL` is now stateful across calls** (`interactive_turn_stream_v2.go:140`).
  Two closure-local flags: `resolvedOnce` (a session file has been returned at least
  once) and `sawEmpty` (an earlier call found no file). The first file returned *after*
  one or more not-found results — and before any file has been returned — is a
  **cold-start** file → `startOffset = 0` (`:191-195`). A file present at the first look
  (warm `--continue` resume) or any file after one has already been returned (a `/clear`
  rotation) → `startOffset = size`, unchanged. Signature is unchanged; only the offset
  value differs.
- **Both error arms count as a cold-start signal.** The empty-dir path (`no session
  jsonl found`, `:186-188`) *and* the unreadable/not-yet-created-dir path (`read claude
  sessions dir`, `:150-151`) set `sawEmpty` when no file has been returned yet — claude
  creates the project dir lazily on first input, so the live race can surface through
  either. (See Lessons.)
- **Cold start ⇒ offset 0 is the right semantics, not just a workaround.** A fresh relay
  session (mobile#421: fresh `$HOME`; relay claude runs under `--continue`, which defers
  JSONL creation until first input) has no transcript at first subscribe → resolve#1
  not-found → the subscriber retries every `subscribeRetryDelay` (500 ms). The phone's
  prompt then lands and claude writes the user turn **+** the reply in one go, so
  resolve#2 finds a brand-new file whose **whole content IS the current turn**. Returning
  `size` there tails from EOF, *past* the in-flight reply — the live drop. Offset 0 tails
  from the file's start; there is no historical transcript to skip because a resumed
  transcript would already exist on disk (→ warm path).
- **Subscribe-success diagnostic** (`producer.go:218`) — `log.Debug("turnbridge:
  subscribed to session jsonl", "path", path, "offset", off, "cold_start", off == 0)`.
  Logs only the path (a UUID filename under the trusted dir), the offset, and a derived
  bool — **never JSONL bytes** (substrate seal). Lets the operator confirm during the live
  AC-3 run that the resolve-retry path fired and the subscribe landed at offset 0.

## Patterns established

- **The two-candidate fork was resolved by evidence to in-repo (candidate 2), and the
  fixture location was re-targeted within the fork.** AC-1's literal ask was a RED-before
  fixture in `internal/turnbridge/mapper_test.go` driving `mapEntry`. The diagnosis
  showed `mapEntry` already maps the reply shape, so a mapper fixture would be
  **vacuous-green** — the exact failure the ticket's own Technical Notes warned against.
  The gate therefore lives in `cmd/pyry/interactive_turn_stream_v2_test.go` against the
  resolver offset. The ticket explicitly authorized this ("whichever the captured shape
  shows" → candidate 2 = the in-repo tail-offset). Three independent confirmations ruled
  out the extractor: (1) real claude 2.1.158's on-disk assistant line is exactly the
  `{type:text,text:…}` shape `AssistantText`/`mapEntry` already map; (2) the fakeclaude
  structured e2e `TestTwoPhoneStructured_InteractiveReceivesStream` proves the mapper
  path and stays green; (3) the same e2e's setup comment self-documents the race and
  pre-creates the JSONL to dodge it. (`[[po-fix-location-fork-decided-by-diagnosis]]`.)
- **The discrimination lives in the resolver closure, not the subscriber — for
  testability.** The subscriber (`NewSessionSubscriber`) conceptually "owns the timeline,"
  but its only seam is `sess.Events` on a concrete `*tuidriver.Session`, not fakeable
  without a new interface. The resolver already has a pure `t.TempDir()` test suite
  (`TestResolveLatestSessionJSONL_*`), so putting the cold-start flag there buys a
  deterministic, seam-free regression test for a small purity-doc update — Simplicity
  First.
- **Single-goroutine invariant, asserted in the doc comment.** `resolvedOnce`/`sawEmpty`
  are read and written only inside the returned closure, which `NewSessionSubscriber`
  invokes from the single `Producer.Run` goroutine (Run → subscribe → resolve). No mutex —
  the same invariant the `OnEvent`/`flushDelta` closures rely on. The closure comment
  states "Do NOT call this resolver from multiple goroutines."
- **Offset 0 is structurally confined to brand-new files (the `security-sensitive`
  crux).** A `--continue` resume transcript present at the first resolve takes the warm
  path → `offset = size`, so a prior session's content is **never** replayed to the
  internet-exposed phone. `TestResolveLatestSessionJSONL_WarmStartTailsFromSize` is the
  **mandatory** deterministic guard (not optional). The emitter additionally scopes events
  to `sup.CurrentConversation`, stamped by `send` before the cold-start tail reads, so no
  cross-conversation misattribution; non-assistant preamble lines drop in `mapEntry`.

## Tests

- **`TestResolveLatestSessionJSONL_ColdStartTailsFromZero`** (`:204`, the CI regression
  gate) — empty `t.TempDir()` → resolve#1 errors; write a non-zero `<uuid>.jsonl` →
  resolve#2 returns that path and **offset 0**; then a newer rotation file → returns its
  **size** (post-cold-start is no longer a cold start). RED on the pre-fix code (returned
  `size`), GREEN after.
- **`TestResolveLatestSessionJSONL_WarmStartTailsFromSize`** (`:251`, mandatory security
  guard) — a transcript present *before* the resolver is constructed → first resolve
  returns its **size**, proving a `--continue` resume is not replayed from 0.
- The six existing `TestResolveLatestSessionJSONL_*` tests (NewestWins, ReEvaluatesPerCall,
  TieBreak, IgnoresNonSessionEntries, EmptyDirErrors, UnreadableDirErrors) stay green;
  `ReEvaluatesPerCall` is the key compat check (warm first resolve → size; rotation → size).
- **AC-3 (operator-verified, out of CI scope).** The live two-phone rung-3 e2e
  `InteractiveStreamE2ETest` (pyrycode-mobile#421) — the `"ping"` reply renders in the
  thread — runs on the live mobile stack the agent cannot drive. The CI gate for this
  ticket is the resolver regression test; the live render is the operator's confirmation.

## Lessons learned

- **The drop log was a red herring.** `producer.go:132` (`kind=9`) logged real drops, but
  they were the ordinary non-representable lines (echoed user prompt, init metadata, the
  #668 transcript-growth line), **not** the reply. Diagnosing from the drop log alone
  would have pointed at the mapper/extractor — the wrong fork. The reply line was never
  read; the tail started past it.
- **The cold-start signal can arrive as either error path.** An empty project dir (`no
  session jsonl found`) *or* a not-yet-created dir (`read claude sessions dir`), since
  claude creates the project dir lazily. The fix treats both as "no file yet," so it is
  robust to which one the live race hits. (Lifted from the PR body's Lessons.)
- **Why CI was green while live was red.** The deterministic fakeclaude harness never hits
  the race because `relay_two_phone_structured_test.go` *pre-creates* `<uuid>.jsonl`
  before the daemon starts (the #642/#661 cold-start workaround) — so its first resolve
  succeeds immediately at a tiny offset. Real claude gets no pre-create and exposes the
  race. This fix moves the protection from the harness into production; the harness still
  pre-creates for determinism.
- **Incidental realclaude golden churn (not feature work).** Code-review's `make
  e2e-realclaude` run regenerated `internal/e2e/realclaude/testdata/permission_protocol_v2.1.158.json`
  (run-specific temp-dir paths, `session_id`, `uuid`) and auto-committed it (`bda0c72`).
  Same noise documented in [603.md](603.md) / [642.md](642.md) / [661.md](661.md); not
  part of #671's diff-of-interest.

## Out of scope / forward notes

- **Post-`/clear` first-reply race (open question).** A `/clear` rotation re-subscribes
  against the same now-warm closure → the new file is tailed from EOF and could skip the
  first post-`/clear` reply by the same race. **Unobserved** (mobile#421's e2e has no
  `/clear`) — out of scope per Evidence-Based Fix Selection. If it surfaces, the escalation
  is per-attempt (subscriber-owned) cold-start tracking; file a follow-up rather than
  pre-building.
- **`--continue` file-fork vs append (open question).** If a future capture shows claude
  writing the reply to a *different* `<uuid>.jsonl` than the one tailed (a resume-fork the
  most-recently-modified pick missed), that is still an in-repo resolver/file-selection
  fix, still S — not the extractor.
- **Guarded fallback (did not fire).** Had the operator's AC-3 run shown the reply line
  *itself* dropping at `producer.go:132` (`mapEntry` returns `ok=false` for the reply's
  own `type:"assistant"` text line), the true cause would have been the extractor →
  `needs-rework:po` for a coordinated tui-driver change + tagged release + `go.mod` bump.
  The three confirmations made this the strongly-expected non-outcome.

## Review notes

- **Code-review PASS** (PR [#676](https://github.com/pyrycode/pyrycode/pull/676)).
  Real-claude e2e column green (`ok …/internal/e2e/realclaude 163.294s`, no flake). One
  **non-blocking NIT**: the diagnostic logs `cold_start` as the proxy `off == 0`, so a
  degenerate size-0 *warm* file (claude created the JSONL but had not yet written the turn)
  would log `cold_start=true` — diagnostic-only and a near-impossible state, no action.
