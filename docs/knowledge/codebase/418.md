# Ticket #418 — Fixture-walk regression test for permission-protocol-spike null findings

Pure JSON parse + assert test that pins the four null findings of the [#383](383.md) `--permission-prompt-tool stdio` spike as a regression contract against the committed `internal/e2e/realclaude/testdata/permission_protocol_v*_*.json` fixtures. Closes the regression-alarm gap left by #383's AC#4: the spike test pinned the on-disk artefact, not its shape, so a future `claude` release that started emitting permission events on stdio (or began enforcing `--allowed-tools` under that argv) would regenerate the fixtures silently and flip the mobile-relay design implication documented in [`features/permission-protocol-spike.md`](../features/permission-protocol-spike.md) without anyone noticing. This test makes the flip loud.

## Implementation

- **`internal/e2e/realclaude/permission_protocol_regression_test.go` — new file**, build tag `//go:build e2e_realclaude`, package `realclaude`, single exported test `TestRealClaude_PermissionProtocol_RegressionFixtures`. ~183 LoC. Stdlib only (`encoding/json`, `os`, `path/filepath`, `regexp`, `slices`, `testing`); no edits to `fixtures.go`, no production-source changes.
- **Glob-driven, not hardcoded.** Top-level constant `fixtureGlob = "testdata/permission_protocol_v*_*.json"` is the source of truth for which modes are covered. `filepath.Glob` failing or returning zero matches is a `t.Fatalf` — a deleted fixture set must be a loud failure, not a silent green. A future spike rerun that adds a new mode (e.g. `--permission-mode foo`) drops a new fixture file in `testdata/` and this test picks it up automatically.
- **Per-fixture subtest naming** via `t.Run(filepath.Base(path), …)` so the `go test` failure header names the offending fixture directly (AC#7). `t.Parallel()` on the inner subtests — independent reads, cheap I/O, no shared state.
- **Narrow parse strategy.** Three minimal structs decode only the fields each assertion needs, mirroring `allowed_tools_enforcement_test.go:75-94`'s style:
  - Top-level: `StdoutEvents []json.RawMessage`, `ExitCode int`, `ContextDeadlineTripped bool`.
  - Event header (used in a single loop over `StdoutEvents`): `Type string`, `Subtype string` — dispatches by event kind to capture `system/init` and `result/success` raw bytes and to flag any forbidden-typed event.
  - `system/init` body: `Tools []string`, `PermissionMode string`.
  - `result/success` body: `PermissionDenials []json.RawMessage`.
- **Single pass over `stdout_events`.** The loop simultaneously (a) flags every event whose top-level `type` is in `forbiddenEventTypes = {"control_request", "permission_request", "tool_permission_request"}` — capturing ALL hits via `t.Errorf` so a release that introduces multiple gate-event types lists them all in one run, not just the first; (b) captures the first `system/init` event's raw bytes; (c) captures the first `result/success` event's raw bytes. Per-line `json.Unmarshal` failure is a per-event `t.Errorf` (not `t.Fatalf`) so one bad line cannot mask the other assertions — same shape as `allowed_tools_enforcement_test.go:60-67`.
- **Filename → expected mode.** `expectedPermissionModeFromFilename(name) (string, error)` uses `fixtureNameRE = regexp.MustCompile(\`^permission_protocol_v[^_]+_([A-Za-z]+)\.json$\`)` to pull the mode token, then applies the **`auto` → `default` synonym** (spike finding #3) at the helper so the assertion site compares apples-to-apples against `init.permissionMode`. Non-matching filename returns a `*fixtureNameError` and the subtest `t.Fatalf`s naming the filename and regex — covers "future spike rerun added a fixture with an unexpected filename shape".
- **AC-to-assertion mapping (inside each subtest):**

  | AC | Assertion | Failure message |
  |----|-----------|-----------------|
  | #2 | No `forbiddenEventTypes` member in any `stdout_events[i].type` | "forbidden event type observed: stdout_events[i].type = q (spike finding #1 flipped: permission gate fired on stdio)" |
  | #3 | `init.tools` contains `"Bash"` via `slices.Contains` | "init.tools missing q (spike finding #4 flipped: --allowed-tools may now gate the registry)" |
  | #4 | `len(result.permission_denials) == 0` | "result.permission_denials non-empty: len=N (spike finding #1/#2 flipped: a gate fired and denied something)" |
  | #5 | `init.permissionMode == expectedPermissionModeFromFilename(name)` | "init.permissionMode = q, want q (spike finding #3 flipped: mode echo changed; for filename token 'auto' the expected echo is 'default')" |
  | #6 | `exit_code == 0`, `context_deadline_tripped == false`, `len(stdout_events) >= 7` | "exit_code = N, want 0 (structural sanity check)" / "context_deadline_tripped = true, want false …" / "len(stdout_events) = N, want >= 7 …" |
  | #7 | Every failure message starts with `<fixture-filename>:` and names which finding flipped | (above) |

- **`t.Errorf` over `t.Fatalf` for the per-finding assertions** is the spec's recommended choice and the one taken: a single subtest run reports every flipped finding simultaneously, which is the load-bearing diagnostic shape for a release where `claude` changes multiple permission-protocol behaviours at once. `t.Fatalf` is reserved for "cannot proceed at all" — unreadable file, malformed top-level JSON, unparseable filename, missing inner JSON header.
- **Cost / runtime.** Zero real-API cost; pure on-disk parse + assert. Six fixtures × a handful of `json.Unmarshal` calls ≈ a few ms aggregate under `make e2e-realclaude`. Excluded from `make check` by the build tag; no CI run-time impact on the default path.

## Patterns established

- **Regression contract over captured artefact.** The #383 spike test pins what is on disk (the file exists and is non-trivial); this test pins what *shape* the on-disk content has. Both are useful — the spike's purpose is the artefact itself, this ticket's purpose is the assertion contract over the artefact. When the artefact's "null findings" are load-bearing for a downstream design (here: the mobile-relay deny-by-default posture), the regression contract is what makes a future flip noisy. Pair shape with cite: the failure message names "spike finding #N flipped" so the spike-runner is pointed directly at the writeup section to revisit, not just told "your fixture changed."
- **Filename glob as source of truth for matrix coverage.** ACs explicitly forbade hardcoding the six mode names, and the test honours that with `filepath.Glob` over a single literal pattern. The win is composability with the manual-sweep discipline #383 established: a spike-runner adding a new mode (`auto2` if claude ever ships one) drops a new fixture and this test picks it up with no code change. Zero-match is a hard fail by design — silent green on an empty matrix would defeat the entire regression contract.
- **`t.Errorf` for per-finding assertions, `t.Fatalf` only for structural blockers.** When the goal is "report every regression in one run" (multi-flip release), `t.Errorf` is the correct primitive. Reserve `t.Fatalf` for cases where subsequent assertions are not meaningful — unparseable file, missing structural anchors. This shape lets a `claude` release that flipped, say, both finding #1 (gate fires) and finding #3 (auto/default split) surface both in one CI log instead of forcing two passes.
- **Narrow `json.RawMessage` parse over full struct.** Fixtures contain many fields this test does not care about; decoding the whole shape would couple the regression test to spike-record format churn (which is allowed to evolve under `permission_protocol_spike_test.go`). Decoding only the four shapes the assertions need keeps the coupling surface small — a future record-field addition or rename in the spike test does not break this regression contract.

## Lessons learned

- **The spike's "test PASSES on any non-structural outcome" shape leaves a regression-alarm gap that needs a separate test, not an extension to the spike.** Tempting alternative: bolt assertions onto `TestRealClaude_PermissionProtocol_Spike` itself so the spike fails when the protocol changes. Wrong call — the spike's value is the captured artefact regardless of behaviour, and adding assertions would force the spike to fail on a real protocol change instead of recording it. Splitting into (a) capture-and-pass spike + (b) parse-and-assert regression keeps both jobs clean. Spec called this out; #418 is the realisation.
- **`forbiddenHit` flag is dead code post-loop.** Wrote `forbiddenHit bool` to track whether any forbidden-typed event was seen, then realised `t.Errorf` inside the loop already reports each hit individually. Kept the `_ = forbiddenHit` line with a "kept for clarity" comment rather than removing it — small enough to not bother, signals to a future reader that the loop's reporting is the assertion, not a post-loop check. A reviewer trimming this is fine.
- **`auto → default` belongs in the filename-parser, not at the assertion site.** First sketch had the synonym applied via a conditional at the assertion (`if expectedMode == "auto" && got == "default" { ok }`). Moving it into `expectedPermissionModeFromFilename` is structurally cleaner — the assertion just compares two strings, and the helper's name + signature documents the synonym for any future maintainer who wonders why the `auto` fixture is expected to echo `default`.
- **`*fixtureNameError` is a tiny custom error type for one call site.** Could have used `fmt.Errorf` and lost ~10 LoC. Kept the typed error because the only call site that consumes it is a `t.Fatalf` that wants a precise message; the typed error makes the test's intent (the regex contract on filenames) explicit. If a third call site materialises and wants to discriminate the error, the type is already there.

## Out of scope (deferred)

- **Re-running the real-API spike.** Covered by `TestRealClaude_PermissionProtocol_Spike` (#383); this ticket is the assertion contract, not the capture.
- **Asserting on `tool_result` payload bodies.** Those reflect the spike-runner's machine-local POSIX environment per the codebase doc; AC explicitly excludes them.
- **The assertion-based permission-protocol test referenced by #383 AC#4.** That follow-up is filed only if a real permission event fires; none has, so it is not in scope here. If a future spike rerun captures a real event (registered MCP tool as the prompt-tool, or a side-channel capture), that's when the assertion-based test gets scoped.
- **Updating `features/permission-protocol-spike.md`'s findings.** The findings doc is correct as-is; this test pins what it already documents. The doc gains a "Regression contract" section pointing back here, but the findings themselves are unchanged.
