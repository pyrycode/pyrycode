# Ticket #423 — `e2e/realclaude`: large tool output > 64 KiB scanner buffer regression guard

Tenth consumer of the `internal/e2e/realclaude` fixture trio (#372 + #373 + #409). One new test — `TestRealClaude_LargeToolOutput_ExceedsDefaultScannerBuffer` — drives a real claude session through a single Bash invocation producing ~80 KiB of stdout in one `tool_result` content block and pins four contracts together: trailer reports `subtype="success"` + `stop_reason="end_turn"`, the on-disk JSONL `tool_result` content block exceeds 70 KiB (headroom under the ~80 KiB target), pyry's stdout-forwarded `tool_result` content block has **byte-equal length** to the on-disk twin (the regression sensor: any non-zero delta means a scanner truncated the forwarding path), and stderr does NOT contain `bufio.Scanner: token too long`. Orthogonal to #421's long-session test, which fires when many short lines accumulate; this one fires when **a single line on pyry's stream-json stdout exceeds 64 KiB**.

See [features/e2e-realclaude.md](../features/e2e-realclaude.md) for the suite contract.

## Implementation

- **`internal/e2e/realclaude/large_tool_output_test.go`** (~252 LoC including the package-level comment block, two prompt constants, and inline rationale), `//go:build e2e_realclaude`, `package realclaude`. Single top-level `TestRealClaude_LargeToolOutput_ExceedsDefaultScannerBuffer` + four file-local helpers (`scanLargeStdoutLines`, `findResultTrailer`, `findBashToolResultBlock`, `findToolResultContentByID`) + two file-scope prompt constants. **Zero edits** to `fixtures.go`, `tool_loop_test.go`, or any peer test file — purely additive.
- **Flow.** `WithWorktreeAuthenticated(t)` → `RunPyryAgentRun(t, RunOpts{AllowedTools:["Bash"], MaxTurns:2, Effort:"low", Model:"claude-haiku-4-5", Prompt:largeToolOutputUserPrompt, SystemPrompt:largeToolOutputSystemPrompt})` → `scanLargeStdoutLines(result.Stdout)` (1 MiB-capped Scanner) → trailer assertions via `findResultTrailer` → `ReadJSONL` walk via `findBashToolResultBlock` to capture (tool_use_id, jsonl content bytes, block index) → matching `tool_result` extraction from pyry's stdout via `findToolResultContentByID` → byte-length equality assertion → stderr substring tripwire.
- **Prompt design — deterministic, not random.** Two file-scope constants: `largeToolOutputSystemPrompt` ("You are an e2e regression-guard test. … Do not modify or paraphrase the command — run it exactly as given.") and `largeToolOutputUserPrompt` asking the model to invoke Bash exactly once with `printf '%80000s' '' | tr ' ' 'A'`. The `printf`/`tr` pipeline emits exactly 80,000 literal `A` characters on a single line (no trailing newline). At ~80 KiB it clears the 64 KiB stdlib cap by ~16 KiB and stays comfortably under the 1 MiB extended cap and the 16 MiB on-disk reader cap. **Deterministic preferred over `/dev/urandom`** per the AC so any future fixture-snapshot work doesn't churn. The `"exactly as given"` wording is load-bearing — paraphrasing risks fewer bytes and a missed 70 KiB threshold.
- **`MaxTurns: 2`** is the structural minimum for the loop `assistant tool_use → user tool_result → assistant text → end_turn`. Bumping to 3 is the documented recourse if a future haiku revision collapses turns or refuses the literal command; do NOT weaken the byte-length thresholds or the exact-match assertion.
- **`scanLargeStdoutLines(stdout []byte) ([]json.RawMessage, error)`** (~13 LoC, file-local). Wraps `bytes.NewReader(stdout)` in a `bufio.Scanner` and applies `scanner.Buffer(make([]byte, 0, 64*1024), 1024*1024)` — the canonical extension pattern from `permission_protocol_spike_test.go:133`. Each returned line is a fresh `make([]byte, len(scanner.Bytes())) + copy` so callers can retain it past subsequent reads (Scanner reuses its internal buffer between `Scan` calls). `scanner.Err()` is propagated so a **future buffer-exhaustion regression** (a line above 1 MiB) fails loudly rather than silently truncating.
- **Why a local scanner — NOT a call to `parseResultTrailer`.** `parseResultTrailer` (`tool_loop_test.go:211`) uses the stdlib's 64 KiB default. With an 80 KiB `user`/`tool_result` line on stdout **before** the trailer line, the default scanner returns `Scan() == false` on the long line, exits the loop, and the function returns `"no type:result line in stdout"` — a false negative that would mask the very regression this test exists to catch. The local `findResultTrailer` walks the pre-scanned 1 MiB-capped lines instead. **The shared `parseResultTrailer` helper is NOT edited** — keeping the regression scope to one file, per the spec.
- **`findBashToolResultBlock(events []JSONLEntry) (toolUseID, content, blockIdx)`** (~30 LoC, file-local). Single pass over JSONL events: on each `assistant` entry, decode content blocks via the shared `parseContentBlocks`, capture the first `{Type:"tool_use", Name:"Bash", ID:!=""}`. On subsequent `user` entries, find the first `{Type:"tool_result", ToolUseID==bashID}` and return its `Content` (`json.RawMessage`) plus block index. Returns `(toolUseID, nil, -1)` if `tool_use` seen but no matching `tool_result`; `("", nil, -1)` if no Bash `tool_use` at all. Per-line parse errors skipped silently (matches `tool_loop_test.go:75-80`).
- **`findToolResultContentByID(lines []json.RawMessage, toolUseID string) json.RawMessage`** (~12 LoC, file-local). Walks pre-scanned stdout lines, decodes each via `parseContentBlocks`, returns the first `tool_result` block whose `tool_use_id` matches. `nil` if no match. Per-line parse errors are silently skipped — user envelopes with a plain-string `content` (not an array) fail `parseContentBlocks`'s array-decode and are correctly ignored.
- **Exact byte-length match, not a tolerance band.** The emitter writes `ev.Raw + '\n'` verbatim (`internal/agentrun/streamjson/emitter.go:149-150`). The on-disk JSONL line and the pyry-stdout twin are byte-equal modulo the trailing newline. The content-block bytes (`json.RawMessage`) extracted from either side must be byte-identical because the content array passes through the emitter unmodified. Comparing `len(stdoutContent) == len(jsonlContent)` is the strongest possible assertion; **do NOT soften to a tolerance band** — any non-zero delta means a scanner / truncator was wired into the forwarding path, and that IS the regression being guarded against. The failure message names this directly ("this is the regression sensor for #423; a scanner in pyry's stdout forwarding path has truncated the `tool_result` content block").
- **Invariant to `Content` shape.** Claude may emit `tool_result.content` as either a bare JSON string (common for raw Bash stdout) or a nested array `[{type:"text",text:"..."}]`. Both shapes are byte-equal through `Emit`, so comparing raw `json.RawMessage` lengths between disk and stdout is invariant — the test does NOT decode `Content` further. The 70 KiB threshold is on the raw byte length including JSON framing (quotes for the string shape, the wrapping array for the nested shape) and absorbs both. The AC's 70 KiB floor is well below the ~80 KiB target so framing overhead doesn't push the threshold near the wire.
- **Forward-defensive stderr tripwire.** `!bytes.Contains(result.Stderr, []byte("bufio.Scanner: token too long"))`. Pins the literal stdlib error text verbatim — paraphrasing ("scanner overflow", "buffer too small") would silently disable the trip. Mirrors `long_session_test.go:135-139`. Today no production `bufio.Scanner` is on the relevant path, so the tripwire passes trivially; its job is to fire the day someone wires a stdlib Scanner into pyry's stdout/stderr forwarding without bumping the buffer.
- **Failure-message discipline.** Each `t.Fatalf` names a focused diagnostic (JSONL path, block index, observed byte length, the stdlib error text where relevant) and either a remediation hint or the regression-class flag. Examples:
  - `"no Bash tool_use observed in JSONL assistant entries (model may have refused to run the literal command; sharpen the system-prompt's \"exactly as given\" wording or bump MaxTurns to 3)"` — distinguishes "test setup wrong" from "production broken".
  - `"JSONL tool_result content byte length = %d, want > 70000 (target ~80 KiB; model may have paraphrased the printf/tr pipeline — inspect the JSONL for the actual command and stdout it ran)"` — names the AC threshold and points the maintainer at the JSONL.
  - `"tool_result content length mismatch: stdout=%d bytes, jsonl=%d bytes — this is the regression sensor for #423; a scanner in pyry's stdout forwarding path has truncated the tool_result content block (the emitter is contracted to re-emit ev.Raw verbatim, see internal/agentrun/streamjson/emitter.go:149-150)"` — names the regression class AND the production invariant that's been violated.
- **Cost.** One real haiku call per run, ~$0.02 (large `tool_result` inflates token count above peer tests but is bounded). `t.Parallel()` is NOT called — matches the existing realclaude convention.

## Reuses (no re-implementation)

- `realclaude.WithWorktreeAuthenticated` (#409) — opt-in real-API fixture, **tenth consumer**. Architect chose this over plain `WithWorktree` because the assertions depend on claude actually invoking Bash with the literal command; a synthetic-auth envelope would skip the `tool_use` event entirely and the "no Bash tool_use observed" assertion would fire on a non-regression.
- `realclaude.RunPyryAgentRun` + `realclaude.RunOpts` + `realclaude.RunResult` (#373) — synchronous subprocess invoker; `result.Stdout` and `result.Stderr` are full captures that the local scanner walks.
- `realclaude.ReadJSONL` + `realclaude.JSONLEntry` alias (#372) — on-disk JSONL walk for the `tool_use` / `tool_result` discovery.
- `realclaude.parseContentBlocks` + `contentBlock` (#376, extended #382) — content-block decoder; **sixth consumer**. The struct's existing fields (`Type`, `Name`, `ID`, `ToolUseID`, `Content`) cover exactly what `findBashToolResultBlock` and `findToolResultContentByID` need — `Content json.RawMessage` (extended in #382) is precisely the field that makes the byte-length equality assertion correct. **No struct extension required.**
- `realclaude.resultTrailer` (#376) — trailer struct (the local `findResultTrailer` decodes into it directly; the `parseResultTrailer` function itself is bypassed for the 64 KiB-cap reason above).
- `realclaude.jsonlPathFor` (#364) — diagnostic helper for resolved JSONL path strings; **tenth consumer**.
- `realclaude.truncate` (#381) — 1 KiB-cap byte-slice formatter for failure messages; **sixth consumer**.

## Patterns established

- **A local 1 MiB-capped `bufio.Scanner` is the right tool when a test asserts on stdout lines whose size exceeds the stdlib default — do NOT widen the shared helper.** The shared `parseResultTrailer` lives in `tool_loop_test.go:211` and uses the stdlib's 64 KiB default; widening it package-wide would couple every test to a shape only this one needs, and would silently mask the day a future regression downgrades the cap (because the package-wide scanner would still parse, just slower). Defining a file-local `scanLargeStdoutLines` instead keeps the regression scope to one file and pins the canonical extension pattern (`make([]byte, 0, 64*1024), 1024*1024`) at the site that actually depends on it. Generalises to any large-input regression sensor that needs a different parser config than the shared helpers.
- **The contract under test names the shape of the assertion, not the value.** Pyry's emitter contract is "re-emit `ev.Raw` verbatim" (`internal/agentrun/streamjson/emitter.go:149-150`); the on-disk JSONL line and the pyry-stdout twin are byte-identical modulo the trailing newline. The strongest assertion that pins this contract is **exact byte-length equality**, not a tolerance band ("within 1%" or "within 10 bytes"). Tolerance bands have to be justified against an absent transformation step; here there is none, so the tolerance is zero. **Document the absence of transformation in the comment above the assertion**, naming the emitter line so a future reader can verify the contract from the source — and so a future PR that introduces a transformation step has to update both this assertion and the named emitter line together. Generalises to any byte-identity assertion that rides on a "verbatim re-emit" contract.
- **Invariance to inner-structure variation is a feature of comparing `json.RawMessage` byte lengths, not a bug.** Claude's `tool_result.content` may be a bare JSON string OR a nested array `[{type:"text",text:"..."}]`. Decoding the content (`json.Unmarshal` past `RawMessage`) into a string would force the assertion to special-case each shape and would re-introduce a transformation step on the test side. Comparing raw `RawMessage` byte lengths from each side instead is shape-invariant: whichever shape claude emits, both sides carry the same bytes through `Emit`. Generalises to any test that asserts on a structured payload whose inner shape is the producer's choice but whose pass-through invariant is the contract under test.
- **Failure-message hints that distinguish "test setup wrong" from "production broken" reduce the maintenance cost of stochastic-input tests.** A real-`claude` test that asks the model to run a literal command can fail in three ways: production regressed (the sensor's purpose), the model paraphrased / refused the prompt (test setup degraded), or the suite plumbing broke (`ExitCode`, `SessionID`). Each `t.Fatalf` here names which class it suspects ("model may have refused to run the literal command; sharpen the system-prompt's \"exactly as given\" wording or bump MaxTurns to 3" vs "this is the regression sensor for #423; a scanner has truncated the forwarding path") so the next maintainer can route the response correctly at the point of failure. Generalises to any stochastic-input test where failure-mode ambiguity costs more in triage time than the assertion saves in coverage.

## Lessons learned

- **`parseResultTrailer` is unsafe to call from a test whose stream contains a >64 KiB line earlier.** The bug class is silent: `bufio.Scanner` returns `Scan() == false` (no error) when a single token exceeds its buffer; the function exits its loop, returns "no type:result line in stdout", and the test reports a misleading failure that points at the model rather than at the truncated scan. First instinct on writing this test was "reuse `parseResultTrailer`"; the spec called this out explicitly because the failure mode is silent. The local `findResultTrailer` over pre-scanned 1 MiB-capped lines is the safe shape. Worth reading `parseResultTrailer`'s buffer setup once before writing any test whose stdout will carry lines past 64 KiB.
- **`Content json.RawMessage` was the load-bearing struct field — pre-#382 the test would have needed a struct extension.** The `contentBlock` struct (originated #376) initially carried `Text string` but not `Content`. #382 added `Content json.RawMessage` for the `tool_result` body (string OR array of nested blocks per the stream-json contract); that field is precisely what makes #423's byte-length comparison work without re-implementing the decoder. The lesson is forward-defensive: when extending a shared struct for one test's needs, name the broader class the new field covers (here, the "structured-vs-string body" variance in `tool_result`) so future tests can reuse it without further extensions. Generalises to any incremental extension of a shared test struct.
- **`scanner.Err()` propagation matters, even when the test doesn't expect to exceed the cap.** If a future regression bumps the tool output past 1 MiB (or if the model glues two tool_results onto one stream-json line), the local scanner's `Err()` will surface `bufio.Scanner: token too long`. Returning `(nil, err)` from `scanLargeStdoutLines` and `t.Fatalf`-ing on the caller side fails the test loudly. Silently discarding the error would re-introduce the exact failure mode the test is meant to catch, just at a different threshold. Generalises to any defensive Scanner setup: always propagate `scanner.Err()`.

## Out of scope (deferred)

- **Streaming partial messages > 1 MiB.** Different concern — partial-message threshold; out of #423.
- **JSONL events > 1 MiB.** Separate failure mode beyond Bash output; the on-disk reader has a 16 MiB cap (`internal/agentrun/jsonl/reader.go:27-30`), so the read path is structurally safe at this scale.
- **Base64-encoding correctness in `tool_result`.** Claude's responsibility, not pyry's; out of #423.
- **Extending buffer caps on the other Scanner sites listed in the issue context** (`mcp_smoke_test.go:274`, `allowed_tools_enforcement_test.go:162`, `fixtures.go:281`, `tool_loop_test.go:212`). A separate fix ticket if warranted; this ticket is regression coverage for the one site that already extends (`permission_protocol_spike_test.go:133`).
- **Promoting `scanLargeStdoutLines` / `findResultTrailer` / `findBashToolResultBlock` / `findToolResultContentByID` to `fixtures.go`.** Single-file consumer in #423; the helpers stay file-local until a second test needs the same shape, matching `resilience_test.go`'s precedent for raw-exec helpers and #422's `spawnPyryAgentRun`/`processesInProcessGroup`.
