# Ticket #533 — `self-check-daily` posts Discord alert on workflow failure

The `agent-run self-check (daily)` workflow at `.github/workflows/self-check-daily.yml` is the deterministic safety net for the per-agent tool-allowlist contract ([#375](https://github.com/pyrycode/pyrycode/issues/375)). Between 2026-05-15 and 2026-05-23 it correctly went red the entire ~9-day life of the workflow while two regressions ([#526](526.md) — `selfcheck.go` not passing `AllowedTools` to `ptyrunner.Config`; and the pre-#526 prior cause) shipped to main undetected. The signal worked; the alerting surface (red README badge + Actions tab entry) did not reach the operator's actively-monitored channels. This ticket converts the passive badge into an active Discord push so a regression of the same shape gets investigated within a day, not the workflow's whole lifetime.

## Implementation

- **`notify-failure` job appended to `self-check-daily.yml` (+32 LOC; one file modified).** Runs only when `self-check` reports a terminal failure (`needs: self-check` + `if: failure()` at the job level — chosen over a step-level `if: failure()` inside the same job because the latter wouldn't fire on `timeout-minutes: 5` exhaustion, which is one of the failure modes the alert needs to cover).

- **Payload built with `jq -n --arg`, not string interpolation.** Four `--arg`s (`workflow`, `url`, `sha`, `date`) feed a single multi-line markdown `content` field. POSTed via `curl --fail-with-body -sS -H 'Content-Type: application/json' -X POST -d "${PAYLOAD}" "${DISCORD_WEBHOOK_URL}"`. No external GitHub Action dependency — matches CLAUDE.md's "stdlib over dependencies".

- **Required field sourcing.** Workflow name = `${{ github.workflow }}`; run URL = `${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}` (wrapped in `<...>` in the markdown to suppress Discord's link-preview embed, keeping the alert compact in a busy channel); commit SHA = `${SHA:0:12}` (12-char prefix for readability); date = shell-generated `date -u +'%Y-%m-%d %H:%M UTC'` (NOT `${{ github.event.repository.updated_at }}` — that field is the wrong thing on a `schedule` trigger and was the spec's flagged placeholder).

- **Secret-missing soft-fail.** `if [ -z "${DISCORD_WEBHOOK_URL}" ]; then echo "::warning::DISCORD_WEBHOOK_URL secret not set — alert suppressed"; exit 0; fi` ahead of the `jq` + `curl` chain. Until the operator wires the secret, every real `self-check` failure would otherwise produce *two* red signals (the original red badge plus a red `notify-failure` job going red on a missing webhook URL), doubling the noise the ticket is explicitly trying to reduce. Discord-API rejection (rate-limited, malformed payload, webhook revoked) goes the other way — `--fail-with-body` ensures the job exits non-zero so "the alert plumbing itself is broken" surfaces alongside the original badge. No retry; the next day's run alerts again if the failure persists.

- **Header comment extended, not replaced.** The existing "belt-and-suspenders safety net" paragraph stays intact. The previous trailing sentence `"A stronger pager hookup is a follow-up if false-positive rate is acceptable"` was removed (this ticket *is* that follow-up — the sentence is no longer truthful) and replaced with a four-line "Alerting contract (#533)" paragraph documenting the Discord post, the operator action ("investigate the linked run before trusting any new dispatcher activity"), and the secret-unset behaviour. A new contributor reading the workflow file knows exactly where the alert lands without leaving the file.

- **Out of scope, preserved.** No change to the `self-check` job body (the ticket's "Do NOT alter the existing self-check body" constraint). No change to `.github/workflows/ci.yml` or `.github/workflows/release.yml` — they had no pre-existing webhook-post pattern to mimic and aren't in scope for this ticket. The release-tag gate is a sibling ticket. No Go code touched.

- **No automated tests.** Workflow YAML; the test harness is the workflow itself. Developer-side verification: `actionlint` clean; `go vet ./...` + `go build ./cmd/pyry` clean (defensive — no Go code touched); local dry-run of the `jq` payload construction renders the expected Discord markdown; local dry-run of the secret-unset branch logs the warning and exits 0. End-to-end "the alert actually fires" verification is the operator's after they wire `DISCORD_WEBHOOK_URL`.

## Patterns established

- **Job-level `if: failure()` + `needs:` over step-level for workflow failure-alert jobs.** A step-level `if: failure()` only fires when a *prior step in the same job* failed — it does not fire when the job is cancelled or when `timeout-minutes` exhausts. Splitting the alert into its own job with `needs: <main-job>` + `if: failure()` at the *job* level captures all three terminal states (failed step, cancellation, timeout). **Generalises: when a CI workflow needs to fire a follow-up step strictly on terminal failure of a primary job, prefer a separate job gated by `needs:` + `if: failure()` over a step inside the primary job — covers the timeout/cancellation failure modes that step-level `if:` silently misses.**

- **Soft-fail on missing alert-channel secret to avoid doubling failure signals.** A naive "POST and let it 401" approach produces two red badges per real failure (the original + the alert job going red on the unset secret), which makes the alert *worse* than the badge it was trying to replace until the secret is wired. Soft-failing on the unset secret (`exit 0` with a `::warning::` line) contracts the noise window to "until the operator clicks Settings → Secrets". Strict-fail on the *Discord API rejection* path (`--fail-with-body`) is deliberately asymmetric: a revoked webhook or rate-limit is a real plumbing problem the operator wants to know about; a not-yet-set secret is a soft state transition that the PR description already calls out as a merge-time chore. **Generalises: when adding a failure-notification path that depends on a secret the operator has to provision separately from the code change, soft-fail on the unset secret (warning + exit 0) and strict-fail on every other downstream rejection — avoids the alert-plumbing self-amplifying the very failure signal it's trying to surface.**

- **Build webhook JSON payloads with `jq -n --arg`, not shell string interpolation.** `jq --arg` properly JSON-escapes the input (backticks, quotes, backslashes, control characters). String interpolation into a heredoc or `-d "{...$VAR...}"` fails the moment a commit-message-derived field or a future error-string field contains an unescaped quote, double-quote, or backslash. The cost is one extra binary on the runner — both `jq` and `curl` are present on `ubuntu-latest` by default. **Generalises: when POSTing JSON to any external webhook from a CI workflow, build the payload with `jq -n --arg field "$VAR"` not shell interpolation — protects against the day a field starts containing user-controlled or commit-message-derived bytes.**

## Lessons learned

- **The ticket's failure-window assertion was wrong, and the spec asked the developer to honestly audit rather than fabricate a match — the audit strengthened the ticket's case instead of weakening it.** The ticket asserted the failure window started 2026-05-19 and named `058c569` ("feat(streamjson): synthesise leading system/init envelope from runner inputs") as the trigger. The developer's audit per the spec's `gh run list --workflow=self-check-daily.yml --limit 100` instruction found: (a) `058c569` was actually merged 2026-05-21 13:47 UTC, not 2026-05-19, and unrelated to the `AllowedTools`-not-plumbed bug #526 fixed; (b) the workflow had failed *every run* since its creation 2026-05-14 — a ~9-day failure window, not 5+ days; (c) the regression-introducing commit was more likely `b2d6fd9` ("agent-run: adapt --self-check to exercise ptyrunner") merged 2026-05-20 12:41 UTC, but failures predated it too, suggesting the original cause may have been in `a209572` or even the initial workflow rollout `8f473b0` itself. The honest finding strengthens the ticket: the safety net failed silently not for 5 days but for the workflow's *entire 9-day life*, while a regression shipped to main undetected. **Generalises: when a ticket asserts a specific timeline or commit attribution as load-bearing context for "why this fix" framing, and the assertion is verifiable with `gh run list` / `git log`, audit it before mirroring it into the PR body — an honest audit that overturns the ticket's framing usually strengthens (not weakens) the case for the fix, because the original framing was conservative.** The spec's "Honest audit > fictional alignment" instruction is the decision rule.

- **`${SHA:0:12}` is bash parameter-expansion, not POSIX sh.** The code-review surfaced this as a `[NIT]`: `ubuntu-latest`'s default `run:` shell is `bash -e -o pipefail {0}` so the current code is fine, but if a future maintainer ever sets `shell: sh` on this job (or moves the snippet to a workflow that uses `sh`), the line silently dies at parse time — dash treats `${var:0:12}` as a syntax error and the job exits before posting the alert. Code review classified as flag-don't-fix because the GitHub Actions default-shell contract is stable and there's no current reason to switch. **Generalises: when a CI workflow snippet relies on bash-specific syntax (`${var:0:N}` substring extraction, `[[ ]]`, arrays, process substitution `<(...)`), either pin `shell: bash` explicitly on the step or document the bash-dependency in a comment so a future `shell: sh` pin doesn't silently break the path. For pyrycode's GitHub Actions today, the implicit `bash` default is sufficient and the dependency is invisible.**

- **The alerting failure was a *visibility* failure, not a *detection* failure — the fix lives at the surfacing layer, not the test layer.** The pre-#533 self-check ran daily, went red daily, exited non-zero daily — the deterministic guard worked exactly as designed for the entire 9-day failure window. What didn't work was the operator's mental model that "a red badge on a daily workflow will be noticed". The dispatcher pipeline + boards generate enough routine activity to drown out a passive red badge that requires the operator to *go look*; an active push into a channel the operator already monitors for dispatcher events is the cheap fix. **Generalises: when a safety net catches a regression but the regression still ships, the failure mode is "alert delivery", not "alert detection" — don't strengthen the test, strengthen the channel it fires into. Cost of an active push (one curl-step, one secret, one header-comment paragraph) is much lower than the cost of strengthening detection that was already working.** This is the alerting-side mirror of the "Evidence-Based Fix Selection" pipeline principle (don't ship a defense for an unobserved failure mode): the failure mode *was* observed (5 days of unread red badges → manual catch on day 9), so the defense — making the badge unmissable — is warranted.

## Out of scope (follow-ups)

- **Operator action required at merge.** The PR description (per AC) calls out that `DISCORD_WEBHOOK_URL` must be added to repo secrets (Settings → Secrets and variables → Actions) before merge — without the secret, the new step logs the warning and no-ops on the first failure. Tracked in the PR body, not in code.
- **Alert deduplication across consecutive failures.** A persistent failure produces one alert per day. Acceptable today — this is the "actionable signal" the ticket asked for. If alert fatigue emerges, file a follow-up to gate on "first failure after a run of green" (e.g. check the previous run's conclusion via `gh run list --json conclusion`), but do not preempt that here.
- **Release-tag gate.** Called out as out-of-scope by the ticket's Technical Notes. Sibling ticket, not in #533.
- **Embed vs plain `content`.** Discord embeds (richer formatting, color stripe) are an alternative. AC names required fields, not formatting. Plain `content` shipped per spec recommendation ("stdlib over dependencies", one fewer JSON shape to maintain). Switch to embed only if the operator finds plain-text fatigue or wants per-severity color coding.
