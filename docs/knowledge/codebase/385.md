# Ticket #385 — `e2e/realclaude`: cache-hit verification + max-turns boundary tests

Seventh consumer of the `internal/e2e/realclaude` fixture trio (#372 + #373 + #409). Two budget-guardrail tests in a new file that pin cost-relevant guarantees the suite did not previously exercise end-to-end: prompt-cache alignment across `pyry agent-run` invocations within the 1-hour TTL, and `--max-turns=N` enforcement at the budget boundary.

Motivated by two silent-regression classes the prior consumers don't cover: (a) a change that breaks the Anthropic prompt-cache key alignment (dynamic content sneaking into the system prompt, per-invocation tool-list churn) would silently 5-10x API spend on every nightly run with no failing assertion; (b) the only structural pin on `--max-turns` enforcement today lives at the emitter unit-test layer (`internal/agentrun/streamjson/emitter_test.go:225`) — nothing asserts end-to-end that pyry's budget Counter actually caps a run at N turns against the real claude CLI. No production-code change; the only non-test edit is a three-field extension of the file-local `resultTrailer` struct in `tool_loop_test.go` (and one new sub-struct alongside it).

See [features/e2e-realclaude.md](../features/e2e-realclaude.md) for the suite contract.

## Implementation

- **`internal/e2e/realclaude/budget_test.go`** (~199 LoC, new). `//go:build e2e_realclaude`, `package realclaude`. Two top-level `Test…` functions, four file-level constants (`cacheHitSystemPrompt`, `cacheHitUserPrompt`, `maxTurnsSystemPrompt`, `maxTurnsPrompt`), and one file-local `truncateStdout([]byte) string` helper. No edits to `fixtures.go`, no new exported fixture surface, no production-source changes.
- **`TestRealClaude_CacheHitWarmsAcrossRuns`** (`budget_test.go:53-112`). Runs the same `RunOpts` skeleton twice through `RunPyryAgentRun` against `WithWorktreeAuthenticated(t)`, parses both trailers, then asserts:
  - **Primary** (`t.Fatalf`): `secondTrailer.Usage.CacheReadInputTokens > 0` — the second invocation hit the prompt cache. A regression that breaks cache-key alignment will show `== 0` here.
  - **Diagnostic** (`t.Errorf`, not `t.Fatalf`): `firstTrailer.Usage.CacheCreationInputTokens > 0` — the first run paid the cache-creation cost. **Soft signal only.** A `0` here with the primary still passing means the cache was already warm from an earlier nightly (the second run hit a pre-existing cache); a `0` here paired with a `0` primary means the system prompt is below the per-model implicit-cache threshold (~2048 tokens on Haiku 4.5 per Anthropic public docs as of 2026-05) and needs padding. **Order is load-bearing**: primary first so a real cache-alignment regression surfaces with the correct failure message; diagnostic second via `t.Errorf` so it doesn't mask the primary on a passing run.
- **`cacheHitSystemPrompt`** (5-sentence string concatenation, ~300 tokens): deterministic across both runs (single `const`, no dynamic content), sized to clear the implicit-cache minimum by margin, discourages tool use and verbose replies. **Do NOT include the literal date, run id, or any other dynamic value** — that is exactly the regression class this test is designed to catch; the doc comment above the const names this constraint explicitly so a future maintainer doesn't "improve" it with a timestamp.
- **`cacheHitUserPrompt = "Reply with a single word: ok."`** — keeps haiku/low on a single end-of-turn rail.
- **`RunOpts` for cache-hit**: `Workdir: workdir` (reused for both runs — incidental; the cache is per-API-key per-content-hash, not per-workdir), `Prompt: cacheHitUserPrompt`, `SystemPrompt: cacheHitSystemPrompt`, `AllowedTools: []string{"Read"}` (minimum non-empty satisfying `validateRunOpts`), `MaxTurns: 1`, `Effort: "low"`, `Model: "claude-haiku-4-5"`. Two haiku/low calls, 1 turn each, ~$0.02 per nightly.
- **`TestRealClaude_MaxTurnsHonored`** (`budget_test.go:146-189`). `RunPyryAgentRun` against `WithWorktreeAuthenticated(t)` with a prompt that natural-completion would require ≥5 turns and `MaxTurns: 2`. After standard run-level asserts (`ExitCode == 0`, `SessionID != ""`) and trailer parse, asserts five fields:
  - `trailer.Subtype == "error_max_turns"` — the budget-exhaustion subtype.
  - `trailer.TerminalReason == "max_turns"` — the structured terminal-reason signal.
  - `trailer.NumTurns == 2` — **exact match**, not `>=`. The budget caps at exactly `--max-turns`; an off-by-one fires here.
  - `trailer.StopReason != "end_turn"` — natural completion would mean the model collapsed the 5-step prompt into ≤2 turns; the assertion is phrased as `!= "end_turn"` per the AC because the precise value depends on what state the model was in when SIGTERM landed (typically `tool_use`).
  - `trailer.IsError == true` — the wire-level `is_error: true` flag is the explicit AC signal of "budget exhaustion, not natural completion" beyond the subtype string.
- **`ExitCode == 0` is correct on the max-turns test.** `pyry agent-run` exits 0 on a successfully-emitted result trailer regardless of trailer `is_error`. The budget-exhaustion signal lives in the trailer fields, NOT the subprocess exit code. A code comment names this explicitly so the next maintainer doesn't "fix" the assertion to `!= 0`.
- **`maxTurnsPrompt` shape** (5-line numbered list + explicit "do NOT combine" guidance): forces haiku/low into one Bash invocation per step. Confirmed empirically across siblings (#376, #382) that haiku at `--effort=low` with `--allowed-tools=Bash` on a "do five things" instruction uses the tool once per turn for each step. Each step consumes one assistant turn (`tool_use`), so 5 steps + a final summary turn = 6 turns of work; capped at `--max-turns=2`, the run terminates after exactly 2 assistant entries with the budget signal.
- **Tool-call-collapse risk pinned in a comment above the const.** If a future haiku revision fires all five `echo`s in one `tool_use` block (or otherwise completes in ≤2 turns naturally), the `Subtype == "error_max_turns"` and `TerminalReason == "max_turns"` assertions fail loudly — surfacing as a clear test failure, not a silent pass. The mitigation is to bump `maxTurnsPrompt` to force more turns (e.g. 8 sequential `read X.txt` calls with seeded file content the model must enumerate by name), not to weaken the assertion. The maintainer who hits this failure on a model upgrade reads the comment, knows what to tune.
- **`RunOpts` for max-turns**: `AllowedTools: []string{"Bash"}`, `MaxTurns: 2`, `Effort: "low"`, `Model: "claude-haiku-4-5"`. ~$0.02 per nightly (matches the AC's $0.04 combined estimate when added to the cache-hit test).
- **`internal/e2e/realclaude/tool_loop_test.go`** — three-field extension to `resultTrailer` (`tool_loop_test.go:189-198`) plus one new sub-struct `resultTrailerUsage` (`tool_loop_test.go:200-205`). All three new fields (`IsError bool`, `TerminalReason string`, `Usage resultTrailerUsage`) are `omitempty`-tagged; pre-#385 consumers (#376/#381/#382/#384) that don't read them decode unchanged. The `resultTrailerUsage` sub-struct emits all four token-count fields (`input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`) with `omitempty`. Doc-comment on `resultTrailer` gained a two-line note naming the #385 additions so a future reader sees why three fields decode-but-aren't-read by older tests. **Single-source extension over duplication**: defining the new fields in a separate `budgetTrailer` struct inside `budget_test.go` (the spec's named alternative) would have meant two parsers diverging silently on a future emitter field rename; one struct is cheaper to keep aligned.
- **`truncateStdout([]byte) string` helper** (`budget_test.go:194-199`). 1 KiB cap with `... (truncated)` suffix, mirrors the inline pattern from `tool_loop_test.go:127` and the file-local `truncate` from `per_agent_test.go` (#381). Kept file-private rather than reaching for the existing `truncate` from #381 because the spec explicitly forbids touching `fixtures.go` and the existing helper is its own file's private. Promote to a shared file-private export if a third trailer-failure-message file shows up.

## Reuses (no re-implementation)

- `realclaude.WithWorktreeAuthenticated` (#409) — both tests. The cache-hit test reuses the same `workdir` for both runs (incidental); the max-turns test uses one workdir for the single run.
- `realclaude.RunPyryAgentRun` (#373) — both tests. Cache-hit calls it twice against the same `opts` struct (`RunPyryAgentRun` rewrites `prompt.txt`/`system.txt` on each call, so the same struct produces deterministic, identical-bytes invocations); max-turns calls it once.
- `realclaude.parseResultTrailer` + `resultTrailer` (#376) — **fifth consumer** (after #376, #381, #382, #384). Same stdout-line scanner; the three new fields extend the shared struct in-place. Both tests parse the trailer (cache-hit parses both runs' trailers).

Deliberately NOT reused:
- `realclaude.parseContentBlocks` + `contentBlock` (#376) — neither test inspects content blocks; the property under test is the trailer shape only.
- `realclaude.ReadJSONL` (#372) — both tests are trailer-only; the spec explicitly forbids reaching into `internal/agentrun/jsonl` for the cache-hit assertion ("the trailer is what nightly cost-regression dashboards aggregate over").
- `realclaude.jsonlPathFor` (#364) — neither test walks the on-disk JSONL, so the file-private helper isn't called here.

## Patterns established

- **Diagnostic-only assertion via `t.Errorf` for a soft signal that should never mask the primary.** The cache-creation-on-first-run check (`firstTrailer.Usage.CacheCreationInputTokens > 0`) is informational: a `0` here with the primary passing means the cache was already warm from an earlier nightly, not a regression. Using `t.Errorf` instead of `t.Fatalf` keeps it surfacing on a green run without escalating to a real failure. The pattern generalises: when an assertion encodes a tuning signal rather than a contract, demote it to `t.Errorf` and document the interpretation matrix (0+pass vs. 0+0) in the comment above the check. The reader who hits a `t.Errorf` knows to read the comment, not file a regression.
- **Comment the next maintainer's escape hatch on a model-dependent assertion.** `TestRealClaude_MaxTurnsHonored` assumes haiku/low won't collapse five sequential `echo`s into one tool_use. That assumption is empirical, not contractual; a future model revision could break it. The right move is not to weaken the assertion (which would hide the regression class the test exists to catch) but to inline a one-line comment above `maxTurnsPrompt` that names the failure mode and the fix (bump the prompt to force more turns). The maintainer who hits the failure on a model upgrade reads the comment, knows what to tune, doesn't relax the contract. Generalises: any assertion that depends on a third-party producer's behaviour should carry an inline "if this fails, here's how to fix it without losing the property" hint.
- **Extend the package-private parser struct in-place, with `omitempty`-tagged additions, rather than duplicating the parser.** The trailer shape is single-source; two parsers (the existing one in `tool_loop_test.go` and a hypothetical `budgetTrailer` in `budget_test.go`) would diverge silently on a future emitter field rename. Adding three `omitempty`-tagged fields to the existing `resultTrailer` and a new `resultTrailerUsage` sub-struct alongside it (same file, same package) is the minimum-friction extension: edit count 1 file / 1 struct / 1 new sub-struct / ~8 added lines, zero behaviour change to the four pre-#385 consumers (#376/#381/#382/#384) that don't read the new fields. Generalises: when a test family shares a wire-format parser, extend the parser at its origin file even from a new consumer, never duplicate.
- **`ExitCode == 0` and "budget exhaustion" can coexist; pin the rationale in a code comment.** A reader scanning `TestRealClaude_MaxTurnsHonored` might naively "fix" `ExitCode == 0` to `!= 0` because the test is about max-turns enforcement. But `pyry agent-run` exits 0 on a successfully-emitted result trailer regardless of trailer `is_error`; the budget signal lives in the trailer fields, NOT the exit code. The comment above the assertion makes the rationale legible. Generalises: when a process exit code is divorced from the property the test is checking, name the divorce inline so a future change to the assertion is a deliberate decision, not a misread.

## Lessons learned

- **The trailer's `usage.cache_*_input_tokens` is aggregated across all assistant messages by `streamjson/emitter.go:216-221`, not per-message.** The on-disk JSONL also carries per-assistant `usage` blocks (`reader.go:152-157`), and either could in principle satisfy the AC. Use the trailer on stdout, not the on-disk JSONL, because (a) the trailer is what nightly cost-regression dashboards read (same envelope as `total_cost_usd`), (b) it's the assertion target across siblings #376/#381/#382/#384, and (c) the on-disk path requires walking events and summing across assistants — duplicating what the emitter already aggregated. Spec-level rule: "the spec explicitly forbids reaching into `internal/agentrun/jsonl` for this assertion." Generalises: when an aggregated and a raw-event source both report the same value, assert on the aggregated one if there's a dashboard reading it — pins two ends of the cost path with one assertion.
- **`omitempty` on `IsError bool` is load-bearing.** Without it, every pre-#385 test that decodes a trailer where the upstream emitter set `is_error: false` would decode the field as `false` whether the wire emitted it or not — fine in practice, but the `omitempty` tag preserves the "field absent vs. field present, value false" distinction symmetric with `PermissionDenials *[]json.RawMessage` from #376. The pattern is consistent across all five trailer fields; deviating on `bool` would create a single asymmetry the next maintainer would have to reason about. Generalises: when extending a shared parser struct, match the surrounding `omitempty` discipline even for bools — symmetry beats micro-optimisation.

- `make check` (vet + test + staticcheck) — unaffected.

## Out of scope (deferred)

- **Nightly cost-regression CI assertion** that compares `cache_read_input_tokens` to a historical baseline. The cache-hit test pins the qualitative "cache warms" property; a quantitative drift assertion is filed as a follow-up only if observed drift in the nightly dashboard warrants it.
- **Asserting `total_cost_usd` is populated.** The trailer carries the field but pyry's emitter zeroes it (`emitter.go:215` per the spec). A future cost-dashboard ticket may pin this once the field is populated upstream.
- **Re-validating the `--max-turns` flag value parser.** Already pinned at `cmd/pyry/agent_run_test.go`; not re-asserted here.
- **Promoting `truncateStdout` / `parseResultTrailer` to fixture exports.** Each is a file-private helper consumed by 1-5 sibling tests in the same package. YAGNI bar not yet met after this seventh consumer — every consumer still lives in the same package, so file-private suffices.
- **Probing Anthropic's cache shape directly** (cache-breakpoint markers, TTL boundary, eviction). Pyry consumes claude's cache reporting passively; how the cache is constructed is upstream's concern.
