# Ticket #679 — turn bridge follows the active conversation's bound session

The capstone of EPIC #672's outbound migration (after [#678](678.md) routed inbound turns per-conversation and [#687](687.md) re-keyed the outbound *cursor/attribution* to the active conversation). Before #679 the structured interactive turn stream's producer still **tailed by recency** — it subscribed over the **bootstrap** supervisor and resolved the *newest* `<uuid>.jsonl` in the shared sessions dir (`resolveLatestSessionJSONL`). With a per-conversation bound session writing alongside the bootstrap (or another conversation's session writing more recently), "newest JSONL" no longer identifies the conversation being interacted with — a reply could be tailed from whichever session last wrote, **cross-streaming another discussion's output into the reply**. #679 makes the producer **follow the active conversation**: it re-keys the subscription host, the JSONL resolver, and the PTY-state source to the active conversation's **bound session**, resolves that session's transcript **by its bound session id** (mtime-independent), and re-subscribes when the active conversation (or its session) changes. This closes the cross-conversation confidentiality gap (`security-sensitive`).

See the evergreen feature docs: [features/turnbridge-package.md § The follow-active subscriber](../features/turnbridge-package.md#the-follow-active-subscriber-newtargetsubscriber-679) and [features/conversation-session-binding.md § Edge cases & limitations](../features/conversation-session-binding.md#edge-cases--limitations).

## Implementation

4 production `.go` files + 3 test files; +1056/-172. No `Producer` change, no `internal/sessions`/`reconcile.go` change (AC4). The whole re-key rides the **existing** `Producer.Run` re-subscribe loop (subscribe → drain → channel-closed → subscribe).

- **`turnbridge.Target` + `TargetResolver` + `NewTargetSubscriber`** (`internal/turnbridge/producer.go`). The old `NewSessionSubscriber(host, resolve, …)` baked in one fixed host + one resolver captured once; it is **removed** and replaced by a target-driven subscriber. `Target{Host, Resolve, Switch}` is what one (re)subscription needs, resolved fresh per subscription by a `TargetResolver func(ctx) (Target, error)` callback (supplied by the caller, so turnbridge stays import-neutral — no `sessions`/`conversations` import). `Switch <-chan struct{}`, when non-nil, tears the subscription down so `Run` re-subscribes onto the now-current Target. Two new exported types; the package now exports six (`Producer`, `Config`, `Subscriber`, `SessionHost`, `Target`, `TargetResolver`).
- **Two-loop subscriber body.** `NewTargetSubscriber` has an **outer** loop (calls `resolve` once per re-subscription to snapshot a `Target`) and an **inner** loop (runs `WaitForPTY → resolve-JSONL → Events` for that Target, transient retries reusing `target.Resolve` so its per-subscription cold/warm offset state persists). A switch (`target.Switch` fires) cancels the per-subscription `subCtx`, aborting any in-flight pre-stream wait and breaking to the outer loop via `continue resubscribe` to re-snapshot. **Switch-abortable waits** (`WaitForPTY`, `WaitForSessionJSONL`, `Events` all take `subCtx`) are the one new invariant — load-bearing for AC3 (otherwise the producer wedges forever in `WaitForPTY` of a stale evicted session after a switch).
- **`resolveTarget`** (`cmd/pyry/interactive_turn_stream_v2.go`) — the follow-active `TargetResolver`, closing over `active`, `boundHost`, `sup` (bootstrap), `claudeSessionsDir`. Snapshots the active conversation via `active.watch()` (id **and** the change channel, atomically) and maps it: `convID == ""` (no route yet) → bootstrap host + `resolveLatestSessionJSONL` (the unchanged AC4 path; `Switch` still set so the first route re-subscribes); `convID != ""` resolvable → bound session's supervisor + `resolveBoundSessionJSONL(dir, sessionID)`; `convID != ""` **unresolvable** → an **error** (subscriber backs off + retries) — **never** a bootstrap fallback under a non-empty cursor (that would cross-stream).
- **`resolveBoundSessionJSONL(dir, sessionID)`** (`cmd/pyry/interactive_turn_stream_v2.go`) — sibling of the recency resolver that tails a **fixed** `<sessionID>.jsonl` instead of scanning for the newest. Validates `sessionID` against `jsonlStemPattern` (the UUID-stem regex) **before** `filepath.Join` (path-safety / traversal guard), then mirrors the recency resolver's per-subscription cold/warm offset rule (absent-then-appears → offset 0 cold start, #671; present at first look → offset = size warm). Because the path is fixed by id, **mtime is never consulted** (AC2).
- **`boundHost`** (`cmd/pyry/main.go`, built in `runSupervisor` where `pool`/`convReg` are concrete) — the `boundHostFunc` lookup `convReg.Get → CurrentSessionID (non-empty guard) → pool.Lookup → sess.Supervisor()`. Returns `(nil, "", false)` on unknown conversation, empty binding, or `Lookup` miss. `*supervisor.Supervisor` satisfies `turnbridge.SessionHost` structurally, so the bound supervisor is **both** the subscription host and the PTY-state source (couplings 1 & 2 collapse). Threaded as one new param down `startRelay → startRelayV2 → startInteractiveTurnStreamV2`.
- **`activeConversation` change signal** (`cmd/pyry/main.go`) — the #687 holder gains `changed chan struct{}` (lazy-init under `mu`), a fire-on-**change** tweak to `set` (close+replace `changed` only when the id differs — consecutive messages to the *same* conversation do **not** re-subscribe), and a `watch() (id, <-chan struct{})` method that snapshots id + channel atomically. Mirrors the supervisor's own `activeCh` replace-on-transition pattern.

## Patterns established

- **Generalise a fixed-host subscriber into a per-subscription `Target` resolved by a callback.** Rather than add follow-active machinery, #679 turned the single captured `(host, resolve)` into a `Target{Host, Resolve, Switch}` that a `TargetResolver` yields fresh on each (re)subscription. The follow-active re-key then needs **no `Producer` change** — a switch closes the stream, the existing `Run` loop re-subscribes, and the fresh resolver snapshots the now-active session. turnbridge stays import-neutral (the conv→session knowledge lives in the `cmd/pyry` callback). `NewSessionSubscriber` was removed outright (not kept as a wrapper) because its single-host behaviour *is* the `convID == ""` branch of `resolveTarget` — AC4 is preserved by a tested branch, not dead code.
- **Discriminate the cancel cause *before* calling your own `cancel()`.** In the subscriber's error arms, a transient resolve error (file not present yet → backoff + retry) and a switch (re-snapshot) must be distinguished by `subCtx.Err()` — but calling the local `cancel()` first makes `subCtx.Err()` unconditionally non-nil, masking a transient as a switch. The fix checks `ctx.Err()` (parent) then `subCtx.Err()` (switch) **before** `cancel()`. (Code-review verified this ordering is load-bearing.)
- **The `CurrentSessionID == "" ` guard before `pool.Lookup` is the confidentiality check, reused outbound.** `boundHost` rejects an unbound conversation *before* `Lookup`, because `Pool.Lookup("")` returns the **bootstrap** (the #678 hazard). On the outbound path the same guard prevents an unbound conversation from resolving to the bootstrap and cross-streaming — the inbound (#678) and outbound (#679) halves share the exact same load-bearing line.
- **Fire-on-change, not fire-on-set.** The active-conversation signal closes `changed` only when the id *differs*, so consecutive turns to the same conversation keep one tail open (as the bootstrap did) and only a genuine switch forces a re-subscribe. Close+replace is single-shot per channel instance, mirroring `supervisor.transitionTo`.

## Lessons learned

- **The cold/warm offset state (`resolvedOnce`/`sawEmpty`, the #671 cold-start gate) must persist across a subscription's *transient* retries (file-not-yet-present backoff) but reset on a *re-subscription* (switch / session-end).** That forces a two-loop subscriber: an outer loop that calls the `TargetResolver` once per re-subscription, and an inner loop whose transient retries reuse the same per-target resolver closure. A flat "re-resolve on every retry" loop silently reintroduces the #671 cold-start drop for bound sessions (the fresh closure forgets it saw the file absent).
- **The unresolvable-bound branch of `resolveTarget` must return an error (→ retry), never a bootstrap-`Target` fallback:** the emitter stamps the live `conversation_id`, so tailing any other transcript under a non-empty cursor would cross-stream. The bootstrap/recency path is reachable only when the cursor is empty (no route yet), where the emitter drops every event anyway.
- **The composed switch-on-success-path re-subscribe is covered transitively, not end-to-end (code-review NIT, non-blocking).** The switch-abort is unit-tested pre-stream (`NewTargetSubscriber_SwitchAbortsWait`, blocked in `WaitForPTY`) and the channel-close re-subscribe via `scriptedSubscriber`; the full composition (switch cancels `subCtx` on the live-stream success path → `Events` channel closes → `Run` re-subscribes → reply flows on stream 2) is covered across those two tests but not in one end-to-end test. Coverage is genuinely present; a single composed producer-level switch test would be a clean follow-up.

## Out of scope (deferred)

- **Two-conversation structured e2e.** A full e2e — two bound sessions writing two JSONLs concurrently (the other newer), assert the phone receives *only* the bound conversation's reply — would prove AC2 end-to-end but needs the daemon to spawn per-conversation sessions under fakeclaude with two live structured JSONLs (an extension of `relay_two_phone_structured_test.go`, inheriting its cold-start offset race). The security-relevant decision (*which file is tailed*) is fully pinned by the deterministic by-id unit test + the producer-level switch test. Recommended as a follow-up harness ticket.
- **Multi-operator cross-conversation isolation.** Concurrently delivering *distinct* per-conversation reply streams to *multiple* connections (two phones each viewing a different conversation). The structured stream fans out to all interactive conns by capability — there is no connection→conversation binding for output. This ticket covers the **single** active reply stream following the operator's current conversation; the connection-level focus model is undesigned (likely cross-repo) and deferred until the usage is real.
- **Switched-away session-end watcher lifetime.** A switched-from session's `sess.Wait` watcher lives until that session ends (bounded by live-session count ≤ `ActiveCap`). Adding a per-subscription `sess.Wait` cancel is unwarranted with no observed failure.
