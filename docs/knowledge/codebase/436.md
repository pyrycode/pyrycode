# Ticket #436 — `cmd/pyry pair preflight` empty-registry gate for v2 cutover

Adds a dedicated `pyry pair preflight` sub-verb that the v2 release workflow invokes before flipping the cutover flag: exits 0 when the device registry is empty (gate passes), 2 when any paired device exists (gate fails — one stderr line with the count), and 1 on registry I/O / malformed JSON (the check itself failed). v1 pair records have no `server_static_pubkey` (#432) and v2 has no migration tooling per [ADR 024](../decisions/024-noise-ik-mobile-e2e.md), so any non-empty registry at v2 cutover would silently break every paired phone with WS code `4426` and no recovery path other than `pyry pair revoke && pyry pair` per device.

## Implementation

- **Dedicated sub-verb, not a flag on `pair list`.** `pyry pair preflight [-pyry-name=<instance>]` — `pair list`'s default tabular stdout output is unchanged so out-of-tree scripts parsing it are unaffected (issue body Technical Note). Smuggling silent-on-stdout + non-zero-exit behaviour through a `--fail-if-any` flag on `pair list` would have been more surprising than a clearly-named sibling; CI lines literally read `pyry pair preflight`, which is self-describing.
- **One file touched in production code:** `cmd/pyry/pair.go` — ~47 LOC. Three additions: a `case "preflight":` arm in `runPair`, the `pairPreflightArgs` + `parsePairPreflightArgs` parser pair (clone of `pairListArgs`/`parsePairListArgs` — same `-pyry-name`-only shape, no positionals), and `runPairPreflight` itself. `pairVerbList` constant updated from `"list, revoke"` to `"list, revoke, preflight"` in lockstep (the doc-comment above it already pinned the rule).
- **Read path reuses `devices.Load` + `registry.List()` verbatim** — same two-line preamble as `runPairList`. **Not extracted into a shared helper:** the duplication is two lines, and per Pyrycode's per-registry duplication policy this is the first sibling reader — not enough pressure for extraction. A shared helper would either take a post-load callback (loses readability) or leak the loaded `*Registry` (splits lifecycle).
- **Pure verdict helper `preflightVerdict(count int) (exitCode int, stderrLine string)`** owns the byte-exact stderr-line contract. Returns `(0, "")` on `count == 0`, `(2, "pyry pair preflight: <N> paired device(s); v2 release gate requires zero.")` on `count > 0`. Format-frozen — no singular/plural branch ("device(s)" is deliberate; CI output pluralisation is not worth the code path). Extracting the verdict from `runPairPreflight` lets `TestPreflightVerdict` pin the stderr-line bytes without driving `os.Exit` through a subprocess.
- **Exit-code triple deliberately split — 0 / 1 / 2 — to mirror `grep(1)`:**
  - `0` — registry empty (gate passes). `runPairPreflight` returns nil.
  - `1` — registry I/O error or malformed JSON. `runPairPreflight` returns `fmt.Errorf("pair preflight: %w", err)`; `main.run` prefixes with `pyry: ` and `os.Exit(1)`s. Release tooling treats this as "the check itself failed" — investigate, retry.
  - `2` — registry non-empty OR usage error (parse failure, unknown flag, unexpected positional). `runPairPreflight` calls `os.Exit(2)` directly to bypass the `pyry: ` prefix, emitting either the verdict line or `pyry pair preflight: <err>` + usage line. Release tooling treats this as "the gate caught what it was supposed to catch" — operator must `pyry pair revoke` per device first.
- **Cold-start contract preserved.** Missing `devices.json` and zero-byte `devices.json` both return an empty `*Registry` with nil error from `devices.Load` (registry contract at `internal/devices/registry.go:40-47`); both collapse to exit 0 with no on-disk side effect. `TestRunPairPreflight_EmptyRegistry` and `TestRunPairPreflight_ZeroByteRegistry` pin this directly.
- **Stdout silent on every branch.** Allows release tooling to redirect stdout to `/dev/null` without losing the gate-failure message (which goes to stderr).
- **Doc update in `docs/protocol-mobile.md` § *Pre-flight: `pyry pair list` empty check*** — section heading anchor unchanged so the cross-reference at line 9 (`see [Pre-flight](#pre-flight-pyry-pair-list-empty-check) and #436`) still resolves; only the body changes. Names the verb, documents the 0/1/2 contract with operator-facing implications, shows the intended CI invocation (`if ! pyry pair preflight; then exit 1; fi`), and explicitly flags that the v2 release-flag mechanism does not yet exist (issue body AC #4 conditional) so this section is the load-bearing artefact until one lands.

## Patterns established

- **Per-verb policy decisions live in pure helpers, not inside the runner.** Extracting `preflightVerdict` from `runPairPreflight` is what makes the byte-exact stderr-line contract unit-testable. The runner becomes the thin shell: parse → load → verdict → dispatch. This is a tighter version of the same shape `pyry pair revoke` uses today, where the not-found policy decision sits inline at `runPairRevoke`'s line 339-342 with no helper to test — `TestRunPairRevoke_*` covers the I/O paths but the not-found stderr text is only covered in e2e. The preflight verb does it cleaner: the policy decision is fully covered in unit tests, the runner only adds the `os.Exit(2)` dispatch (two-line trivial wiring).
- **Exit codes mirror Unix utility conventions (`grep` 0/1/2) when CI consumes the binary.** The 1-vs-2 split is load-bearing for release tooling: "the check itself failed" needs human investigation; "the gate fired" needs the operator-instructed remediation (`pyry pair revoke`). Conflating both at exit 1 would have been simpler in code but lossier at the CI boundary. Future CLI verbs intended for CI gating should follow the same convention.
- **Opt-in release gates layer on top of human-facing verbs without disturbing them.** `pair list`'s tabular output is operator-facing and may be parsed by out-of-tree scripts; `pair preflight` is machine-facing with a byte-pinned stderr message. The two surfaces share read code (`devices.Load` + `registry.List()`) but diverge entirely in output contract. Cleaner than adding flags that mutate one verb's output for CI consumption.

## Lessons learned

- **Don't extract on the first sibling reader.** The architect's spec explicitly flagged the two-line `devices.Load` + `registry.List()` preamble in `runPairPreflight` as duplicating `runPairList`'s preamble, and explicitly said *don't extract*. The developer respected this. PROJECT-MEMORY.md § *Atomic-write recipe* names the project policy ("duplicated until a Nth instance forces extraction") but it applies more broadly: a shared helper for two lines that compose differently downstream (one renders a table, the other counts and gates) would either grow callback parameters or leak shared state. The pressure threshold is the second sibling reader for read-then-render code, not the first.
- **`os.Exit(N)` paths are unit-testable through a pure verdict helper, not through `TestHelperProcess`.** The spec made the call explicitly: `TestPreflightVerdict` covers the byte-exact stderr message + exit-code mapping, and the two-line `os.Exit(exitCode)` dispatch after the helper is trivial wiring that doesn't need a subprocess test. The alternative (a `TestHelperProcess` re-exec to capture `os.Exit(2)` from inside `runPairPreflight`) would have re-verified the dispatch lines without buying coverage of the actual contract. Lesson generalises: when a function ends in `os.Exit`, extract the policy decision into a pure helper *first*, then the test surface follows.

## Out of scope (deferred)

- **The v2 release-flag mechanism itself.** Pyrycode has no feature-flag mechanism distinct from version bumping at the time of this ticket; AC #4 is satisfied by documenting the intended invocation in `docs/protocol-mobile.md`. When the release-flag mechanism lands, its CI workflow should invoke `pyry pair preflight` and abort on non-zero exit.
- **Migration tooling for existing v1 pair records.** Explicitly rejected by the v2 spec (ADR 024). Operators with paired devices at v2 cutover must `pyry pair revoke` per device first.
- **Race against running daemon.** Same caveat as `pyry pair list` — the verb reads `devices.json` once; concurrent `pyry pair` or `pyry pair revoke` invocations on the same machine could write between read and exit. Out of scope for the same reason as the parent verb family (the file-lock fix belongs at the `internal/devices` boundary, not the CLI layer).
