# Ticket #383 — `--permission-prompt-tool stdio` protocol spike

Captured-trace test that invokes `claude` directly (not `pyry agent-run`) with `--permission-prompt-tool stdio` plus a chosen `--permission-mode`, and writes every stdout stream-json line to a fixture under `internal/e2e/realclaude/testdata/`. Built for the pyrycode-mobile permission-relay design, which needs to know the on-the-wire shape of permission events before mobile-side handling is designed.

## Implementation

- **`internal/e2e/realclaude/permission_protocol_spike_test.go` — new file**, build tag `//go:build e2e_realclaude`, no exported symbols. One test: `TestRealClaude_PermissionProtocol_Spike`. Uses `WithWorktreeAuthenticated(t)` (from [#409](409.md)) so the test skips cleanly when `ANTHROPIC_API_KEY` is unset and runs against the real Anthropic API otherwise.
- **Argv divergence from `pyry agent-run`'s canonical `buildClaudeArgs`** (`cmd/pyry/agent_run.go:254`):
  - NO `--dangerously-skip-permissions` — that flag suppresses permission gates and would defeat the spike.
  - ADDS `--permission-prompt-tool stdio` and `--permission-mode <mode>`.
  - OMITS `--append-system-prompt-file` and `--effort` — minimal input surface across reruns.
- **Stdin envelope** matches `internal/agentrun/streamrunner` production shape: `{"type":"user","message":{"role":"user","content":[{"type":"text","text":"..."}]}}`. The architect's open-question fallback (`{"type":"user","content":"..."}`) was not needed — the production shape was accepted with no stderr error.
- **Process I/O**: explicit `cmd.StdinPipe()` + `cmd.StdoutPipe()` + `bytes.Buffer` for stderr. Stdin gets one envelope + newline, then close. A single reader goroutine scans stdout via `bufio.Scanner` with a 1 MiB line cap (stream-json model output can outgrow the 64 KiB default — matches `parseResultTrailer`'s note at `tool_loop_test.go:189`). The test goroutine waits on `cmd.Wait()` then on the reader goroutine's `done` channel before reading the slice — no lock, single-writer discipline.
- **Hard timeout** `90 * time.Second` via `context.WithTimeout`. The test does NOT respond to permission events (rationale in spec); claude may block forever waiting for a response, and the timeout is the bound that lets the test finish. Recorded as `context_deadline_tripped: true` in the fixture; the test PASSES regardless.
- **Failure shape**: structural failures (process can't start, fixture can't write, claude not on PATH → `t.Skipf`) fail the test; behavioral signals (claude crashed, hung, exited non-zero) get recorded into the fixture and pass. The only test-fatal behavioral case is "deadline tripped AND zero stdout events captured" — that means the run produced no signal at all.
- **Fixture path**: `internal/e2e/realclaude/testdata/permission_protocol_v<slug>.json`. `slug` is the leading whitespace-split token from `claude --version` (e.g. `2.1.143` from `2.1.143 (Claude Code)`), lower-cased, with non-`[a-z0-9._-]` chars replaced by `_`, capped at 32 chars. Falls back to the trimmed full output if the parse yields no fields. Atomic-write: `os.WriteFile(path+".tmp", ...)` → `os.Rename(tmp, path)` (matches `PROJECT-MEMORY.md` "Atomic-write recipe").
- **Fixture record shape** (`fixtureRecord` struct, file-private): records the full untrimmed `--version` raw output, the extracted version token, the verbatim argv slice, the permission mode, the allowed-tools list, the stdin envelope, every stdout line as a `json.RawMessage`, stderr capture truncated at 8 KiB, exit code, deadline-tripped bool, and run duration in ms. Indented with two spaces so diffs are readable.
- **Matrix sweep is manual**, per spec. The `permissionMode` constant at the top of the test file is the single knob. The spike-runner edits it, reruns, then renames `permission_protocol_v<version>.json` to `_<mode>.json` before the next run. The committed fixtures (six of them, one per mode) are the artefacts of this discipline at v2.1.143.
- **`truncateString`, not `truncate`** — `per_agent_test.go:138` already exports a `truncate(b []byte) string` at package scope. To avoid the redeclaration collision (caught by `go vet -tags e2e_realclaude`), this file uses `truncateString(s string, max int) string`. No attempt was made to lift the helper into a shared utility — file-local is enough until a fourth consumer materialises.
- **`packageDir(t)` returns `os.Getwd()`**, not `runtime.Caller`-based discovery. Go test runs each package's tests in its own source directory, so `$CWD` is the source dir at test time. `runtime.Caller` would resolve to a temp build path on some toolchain configurations.
- **`WithWorktreeAuthenticated`, not `WithWorktree`**: the spike's purpose is to capture the protocol when claude actually reaches the model. Without a real API key, `claude` emits a synthetic `assistant` envelope with `"text":"Not logged in · Please run /login"`, `"apiKeySource":"none"`, exit 1, ~400 ms — zero `tool_use`, zero permission events. Confirmed by an early local run during development. `WithWorktreeAuthenticated` is the right gate for any realclaude test that needs more than an argv-shape probe.

## Findings (cross-link)

[`features/permission-protocol-spike.md`](../features/permission-protocol-spike.md) has the full writeup. The headline:

- **No permission-gate event was emitted on stdout in any of the six `--permission-mode` values** (`default`, `acceptEdits`, `auto`, `plan`, `dontAsk`, `bypassPermissions`).
- **`--allowed-tools Read` did not prevent Bash invocation** in any mode under this argv. The allowlist is not enforced when `--permission-prompt-tool stdio` is in effect and `--dangerously-skip-permissions` is absent.
- The most likely interpretation is that `--permission-prompt-tool <tool>` expects an MCP tool name (e.g. `mcp__permissions__approve`), and the literal `stdio` doesn't resolve, so claude silently no-ops the gate.
- **Per AC, no follow-up issue is filed** — the "filed only if a real event fires" branch is not triggered.

## Patterns established

- **Spike tests live next to behavior tests, with one captured artefact per dimension swept.** The spike-runner's discipline is "edit one constant, rerun, rename the canonical fixture per dimension before the next overwrite." Resist embedding the matrix in code (a `for mode := range modes` loop) — the cost is per-call to a real API, the budget is a discrete operator decision per mode, and the loop hides the rename discipline that's load-bearing for diffing across reruns.
- **Captured-trace tests pass on any non-structural outcome.** The test's only assertion is "the run produced something worth capturing." Any behavioral signal (crash, hang, non-zero exit, deadline trip) gets recorded into the fixture and passes. Only "produced nothing and we hit the deadline" fails. This is the right shape for tests whose value is the on-disk artefact, not a green-or-red gate.
- **Stdin envelope shape question is answered by trying the production shape first.** The architect's spec listed two candidate input shapes (production vs. simpler fallback) as an open question. The production shape worked on the first try; the fallback wasn't tested. Generalisable rule for stream-protocol probes: start with the shape known to work in production, fall back to spec inferences only if startup-time stderr surfaces a parse error.
- **Captured fixtures embed real environment-leaky context.** The `tool_result` content in the default-mode fixture contains the macOS user's POSIX name (`drwxr-xr-x@ 4 juhanailmoniemi  staff  ...`). That's not a defect — the spike runs Bash against the worktree and Bash's output reflects the runner's environment. Reviewers diffing fixtures across spike-runners on different machines should expect machine-local variance in the `tool_result` payloads; the load-bearing data is the *event shape*, not the payloads.

## Lessons learned

- **The `init` envelope's `tools` array is the full tool registry, not the `--allowed-tools` allowlist.** A reader inspecting the init envelope and seeing `Bash` listed alongside `Read` might conclude the allowlist isn't being honored. The allowlist is enforced (when it's enforced at all) at tool-use time, not surfaced in the bootstrap event. If a future ticket needs to verify the boundary's allowlist behavior from the init envelope, that won't work — verify from the `result` trailer (`permission_denials`) or from the `tool_use` events themselves.
- **`--permission-mode auto` is a synonym for `default`.** Claude's `--help` lists both as separate choices, but the `init` envelope echoes `permissionMode: "default"` when launched with `--permission-mode auto`. The pyrycode-mobile design should treat them as the same mode.
- **`--permission-prompt-tool stdio` does not behave like an undocumented flag with documented semantics — it behaves like a no-op.** The architect's spec assumed the flag was the VS Code extension's permission channel; the captured traces show no events on stdout under any mode. Either the VS Code extension passes additional flags / registers an MCP tool we didn't replicate, or the protocol is on a side channel (FD 3+, IPC). The mobile design cannot rely on this flag in isolation.
- **OAuth-token-as-`ANTHROPIC_API_KEY` works for real API calls.** During development the macOS keychain held a Claude Code OAuth token (`sk-ant-oat01-...`); exporting that token as `ANTHROPIC_API_KEY` produced a fully-authenticated probe (apiKeySource: `ANTHROPIC_API_KEY`, real model output). The Anthropic API accepts the OAuth token in the same header as a long-lived API key. This isn't load-bearing for the spike, but it matters if a contributor without a personal API key needs to reproduce a spike locally — the keychain token is a viable source.

## Out of scope (deferred)

- **Assertion-based permission-protocol test.** Per AC, this is filed as a follow-up only if a real permission event fires. None fired, so no follow-up. If a future spike (registered MCP tool as the prompt-tool, or a side-channel capture) surfaces real events, that's when the assertion-based test gets scoped.
- **Side-channel capture (FD 3+, Unix-domain socket, IPC).** Not in the spec, not in the AC. The current spike's read surface is `cmd.Stdout` only. Extending it would require redesigning the I/O contract and is out of scope here.
- **`docs/knowledge/INDEX.md` update.** Per `docs/PROJECT-MEMORY.md` and the developer CLAUDE.md guard, INDEX.md is the documentation phase's exclusive write surface. The new `features/permission-protocol-spike.md` entry is left for the docs phase to append.
