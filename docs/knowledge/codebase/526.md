# Ticket #526 — selfcheck wires `canonicalAllow` into `ptyrunner.Config`

`pyry agent-run --self-check` had been DOA on every post-#470 build: the synthetic init envelope added by commit `058c569` ("feat(streamjson): synthesise leading system/init envelope from runner inputs") tightened `ptyrunner.Config` to require a non-nil `AllowedTools` (runner.go:245), but the selfcheck's `ptyrunner.Config` struct literal at `selfcheck.go:196-215` was never updated, so the install-time canary crashed with `pyry: agentrun: self-check: ptyrunner: AllowedTools required` before claude was ever spawned. Production dispatcher path (`runAgentRunPty` in `cmd/pyry/agent_run.go`) was unaffected — it builds Config from `parsed.allowedTools`, which `parseAgentRunArgs` requires. One-line fix in the selfcheck literal, plus a regression test that pins the call-site contract.

## Implementation

- **Production change is one struct-literal field.** `internal/agentrun/selfcheck/selfcheck.go:201` adds `AllowedTools: canonicalAllow,` adjacent to `SettingsPath` inside the `errgroup` goroutine that calls `ptyRun`. Placement is deliberate: `SettingsPath` is the only other Config field that carries the deny-default allow-list (via the per-spawn settings file written by `settingsWrite(canonicalAllow)` at selfcheck.go:172), so keeping the two visually paired makes the coupling obvious to future readers. The pre-existing `canonicalAllow = []string{"Read"}` constant at `selfcheck.go:77` is reused — its doc comment explicitly forbids parallel literals because the spawn's allow-list and the per-spawn settings file MUST agree byte-for-byte for the deny-default invariant to hold. Net diff: +1 line in `selfcheck.go`.

- **Regression test pins the call-site contract.** `internal/agentrun/selfcheck/selfcheck_test.go` adds `TestSelfCheck_PassesCanonicalAllowToPtyRunner` (+35 LOC, +1 `reflect` import). The test installs the seam, replaces `ptyRun` with a closure that captures `cfg.AllowedTools` into a test-local `observedAllow []string`, writes `passLine + "\n"` to `cfg.Stdout` so the watcher sees end-of-turn and the call returns PASS (mirrors `TestSelfCheck_Pass` at selfcheck_test.go:71-102), then asserts `observedAllow != nil` (separate from the equality check — guards against a future regression where someone passes `[]string{}` "to be safe", which would silently break the deny-default invariant without tripping runner.go:245) and `reflect.DeepEqual(observedAllow, canonicalAllow)`. The non-nil assertion is the load-bearing one: runner.go:245's actual failure mode is the nil check, and an empty-slice "fix" would mask the regression.

- **No runner-side change.** The `cfg.AllowedTools == nil` required-field check at `internal/agentrun/ptyrunner/runner.go:245-246` is correct as written — the bug was in the caller. Tightening or relaxing that check was explicitly out of scope.

- **No mock-seam restructure.** The existing `var ptyRun = ptyrunner.Run` seam at `selfcheck.go:62` is unchanged. The architect's spec considered injecting the real `runner.go` required-field validation into the mock by default (to catch this exact silent-drift pattern globally) and explicitly out-of-scoped it as a structural change affecting nine sibling tests. The new test exercises the contract at one specific call site rather than at the harness layer.

## Patterns established

- **Test-seam mocks that accept any Config silently un-enforce required-field contracts.** This bug's blast radius (the install-time canary was unusable on every post-#470 build for the ~4-day window between the `058c569` tightening and #526) is a direct consequence of the mock seam (`ptyRun = func(ctx, cfg) error { ... }` returning `(Result{...}, nil)` regardless of Config) never exercising the real `runner.go:245` nil-check. The fix shape — a dedicated regression test that captures the Config field and asserts the call-site contract — is the right XS pattern for the same shape elsewhere. **Generalises: when a package mocks an external function via a package-level seam, any field that becomes required downstream needs an explicit call-site capture-test in the consumer package, not just the producer package's nil-check.** The producer's contract test verifies "the runner rejects bad Configs"; the consumer's capture test verifies "the consumer never builds a bad Config". Both are needed; only the latter catches silent drift in the consumer.

- **Reuse the named constant; never re-literal a coupled invariant.** The `canonicalAllow` constant doc-comment explicitly says "coupling these values prevents a future caller from breaking the invariant by widening Config.AllowedTools" — i.e. the constant is *named* (not just hoisted) precisely so its identity threads through the deny-default contract. The fix passes `canonicalAllow` (not `[]string{"Read"}`) into the Config literal because a parallel literal would let one half of the pair drift while the other stayed pinned. **Generalises: when a constant's doc-comment names an invariant it carries, any new use-site should pass the constant by name even if the literal value is small — the cost is zero LOC, the regression-prevention value is one bug-class.**

## Lessons learned

- **The install-time canary masked a deeper bug for ~4 days.** The PR's manual smoke (AC3) revealed that even after the fix, `pyry agent-run --self-check` does NOT print PASS against `claude 2.1.150` — it correctly detects `Bash` being invoked despite the deny-default settings file and prints `FAIL — deny-default whitelist did NOT enforce`. That is the install-time canary doing its job (the regression it was designed to catch is real), but it was invisible from the #470 cutover until #526 because the `AllowedTools required` crash stopped execution before the runtime detector could fire. Filed as **#528** (separate ticket, likely in `internal/agentrun/settings/settings.go` JSON shape or `internal/agentrun/ptyrunner/runner.go` argv pair). **Generalises: when a startup-phase check is broken, every later-phase check is also untested by extension — the visible bug's blast radius understates the latent-bug surface that's been hiding behind it.** Future maintainers reading a "fixed the obvious thing, but the next layer surfaced X" PR description should treat X as a strong signal that other later-phase checks may also be quietly broken downstream of any earlier-phase break.

- **AC3 of the form "exits 0 and prints PASS" presumes the only obstacle is the change at hand.** The architect's spec wrote AC3 as "exits 0 and prints the existing PASS message" — true in a counterfactual world where the canary's only obstacle was the missing field. In reality, the missing-field crash had been masking a downstream regression. Per scope discipline (developer CLAUDE.md), the developer stopped at the discovery and filed #528 instead of expanding #526 to fix it. The PR's "Lessons learned" section captured the meta-pattern verbatim and is the immediate source for this section. **Generalises: when a spec's AC includes "manual smoke against real X" and the manual smoke surfaces a separate failure mode not caused by the change at hand, the right move is to file the follow-up and document the discovery in the PR, not to expand scope or call the PR blocked.** Same shape as the [#513](513.md) NIT-deferral pattern (in-place fix is cosmetic, next contributor picks it up at zero cost) but at the AC-completion boundary rather than the doc-freshness boundary.
