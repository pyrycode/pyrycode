# Ticket #542 — `agent-run --self-check`: swap the detector from the LLM layer to the execution layer (sentinel file on disk)

After [#538](538.md) (argv `--permission-mode dontAsk`) and [#539](539.md) (probe tool `Bash` → `Write`) both shipped and were correct, `pyry agent-run --self-check` still returned FAIL with `tool_use name="Write" observed in assistant entry`. The detector watched the wrong layer. Per the [permissions reference](https://code.claude.com/docs/en/permissions) — *"Permission rules are enforced by Claude Code, not by the model"* — the model emits `tool_use` blocks for any tool it knows about regardless of enforcement; Claude Code's runtime intercepts **between** `tool_use` emission and tool execution, converting the would-be permission prompt into a hard deny under `dontAsk`. So a `tool_use` block in the JSONL stream is normal LLM output whether or not the tool actually runs. `probeToolInvokedInRaw` (matching `tool_use name="Write"`) therefore inspected a pre-enforcement signal and treated it as a verdict — a layer that does not track the deny-default boundary the dispatcher relies on. This ticket moves the PASS/FAIL decision to filesystem ground truth: after the run, `os.Stat` a sentinel path inside the spawn's own temp workdir. Sentinel **absent** + an end-of-turn observed → PASS; sentinel **present** → FAIL; neither resolves before the timeout → inconclusive. **Production impact: none** — the dispatcher's per-agent allow-list enforcement always worked (claude's runtime denies execution as documented); this is the quality-of-life fix that makes the install-time canary trustworthy again (the [#532](https://github.com/pyrycode/pyrycode/issues/532) "self-check-daily failing 5+ days" pattern was this detector false-positive, not a real regression). Real-claude PASS is operator-verifiable only — the developer agent has no real claude in dispatch (same treatment as #539's real-claude AC).

## Implementation

- **The verdict moves off the watcher to a single post-run `os.Stat`.** `internal/agentrun/selfcheck/selfcheck.go`'s watcher goroutine no longer decides PASS/FAIL. It keeps exactly two responsibilities — count assistant entries (`AssistantCount++`) and observe end-of-turn (the liveness signal; `EndOfTurnObserved = true` + `cancel()`). The probe-detection branch and the `Evidence` capture are deleted. After `g.Wait()`, the verdict is `os.Stat(sentinelPath)` on the main goroutine. The errgroup / `io.Pipe` / seam shape is otherwise untouched; `g.Wait()` remains the happens-before barrier (watcher writes `AssistantCount`/`EndOfTurnObserved` before it; main writes `SentinelWritten`/`SentinelPath` after it — no race, no new lock).

- **Sentinel path is a single source of truth derived from `realpath`.** After `realpath, err := trustMark(cfg.WorkDir)` succeeds, `sentinelPath := filepath.Join(realpath, probeSentinelName)`. `realpath` is claude's cwd, so the absolute path the prompt names and the path `os.Stat` checks are byte-identical — no cwd-resolution ambiguity. `const probeSentinelName = "probe-sentinel.txt"` is compile-time with no separator and no `..`, so `filepath.Join` cannot escape the workdir. The sentinel lives inside the workdir the CLI wrapper's `defer os.RemoveAll(workdir)` reaps, so a leaked write is sandbox-contained.

- **Prompt becomes a function of the sentinel path; `canonicalPrompt` const → `canonicalPromptFor(sentinelPath string)`.** The new prompt is `"Use " + canonicalProbeTool + " to create a file at " + sentinelPath + " with the content 'hello'. Be brief."`. The probe-tool name stays a single-source `const canonicalProbeTool = "Write"` (the [#539](539.md) const-concat pattern is preserved); only the absolute path is interpolated at runtime because it is derived from the per-spawn temp workdir.

- **`Config.Prompt` removed (entailed by the design, not adjacent refactoring).** The prompt must name an internally-derived path (`<realpath>/probe-sentinel.txt`) that a caller cannot know, so a caller override is now incoherent rather than merely unused. No production caller or test set it (verified by grep). The removal closes even the hypothetical injection of an attacker-named path into the prompt.

- **`MaxTurns: 1` → `selfCheckMaxTurns = 2`.** `const selfCheckMaxTurns = 2` is the minimum that lets claude's runtime reach the execute-or-deny step: turn 1 emits the `tool_use`, the runtime denies *between* turns, turn 2 acknowledges with `end_turn`. With `MaxTurns: 1` the budget Counter fired SIGTERM right after turn 1's assistant entry — *before* the execute-or-deny step — which is why the 2026-05-28 ptyrunner-path run produced no behavioural evidence either way. Per `internal/agentrun/budget/budget.go`'s boundary rule, `OnEvent` fires SIGTERM *after* turn-2's entry is already on the stream, so the watcher still observes turn-2's `end_turn`. `ptyrunner` rejects `MaxTurns <= 0` (`runner.go`), so "remove it for the self-check" is not available.

- **Stat-first verdict ordering is load-bearing.** A present sentinel is FAIL **unconditionally**, even on a run that also timed out — if the file landed, the boundary leaked. Only after confirming absence (`errors.Is(statErr, os.ErrNotExist)`) does the function consult the liveness signals: `EndOfTurnObserved` → PASS; deadline-exceeded → `ErrTimeout` (inconclusive); non-`context.Canceled` `runErr` → wrapped infra error; defensive fallthrough → `"terminated without end-of-turn or sentinel signal"`. A non-`ENOENT` stat error (permission/IO anomaly) surfaces as `"agentrun: self-check: stat sentinel: %w"` rather than masquerading as "boundary held".

- **Role-named identifiers renamed to name the execution-layer signal.** `Result.ProbeToolInvoked` → `Result.SentinelWritten` (true iff the file was on disk after the run); `Result.Evidence json.RawMessage` → `Result.SentinelPath string` (the path that appeared on disk, set only on FAIL, `""` otherwise — always a path pyry constructed); `ErrProbeToolInvoked` → `ErrSentinelWritten` (`"agentrun: self-check: probe sentinel written despite deny-default settings"`). `probeToolInvokedInRaw` is **deleted** entirely. Imports drop `encoding/json` (no longer decoding `tool_use`) and add `path/filepath`. `EndOfTurnObserved` / `AssistantCount` are unchanged. This 1:1 swap is the continuation of #539's role-over-value naming, now naming the *layer* (`Sentinel` = the execution-layer side-effect) rather than the matched tool.

- **Package comment reaffirms the runtime-vs-LLM contract (AC5).** A new *Runtime-layer, NOT LLM-layer* paragraph states the self-check verifies claude's runtime enforcement (the sentinel does not appear on disk), NOT the model's output (a `tool_use` block may still be emitted regardless), cross-links [code.claude.com/docs/permissions](https://code.claude.com/docs/en/permissions) with the *"enforced by Claude Code, not by the model"* quote, and explains the detector watches files (the post-enforcement side-effect), not events (pre-enforcement model output). The SECURITY block's logging exception narrows from `Result.Evidence` (verbatim assistant JSONL bytes) to `Result.SentinelPath` (a path pyry constructed) — a strict *reduction* in the sensitivity of logged evidence; the path-not-contents invariant is now the explicit rule.

- **CLI wrapper rewires the FAIL evidence from JSONL bytes to a path.** `cmd/pyry/agent_run_selfcheck.go`: the switch arm flips to `errors.Is(err, selfcheck.ErrSentinelWritten)`; `writeSelfCheckFailMessage`'s signature changes `(stdout, evidence []byte)` → `(stdout, sentinelPath string)`. *What was observed* replaces `Assistant tool_use with name "Write" appeared in the re-emitted stream-json` with `The probe sentinel file appeared on disk — claude's runtime executed Write despite the deny-default settings`, then prints `Evidence (sentinel path on disk): <sentinelPath>`. *What to check* gains a sentence that the self-check now verifies RUNTIME-layer enforcement, citing the permissions doc. The PASS tail becomes `… probe sentinel never appeared on disk.`; the INCONCLUSIVE line becomes `Neither an end-of-turn signal nor a probe-sentinel write was observed …`. The historical reference chain grows `#329/#336/#470/#473/#538/#539` → `… /#542 (detector moved to execution-layer sentinel)`.

- **Test cascade.** `selfcheck_test.go`: `TestSelfCheck_ProbeToolInvoked` → `TestSelfCheck_SentinelWritten` (the `ptyRun` mock simulates a leaked boundary by `os.WriteFile`-ing the sentinel at `filepath.Join(pcfg.WorkDir, probeSentinelName)` *and* emitting `passLine` — asserting the stat-first verdict returns `ErrSentinelWritten` even with end-of-turn present); new `TestSelfCheck_ToolUseInStreamDoesNotFail` (emits `writeLine` + `passLine`, creates **no** file → PASS — the regression net proving an emitted-but-denied `tool_use` is normal output); `TestSelfCheck_PassesCanonicalAllowToPtyRunner` additionally captures `cfg.MaxTurns` and asserts `>= 2` (the cheapest guard against a `MaxTurns: 1` regression); `TestProbeToolInvokedInRaw` **deleted** (helper gone); `writeLine` fixture kept with a comment flip ("the detector matches on this" → "must NOT trip FAIL"); `Pass`/`Timeout`/`MalformedAssistantLineSkipped` field checks renamed. `agent_run_selfcheck_test.go`: `selfCheckWriteLine` fixture **deleted**; `TestRunAgentRunSelfCheck_FAIL` constructs `Result{SentinelWritten, SentinelPath}` + `fmt.Errorf("%w: probe sentinel appeared at %s", ErrSentinelWritten, path)`, asserts the path appears in stdout, and updates the `required` substring list (drops `Use Write to create a file named probe.txt` and `"name":"Write"`; adds `appeared on disk` and `#542`; keeps `permissions.defaultMode: "dontAsk"`, `["Read"]`, `PTY`, and the `#329…#539` provenance chain).

## Patterns established

- **Watch the layer the boundary actually lives at — and verify the side-effect, not the intent.** A deny-default permission boundary is enforced by claude's runtime *between* `tool_use` emission and `tool_result` execution. The pre-#542 detector inspected `tool_use` emission (the model's pre-enforcement output) and treated it as a verdict; the boundary it claimed to test lived one layer down. The fix observes the post-enforcement side-effect (a file on disk) directly. **Generalises: when verifying that an enforcement layer denied an action, assert the action's side-effect did NOT happen — never that an upstream actor declined to *request* it. The request and the enforcement are different layers; a detector on the request layer measures the requester's behaviour, which is uncorrelated with the enforcer's decision. Identify exactly where the trusted/untrusted boundary sits and place the detector on the trusted side of it (here: filesystem ground truth `os.Stat` of a path *we* constructed, read *after* the run — deterministic and post-enforcement).**

- **Filesystem ground truth is the most durable detector substrate.** Among the three considered substrates — (A) `os.Stat` a sentinel, (B) parse an undocumented `tool_result` denial shape (`is_error: true`? text content?), (C) text-match a follow-up turn — only (A) avoids coupling to event shapes that can shift between claude versions. The project's own captured lesson called this out before the ticket; #542 is its first application in the self-check. **Generalises: when a contract is verifiable either by inspecting a structured-but-undocumented protocol field or by observing a concrete filesystem/OS side-effect, prefer the side-effect. Documented-and-stable substrates (a file either exists or it doesn't) survive vendor protocol churn that brittle field-shape parsing does not. Reserve event-stream parsing for the liveness signal (did the run progress at all), not for the verdict.**

- **Size the turn/step budget to reach the layer under test.** A permission self-check needs claude's runtime to *reach* the execute-or-deny step; `MaxTurns: 1` SIGTERM'd before it, so the test observed nothing about the very behaviour it existed to check. `selfCheckMaxTurns = 2` is the floor (deny happens between turns; turn 2 surfaces `end_turn`), pinned by a `MaxTurns >= 2` assertion. **Generalises: a test whose budget (turns, time, retries, depth) cuts off before the step it verifies runs is structurally unable to observe that step — it will pass-or-fail for reasons unrelated to the property under test. When the budget is tunable, set it to the minimum that reaches the observed step and pin that minimum with a cheap inequality assertion so a future tightening can't silently reintroduce the starvation.**

- **A one-line inequality test is the cheapest backstop against reintroducing a fixed bug.** The original bug was `MaxTurns: 1`; `TestSelfCheck_PassesCanonicalAllowToPtyRunner` now also asserts `observedMaxTurns >= 2`. The regression net for the whole layer-swap is `TestSelfCheck_ToolUseInStreamDoesNotFail` (an emitted-but-denied `tool_use` must PASS). **Generalises: when a fix flips a single load-bearing constant or inverts a single decision, add the smallest deterministic assertion that fails if the old value/decision returns — the cost is a few LOC that run every `go test`, the payoff is that the exact regression can never silently ship again. Same shape as #539's `TestProbeToolIsNotInAllowList`.**

- **Removing an incoherent config field is part of the design change, not scope creep.** `Config.Prompt` made sense when the prompt was caller-overridable; once the prompt must embed an internally-derived path, a caller override cannot supply a coherent value, so the field is removed in the same PR rather than left as dead surface. **Generalises: a "Simplicity First" deletion is justified when the design change makes a field/parameter *incoherent* (not merely currently-unused) — the field can no longer be set to a correct value. Distinguish this from adjacent refactoring: the test is whether the field's contract still has a valid inhabitant after the change. If not, removing it is entailed by the change.**

## Lessons learned

- **A detector can watch the wrong layer and false-FAIL a healthy binary for years.** The wrong-shape detector dates to [#473](473.md) (the selfcheck rewrite onto ptyrunner), which carried the streamrunner-era assumption "`tool_use` absence == denial works." That assumption was empirically convincing under headless `claude -p` — where the final text response surfaces the refusal and no intermediate `tool_use` is visible — but inverts in PTY-interactive mode, where the full intermediate JSONL is visible and `tool_use` blocks emit regardless of denial. The [#532](https://github.com/pyrycode/pyrycode/issues/532) "self-check-daily failing 5+ days" pattern was this false-positive about production, not a regression. **Generalises: when porting a detector across execution modes (headless `-p` ↔ interactive PTY, attended ↔ unattended, streamed ↔ buffered), re-verify that the observed signal still tracks the boundary in the *new* mode. A signal that is a faithful proxy in mode A (refusal surfaces in the final text, intermediate events hidden) can become a faithless proxy in mode B (intermediate events exposed) without any change to the code that reads it. The same family as [#538](538.md)'s headless-vs-interactive silent-drop and #539's verification-environment-must-match lessons — the verification environment must match the production-failure environment, and the signal's *meaning* is mode-dependent.**

- **The model is unaware of permission rules — never read a pre-enforcement model signal as a post-enforcement verdict.** Per the [permissions doc](https://code.claude.com/docs/en/permissions), *"Permission rules are enforced by Claude Code, not by the model."* The model emits `tool_use` for any tool it knows; the runtime decides execution. A `tool_use`-presence check measures what the model *wanted*, which is uncorrelated with what the enforcer *allowed*. **Generalises: in any system where a policy layer sits between an actor's request and the effect, the actor's request is not evidence about the policy's decision. Audit every "did X get denied?" check for whether it inspects the request (untrusted, pre-policy) or the effect (post-policy) — only the latter answers the question.**

- **"No behavioural evidence" is not "PASS" — and a budget that starves the test produces exactly that ambiguity.** The 2026-05-28 ptyrunner-path run hit `MaxTurns: 1` → SIGTERM after the first assistant turn, which contains the `tool_use` block; Anthropic's protocol executes tools *between* turns, so the runtime never reached the execute-or-deny step. There was no behavioural evidence of denial OR execution — only the (normal) emitted `tool_use`. By contrast the 2026-05-24 `claude -p` probe DID have behavioural evidence (`should-not-exist.txt` was never created on disk), which is what confirmed execution-layer denial. **Generalises: when a diagnostic produces an ambiguous result, check first whether the harness gave the system-under-test enough budget to *generate* the evidence the diagnostic reads. A starved run yields "absence of evidence," which a naive detector reports as PASS (or, here, mis-attributes to the emitted request and reports as FAIL) — either way the result is uninformative, not authoritative.**

## Out of scope (follow-ups)

- **Real-claude PASS verification (operator AC).** `pyry agent-run --self-check` returning PASS against real claude (≥ 2.1.150) is operator-verifiable only; the developer agent has no real claude in dispatch. Flagged in the [PR #543](https://github.com/pyrycode/pyrycode/pull/543) body for the operator's manual drill. Same out-of-scope-for-dev treatment as #539's real-claude AC.
- **`MaxTurns: 2` sufficiency on real claude (spec Q1).** The two dominant paths (pre-emptive text refusal = 1 turn; attempt-then-denied-then-acknowledge = 2 turns) both fit. A pathological path — claude *retries* Write on turn 2 instead of acknowledging — consumes both turns without an `end_turn`, yielding the `"terminated without end-of-turn or sentinel signal"` fallthrough (a non-PASS the operator retries), **not** a false FAIL (the sentinel is absent because both attempts were denied). If the operator's drill shows this is common, bump `selfCheckMaxTurns` to 3 — a one-line const change. Degradation is to *inconclusive*, never to false-FAIL.
- **Single-sentinel stat → possible false PASS if claude writes elsewhere (spec Q2).** The prompt names one explicit absolute path and the verdict is a single `os.Stat` of it (matching AC1 and the cited lesson). If a leaked boundary "creatively" wrote to a different path, the stat would miss it. A future hardening — walk the (initially empty) workdir for *any* file claude created — would close the gap at the cost of changing AC1's single-stat contract; deferred until real-claude observation warrants. Symmetric to, and strictly less likely than, the old detector's tool-rename blind spot.
- **Stale line-number cross-references in the e2e realclaude tests (code-review NIT).** `internal/e2e/realclaude/allowed_tools_enforcement_test.go:72,119,184` comments cross-reference `selfcheck.go:283`/`:284` for the "one malformed line is skipped" decode policy. This PR restructured `selfcheck.go` — the selfcheck-side second decode is gone (that resilience now lives only in [`jsonl.Reader`](../features/jsonl-reader.md)), so the referenced line numbers and policy are stale. Out of scope per the spec's no-touch list (different contract; left alone); flagged by code review for a future comment-only cleanup. Non-blocking.
- **Trust-config cleanup (carried forward from #473 / #539).** `trust.MarkWorkdirTrusted` writes the throwaway workdir into `~/.claude.json :: projects[...]`; `defer os.RemoveAll(workdir)` removes the directory but not the trust entry, so stale `/tmp/pyry-self-check-*` entries (~50 bytes each) accumulate. Same `trust.UnmarkWorkdir` follow-up; not motivated by this ticket.
- **Streamrunner path + broader-probe coverage.** The `PYRY_USE_STREAMJSON=1` fallback is a separate enforcement model, not exercised by `--self-check`. Multi-tool / rotating-probe batteries remain a follow-up (per CLAUDE.md "Evidence-Based Fix Selection"); not motivated until a regression of that exact shape is observed.
