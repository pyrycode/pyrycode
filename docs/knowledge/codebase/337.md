# Ticket #337 — `pyry agent-run` subcommand skeleton + flag parsing

Phase B-1 of #329's spike: lands the `agent-run` verb scaffold with the full flag surface but no behaviour. The dispatcher's eventual replacement for `claude -p` needs a stable, tested argument-parsing entry point so the three behaviour-adding split-offs from #331 (trust-merge, settings-file, spawn) review against fixed-shape `agentRunArgs` rather than re-litigating the surface each time.

## Implementation

- `cmd/pyry/agent_run.go` (new, ~180 LOC) — `agentRunArgs` unexported struct, `parseAgentRunArgs(args) (agentRunArgs, error)` doing parse → validate → return, `splitAllowedTools(raw) []string` pure tokeniser (`strings.FieldsFunc` over `r == ',' || unicode.IsSpace(r)` + trim + drop empties), `validEfforts` package-level `map[string]bool` for the `--effort` enum, `requireRegularFile` / `requireDir` helpers, and a stub `runAgentRun` that consumes the parsed struct as `_ = parsed` and prints one confirmation line on stdout. See [pyry-agent-run-command.md](../features/pyry-agent-run-command.md) for the full feature surface.
- `cmd/pyry/main.go` — three edits: `case "agent-run": return runAgentRun(os.Args[2:])` in the top-level dispatch switch (placed after `update`, matching the daemon-free verb cluster), one help-text entry in `printHelp()`, one bullet in the package-doc verb list.
- `cmd/pyry/agent_run_test.go` (new) — table-driven, `t.TempDir()`-backed `validArgsFixture` builds the happy-path argv with real on-disk prompt-file / system-prompt-file / workdir; `argvWithout(flag)` / `argvReplacing(flag, value)` helpers let each error row override a single flag and reuse the rest. Tests target `parseAgentRunArgs` directly (not `runAgentRun`) so the confirmation-line printout — which siblings will overwrite — is not pinned.

## Patterns established

- **Daemon-free standalone verb shape.** `agent-run` mirrors `runUpdate`'s structure: own `flag.NewFlagSet`, no `parseClientFlags` (does not dial the control socket), returns errors that `main()` wraps as `pyry: <err>` and exits non-zero. Verbs in this cluster do not accept the global `-pyry-*` flags; if a sibling needs one (e.g. `-pyry-name` for per-instance logging), add it then.
- **One-line errors that name the offending flag.** All validation failures wrap as `fmt.Errorf("agent-run: --<flag>: <reason>")` so the top-level prefix renders as `pyry: agent-run: --<flag>: <reason>`. Stat-style underlying errors (`ENOENT`, `EACCES`) flow through via `%w`; tests assert only the flag-name substring, leaving the OS-error text free to vary across platforms.
- **Frozen field names for downstream consumers.** `agentRunArgs`'s field names (`promptFile`, `systemPromptFile`, `allowedTools`, `maxTurns`, `effort`, `model`, `workdir`, `outputFormat`) are the contract the trust-merge / settings-file / spawn siblings read against. Renames in those tickets are out of scope; cleanup tickets land separately so a behaviour slice doesn't double as a surface migration.
- **Tokeniser as a pure, separately-tested helper.** `splitAllowedTools` has its own focused test independent of the full parse path (`empty → []`, `,,Read,, → ["Read"]`, plus the three documented shapes). The full-parse path exercises the comma / space / mixed forms again through `--allowed-tools` to pin the end-to-end behaviour. Pure functions earn their own unit table even when a higher-level test would technically cover them — keeps the contract debuggable without a tempdir.
- **`flag.ContinueOnError` + custom `fs.Usage`.** `pyry agent-run --help` prints a one-line synopsis and `fs.PrintDefaults()` rather than falling through as a parse error. Optional per the AC but cheap, and matches `pair.go`'s usage line.
