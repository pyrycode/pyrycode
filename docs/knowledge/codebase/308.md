# Ticket #308 — net: auth-gate first frame + WS 4401 close on reject

Plugs `relay.AuthenticateFirstFrame` (#249, idle) into the dispatcher
scaffold (#307, empty handler table) so the very first inbound frame on
every new `conn_id` is validated against the local device registry, and
extends the binary↔relay wire so a reject can atomically deliver the
`auth.invalid_token` error envelope plus a WS `4401` close. Before this
ticket, no code consumed the predicate and an unpaired phone could send
arbitrary frames against the empty handler table (they got
`protocol.unsupported`, but the binary never closed the WS).

## Implementation

### Wire-protocol additions (`internal/protocol/envelope.go`)

Two optional fields on `protocol.RoutingEnvelope`, both `omitempty` so
every pre-#308 fixture and on-wire byte sequence is byte-identical:

- **`Token string`** — phone→binary direction only. Populated by the
  relay from the phone's `x-pyrycode-token` HTTP header at WS upgrade,
  on the **first** frame for a given `conn_id` only; empty on every
  subsequent frame and on every binary→phone frame. The binary's
  `AuthenticateFirstFrame` is the only consumer.
- **`CloseCode uint16`** — binary→relay direction only. When non-zero,
  asks the relay to forward `Frame` (if non-empty) to the phone and then
  close that phone's WS with this WS close code. The dispatcher ignores
  `CloseCode` on inbound (phone→relay→binary) frames: a malicious relay
  cannot induce a self-close that way.

Both fields carry security-load-bearing doc-comments. `Token` flags that
no code at any layer may log it; `CloseCode` flags the direction
asymmetry. `docs/protocol-mobile.md` § Routing envelope grows two short
paragraphs covering both fields with the same direction restrictions.

### Dispatcher first-frame gate (`internal/dispatch/dispatch.go`)

`Config` grows one optional field plus its two types:

```go
type Config struct {
    Frames         <-chan protocol.RoutingEnvelope
    OutboundBuffer int
    Logger         *slog.Logger
    FirstFrame     FirstFrameGate           // optional; nil disables gating
}

type FirstFrameGate func(ctx context.Context, env protocol.RoutingEnvelope) FirstFrameOutcome

type FirstFrameOutcome struct {
    Response  protocol.RoutingEnvelope
    CloseConn bool
    Code      uint16   // WS close code, required when CloseConn is true
    Err       error    // non-nil → dispatcher emits protocol.malformed, no Response
}
```

`connState` gains two flags, both serialised through `d.mu` only when
they cross goroutines:

- `gateRun bool` — single-writer (per-conn goroutine). The for-loop
  checks it on every iteration; the first iteration sets it to `true`
  before invoking the gate, so a malformed first frame consumed by the
  gate cannot loop back through the gate forever.
- `closed bool` — written by the per-conn goroutine immediately before
  it returns on a `CloseConn` verdict; read by the demux under `d.mu`
  to drop further frames for that conn (the relay-side WS has been
  asked to close; any straggler is a race the demux can't deliver
  anywhere).

The reject path publishes **one** routing envelope onto `Outbound()`
carrying `Frame=<error>` *and* `CloseCode=4401` together, so the
forwarder's single `conn.Send(env)` is the atomic wire op. There is no
sequence of (send error) then (close conn); both ride one envelope.

### Per-conn id counter advance on accept

When the gate accepts, `AuthenticateFirstFrame` emits the `hello_ack`
envelope with `ID=1`. The dispatcher's accept arm therefore advances
`Conn.NextID()` one tick so the next binary-originated frame on the
same conn gets `ID=2` — the AC #2 invariant. The advance is a discard:
`_ = st.conn.NextID()`. Without it, the first handler reply on this
conn would collide with the gate's outbound `ID=1`.

### Closed-conn drop in the demux (`routeConn`)

`routeConn` is the single critical section that gates per-conn state.
After #308 it returns `(nil, true)` when an entry exists with
`closed == true` — the demux logs at Debug and drops the frame without
attempting a channel send into a dead per-conn goroutine. The flag
flip happens **after** the close-intent envelope is published onto
`Outbound()` and **under `d.mu`**, so the demux's lookup-and-check is
serialised against the per-conn goroutine's exit-and-mark. This is the
MUST-FIX from the architect's security review made structural; without
it, a frame arriving for a just-closed conn would block the demux on
`st.input <- env` with no receiver.

### `Connection.CloseConn` (`internal/relay/connection.go`)

The AC-mandated explicit surface:

```go
func (c *Connection) CloseConn(connID string, code uint16) error
```

Builds a close-only `RoutingEnvelope{ConnID, CloseCode}` (no `Frame`),
marshals, forwards via `transport.Client.Send`. The dispatcher's
reject path does **not** call `CloseConn` — it publishes the
`Frame+CloseCode` envelope onto `Outbound()` and the existing
forwarder handles it. `CloseConn` is the surface reserved for direct
callers that want close-without-payload (none today; the
idle/inactivity sweep hinted at in #307's Open Questions is the
anticipated future consumer).

### Daemon wiring (`cmd/pyry/relay.go`)

`startRelay` grows two inserts between `relay.Connect` and
`dispatch.New`:

1. **Registry load.** `devices.Load(resolveDevicesPath(instanceName))`
   reads `~/.pyry/<sanitized-instance>/devices.json`. ENOENT yields an
   empty registry (every phone rejects until `pyry pair` runs);
   malformed JSON wraps and fails fast as `load device registry: %w`.
2. **Gate closure.** `authGate(registry, string(serverID), logger)`
   returns a `dispatch.FirstFrameGate` that calls
   `relay.AuthenticateFirstFrame(env, env.Token, registry, serverID,
   logger)` and folds the result into `dispatch.FirstFrameOutcome`,
   mapping `outcome.CloseConn` to `Code: uint16(relay.StatusUnauthorized)`
   (4401). The closure NEVER logs `env.Token`.

`dispatch.New` is then called with `FirstFrame: authGate(...)`. The
three-goroutine wiring (dispatcher / outbound forwarder / wait
classifier) and cleanup chain from #307 are unchanged.

### e2e infrastructure additions

**`internal/e2e/internal/fakerelay/fakerelay.go`:**

- `phoneConn` grows `token string`, `firstFrameSent bool`, and a
  per-conn `tokMu sync.Mutex`. The token is captured at upgrade time
  from the `x-pyrycode-token` header; `phoneRecvPump` reads-and-clears
  it under `tokMu` on the first frame for each conn, embedding
  `Token: <captured>` on that one outbound `RoutingEnvelope` and
  leaving it empty on every subsequent frame.
- `binaryRecvPump` honours `env.CloseCode`. Per-conn frames queue a
  `phoneSend{frame, closeCode}` tuple onto `pc.sendCh`; the
  `phoneSendPump` writes `frame` first (preserving "phone sees error
  envelope before close") and then issues `pc.conn.Close(closeCode)`.
  Serialising the close through the same goroutine that writes the
  frame eliminates the race that a direct `conn.Close` at the
  `binaryRecvPump` site would have introduced against an in-flight
  write.

**`internal/e2e/internal/fakephone/fakephone.go`:**

- `Receive`'s error path captures the peer close status via
  `websocket.CloseStatus(err)` (skips `-1` for non-CloseErrors) into
  `lastCloseStatus` under the existing mutex.
- New method `LastCloseStatus() (websocket.StatusCode, bool)` exposes
  the captured value for test assertions.

## Patterns established / reinforced

- **Single-envelope close-with-frame.** Atomicity of "write error then
  close" comes for free when the wire spec carries both intents on one
  envelope. No race window between the error envelope's `conn.Send`
  and a separate `CloseConn` call. The same mechanism extends naturally
  to future "binary disconnect this phone" intents (idle/inactivity
  sweep, force-revoke).
- **Direction-restricted optional wire fields.** `Token` (phone→binary,
  first frame) and `CloseCode` (binary→relay) are both
  direction-asymmetric. The doc-comments on `protocol.RoutingEnvelope`
  state the direction explicitly; the dispatcher enforces by ignoring
  `CloseCode` on inbound frames (test-pinned).
- **Closed-conn drop guards the demux under the same mutex.** The
  per-conn goroutine's exit-and-mark and the demux's lookup-and-check
  share `d.mu`, so a frame arriving for a just-closed conn cannot
  deadlock the demux on a channel send into a dead goroutine. This is
  the structural fix for the architect's MUST-FIX security finding.
- **Gate runs on the per-conn goroutine, not the demux.** A slow gate
  stalls at most one conn; the dispatcher's fan-in is never blocked
  by gate work. `AuthenticateFirstFrame` is in-memory only so "slow
  gate" is hypothetical, but the architectural property matters once a
  future gate touches I/O.
- **`gateRun` consumed on Err.** A malformed first frame burns the
  gate's "first frame" status — the conn doesn't loop back through
  the gate forever. Subsequent frames go through the regular handler
  table with an empty `Token`, where they will reject via the
  handler-table fall-through. Trade-off documented on `FirstFrameOutcome.Err`.

## Lessons learned

### One envelope beats a (send, close) sequence

The spec considered both shapes for the reject path: (a) dispatcher
publishes the error envelope, then separately calls `CloseConn`; (b)
dispatcher publishes one envelope carrying both. (b) ships because
sequence (a) opens a window where the error envelope is in flight on
the wire and a transport-level disconnect could swallow it before the
close intent ever reaches the forwarder — the phone sees the WS drop
with `1006` (abnormal closure) instead of `4401` (`auth.invalid_token`),
losing the actionable diagnostic. (b) makes "frame + close" a single
forwarder iteration, so either both succeed or both fail (the forwarder
drops the envelope on transport disconnect and the relay's connection
state implicitly closes the phone WS via cascade — same eventual
outcome, no diagnostic ambiguity).

### `gateRun` is single-writer; `closed` needs the mutex

`gateRun` is read and written only on the per-conn goroutine, so
naive struct-field access is race-free under `-race`. `closed` is
written there too — but read by the demux on `routeConn`'s entry —
so it must traverse `d.mu`. Putting both fields on the same struct
makes it tempting to think they share concurrency posture; they don't.
The doc-comment on `connState` calls this out, and the demux's
"check closed under mutex, then send to input" sequence in `routeConn`
+ "exit-and-mark under mutex" in `runConn` pin the invariant.

### `AuthenticateFirstFrame`'s non-hello-first-frame path is intentional

`AuthenticateFirstFrame` decodes `env.Frame` to read `helloID` but
runs `reg.Validate(token)` unconditionally regardless of the inner
envelope's `type`. A phone sending (say) `send_message` as its first
frame on a fresh `conn_id` therefore goes through the same
`auth.invalid_token` reject + 4401 close path as a bad-token hello.
There is no protocol-valid first frame on a fresh conn except
`hello`, so this is the correct posture — pinned by
`TestAuthenticateFirstFrame_*` in #249.

## Security review notes (from the architect's spec)

The architect flagged two SHOULD-FIX items + one MUST-FIX during the
spec's security review; all three are addressed in code:

1. **MUST-FIX**: closed-conn check under `d.mu` immediately before
   sending to `st.input` — implemented in `routeConn` (returns
   `(nil, true)` when `st.closed` is observed; demux drops and logs at
   Debug).
2. **SHOULD-FIX**: dispatcher and gate closure never log
   `RoutingEnvelope.Token` — pinned by doc-comments on
   `RoutingEnvelope.Token`, `Config.Logger`, `FirstFrameGate`, and
   `authGate` in `cmd/pyry/relay.go`. No call site references `Token`
   in any log call.
3. **SHOULD-FIX**: dispatcher ignores `CloseCode` on inbound frames —
   pinned by doc-comment on `Config.FirstFrame` and by
   `TestFirstFrameGate_IgnoresInboundCloseCode`.

## Out of scope (deferred)

- **Registry hot-reload.** `devices.Load` runs once at daemon startup;
  a device revoked via `pyry pair revoke` continues to authenticate
  until daemon restart. The mitigation is the existing
  `pyry pair revoke` + restart workflow; an in-process reload
  primitive is a future hardening pass.
- **Per-`Type` gates / middleware chain.** This slice hard-codes one
  gate per dispatcher (auth). A future slice that needs (say) a
  backfill rate-limiter would generalise the hook to a list; out of
  scope here.
- **Verb handlers.** `list_conversations` (#303), `send_message`
  (#304), `register_push_token` (#305) still register their routes
  in separate slices. Until they land, a successful auth gate is
  followed by every subsequent frame falling through to
  `protocol.unsupported`.
- **Token rotation / replay defence.** A compromised relay can replay
  a captured token from a legitimately-paired phone until that device
  is revoked. The protocol's threat model documents this as accepted
  risk for v1 with `pyry pair revoke` as the mitigation.
