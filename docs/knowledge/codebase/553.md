# Ticket #553 — agent-run: chip-gate prompt re-delivery (hasPastedChip + #227 protection)

Gate the [`ptyrunner`](../features/ptyrunner-package.md) prompt-commit recovery loop so it
re-delivers a prompt **only** when there is positive evidence the paste is still uncommitted —
the `Pasted text` input-box chip. PR #547 (Mode B paste-recovery) re-delivered **unconditionally**
whenever `promptDidCommit` timed out; under a slow MCP cold-start (~25% of runs) neither commit
signal is up inside the window even though the paste *did* commit, so the unconditional re-deliver
fired a destructive `ClearInputLine()` + repaste on an in-flight turn — **regression #227**. The
chip is the discriminator that tells a genuine wedge from a committed-but-slow turn.

This was a **faithful re-implementation from an already-validated design**, not a redesign. The
2026-05-30 work (fix + integration tests, proven clean RED→GREEN) was lost uncommitted on a wiped
local branch; the design survived in the spec + live-probe evidence. The evidence base is **closed**
— the N=60 probe was not re-run.

## Implementation

Two small, additive production changes in `internal/agentrun/ptyrunner/runner.go`; the bulk of the
diff is test surface.

- **`hasPastedChip(snap []byte) bool`** (`runner.go:570`) — pure detector next to `promptDidCommit`:
  `bytes.Contains(tuidriver.StripANSI(snap), []byte("Pasted text"))`. Mirrors `tuidriver.IsThinking`'s
  one-liner shape; total over a possibly-empty snapshot. Required adding `"bytes"` to the import block
  (not previously imported).
- **Gated re-deliver branch** (`runner.go:365`) — replaces the single unconditional
  `logger.Warn("ptyrunner: prompt did not commit; re-delivering", …)` inside the existing retry loop
  (`runner.go:352`). The loop structure, attempt cap (`maxPromptAttempts = 3`), `ClearInputLine` on
  attempt > 1, `WritePrompt`, the `promptDidCommit` check, the `committed` flag, and the post-loop
  `…may wedge` backstop are all **unchanged**. Decision table:

  | `promptDidCommit` | `Pasted text` chip? | action | `committed` | WARN logged |
  | --- | --- | --- | --- | --- |
  | true | — | break | true | (none) |
  | false | **no** | break (committed-but-slow) | **true** | `…input box empty (no pasted-text chip) — assuming committed-but-slow, not re-delivering` |
  | false | **yes** | loop → re-deliver | false | `ptyrunner: prompt uncommitted (pasted-text chip present); re-delivering` |

  The two WARN strings are **test observables — verbatim, not paraphrased.** Setting `committed = true`
  on the no-chip path is evidence-based (no-chip ⟺ committed, 0 counterexamples) and suppresses the
  misleading `…may wedge` backstop warn. **Either break path still falls through to
  `WaitForSessionJSONL`**, so a committed-but-slow turn completes cleanly once its lagging JSONL lands —
  the gate only decides whether to *re-paste*, never whether to *wait*.

- **Discriminator invariant (load-bearing for the tests):** the substring
  `(pasted-text chip present); re-delivering` is unique to the wedge line and absent from the
  committed-but-slow line (which ends `…not re-delivering`). Both messages contain the word
  "re-delivering"; the tests key on the parenthesized marker, never on "re-delivering" alone.

### Evidence base (closed — do NOT re-run)

N=60 live probe on claude 2.1.158 established the discriminator: the `Pasted text` input-box chip is
present ⟺ the paste is uncommitted (wedge & chip = 9/9, wedge & no-chip = 0, no-wedge & chip = 0 —
zero counterexamples; `"Pasted text"` validated as the exact anchor). A false negative from the
detector costs at most one skipped re-delivery (still backstopped by the JSONL wait + watchdog); a
false positive costs one extra non-destructive re-paste on a still-garbled line. Both are bounded by
`maxPromptAttempts`.

## Patterns established

- **Observe loop decisions via the captured `cfg.Logger`, not a fake-claude stderr sentinel.** The two
  integration tests (`TestRun_CommitWedge_ChipPresent_ReDelivers`,
  `TestRun_CommitSlow_NoChip_DoesNotReDeliver`) wire `cfg.Logger` to a `loggerSyncWriter` +
  `slog.NewTextHandler(buf, {Level: LevelWarn})` and assert on `buf.String()` — copying the existing
  `TestRun_MaxTurnsExhaustion_NoBenignWarns` pattern. A fake-claude `os.Stderr` sentinel does **not**
  work here: under the PTY the helper's stderr is not wired into the parent's `cfg.Stderr`, so it
  observes nothing. (This was the first attempt's mistake — see Lessons.)
- **Fall-through-safe fixture timing beats tight timing.** The new fake-claude modes hold the session
  JSONL body back by `commitModeJSONLDelay = 500ms` (`helper_test.go`), which only needs to exceed the
  test's `PromptCommitTimeout = 200ms` so the parent's first commit window elapses with no JSONL and the
  gate is exercised (300 ms margin). Correctness does **not** depend on the wedge committing inside the
  `maxPromptAttempts × PromptCommitTimeout` retry budget — control always falls through to
  `WaitForSessionJSONL`, which picks up the delayed body and completes the run cleanly. That fall-through
  is what keeps the suite `-race -count=5` stable.
- **Shared JSONL-write helper.** `writeSessionJSONLBody(path, body string)` (`helper_test.go`) was
  extracted from the `jsonl` goroutine (`MkdirAll → OpenFile(O_CREATE|O_WRONLY|O_APPEND,0o600) →
  WriteString → Sync → Close`) and is now shared by `jsonl`, `jsonl_exit143`, `commit_wedge_chip`, and
  `commit_slow_nochip`. The no-delay modes were refactored to call it, proving the extraction is
  behaviour-preserving.

## Testing

- **Unit — `TestHasPastedChip`** (6 cases): chip-present plain, no-chip, **ANSI-escaped chip** (anchor
  split by a CSI reset — only matches after `StripANSI`, proving that step is load-bearing), empty, nil,
  and a near-miss substring (`"Paste text"` → false).
- **Integration** — `commit_wedge_chip` (renders `[Pasted text +3 lines] ❯` at idle; the chip carries
  no `✻`, so `IsIdle` still fires) asserts the captured WARN **contains** the wedge marker;
  `commit_slow_nochip` (renders `❯` only) asserts the WARN **contains** the committed-but-slow marker
  **AND does NOT contain** the wedge marker (the #227 protection).
- **RED→GREEN demonstrated:** with the gate disabled (`if false && !hasPastedChip(…)`),
  `TestRun_CommitSlow_NoChip_DoesNotReDeliver` FAILS (committed-but-slow turn falls through to the
  destructive re-deliver); restored, both PASS. A grep for `if false` confirmed no gate-disabling toggle
  remains.
- **Green / race-clean:** `go build ./...` and `go vet ./internal/agentrun/ptyrunner/` clean;
  `go test ./internal/agentrun/ptyrunner/` →
  `ok  github.com/pyrycode/pyrycode/internal/agentrun/ptyrunner  3.321s`.

## Lessons learned

- **A PTY fake-claude's stderr is invisible to the parent — assert on `cfg.Logger`, not a stderr
  sentinel.** The first (lost) attempt tried to observe the re-deliver decision via a fake-claude
  `os.Stderr` write and saw nothing, because the helper's stderr isn't wired into the parent's
  `cfg.Stderr` under the PTY. The decision is the runner's own `logger.Warn`, so the only reliable probe
  is the captured logger. Any future test that needs to observe an in-`Run` branch should follow the
  `loggerSyncWriter` pattern.
- **The destructive failure mode is re-pasting a turn that already committed**, not failing to re-paste
  one that wedged. Skipping a needed re-delivery is recoverable (JSONL wait + watchdog backstop, and the
  dispatcher retries `num_turns=0`); re-pasting an in-flight turn corrupts a live session (#227). The gate
  is deliberately biased to **only act on positive evidence** of an uncommitted paste.
- **The evidence base is closed.** The chip discriminator rests on a one-time N=60 probe (claude 2.1.158);
  the design is fixed. A future maintainer extending this should not re-derive it live — re-validate only
  if claude's input-box chip rendering changes.
