# Ticket #297 — relay roundtrip e2e: appendix-flow happy path

Pure consumer test on top of the daemon-side wire-protocol stack. One
new file (`internal/e2e/relay_roundtrip_test.go`, ~310 LOC including the
one-helper `roundTripTS`), build tag `e2e`, single test function
`TestRelay_Roundtrip_Appendix`. **Zero production-code changes** —
every package the test consumes (`fakerelay`, `fakephone`, `fakeclaude`,
the daemon's `relay.Connect` → `dispatch` → `handlers` chain, the
`Bridge` assistant-turn observer from #311) was already shipped by
sibling slices.

The gap this ticket fills: four sibling e2e tests already pin each
appendix step in isolation, but none of them drives all of them on a
**single** phone connection and asserts the cross-step invariants that
only become observable when the same conn carries a mixed verb set.

| Existing test | Step pinned |
|---|---|
| `TestRelay_Hello` | step 2 — binary `hello` |
| `TestRelay_SendMessage_AckAndPTYDelivery` | steps 3, 5a — phone `hello` + `send_message` + PTY delivery |
| `TestRelay_AssistantTurn_BroadcastsMessageEnvelope` | step 5b — `message` echo |
| `TestRelay_RegisterPushToken_AckAndPersists` | step 6 — `register_push_token` + on-disk persistence |

## What the test drives

Walks `docs/protocol-mobile.md` § Appendix (lines 712+) verbatim on one
phone conn:

1. `pyry pair --pyry-name=test --name=phone-a` (via `RunBareIn`) →
   mint pairing token through the existing CLI surface; never
   re-implements the pairing flow.
2. Seed `~/.pyry/test/conversations.json` with one fixed-UUID row
   (`knownConvID = "77777777-…"`, `cwd=home`, fixed `last_used_at`).
   This row is what `list_conversations` must report **and** what
   `send_message` targets — single conv keeps the test free of
   ordering assertions that belong on `list_conversations`'s own
   slice.
3. `StartRotationWithRelay(t, home, sessionsDir, initialUUID,
   neverCreatedTrigger, stdinLog, fr.URL()+"/v1/server",
   "PYRY_FAKE_CLAUDE_ASSISTANT_TRIGGER="+asstTrigger)` spawns the
   daemon wired to a `fakerelay` URL. The rotation trigger path is
   deliberately never-created — this test isn't about rotation; the
   harness flavour is chosen because step 5 needs the assistant-turn
   bridge (and therefore a fakeclaude with a trigger env var to
   produce the assistant chunk on cue).
4. Poll `fr.LastBinaryHello(serverID)` with a 5s deadline + 20ms
   tick until the binary's `hello` is observed. No content
   assertions here — `TestRelay_Hello` owns the binary-hello shape.
5. `fakephone.Dial(ctx, fr.URL(), serverID, pairPayload.Token,
   "phone-a")` with a 3s ctx.
6. Send phone `hello` (`id=1`) → receive `hello_ack` with
   `InReplyTo==1`.
7. Send `list_conversations` (`id=2`) → receive `conversations` with
   `InReplyTo==2`, decode payload, assert `len==1` and
   `Conversations[0].ID == knownConvID`.
8. Send `send_message` (`id=3`, `conversation_id=knownConvID`,
   `message_id="u-1"`, `text=knownUserText`) → receive `ack` with
   `InReplyTo==3`.
9. Write `knownAssistantText` to `asstTrigger`. The fakeclaude PTY
   emits the marker, the `Bridge.outputObserver` from #311 fires the
   broadcast, and a `message` envelope arrives at the phone. The
   drain loop tolerates TUI prelude chunks (banner, clear) by
   accepting any frame whose `Type != message` or whose
   `MessagePayload.Text` doesn't contain the marker — pattern
   verbatim from `relay_assistant_turn_test.go`. 5s outer budget,
   per-receive deadline derived from `time.Until(drainDeadline)`.
10. Send `register_push_token` (`id=4`, platform=`fcm`,
    token=`fcm-token-xyz`) → receive `ack` with `InReplyTo==4`. No
    on-disk reload — `TestRelay_RegisterPushToken_AckAndPersists`
    owns that.

Every received envelope (steps 6, 7, 8, 9, 10) is collected into a
`received []protocol.Envelope` slice during the run. Two cross-step
invariants run as one pass after step 10.

## Cross-step invariants

- **Strictly monotonic binary-side `id` across the full sequence.**
  Five entries, indices 1..4 each greater than its predecessor. The
  per-slice tests can only pin `>= N` against a fresh conn; this is
  the first place that sees `id` advance across a mixed set of verbs
  (`hello_ack`, `conversations`, `ack`, `message`, `ack`) and can
  assert strict ordering. The `message` envelope (server-initiated,
  no `in_reply_to`) is in the chain too — the dispatcher's per-conn
  counter advances on every emitted frame regardless of trigger
  source.
- **`in_reply_to` chain across four req/resp pairs.** Each of the
  request envelopes (`hello`, `list_conversations`, `send_message`,
  `register_push_token`) has its response's `InReplyTo` checked
  inline against the request's `ID`. The step-9 `message` echo's
  `InReplyTo` is asserted **nil** (server-initiated).
- **`conversation_id` stability `send_message` → `message`.** The
  request carried `knownConvID`; `MessagePayload.ConversationID` on
  the echo is `== knownConvID` byte-for-byte. `ack` carries no
  `conversation_id` (`AckPayload` is `struct{}` by spec) — bridged
  across that gap by the `in_reply_to` of step 8 plus the `id`
  monotonicity check.
- **`ts` round-trips through JSON unchanged.** For every received
  envelope: `!env.TS.IsZero()` and `roundTripTS(env.TS).Equal(env.TS)`
  where `roundTripTS` marshals through a `struct { TS time.Time }`.
  The monotonic-clock reading is stripped by `json.Marshal` —
  `time.Time.Equal` is the only correct comparison, never `==` or
  `reflect.DeepEqual` (see PROJECT-MEMORY.md § "`time.Time`
  round-trip discipline").

## What the test deliberately does NOT do

Each of these is owned by an adjacent test; duplicating them here
would erase coverage attribution and inflate the test past its
~200 LOC hedge.

- Reload `devices.json` after `register_push_token` →
  `TestRelay_RegisterPushToken_AckAndPersists`.
- Assert the user-turn text reaches fakeclaude's stdin →
  `TestRelay_SendMessage_AckAndPTYDelivery` via the stdin log
  (#323).
- Assert `auth.invalid_token` / WS close 4401 → `relay_auth_test.go`
  (out of scope per the ticket; the dispatcher slice that introduced
  the gate owns the negative case).
- Multiple phones, reconnect, backfill, ordering of multiple
  conversations.

## Patterns established (or reinforced)

- **Collect-then-assert for cross-step invariants.** Per-step shape
  assertions (`Type`, `InReplyTo`) stay inline at the point they
  happen — they read like the protocol and fail with the smallest
  scope. Cross-step invariants (monotonic `id`, `ts` round-trip)
  run as one pass after the sequence completes, walking the
  `received` slice. This split keeps the test linear and the
  failure messages local while still asserting the global
  properties no per-slice test can see.
- **Pure consumer e2e tests are the right place for cross-cutting
  invariants the per-slice tests can't observe.** The four sibling
  tests each pin one slice end-to-end; this one stays at the
  protocol-envelope tier (no on-disk reload, no fakeclaude-stdin
  assertion) and pins only what's invisible without the full
  appendix sequence. Holds the test at ~310 LOC, well under the
  `needs-rework:po` trip-wire at ~200 LOC of *new* test logic
  (the budget excluded the helper + comments + the
  delete-on-error scaffolding).
- **`StartRotationWithRelay` is the right harness flavour when an
  assistant-turn echo is in the test plan.** The rotation
  machinery is unused (trigger path deliberately never-created)
  but the harness's fakeclaude with the assistant-trigger env var
  is what the step-9 echo needs. The naming was awkward enough to
  briefly tempt extracting a `StartWithRelayAndAssistantTrigger`
  helper — resist; the harness already accepts variadic env, and
  one consumer doesn't justify a new factory.

## Lessons learned

### One file, one test, one helper — resist the urge to factor

The architect's spec explicitly called out "no helpers extracted;
the existing helpers (`shortHome`, `relayTestLogger`,
`readPersistedServerID`, `StartRotationWithRelay`,
`decodePairPayload`, `mustJSON`) cover everything." The only new
helper (`roundTripTS`, ~6 lines, single caller) lives inside the
test file. There's exactly one consumer of this composition and a
second consumer would be a different test in spirit anyway. The
pull toward `relayRoundtripHelpers.go` was real and was the wrong
move — keeping it inline made the test readable as a literal
walk of the appendix.

### Drain-until-marker, not count-and-receive

Step 9's drain loop is the only place the test reads multiple
frames without knowing the count up front: the supervisor's PTY
may emit prelude chunks (TUI banner, clear) before the assistant
marker chunk. The deadline-bounded `for { phone.Receive(remaining)
… }` loop pattern is verbatim from
`relay_assistant_turn_test.go:159-190`. Trying to assert "exactly
one `message` frame arrives" would be brittle against any
TUI-prelude churn; "at most one `message` frame whose payload
contains the marker arrives within 5s" is the right shape and is
robust against unrelated changes to the fakeclaude warmup.

### Per-receive deadline derived from outer budget

`phone.Receive` deadlines on steps 6–8 and 10 are a flat `3 *
time.Second` — these are point-to-point synchronous req/resp and
a longer timeout would only obscure the real failure. Step 9 is
different: the drain budget is `5 * time.Second` outer, and each
inner `Receive` gets `time.Until(drainDeadline)` so the loop
terminates exactly at the outer deadline regardless of how many
prelude chunks arrived. If anyone tightens this, they need to
measure under `-race` on the slowest CI runner first.

### Test-only markers are NOT secrets

The test file opens with a four-line comment block warning that
`knownUserText` and `knownAssistantText` are test-only markers
that flow through the fake relay and get echoed verbatim in
failure messages. Anyone copy-pasting the test as a template for
a different protocol slice needs to know not to paste a real
token / device-name / push-token in their place — the marker
constants are designed for grep, not for security.

## Out of scope (carried forward, per the ticket's hedge)

- The real `pyrycode-relay` binary (fakerelay harness is
  sufficient for the daemon side).
- TLS termination — test runs `ws://` only; the
  `AllowInsecureScheme` / `PYRY_ALLOW_INSECURE_RELAY=1` seam from
  #301 is what lets the harness through `Connect`'s `wss://`-only
  guard.
- APNs / FCM push delivery — `register_push_token` persists the
  token; remote push delivery is operational, not protocol.
- Invalid-token negative case — owned by the dispatcher slice's
  `relay_auth_test.go` per the ticket's Out-of-Scope.
- Multi-phone fan-out, reconnect-and-resume, backfill state. The
  appendix as written is single-phone; multi-phone semantics need
  their own appendix section before they can have an e2e test.
