# Ticket #376 — `e2e/realclaude`: tool-loop integrity test (multi-turn with Bash)

Third consumer of the `internal/e2e/realclaude` fixture trio (#372 + #373). Positive-path counterpart to #365: that test verifies a Bash invocation is *blocked* under `--allowed-tools=Read`; this verifies that under `--allowed-tools=Bash` the model successfully drives the full internal tool loop (assistant `tool_use` → user `tool_result` → final assistant `text` → `end_turn`) end-to-end through `pyry agent-run`.

Pins both the on-disk JSONL content-block correlation (tool_use id ↔ tool_result `tool_use_id` ↔ subsequent assistant text block) AND pyry's stdout result-trailer shape (`subtype=="success"`, `stop_reason=="end_turn"`, `num_turns>=2`). A regression here means claude's stream-json contract has shifted in a way that silently breaks pyry's tool-loop assumptions — e.g. tool execution moving from internal to caller-supplied, or event-ordering changes. No production-code change.

See [features/e2e-realclaude.md](../features/e2e-realclaude.md) for the suite contract.

## Implementation

- **`internal/e2e/realclaude/tool_loop_test.go`** (~202 LoC, new). `//go:build e2e_realclaude`, `package realclaude`. One test (`TestRealClaude_ToolLoopIntegrity`) + two file-local helpers (`parseContentBlocks`, `parseResultTrailer`) + two struct types (`contentBlock`, `resultTrailer`). No new fixture surface, no new exports.
  - **Four-phase shape extended to five** — worktree allocate → **seed two text files** → run → run-level asserts → property assertion (two sub-phases: on-disk content-block walk + stdout result-trailer parse). The seeding step is new vs. #364/#365 because the Bash invocation needs deterministic targets to list and summarise; absent seeds, the model has no fixed observable to exercise.
  - **Seeded fixtures.** `hello.txt` → `"hello\n"`, `world.txt` → `"world\n"`, written via `os.WriteFile(filepath.Join(workdir, name), payload, 0o600)`; `t.Fatalf("seed %s: %v", ...)` on write error. The files exist so the Bash invocation has something to enumerate. They are NOT asserted against — the test pins protocol shape, not response content.
  - **`RunOpts` shape.** `AllowedTools: []string{"Bash"}`, `MaxTurns: 3` (assistant tool_use → tool_result → final assistant text; the third turn is slack against haiku occasionally running a verification command), `Effort: "low"`, `Model: "claude-haiku-4-5"` (full literal id, matching #364/#365). One real haiku call per run, ~$0.02.
  - **Prompt is intentionally explicit.** `"Use the Bash tool to run \"ls -1\" in the current directory, then tell me how many .txt files you see."` — phrases Bash as the obvious tool and gives the model a concrete numeric question to answer in its final assistant turn, exercising both the `tool_use`→`tool_result` hop AND the subsequent text turn. Mirrors #365's "prompt is an enforcement-test fixture" discipline.
  - **`SystemPrompt` is minimal and neutral** — `"You are an e2e regression-guard test. When asked to inspect the filesystem, use the Bash tool."`. Non-empty is load-bearing (`validateRunOpts` rejects empty). The "use the Bash tool" line is a gentle nudge, not a hard requirement; the prompt itself does the heavy lifting.
  - **On-disk traversal is a single-pass state machine.** Three flags (`bashToolUseID string`, `sawToolResult bool`, `sawFinalText bool`) and one walk over the `events` slice from `ReadJSONL`. For `e.Kind == "assistant"`: if no Bash tool_use captured yet, scan content blocks for the first `{Type:"tool_use", Name:"Bash"}` with non-empty `ID`; otherwise (after a tool_result has been seen), scan for any `{Type:"text", Text:non-empty}` and set `sawFinalText`. For `e.Kind == "user"` with `bashToolUseID` already captured: scan content blocks for `{Type:"tool_result", ToolUseID==bashToolUseID}` and set `sawToolResult`. Post-loop, three sequential `t.Fatalf`s (Bash use → result → final text) so the first missing piece is the one reported.
  - **`tool_use` / `tool_result` are content blocks INSIDE assistant/user entries, NOT top-level events.** The on-disk JSONL emits `tool_use`/`tool_result` as content blocks under `message.content[]` on `assistant`/`user`-kinded entries. `jsonl.Event.Kind` whitelists strings including `"tool_use"`/`"tool_result"`, but those values do not appear on disk for real-claude runs; the assertions match against the **nested content blocks** parsed out of `Event.Raw`, not against `Event.Kind`. This is the load-bearing insight from the architect's spec — getting it wrong would have produced a test that always fails.
  - **`parseContentBlocks(raw json.RawMessage) ([]contentBlock, error)`** — file-private. Decodes the verbatim line into `{Message:{Content:[]contentBlock}}` where `contentBlock` carries only the five fields under assertion (`Type`, `Name`, `ID`, `Text`, `ToolUseID`) with `omitempty` so one struct shape handles `tool_use`, `tool_result`, and `text` blocks. Returns `(nil, err)` on malformed JSON; callers treat parse errors as a per-line skip (mirrors `bashInvokedInRaw` from #365 and `selfcheck.go:283`).
  - **Result-trailer source: stdout, NOT on-disk JSONL.** The `type:"result"` event is emitted exclusively on pyry's stream-json **stdout** by `streamjson/emitter.go`'s `Close` — it is NOT present in the on-disk `~/.claude/projects/<...>.jsonl`. The acceptance criteria mention "the `result` event" without disambiguating; the spec and this implementation resolve it: result-trailer assertions read `result.Stdout`, not `events`. Getting this wrong is the second hidden landmine in the AC.
  - **`parseResultTrailer(stdout []byte) (*resultTrailer, error)`** — file-private. `bufio.Scanner` over `bytes.NewReader(stdout)`, attempts `json.Unmarshal` on each line into `resultTrailer`, returns the first line where `Type == "result"`. Non-JSON and non-result lines skip silently. Returns `errors.New("no type:result line in stdout")` if no result line is found; caller `t.Fatalf`s with the first 1 KiB of stdout (matches the SessionID-missing failure-message pattern at `allowed_tools_enforcement_test.go:45-50`). The default `bufio.Scanner` 64 KiB line limit is sufficient — pyry's emitter trailer is ~400 bytes — but a comment in the helper notes that a future stream-json line exceeding 64 KiB would require `scanner.Buffer(make([]byte, 1<<20), 1<<20)`.
  - **`resultTrailer.PermissionDenials` is pointer-typed `*[]json.RawMessage` with `omitempty`.** Pyry's emitter (`streamjson/emitter.go:251-275`) emits eleven trailer fields and `permission_denials` is NOT in that set — it is a claude-only field that pyry's re-emitter drops. The conditional check (`if trailer.PermissionDenials != nil && len(*trailer.PermissionDenials) != 0`) is therefore dormant today; it is retained per the AC ("if present in the schema") as a forward-compat guard. Pointer-typed so callers distinguish "field absent" (`nil`) from "field present, empty slice" (non-nil, len 0).
  - **Result-trailer assertions (four).** `Subtype == "success"`, `StopReason == "end_turn"`, `NumTurns >= 2` (a tool-loop run necessarily has ≥2 assistant entries; ≥ absorbs any post-tool clarification turns within `MaxTurns: 3`), and the dormant `permission_denials` empty-check. Each on its own `t.Fatalf` with the observed value embedded so a single failure pinpoints the field.
  - **Decode-error policy on per-line parses: skip silently.** A single malformed assistant or user line must not turn a PASS into an inconclusive — mirrors `bashInvokedInRaw` (#365) and `selfcheck.go:283`. By contrast, **missing the entire result trailer is a hard `t.Fatalf`** — the trailer is a structural guarantee of every successful run, so its absence is a real regression.
  - **Failure-message discipline.** On-disk-walk failures cite the resolved JSONL path via the file-package-private `jsonlPathFor` helper (#364, now three consumers). Stdout-trailer failures embed the first 1 KiB of stdout. Never echo `e.Raw` (consistent with #365 and `selfcheck.go:217-219`).

## Patterns established

- **Distinguish on-disk JSONL events from stdout stream-json events when asserting protocol shape.** The on-disk `~/.claude/projects/.../<sessionID>.jsonl` and pyry's stdout stream-json are two different streams with **different event sets**. On-disk has `assistant`/`user`/`system` only; stdout additionally carries the `type:"result"` trailer (emitted by `streamjson/emitter.go.Close`). When a test reads "the X event," the spec must name which stream produces it, and the test must read from the matching source. Generalises to any future protocol-shape regression-guard.
- **Tool-use and tool-result are content blocks under `message.content[]`, not top-level entries.** When asserting tool-loop shape on real-claude on-disk JSONL, the assertions must parse `Event.Raw` for nested content blocks, not switch on `Event.Kind`. `jsonl.Event.Kind`'s whitelist technically permits `"tool_use"`/`"tool_result"` as top-level kinds but that path is unreachable for production claude. Generalises to any tool-loop assertion against real-claude.
- **Pointer-typed `*[]T` with `omitempty` to distinguish "field absent" from "field present but empty".** When an AC says "the field, if present, must be empty," and the field is genuinely optional in the schema, use a pointer to a slice. `nil` means absent; non-nil, `len == 0` means present and empty; non-nil, `len > 0` means present and non-empty. Generalises to any forward-compat guard for an optional collection.
- **Single-pass state-machine traversal beats multiple grep-the-slice passes.** When a test asserts a sequence of related properties (Bash tool_use captured → matching tool_result observed → subsequent text turn), encode the sequence as three flags advanced by one loop over the events, not as three independent passes. The state machine produces directional failure messages (the first missing piece is the one reported) and reads in event order so an out-of-order regression surfaces correctly. Generalises to any sequence-property assertion over a stream of events.

## Lessons learned

- **AC ambiguities benefit from explicit spec resolution before implementation.** The AC's "result event" mention did not disambiguate on-disk vs. stdout, and the on-disk JSONL doesn't carry a `type:"result"` line at all. The architect's spec named the source explicitly (stdout, via `result.Stdout`), and that single decision determined the shape of `parseResultTrailer` and where its assertions plug in. When an AC names a field or event, verify which producer emits it and pin that in the spec; don't push the disambiguation onto the developer's first failed test run.
- **`permission_denials` is a claude-only field; pyry's re-emitter drops it.** Future work on the stream-json trailer pipeline should either (a) decide to propagate `permission_denials` through to pyry's stdout, in which case this test's dormant branch fires for free, or (b) consciously continue dropping it and document why in a feature doc. Today the field is invisible to anything reading pyry's stdout; the dormant guard is harmless but not load-bearing.

## Out of scope (deferred)

- **Per-line tests of `parseContentBlocks` / `parseResultTrailer`.** There is no second-order test of the test — the helpers' correctness is bounded by what the production claude CLI emits and the assertion path within the test itself. Mirrors the shape decision in #365 for `bashInvokedInRaw`.
- **Assertions on the assistant's final text content.** The property is "a subsequent text block exists and is non-empty," nothing more. The test pins protocol shape, not response quality; widening to content assertions would couple the test to haiku's wording and add flakiness without strengthening the contract.
- **Other tools (Read, Edit, Grep, Glob).** Bash is the canonical tool-loop probe because it has a side-effecting verb and produces a deterministic enumeration; adding other tools would multiply cost without proving a different protocol property. If a future protocol divergence shows up only for read-shaped tools, *that* ticket adds the row.
- **Promoting `jsonlPathFor` to an exported fixture helper.** Three consumers in the same package today; package-private access still suffices. Promote when a sibling-package test needs the same lookup.
- **Extending the `bufio.Scanner` line-buffer.** Default 64 KiB is sufficient against pyry's ~400-byte trailer; bump only when a stream-json line genuinely exceeds the limit.
