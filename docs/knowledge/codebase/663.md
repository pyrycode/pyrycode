# Ticket #663 — Clamp the caught-up replay watermark to the newest retained id (#647 MUST FIX)

Resolves the code-review **MUST FIX** that shipped unresolved with [#647] / PR #651
(merged 2026-06-08) and was recorded in [codebase/647.md] § "⚠️ Known issue".
`replayMissed` set the per-connection dedup watermark `s.replayThrough` straight
from the **untrusted** remote `afterID` (`hello.last_event_id`) even in the
**caught-up** branch — and the paired `forwardEnvelope` guard drops any live frame
whose `EventID <= replayThrough`. An `afterID` outside the current conversation's
id space (a stale cross-`/clear` id, or a hostile `2^64-1`) therefore became a
remote-triggerable mute switch on the connection's own live stream — the exact
silent-gap the replay feature exists to prevent. The fix clamps the caught-up
watermark to server-known reality: `s.replayThrough = min(afterID, NewestID(convID))`.
Server-internal logic fix only — **no wire change** (`last_event_id` / `resync`
already exist). `security-sensitive` (inbound trust boundary). Blocks
pyrycode-mobile#416. EPIC #596 Phase 2 structured streaming, [ADR 025]
§ Backpressure / replay.

## Implementation

Two production touch points, both additive/minimal; **0 new types, 0 wire surface,
0 constructor call sites changed**. The production delta is two lines of logic; the
**regression coverage is the deliverable** (it closes the precise gap the green
#647 suite missed).

| File | Change |
|---|---|
| `internal/eventring/ring.go` | **+`NewestID(convID) uint64`** — additive accessor returning `c.nextID - 1` for a known conversation, `0` for an unknown one. Mutex-guarded by the ring's existing `sync.Mutex` (same safe-off-the-emitter-goroutine guarantee as `After`). It is the clamp source. |
| `internal/relay/v2session.go:1493` | **the one-line clamp** — `s.replayThrough = min(afterID, newest)` replaces the unconditional `= afterID`, with `newest := ring.NewestID(convID)` read **before** `After` (`:1476`). The gap branch (resync, watermark untouched) and the replay loop (`= ev.ID` per frame) are unchanged. |

Test files (excluded from the production count): `internal/eventring/ring_test.go`
(`TestNewestID`, 4 subtests), `internal/relay/v2session_replay_test.go` (the
`reconnectOpenLive` helper + 3 live-delivery regression tests, ~200 LOC).

Why `NewestID == nextID - 1` is sound: `nextID` advances on **every** `Append`
independent of retention, and the **newest event is never evicted** (eviction takes
the oldest first — [eventring-package.md] § Eviction policy). So `nextID - 1` is
always the highest *retained* event's id, and it equals `After`'s own internal
`latestID` caught-up boundary — the clamp is consistent with `After`'s
classification by construction.

Why an accessor, not extending `After`'s signature: `After` has 18 call sites (1
production + 17 test); adding a return value cascades a mechanical arity bump across
all of them (over the 10-call-site edit-fan-out red line) for zero correctness gain
over a clean additive method. (Codegraph confirmed `NewestID` is a genuinely new
symbol — no collision, `After` untouched, blast radius nil.)

## Patterns established

- **A dedup/drop watermark derived from remote input must be clamped to
  server-known reality, not just shape-validated by its type.** `*uint64` decode
  stops non-integers, but a syntactically valid *out-of-conversation-range* id is
  just as dangerous as a malformed one for a "≤ watermark ⇒ drop" guard. The fix
  shape — `min(remoteCursor, serverKnownBound)` where the bound is internal state
  (`NewestID`) — generalises to any future watermark/guard pair. This is the
  resolution of the lesson [codebase/647.md] § "Patterns established" flagged: the
  watermark pattern was sound, the *input* was not.
- **Read the clamp bound BEFORE the classifier, so staleness only ever
  under-clamps (delivers more), never over-clamps (suppresses).** `newest` is
  captured before `After`; any event the emitter `Append`s in the window carries an
  id `> newest` and falls through to the live stream rather than under the clamp.
  For a "never a silent gap" guarantee, bias every race toward delivery. The two
  ring reads (`NewestID` then `After`) are separate lock acquisitions, not atomic —
  intentionally; the read order makes it correctness-equivalent to an atomic
  single-read (the architect's reason for rejecting the extend-`After` alternative).

## Lessons learned

- **The truest RED for a "silent suppression" defect asserts downstream
  *delivery*, not the absence of the intermediate artifact.** The green #647 suite
  checked "no replay frames" in the caught-up branch and never asserted the
  connection's *live* stream still flowed afterward — so it passed against the bug.
  Every regression test here drives a live `mgr.Push` after the caught-up handshake
  and asserts the frame is forwarded and decrypts, and each was demonstrated RED
  against the bug before the fix. (From the PR body's Lessons learned.)
- **The `/clear`-rotation regression only bites when the target conversation is
  non-empty at reconnect.** An empty/unknown `conv-B` classifies `After(B, stale)`
  as a **gap** (`afterID > 0` on an unknown conv → resync, `replayThrough` left at
  0) — a path that already delivers live events *even with the bug present*. The
  caught-up branch (where the defect lives) is only reachable when the conversation
  already holds at least one retained event with `latestID < afterID`. A test that
  let `conv-B` be empty would reproduce the exact false confidence the green suite
  suffered. (From the PR body's Lessons learned.)
- **`tc := tc` loop-variable capture is redundant under the repo's `go 1.26.2`**
  (per-iteration loop vars since Go 1.22). Harmless and a common defensive idiom;
  flagged NIT-only by code review, no change needed.

## Testing

The two-line production change is verified by tests that are the actual point of
the ticket.

`internal/eventring/ring_test.go` — `TestNewestID` (4 subtests): unknown
conversation → 0; after N appends → N; **after eviction** (`New(2)` + 5 appends) →
still 5 (the load-bearing "counter advances independent of retention, newest never
evicted" property the clamp rests on); conversation isolation (A=3, B=7
independent).

`internal/relay/v2session_replay_test.go` — three live-delivery regression tests,
all built on a new `reconnectOpenLive` helper (the `OtherConnsUnaffected`
inline-manager + `mgr.Push` + per-conn frame-counting pattern; the existing
`reconnectScenario` discards the manager). The helper decrypts the handshake's
replay frames to keep the initiator's recv `CipherState` nonce aligned, so live
frames pushed afterward decrypt cleanly (the guard drops *before* seal, never
gapping the nonce):

- **`…_OutOfRangeLastEventID_LiveStreamDelivered`** (AC-2/AC-5): ring `conv-A`
  ids 1..5; reconnect with `last_event_id` = 99 (plain out-of-range) and
  `math.MaxUint64` (hostile) — caught-up, no replay — then `mgr.Push` a live
  `EventID=6` and assert it is forwarded and decrypts to `{"n":6}`. Under the bug
  (`replayThrough = afterID`) this frame is silently dropped; with the clamp
  (`replayThrough = 5`) it flows.
- **`…_ClearRotation_LiveStreamDelivered`** (AC-3): `conv-B` pre-populated with ids
  1..3 (**non-empty** — see Lessons), cursor → B, reconnect with stale
  `last_event_id = 100`; caught-up → `replayThrough = 3`; push live ids 4, 5 and
  assert both delivered.
- **`…_SameConversation_DedupPreserved`** (AC-4): `conv-A` ids 1..5, reconnect
  `last_event_id = 2`, replay 3,4,5 (`replayThrough` ends at 5); a re-delivered
  `EventID=3` is dropped (≤ 5) while `EventID=6` flows — proves the clamp did not
  loosen real same-conversation dedup (`min(2,5)=2`, loop advances to 5).

`make check` (vet + `-race` + staticcheck + substrate-guard) and `make build`
green. Real-claude e2e (`-tags e2e_realclaude`) passed clean (163.5s, no flake).
The `internal/relay` + `internal/eventring` unit suites remain the behaviour oracle;
a live two-phone reconnect-replay e2e stays blocked on the structured-receive
harness gap (#603 / #642), the same inherited caveat as #647.

## Concurrency model

Unchanged access regime — the fix only changes the *value* assigned to
`replayThrough`, not how the field is accessed.

- **No new goroutine, no new lock, no new shared field.** `NewestID` is guarded by
  the ring's existing `sync.Mutex` (one extra rare-path O(1) acquisition per
  reconnect). `replayThrough` stays Run-owned (written in `replayMissed`, read in
  `forwardEnvelope`, both on the manager's single `Run` goroutine).
- **The two ring reads are deliberately non-atomic; the read order is the
  correctness argument** — `newest` first means a concurrently-appended event
  (id `> newest`) always reaches the live stream. The one benign edge (a
  pending-broadcast event with id `== newest` in the deferred cross-conversation
  case) is recoverable via the resync-on-`conversation_id`-change path and is
  identical under an atomic single-read design. The legitimate same-conversation
  path is race-free (`afterID == newest` there, so `min` selects `afterID`).

## Out of scope (unchanged from #647)

- The `backfill_since` full-reload handler the phone calls on a resync marker
  (needs a message-history store that does not exist; `internal/conversations` is
  metadata-only). Cross-`/clear` *catch-up* (vs. the live-stream delivery this
  fixes) remains deferred `backfill_since` territory — but the live stream the
  phone uses to *detect* the rotation via `conversation_id` is no longer
  suppressed, which is what this ticket restores.
- A live two-phone reconnect-replay e2e (#603 / #642 harness gap).
