# Ticket #583 — remove the now-dead fakerelay binary-hello path

The e2e fakerelay harness (`internal/e2e/internal/fakerelay`) carried the
*receiving* half of a handshake the binary no longer performs. [#582](582.md)
retired the binary↔relay `hello`/`hello_ack` ceremony in the real binary — the
leg is now established on WS upgrade with header-based server-id registration
(`x-pyrycode-server`), and no relay-leg hello is sent. That left the harness's
`handleBinaryDirect` hello→hello_ack synthesis branch unreachable, its
`lastBinaryHello` map never written, and its `LastBinaryHello` accessor with no
readers ([#581](581.md) had already moved e2e readiness onto the header-based
`WaitBinary`). This ticket deletes that dead machinery. Pure deletion — no new
types, no behaviour change, no signature changes. It is the final slice of
[#569](https://github.com/pyrycode/pyrycode/issues/569)'s Strangler Fig.

## Implementation

Deletion-only across two files in `internal/e2e/internal/fakerelay/`
(net −85 production / −50 test lines):

- **`fakerelay.go` — the cut (AC #1).** Removed: the package-doc deviation
  bullet describing binary-direct hello→hello_ack synthesis; the
  `Server.lastBinaryHello map[string]protocol.Envelope` field + doc; the
  `New` constructor's `lastBinaryHello: make(...)` initializer; the
  `binaryRecvPump` `if env.ConnID == "" { handleBinaryDirect(...); continue }`
  dispatch branch and its explanatory comment; the `handleBinaryDirect` method
  in full; and the `LastBinaryHello` accessor + doc.
- **Stray no-conn_id frames now fall through to the existing
  unknown-`conn_id` Debug-drop** (`fakerelay.go:400-403`: log + `continue`).
  No special case is needed — `s.phones[""]` is always a miss, so there is no
  nil-phone deref and no goroutine leak. After #582 the binary↔relay leg sends
  no such frames anyway; the catch-all matches the pump's established
  "drop the offending frame, keep serving" philosophy.
- **`fakerelay_test.go` (AC #2).** Removed `TestBinaryHello_GetsHelloAck` — the
  sole exerciser of both the hello-ack synthesis and `LastBinaryHello`.
  `TestWaitBinary` (happy + timeout) and `TestForceCloseBinary` stay (AC-3
  `WaitBinary` coverage); `TestPhoneToBinary_FrameWrappedWithConnID` /
  `TestBinaryToPhone_FrameUnwrapped` / `TestConnIDIncrementsPerPhone` stay —
  they route frames whose *content* is `{"type":"hello"}` **with a conn_id**
  through the phone-leg path (ordinary phone traffic, not a binary-direct
  hello) and are the AC-3 "binary↔relay leg still reaches frame-forwarding"
  coverage.
- **Zero import changes**, exactly as the spec predicted: `protocol.RoutingEnvelope`
  keeps the `protocol` import live in both files; `time` stays live via
  `WaitBinary`. Only `protocol.Envelope`/`TypeHello`/`TypeHelloAck`/
  `HelloAckPayload` references lived inside the deleted code.

### What stayed (untouched)

- **Header-based server-id registration** — `handleBinary` reads
  `X-Pyrycode-Server` → `s.binaries`. This is what makes `WaitBinary` work.
- **`WaitBinary`, `ForceCloseBinary`, `RejectNextBinaryWith4409`** — server-side
  e2e hooks unrelated to hello.
- **Phone-leg handling** — `handlePhone`/`phoneRecvPump`/`phoneSendPump`,
  token-injection, `CloseCode` honor. The phone↔binary hello travels as
  Noise_IK early-data (E2E-encrypted, relay-blind) and never went through the
  binary-direct path.

## Patterns established

- **A test harness should not carry machinery for a protocol step the
  production code no longer performs.** Keeping `handleBinaryDirect` alive
  would have let the harness silently absorb a relay-leg hello that a real
  binary can never send, masking a regression rather than exposing one. The
  honest harness now mirrors the production relay: content-blind on the
  binary↔relay leg, registering by header on upgrade.
- **Strangler Fig completed.** This is the last of #569's three-step removal:
  **[#581](581.md) migrated the readers** (e2e readiness off `LastBinaryHello`
  onto header-based `WaitBinary`) → **[#582](582.md) retired the handshake** in
  the binary → **#583 removes the now-dead harness hello-receiving path +
  `LastBinaryHello`.** Migrating consumers first meant retiring the handshake
  (#582) cascaded no `t.Fatal`s, and this final deletion touched only the two
  fakerelay files — every reference was already inside them.

## Lessons learned

- **Code review can be clean yet BLOCKED on an environment gap, not a code
  defect.** The judgment review passed step-for-step (zero dangling refs;
  `go test -tags e2e -race ./internal/e2e/internal/fakerelay/...` green), but
  the mandated `make e2e-realclaude` gate could not run: the realclaude fixture
  rebuilds `pyry` in an isolated temp `GOMODCACHE`, forcing a clean re-fetch of
  the **private** dep `github.com/pyrycode/tui-driver`, and the review
  environment has no non-interactive git credentials for `github.com/pyrycode/*`.
  Reviewer correctly escalated to the operator (provision credentials / confirm
  a credentialed CI run) rather than routing a false `needs-rework:developer` —
  nothing in the diff (which never touches `agentrun/budget`, where the failing
  import lives) is broken. *Pipeline-environment lesson: the realclaude gate
  needs git credentials for private `pyrycode/*` modules in the review env.*
- **Incidental testdata churn (recurring footnote from #581/#582):**
  `internal/e2e/realclaude/testdata/permission_protocol_v2.1.158.json` changed
  in the code-review auto-commit — identifier-only (`session_id`, message/`toolu`
  ids, timing/cost), no structural change. It belongs to a realclaude test, not
  to this ticket.

## Testing

- `go test -race ./...` and `go vet ./...` pass (the fakerelay package is part
  of the e2e suite; the default build must not regress).
- `go test -race -tags e2e ./internal/e2e/...` + `go vet -tags e2e ./internal/e2e/...`
  pass — fakerelay unit tests and the relay round-trip e2e tests confirm real
  pyry binaries still complete header-based relay registration without the
  synthesized `hello_ack`. `go build ./cmd/pyry` succeeds.
- `grep -rn 'LastBinaryHello|lastBinaryHello|handleBinaryDirect'` across all Go —
  zero remaining references.

## Out of scope

- **Removing the `HelloServerPayload` / other now-orphaned handshake types** —
  separate follow-up cleanup flagged in [#582](582.md)'s Lessons learned. The
  `TypeHello`/`TypeHelloAck`/`HelloAckPayload` types stay: the *phone↔binary*
  Noise leg still uses them.
- **The `/v1/...` → `/v2/...` route rename** — cosmetic on a content-blind relay;
  explicitly not performed (per #582 AC).
