# Ticket #523 ŌĆö transport: harden RacingSendError test ctx budget against -race contention

Test-only fix for `TestFatalCloseCodes_HaltsReconnect_RacingSendError`, observed flaking once during PR #522's CI with the signature `wssclient_test.go:862: Connect did not return before ctx deadline` shape at a 5.01s duration (== the test's 5s ctx-deadline literal). The architect's spec (`specs/architecture/523-racing-send-error-flake.md`) prescribed a diagnose-first protocol with three branches: 2a (no repro ŌåÆ test-side bump), 2b (matrix A/B repro ŌåÆ instrument and pick from three sub-cases), 2c (matrix-C-only repro ŌåÆ bump `closeFrameGrace`). Diagnosis landed on **sub-case iii** (test-side race between the connect goroutine's buffered `connectErr` send and the test's `<-ctx.Done()` arm) by exclusion: bumping `closeFrameGrace` from 50ms ŌåÆ 250ms ŌåÆ 500ms did not measurably change the failure rate, so the production-side grace was ruled out as the operative knob and the fix is purely in the test.

## Implementation

### `internal/transport/wssclient_test.go` (+23/-9 LOC, no production change)

- **Ctx-deadline bumped 5s ŌåÆ 15s** at `wssclient_test.go:814`, with a three-line comment citing #523 and identifying the surface as `-race + parallel self-contention` (not production-side grace expiry).
- **Result-check select tightened** at `wssclient_test.go:857-879`. Previously: a single `select { case err := <-connectErr: ...; case <-ctx.Done(): t.Fatal(...) }` ŌĆö if the goroutine's `connectErr <-` send landed in the same scheduling window as ctx.Done becoming ready, Go's `select` would pick ctx.Done and `t.Fatal` even though `Connect` had returned `ErrFatalClose` successfully. Now: if the outer `select` picks `<-ctx.Done()`, an inner `select { case err = <-connectErr: ; case <-time.After(500*time.Millisecond): t.Fatal(...) }` gives the goroutine 500ms to publish its already-computed return value before declaring a real deadline failure. The `errors.Is(err, ErrFatalClose)` / `CloseStatus == 4409` assertions move outside the select so both paths run them.

### Why no production code changed

The diagnose-first protocol the spec prescribed had a clear branching point on whether the failure rate moved with `closeFrameGrace`. The developer's evidence:

| `closeFrameGrace` | Matrix | Failure rate |
| --- | --- | --- |
| 50ms (default) | A, `-count=500` | 2/500 |
| 250ms | C (3-yes loaders) | 3/500 |
| 500ms | C (3-yes loaders) | 2/500 |

A 10├Ś change in the candidate fix produced a statistically indistinguishable failure rate. Per the pipeline's "Evidence-Based Fix Selection" rule, that's the signal to abandon the production-side candidate and re-examine the failure mode ŌĆö which the developer did, landing on the test-side ctx-deadline race instead. Bumping a wire-internal timing constant that doesn't move the metric would have been a defense ahead of evidence.

### Sibling-contract pins remain green

- `TestAwaitCloseStatus_GraceBranchPreservesCloseError` (helper-level pin from #290 ŌĆö close-status preference orderings A/B/BŌĆ▓).
- `TestFatalCloseCodes_HaltsReconnect` (peer-side `TestFatalCloseCodes_HaltsReconnect_RacingSendError` partner).

Both pass under `-race -count=10` post-fix. The close-status preference invariant ŌĆö *"when the peer sends a fatal close, `Connect` classifies it as fatal regardless of which pump returns first"* ŌĆö is unchanged.

## Patterns established

- **A failing test whose elapsed time aligns precisely to its ctx-deadline literal is the test-budget shape, not a production stall.** When the diagnostic frame shows elapsed Ōēż 100ms off the test's deadline literal, look at the test's `select` shape and the goroutine that publishes the result before reaching for a production-side timing knob. The `-race -count=N` + `t.Parallel` test runner is itself a contention stress matrix ŌĆö matrix A at `-count=500` with no external load reproduces at the same rate as matrix C with 3 saturating loaders, because the parallel scheduler pressure from N concurrent test instances is the dominant factor independent of background CPU load.
- **The diagnose-first stress-matrix protocol from #290 generalises to other transport flakes.** Phase 1 reproduces under three matrices (A: default GOMAXPROCS; B: GOMAXPROCS=1; C: with background `yes` loaders), Phase 2 branches on which matrix surfaces the failure. The protocol's value is forcing the developer to rank the candidate knobs *before* coding ŌĆö and to ratchet the leading candidate to confirm it actually moves the metric. When ratcheting reveals a flat failure rate, that's the signal to switch branches, not to ship the no-effect change anyway.
- **Buffered-channel-then-select-with-ctx.Done is a common test antipattern under `-race`.** Any test of the shape "goroutine computes result, sends to buffered channel, main select-races channel vs. ctx.Done" can pick the ctx.Done arm at exactly the moment the buffered send becomes visible, producing a deadline-shape failure even when the work succeeded. The fix shape used here (drain the channel non-blockingly with a short follow-up timer after ctx.Done fires) is reusable for the same pattern elsewhere.

## Lessons learned

### A production-side knob that doesn't move the failure rate when ratcheted is not the bottleneck

The spec correctly framed `closeFrameGrace` as the candidate fix for matrix-C reproductions (sub-case 2c-i). The developer's ratchet sweep ŌĆö 50ms / 250ms / 500ms ŌĆö produced ~2/500 failure rates across all three, statistically indistinguishable. The correct read of that data is "the knob doesn't move the metric, so this is the wrong candidate," not "let me ship the 250ms anyway because it's a real knob in a real place." Future maintainers reading this: "Evidence-Based Fix Selection" doesn't just mean "have evidence before shipping" ŌĆö it also means "if the evidence flips the candidate, switch branches and don't ship the original."

### Matrix-C residual flake at the 15s budget is filed, not chased

After the 5s ŌåÆ 15s bump, matrix C (`-race -count=500` + 3 saturating `yes` loaders) still surfaces ~1/200 failures, but the error message changes from `Connect did not return before ctx deadline` (the test-budget race shape) to `Connect returned context deadline exceeded` (a real `Connect`-returned ctx-deadline error). That's a separate, deeper limitation under extreme synthetic contention (`-race` + 3-core saturation + `t.Parallel` self-contention) well beyond realistic CI conditions. Per scope discipline, this is a follow-up to file rather than expand-#523 territory. If it ever surfaces in real (non-synthetic) telemetry, the existing 290-style stress-matrix protocol applies ŌĆö the diagnose path is documented.
