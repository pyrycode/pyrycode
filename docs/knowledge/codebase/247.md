# Ticket #247 — `net`: WSS client with auto-reconnect backoff

Phase 3 Track C transport mechanic. New `internal/transport` package exposes a long-lived WSS client with native ping/pong heartbeat, exponential-with-jitter reconnect backoff, and stability-reset after a stable connection. The client is **generic over frame payload** — it accepts and emits `[]byte`, knows nothing about pyrycode's protocol envelope (#255), per-type payloads (#256), handshake, routing, or close-code interpretation. Protocol semantics layer on top in #248 (dispatcher); per-type payload decode in #256. The wire-spec source-of-truth for the cadence is `docs/protocol-mobile.md` § Heartbeat + § Reconnect — when that doc changes, this package changes.

## Implementation

- **Single production file (`internal/transport/wssclient.go`, ~375 LOC) + single test file (~520 LOC).** `Client`, `Config`, four sentinels (`ErrNotConnected`, `ErrClosed`; `ErrDisconnected` from the spec was elided — the recv-pump returning on conn-drop causes the next `Receive` to block until reconnect, per the spec's "deferred to #248" guidance). Methods: `New(Config) *Client`, `(*Client).Connect(ctx) error` (blocking lifecycle, returns `ctx.Err()` / `ErrClosed`, never `nil`), `Send([]byte) error`, `Receive(ctx) ([]byte, error)`, `Close() error` (idempotent via `sync.Once`).
- **WebSocket dep: `github.com/coder/websocket v1.8.13`** (architect's call — context-first API, ~2k LOC, MIT, actively maintained; `gorilla/websocket` was archived Dec 2022). First network-protocol dep in the project; pinned, no transitive deps. Single import point in `internal/transport`; no other package reaches for it.
- **Wire-spec constants pinned at package scope.** `pingInterval = 30s`, `pongTimeout = 30s`, `reconnectInitial = 1s`, `reconnectMax = 30s`, `stabilityResetMin = 60s`, `maxFrameBytes = 1 << 20` (1 MiB DoS cap via `conn.SetReadLimit`). These five timing constants are **mirrored as `Client` fields** (`pingInterval`, `pongTimeout`, `reconnectInitial`, `reconnectMax`, `stabilityReset`) set to the package constants in `New`; the test-only `newClientForTest` substitutes shorter values per-Client so cadence assertions run in tens of milliseconds, not tens of seconds. No build tags, no test-only setters on `Config`, no globals — the override path is "construct via `newClientForTest`," used only by the `_test.go`.
- **`Connect` is the blocking lifecycle (over fire-and-forget that returns after first dial).** Caller does `go client.Connect(ctx)`. Matches `supervisor.Run(ctx)` shape. One goroutine per Client, owned by the caller; cancelling `ctx` (or `Close`) tears down dial loop + pumps + ping loop atomically. The alternative shape (Connect returns after first success, reconnect runs in an internal goroutine) was rejected because (a) it hides goroutine lifecycle inside the type, (b) "Connect returns" semantics are ambiguous between "first dial succeeded" and "dial loop is running."
- **Per-conn fan-out: three goroutines (`recvPump`, `sendPump`, `pingLoop`) under a child ctx.** `serve(parent, conn)` builds `ctx, cancel := context.WithCancel(parent)`, spawns the three pumps with a 3-buffered `errCh`, **then calls `c.setConn(conn)` AFTER `go func()` lines** (per `docs/lessons.md:290` — install lifecycle goroutines BEFORE making conn observable to concurrent `Send` callers, otherwise a `Send` racing in sees `c.conn != nil` and writes to `sendCh` but no pump is reading). Waits on first errCh return → cancels → drains the other two before returning to the outer dial loop. No goroutine outlives its `serve` call.
- **`Send` and `Receive` are pure channel proxies; `sendCh` and `recvCh` are unbuffered.** Backpressure is the caller's problem: a slow consumer blocks `Receive` (which blocks `recvPump`'s channel-send, which fills the conn's read buffer, which the relay eventually feels). Adding a buffered channel would require a drop policy this layer has no signal to choose — the dispatcher (#248) consumes envelopes ordered by `id`, so dropping a frame would break the protocol's monotonic-id invariant. Defer until there's a measured need.
- **Backoff is a `*Client` method with a per-Client RNG.** `(*Client).backoff(attempt int) time.Duration` computes `reconnectInitial << (attempt-1)` capped at `reconnectMax`, then applies `0.8 + 0.4*rng.Float64()` (±20% jitter). `rng` is unexported `*rand.Rand` seeded from `time.Now().UnixNano()` in `New`; tests substitute a deterministic seed via `newClientForTest`. `rngMu` guards the RNG (Go's `math/rand.Rand` is not concurrent-safe; production access is single-goroutine from the dial loop, but the mutex is cheap and pins the contract). `math/rand` over `crypto/rand` is intentional — jitter is anti-thundering-herd noise, not adversary-resistant randomness.
- **Stability reset is structural to the dial loop.** `Connect` records `connectedAt := time.Now()` on each successful dial, runs `serve(ctx, conn)`, computes `uptime := time.Since(connectedAt)` on return, and **resets `attempt = 1` iff `uptime >= stabilityReset`** else increments. Tests pin both directions: `TestBackoff_Sequence` covers the pure cadence (attempts 1..10, base ∈ `[base*0.8, base*1.2]`); `TestBackoff_ResetAfterStableConnection` drives the dial loop with a synthetic `dialFn` (2 failures → 1 success held past `stabilityReset` → ForceClose → assert the 4th attempt's gap is consistent with attempt-1 backoff, not attempt-4).
- **`dialFn` is a function-typed injection seam on `*Client`.** Production wires `c.dialFn = c.realDial` (which calls `websocket.Dial` + applies `SetReadLimit(maxFrameBytes)`); tests substitute a closure that synthesises dial failures or dials a real `httptest`-hosted relay without going through DNS / TLS. Function-typed seam over interface-based dial-strategy because there's exactly one production implementation and one test variant per test — same pattern as `updateOptions.runRestart` / `updateOptions.probeRestart` (#190).
- **Per-frame `WriteTimeout`, no `ReadTimeout`.** `Config` exposes only `WriteTimeout time.Duration` — bounds each `sendPump` write via `context.WithTimeout`. `ReadTimeout` from the AC body was **dropped** during architect's security review: the inactivity contract is the ping/pong heartbeat (a stuck conn fires pong-timeout within 60s worst case, which cancels the ctx the recv-pump shares, which unblocks `conn.Read`). A separate `ReadTimeout` would shadow the heartbeat and produce confusing failure modes if set lower than `pongTimeout`. `Config` shape is intentionally minimal: `URL string`, `Headers http.Header`, `WriteTimeout time.Duration`, `Logger *slog.Logger` (required, nil panics at `New` time).
- **Test surface: nine tests against a single `httptest.NewServer` relay helper.** `newTestRelay(t)` stands up a `coder/websocket.Accept` upgrader with hooks for ping-counting (`OnPingReceived` callback returns `false` to suppress pong), forced-close (`ForceClose` calls `conn.CloseNow` on every tracked conn), and an echo loop. `relayCtrl.connectedCh` / `connCount` / `pingCount` expose observable state for assertions. `TestBackoff_Sequence`, `TestBackoff_ResetAfterStableConnection`, `TestPing_FiredAt30s` (skipped under `-short`), `TestPongTimeout_TriggersReconnect`, `TestClose_OnContextCancel`, `TestClose_Idempotent`, `TestSmoke_HttptestEchoServer`, `TestSend_ReturnsErrNotConnected_BeforeConnect`, `TestSend_ReturnsErrClosed_AfterClose`, `TestReceive_ReturnsErrClosed_AfterClose`, `TestNew_PanicsWithoutLogger`. All `t.Parallel()`.
- **Close-code on reconnect is `1011` (internal error), not `1006`.** WebSocket `1006` is received-only — it indicates the conn dropped without a close frame, the client cannot actively send it. The spec's "`1011` or `1006`" wording is a description of the two observable peer states, not a choice; this client sends `1011` on graceful close after pong timeout / serve return.
- **Inherits Go's `crypto/tls` defaults** via `coder/websocket.Dial` → `http.DefaultClient` → `http.DefaultTransport`. TLS 1.2 minimum, modern cipher suites, hostname verification against the URL's host. No `MinVersion: tls.VersionTLS13` override yet (spec § TLS says "standard TLS verification"); a future hardening pass can pin TLS 1.3 once relay support is confirmed.

## Out of scope (deferred)

- **WS handshake protocol semantics** (`hello`/`hello_ack`, server-id negotiation, role-based type restriction) — #248.
- **`protocol.Envelope` marshal/unmarshal** — #248 wraps this client's `[]byte` Send/Receive with envelope I/O.
- **Close-code interpretation** (`4409` server-id conflict, `4401` auth invalid, `4404` no server) — recv-pump reports a generic Read error on any close; the dispatcher decides retry semantics. v1: just reconnect on any drop.
- **`Connected() <-chan struct{}` signal channel** — deferred to #248 once consumer reaction shape is known (closed-channel-per-conn vs. signal-on-channel vs. callback). Naming it pre-emptively would commit to a shape blindly. #248 can build "wait for conn" on top of the existing `Send` returning `ErrNotConnected`.
- **TLS pinning, custom verifier, `InsecureSkipVerify`** — spec § TLS explicit; v2-or-later.
- **Connection multiplexing** — one Client = one conn.
- **Per-message metrics, frame counters, latency histograms** — observability ticket separate.
- **`per-message-deflate` compression** — not in v1 wire spec.
- **`Send(ctx, frame)` variant** — `Send` already exits via `closeCh`; ctx variant adds no signal.
