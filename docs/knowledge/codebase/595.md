# Ticket #595 — Rewire `pyry attach` onto the tui-driver mirror; phone + local terminal coexist (two heads)

Locks the **two-heads coexistence invariant** behind deterministic tests and records the Phase-1
input expectation. After [#593] the supervisor hosts claude through a [tui-driver](https://github.com/pyrycode/tui-driver)
`Session` and the existing `Bridge` already fans `Session.MirrorOutput()` → `Bridge.Write` → **both**
the local attach head (`b.output`) **and** the phone observer head (`b.outputObserver`, set by
`cmd/pyry/assistant_turn_v2.go`); after [#594] the phone's input path is reliable. So both heads
*already function* — #595 does not re-plumb anything. Its job is to prove, deterministically, that a
local `pyry attach` and a connected phone observe and drive the **one** session at once without
corrupting each other, and to make the Phase-1 "no arbitration, no echo ownership" input contract
explicit. This is task **T4** of Phase 5 / Phase 1 in [ADR 025](../decisions/025-mobile-remote-head-interactive-session.md).
Architect override **S → XS**: the production delta is one doc-comment; everything else is tests.

Not `security-sensitive` (confirmed, no label): behaviour-preserving plumbing of the **local, trusted**
attach head, adding no untrusted-input handling. Phone confidentiality (raw screen bytes must never
reach the phone) is realised in [#596]; two-heads permission ownership / first-answer-wins is [#597].

## Implementation

### The one production artifact: name the two-heads model (`bridge.go`)

The only production change is a ~8-line doc-comment on the `Bridge` struct's output fields
(`internal/supervisor/bridge.go:61`), naming the model the tests pin:

> `output` is the local attach head — at-most-one, guarded by `attached` (`ErrBridgeBusy` on a second
> `Attach`); `outputObserver` is the phone observer head — a non-owning tap set by
> `assistant_turn_v2.go`. `Write` fans the same bytes to both. The two seams are independent: neither
> reads nor mutates the other's field, so neither head can corrupt the other. Attach is **not**
> re-seated onto a separate mirror seam — this shape *is* the intended two-heads surface.

The field-comment renames (`output` → "local attach head", `outputObserver` → "phone observer head")
make the two seams self-describing at their declaration. No logic, no new types, no signature change.

### Recorded decision: the existing two-seam shape IS the Phase-1 two-heads model

The ticket asked whether attach should re-seat onto a dedicated mirror seam separate from the phone
observer. **It should not.** The two seams are already independent by construction — `Bridge.Attach`
sets `b.output` (at-most-one), `Bridge.SetOutputObserver` sets `b.outputObserver` (non-owning tap),
and `Bridge.Write` snapshots both under `b.mu`, invokes the observer first, then the attached writer,
with the same bytes. Re-seating would be a refactor with **zero behavioural change** — output already
reaches both heads. Per Simplicity-First + Evidence-Based Fix Selection (no observed failure justifies
the re-plumb), the decision is recorded in the doc-comment, not in code.

### AC1 — local attach round-trips against the mirror (e2e regression lock)

`internal/e2e/attach_pty_test.go`'s `TestE2E_Attach_RoundTripsBytes` already runs against the
#593-hosted supervisor and passes; #595 adds a doc-comment naming it the AC1 regression lock — a byte
written to the attach client's PTY master is forwarded as raw input to claude **and** claude's mirror
output (`Session.MirrorOutput` → `Bridge.Write`) renders back to the same master. With the echo helper
the two directions are one observable event, so a single round-trip proves both. The across-restart
variant is `TestE2E_Attach_SurvivesClaudeRestart`. No test rewrite — the smaller change per the spec's
open question.

### AC2 — output coexistence (unit, `bridge_test.go`)

Three new tests fill the gap the existing single-seam tests miss (none set **both** seams):

- **`TestBridge_OutputCoexistence_BothHeadsReceiveSameBytes`** — with `b.output` (via `Attach`, input
  pump parked on an `io.Pipe` so the writer stays bound) **and** `b.outputObserver` both set,
  `Write("alpha")` then `Write("beta")` → the attached writer holds `"alphabeta"` AND the observer was
  invoked with `"alpha"` then `"beta"`, byte-identical. (Local attach unaffected by an active phone
  observer; phone fan-out keeps delivering while a local attach is bound — both directions of AC2.)
- **`TestBridge_OutputCoexistence_FaultingLocalSinkDoesNotStarveObserver`** — an always-erroring
  attached writer + a healthy observer → `Write` swallows the local error (returns `n == len(p)`) and
  the observer still receives the chunk intact. A mid-detach attach cannot starve the phone.
- **`TestBridge_OutputCoexistence_ObserverCopyContract`** — one caller buffer reused across `Write`s
  (overwritten in place between calls, exactly the supervisor's `MirrorOutput → Write` shape) → a
  correctly-copying observer's recording is stable across the reuse. Pins the emitter's copy contract
  the fan-out relies on.

These are **pure `Bridge`-level** invariants — no claude, no relay, no `WriteUserTurn`.

### AC3 — input coexistence + the Phase-1 expectation (integration, `supervisor_test.go`)

`TestSupervisor_InputCoexistence_BothHeadsReachChildContiguously` drives a real `Supervisor` in
service mode (a `Bridge` + a fake child via `helperConfig` + `TestHelperProcess` mode
`stdin_to_file`, which records the child's PTY input to a file). It injects a `deliverFn` that writes
straight to `Session.AttachInput` — **bypassing #594's `WaitReady` idle-gate** so a phone turn commits
without a live claude — then races a local keystroke marker (`Bridge.Attach`'s reader → `Bridge.Read`
→ `sessionWriter` → `AttachInput`) against a phone-turn marker (`WriteUserTurn` → injected `deliverFn`)
into the one PTY. Assertions: both markers appear in the child's stdin log, **each as a contiguous
substring** (turn integrity — neither head's single write split by the other), and
`CurrentConversation()` reflects the phone turn (the local path never touches `convMu`). Runs under
`-race`: the detector is the evidence that the two input paths' supervisor-level bookkeeping
(`convMu`, `sessMu`, the `Bridge` channel) shares no mutable state and has no data race.

The Phase-1 expectation is made **explicit** in the test's doc-comment: the two heads share one input
stream with **no arbitration and no echo ownership**. Each *single* PTY write is atomic against the
other (serialized by `*os.File`'s `fdMutex`); multi-write delivery sequences may interleave at sub-turn
granularity, which is acceptable in Phase 1 (one human, not two heads typing the same instant) and is
arbitrated in [#597]. The test deliberately asserts **neither** an ordering between the two markers
**nor** any echo ownership.

## Patterns established

- **Lock the invariant where the fan-out actually happens, not at the daemon edge.** The two-heads
  coexistence is proven deterministically at the `Bridge` (AC2) and at the supervisor's `deliverFn`
  seam (AC3), not via a flaky live-claude daemon e2e. The injected `deliverFn` lets a phone turn commit
  without `WaitReady`, so coexistence is observable with synthetic markers and zero claude-screen
  literals — keeping `cmd/substrate-guard` green by construction (AC4) and the test the *better fabric*
  than the #603-blocked fakeclaude e2e.
- **Record the no-re-plumb decision in a doc-comment, not a refactor.** When the right answer to "should
  we re-seat this onto a dedicated seam?" is "no — the existing shape already is the model," the
  artifact is a named invariant at the declaration site, not behaviour-neutral churn (Simplicity-First
  + Evidence-Based Fix Selection).
- **No PTY-write mutex added (evidence-based).** No interleaving failure has been observed; the only
  correct place for such a lock is tui-driver (it owns `s.pty`), not pyrycode; and two-heads
  arbitration is [#597]'s job. Shipping a lock now would defend an unobserved failure mode — deferred.

## Lessons learned

- **`PhaseRunning` is set in `onSpawn`, which `runOnce` calls *before* `setSession`** — so
  `waitForPhase(PhaseRunning)` does **not** guarantee a registered session. A test that drives
  `WriteUserTurn` right after must gate on `WaitForPTY`, not just the phase.
- **Input-coexistence turn integrity is observable deterministically without live claude.** An
  injected `deliverFn` that writes straight to `Session.AttachInput` (bypassing the `WaitReady`
  idle-gate) lets a phone turn race a local attach keystroke into the one PTY, and the `stdin_to_file`
  helper log proves each single PTY write lands contiguous — the better fabric than a fakeclaude e2e
  (which is [#603]'s job).

## The #603 boundary (scope fence)

The daemon-level two-heads e2e (real fan-out + real `send_message` ack, both heads live) is **deferred
to [#603]**: `internal/e2e/internal/fakeclaude` renders no claude TUI glyphs, so #594's `WaitReady`-gated
`WriteUserTurn` cannot confirm a commit against it (the `relay_v2_daemon_test.go` phone-echo oracle and
four siblings remain `t.Skip`-ped on #603). Teaching fakeclaude those glyphs needs a `cmd/substrate-guard`
allowlist exemption — a seal decision owned by #603. **#595 does not un-skip, re-enable, or modify those
tests**, and proves coexistence at the seam instead. The full live-claude two-heads gate is verified
**manually** per ADR 025 § Verification, not by a fakeclaude e2e in this ticket.

## Out of scope (deferred to siblings)

- **[#596] (Phase 2)** — typed event stream / phone-confidentiality conversion (raw chunks → typed wire
  events). #595 **must not** touch `assistant_turn_v2.go`'s coarse #589 fan-out: removing it before
  #596 lands would make the phone go dark and break the Phase-1 gate.
- **[#597] (Phase 3)** — two-heads permission/modal ownership, first-answer-wins, `interrupt`/`SendEsc`,
  and any input arbitration between the heads.
- **[#603]** — fakeclaude TUI glyphs / seal allowlist / restoring the skipped `send_message` e2e.
