# Ticket #482 — agent-run: ptyrunner ↔ streamrunner real-claude byte-equivalence smoke

Adds two real-claude e2e tests under `internal/e2e/realclaude/` that guard the dispatcher-visible wire-shape equivalence between the two claude-spawn primitives: [`internal/agentrun/ptyrunner`](https://github.com/pyrycode/pyrycode/issues/471) (interactive-TUI, default post-[#470](https://github.com/pyrycode/pyrycode/issues/470)) and [`internal/agentrun/streamrunner`](https://github.com/pyrycode/pyrycode/issues/390) (stream-json subprocess, retained as the `PYRY_USE_STREAMJSON=1` rollback baseline). Sub-issue of [#329](https://github.com/pyrycode/pyrycode/issues/329) (tracking). Siblings: [#479](https://github.com/pyrycode/pyrycode/issues/479) (budget + watchdog, blocker), [#478](https://github.com/pyrycode/pyrycode/issues/478) (JSONL tail + emit), [#469](https://github.com/pyrycode/pyrycode/issues/469) / [#475](https://github.com/pyrycode/pyrycode/issues/475) / [#476](https://github.com/pyrycode/pyrycode/issues/476) (trust + settings helpers), [#470](https://github.com/pyrycode/pyrycode/issues/470) (cmd/pyry cutover, already on main). The ticket body framed #482 as a pre-cutover gate, but the cutover landed first (operator decision: the unit-test coverage on #478/#479 was strong enough that the empirical-validation gate could land in parallel rather than block). The test remains load-bearing as a regression baseline against future drift in either runner. Scope stays surgical: one new file, ~460 LOC, zero edits to any production code path.

## Implementation

- **One new file, two test functions, package-private helpers only.** `internal/e2e/realclaude/ptyrunner_byte_equivalence_test.go` (~457 LOC) holds both tests; both are gated by the established `//go:build e2e_realclaude` tag so default `go test ./...` skips them and `make e2e-realclaude` opts in. No new exported identifiers, no new fixture helpers in `fixtures.go`, no new testdata. The argv-flags test sits first in the file so contributors without `ANTHROPIC_API_KEY` can verify the cheaper half locally (`TestPtyRunnerVsStreamRunner_StructuralEquivalence` `t.Skip`s via `WithWorktreeAuthenticated` when the key is absent; the argv-flags test runs unconditionally with only the `claude` binary on PATH).

- **`TestPtyRunnerArgvFlagsExistInClaudeHelp` runs `claude --help` once and asserts every flag `ptyrunner.buildArgs` emits is documented.** The 10s `context.WithTimeout` wraps the `claude --help` invocation; the `exec.LookPath("claude")` gate at the top mirrors `smoke_test.go:12-24`'s install-or-adjust-PATH message verbatim. The literal flag list `[]string{"--session-id", "--settings", "--permission-mode", "--append-system-prompt-file", "--model", "--effort"}` is the SAME six flag names `ptyrunner.buildArgs` returns at `internal/agentrun/ptyrunner/runner.go` lines 395-404 — a block comment above the slice pins this and points at the source-of-truth line range. Drift in either direction (a flag renamed in `buildArgs` without updating the test, OR a future claude release dropping one of the flags) fails the assertion. The AC offered a choice between pinning a known-good claude version (2.1.144) and pinning the flag set; the flag-set pin is more durable across upgrades — what we actually care about is "does buildArgs's argv still produce a recognised claude invocation," not "is claude exactly at version X."

- **One flag-to-shorthand alternates table absorbs a documentation-only drift discovered on the first run.** As of claude 2.1.144, `--help` documents `--append-system-prompt-file` only via the bracketed shorthand `--append-system-prompt[-file]` (meaning both `--append-system-prompt` and `--append-system-prompt-file` are valid invocations, but the literal `--append-system-prompt-file` token is absent from the help text). The flag works at runtime — every existing realclaude test that goes through streamrunner argv passes it — so the check accepts either form via `ptyRunnerArgvFlagAlternates map[string]string{"--append-system-prompt-file": "--append-system-prompt[-file]"}`. A future claude release dropping BOTH forms still trips the assertion (the contains-check tries the canonical form, then the alternate, then fails). The alternates table is the documented escape hatch for help-text shorthand variations, not for runtime flag renames — if claude actually renames a flag, both branches fail and the test forces a `buildArgs` audit.

- **`TestPtyRunnerVsStreamRunner_StructuralEquivalence` runs both pipelines sequentially against real claude with the same prompt + budget.** `WithWorktreeAuthenticated` pins HOME to a per-test `t.TempDir()` and `t.Skip`s on missing `ANTHROPIC_API_KEY`; two workdirs `home/stream-work` and `home/pty-work` live under the pinned HOME so each runner's `~/.claude/projects/<encoded>/<sid>.jsonl` lands in its own subtree. Each runner gets its own 90s `context.WithTimeout` (sequential, not shared, so a slow first run can't eat the second's budget). Shared inputs: `model = "claude-haiku-4-5"`, `effort = "low"`, `maxTurns = 1`, `allowedTools = ["Read"]`, `promptText = "Reply with the single word OK and nothing else."`, identical `systemText` written once to `home/system.txt` and pointed at by both runners. Haiku + low-effort + max-turns=1 minimises both wall-clock (~30-60s total, two 5-15s real turns) and the surface area where LLM stochasticity can shift the type sequence.

- **The two runs are sequential, not parallel.** A shared `~/.claude.json` would see two atomic writes from competing trust pre-marks; sequential simplifies the test and avoids racing on the trust file. ptyrunner pre-marks trust on `workdirPty` only (the streamrunner path uses `--dangerously-skip-permissions` and doesn't touch `~/.claude.json`). The trust pre-mark returns `realpath`, which becomes `ptyrunner.Config.WorkDir` directly — handing claude a symlinked path when the trust pre-write keyed the realpath would miss the modal-check and re-render the trust modal (codebase/470.md pins this constraint). `HomeDir` on the ptyrunner Config is intentionally left empty — `WithWorktreeAuthenticated`'s `t.Setenv("HOME", home)` already pins `os.UserHomeDir()` so the JSONL watcher reads from `home` exactly as in production; setting `HomeDir` would test a different code path.

- **`streamRunnerArgs` is a file-local 12-line helper that mirrors `cmd/pyry/agent_run.go:buildStreamRunnerClaudeArgs` verbatim.** `cmd/pyry` is `package main` so the helper is not importable; a block comment above the local copy names the source-of-truth file and explains the duplication. If either side drifts (the cmd-side helper rename, the test-side copy stale-bit-rots), the byte-equivalence test fails on the next run because the streamrunner side stops emitting compatible argv. The architect explicitly de-scoped refactoring `cmd/pyry/agent_run.go` to export the helper — that's scope creep and the local helper serves a different purpose (test-local pinning vs runtime argv construction).

- **`extractShapes` reads ONLY `type` + `subtype` per envelope.** The shape comparison never materialises a normalised form of either stream; it walks newline-delimited input, unmarshals each non-empty line into `struct { Type, Subtype string }`, appends an `envelopeShape{Type, Subtype}`. Malformed JSON returns an error with the offending line embedded (256-byte truncated). The normalisation table in the file-doc comment is for the *reader* (and for future debugging when the test fails) — the assertion code itself never strips fields from raw bytes. Field-level invariants 7-10 use targeted per-field decodes; `bytes.Contains` for the prompt-text check.

- **`compareShapes` asserts eight things in order with a single fatal-formatter.** (1) `len(streamShapes) >= 4` (init+user+assistant+result minimum). (2) `len(ptyShapes) >= 4`. (3-4) Both first elements: `Type=="system" && Subtype=="init"`. (5-6) Both last elements: `Type=="result"`. (7) The two `result` subtypes agree (success vs error_max_turns — whichever the LLM produced, both pipelines should agree). (8) Element-by-element `reflect.DeepEqual` of the full sequences with a first-divergent-index hunt on mismatch. The `fail` closure is the only fatal-exit path: it prints both shape sequences pretty-formatted (`[%2d] %-15s %s` per line), 1024-byte truncated prefixes of both raw stdouts, AND both stderr buffers — a regression is diagnosable from a single test-output read with no `-v` re-run needed.

- **Field-level invariants 7-10 use `t.Errorf`, not `t.Fatalf`, so all four run on a single failure.** `checkInitModel` (both streams' first `system/init` line carries `model=="claude-haiku-4-5"`), `checkUserContains` (both streams' first `user` line contains the prompt text — `bytes.Contains` is robust to content-string vs content-array shape variation since the prompt has no JSON-escape-sensitive characters), `decodeResultTrailer` (both streams' `result` trailer agrees on `is_error` AND `num_turns`). Decoding the trailer into the test-local `byteEquivResultTrailer{IsError bool, NumTurns int}` is enough for the cross-stream comparison; the much-larger field-set delta between the two pipelines (api_error_status, modelUsage, permission_denials, fast_mode_state, uuid, usage.server_tool_use — all in streamrunner's pass-through, all deliberately omitted by ptyrunner's `streamjson.Emitter` per `internal/agentrun/streamjson/emitter.go:251-273`) is documented in the normalisation table and intentionally NOT compared.

- **`Stderr` is captured into per-run `bytes.Buffer`s, not `io.Discard`.** The architect spec offered either; capture-on-failure was the pick because the failure message benefits from having the stderr in-band — operators reading a CI test-output don't have a separate stderr stream to consult. On the success path the buffers are unread (no `t.Logf` call); on failure the assertion includes them in the `fail` closure's output. The cost is one bytes.Buffer per run that the GC will reap; the benefit is single-read diagnosability of network/auth/exec failures.

- **`byteEquivResultTrailer` carries a deliberate test-local prefix to avoid a `go vet` collision.** The realclaude package already defines a `resultTrailer` type in `tool_loop_test.go` for a different cross-stream comparison; without the prefix, the second definition triggers a duplicate-declaration error in the same `package realclaude` build unit (both test files share the `e2e_realclaude` build tag). Prefixing the new type with the test name (`byteEquivResultTrailer`) keeps both alive without renaming the older identifier or factoring out a shared type that neither test needs. Generalises: when adding a `_test.go` file to a package that already has typed helpers, audit for collisions in the same build tag.

## Patterns established

- **Empirical wire-shape equivalence as a regression baseline for parallel spawn primitives.** When two paths produce the same dispatcher-visible wire shape from different internals (here: stream-json from claude's stdout vs synthesised stream-json from a JSONL tail), pair the unit-test coverage with a single real-claude e2e test that runs both pipelines with shared inputs and asserts structural equivalence at the dispatcher boundary. The unit tests prove each side internally; the e2e test catches drift between the sides that unit tests can't see. Generalises whenever a runtime path forks into multiple implementations that share a downstream contract.

- **Structural-equivalence over byte-equivalence when field sets differ by design.** When two pipelines emit the same wire SHAPE but different field SETS (here: streamrunner forwards claude's native trailer verbatim, ptyrunner's emitter declares a strict subset), comparing bytes would require either deleting fields from one side post-hoc or synthesising them on the other — both are wrong. Extract the shape sequence (here: `(type, subtype)` pairs per envelope) and assert element-by-element equality, then pin a small set of dispatcher-visible content invariants (model, prompt text, error flag, turn count) via targeted decodes. The test never materialises a normalised form of either stream; the normalisation table in the doc-comment is for the reader, not for the assertion. Generalises whenever a wire-shape comparison spans two implementations with disjoint-by-design field sets.

- **Pin the flag SET, not the binary version, for argv-vs-help drift detection.** When asserting that an external binary still recognises a wrapper's argv shape, walk the wrapper's known flag list against the binary's help output rather than pinning a known-good binary version. Flag renames in either the wrapper or the upstream binary are what the test should catch; version pins go stale every release and convert maintenance into a chase. Pair the literal flag-list with an alternates map for documentation-only shorthand variations (e.g. claude's `--append-system-prompt[-file]` bracketed shorthand) so the test survives upstream help-text reformats but still trips on actual flag removals. Generalises whenever a test guards an argv-shape contract against an external CLI.

- **`fail` closure as a single fatal-formatter for diagnostic-heavy assertions.** When an assertion has many failure modes that all want to print the same diagnostic context (here: both shape sequences + both stdout prefixes + both stderr buffers), declare a single `fail func(reason string)` closure inside the assertion function that composes the message once and routes every failure mode through it. The body becomes a sequence of `if !invariant { fail(reason) }` checks and the test-output format stays uniform across all failure modes. Generalises whenever an e2e assertion has more than two ways to fail and the operator-facing diagnostic context is the same in each case.

## Lessons learned

- **`resultTrailer` already exists in `tool_loop_test.go` in the same realclaude package.** First draft of the result-trailer decode helper hit a duplicate-declaration error at `go vet` time because both `_test.go` files share the `e2e_realclaude` build tag and the `package realclaude` namespace. Resolved by prefixing the new type with the test-name (`byteEquivResultTrailer`) — cheaper than renaming the older identifier or factoring out a shared type that neither test independently needs. Pattern reinforced: when adding a `_test.go` file to a package that already has typed helpers, run `go vet -tags=<build-tag> ./internal/e2e/realclaude/...` BEFORE committing the new file and audit for symbol collisions in the same build-tag bucket.

- **`claude --help` has begun documenting `--append-system-prompt-file` only via the bracketed shorthand `--append-system-prompt[-file]`.** The first run of `TestPtyRunnerArgvFlagsExistInClaudeHelp` failed because the literal token `--append-system-prompt-file` is absent from claude 2.1.144's help output — the help text uses the canonical-and-aliases shorthand syntax. The flag works at runtime (every existing realclaude test going through streamrunner argv passes it), so the fix is a small alternates table in the test that accepts either form. Future argv-shape tests over `claude --help` text need this kind of alternates table for documentation-only shorthand variations. The flag still works at runtime — the alternates table is the right granularity (per-flag override, not a blanket relaxation) because it documents WHICH flag has a known shorthand alternative and leaves the rest strict.
