# Ticket #696 â€” per-conversation scratch `Cwd`: expand `~` to `$HOME` and create the dir before spawn

The mobile rung-3 live e2e (pyrycode-mobile#421) surfaced two daemon bugs in #685's per-conversation spawn-dir resolution. A phone cannot know the daemon's absolute home, so it sends the default conversation `Cwd` as `~/.pyrycode/scratch` meaning "the daemon's home". `resolveSpawnDir` (the `create_conversation` phone path) rejected it twice over: (1) it never expanded `~`, so `filepath.Abs("~/.pyrycode/scratch")` produced `<cwd>/~/.pyrycode/scratch` and `lstat`'d a literal `~` segment under the process cwd; (2) even with `~` expanded, `confineWorkdirToHome` canonicalises with `EvalSymlinks`, which fails for a path that doesn't exist yet, and nothing created the scratch dir â€” a first-time user has no `~/.pyrycode`. Both rejected as `protocol.malformed`, so the thread never opened.

#696 makes the default-scratch `Cwd` **resolve under `$HOME` and be created before spawn**, deliberately reversing #685's then-correct "a non-existent requested path is rejected" posture *for this case only*. Confirmed live: a local build expanding `~` plus a manual `mkdir -p ~/.pyrycode/scratch` cleared both and #421 went green. #670 (trust pre-mark) and #671 (reply bridge) are confirmed working in the same run; these were the next blockers behind them.

See the evergreen feature doc: [features/conversation-session-binding.md Â§ `Cwd` is the validated, trust-marked spawn workdir](../features/conversation-session-binding.md#cwd-is-the-validated-trust-marked-spawn-workdir-685).

## Implementation

2 production-relevant files: `cmd/pyry/main.go` (+~115 LOC, two new helpers + `resolveSpawnDir` wiring) and `cmd/pyry/conversation_spawndir_test.go` (+159 LOC, 5 new cases). No new types, no cross-package work, no protocol change. (An incidental `internal/e2e/realclaude/testdata/permission_protocol_v2.1.158.json` churn â€” temp-path/UUID drift from a real-claude e2e re-run captured by the code-review auto-commit â€” is unrelated to the fix.)

The whole change lives inside `resolveSpawnDir`. **`confineWorkdirToHome` is left byte-for-byte unchanged**, and so is the daemon-startup caller (`runSupervisor`, `main.go:547`): only the phone path receives a `~`-prefixed / not-yet-existing value, and the operator's `-pyry-workdir` is shell-expanded and must already exist (a missing daemon workdir should stay a loud startup failure). Two new unexported helpers, both siblings of `confineWorkdirToHome`/`withinDir` and consumed only by `resolveSpawnDir`:

- **`expandTilde(p string) (string, error)`** â€” minimal, leading-only home expansion. `p == "~"` â†’ `os.UserHomeDir()`; `"~/"`-prefix â†’ `filepath.Join(home, p[2:])`; anything else (absolute, relative, `~user`, `~foo/bar`) â†’ returned **verbatim**; `UserHomeDir` failure â†’ error. No `$VAR` / arbitrary env expansion (out of scope, larger attack surface). `~user` is intentionally *not* resolved to another user's home â€” it passes through as a literal segment and fails the later confinement/existence check as a deterministic reject. Pure string + `filepath.Join`; no shell interpretation.

- **`confineWorkdirToHomeCreating(workdir string) (string, error)`** â€” the create-aware variant of `confineWorkdirToHome`. Same return contract (canonical `$HOME`-confined realpath on success), but tolerates a not-yet-existing leaf/parents by canonicalising the **longest existing ancestor** and creating the rest *only after* the `$HOME` check passes:
  1. `homeReal = EvalSymlinks(os.UserHomeDir())`; `absWork = filepath.Abs(workdir)`.
  2. Split `absWork` into `(existing, rest)` â€” `existing` is the longest leading ancestor that exists by **`os.Lstat`** (a symlink counts as existing, so it is resolved, not stepped over); `rest` is the not-yet-existing suffix (`""` when the whole path exists).
  3. `existingReal = EvalSymlinks(existing)`; `candidate = filepath.Join(existingReal, rest)` â€” the only unresolved part is `rest`, which doesn't exist yet so contains no symlinks.
  4. **Containment check #1 (pre-creation):** `if !withinDir(homeReal, candidate)` â†’ reject with the same content-free `"â€¦ resolves outside the home directory â€¦"` error `confineWorkdirToHome` returns. **No directory is created on this path.**
  5. If `rest != ""`: `os.MkdirAll(candidate, 0o700)` (operator-private). A failure (file in the way, EACCES) â†’ reject.
  6. **Containment check #2 (post-creation re-confine):** `final = EvalSymlinks(candidate)` (now exists) and re-check `withinDir(homeReal, final)` â†’ reject if escaping; else return `final`.

  When `rest == ""` (path already exists) this reduces **exactly** to `confineWorkdirToHome`: `existing == absWork`, `candidate == final == EvalSymlinks(absWork)`, no `MkdirAll`. So the AC#4 happy path is byte-identical to today and the 5 existing `resolveSpawnDir` tests pass unchanged (each has an existing or escaping path â†’ `rest == ""`).

- **`resolveSpawnDir` wiring** (`cmd/pyry/main.go:606`). Keeps its three-part contract (`""` â†’ `("", nil)`, no `trustMark`; set â†’ confine+trust; escape â†’ `ErrSpawnDirRejected`; `trustMark` error â†’ plain/retryable). The only change is between the empty-check and the trust-mark: `expanded = expandTilde(requested)` then `realpath = confineWorkdirToHomeCreating(expanded)` (was `confineWorkdirToHome(requested)`). `expandTilde`'s own error (UserHomeDir failure) is also wrapped as `ErrSpawnDirRejected` â€” a deterministic resolve failure, consistent with confine errors.

- **Tests** (`cmd/pyry/conversation_spawndir_test.go`, reusing `installRecordingTrustMark` + `t.Setenv("HOME", â€¦)`, non-parallel):
  - `TestResolveSpawnDir_BareTilde_ExpandsToHome` â€” `~` â†’ `EvalSymlinks(home)`, trustMark called once, no `~` literal survives (AC#1).
  - `TestResolveSpawnDir_TildePrefixMissing_ExpandsCreatesResolves` â€” `~/.pyrycode/scratch` with no `.pyrycode` â†’ expands, creates under `$HOME`, the resolved realpath exists + `IsDir()` + equals `EvalSymlinks(home)/.pyrycode/scratch` (AC#1+AC#2).
  - `TestResolveSpawnDir_SymlinkedAncestorEscaping_RejectedNotCreated` â€” under `$HOME` `link -> outside`, `~/link/scratch` (suffix missing) â†’ `ErrSpawnDirRejected`, trustMark called **0** times, **and `outside/scratch` was never created** (`os.IsNotExist`).   - `TestResolveSpawnDir_DeepMissingChain_CreatesAllParents` â€” `~/a/b/c` all-missing â†’ full chain created `0o700`, resolves under `$HOME`. Guards the longest-existing-ancestor walk past one level.
  - `TestResolveSpawnDir_TildeUser_NotExpanded` â€” `~root/x` â†’ rejected, trustMark 0 times. Pins the leading-`~/`-and-bare-`~`-only expansion boundary.
  - The 5 existing cases (empty, within-`$HOME`, outside-`$HOME`, symlink-under-home-pointing-out, trustMark-error) stay green unchanged â€” they exercise the `rest == ""` reduction.

## Patterns established

- **Don't widen a shared helper for a one-caller bug â€” clone the new behaviour into a path-specific variant.** `confineWorkdirToHome` is shared by the phone path *and* daemon startup, but only the phone path has the bug and only it receives a `~`-prefixed / not-yet-existing value. The create-aware logic went into a new `confineWorkdirToHomeCreating` consumed only by `resolveSpawnDir`, leaving `confineWorkdirToHome`'s 8 callers (2 production, 6 test) and the daemon-startup contract byte-for-byte untouched. Blast radius: one production file + one test file. (Echoes [[architect-trust-premark-is-cmd-layer-job]] â€” keep the spawn primitives dumb and the policy at the caller.)
- **Build the containment candidate from the symlink-resolved ancestor, and check `$HOME` *before* you create.** The naive "containment-check the `filepath.Abs` (un-symlink-resolved) path, then `MkdirAll`" the ticket body sketches is exploitable: `~/link -> /tmp/evil`, requested `~/link/scratch`, the textual path *looks* inside `$HOME`, passes the check, and `MkdirAll` follows the symlink to create `/tmp/evil/scratch`. The fix is that check #1 runs against `existingReal + rest` (symlinks in the existing portion already resolved), so a symlinked ancestor is resolved *out* before the bound and the dir is never created. Same realpath-before-bound discipline `confineWorkdirToHome` already uses.
- **`os.Lstat`, not `os.Stat`, for the longest-existing-ancestor walk (security-load-bearing).** `os.Lstat` reports a symlink itself as existing (without following it), so a symlinked ancestor is selected as `existing`, then `EvalSymlinks` resolves it and check #1 catches the escape. With `os.Stat` a dangling symlink would read as not-existing, the walk would step *over* it, and a textual candidate under `$HOME` could pass check #1 before `MkdirAll` followed the symlink outside.
- **Minimal expansion keeps the attack surface small.** Only leading `~`/`~/` â†’ `$HOME`; no `$VAR`, no `~user`. `expandTilde` does no shell interpretation, so there is no shell-expansion attack surface, and `~user` deterministically rejects rather than silently resolving to another user's home.

## Lessons learned

- **`EvalSymlinks` cannot canonicalise a path that does not exist** (lifted from the PR's Lessons learned). The create-aware confiner resolves the **longest existing ancestor** (probed with `os.Lstat`, not `os.Stat`, so a symlinked ancestor is resolved rather than stepped over) and runs the `$HOME` containment check on `existingReal + rest` **before** `MkdirAll`. A textual abs-path check before creation would let `~/link -> /tmp/evil` create outside `$HOME`. This is why the ticket is `security-sensitive` and why `TestResolveSpawnDir_SymlinkedAncestorEscaping_RejectedNotCreated` asserts the escaping target is **never created**, not merely that the call errors.
- **#696 reverses #685's "non-existent path is rejected" posture â€” but narrowly.** Only the phone path, only the default-scratch case, and only *after* the `$HOME` gate. #685's lesson ([codebase/685.md](685.md) Â§ Lessons learned) that "a non-existent requested path is rejected, same as an escape" no longer holds for `resolveSpawnDir`; it still holds for the daemon-startup `confineWorkdirToHome` caller (unchanged).
- **The confineâ†’`MkdirAll`â†’re-confineâ†’chdir TOCTOU window is the same one #685 accepts; `MkdirAll` does not widen it.** Creation is gated on check #1 (the symlink-present-at-check-time case), and check #2 re-confines the realpath before trust+spawn. Closing the residual race entirely (`openat2`/`RESOLVE_BENEATH`) is out of scope and unobserved â€” an attacker must already control the operator's `$HOME` to win it.

## Out of scope (deferred)

- **Widening `~`-expansion / `MkdirAll` to the daemon-startup path** â€” deliberately *not* done. The operator's `-pyry-workdir` is shell-expanded and must pre-exist; a missing daemon workdir stays a loud startup failure.
- **`MkdirAll`-failure retryability.** A transient `MkdirAll` failure (disk full, EACCES) is classified **non-retryable**, same as every other confine error today. Distinguishing it would need the handler to inspect the error â€” out of this slice's lane. The phone re-issuing the same `Cwd` against a full disk failing identically is acceptable.
- **Recording the realpath vs the raw `Cwd` in the conversation row** â€” unchanged from #685; no consumer needs the row to equal claude's actual cwd.
- **A dedicated `conversation.cwd_rejected` error code** â€” still reuses `protocol.malformed` (a v2 protocol-vocabulary change is out of scope; see [codebase/685.md](685.md)).
