# Ticket #463 — `cmd/pyry`: `pyry rekey <conn_id>` operator verb (manual re-key, CLI-side)

Slice B2 of the [#460](https://github.com/pyrycode/pyrycode/issues/460) split (itself a re-split of #451). [#459](459.md) shipped the control-socket wire contract + `control.Rekey` client helper, [#462](462.md) shipped `(*relay.V2SessionManager).Rekey` and the `emitRekeyRequest(reason)` refactor; this slice ships the operator-facing CLI surface that ties them together. `pyry rekey <conn_id>` dials the control socket via `control.Rekey`, surfaces the result as a one-line stderr message + exit code, and the v2-protocol `payload.reason = "manual"` value (`docs/protocol-mobile.md` § Re-key) becomes operator-triggerable. The verb is operator-facing only — never exposed over the relay wire; filesystem perms on the `0600`-mode control socket are the only authentication. Production wire-up of `*V2SessionManager` into `cmd/pyry` is still a separate ticket (`NewV2SessionManager` has no production caller as of 2026-05-17), so until that lands the verb returns slice A's `rekey: no rekeyer configured` reject — matching the `pair preflight` "shipped ahead of cutover" precedent.

## Implementation

- **One new file + one modified file** — `cmd/pyry/rekey.go` (new, +136 LOC: `rekeyArgs` + `parseRekeyArgs` + `rekeyVerdict` + `runRekey` + private `isServerReject` helper) and `cmd/pyry/main.go` (+5 LOC: one `case "rekey":` dispatch arm next to `case "pair":` and `case "install-service":`, plus one help-text line under `pair revoke`). Tests: `cmd/pyry/rekey_test.go` (new, +307 LOC: parser table-test, verdict table-test, three AC-mandated scenario tests + local `fakeRekeyer` / `rekeyTestResolver` / `startControlServerWithRekeyer` / `shortSockTempDir` fixtures). No new exported symbols.

- **`runRekey` distinguishes three error classes for the prefix shape** — typed `ErrConnNotFound`, known server-side rejects (`"rekey: no rekeyer configured"`, `"rekey: missing connID"`), and everything else (transport). The first two route through `rekeyVerdict` + `os.Exit(1)` so the operator sees a clean `pyry rekey: <message>` line; the third returns `fmt.Errorf("rekey: %w", err)` so main's outer `pyry: ` prefix produces `pyry: rekey: <transport err>`. The split is load-bearing: routing server-side rejects through the wrapped-return path would produce `pyry: rekey: rekey: no rekeyer configured` (double `rekey:`, with the outer `pyry: ` prefix added), but the AC pins the no-rekeyer-reject shape as `pyry rekey: rekey: no rekeyer configured` (single outer prefix, double `rekey:` from the surfaced verbatim server message). See § Lessons learned for why the architect's recommended "Option B everywhere" prose was rejected in favour of the spec's decision-matrix table.

- **`isServerReject` matches by message-prefix because slice A reserves the typed-sentinel slot for `ErrConnNotFound` only.** `control.Rekey` reconstructs the wire `ErrCodeConnNotFound` into the typed `control.ErrConnNotFound`; every other server-side error (the slice A guards, slice B1's `ErrSessionNotOpen`) comes back as a plain `errors.New(resp.Error)` with no `ErrorCode`. The helper recognises the exact strings `"rekey: no rekeyer configured"` and `"rekey: missing connID"`. Not `errors.Is` — the helper exists because the producer side did not opt to define typed sentinels for these guards (slice A's deliberate choice, [#459](459.md)'s spec § "no `ErrCodeSessionNotOpen` defined"). The helper's doc-comment promises that if slice B later defines typed sentinels for the no-rekeyer / missing-connID guards (or the `ErrSessionNotOpen` variant), the body collapses to a couple of `errors.Is` calls. `slice B1`'s [`ErrSessionNotOpen`](462.md) surfaces as `"relay: session not open"` and is **not** matched by `isServerReject` — it currently routes through the wrapped-return path and surfaces as `pyry: rekey: relay: session not open`. The shape is consistent with "untyped server text" until a wire code lands.

- **`rekeyVerdict(connID, err)` is the pure-formatter seam.** Returns `(0, "")` on `err == nil`, `(1, "pyry rekey: conn_id %q not found")` on `errors.Is(err, control.ErrConnNotFound)` (so `fmt.Errorf("wrapped: %w", control.ErrConnNotFound)` flows through with no producer-side cooperation needed), `(1, "pyry rekey: <err.Error()>")` for everything else. Mirrors `preflightVerdict` in `cmd/pyry/pair.go` — extracted so the unit tests can drive the formatter without intercepting `os.Exit`. The `%q` quoting on the not-found path Go-quotes operator-supplied conn-ids containing newlines, ANSI escapes, or other terminal-control bytes (the architect's security review names this as a paranoid-operator defence for piping pyry's stderr into another shell).

- **`runRekey`'s structure** — `parseClientFlags` peels `-pyry-name` / `-pyry-socket`, `parseRekeyArgs` validates the sole positional, `context.WithTimeout(ctx, 30*time.Second)` bounds the round trip (matches `runSessionsRm`'s budget), `control.Rekey` does the dial+encode+decode, then the three-way branch above on the returned error. No new goroutines, no logging — the runner is straight-line foreground code. The 30s context bounds the dial + write + decode; slice B1's `Rekey` returns synchronously once the manager has enqueued, so the timeout is a generous ceiling on the local round trip, not on the actual handshake (the handshake runs asynchronously in the v2 manager's `Run` goroutine).

- **`parseRekeyArgs` mirrors `parsePairRevokeArgs` verbatim** — `flag.NewFlagSet("pyry rekey", flag.ContinueOnError)`, `SetOutput(io.Discard)`, no flags declared on the FlagSet (any `--foo` is unknown by construction), arity rule `0 → "missing conn_id"` / `1 → connID = fs.Arg(0)` / `≥2 → "unexpected positional <…>"`. Parse errors and arity errors flow through `runRekey`'s `os.Exit(2)` + usage-line arm; unknown flags surface the `flag.ContinueOnError` "flag provided but not defined" string verbatim. The "unexpected positional" check (rather than `Args()` length only) names the offending arg so the operator sees which token was extraneous.

- **Help-text placement under `pair revoke`** — `pyry rekey <conn_id> [flags]` lands between `pair revoke <name>` and `install-service`, matching the operator-facing / mobile-protocol-adjacent corridor in the `printHelp` body. Three-line description: trigger an immediate Noise re-key on the named v2 conn (operator rotation; control-socket only). No `--all` / `--dry-run` / confirmation-prompt flags — explicit non-goal per the ticket Out of scope.

- **`Server.SetRekeyer(nil)` is the production "no rekeyer configured" state** — `startControlServerWithRekeyer(t, nil)` in the test fixture exercises exactly that path (the test helper skips the `srv.SetRekeyer(r)` call when `r == nil`, leaving the server's `rekeyer` field as the zero value). This pins the slice A guarantee that the daemon-wire-up gap (no `ctrl.SetRekeyer(v2mgr)` in `cmd/pyry/main.go`) surfaces as `"rekey: no rekeyer configured"` end-to-end through the verb. The test that exercises it (`TestRunRekey_NoRekeyerConfigured_ExitsOnePrintsVerbatimReject`) drives `control.Rekey` directly and feeds the result into `rekeyVerdict` — the seam pattern below.

- **The three AC-mandated scenarios are unit-testable via the verdict seam.** `TestRunRekey_BogusSocket_ReturnsWrappedError` mirrors `TestRunSessions_RmDispatch` verbatim — invokes `runRekey` against a bogus socket path and asserts the returned `err.Error()` contains the `rekey:` substring (no `%v` formatting accident). Calls `runRekey` directly because the transport-error path returns a wrapped error rather than `os.Exit`-ing. `TestRunRekey_UnknownConnID_ExitsOnePrintsTypedMessage` and `TestRunRekey_NoRekeyerConfigured_ExitsOnePrintsVerbatimReject` cannot call `runRekey` directly because the typed-not-found / server-reject paths end in `os.Exit(1)`, which would kill the test process — instead they stand up a real `control.Server` via `startControlServerWithRekeyer`, call `control.Rekey` directly to exercise the real wire round-trip (`Response.ErrorCode = ErrCodeConnNotFound` mapping, sentinel reconstruction, plain-`errors.New(resp.Error)` for the guard), then feed the result into `rekeyVerdict` to assert the formatter output. The wire round-trip IS exercised; only the `os.Exit` final step is bypassed.

- **No `TestHelperProcess`-style subprocess test for `os.Exit(1)` itself.** Verified by inspection — the verdict-helper seam makes the formatting + exit code unit-testable without intercepting `os.Exit`. A subprocess test would add ~40 LOC of harness for one assertion; if the dispatch ever regresses, `runRekey`'s body is < 30 LOC and reviewing the diff is faster than a subprocess assertion. Slice A's `TestServer_Rekey_*` family + slice B1's `TestV2Session_RekeyManual_*` family + this slice's verdict + scenario tests collectively pin the end-to-end shape; the only un-asserted edge is "does `os.Exit(1)` actually fire", which is structurally trivial.

- **Local re-implementation of `internal/control/rekey_test.go`'s fixtures.** `fakeRekeyer`, `rekeyTestResolver`, `startControlServerWithRekeyer`, and `shortSockTempDir` are all defined in `package main`-scoped `_test.go` files in slice A's home package; Go's package-private `_test.go` visibility rule means they cannot be imported across package boundaries. The re-implementation in `cmd/pyry/rekey_test.go` is structurally identical — `fakeRekeyer` is a single-shot mu-guarded stub with a `calls` slice and `returnErr` field; `rekeyTestResolver` satisfies `control.SessionResolver` with `errors.New("not used")` stubs (the resolver path is never exercised by `VerbRekey`); `startControlServerWithRekeyer` skips the `SetRekeyer(r)` call when `r == nil` so the same helper drives both the typed-error and no-rekeyer-configured tests; `shortSockTempDir` uses `os.MkdirTemp("/tmp", "pyryrekey")` rather than `t.TempDir()` because macOS's `/var/folders/...` tempdir paths combined with long test names blow past the 104-byte `sun_path` limit (mirrors `internal/control/server_test.go:shortTempDir`'s exact rationale).

## Patterns established

- **Three-way error routing in a control-socket verb runner.** When a verb has both typed sentinels AND known untyped server-side rejects AND transport errors, the routing splits three ways: typed sentinel → `os.Exit(1)` via verdict helper (clean `pyry <verb>:` prefix), known server reject → `os.Exit(1)` via verdict helper (clean prefix, message surfaced verbatim — produces a double-`<verb>:` when the surfaced text also starts with the verb name), transport error → `return fmt.Errorf("<verb>: %w", err)` (outer `pyry: ` prefix added by main). The split is load-bearing for the operator-readable message shape: routing server rejects through the wrapped path would prepend an outer `pyry: ` that the AC's stderr-format pin forbids. The detection of "known server reject" is by message-prefix today (slice A intentionally reserves typed-sentinel slots for the conn-not-found contract only); if slice B later defines typed sentinels for the other guards, the helper collapses to `errors.Is` calls. Generalises whenever a slice consuming a producer's stable-but-untyped error surface needs to discriminate at the consumer.

- **Pure-formatter seam (`<verb>Verdict`) keeps `os.Exit` paths unit-testable.** A function with signature `<verb>Verdict(args ..., err error) (exitCode int, stderrLine string)` extracts the format-then-decide logic from the runner. The runner calls the formatter, then writes the line and `os.Exit`-s. The unit test calls the formatter directly with a constructed error and asserts on the return tuple — no subprocess, no `os.Exit` interception. The pattern pre-dates this slice (`preflightVerdict` in `cmd/pyry/pair.go` shipped first); this slice adopts it for `rekey`. Generalises whenever a CLI verb has multiple `os.Exit(1)` arms with different formatted messages — extract the formatting tuple and the runner becomes a thin "call → print → exit" wrapper. The formatter's purity also gives `t.Parallel()` for free.

- **Local re-implementation of cross-package `_test.go` fixtures.** When a test in package A needs a fixture (fake, helper, server-stand-up function) defined in package B's `_test.go`, Go's package-private `_test.go` visibility rule blocks the import. The fix is to re-implement the same shape locally in A's `_test.go`; both files defining the same shape is acceptable when the alternative is exporting the fixture from B (which would pollute B's public API with test-only types). The two implementations should be structurally identical so a reviewer can verify by inspection; cross-reference the original file in a doc-comment so the re-implementation isn't mistaken for a divergent fork. Generalises whenever a CLI/wiring package needs to stand up an in-process instance of a lower-level package's server for end-to-end verb tests.

- **Skip the dependency install in a fixture by passing nil through the setter.** `startControlServerWithRekeyer(t, nil)` exercises the "no rekeyer configured" production state without a separate fixture variant. The setter's contract (slice A's `SetRekeyer` doc-comment) explicitly permits clearing-by-nil for tests; production never calls it that way. The fixture's branch `if r != nil { srv.SetRekeyer(r) }` keeps a single helper driving both the typed-error path and the no-rekeyer-configured path, parameterised only on the rekeyer argument. Generalises whenever a setter-installed optional dependency has both an "installed" and a "not installed" production-meaningful state — pass nil to the fixture rather than maintaining two helpers.

## Lessons learned

- **An architect's prose recommendation can contradict the architect's own decision-matrix table.** The spec's § "Transport error path" recommended Option B ("`return fmt.Errorf("rekey: %w", err)` for everything except `ErrConnNotFound`, which uses `os.Exit(1)`") and argued it "still satisfies" the AC because the resulting stderr `pyry: rekey: <err>` "contains the `pyry rekey:` substring prefix." But the AC's explicit pin for the no-rekeyer-reject case is `pyry rekey: rekey: no rekeyer configured` — a SINGLE outer prefix, with the inner-message's own `rekey:` accounting for the second occurrence. Option B would produce `pyry: rekey: rekey: no rekeyer configured` — three `rekey:` segments. The architect's decision-matrix table at the end of the same § correctly listed Option A (os.Exit(1) clean prefix) for the no-rekeyer-reject row, contradicting the prose recommendation immediately above. The developer noticed the contradiction and implemented the table (Option A for server rejects, Option B for transport errors), threading the discrimination through `isServerReject`. The lesson generalises: when a spec's prose and its summary table disagree, the table usually carries the load-bearing detail (it was written more carefully because it had to be exhaustive); trust the table and flag the prose drift as a spec defect rather than reaching for the prose. The architect's prose was a remnant of an earlier draft that had not yet split server-rejects from transport-errors.

- **Message-prefix matching is a legitimate consumer-side workaround for missing typed sentinels.** Slice A intentionally reserved the typed-sentinel slot for `ErrConnNotFound` only — the no-rekeyer-configured and missing-connID guards surface as plain `Response{Error: "..."}` with no `ErrorCode`. The consumer (`pyry rekey`) needs to distinguish these from transport errors to route the prefix correctly. `errors.Is` won't work (no typed sentinel); the wrapped-return path can't be the catch-all (would violate the AC stderr-pin). String matching on the known guard prefixes is the cheapest correct mechanism. The lesson: when a producer-side decision deliberately leaves typed sentinels out, consumers can compensate with prefix-match — but the consumer's `isServerReject` helper should doc-comment the eventual-typed-sentinel path so a future contributor knows the helper collapses to `errors.Is` calls if/when the producer side adds the sentinels. The string-equality posture is not a permanent design choice; it's a bridge until the producer surface evolves. Generalises whenever a stable-text-but-untyped-error surface needs consumer-side classification.

- **`t.TempDir()` is not always Unix-socket-safe on macOS.** The fixture's local `shortSockTempDir` uses `os.MkdirTemp("/tmp", "pyryrekey")` rather than `t.TempDir()` because macOS's `/var/folders/<random>/T/...` tempdirs combined with a long test name (e.g. `TestRunRekey_NoRekeyerConfigured_ExitsOnePrintsVerbatimReject`) can exceed the 104-byte `sun_path` limit, surfacing as `bind: invalid argument` at `net.Listen`. The mirror exists already in `internal/control/server_test.go:shortTempDir`. The lesson generalises: tests that bind a Unix-domain socket on macOS need a short-prefixed tempdir, and `/tmp/<short-prefix>` is the canonical fix. `t.TempDir()` is fine for files and other I/O — only socket paths trip the limit. A future contributor who switches to `t.TempDir()` for "consistency" will see a CI failure on macOS but not on Linux (where `sun_path` is 108 bytes AND the tempdir prefix is shorter `/tmp/...` by default); flag the choice with an inline comment so the rationale survives.
