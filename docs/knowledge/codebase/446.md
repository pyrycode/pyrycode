# Ticket #446 — `internal/relay` v2 `noise_msg` application dispatch + tampered-frame teardown

Fills the open-state `noise_msg` cell of the [v2 transition table](../features/v2-session-manager.md#transition-table) that [#445](445.md) left as `drop (open-state dispatch deferred)`. The slice adds three load-bearing behaviours to `(*V2SessionManager).handleNoiseMsg`'s `V2StateOpen` case: AEAD-decrypt → JSON-unmarshal → dispatch through the existing [`internal/relay/handlers/`](../features/relay-package.md) chain (AC #1); a single new close-code path at 4421 for tampered/replayed/truncated `noise_msg` frames (AC #2); and structural session-entry cleanup on that close path so the next `noise_init` for the same `conn_id` lazy-creates a fresh `awaitingInit` (AC #3). The handler chain in `internal/relay/handlers/` does not change — same shape, same dispatcher, same payloads — only the envelope bytes' provenance changes (`s.recv.Decrypt` instead of straight off the WS). Follow-on to [#445](445.md); preconditioned by [#433](433.md) (`internal/noise` `CipherState`) and [#318](318.md) (`*dispatch.Conn.Auth()`). The production cutover in `cmd/pyry/relay.go` is still NOT in this slice — explicitly deferred to a follow-up gated on [#436](436.md) (release-flag preflight).

## Implementation

- **Two production source files modified** — `internal/relay/v2session.go` (+~85 LOC) and `internal/dispatch/dispatch.go` (+~20 LOC for `Route`/`NewConn`, ~10 LOC mechanical move for `sendError`/`handleOne`). Within the spec's size:S budget (~75–110 LOC production).
- **`dispatchAppFrame(ctx, s, plaintext)`** is the new ~40-LOC function the open-state branch calls after a successful `s.recv.Decrypt`. It allocates a per-frame `outbound chan protocol.RoutingEnvelope` of capacity `handlerOutboundBuf = 8`, constructs a per-frame `*dispatch.Conn` via the new `dispatch.NewConn(s.connID, outbound, s.device)`, calls `dispatch.Route(ctx, m.cfg.Logger, conn, m.cfg.Handlers, plaintext)` synchronously, then drains the channel non-blockingly: `for { select { case env := <-outbound: ...; default: return } }`. For each captured reply: `s.send.Encrypt(reply.Frame)` → `marshalInnerFrameV2(TypeNoiseMsg, ciphertext)` → emit via `m.send`. The reply's `CloseCode` is ignored — handlers do not signal closes through `c.Send`/`c.Reply`; that routing-envelope field is reserved for the manager's own `closeWith` emissions.
- **`handlerOutboundBuf = 8`** is the per-frame channel capacity. The three production handlers (`send_message`, `list_conversations`, `register_push_token`) emit exactly one reply per invocation; `Route` emits at most one error reply (malformed/unsupported/unknown_type/no-handler). 8 is a generous safety margin documented as the synchronous-handler assumption in the field doc-comment on `V2SessionConfig.Handlers`.
- **The per-frame `outbound` channel is intentionally NOT closed** after the drain. A handler that accidentally forks a goroutine retaining `*dispatch.Conn` would write to a closed channel and panic on the sending side; the non-blocking drain over an open channel survives such a misbehaving handler. The leaked channel is bounded by its capacity (8 envelopes) and reclaimed by GC once the forked goroutine exits. This trade-off is documented on `dispatchAppFrame` and on `V2SessionConfig.Handlers`.
- **AEAD-failure teardown reuses `closeWith`** — no parallel cleanup path. `m.closeWith(ctx, s, StatusProtocolMismatch, nil)` already deletes the session from `m.sessions` (`v2session.go` close path from [#445](445.md)); reusing it means AC #3 is satisfied structurally rather than by a hand-written `delete(m.sessions, s.connID)` next to the close. Pinned by `TestV2Session_OpenState_TamperedNoiseMsg_4421`'s post-stop `mgr.sessions[v2TestConnID]` absence assertion.
- **No error text in the `v2.aead.fail` log line.** flynn/noise's `Decrypt` returns errors that may carry counter indices or AEAD-tag positions; these are not operator-actionable and stay out of the log channel per the spec's security review. Logged fields: event class `v2.aead.fail`, `conn_id`, `close_code=4421` only. Mirrors the `MUST NOT log AEAD bytes / counter state` rule from [#445](445.md).
- **"Handler chain unreachable on AEAD failure" is pinned with an `atomic.Bool` side-effect flag** inside `TestV2Session_OpenState_TamperedNoiseMsg_4421`. The test registers a handler that stores `true` when invoked; after the tampered-frame close + cleanup, the test asserts the flag is still false. The spec named this assertion as a separate hypothetical `TestV2Session_OpenState_TamperedFrame_HandlerNotReached`; the developer folded it into the existing tamper test rather than spawning a separate fixture for the same setup. Structurally equivalent, fewer lines, same regression guard.
- **`s.device = &device` is the one-line edit to `handleNoiseInit`.** Captured right before `s.state = V2StateOpen` in the token-accept branch. The `device` local already holds the matched `devices.Device` value returned from `cfg.Devices.Validate`; taking `&device` into a fresh stable pointer is the canonical Go escape pattern. Handler closures consult the matched device via `c.Auth()` exactly like v1 verb handlers do (`internal/relay/handlers/send_message.go` calls `c.Auth()` directly).
- **`V2SessionConfig.Handlers map[string]dispatch.Handler` is optional.** `nil` or empty means no app handlers are registered and every open-state envelope falls through to a sealed `protocol.unsupported` reply via `dispatch.Route`'s "no handler registered" path. Constructor validation needed no new check. Production wires `Handlers` via the daemon (cutover follow-up) — the same handler functions that v1 registers against `Dispatcher.Register`.
- **The handler closure's `c.Auth()` returns `s.device`'s pointer-equal `*devices.Device`.** Same lifetime relative to the first-frame gate as v1's `Conn.auth` slot. Device revocation after handshake does NOT tear down the active conn — same posture as v1, intentional, called out in the security review.
- **`Dispatcher.handleOne` body extracted to the new exported `Route`.** Mechanical refactor (~10 LOC moved). `handleOne` becomes a one-line wrapper: `Route(ctx, d.cfg.Logger, c, d.handlers, routing.Frame)`. Method stays for the `runConn` caller — same signature, same call sites. `Dispatcher.sendError` similarly extracted to a package-private free function `sendError(ctx, logger, conn, ...)` so both `Route` and `runGate` call the same code path. The refactor preserves observable behaviour; existing `internal/dispatch/dispatch_test.go` tests stay green and serve as the regression guard.
- **`dispatch.NewConn` is the production-allowed sibling of `NewTestConn`.** Same implementation (three field assignments), different policy. `NewTestConn`'s doc-comment retains the "Test fixtures only — do not call from production code" restriction; `NewConn`'s doc-comment names the production callers (v2 manager today) and the contract (the caller owns `outbound` and is responsible for draining it). Two constructors exist so a reviewer scanning `cmd/` for `NewTestConn` calls still gets a single-word grep that flags misuse; production callers that need their own per-conn goroutine use the policy-matching constructor.
- **No new ADR.** The architectural decision (Mobile Protocol v2 / Noise_IK) is [ADR 024](../decisions/024-noise-ik-mobile-e2e.md); the open-state dispatch shape is a direct read-off the v1 dispatcher's handler-table semantics, not a new architectural choice. The spec at `docs/specs/architecture/446-v2-noise-msg-application-dispatch.md` captures the seam-fill design and the reuse-vs-duplication reasoning.
- **No `cmd/pyry/relay.go` changes.** Production daemon still constructs `dispatch.Dispatcher` against `/v1/server`. The v2 manager remains reachable through test wiring only. The cutover follow-up re-wires `cmd/pyry/relay.go` to construct `V2SessionManager` instead of `Dispatcher` and registers handlers against `V2SessionConfig.Handlers`; gated on [#436](436.md)'s preflight check.

## Patterns established

- **Re-export the seam, don't duplicate the dispatch logic.** The v1 dispatcher already implemented the malformed → `protocol.malformed`, unsupported → `protocol.unsupported`, unknown-type → `protocol.unknown_type`, no-handler → `protocol.unsupported` ladder. Rather than copy that ladder into v2, the slice extracts the body of `Dispatcher.handleOne` into a free function `Route(ctx, logger, conn, handlers, frame)` that both the v1 dispatcher (now a 1-line wrapper) and the v2 manager call. Same for `sendError`. The pattern: when a second caller needs the same per-frame error-envelope semantics, expose the existing implementation as a free function over a caller-supplied `*Conn` and handler table rather than duplicating it — duplication invites drift (one ladder grows a `protocol.unauthorized` case, the other doesn't). The cost is two new exported functions (`Route`, `NewConn`); the value is one canonical decode-and-dispatch ladder maintained in one place.
- **Two factory functions, same body, different policies.** `NewTestConn` and `NewConn` have identical bodies (three field assignments). The duplication exists for grep ergonomics: a reviewer searching `cmd/**/*.go` for `NewTestConn` calls catches misuse in a single regex, whereas a single dual-policy constructor would force the reviewer to read every call site to determine intent. The pattern generalises whenever a test-only seam acquires a production-allowed sibling: keep the two constructors distinct in name and doc-comment even when the body is identical — the policy boundary is what's load-bearing, not the implementation.
- **Drain a per-frame outbound channel non-blockingly; never close.** The per-frame `outbound chan` in `dispatchAppFrame` is drained via `select { case env := <-outbound: ...; default: return }` after `Route` returns — bounded work, no goroutine blocking on the channel's drain. Crucially the channel is NOT closed: a misbehaving handler that forks a goroutine retaining the `*dispatch.Conn` would panic on a closed-channel send. The leak is bounded by the channel's capacity (8 envelopes) and reclaimed by GC once the goroutine exits. The pattern generalises: when a producer-consumer rendezvous is per-frame (the consumer dies after `Route` returns), the consumer must drain non-blockingly AND must not close the channel — otherwise any forked producer becomes a panic site.
- **Reuse `closeWith` rather than adding a parallel cleanup path.** `closeWith` already emits the close-only routing envelope AND deletes the session entry under the single-dispatch-goroutine ownership contract. AC #3 (local cleanup on AEAD-failure close) is then satisfied structurally by *not adding any new cleanup code*. The pattern: when a new failure mode lands on an existing close-and-clean primitive, route the new path through the existing primitive rather than re-implementing the cleanup at the call site — the cleanup contract stays in one place and the cell of the transition table inherits the existing invariant for free. This mirrors [#445](445.md)'s `closeWith` design intent: the close-and-delete sequence is the atomic unit, not two separate operations.
- **Capture the device snapshot at handshake-time, surface through `*dispatch.Conn.Auth()`.** Same lifetime model as v1's first-frame gate: the `*devices.Device` pointer is set once before any handler-dispatch code runs and reads through the well-known `c.Auth()` accessor. v2's `s.device = &device` in the token-accept branch is the structural mirror of v1's `setAuth` call in `runGate`'s accept branch. The pattern: when adding a second auth-protected dispatch path, mirror v1's "set once on accept, expose through the canonical accessor" shape — handlers don't care whether the auth came from the v1 gate or the v2 handshake; they only call `c.Auth()`.

## Lessons learned

- **The architect spec deliberately chose synchronous handler dispatch despite the head-of-line blocking risk.** A slow handler stalls the manager's dispatch loop for ALL `conn_id`s — worst-case `send_message`'s 30s `Activate` timeout. v1 absorbs this via `Dispatcher.runConn`'s per-conn goroutine fan-out; v2 does not. The lesson generalises: when an incremental slice would land smaller by deferring a known performance follow-up, name the follow-up explicitly in the docs (here: "per-conn fan-out, priority concern before production cutover") AND name what cannot ship until the follow-up lands (here: the `cmd/pyry/relay.go` cutover). Pretending the synchronous shape is fine for production would invite a stalled-binary incident at the worst possible time — the cutover.
- **AC #3 looked like new code; it was actually a test on existing code.** The spec called out "Local cleanup on binary-initiated close" as a third acceptance criterion. The natural reading is "write the cleanup code"; the correct reading is "verify the existing `closeWith` already deletes the session." The spec is explicit about this — "the developer must verify and pin this with a test, not add a parallel cleanup path" — but it would be easy to skim past and end up with a `delete(m.sessions, s.connID)` next to the `closeWith` call. The lesson: when an AC reads as new behaviour but the spec explicitly redirects to "reuse the existing primitive," the test IS the implementation — the absence of new code is the load-bearing artifact. The pattern recurs whenever a new failure mode is described in user-visible terms ("session is cleaned up") but the cleanup primitive already exists; the work is verification, not addition.
- **The fresh-`noise_init`-after-tampered-close case is provably unit-shape, not e2e-shape.** The natural e2e formulation is "phone reconnects with the same `conn_id` after the tamper-close." But fakerelay assigns a new `conn_id` per `/v1/client` dial — forcing a `conn_id` reuse at the e2e layer requires invasive harness changes (a new `fakerelay` API for `conn_id` injection, or a test-only env var). The spec recognised this and explicitly assigned the assertion to the unit test layer (`TestV2Session_OpenState_FreshNoiseInitAfterAEADClose`), where `conn_id` reuse is a deterministic test fixture rather than a fakerelay extension. The lesson: when an e2e test would require harness invasiveness that exceeds the test's behavioural value, accept the asymmetry — pin the invariant where it's deterministic (unit) and rely on the e2e suite for the natural-flow case (tamper → 4421 close, observed from the phone side). Document the decision in the test's comment so a future maintainer doesn't try to "fix" the asymmetry by extending fakerelay.
- **Empty plaintext is not a security event; it's an envelope-shape event.** `s.recv.Decrypt` on a well-formed empty-payload `noise_msg` returns `[]byte{}` cleanly (no AEAD failure). The empty bytes flow into `dispatch.Route`'s `json.Unmarshal` which fails decode, and Route's existing malformed-envelope branch emits a sealed `protocol.malformed` reply. The handler chain is never reached. No new code path needed — the existing v1 ladder handles empty payloads correctly, and the v2 manager inherits that behaviour for free. The lesson generalises: when the AEAD layer terminates at a `Decrypt` call whose successful return passes plaintext to a JSON-shaped consumer, the consumer's existing shape-validation is the next line of defence — don't add a redundant length check at the AEAD boundary.

## Out of scope (filed elsewhere or deferred)

- **Production cutover of `V2SessionManager` into `cmd/pyry/relay.go`** — re-wires the daemon to construct `V2SessionManager` instead of `Dispatcher` and registers handlers against `V2SessionConfig.Handlers`. Gated on [#436](436.md)'s preflight check. Sibling follow-up.
- **Per-conn fan-out for handler dispatch** — one goroutine per `conn_id` with a per-session mutex guarding `s.send`/`s.recv` (mirroring `dispatch.Dispatcher.runConn`). Priority concern before production cutover, NOT in this slice. Tracked in [`features/v2-session-manager.md`](../features/v2-session-manager.md) § Out of scope.
- **Re-key timer + `rekey_request` envelope handling** — #435.
- **`V2Session` cleanup on phone-initiated WS close** — relay→binary "phone disconnected" forward signal does not exist on the v2 wire today. AEAD-failure teardown (this slice) is the only binary-initiated cleanup path; phone-initiated reconnects still cannot trigger local cleanup. Bounded by per-binary-leg lifetime.
- **Per-phone-conn 10s handshake timeout** — requires a relay→binary "phone connected" signal that does not exist in the v2 wire today.
- **Revocation propagation to active conns** — `s.device` snapshot does not refresh after handshake; tears down at the next WS recycle, same posture as v1's `Conn.auth`.
- **Removal of the v1 `RoutingEnvelope.Token` field** — v1 dispatcher still consumes it; removal lives in the eventual v1→v2 hard-cutover slice.
- **Changes to `internal/relay/handlers/` payload shapes** — explicitly out of scope per the ticket; handler chain is consumed unchanged.
- **Phone-side and relay-side endpoint changes** — production relay's `/v2/server` route is its own team's work; only fakerelay continues to mirror it.
