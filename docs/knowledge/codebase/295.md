# Ticket #295 — `net/e2e`: fakerelay harness package

Phase 3 Track C, e2e tooling. New `internal/e2e/internal/fakerelay` package — an in-process WebSocket server that speaks the **routing half** of the mobile↔relay protocol (`docs/protocol-mobile.md` § Authentication, § Routing envelope). It exists so daemon-side e2e tests can exercise the full WS roundtrip (binary ↔ relay ↔ phone) without depending on the `pyrycode-relay` binary or live infrastructure. Sibling ticket ships the fake-phone client; a third ticket consumes both for the appendix-flow roundtrip.

## Implementation

- **One production file (`fakerelay.go`, ~425 LOC) + one test file (`fakerelay_test.go`, ~400 LOC).** Matches `fakeclaude`'s single-file shape; the architect's ~200 LOC budget was overrun (handler + serve + two pumps + cleanup-cascades-on-binary-loss adds up), but no second file would have clarified anything.
- **Surface:** `func New(*slog.Logger) *Server` (returns running, logger required, nil panics), `(*Server).URL() string` (ws://… base — callers append `/v1/server` or `/v1/client`), `(*Server).Close() error` (idempotent via a `closed bool` guard, always returns nil). No `Config` struct — every test wants "boot it, get a URL, close it"; a future ticket can add `NewWithConfig` non-breakingly.
- **`coder/websocket` on both server and client side** (matching #247). `websocket.Accept(w, r, nil)` for upgrades, `conn.SetReadLimit(maxFrameBytes = 1<<20)` mirrors `internal/transport`'s 1 MiB ceiling. The AC's "gorilla/websocket or nhooyr.io/websocket" wording was a stale carryover — resolved to `coder/websocket`.
- **Routing data model.** Two maps under a single `sync.Mutex`: `binaries map[string]*binaryConn` (serverID → binary) and `phones map[string]*phoneConn` (connID → phone). Lookup keys differ; phone-without-binary is invalid; separate maps make the invariants local. `connSeq uint64` increments under `s.mu` for deterministic `c-1`, `c-2`, … conn-ids (over UUIDs — makes consumer-test assertions readable).
- **`/v1/server` upgrade contract.** Read `x-pyrycode-server` → empty rejects with HTTP 400. Pre-upgrade claim check under `s.mu` → already-claimed rejects with HTTP 409. `websocket.Accept` → on failure, return (Accept already wrote the response). Re-check claim under `s.mu` AFTER Accept (closes the race where two binaries upgrade concurrently for the same id); race-loser closes with `StatusPolicyViolation` "server-id claimed". Install in `binaries`, run `serveBinary`. On cleanup: delete from `binaries` AND tear down every phone whose `serverID` matches — the load-bearing cleanup-cascades-on-binary-loss path that satisfies "no goroutine leaks under `go test -race`."
- **`/v1/client` upgrade contract.** Read three required headers (`x-pyrycode-server` / `x-pyrycode-token` / `x-pyrycode-device-name`) → any empty rejects with HTTP 400. Under `s.mu`: if `binaries[serverID]` absent, reject with HTTP 503 (the AC bullet 5 "no binary online" case). `websocket.Accept`. Re-check binary presence under `s.mu` AFTER Accept (closes the race where the binary drops between pre-check and accept); allocate `connID = "c-<seq>"`, install in `phones`, run `servePhone`. On cleanup: delete from `phones`.
- **Two goroutines per accepted conn: recvPump + sendPump.** No ping-loop goroutine on the server side — `coder/websocket` auto-responds to pings. `serveBinary` / `servePhone` spawn both under a child ctx, wait for the first pump to return, cancel the other, drain the second, close `done` AFTER both pumps unwind (so anyone holding the conn + waiting on `done` knows the sendCh receiver is gone — backstop for the "binary disappears mid-route" path in `phoneRecvPump`).
- **`sendCh` is unbuffered on both sides — intentional backpressure.** A slow binary that doesn't read from its WS conn → its sendPump blocks on `conn.Write` → phones writing to it block on `binary.sendCh`. Consumer tests that want to stall the binary observe a stuck phone-send rather than silently-dropped frames. The pump select also watches `ctx.Done()` and the peer's `done` channel so a binary disconnect doesn't strand a phone-side recv-pump on a send to nowhere.
- **Wrap / unwrap uses `protocol.RoutingEnvelope` from #255** — `Frame json.RawMessage` survives marshal byte-for-byte so the wrapper splices instead of re-parsing. Phone→binary path: `phoneRecvPump` reads, `json.Valid(data)` gate (non-JSON phone frame tears the phone down with a Debug log + return — the wrapper assumes a JSON value, the consumer test relies on legible wrapped output), marshal `RoutingEnvelope{ConnID, Frame}`, ship to `binary.sendCh`. Binary→phone path: `binaryRecvPump` reads, unmarshal as `RoutingEnvelope`, look up `phones[ConnID]`, ship `Frame` to `phone.sendCh`. Malformed wrapper / unknown conn_id is **Debug-log-and-continue** (not close-the-binary) — surfaces as a missing receive in the consumer test rather than a relay-side shutdown masking the cause.
- **Lock discipline.** Only `s.mu`. Held only for map mutations + `connSeq` increment + lookups. Never held across `conn.Read`, `conn.Write`, or channel sends. No ordering graph (only one mutex).
- **Shutdown sequence (`Close`).** Under `s.mu`: set `closed=true`, snapshot every binaryConn + phoneConn, release. Force-close each conn (`cancel()` + `conn.Close(StatusNormalClosure, "server closing")`) — unblocks the per-conn pumps' `Read`/`Write`. Then `s.http.Close()` (waits for in-flight HTTP handlers). The `closed` flag is also re-checked inside each handler under `s.mu` after Accept so a handler racing `Close` aborts cleanly instead of installing a doomed conn.
- **Tests.** Eleven `Test*` functions, all `t.Parallel()`-able, stdlib `testing` only. Shared helpers (`dialBinary` / `dialPhone` / `mustReadJSON`) live alongside. Coverage: empty-server-header reject (400), first-claim-wins (409 on duplicate, immediate reuse after disconnect — no grace period), three-headers-required for phone (400 sub-cases), phone-without-binary reject (503), phone→binary frame wrap with `c-1` conn_id assertion, binary→phone frame unwrap, conn_id increments per phone, phone closed when binary goes (cleanup-cascades), `Close` returns within 1s with no goroutine leak (capture `runtime.NumGoroutine` baseline + assert tolerance after Close), idempotent `Close`.

## Patterns established

- **`internal/e2e/internal/<harness>/` for in-process test peers.** Both `fakeclaude` (#122, supervised-child stand-in) and now `fakerelay` (this ticket, WS-server stand-in) live here. The double-`internal/` visibility-fence is the structural signal — only e2e-package code can import them. Single-`.go`-file shape carries over from `fakeclaude`.
- **Pre-upgrade HTTP status as the rejection channel** for a fake-relay harness. The production wire spec uses post-upgrade WS close codes (`4409`/`4404`/`4401`); the harness uses HTTP 400/409/503 because the status surfaces directly in `websocket.Dial`'s returned error, which is far simpler for consumer tests to assert on than the close-code dance. Documented as a deliberate deviation in the package comment so future maintainers don't mistake the fake for the spec.
- **Re-check race-sensitive invariants AFTER `websocket.Accept`.** Both `/v1/server` (server-id still unclaimed?) and `/v1/client` (bound binary still present?) check under `s.mu` twice: pre-Accept for the cheap-reject fast path, post-Accept for the race window between pre-check release and Accept's response write. Race-loser closes the freshly-accepted conn with the appropriate WS close code rather than installing a doomed entry.
- **Cleanup-cascades-on-dependency-loss** is the only way to satisfy "no goroutine leaks under `-race`" for a harness where one accepted conn (phone) is structurally dependent on another (binary). The serve-loop cleanup walks `phones` for matching `serverID` and cancels each.
- **`Close()` pump unwind via WS-conn force-close, not via a separate signal channel.** Single-signal-path keeps the shutdown reasoning local: close the conn, pumps see Read/Write errors, serve goroutines return, `httptest.Server.Close` waits for handlers. No need for a `closeCh` that pumps observe directly.

## Out of scope (deferred)

- **Fake phone client** — sibling ticket. Mirror shape: in-process WS client that speaks the phone↔relay envelope protocol.
- **Roundtrip e2e test** — third ticket. Stands up fakerelay + fakephone, points the daemon's WSS client (#247–#250 chain) at the fakerelay, asserts the appendix-flow envelope sequence end-to-end.
- **30-second server-id release grace period** from `docs/protocol-mobile.md` § Authentication. AC explicitly pins immediate release.
- **Production WS close codes** (`4409`/`4404`/`4401`). Pre-upgrade HTTP 409/503 is the harness's deliberate substitution.
- **Token-contents validation.** Relay accepts any non-empty token; the binary owns validity per #249.
- **`hello` / `hello_ack` envelope dispatch.** The harness is the routing seam only; envelope dispatch is a separate layer (production: `internal/relay`).
- **TLS termination, certificate pinning.** Loopback `ws://` only.
- **Persistence, rate limiting.** Maps live for the `Server`'s lifetime; no throttling.
- **`Config` struct overrides** (`connSeq` start, buffer sizes, etc.). Add when a consumer ticket needs it.
- **Inspection API** (`Server.Connections()` etc.). Tests assert behavior via the WS endpoints, not by reaching in.

## Lessons learned

- **The AC wording "matching #247's choice" was authoritative; the AC body's "gorilla/websocket or nhooyr.io/websocket" was a stale carryover.** Both libraries had been considered earlier; #247 settled on `coder/websocket`. Flag for PO to update the issue-template wording so future tickets don't echo the stale alternatives.
- **A naive `Server.Close()` that only force-closes WS conns leaks goroutines** when a phone is mid-`select` on a binary's `sendCh` (phoneRecvPump) or a binary is mid-`select` on a phone's `sendCh` (binaryRecvPump). The fix is the peer-`done`-channel-in-select pattern — every channel-send in a routing path watches both `ctx.Done()` AND the destination conn's `done`. Without the `done` arm, a phone send to a binary whose recvPump has already returned (so its sendPump is being cancelled) blocks forever; with it, the phone-side gives up and the recvPump returns, letting cleanup complete.
