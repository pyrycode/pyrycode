# Ticket #452 ŌĆö `internal/noise` + `internal/relay`: capture initiator peer-static at v2 IK handshake

Pure-data slice: exposes the initiator's static public key from [`internal/noise`](../features/noise-package.md)'s `Responder` and captures it on the per-session [`V2Session`](../features/v2-session-manager.md) at the initial Noise_IK handshake, so a future re-key responder slice can compare a fresh-handshake initiator's static against the pinned value and reject mismatched peers at WS close code 4426. No production code path reads the captured field in this slice; the field exists to be consumed by [#453](#) (re-key responder peer-continuity check). Extracted from the abandoned #449 super-slice ŌĆö design lifted verbatim from `docs/specs/architecture/449-v2-rekey-responder.md` on the orphan `feature/449` branch.

## Implementation

- **Two production files modified** ŌĆö `internal/noise/noise.go` (+20 LOC: one new method + doc-comment) and `internal/relay/v2session.go` (+19 LOC: one new struct field + one capture-site assignment + comments). Tests +100 LOC across `internal/noise/noise_test.go` and `internal/relay/v2session_test.go`. Inside the spec's XS budget.
- **`(*noise.Responder).PeerStatic() []byte`** ŌĆö one-statement body: `return append([]byte(nil), r.hs.PeerStatic()...)`. Returns a defensive copy of flynn/noise's internal `HandshakeState.rs` field, which is populated by `ReadInit` when the IK message 1's encrypted-`s` token is DH-decrypted. Mirrors the defensive-copy posture of `NewResponder`'s `StaticKeypair.Private: append([]byte(nil), staticPriv...)` ŌĆö same idiom, same posture for bytes flowing OUT as for bytes flowing IN.
- **Pre-`ReadInit` contract: no panic, returns zero-length slice.** Documented on the method. flynn's `s.rs` is nil before `ReadMessage` populates it, so `append([]byte(nil), nil...)` yields a zero-length slice. The wrapper pins this as its own contract independently of flynn's internals ŌĆö a future flynn upgrade that changes `s.rs`'s initial value would surface as a unit-test failure rather than a silent behavioural change. The alternative (panic, or `(nil, ErrHandshakeIncomplete)`) was rejected: a panic would propagate to the v2 manager's single dispatch goroutine and tear down the entire mobile relay path, and an error return would bloat the call site for a contract that is structurally maintained by the call order.
- **No new method on `Initiator`.** The initiator already knows its peer's static ŌĆö it is the constructor argument. Adding the symmetry would be dead code and invite confusion about which side's value is "trusted".
- **`V2Session.peerStatic []byte`** field, declared in the same field block as `device`. Set exactly once per V2Session in `handleNoiseInit`, immediately after `Responder.ReadInit` returns nil ŌĆö at the earliest authenticated point, before the early-data JSON unmarshal, hello-type check, token validation, or `WriteResp`. Reasoning: by the time `ReadInit` returns nil, flynn has MAC-verified and decrypted the initiator's static; the value is authentic regardless of whether the subsequent steps accept the hello.
- **The capture site precedes any branch that calls `closeWith`.** A token failure later in the function tears the session down via `closeWith` and `delete(m.sessions, s.connID)` ŌĆö the captured field is dropped along with the session entry, which is correct (a failed handshake leaves no peerStatic to compare against on a future re-key).
- **The field is set but never read in this slice.** `go vet` and `staticcheck` do not warn on unused struct fields, so no suppression needed. The v2session integration test reads the field via white-box access (same-package test), which keeps `go test -race` quiet on the unused warning across the build matrix.
- **Lifetime contract pinned in the field doc-comment.** A successful re-key (added in #453) MUST NOT overwrite this value ŌĆö its lifetime is the V2Session's, pinning the original peer's identity for the entire session. The re-key responder will call `bytes.Equal(newResp.PeerStatic(), s.peerStatic)` and reject mismatches at WS close code 4426.
- **SECURITY discipline extended to the new field.** The doc-comment names it explicitly: the captured value is a public key (not a secret) but is identity-bearing, and MUST NOT appear in any logged field. The package's no-key-bytes-in-logs discipline extends naturally to per-session identity pins. Not persisted to disk; lifetime is the V2Session.
- **No new ADR.** The architectural decision (Mobile Protocol v2 / Noise_IK + per-session peer-static pinning) is [ADR 024](../decisions/024-noise-ik-mobile-e2e.md); this slice ships the data-capture half of Threat #3's residual-risk-claim implementation. The orphan spec on `feature/449` resolved every design question this slice needed.

## Patterns established

- **Defensive-copy posture is symmetric: bytes IN and bytes OUT both `append([]byte(nil), ŌĆ”...)`.** The wrapper's `NewResponder` already defensive-copies the caller-supplied `staticPriv` before handing it to flynn's `DHKey` (a caller-side mutation can't corrupt the live handshake). The new `PeerStatic()` accessor mirrors that posture for the reverse direction ŌĆö bytes leaving the wrapper are equally insulated from caller mutation, so a caller that stashes the slice and later mutates it cannot corrupt flynn's internal state nor poison a downstream `bytes.Equal` comparison. The pattern generalises: every byte-slice boundary at a wrapper edge gets a defensive copy in both directions; the cost is `O(KeyLen)` per call, paid once per handshake; the benefit is that callers can store the bytes without coordination with the wrapper's lifetime.
- **Inert data capture lands one slice before its consumer.** AC #4 says "field is unread within this slice ŌĆö no behavioural change observable to the wire". The capture site exists today to be read tomorrow (#453). Splitting the pure-data exposure from the read+comparison logic lets the security-review surface for this slice reduce to "does the capture happen at the right point, under the right defensive-copy posture" ŌĆö without entangling re-key state-machine semantics. The pattern: when a follow-on slice will need data captured at a specific authenticated point, ship the inert capture first; the security review for the data slice is small and self-contained, and the follow-on slice's review focuses solely on the consumer logic.
- **Pin a wrapper contract independently of the upstream library's implementation detail.** The pre-`ReadInit` zero-length return matches flynn's `s.rs == nil` initial state today, but the wrapper documents and tests its own contract rather than re-stating flynn's. A future flynn upgrade that initialised `s.rs` to a non-nil placeholder would surface as a `TestResponder_PeerStatic_BeforeReadInit_ReturnsZeroLength` failure ŌĆö caught at CI, not at the next caller's runtime. The pattern: when a wrapper exposes a behaviour that is currently a consequence of the upstream library's internal field state, declare and test the wrapper's contract directly so an upstream change cannot silently break callers.

## Lessons learned

- **The "field is set but never read" warning class isn't a Go warning class.** Initial concern: would `go vet` or `staticcheck` flag the inert `peerStatic` capture across `go test ./...`? They do not ŌĆö both tools tolerate unused struct fields (only unused locals/parameters/imports trip them). The same-package white-box test additionally reads the field, which would have suppressed the warning anyway. The lesson: when an AC explicitly forbids a behaviour-observable change ("set but never read in production"), verify the toolchain's warning posture once and document it in the spec ŌĆö the developer doesn't need a `//nolint` suppression, and the reviewer doesn't need to verify one isn't missing.
- **Orphan specs on abandoned branches are first-class context, not noise.** #452 was extracted from #449, whose super-slice was closed `NOT_PLANNED` after the PO decided the rework was too coarse. The architect spec at `docs/specs/architecture/449-v2-rekey-responder.md` lives on the unmerged `feature/449` branch and contains the verbatim design notes for the accessor signature, defensive-copy body, field-lifetime contract, and capture-site placement that this slice ships. Re-deriving any of those decisions from scratch risks a subtle drift (e.g. a different defensive-copy idiom, a different capture-site line). The lesson: when a slice is extracted from an abandoned super-slice, treat the abandoned spec as the canonical design document for the carved-out slice ŌĆö read it first, lift the decisions verbatim, and name it explicitly in the new spec's "Files to read first" so the developer doesn't re-derive. The orphan branch is a feature of the carve-out, not a problem to route around.
- **Public-key bytes are not secret, but they ARE identity-bearing.** The instinct on first read: "it's a public key, why does the doc-comment say MUST NOT log?" The reason: per-session identity pinning is what makes the relay-MITM mitigation cryptographic rather than policy. A binary log that emits `peerStatic` makes a `paired_devices.json` enumeration attack trivial for an attacker who already has read access to the log channel ŌĆö the log becomes a parallel device registry. The package's no-key-bytes-in-logs discipline (designed for private keys) extends naturally to identity pins; the SECURITY doc-comment on the new field explicitly states this so the next reviewer doesn't relax it on the "but it's public" instinct. Code review enforces.

## Out of scope (filed elsewhere or deferred)

- **The re-key responder path itself** (`handleRekeyInit`, atomic `CipherState` swap, peer-static continuity check via `bytes.Equal(newResp.PeerStatic(), s.peerStatic)`, AEAD-mismatch teardown verification) ŌĆö **#453**, blocked by this slice. PO recorded the `blockedBy` edge in the split-summary on #449.
- **The `rekey_request` control-envelope discriminator and its handler chain branch** ŌĆö **#454**, independent of this slice and unblocked.
- **Initiator-side exposure of peer static** ŌĆö the initiator already knows it (constructor argument); no accessor needed, no parallel field on `V2Session` for the initiator role (none exists; `V2Session` is responder-only).
- **Persistence of `peerStatic` across binary restarts** ŌĆö out of scope for v2; ADR 024 ┬¦ Re-key policy specifies in-memory only. A binary restart drops the pin; the next initial handshake re-pins for the new session.
- **Constant-time comparison concerns on the future read** ŌĆö `bytes.Equal` (lands in #453) is acceptable: both operands are attacker-known public values, so variable-time leakage carries no secret. Documented in the #453 spec ahead of time to prevent a "should be `subtle.ConstantTimeCompare`" review nit on a non-issue.
