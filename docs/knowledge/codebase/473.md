# Ticket #473 — agent-run: adapt `--self-check` to exercise ptyrunner

Path-swap rewrite of [`internal/agentrun/selfcheck`](../features/agentrun-selfcheck-package.md) so the deny-default verification spawns claude via [`ptyrunner.Run`](../features/ptyrunner-package.md) instead of `streamrunner.Run`. After the [#470](https://github.com/pyrycode/pyrycode/issues/470) cutover the dispatcher's production path is ptyrunner; the selfcheck now verifies that same enforcement contract (per-spawn deny-default settings file with `permissions.defaultMode: "deny"` + `permissions.allow: ["Read"]` under `--settings <path> --permission-mode default`, in an interactive-TUI PTY) instead of the legacy streamrunner shape (`-p` + `--allowed-tools Read --dangerously-skip-permissions`). The empirical contract (#329 Phase A spike) is unchanged — claude refuses Bash under the deny-default whitelist — but the verification target moves with production. Sub-issue of [#329](https://github.com/pyrycode/pyrycode/issues/329) (tracking). Composes [`trust.MarkWorkdirTrusted`](../codebase/475.md) + [`settings.WriteSettings`](../codebase/476.md) + `sessions.NewID` + [`ptyrunner.Run`](../codebase/471.md) through four package-level seam vars, mirroring the production [`runAgentRunPty`](../codebase/470.md) wiring at smaller scale.

## Implementation

- **`SelfCheckDenyDefault` composes four collaborators behind seam vars; tests mock the entire spawn in-process.** The package gains four unexported `var` assignments at init — `trustMark = trust.MarkWorkdirTrusted`, `settingsWrite = settings.WriteSettings`, `newSessionID = func() (string, error) { sid, err := sessions.NewID(); return string(sid), err }`, `ptyRun = ptyrunner.Run` — same pattern `cmd/pyry/agent_run.go`'s production ptyrunner path uses. Production never assigns; per-test `installSeams(t)` captures + restores via `t.Cleanup` and installs benign default overrides so no test accidentally hits `~/.claude.json`, `os.TempDir()`, or real ptyrunner. The default `ptyRun` override `t.Errorf`s if invoked so a test that forgets to set it surfaces loudly instead of silently passing. Same shape as `cmd/pyry/agent_run_test.go:installFakeSeams` from [#470](https://github.com/pyrycode/pyrycode/issues/470); no new convention.

- **`Config` / `Result` / `ErrBashInvoked` / `ErrTimeout` / `bashInvokedInRaw` shapes preserved verbatim.** The CLI wrapper's call site (`cmd/pyry/agent_run_selfcheck.go:51`) is unchanged: it still calls `selfCheckFn(ctx, Config{ClaudeBin, WorkDir})`. The sentinel message strings (`"Bash invoked despite deny-default settings"`, `"overall timeout"`) preserve verbatim from [#336](https://github.com/pyrycode/pyrycode/issues/336) — the word "settings" in `ErrBashInvoked` is now mechanism-accurate again (it was already accurate post-#336, became aspirational under [#375](https://github.com/pyrycode/pyrycode/issues/375) when the streamrunner path used `--allowed-tools` instead, and is again literally true under the ptyrunner settings file). No exported-surface migration cost for the CLI wrapper or the daily CI workflow's exit-code contract.

- **`runAgentRunPty`-shaped sequence inside the helper: four I/O calls + one delegated `ptyRun`, no goroutines around the composition.** Order: `trustMark(cfg.WorkDir)` → wrap as `"mark workdir trusted: %w"`; `settingsWrite(canonicalAllow)` → wrap as `"write settings: %w"`, then `defer func() { _ = os.Remove(settingsPath) }()` registered AFTER the err-check; `newSessionID()` → wrap as `"mint session id: %w"`; then `ptyRun(gctx, ptyrunner.Config{...})` runs inside an errgroup with the JSONL watcher. The defer-LIFO discipline that ticket [#470](https://github.com/pyrycode/pyrycode/issues/470) pinned for `runAgentRunPty` carries through verbatim — `TestSelfCheck_SettingsCleanedOnLaterFailure` is the selfcheck-side equivalent of `TestRunAgentRun_PtyPath_SettingsRemovedOnFailure`, both pinning that the settings tempfile is removed via `os.Stat == ErrNotExist` against a real `os.CreateTemp` path even when a later step (sessionID mint) returns error.

- **`canonicalAllow = []string{"Read"}` is package-level and hard-coded; no `Config.AllowedTools` exposure.** The verification is "deny-default refuses tools NOT in the allow list", and the chosen tool `"Bash"` MUST NOT be in the allow list. Coupling the two values prevents a future caller from breaking the invariant by widening the allow list to include Bash. The `canonicalPrompt` (`"Use Bash to echo hello. Be brief."`) is unchanged from [#336](https://github.com/pyrycode/pyrycode/issues/336)/[#375](https://github.com/pyrycode/pyrycode/issues/375).

- **`ptyrunner.Config.SystemPrompt` set to `"/dev/null"`.** ptyrunner's required-field validation rejects an empty `SystemPrompt`; the selfcheck has no production system prompt to inject and the canned exhibit prompt is self-contained. `/dev/null` is a portable 0-byte readable character device on Linux + macOS (the only targets per project CLAUDE.md); claude's `--append-system-prompt-file` reads it as empty bytes and appends nothing. One fewer tempfile to create + clean up per selfcheck run vs. the rejected alternative (`os.CreateTemp` empty file + `defer os.Remove`, ~6 LOC heavier).

- **`ptyrunner.Config.WorkDir` is the symlink-resolved `realpath` from `trustMark`, not `cfg.WorkDir`.** Same realpath-not-parsed-workdir contract pinned for `runAgentRunPty` in [#470](https://github.com/pyrycode/pyrycode/issues/470): claude resolves the workdir before keying `projects[<realpath>]` in `~/.claude.json`, so handing claude a symlinked path when the trust pre-write keyed the realpath would miss the modal-check and re-render the trust modal. Selfcheck mirrors the contract structurally by threading `trustMark`'s return value into `ptyrunner.Config.WorkDir`. The default benign `trustMark` mock returns the workdir verbatim; tests that exercise the wrapper-error path force a non-nil error instead of asserting realpath identity (covered by `TestRunAgentRun_PtyPath_WorkDirIsTrustResolvedRealpath` in [codebase/470.md](470.md)).

- **`MaxTurns: 1, Model: "sonnet", Effort: "low"`** — frozen by #329/#336 and not exposed as `Config` fields. Bounds the ptyrunner budget Counter at one turn and keeps the canonical prompt's wall-clock under the 90s `defaultSelfCheckTimeout`. The ptyrunner spawn itself can take 5-15s (idle wait + JSONL tail arming); the canonical prompt + sonnet/low generates a single turn in ~3-8s; total wall-clock is comfortably under 30s in the happy path.

- **Two-goroutine errgroup with documented defer LIFO is preserved verbatim from #336/#375.** Goroutine A runs `ptyRun(gctx, ptyrunner.Config{...})` with `Stdout: pw`, `Stderr: io.Discard`, deferring `pw.Close()` so the watcher sees EOF when the spawn exits. Goroutine B wraps `pr` in `jsonl.NewReader`, loops on `reader.Next()`, filters `ev.Kind == "assistant"`, runs `bashInvokedInRaw` on `ev.Raw`, sets `result.BashInvoked` + `cancel()` on hit, sets `result.EndOfTurnObserved` + `cancel()` on first `ev.EndOfTurn`, deferring `pr.Close()`. The streamjson.Emitter's trailing `type:"result"` line is filtered naturally by the `ev.Kind != "assistant"` continue; no new parser. Outcome priority unchanged (`BashInvoked` > `EndOfTurnObserved` > `DeadlineExceeded` > wrapped infra > defensive fallthrough). `ptyrunner.Run`'s internal teardown (cancel → wg.Wait → counter.Stop → emitter.Close → sess.Close per its own defer LIFO) is inherited.

- **Error-message hygiene is load-bearing — wrappers format with `%w` only.** The package doc-comment pins it: `"mark workdir trusted"`, `"write settings"`, `"mint session id"` wrappers MUST NOT substitute workdir realpath, settings tempfile path, or session id into the message. The underlying error from `trust.MarkWorkdirTrusted` / `settings.WriteSettings` / `sessions.NewID` already names the failing operation; pyry's wrapper adds only the `"agentrun: self-check: …"` namespace prefix. Rationale: the dispatcher's log aggregator forwards error strings; sessionID is benign UUID but settingsPath leaks tempfile paths and workdir realpath leaks operator cwd. Both are operationally diagnostic — not exfiltration-grade — but pinning the discipline at the spec level prevents the `fmt.Errorf("write settings to %s: %w", settingsPath, err)` reflex during implementation. Caught and folded in during architect security-review.

- **CLI wrapper gains `selfCheckFn` + `selfCheckGetVersion` seams; shell-wrapper helper-process machinery deleted on both sides.** `cmd/pyry/agent_run_selfcheck.go` adds `var selfCheckFn = selfcheck.SelfCheckDenyDefault` and `var selfCheckGetVersion = captureClaudeVersion`; the CLI calls `selfCheckFn(ctx, cfg)` and `selfCheckGetVersion(claudeBin)` so tests mock the entire selfcheck at the boundary instead of spawning a fake claude binary. Removes the `TestSelfCheckHelperProcess` re-exec + `selfCheckHelperWrapper` shell script (~80 LOC) and the `TestSelfCheckCLIFakeClaude` + `configureSelfCheckFakeClaude` machinery — those existed only because the old selfcheck needed a real fake-claude binary on disk. Net LOC change in `cmd/pyry/agent_run_selfcheck_test.go` is negative (~135 LOC removed, ~20 LOC added). The `captureClaudeVersion` helper is left unchanged in production; tests stub it via the `selfCheckGetVersion` seam (architect-recommended option (a) from the spec's open questions).

- **`TestRunAgentRunSelfCheck_FAIL` forbidden-substring pins inverted to required-substring pins.** Pre-#473 the FAIL message described the streamrunner argv shape (`--allowed-tools "Read" --dangerously-skip-permissions`) and the test asserted that PTY/settings-file substrings were ABSENT (a regression guard against drift). Post-#473 those substrings ACCURATELY describe what selfcheck exercises and MUST be present. The new required list: `permissions.defaultMode: "deny"`, `["Read"]`, `PTY`, plus ticket references `#329` / `#336` / `#470` / `#473`. Verbatim `"name":"Bash"` Evidence substring stays as the FAIL-Evidence anchor.

- **Eleven test cases cover the rewrite's surface.** `TestSelfCheck_Pass` (ptyRun mock writes `passLine` + brief hold so the watcher consumes the line before pw closes — pins the buffer-races-pw-close failure mode), `TestSelfCheck_BashInvoked` (two-line fixture: `bashLine` + `passLine`; the trailing pass confirms the detector trips on the first line rather than falling through to end-of-turn), `TestSelfCheck_Timeout` (ptyRun mock blocks on `<-ctx.Done()` mirroring ptyrunner.Run's ctx-cancel-collapse contract), `TestSelfCheck_MalformedAssistantLineSkipped` (`"{not valid json\n" + passLine + "\n"`; jsonl.Reader's log-and-skip resilience inherits to PASS), `TestSelfCheck_ConfigValidation` (table over empty `ClaudeBin` / empty `WorkDir`), `TestSelfCheck_TrustMarkFailure` / `_SettingsWriteFailure` / `_SessionIDFailure` (wrapped-error string pins: each asserts BOTH the namespace prefix substring AND the underlying error string is preserved through `%w`), `TestSelfCheck_SettingsCleanedOnLaterFailure` (the defer-ordering invariant — `settingsWrite` mock calls real `os.CreateTemp`, `newSessionID` mock forces error, assertion is `os.Stat == ErrNotExist`), `TestSelfCheck_PtyRunnerError` (`ptyRun` returns `ptyrunner.ErrTrustModalDetected`; asserts `errors.Is` survives the wrap so the operator sees the sentinel's embedded remediation hint), and `TestBashInvokedInRaw` (six-row table preserved verbatim from #336).

- **Real-claude smoke (AC #4) is a manual operator drill, not CI.** The PR body notes the smoke as deferred-to-developer: `PYRY_CLAUDE_BIN=<path> ./pyry agent-run --self-check` against claude 2.1.144 expected to PASS. claude is not bundled with the repo and reaches the live Anthropic API; per-PR CI continues to exercise the dispatcher's production path via [#482](https://github.com/pyrycode/pyrycode/issues/482)'s real-claude byte-equivalence test (different scope — ptyrunner-side wire-shape equivalence vs. selfcheck's deny-default refusal), and the daily `.github/workflows/self-check-daily.yml` cron continues to run the selfcheck end-to-end against real claude. The daily workflow's exit-code contract is unchanged (`0` PASS, non-zero FAIL/INCONCLUSIVE) so no workflow file edit is required for the cutover; the daily workflow now structurally validates the production ptyrunner path instead of the (no-longer-production) streamrunner path.

## Patterns established

- **Match the verification target to the production path, not to the historical contract.** Two earlier selfcheck rewrites ([#336](https://github.com/pyrycode/pyrycode/issues/336) → [#375](https://github.com/pyrycode/pyrycode/issues/375) → [#473](https://github.com/pyrycode/pyrycode/issues/473)) all preserved the same empirical claim ("claude refuses Bash under deny-default whitelist") while swapping the verification mechanism to follow production. The principle generalises: when the production code path changes and an early-warning safety net exists for that path, rewriting the safety net to follow production is cheaper than maintaining two divergent enforcement contracts in claude — and the verification has higher fidelity (the safety net catches regressions in the path the dispatcher actually uses, not the historical or fallback path).

- **Mirror the production helper's composition shape at the selfcheck boundary.** `runAgentRunPty` (cmd-side) and `SelfCheckDenyDefault` (helper-side) now share the same four-call composition: `trustMark` → `settingsWrite` + defer cleanup → `newSessionID` → `ptyRun`. Both wire the same package-level seam-var pattern for tests. Both pin the `defer os.Remove(settingsPath)` AFTER the err-check, both pin the realpath-not-parsed-workdir invariant, both preserve the `%w`-only error-wrapper hygiene. Generalises: when a CLI verb and a sibling helper both compose the same set of primitives, mirror the wiring and the test seams structurally — the two stay aligned through future ptyrunner.Config additions because each addition shows up in both call sites as the same source-line edit.

- **Invert forbidden-substring tests into required-substring tests when the path swaps.** The predecessor's `TestRunAgentRunSelfCheck_FAIL` pinned that `permissions.defaultMode` / `PTY` / settings-file substrings were ABSENT — a regression guard for the streamrunner path. After the swap those substrings ARE accurate and MUST be present. Pattern: when a UI-string test pins absence of vocabulary that should never appear, and the underlying implementation later acquires that vocabulary as production reality, inverting the test (turning forbidden-list into required-list) is the right move — the test still anchors the operator-facing prose to the actual implementation, just with the polarity flipped.

- **`/dev/null` over an empty tempfile when a required-but-empty-content path field appears in a downstream struct.** ptyrunner's required `SystemPrompt` field exists for production callers that always have a system prompt to inject; the selfcheck doesn't and shouldn't be forced to create + manage an empty tempfile. `/dev/null` is portable across pyry's targets (Linux + macOS) and reads as zero bytes. The pattern generalises whenever a downstream API requires a non-empty path the upstream caller has no content for; cleanup-free, ~6 LOC lighter than the tempfile alternative.

## Lessons learned

- **Buffer-races-pw-close was the only non-obvious failure mode introduced by mocking `ptyRun` in-process.** The pre-rewrite shell-wrapper helper-process pattern had natural pacing — claude (the fake binary) wrote to stdout, then exited, and the OS-level pipe drain happened before the next test step. The in-process `ptyRun` mock can write `passLine` + `\n` and return *immediately*, racing the watcher goroutine's `jsonl.Reader.Next()` against the deferred `pw.Close()` that happens at the end of the spawn-goroutine's function body. If `pw.Close()` lands before the reader has consumed the buffered line, `Next()` returns `io.EOF` cleanly but the assistant entry is lost. Fixed by adding a 50ms `select { case <-ctx.Done(): case <-time.After(50 * time.Millisecond): }` hold inside each mock that writes a line — the watcher reliably observes the line, and the test's `OverallTimeout` still bounds the wall-clock. Pattern reinforced: when replacing a subprocess-based test fixture with an in-process mock that shares an `io.Pipe`, the mock must hold the write end open long enough for the reader to drain, OR the test must read from `pr` directly instead of going through the helper's watcher. The 50ms hold is the cheaper option for this surface.

- **`installSeams`' default `ptyRun` override `t.Errorf`s rather than returning silent nil.** First-draft default returned `nil` so seams that the per-test body forgot to override would coast through, producing PASS Results with no assistant events — which looked like INCONCLUSIVE because `EndOfTurnObserved` stays false. The diagnostic confusion ("why does this PASS test fail with `EndOfTurnObserved: false`?") landed harder than a clean "unexpectedly invoked" failure. The pattern: when defaulting a seam to a no-op, fail loudly inside the no-op if the helper is composed in a way that requires the seam to do meaningful work. Same shape as `cmd/pyry/agent_run_test.go:installFakeSeams`'s default `ptyRun` (which Errored from day one in [#470](https://github.com/pyrycode/pyrycode/issues/470)) — that prior art is what flipped the choice here on second draft.

- **The package doc-comment pinning `%w`-only error wrappers prevents the substitute-the-path reflex during implementation.** The architect's security review caught the risk before code landed and folded the "Error-message hygiene" subsection into the spec. Without that pin in the spec, the natural `fmt.Errorf("write settings to %s: %w", settingsPath, err)` shape would have shipped — diagnostically nice for the developer, mildly leaky for the operator's log aggregator (tempfile paths + workdir realpath). The package doc-comment carries the rule forward; future contributors editing the helper see it at the top of the file. Generalises: when a security review folds a discipline into the spec, the discipline belongs in the package doc-comment too — specs are write-once, doc-comments are read-many.
