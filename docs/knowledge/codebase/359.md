# Ticket #359 — replace stale "scaffold only — no spawn yet" disclaimer in `pyry agent-run --help`

Replaces the misleading one-liner printed by `pyry agent-run --help` (which still claimed the verb was an unimplemented scaffold long after #391 wired it onto the stream-json subprocess pipeline) with a short behavioural description sourced from the canonical `runAgentRun` / `buildClaudeArgs` doc comments. Locks the new prose with a sibling unit test so a future stale-disclaimer regression fails CI. No runtime behaviour change.

## Implementation

- **`cmd/pyry/agent_run.go`** — replaced the single-line `fmt.Fprintln` inside the `fs.Usage` callback (the stale `"Drive a single supervised claude turn headlessly (scaffold only — no spawn yet)."`) with a `fmt.Fprintln(fs.Output(), agentRunUsageDescription)` referencing a new package-level constant. The constant is a multi-line raw-string literal beginning `"Drive a single supervised claude turn headlessly."` followed by a 6-line behavioural summary covering: spawns claude as a stream-json subprocess (no PTY, no JSONL watcher); delivers the user prompt as a stream-json envelope on claude's stdin; forwards claude's stream-json stdout byte-for-byte (system init, assistant deltas, result events); `--max-turns` is passed through to claude (claude enforces it); `--dangerously-skip-permissions` paired with `--allowed-tools` is the authoritative tool gate. The constant's doc comment explicitly names `runAgentRun`'s and `buildClaudeArgs`'s doc comments as the canonical sources to keep in sync.
- **No production behaviour change.** `runAgentRun`, `parseAgentRunArgs`, `buildClaudeArgs`, `splitAllowedTools`, `requireRegularFile`, and `requireDir` are untouched. The flag set (names, types, default values, per-flag help strings) is untouched. Only the `fs.Usage` callback's description string was lifted into a named constant.
- **`cmd/pyry/agent_run_test.go`** — added `TestAgentRunUsageDescription` (single non-table function in the file's stdlib-only idiom). Asserts (a) `!strings.Contains(agentRunUsageDescription, "scaffold only")` as the regression-guard against AC #1, (b) the constant contains all three load-bearing substrings `"stream-json"`, `"--max-turns"`, `"--allowed-tools"` via a table over a `[]string`. Asserts directly on the constant rather than capturing `fs.Output()` because the constant is the sole source of the prose printed between the `Usage:` header and `fs.PrintDefaults()` — flipping it flips production and test in lockstep, with zero need to mutate `os.Stderr` globally or change `parseAgentRunArgs`'s signature (AC #4 forbids the latter).
- **`.pyry-agent-run-settings.json`** — a stray copy was committed at the worktree root and removed in `54de76a` (follow-up tidy-up unrelated to the help-text change). The file is part of the legacy PTY-drive path #391 cut over from; it should never appear in the repo.

## Patterns established

- **Lock user-facing text via a named constant + sibling test, not via stderr capture.** When an AC requires asserting the rendered content of a flag-set's `--help`, prefer lifting the description into an unexported package-level constant referenced by the `fs.Usage` callback over the alternatives. Same-package tests read the constant directly; no signature changes, no global-state mutation, no flag-parser invocation. The shape extends to every other CLI verb that wants a stale-disclaimer regression guard.
- **Pin load-bearing substrings, not whole strings.** The regression test asserts on three short substrings (`"stream-json"`, `"--max-turns"`, `"--allowed-tools"`) plus a negative pin on `"scaffold only"`, not on the full prose. This catches the specific failure modes the AC names (stale disclaimer survives; a copy-edit drops a load-bearing behaviour) without breaking every time a comma moves. The doc comment on the constant tells future editors which words are pinned and why.

## Lessons learned

- **Stale `--help` text outlives every other kind of stale doc.** The "(scaffold only — no spawn yet)" parenthetical survived from the v0.12.0-era scaffold through #332 (PTY spawn), #391 (stream-json cutover), and #375 (self-check rewrite) because no test had a reason to read it. Behavioural tests cover the wire — they don't cover the signage. The fix isn't "remember to update help text"; it's "lock the signage to a substring the runtime cannot describe without". Once one CLI verb has the pattern, replicating it across the others is a follow-up worth scheduling proactively, not a per-ticket scramble.

## Out of scope (deferred)

- Applying the same constant + substring-lock pattern to other `pyry` verbs (`pyry`, `pyry pair`, `pyry update`, `pyry sessions`, ...). Each verb's `fs.Usage` callback still prints inline string literals; a sweep ticket can lift them in one move.
- Any change to `runAgentRun`'s doc comment, `buildClaudeArgs`'s doc comment, or the per-flag help strings (the third argument to `fs.String` / `fs.Int` at the top of `parseAgentRunArgs`). All three are already accurate post-#391 and are the canonical sources the new `--help` body draws from.
