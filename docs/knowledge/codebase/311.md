# Ticket #311 — assistant-turn → `message` envelope bridge

Outbound counterpart to #322's inbound `send_message` path. When the supervised
claude child emits an assistant turn on its PTY, the binary fans out a
`message` envelope (stamped with `Supervisor.CurrentConversation()`) to every
active phone conn. Together #322 + #311 close the bidirectional mirror loop:
phone → `send_message` → PTY → claude → PTY → `message` → phone.

## Implementation

Four parts: a one-method hook in `internal/supervisor/bridge.go`, a snapshot
accessor in `internal/dispatch/dispatch.go`, the emitter wiring in `cmd/pyry`,
and a fakeclaude test trigger.

### 1. `Bridge.SetOutputObserver` — PTY output tee

`internal/supervisor/bridge.go` grew one unexported field
(`outputObserver func([]byte)`, guarded by `b.mu` alongside `b.output`) and one
exported setter. Inside `Write`, the observer fires on every PTY chunk **before**
the existing forward-to-attached-writer code:

```go
b.mu.Lock()
out := b.output
obs := b.outputObserver
b.mu.Unlock()
if obs != nil {
    obs(p)
}
```

Contract: observer runs on the supervisor's PTY-drain goroutine and MUST NOT
block, panic, or retain `p` past return (the supervisor's `io.Copy` reuses the
read buffer). Production observers enqueue + drop on overflow. No new lock
order — the observer field rides the same `b.mu` already held to read
`b.output`. `Write` still always returns `(len(p), nil)`; the load-bearing
"never wedge the PTY drain" invariant is preserved.

### 2. `Dispatcher.ActiveConns()` — broadcast eligibility snapshot

`internal/dispatch/dispatch.go` adds a snapshot accessor:

```go
func (d *Dispatcher) ActiveConns() []*Conn
```

Returns `*Conn` pointers for every conn that is **gate-completed and
not-closed**. Holds `d.mu` briefly to iterate `d.conns`; no new lock order.
Returned slice is fresh; callers may retain it. A conn that closes between
snapshot and `Send` lands its frame on the still-live shared `outbound` chan
and is harmlessly dropped at the transport layer (#307 reconnect semantics).

The `gate-completed` filter is load-bearing — see "gateStarted /
gateCompleted split" below.

### 3. `gateStarted` / `gateCompleted` flag split

`connState` previously carried a single `gateRun bool` that served two
unrelated purposes (don't re-run the gate locally; tell the demux a conn is
"alive"). Splitting closes a broadcast race against the auth gate's literal-id=1
hello_ack:

| Flag             | Writer / reader        | Lock     | Purpose                                                                |
|------------------|------------------------|----------|------------------------------------------------------------------------|
| `gateStarted`    | per-conn goroutine only| none     | Prevent re-entering the gate path on subsequent `runConn` iterations.  |
| `gateCompleted`  | per-conn → ActiveConns | `d.mu`   | Mark conn broadcast-eligible. Set ONLY AFTER `runGate`'s NextID advance returns. |
| `closed`         | per-conn → demux       | `d.mu`   | Unchanged from #308.                                                   |

The race the split closes: `relay.AuthenticateFirstFrame` emits `hello_ack`
with **literal** `ID=1` (not via `c.NextID()`). `runGate` publishes that
envelope onto `d.outbound` and **then** runs `_ = c.NextID()` so the next
binary-originated frame on the conn gets `id=2`. A broadcast call to
`c.NextID()` that ran in the window between gate-entry and that advance would
claim `id=1` for its `message` envelope and collide with `hello_ack` on the
wire (two envelopes both stamped id=1). Filtering `ActiveConns` on
`gateCompleted` — which is published under `d.mu` **after** `runGate`
returns — makes the collision structurally impossible: when `ActiveConns`
returns a conn, the per-conn id counter has already advanced past 1.

The gate-disabled tail of `runConn` sets `gateCompleted = true` immediately
because there is no `hello_ack` competing for `id=1` in that configuration —
the existing dispatcher unit tests use gate-disabled config and pin that path.

The split was introduced after PR #326's code review caught that the initial
fix (a single `gateRun` flag flipped to `true` at the **top** of the gate
path) did not actually close the race — `gateRun = true` ran **before**
`runGate`, leaving the same window open. See spec
[`311-assistant-turn-message-bridge.md`](../../specs/architecture/311-assistant-turn-message-bridge.md)
§"Security review [Concurrency]" for the full three-pass audit trail.

### 4. `cmd/pyry/assistant_turn.go` — the emitter

New file. One unexported type plus a wiring function:

```go
type assistantTurnEmitter struct {
    sup    cursorReader      // CurrentConversation() — interfaced for tests
    disp   connBroadcaster   // ActiveConns() — interfaced for tests
    logger *slog.Logger
    in     chan []byte       // buffered, size 16
}

func (e *assistantTurnEmitter) Enqueue(p []byte)            // bridge observer call site
func (e *assistantTurnEmitter) Run(ctx context.Context)     // single drain goroutine
func startAssistantTurnBridge(ctx, sup, bridge, disp, logger) func()  // wiring
```

`Enqueue` copies `p` (the supervisor reuses its read buffer) and non-blockingly
sends onto `in`. Queue full → drop + WARN, never block. The PTY drain is
never delayed.

`Run` reads from `in`. Per chunk:

1. Read `sup.CurrentConversation()`. Empty → DEBUG drop (no inbound
   `send_message` has stamped the cursor yet).
2. Mint a `MessageID` via `conversations.NewID()` (crypto/rand UUIDv4). Failure
   → WARN drop.
3. Build `protocol.MessagePayload{ConversationID, MessageID, Role: "assistant",
   Text: string(chunk)}`, marshal, then for each `c := range disp.ActiveConns()`
   build `Envelope{ID: c.NextID(), Type: TypeMessage, TS: time.Now().UTC(),
   Payload: ...}` and call `c.Send(ctx, env)`. Per-conn `Send` errors are
   DEBUG (transport disconnects are normal).

`startAssistantTurnBridge` registers `emitter.Enqueue` on the bridge,
spawns the Run goroutine, and returns a cleanup that clears the observer
and waits on ctx-cancel. Cleanup deliberately does NOT close `in` — a
PTY-drain `Write` racing with cleanup could call `Enqueue` after the
observer is cleared (the bridge releases `mu` before invoking the
observer); sending to a closed channel would panic. Instead we let
ctx-cancel drain Run and the buffered chunks fall to GC.

### Wiring (`cmd/pyry/relay.go`, `cmd/pyry/main.go`, `internal/sessions/session.go`)

- `startRelay` grows two parameters: `sup *supervisor.Supervisor` and
  `bridge *supervisor.Bridge`. When `bridge != nil` (service mode), it
  calls `startAssistantTurnBridge` after the dispatcher and handlers
  are constructed but before the dispatcher goroutine spawns. Cleanup
  stops the assistant-turn observer first, so no new PTY chunks queue
  while the dispatcher is winding down.
- `main.go` reads the bootstrap session and passes
  `bootstrap.Supervisor()` and `bootstrap.Bridge()` through.
- `(*Session).Supervisor()` and `(*Session).Bridge()` are new
  one-line accessors on `internal/sessions/session.go`. The bridge
  pointer is `nil` in foreground mode; `startRelay` gates on it.

Alternative considered: extend `handlers.TurnWriter` to expose both
handles. Rejected — `TurnWriter` exists to satisfy a narrow inbound
contract, and the bootstrap session already wires both surfaces, so
passing them directly is one fewer indirection.

### Routing topology: broadcast (v1)

The bridge fans every assistant chunk out to **every** currently-active
phone conn. No `conversation_id → conn_id` map. Pyrycode v1 is
single-paired-phone (one token, one conn at a time in practice), and a
subscriber set of size ≤ 1 doesn't warrant the routing table. Future
subscription semantics replace the broadcast call without touching the
supervisor or bridge sides.

### fakeclaude `PYRY_FAKE_CLAUDE_ASSISTANT_TRIGGER`

`internal/e2e/internal/fakeclaude/main.go` adds one optional env var that
names a file watched in parallel with `PYRY_FAKE_CLAUDE_TRIGGER`. When the
file appears, its contents (capped at 64 KiB) are written to `os.Stdout`,
`Sync`'d, then removed. Repeat-firing is supported. Default off — when unset,
no watch and no behavioural drift. Same opt-in-by-env posture as the existing
rotation trigger and stdin-log seam from #323.

`StartRotationWithRelay` grew an `extraEnv ...string` variadic so the
assistant-turn e2e can inject `PYRY_FAKE_CLAUDE_ASSISTANT_TRIGGER=<path>`
without forking the helper.

## Data flow

```
claude PTY output
    │
    ▼
ptmx Read → io.Copy(bridge, ptmx) → Bridge.Write
                                       ├── observer(chunk) → emitter.Enqueue
                                       │                            │
                                       └── attached out.Write       │  (drop on full)
                                                                    ▼
                                                       emitter.Run goroutine
                                                                    │
                                              sup.CurrentConversation()? empty → drop
                                                                    │
                                                       MessagePayload + marshal
                                                                    │
                                                       for c := range disp.ActiveConns()
                                                                    │
                                                       Envelope{ID: c.NextID(),
                                                                Type: TypeMessage,
                                                                Payload, TS}
                                                                    │
                                                       c.Send(ctx, env)  →  dispatch.outbound
                                                                                       │
                                                                            cmd/pyry forwarder
                                                                                       │
                                                                            transport.Client.Send
```

## Concurrency model

- **Producer:** the supervisor's PTY-drain goroutine. Calls
  `Bridge.Write` → `observer(p)` → `emitter.Enqueue(p)`. Enqueue copies
  the chunk and non-blockingly sends onto a buffered channel
  (size 16); drop-on-full logs WARN with `chunk_len` only.
- **Consumer:** one long-lived `emitter.Run` goroutine. Reads from `in`,
  blocks on per-conn `c.Send` — the dispatcher's outbound channel
  provides natural backpressure on the consumer side, which is the
  correct place to block (not on the PTY drain).
- **Cursor read:** `sup.CurrentConversation()` is mutex-guarded under
  `convMu` (leaf-only, #312). Taken once per chunk, briefly.
- **Snapshot semantics:** `ActiveConns` is a stale snapshot. A conn that
  closes between snapshot and `Send` either has its frame queued
  (dropped at transport on disconnect, #307) or blocks until ctx-cancel.
- **Lifecycle:** ctx-cancel → `c.Send` returns ctx.Err → loop returns.
  Cleanup chains: clear observer → wait for Run.

No new locks. The bridge's `mu` guards the observer slot; the
dispatcher's `mu` guards `conns`; the supervisor's `convMu` guards the
cursor. None nest with each other.

## Security posture

- **PTY chunk bytes are NEVER logged at any level** — not in WARN, not
  in DEBUG, not via wrapped errors. Logs carry only `chunk_len`,
  `conversation_id`, `message_id`, and `conn_id`. The marshal-error
  branch in particular avoids `err.Error()` because the encoder echoes
  invalid input bytes into its error text.
- The chunk reaches the phone via `MessagePayload.Text` **verbatim** —
  no decode, strip, or normalisation. ANSI escapes from real-claude TUI
  output reach the phone as-is; render-time escape handling is the
  phone's problem (and a future "real-claude assistant-turn parser"
  slice's concern). Documented in the spec under "Open questions" #3.
- The fakeclaude assistant trigger file's content is bounded to 64 KiB
  to prevent a misbehaving test from emitting unbounded chunks to
  stdout in an environment with no flow control.

- **`internal/e2e/relay_assistant_turn_test.go`** (build tag `e2e`) —
  full wire path: pair → seed conversations.json → fakerelay → daemon →

  assistant trigger → assert `message` envelope on the phone with

  by looping receives until the marker text is observed.

## Lessons learned

### A bool published before its invariant doesn't gate anything

PR #326's review caught that `gateRun = true` was set **before** `runGate`
ran — which is exactly what you want for the "don't re-run the gate"
purpose the flag was originally invented for, but the worst possible
placement for "publish broadcast-eligibility after the gate's NextID
advance." Splitting the flag into two (local `gateStarted` for the local
purpose; cross-goroutine `gateCompleted` for the eligibility purpose)
was less work than trying to make one flag carry both meanings while
also moving its assignment site.

General shape: when one boolean serves two consumers and they disagree
about WHEN the boolean should be set, that's the diagnostic — split it.

### Defensive log fields, not defensive code

The marshal-error branch in `broadcast` cannot fail in practice
(`MessagePayload` is a closed struct of strings) but is logged defensively.
The discipline that matters is the field list, not the branch itself:
log `chunk_len` + `conversation_id` + `message_id` and **never**
`err.Error()`, because `encoding/json`'s error format quotes the input
bytes back into the error string. If the chunk happens to contain
invalid UTF-8, the operator's WARN log gains an attacker-controlled
payload echo.

### Don't close a channel the bridge can still write to

The first natural cleanup shape — "clear observer, close `in`, wait for
Run" — has a panic window: `Bridge.Write` captures the observer under
`b.mu`, releases the lock, **then** invokes the observer. A `Write`
racing with cleanup can hold a reference to `emitter.Enqueue` after the
observer slot is nilled and panic on the next `select { in <- chunk }`.
Letting ctx-cancel drain Run and relying on GC to reclaim the
not-yet-closed `in` chan is the simpler, race-free shape. Same
"channel-close races writer" pattern surfaces in #307's outbound-chan
discipline.

### Per-ticket file convention works through doc-phase race scenarios

This is the first ticket where the doc-phase's worktree was wiped twice
during pipeline iteration (max_turns salvage on commit `37dc167`, spec
rework after PR #326's review on `15d894c`). Per-ticket
`docs/knowledge/codebase/311.md` was rewritten each time without ever
touching a sibling ticket's file — exactly the shared-append safety the
convention was designed to provide. Pre-2026-05-11 frozen blocks on
`PROJECT-MEMORY.md` / `lessons.md` were never touched.

## Out of scope (carried forward)

- **Owner-conn / subscription routing.** Broadcast for v1; replaces the
  inner fan-out loop when subscription semantics land.
- **Real-claude assistant-turn parser.** The bridge tees raw PTY bytes;
  parsing what counts as a "turn" against the real claude TUI is a
  separate slice.
- **Backpressure tuning.** Fixed buffer-16, drop-on-full WARN. No
  coalescing, no per-conversation queueing.
- **Outbound rate-limiting.** Trust the supervised process.
- **User-turn echo to other paired devices.** The protocol allows
  `Role: "user"` echoes for multi-device awareness; only `assistant`
  is emitted here.
