# Ticket #432 — `pyry pair`: include `server_static_pubkey` + 64-bit fingerprint in QR payload

Grows the QR pairing payload from `{server, relay, token}` to `{server, relay, token, server_static_pubkey}` and adds an 8-byte BLAKE2s-256 fingerprint line under the QR for operator-driven visual verification against the phone's confirm-pairing screen. This is the first production consumer of the [#438](438.md) / [#439](439.md) `internal/keys` static-keypair primitive — the binary side of Mobile Protocol v2 ([ADR 024](../decisions/024-noise-ik-mobile-e2e.md)) now has the trust anchor the phone needs for `Noise_IK` before the first message.

## Implementation

- **`internal/pair/payload.go`** — `Payload` gains `ServerStaticPubkey string` with JSON tag `server_static_pubkey`, in last position to match `docs/protocol-mobile.md` § *Pairing flow* / § *Appendix*. Package doc-comment pins `base64.StdEncoding` (padded) as the alphabet — same string form as the on-disk encoding in `internal/keys`. `Decode` extends with three new rejection categories, layered in order after the existing missing/empty checks: missing/empty field → `missing field: server_static_pubkey`, non-base64 → `invalid server_static_pubkey encoding`, decoded length ≠ 32 → `server_static_pubkey wrong length`. All three wrap `ErrInvalidPayload` and preserve the "category only, never input bytes" leak discipline (`internal/pair/payload.go:113-126`).
- **`internal/pair/render.go`** — new exported `Fingerprint(pubkey [32]byte) string` returns `BLAKE2s-256(pubkey)[:8]` formatted as colon-separated lowercase hex (`aa:bb:cc:dd:ee:ff:11:22`, exactly 23 chars). Signature takes `[32]byte` not `[]byte` so callers cannot accidentally pass a buffer of the wrong length. `Render` writes one new line between the encoded-payload line and the instruction line: `Static-key fp: <fp>  (verify this matches the fingerprint shown on your phone)`. Unexported `fingerprintLine(pubkeyB64 string) string` handles the base64-decode + length-32 check and falls back to the literal `Static-key fp: <invalid>` (still on the same line, still with the verify hint) if a caller hand-builds a `Payload` that bypasses `Decode`'s validation — keeps `Render` non-panicking on its contract surface.
- **`cmd/pyry/pair.go`** — new local helper `resolveStaticKeyBaseDir() string` returns `~/.pyry` when HOME resolves, `""` otherwise (mirrors `resolveDevicesPath` / `resolveServerIDPath`'s fallback shape; `keys.LoadOrCreate("", name)` then writes to `./<name>/static_key.json`). In `runPairDefault`, `keys.LoadOrCreate(resolveStaticKeyBaseDir(), sanitizeName(parsed.instanceName))` runs after `identity.LoadOrCreate` and before the `crypto/rand` token mint (slots into the existing "all loads before the random draw" discipline from [ADR 021](../decisions/021-pair-cli-order-of-operations.md)). The `pair.Payload` literal grows a fourth field `ServerStaticPubkey: base64.StdEncoding.EncodeToString(pub[:])` where `pub := staticKey.PublicKey()` is the `[32]byte` by-value accessor. `StaticKey.PrivateKey()` is never read from this call site — grep-enforceable invariant.
- **`golang.org/x/crypto/blake2s`** new dep in `go.mod`. Pulled in via `go get`; minor revision bumps on `x/term` (0.42 → 0.43) and `x/sys` (0.43 → 0.44) ride along. Same hash mandated for `Noise_IK_25519_ChaChaPoly_BLAKE2s` per [ADR 024](../decisions/024-noise-ik-mobile-e2e.md), so the dep is forward-compatible with the upcoming Noise wrapper (#433).
- **Width is 8 bytes (64 bits), not 4.** The truncation width is load-bearing security per `docs/protocol-mobile.md` § *Security review* → *[Cryptographic primitives]*: a 32-bit fingerprint is brute-forceable (≈ 2³² preimage search ≈ a few CPU-hours). The test vector pins the width byte-for-byte; a 4-byte vector is a spec violation, not a test bug.
- **Tests pinned.** `TestEncode_DecodeRoundTrip` / `TestEncode_Format` / `TestEncode_Stable` / `TestDecode_AcceptsTrailingWhitespace` extended with `testStaticPubkeyB64 = "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA="` (32 zero bytes). `TestDecode_Errors` gains six new rows: missing field, empty string, non-base64 (`!!!`), too-short (31 bytes), too-long (33 bytes), plus updated sentinel-leak rows that carry the new field. New `TestFingerprint_FixedVector` hard-codes the BLAKE2s digest of 32 zero bytes (`32:0b:5e:a9:9e:65:3b:c2`) — **not** recomputed at test time from `blake2s.Sum256`, which would be tautological. New `TestFingerprint_LengthAndShape` pins the regex `^[0-9a-f]{2}(:[0-9a-f]{2}){7}$` and length 23 independent of hash value. `TestRender_Format_Happy` / `TestRender_FieldOrder` assert the fingerprint substring + hint substring + ordering (QR → encoded → fingerprint → instruction). New `TestRender_InvalidPubkey_DoesNotPanic` pins the `<invalid>` fallback. New `TestRunPairDefault_PopulatesStaticPubkey` in `cmd/pyry/pair_test.go` captures stdout via an inline `captureStdout` helper (replaces `os.Stdout` with a pipe), decodes the rendered payload via `pair.Decode`, asserts pubkey non-empty + 32 bytes after base64 decode, and runs `runPairDefault` twice asserting **same pubkey across invocations** — verifies `keys.LoadOrCreate` returns the persisted key on second call, not a fresh one. `internal/e2e/pair_test.go` gains a two-line `payload.ServerStaticPubkey == ""` assertion in each happy-path sub-test so the producer side fails loud if the field is ever dropped.

## Patterns established

- **The `(baseDir, daemonName)` shape for `internal/keys` extends to its first production consumer without surprise.** `runPairDefault` resolves `~/.pyry` via a local helper mirroring `resolveDevicesPath` and passes `sanitizeName(parsed.instanceName)` for the daemon name. The intentional mismatch between `cmd/pyry/main.go:sanitizeName` (permissive transformer) and `keys.validDaemonName` (strict rejector) — pinned in [#438](438.md) under *Patterns established* — surfaces here as a clean `pyry: pair: keys: invalid daemon name "Foo"` error rather than a silent-fallback. **No auto-rewrite, no `strings.ToLower`-and-retry, no synchronization between the two functions.** The fix for a non-conforming name is operator-side: rename the instance.
- **Decoder-error category strings extend by category, not by field.** The new `missing field: server_static_pubkey` / `invalid server_static_pubkey encoding` / `server_static_pubkey wrong length` strings reuse the existing `Decode`-error vocabulary patterns. The "never echo input bytes; field names are public protocol vocabulary" discipline scales by adding rows, not by inventing new wrapping shapes. A future field add (v3 envelope?) drops in at the same anchor.
- **Belt-and-suspenders is two layers with different jobs, not one layer twice.** `Decode` rejects malformed pubkeys via the three-rule sequence — that's the load-bearing validator. `Render`'s `<invalid>` fallback is the no-panic safety net for the unreachable-via-Decode path where a caller hand-builds a `Payload`. The two are different fabrics (validation vs. defensive output), serving different threats (round-trip failure vs. caller misuse). Belt-and-suspenders patterns in the project consistently follow this shape: see [#439](439.md)'s `Lstat`-mode-check + `O_NOFOLLOW` for the canonical example.
- **Exported `Fingerprint([32]byte) string` instead of an inline private helper.** The formatter is pure, deterministic, and load-bearing for security — exporting it lets it be tested in isolation against a fixed vector and reused if/when the same fingerprint surfaces in `pyry pair list` output, the eventual mobile-team's binary-fingerprint comparison flow, or operator-facing `pyry status`. The `[32]byte` (not `[]byte`) parameter type makes wrong-length input a compile error, not a runtime error.
- **Test-vector discipline: hardcode the expected value, never derive it from the same primitive at test time.** `TestFingerprint_FixedVector` pins `"32:0b:5e:a9:9e:65:3b:c2"` as a literal. Computing the expected via `blake2s.Sum256(make([]byte, 32))[:8]` inside the test would be a tautology — it would pass against any implementation that runs the call, including one that's silently wrong. Hardcoding makes the test fail loud if the hash primitive or truncation width drifts. The pattern generalises to every spec-pinned cryptographic constant; future test vectors should follow.

## Lessons learned

- **The `runPairDefault` end-to-end test pattern (capture-stdout + `pair.Decode` round-trip) is now the model for any future CLI verb that emits a structured payload.** Earlier `pair_test.go` tests stuck to unit-testing pure helpers (`parsePairArgs`, `resolveRelay`, `renderPairList`) and pushed end-to-end behaviour into the `e2e` build-tagged tests. `TestRunPairDefault_PopulatesStaticPubkey` bridges the gap — it drives `runPairDefault` directly, captures stdout via an inline pipe, and decodes the rendered payload back. Faster than `e2e` (no subprocess spawn) and catches regressions in the verb's wiring without paying the cost of a full binary build. The inline `captureStdout` helper is the only consumer today; lift to a shared `cmd/pyry/internal/testutil` package if/when a second verb needs it. The "two-invocation same-output" assertion shape (running `runPairDefault` twice and asserting the persisted state is reused) is reusable for any verb whose first-run-mints/second-run-loads contract matters.
- **Adding a required field to a wire protocol's decoder makes every fixture-bearing test fail at once.** The Decode-side trailing-whitespace test, the round-trip test, the format test, the stable-order test, the field-order test, the `samplePayload()` helper in render tests, and **every** `TestDecode_Errors` row that fed valid JSON-shapes-but-tested-some-other-rejection — all needed the new field added. This is the right failure mode (loud, structural, impossible to miss) but the surface area is wider than you'd guess from "one new required field." Future required-field adds to the QR payload should expect to touch every `validJSON` constant in the test file plus every render-test that builds a `Payload` literal. The sentinel-leak rows in `TestDecode_Errors` are particularly easy to miss — they construct payloads that intentionally violate something *else* (the server-id rule, the trailing-data rule) but still need a structurally-valid pubkey field to reach those checks.
- **The CR-flagged "no separate doc for v3 deferral" pattern is structurally correct.** The issue body, the architect spec, and this codebase note all list rotation / re-pair / multi-binary-fingerprint-UI as deferred to v3 without filing follow-up tickets. v3 is far enough out (no spec, no schedule) that filed tickets would rot. The pattern: when "out of scope" items are pinned by a parent spec ([ADR 024](../decisions/024-noise-ik-mobile-e2e.md) names rotation as v3 in § *Static keys — binary side*), the deferral lives in the spec, not in a ticket. Compare with [#438](438.md)'s out-of-scope list, which **does** file #439 because the follow-up is the *blocking prerequisite* for production consumers — same pattern, different urgency tier.

## Out of scope (deferred)

- **Phone-side handling of the new field** — pyrycode-mobile track. The binary now emits the field; the phone team consumes it in the Noise_IK initiator path (also deferred — `flynn/noise` wrapper is #433).
- **Static-key rotation / re-pair flow** — v3 per [ADR 024](../decisions/024-noise-ik-mobile-e2e.md) § *Static keys — binary side*. No ticket filed; v3 surface is still un-spec'd.
- **Multi-binary fingerprint comparison UI** — a future `pyry pair list` column or a dedicated `pyry whoami --fingerprint` verb. Out of scope here; the exported `Fingerprint([32]byte) string` is ready when that lands.
- **Aligning `cmd/pyry/main.go:sanitizeName` with `keys.validDaemonName`** — same deferral noted by [#438](438.md). `PYRY_NAME=Foo` produces `~/.pyry/Foo/devices.json` from the existing helpers but errors at `keys.LoadOrCreate(...)` with `ErrInvalidDaemonName`. Operator-facing fix is to rename the instance; tightening the CLI-wide validator is a larger sweep.
