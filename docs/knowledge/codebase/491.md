# Ticket #491 ŌĆö e2e/realclaude: guard plain-`WithWorktree` tests with auth-or-skip

Consumer-side flip in [`internal/e2e/realclaude`](../features/e2e-realclaude.md): 5 test files that call `pyry agent-run` end-to-end against real `claude` now use [`WithWorktreeAuthenticated(t)`](../codebase/490.md) instead of plain `WithWorktree(t)`, inheriting the dual-env-var skip logic landed in [#490](490.md). With neither `ANTHROPIC_API_KEY` nor `CLAUDE_CODE_OAUTH_TOKEN` set in the outer environment the 9 affected tests (4 file-level + 5 `*_RoleLoop` subtests via the shared `runRoleSmokeTest` helper) now skip in milliseconds with the named-variable diagnostic from #490, replacing the prior failure mode where each test hit either a 31 s ptyrunner deadline timeout or a 5 s streamrunner exit-1 fail depending on the default runner. The fixture itself is unchanged; this ticket is purely the 5-call-site update that #490's "follow-up" line item named.

## Implementation

- **Five one-line edits, identical shape, no other changes.** Each affected file replaces exactly one `workdir := WithWorktree(t)` line with `workdir := WithWorktreeAuthenticated(t)`. No imports added (both helpers live in the same package under the same build tag), no signature shifts, no new helpers, no assertions touched. Locations:
  - [`prompt_fidelity_test.go:29`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/prompt_fidelity_test.go) ŌĆö `TestRealClaude_PromptFidelity` (#364)
  - [`prompt_fidelity_unicode_test.go:31`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/prompt_fidelity_unicode_test.go) ŌĆö `TestRealClaude_PromptFidelity_Unicode` (#419)
  - [`tool_loop_test.go:28`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/tool_loop_test.go) ŌĆö `TestRealClaude_ToolLoopIntegrity` (#376)
  - [`allowed_tools_enforcement_test.go:38`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/allowed_tools_enforcement_test.go) ŌĆö `TestRealClaude_AllowedToolsEnforcement` (#365 + #420)
  - [`per_agent_test.go:85`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/per_agent_test.go) ŌĆö `runRoleSmokeTest` shared helper (#381). All 5 `*_RoleLoop` subtests (`PO`, `Architect`, `Developer`, `CodeReview`, `Documentation`) inherit the guard transparently because the helper is the single point of entry ŌĆö one edit covers five subtests.

- **Shape selection: direct call-site replacement, not a new `RequireAuth(t)` wrapper.** The architect spec offered two shapes; Option 1 (call `WithWorktreeAuthenticated` directly) won on three grounds. (1) It avoids duplicating the dual-env-var skip logic that already lives in `WithWorktreeAuthenticated` ŌĆö a second helper would mean two places to maintain the skip message, two places where a future env-var addition (e.g. `ANTHROPIC_AUTH_TOKEN`) would have to land. (2) It makes the test's auth requirement structurally explicit at the call site ŌĆö readers see the fixture name `WithWorktreeAuthenticated` and immediately know auth is required, without scanning above for a separate guard line. (3) A verb-shaped alias like `RequireAuth(t)` would fragment the `grep`-able search surface (`grep WithWorktreeAuth` would miss it).

- **Audit confirmed all 5 files genuinely spawn claude end-to-end.** Per the AC item asking to verify each affected test actually needs auth (not just argv-shape probes), every file invokes `RunPyryAgentRun(t, RunOpts{...})` with the default `UseTestBinaryAsFakePyry=false` (real `pyry agent-run` subprocess), and the assertions land on values that only a real claude response can produce: byte-survival of the prompt literal in the JSONL `user` entry (#364, #419), tool-use ŌåÆ tool-result correlation across the multi-turn loop (#376), the claude-side `--allowed-tools` deny-by-default contract (#365 + #420), and per-role trailer fields `num_turns >= 1` + tail-side `EndOfTurn=true` assistant event (#381's five roles). No test in scope is argv-shape-only ŌĆö the audit risk Option 1 carried is discharged.

- **`TestRealClaude_MalformedStreamJSON` at [`resilience_test.go:177`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/resilience_test.go) is explicitly out of scope despite also calling `WithWorktree(t)`.** Codegraph flagged it as a 16th plain-`WithWorktree` caller during architecture review. Excluded because it spawns claude **directly** via `runClaudeDirect` (not via `pyry agent-run`) with deliberately-malformed stdin and asserts a parse-shaped diagnostic ŌĆö the expectation is that claude's stream-json parser rejects the garbage before the auth check matters. The hazard worth flagging for future readers: that test's stderr-keyword predicate (`json | parse | input | format | envelope`) is loose enough that a "not logged in" diagnostic containing `input` or similar could pass spuriously without auth. Worth a follow-up if observed in practice; not worth expanding XS scope here.

- **Skip-message contract is inherited from #490, not duplicated at the call site.** [`TestWithWorktreeAuthenticated_SkipsAndNamesBothEnvVarsWhenNeitherSet`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures_test.go) (added in #490, lines 104ŌĆō177) already pins the named-variable + Keychain-extraction-recipe substrings via the outer/inner subprocess re-exec pattern. Adding per-file skip-message assertions on the 5 newly-converted tests would couple every test to the message string and turn a single message tweak into a 6-file edit. The 5 affected tests get one skip line each when neither env var is set ŌĆö the AC's "single named-variable skip messages" wording is satisfied at the test-binary level (one `--- SKIP: TestName` line per skipped test), not via duplicate contract assertions.

- **Behaviour under auth set in the outer env is unchanged from the prior `ANTHROPIC_API_KEY`-only baseline.** When either credential is present, `WithWorktreeAuthenticated` falls through to `WithWorktree(t)` for `$HOME` isolation, re-pins whichever env var(s) the operator exported via per-variable `if value != ""` guards, and returns the same workdir path the prior plain call would have returned ŌĆö `RunPyryAgentRun`'s `os.Environ()` inheritance carries the credential into the subprocess deterministically. Existing assertions in each test fire unchanged. The behavioural delta visible to operators is exactly the skip path: ~0.00 s skips replace 31 s timeouts and 5 s exit-1 fails.

## Patterns established

- **When a fixture's contract widens, flip consumers to the existing widened helper rather than introducing a second helper.** #490 widened `WithWorktreeAuthenticated` to accept both `ANTHROPIC_API_KEY` and `CLAUDE_CODE_OAUTH_TOKEN`. The natural temptation in #491 was to introduce a `RequireRealClaudeAuth(t)` one-liner that callers could place before `WithWorktree(t)`. Resisting that temptation kept the contract single-sourced: the dual-env-var skip logic, the operator-facing diagnostic, and the per-variable re-pin discipline live in exactly one function ŌĆö the same one that future credential mechanisms (a third env var, a token file path, an OS-keyring lookup) will be added to. Generalises: a fixture-widening landing in one PR + a consumer-flip landing in a follow-up PR is the clean split when the widening preserves the signature and proceed-path semantics; do not introduce a sibling helper "to keep the call sites simple" ŌĆö the original helper's name already encodes the contract.

- **Audit-then-flip discipline for tests that "look like" argv-shape probes but spawn the real upstream.** The 5 affected tests historically called plain `WithWorktree(t)` even though they invoked `pyry agent-run` end-to-end against real claude. That mismatch (auth-required test using the no-auth fixture) was the silent-skip masquerade #490 closed at the fixture layer; #491 closes it at the call-site layer. Before flipping any test from `WithWorktree` to `WithWorktreeAuthenticated`, the audit must confirm the test actually spawns the upstream ŌĆö converting an argv-shape-only test (no real claude invocation) would introduce a spurious auth gate. Concretely: grep for `RunPyryAgentRun(t, RunOpts{...})` with `UseTestBinaryAsFakePyry=false` (or absent ŌĆö the field defaults to false), and verify the test's assertions are on values only a real response can produce. If both conditions hold, flip is safe; if either fails, the test was correctly using `WithWorktree` and should stay.

- **A 16th-caller "while I'm in here" is XS-scope creep, not thoroughness.** Codegraph surfaced one additional `WithWorktree` caller (`TestRealClaude_MalformedStreamJSON`) that didn't fit the ticket's auth-requirement criterion ŌĆö it spawns claude directly with malformed stdin and expects rejection before the auth check matters. Expanding the ticket to also touch that file would have moved the change from 5 files to 6 files, traded an XS estimate for a longer review, and added a defensive-skip to a test whose existing behaviour isn't broken. The right move was to name the caller in the spec's "out of scope" section with the precise reason it doesn't qualify (different spawn path, parse-shaped contract) and the hazard to watch (loose stderr predicate could mask a spurious pass), then stop. Generalises: when a codebase-wide query returns N+1 candidates and one doesn't fit the ticket's criterion, the discipline is to document the asymmetry, not absorb it.

## Lessons learned

- **The PR body's "Lessons learned" section was correctly omitted.** This ticket was mechanical end-to-end: the contract was already pinned by #490's fixture test, the architect spec explicitly forbade per-file skip-message assertions, the code review was PASS with no findings, and CI was green on the first push. There were no non-obvious surprises during implementation. Generalises: not every ticket produces a lesson ŌĆö and inventing one to fill the section is anti-signal for future readers scanning the codebase notes for genuine gotchas. The codebase doc's "Lessons learned" section is allowed to be empty when the work itself produced none; the value is in the lessons that DID land elsewhere (e.g. the audit-then-flip pattern above, which is a pattern, not a lesson, and goes under "Patterns established").
