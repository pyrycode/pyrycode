# Ticket #527 ŌĆö `agentrun.ExitErrIsBenign` filters benign teardown errors from the WARN log surface

`pyry agent-run --max-turns N` emitted one misclassified WARN on every budget-driven termination: `WARN ptyrunner: close failed err="exit status 143"` ŌĆö claude exited from the SIGTERM the budget watcher itself sent (143 = 128 + SIGTERM(15)), and the teardown path framed the expected exit as a failure. The same shape recurred at three more sites in `internal/agentrun/{budget,streamrunner}`. With `max-turns:90` being the routine path for many tickets, this would become a one-WARN-per-run noise floor once tui-driver runtime ships. Fix: a shared `agentrun.ExitErrIsBenign` predicate in the parent package; each of the four documented sites filters its returned error through the predicate before logging ŌĆö benign ŌåÆ DEBUG, non-benign ŌåÆ original WARN unchanged. Companion bug from the same dispatcher-shape verification was [#526](526.md).

## Implementation

- **Predicate is exported from the parent `internal/agentrun` package.** `internal/agentrun/exitclass.go` (+45 LOC) defines `func ExitErrIsBenign(err error) bool` covering five benign classes: `syscall.ESRCH`, `syscall.EPIPE`, `os.ErrClosed`, `*exec.ExitError` with `ExitCode() == 143 || 137`, and `*exec.ExitError` whose `Sys().(syscall.WaitStatus)` reports `Signaled() && Signal() in {SIGTERM, SIGKILL}`. The fifth (signal-killed) branch goes beyond the literal AC ("ExitCode() == 143 or 137") and covers the case where a child without a signal handler reports `ExitCode() == -1` ŌĆö ~6 LOC and one comment for one bug class avoided. `nil` returns `false` per AC (explicitly non-benign). Wrapped errors traverse `errors.Is` / `errors.As`. **Exported** because the three consuming subpackages live under it; the AC's lowercase `exitErrIsBenign` was informal ŌĆö the binding requirement was single-predicate single-source-of-truth, which an unexported name would have broken.

- **Four call sites adopt the same if/else transform.** All four log at DEBUG on the benign branch with structured fields preserved (`count`, `max_turns`, `reason`) ŌĆö drop was permitted by the AC, DEBUG kept for operator observability when debug logging is explicitly enabled:
  - `internal/agentrun/ptyrunner/runner.go:283` ŌĆö `defer sess.Close()`. Captures `*exec.ExitError{ExitCode=143|137}` from claude's SIGTERM-handled exit bubbled through `tui-driver.Session.Close()`.
  - `internal/agentrun/budget/budget.go:138` ŌĆö `cfg.Terminate()` race: budget terminates after claude already exited normally ŌåÆ `ESRCH`.
  - `internal/agentrun/budget/budget.go:201` ŌĆö `cfg.Kill()` after grace window: process exited from SIGTERM during the grace window; SIGKILL hits `ESRCH`.
  - `internal/agentrun/streamrunner/runner.go:170` ŌĆö `stdin.Close()` after envelope write: claude exits early, close hits `EPIPE` / `os.ErrClosed`.

- **The ten unrelated WARN/ERROR sites in the same three packages are untouched.** Pre-run spawn failures, trust/MCP/network modal detection, watchdog hangs, grace-period elapsed (the WARN preceding the kill, not the kill error itself), and mid-write stdin failures all remain unchanged. The predicate is opt-in per call site; scope-verification grep returns exactly the definition + four production call sites (plus test-file references).

- **No import cycle.** `internal/agentrun/workdir.go` (the parent's only other production file) imports only stdlib, so the three subpackages can safely import the parent for the predicate. The ptyrunner package already imports parent subpackages (`budget`, `streamjson`), so the new parent-package import sits naturally alongside.

- **Predicate unit tests (`exitclass_test.go`, +125 LOC)** are table-driven, parallel, stdlib-only. Construct realistic `*exec.ExitError` shapes via a `TestMain` re-exec pattern (`GO_EXITCLASS_HELPER` env modes `exit143` / `exit137` / `exit1` / `block`) ŌĆö `exit143` / `exit137` capture exit codes; `block` is spawned then signal-killed by the test goroutine to produce the `Signaled()==true, ExitCode()==-1` shape. Cases cover every benign class raw + wrapped via `fmt.Errorf("ctx: %w", ...)`, plus negatives for `nil`, plain `errors.New`, `*exec.ExitError{ExitCode=1}`, and wrapped non-benign errors. The re-exec pattern produces real `*exec.ExitError` values without depending on test-side syscalls ŌĆö clean construction.

- **ptyrunner regression test (`TestRun_MaxTurnsExhaustion_NoBenignWarns`, +51 LOC)** wires a captured slog handler at `LevelWarn` into `cfg.Logger` via a new `loggerSyncWriter` (serialises Write ŌĆö slog handlers may run concurrently from defers and timer goroutines), reuses `helperRunCfg` with a new `jsonl_exit143` helper mode that installs a SIGTERM handler exiting with code 143, sets `MaxTurns: 1`, and asserts the captured buffer contains zero of `ptyrunner: close failed` / `budget: terminate failed` / `budget: kill failed`. Exercises sites 1, 2, and 3 in production-realistic shape (`Session.Close` returns `*exec.ExitError{143}`, predicate suppresses).

- **streamrunner regression test (`TestRun_EarlyExitChild_NoBenignStdinCloseWarn`, +46 LOC)** wires a captured slog handler at `LevelWarn` via a sibling `syncWriter`, reuses `helperRunCfg` with a new `exit0_no_read` helper mode that exits 0 without draining stdin, and asserts the buffer contains no `stdin close failed` record. Site 4 coverage ŌĆö see "Lessons learned" for the caveat the code-review surfaced.

- **Existing budget tests untouched.** `TestTerminateError_DoesNotBlockKill` and `TestKillError_IsLogged` pass `errors.New("simulated ŌĆ”")` (plain-text, not sentinel), so the predicate returns `false` and the WARN still fires ŌĆö these tests assert Kill / log behaviour, not log-shape, so they remain valid without modification. The spec's analysis predicted this and it held.

## Patterns established

- **Shared teardown-error predicate as the single source of truth for "what counts as benign exit noise?"** The "the OS returns an error on an operation against an already-exited process, but the goal of the operation was getting the child gone" shape recurs at every teardown layer (kill, signal, stdin close, fd close, Session.Close bubbling exit codes). Each layer used to need its own ad-hoc `errors.Is(err, syscall.ESRCH)` chain or be silently noisy; centralising the predicate means adding a new benign class (e.g. a sixth `errors.Is` target) is a one-file edit, not a four-file edit drifting out of sync. **Generalises: when Ōēź3 sibling sites need to recognise the same OS-level "expected during teardown" error class, define one exported predicate and reference it from each site rather than inlining the `errors.Is` / `errors.As` chain.** The AC's scope-verification clause (`grep -rn 'ExitErrIsBenign' internal/agentrun/` returns exactly the documented sites) is the structural enforcement.

- **Per-site if/else over a `logIfBenign` helper.** Each guard could have collapsed to `logIfBenign(logger, benignMsg, warnMsg, fields, err)` ŌĆö and was considered ŌĆö but the structured-field shape differs at each site (`count` / `max_turns` at budget terminate, `reason` at budget kill, no extra fields at ptyrunner close / streamrunner stdin close), and the WARN message text is grep-load-bearing for existing operator log searches. The if/else preserves both. **Generalises: when a "log either WARN or DEBUG depending on a predicate" transform appears at <5 sites with divergent structured-field shapes, inline the if/else at each site over extracting a helper ŌĆö the per-site code is ~6 LOC, the helper would need to be generic over field shape and the cognitive load of "what fields does this site log?" stays local.**

- **The `Signaled() + Signal()` defensive branch beyond the literal AC.** The architect's spec called this out as borderline against the "Evidence-Based Fix Selection" principle (don't defend against an unobserved failure mode). Decision: ship it. Cost: ~6 LOC + one comment. Justification: the literal AC's `ExitCode() == 143 || 137` covers the case where the child *has* a signal handler that calls `os.Exit(143)` (claude's current behaviour); the signal-killed branch covers the case where the child has *no* handler and the stdlib reports `ExitCode() == -1` with `Signaled() == true` (which is the kernel's default for SIGTERM/SIGKILL). If claude ever drops its signal handler (or the production path ever wraps a process that doesn't have one), the literal-AC predicate would silently reclassify the exit as non-benign and a WARN would reappear. **Generalises: when an evidence-based predicate has a structurally adjacent "same intent, different OS-reported shape" case that costs <10 LOC to cover, including it is preferable to deferring ŌĆö the marginal cost is small and the marginal benefit is one bug class precluded. The "Evidence-Based Fix Selection" principle binds against adding *defensive code paths* for unobserved failure modes, not against making a predicate's structural coverage match its intent.**

## Lessons learned

- **The streamrunner regression test is weak by construction; the predicate's unit coverage carries the load.** The code-review surfaced a `[SHOULD FIX]`: `TestRun_EarlyExitChild_NoBenignStdinCloseWarn` asserts no `stdin close failed` record after the child exits early, but with `PromptBytes = []byte("hi")` (~2 bytes plus the JSON envelope, well under `PIPE_BUF`), the kernel buffers the write even when the read end is gone, so `stdin.Write` returns nil; `os.File.Close` on a pipe's write end then returns nil regardless of readers ŌĆö `EPIPE` surfaces on `Write`, not `Close`. The absence assertion holds even if the predicate never fires. Two hardening options were called out: (a) add the positive sibling assertion (`strings.Contains(out, "stdin write failed")` should hold when the write *does* race the exit) to confirm no collateral damage to the still-WARN site; (b) provoke a deterministic close error (e.g. a `t.Cleanup`-driven double-close, or wire a writer that returns `os.ErrClosed`). Code review classified as test-quality, not correctness ŌĆö the predicate's own unit tests cover every benign error shape thoroughly, so the regression net at the call site is partial belt-and-suspenders. **Generalises: when a regression test asserts "no log of X" and the production path's error surface depends on buffer-size / FD-state racing, the absence assertion is often a no-op masked by the kernel behaviour ŌĆö pair every absence assertion with a positive sibling that proves the path *does* fire when it should.** Filed as a follow-up test-hardening item; the predicate ships correct.

- **The ptyrunner regression test exercises sites 1, 2, and the Close path of 3 in one shot, but does not exercise the actual Kill-failure branch.** `TestRun_MaxTurnsExhaustion_NoBenignWarns` relies on the SIGTERM-handled exit being fast enough that the kill timer never fires ŌĆö confirms sites 1 and 2 in production-realistic shape, but the Kill path's predicate guard (site 3) is exercised only by the predicate's own unit tests, not by the regression. The spec acknowledged this ŌĆö exercising site 3 would require a helper mode that ignores SIGTERM and lets the grace timer fire SIGKILL into an already-exited shell wrapper, doubling the helper-mode surface for marginal additional coverage. **Generalises: when a regression test's natural shape covers N of M sites and the M-N gap requires substantially more test scaffolding for marginal coverage gain, leaning on the predicate's unit tests for the remaining gap is the right XS/S call.** Same reasoning the spec applied; the code-review concurred.

- **The DEBUG message text "child already exited" is accurate for ESRCH and exit-code 143/137 but slightly off for raw `EPIPE` / `os.ErrClosed`** ŌĆö where the literal cause is "broken pipe" or "fd already closed", both evidence of (but not direct observation of) the child being gone. Code review flagged as a NIT for pedantic accuracy; phrasing is fine for grep / log archaeology and was kept as-is. **Generalises: DEBUG messages on benign-error paths are operator-grep targets, not literal-cause descriptions ŌĆö prefer a single grep-target phrase that catches every benign class over per-class precision.** The grep load-bearing property dominates the literal-accuracy property at DEBUG level.

## Out of scope (follow-ups)

- **Streamrunner regression test hardening.** Code review's `[SHOULD FIX]` on `TestRun_EarlyExitChild_NoBenignStdinCloseWarn`: pair the absence assertion with a positive sibling (`strings.Contains(out, "stdin write failed")` when the write *does* race the exit) or provoke a deterministic close error. The predicate's own unit tests cover the error shapes; this is test-quality, not correctness.
- **Budget-package positive regression for the `ESRCH`-on-`Terminate` path.** If a future change wants to assert end-to-end that the predicate path fires at the budget layer (rather than driving it through ptyrunner), add a sibling test that passes `syscall.ESRCH` as the `Terminate` return ŌĆö confirms the predicate path independently of the ptyrunner integration. Not strictly needed today since the predicate's unit tests + the ptyrunner regression cover the same surface, but tightens isolation.
