## Patterns established

- **Allowlist citations point at the SOURCE OF THE DECISION, not the placeholder ticket that scheduled the work.** #506's initial allowlist entries carried bare `// #503` markers — correct as placeholders ("the audit will tell us why"), incorrect as durable documentation once the audit lands. The right citation shape, once the audit closes, is `// <audit-id> <audit-date>: <one-line reason from the audit's per-identifier table>` — the date pins which audit revision the reason came from, and the one-liner reproduces enough of the audit's verdict that a contributor reading the test source has the answer without opening the audit doc. Generalises whenever an allowlist or tolerance table accumulates entries during an open investigation: gate the entries on a ticket ID while the question is live, then sweep all entries to their per-source-of-truth citation when the answer lands. A bare ticket citation that outlives its audit is technical debt; the audit ID + date + one-liner is durable.

- **"Ditto." is a legitimate code-comment idiom when N entries genuinely share one rationale.** The four `expectedPtyRunnerOnly.Events` additions all derive from the same audit verdict (claude local-JSONL housekeeping; dispatcher default-case preview log only). Repeating the full rationale four times would be noise; omitting the citation entirely would break the "every tolerated divergence cites the audit" invariant; an out-of-band block comment explaining "the next four entries share this rationale" would split the citation off from the entry it justifies. `// <audit-id> <audit-date>: ditto.` on entries 2-4 with the full rationale on entry 1 keeps the citation per-entry (so a future single-entry edit doesn't strand siblings under a now-orphaned block comment) AND honest about the duplication. The pattern only works when the entries are visually adjacent and share IDENTICAL rationale; for entries that share only the audit ID but have entry-specific reasons, write the per-entry one-liner.

- **Architect's "leave the scheduling sentence alone" rule applies whenever a future-tense pointer describes the current ticket.** The line 70-72 doc-comment in #506 says "Sibling ticket #505 tunes membership per the #503 audit's per-field decisions" — present-tense documentation of the scheduling relationship between two open tickets. The temptation when picking up #505 is to rewrite it past-tense ("Ticket #505 tuned membership per the #503 audit…") to match the new state. Don't: as #505's diff lands, #506's sentence becomes a true historical record of how the work was sequenced, and a future contributor reading just the test file gets the cross-ticket-relationship context without needing to grep `git log`. Rewriting forward-pointing language past-tense is the test-source equivalent of squashing useful history.

## Lessons learned

- **The architect's spec edits an unchanged pre-#506 anchor (lines 73-97, 128-141) into the post-#506 file. Confirm anchors against the current file before applying.** The ticket body's line-number pointers were anchored to "the post-#506 file" per the architect's prose, but the actual line numbers in the merge-base file as of pickup were a few lines off due to interim formatting — the spec hedge ("if the file shifts before pickup, use the block-name landmarks") was the load-bearing instruction. The merged edits ended up at lines 75-84 (streamrunner allowlist), 96-99 (ptyrunner Events), and 133-146 (envelopeShape doc-comment) — close enough to the spec's 73-86 / 94-97 / 128-141 that block-name search resolved the drift in one pass, but the lesson generalises: when a spec carries line-number anchors AND a block-name landmark, the landmark is the source of truth; the line numbers are a hint. This is the third doc'd instance of "AC line numbers shifted between writing the AC and picking up the ticket" — the pattern is well-established at this point and worth a sentence in any future architect spec template.

- **`go test -race -tags e2e_realclaude -run TestAdditiveDriftAssertion_SelfCheck ./internal/e2e/realclaude/...` is the local-runnable verification target for this ticket family.** The full `TestPtyRunnerVsStreamRunner_StructuralEquivalence` is gated on `ANTHROPIC_API_KEY` and only runs in CI / against a real worktree; a contributor making allowlist-comment-only changes can't exercise it locally without credentials. The `_SelfCheck` sub-test from #506 is hermetic (hand-crafted byte fixtures, no `claude` invocation, microseconds) and exercises the allowlist mechanism directly — so adding map entries can't crash the comparison logic without the self-check surfacing it. Generalises whenever a credentialed e2e test ships a hermetic self-check sibling: that's the assertion-mechanism regression target for changes scoped to the assertion plumbing (allowlist tables, helper signatures, etc.).

- **Code review found zero findings. Spec-faithful execution when the AC is fully prescriptive is the correct path.** The architect spec gave verbatim trailing-comment text for every entry; the developer applied the verbatim text. No improvisation, no "while I'm here" cleanup of adjacent code (e.g. the line 70-72 scheduling sentence — see Patterns established above), no extra entries beyond the four the audit catalogued. Recording this as a lesson because the temptation on a "doc-only" ticket is to expand scope ("the comment alignment column is off; let me fix that too" — gofmt handles it, don't touch it manually) and the right behaviour is to ship exactly what was specified. Generalises to any prescriptive doc-edit ticket: the AC IS the design; deviation needs a separate ticket.
