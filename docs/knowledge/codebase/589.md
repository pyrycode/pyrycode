# Ticket #589 — v2 assistant-turn bridge: fan finished assistant turns to open v2 phones

The return half of every conversation on the encrypted (Noise) path. Before this slice a phone's
`send_message` over v2 reached claude and the assistant replied, but the reply never came back —
`send_message` yielded only an `ack`. This wires the v2 leg of the assistant-turn bridge: it taps
the supervised child's PTY output, reads the conversation cursor `send_message` stamped, builds the
**same** `message` application envelope the v1 bridge ([#311](311.md)) builds, enumerates the
currently-open v2 sessions ([#588](588.md) `ActiveConnIDs`), and `Push`es ([#571](571.md)) the
sealed envelope to each. Together with the inbound v2 `send_message` path this closes the v2 mirror
loop: phone → sealed `send_message` → PTY → claude → PTY → sealed `message` → phone.

## Implementation

### Decision: a parallel v2 emitter, v1 untouched

The ticket left open "generalize the existing emitter vs. add a parallel v2 emitter." The architect
chose **a parallel emitter in a new file `cmd/pyry/assistant_turn_v2.go`; zero edits to
`assistant_turn.go` or `assistant_turn_test.go`.** This satisfies AC#5 ("v1 unchanged") *provably*
rather than by argument — no v1 file is touched, so there is no behavioural drift and no churn in the
v1 unit tests (which assert `env.ID == 1` from `c.NextID()`, an assertion that a shared sink
abstraction would have forced to move). The only cross-file reuse is the **`cursorReader` interface**
(defined in `assistant_turn.go`, same package `main`), referenced directly — free, no v1 edit.

Rejected: generalizing behind an `envelopeSink` interface with a `dispatchSink` (v1) and `v2Sink`.
Cleaner on paper but it rewrites the v1 emitter's signature and its three unit tests — friction
against an AC that exists precisely to keep v1 still. Revisit only if a third transport leg appears
(YAGNI). This mirrors how the manager itself was built: `ActiveConnIDs`/`Push` are *parallel* v2
analogs of v1's `ActiveConns`/`Conn.Send`, not a unified surface (see [#571](571.md)/[#588](588.md),
and the PROJECT-MEMORY "resist over-DRY on duplicated registry primitives" note).

### `cmd/pyry/assistant_turn_v2.go` (new, ~206 LOC)

Mirrors the v1 emitter's shape. All unexported (package `main`); **zero new exported types**.

- **`v2Broadcaster` interface** — the minimal `*relay.V2SessionManager` surface the emitter needs:
  `ActiveConnIDs(ctx) []string` + `Push(ctx, connID, env) error`. Declared at the consumer
  (CODING-STYLE) so the unit tests drive it without a real manager. `*relay.V2SessionManager`
  satisfies it.
- **`assistantTurnEmitterV2`** — fields `sup cursorReader`, `bcast v2Broadcaster`,
  `logger *slog.Logger`, `in chan []byte` (buffered `assistantTurnQueueSize`, shared const with v1),
  and a private `nextID uint64` envelope-ID counter. The counter is a **plain field, not atomic** —
  read/written only on the single `Run` goroutine (`broadcast` is serial there).
- **`Enqueue(p []byte)`** — byte-identical to v1: copies `p` (the supervisor reuses its `io.Copy`
  read buffer), non-blocking send onto `in`, drop-on-full with a WARN carrying only `chunk_len`.
- **`Run(ctx)`** — drains `in` until ctx-cancel or channel close; per chunk calls `broadcast`.
- **`broadcast(ctx, chunk)`** — the one divergent method. Steps 1–3 (cursor read → mint MessageID →
  build+marshal `MessagePayload`) are ~25 LOC duplicated deliberately from v1. Step 4 is the
  v2-specific tail: re-snapshot `ActiveConnIDs(ctx)` and, per conn, `e.nextID++`, build an
  `Envelope{ID: e.nextID, Type: TypeMessage, TS: now, Payload: payloadJSON}`, and `Push` it. A
  per-conn `Push` error is **DEBUG-logged and the loop `continue`s** (a dropped conn must not abort
  the turn for the others — AC#2); `ctx.Err() != nil` returns early (teardown).
- **`startAssistantTurnBridgeV2(ctx, sup, bridge, bcast, logger) func()`** — constructs the emitter,
  registers `emitter.Enqueue` via `bridge.SetOutputObserver`, spawns `Run`, and returns an
  idempotent cleanup that clears the observer (`SetOutputObserver(nil)`) and `<-done`. It **does NOT
  close `in`** — same panic-window reasoning as #311 (a `Write` racing cleanup could `Enqueue` after
  the observer is cleared; sending to a closed channel would panic). ctx-cancel drains `Run`.

### Wiring into `startRelayV2` (`cmd/pyry/relay.go`)

- `startRelayV2`'s signature gained `sup *supervisor.Supervisor, bridge *supervisor.Bridge`
  (mirroring v1 `startRelay`'s tail order). Its **single call site** in `startRelay` (the
  `if v2Enabled` branch) forwards them — `startRelay` already holds both.
- After `mgr` and its `Run` goroutine are up, the bridge is started under a **foreground gate**:
  `if bridge != nil { bridgeCleanup = startAssistantTurnBridgeV2(ctx, sup, bridge, mgr, logger) }`.
  `mgr` (`*relay.V2SessionManager`) is the `v2Broadcaster`. In foreground mode `bridge == nil`
  (`main.go` only builds a `Bridge` when stdin is not a terminal), so the v2 bridge — like v1 — never
  starts there; there is no PTY-output observer surface, and inbound `send_message` still works
  (AC#3).
- The returned `drain` calls `bridgeCleanup()` **before** `<-mgrDone`: stop the observer first so no
  new PTY chunks queue while the manager winds down, then wait for the manager's `Run` to exit.

### Envelope-ID policy

v2 has no per-session outbound envelope counter (manager-internal frames use fixed/`Reply`-allocated
IDs the bridge does not share). Per the [#571](571.md) recommendation, the bridge uses a
**caller-side monotonic counter** (`e.nextID`, ++ once per envelope emitted). It is **not**
coordinated with the dispatch reply path, so a bridge `env.ID` may collide with a solicited reply's
`env.ID` on the same session. **This is acceptable today**: `MessagePayload.MessageID` (a fresh
UUIDv4 per chunk) is the phone's dedup/ordering key; `env.ID` is an envelope sequence hint, not
load-bearing on v2. Escalate to a manager-owned per-session counter only if pyrycode-mobile later
mandates globally-monotonic per-session `env.ID`s (open question, not built).

## Patterns established

- **Parallel-emitter over forced-DRY, again.** The v2 leg duplicates the ~25 LOC application-minting
  core rather than abstracting a shared sink, for one caller per side. The deliberate, bounded
  duplication keeps v1 provably frozen and is consistent with how `Push`/`ActiveConnIDs` were
  authored as parallel v2 analogs. The `cursorReader` interface is the one thing genuinely shared,
  and it costs nothing.
- **Inherited #311 lessons, carried verbatim.** (1) PTY chunk bytes are **never** logged at any
  level — every `broadcast` branch logs only `chunk_len`, `conversation_id`, `message_id`, `conn_id`,
  `event`, and (where present) a `Push` `err` (a transport/sentinel error, never chunk-derived); the
  marshal-error branch omits `err.Error()` because `encoding/json` echoes invalid input bytes into
  its error string. The chunk reaches the phone only via `MessagePayload.Text`, sealed under Noise.
  (2) Cleanup does **not** close `in` (panic window). (3) "Defensive log fields, not defensive code"
  — the marshal branch can't fail in practice but is logged with a safe field list.
- **Authorization is inherited, not re-implemented (`security-sensitive`).** Server output reaches
  only authenticated peers via **two independent, deterministic code-level gates**: `ActiveConnIDs`
  returns only `V2StateOpen` conn IDs (token validated in `handleNoiseInit`'s accept branch), and
  `Push` independently re-gates on `V2StateOpen` (`ErrSessionNotOpen` otherwise). Belt-and-suspenders,
  different fabric — the enumeration filter is primary, the `Push` gate is the net. The bridge adds
  **no third gate of its own**; doing so would duplicate the primitive's contract.
- **Fresh snapshot per chunk is the AC#2 mechanism.** `ActiveConnIDs` is re-called on every
  `broadcast`, so a phone that opens its session between chunk N and N+1 is included in N+1's fan-out;
  a phone whose session was torn down is absent from the snapshot, or — if it drops between snapshot
  and `Push` — surfaces as `ErrConnNotFound` and the loop continues. No subscription map: like v1, the
  v2 bridge broadcasts to *every* open session (pyrycode is single-paired-phone in practice).

## Concurrency model

Identical topology to v1 ([#311](311.md)), with the manager funnel replacing the dispatcher:

- **Producer** — the supervisor's PTY-drain goroutine: `Bridge.Write` → `observer(p)` →
  `Enqueue(p)`. Copy + non-blocking send + drop-on-full; the PTY drain never wedges.
- **Consumer** — one long-lived `Run` goroutine. `broadcast` is serial, so `e.nextID` needs no
  atomic. `ActiveConnIDs` then `Push` both **funnel through the manager's single `Run` goroutine**
  (the package's single-writer idiom; no mutex), which serializes every `s.send.Encrypt` so the noise
  nonce counter never races.
- **Cursor read** — `sup.CurrentConversation()` is `convMu`-guarded (leaf-only, #312); taken once per
  chunk.
- **Teardown** — the same ctx passed to `startRelayV2` unblocks `Run`; cleanup runs post-cancel
  (`main.go` defers it after `pool.Run`). A `broadcast` in flight during teardown passes the cancelled
  ctx to `ActiveConnIDs`/`Push`, both of which have a `ctx.Done` escape arm, so the emitter never
  blocks on a winding-down manager.

No new locks. The bridge's `mu` guards the observer slot; the supervisor's `convMu` guards the
cursor; the manager's `Run` owns `m.sessions`. None nest.

## Lessons learned

- **The v2 receive CipherState nonce is sequential — the e2e phone cannot skip frames.** Unlike v1,
  where the phone could read frames opportunistically, a v2 phone must `Decrypt` **every** binary→phone
  `noise_msg` frame in capture (= seal) order: the receive CipherState's nonce advances per frame, so
  skipping ahead to the marker `message` desynchronises the nonce and every subsequent decrypt fails.
  The e2e therefore decrypts the solicited `ack` and then loops-until-marker through the prelude
  `message`s on the **same in-order cursor**. A shared `decryptInnerEnvelope(t, inner, cs)` helper
  (base64-decode `inner.Data` → `cs.Decrypt` → `json.Unmarshal`) keeps the ack read and the
  loop-until-marker read on that single cursor. (Lifted from the PR's Lessons-learned section.)

## Testing

- **Unit — `cmd/pyry/assistant_turn_v2_test.go` (new, stdlib only, `-race`, package `main`).** Drives
  the emitter against the `cursorReader` seam and a `stubV2Broadcaster` recording pushed
  `(connID, env)` pairs. Scenarios: cursor-empty → no push; fan-out to every open conn (two conns →
  two `Push`es, each `TypeMessage`, `InReplyTo == nil`, decoded `MessagePayload` with the right
  `ConversationID`/`Role: "assistant"`/`Text`, valid UUID `MessageID`); per-conn `Push` failure
  (`relay.ErrConnNotFound` on the middle conn) does not abort the turn (3 attempts, 2 deliveries);
  envelope-ID counter monotonic across two chunks × two conns with distinct UUID `MessageID`s;
  Enqueue copies the chunk (mutate-after-enqueue, pushed `Text` reflects the original); ctx-cancel
  stops `Run` (gated on a `done` channel, no sleeps); mid-turn connect (`{a}` then `{a, b}` → `b`
  receives only the second chunk). All in package `main`, so QA's `go test -race ./...` covers them.
- **E2E — `internal/e2e/relay_v2_daemon_test.go` (new `TestRelayV2_AssistantTurn_BroadcastsMessageEnvelope`,
  build tag `e2e`, AC#4).** Combines the v2 handshake/sealed-frame path with the v1 assistant-trigger
  flow: `pair` → seed `conversations.json` → `StartRotationWithRelay(..., fr.URL()+"/v2/server",
  "PYRY_MOBILE_V2=1", "PYRY_FAKE_CLAUDE_ASSISTANT_TRIGGER=...")` (no harness change) →
  `waitBinaryHello` → `fakephone.Dial` → `driveHandshakeToOpenDaemon` → sealed `send_message` →
  decrypted `ack` (stamps the cursor) → write the assistant trigger → loop-until-marker decrypting
  every inner frame in order; assert the marker `message` has `InReplyTo == nil`, matching

## Out of scope (deferred)

- **Live token streaming.** Finished-message delivery only; per-token streaming stays out of scope
  (pyrycode-mobile#337). Unblocks pyrycode-mobile#336 (session boundaries) + #337 on the server side.
- **Globally-monotonic per-session envelope IDs.** The caller-side `nextID` can collide with the
  dispatch reply path's `env.ID` on the same session; acceptable while `MessageID` is the dedup key.
  Promote into the manager (one counter per `V2Session`) only if the phone protocol later requires it.
- **Real-claude assistant-turn parsing.** The bridge tees raw PTY bytes (ANSI escapes and all) into
  `Text`, exactly as v1; parsing "what counts as a turn" against the real claude TUI is a shared,
  future slice (inherited #311 deferral).
- **Per-conversation routing.** Broadcast to every open session; a `conversation_id → conn_id`
  subscription map (pyrycode-mobile#336) replaces the broadcast call later without touching the
  supervisor or bridge sides.
