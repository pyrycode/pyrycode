# Ticket #439 — `internal/keys` filesystem hardening (0700 dir + 0600 file + `O_NOFOLLOW`)

Locks down the on-disk layer the [#438](438.md) primitive landed: `LoadOrCreate` now refuses to start the daemon when the keystore parent directory has any group/other permission bit set, when `static_key.json`'s mode is anything but exactly `0600`, or when a between-stat-and-open symlink swap would otherwise redirect the read. No auto-`chmod`, no silent fallback — the daemon refuses to start so the operator can fix the mode by hand. This is the hard prerequisite #438 named as the blocker for any production consumer (#432 QR payload, #433 Noise wrapper).

## Implementation

- **Two new `errors.Is`-matchable sentinels** in `internal/keys/static_key.go`: `ErrInsecureKeyDirMode` (parent directory has group/other bits OR path is not a directory) and `ErrInsecureKeyFileMode` (file mode is not exactly `0600`). Kept distinct because the operator's fix differs — chmod the directory vs. chmod the file. Lumping them under a single `ErrInsecurePermissions` would force the operator to read the wrapped message to know which path to chmod, defeating the point of an `errors.Is`-matchable sentinel.
- **`ensureSecureKeyDir(dir string) error`** (unexported, `store.go`). Single responsibility: guarantee `dir` exists, is a directory, and has no group/other bits set. Flow: `os.Stat`; on `fs.ErrNotExist` → `os.MkdirAll(dir, 0o700)` + re-stat; reject `!IsDir()` with `ErrInsecureKeyDirMode`; reject `mode & 0o077 != 0` with `ErrInsecureKeyDirMode`. The re-stat after `MkdirAll` is defensive — under a normal umask `mkdir(2)` can only narrow the requested mode, but the re-stat catches default ACLs, exotic filesystems, and future stdlib changes.
- **`openSecureKeyFile(path string) ([]byte, error)`** (unexported, `store.go`). Opens via `os.OpenFile(path, os.O_RDONLY|syscall.O_NOFOLLOW, 0)`, `io.ReadAll`, wraps any error as `keys: open %s: %w` / `keys: read %s: %w`. The wrapped error names only the caller-supplied path — the resolved link target is deliberately not included so a hostile symlink cannot use the error message as an exfiltration channel.
- **`LoadOrCreate` rewired** from `os.ReadFile` to `Lstat` + strict `0o600` check + `openSecureKeyFile`. Signature unchanged from #438. The order is: `validDaemonName` → `ensureSecureKeyDir` → `Lstat`. On `Lstat` success, reject `fi.Mode().Perm() != 0o600` with `ErrInsecureKeyFileMode` before opening. On `fs.ErrNotExist`, fall through to `mintAndPersist` (its existing `MkdirAll(0o700)` is now a redundant no-op after `ensureSecureKeyDir` already ran; left in place as defense-in-depth survives any future refactor that calls `writeStaticKey` from a different entry).
- **Belt-and-suspenders symlink defence** uses two different fabrics. `Lstat` catches the *static* case — a symlink's `Mode().Perm()` is the symlink's own mode (typically `0755` on macOS / `0777` on Linux), which fails the `== 0600` check before the open ever runs. `O_NOFOLLOW` catches the *dynamic* case — between the Lstat and the open, a hostile process under the same UID replaces the regular file with a symlink; the open returns `ELOOP` instead of redirecting to attacker-controlled bytes. The two layers catch distinct failure modes the other cannot.
- **Mode check uses `& 0o077 != 0`** for the directory, not `!= 0o700`. The security contract is "no group/other access", not "exactly 0700"; a narrower mode (`0600`, `0500`) is permitted by this check — it isn't *insecure*, just non-functional, and the daemon's subsequent operation will fail loudly with `EACCES` if so. We don't gold-plate against narrower modes. The file check, by contrast, is strict `== 0o600` because anything else (including narrower) is a signal something tampered with the file.
- **Wrapping pattern** `fmt.Errorf("keys: %s: mode %#o: %w", path, observedMode, ErrInsecure...)` mirrors `parsePersisted`'s style. The path is operator-facing (tells them which file to chmod); the mode is operator-facing (tells them what the file *currently* is so `0644 → 0600` is obvious); both are non-secret. Private key bytes are secret and must not appear in error context for any reason.
- **`syscall.O_NOFOLLOW` is canonical on both Linux and macOS** — no `golang.org/x/sys/unix` import added. Go's `syscall` package exports the platform-correct value on both targets.
- **No auto-chmod ever.** Spec called this out explicitly as an item the developer must not relax. Auto-`chmod` would remove the loud-failure signal the operator needs: if the package manager unpacked the key as `0644`, silently fixing it would hide a packaging bug; if a hostile sibling process under the same UID weakened the mode, auto-`chmod` hands the attacker exactly the moment of vulnerability they need.

## Patterns established

- **Stat-then-open with loud-failure semantics for security-sensitive on-disk state.** The pattern: `Lstat` (NOT `Stat` — symlinks must not be transparently followed during the check), reject on mode mismatch with a wrapped sentinel naming the path and the observed mode in octal, then open with `O_NOFOLLOW`. No auto-repair, no silent fallback. Applies to any future on-disk credential or capability material; `~/.cache/pyry-relay/`'s TLS cert directory in `pyrycode-relay` is the same shape. The recipe diverges from the registry-style `os.ReadFile` shorthand used elsewhere in `internal/*` because registries trust their own write contract on the create path; this hardening defends the *read* path against drift introduced outside the process (operator chmod, package-manager unpack, restore-from-backup, hostile process under the same UID).
- **Two-fabric belt-and-suspenders.** When pairing security checks, the two layers must catch genuinely distinct failure modes — not the same failure mode twice. `Lstat`-mode-check + `O_NOFOLLOW`-on-open is canonical: the first catches symlink-at-rest, the second catches symlink-swap-during-the-window. A second `Stat` between would be redundant; a checksum on the bytes would be a different fabric again (file-mode vs. content). The TOCTOU window between `Lstat` and `OpenFile` is inherent in any check-then-act filesystem pattern; `O_NOFOLLOW` closes the symlink-swap subspace, and a regular-file-to-regular-file swap during the window is out of scope by design (the attacker would have to write a fully-valid `static_key.json` envelope — a totally different threat model).
- **Distinct sentinels per distinct operator fix.** `ErrInsecureKeyDirMode` and `ErrInsecureKeyFileMode` are two sentinels not one — they exist because the chmod target differs. The `errors.Is`-matchability is load-bearing for callers (and for tests); a single `ErrInsecurePermissions` would force the consumer to parse the wrapped message to know which path to chmod. The pattern generalises: sentinel granularity should match operator-action granularity, not implementation-detail granularity.
- **Permit-narrower / reject-broader for the directory; strict-equality for the file.** Asymmetry is deliberate. The directory's security contract is "no group/other access" (`& 0o077 != 0`), so `0600` or `0500` are permitted; the file's security contract is "exactly `0600`" because anything else — including narrower modes — indicates tampering on a file the daemon itself writes at exactly `0600`. Worth pinning because a future contributor reading just one of the two checks may want to "harmonise" them and silently regress the contract.

## Lessons learned

- **The pre-existing `TestLoadOrCreate_NonENOENTReadErrorIsNotCorruption` keeps passing — but for a different reason after this ticket.** Before: making `static_key.json` itself a directory caused `os.ReadFile` to surface `EISDIR`, and the three-way switch wrapped it as a non-corruption error. After: `Lstat` succeeds on the directory and reports its mode as `0o700` (or whatever the directory was created with); the strict `mode != 0o600` check then rejects it as `ErrInsecureKeyFileMode` *before the read happens*. The test's assertions (`!errors.Is(ErrCorruptKeyFile)`, `!errors.Is(ErrInvalidDaemonName)`) still hold because `ErrInsecureKeyFileMode` is neither; the sentinel-distinction guarantee survives even though the code path changed underneath the test. The lesson generalises: tests that assert "is NOT sentinel X" remain meaningful across refactors that introduce sentinel Y, provided the test was written that way originally. Worth pinning the convention for future security-sensitive checks.
- **`io.ReadAll` on the opened key file is uncapped.** Spec § Security threat #7 (in `docs/specs/architecture/439-keys-filesystem-hardening.md`) explicitly deferred `io.LimitReader` to a future ticket: a hostile process under the daemon's UID could place a multi-MB file at `static_key.json` to OOM the daemon, but the threat model already excludes "hostile process under same UID" as out-of-scope. The CR called this out as an `[INFO]` finding for the same reason — consistent with the spec, not a defect — but the next person grepping for `io.ReadAll` in security-sensitive code paths should know the discussion happened.
- **PR-body filename typo vs. committed filename.** PR #442's body referenced `docs/specs/architecture/439-keys-fs-hardening.md` but the committed file is `docs/specs/architecture/439-keys-filesystem-hardening.md`. CR flagged as cosmetic. Worth a note here because `qmd update` indexes the committed name; future grep for "fs-hardening" via QMD returns no hits while "filesystem-hardening" does.

## Out of scope (filed elsewhere or deferred)

- **`io.LimitReader` (e.g. 64KB) on the read** — spec § Security threat #7. Revisit if the threat model expands to "hostile process under same UID". The on-disk envelope is ~200 bytes; a multi-MB file is the only signal we'd have that something is wrong, and we don't have to defend against it at this layer.
- **Path traversal beyond `daemonName`** — already closed by the #438 allowlist; this ticket doesn't widen the surface.
- **Hardware-backed key storage** — v3 concern, per `docs/protocol-mobile.md`.
- **Auto-`chmod` repair flow** — explicitly forbidden by the spec; would erase the loud-failure signal that is the entire security feature.
