# Ticket #510 — `ptyrunner/watchdog` delegates spinner regex + tick loop to tuidriver

Sibling of [#509](509.md) on the ptyrunner side of the migration: `internal/agentrun/ptyrunner/watchdog.go` shrinks from 89 LOC to a single ~15 LOC glue function that calls `tuidriver.RunWatchdog` and maps a non-nil return into the three pyrycode-specific side effects (`slog.Warn` → `emitter.SetExitReason(streamjson.ExitReasonError)` → `cancel()`). The class-A spinner-seconds regex (`spinnerSecondsRe`), its `parseSpinnerSeconds` helper, the `defaultWatchdogTick = 1 * time.Second` constant, and the local `time.Ticker` loop are all deleted — every one of them is now owned upstream by tui-driver #89 (`tuidriver.ParseSpinner`, `tuidriver.RunWatchdog`, `tuidriver.DefaultWatchdogTick`, `tuidriver.WatchdogOpts`). Net diff: production file -62 LOC (89 → 27); `runner_test.go` -29 LOC (`TestParseSpinnerSeconds` deleted, equivalent coverage upstream); no other file touched. `runWatchdog`'s function signature is preserved byte-for-byte so `runner.go:384`'s call site is untouched — the smallest blast radius for the refactor and the deliberate setup for capstone ticket [#513](https://github.com/pyrycode/pyrycode/issues/513), which collapses the spawn into `Session.Events()`.

## Implementation

- **`watchdog.go` shrinks to one delegation + three side effects.** Final body is `if err := tuidriver.RunWatchdog(ctx, buf, tr, tuidriver.WatchdogOpts{Tick: tick}); err != nil { logger.Warn("ptyrunner: watchdog fired", "err", err); emitter.SetExitReason(streamjson.ExitReasonError); cancel() }`. Three imports drop (`regexp`, `strconv`, and the now-unused fragment of the comment block); five remain (`context`, `log/slog`, `time`, `github.com/pyrycode/tui-driver/pkg/tuidriver`, `github.com/pyrycode/pyrycode/internal/agentrun/streamjson`). The "Discipline: the only thing logged is the tuidriver-generated watchdog error string …" sentence is preserved verbatim in the doc-comment because it remains the load-bearing log-content invariant for the package (also reasserted in `runner.go:22-27`).

- **Function signature kept byte-for-byte.** `runWatchdog(ctx, buf *tuidriver.Buffer, tr *tuidriver.Tracker, emitter *streamjson.Emitter, cancel context.CancelFunc, tick time.Duration, logger *slog.Logger)` — same as pre-#510. `runner.go:384`'s call `runWatchdog(runCtx, sess.Buffer, tracker, emitter, cancel, cfg.WatchdogTick, logger)` is untouched. The spec explicitly forbade "simplifying" the signature (dropping `tick` would lose `cfg.WatchdogTick` plumbing; dropping `logger` would lose the `slog.Default()` fallback at runner.go:254-257). Preserving the signature also keeps capstone [#513](https://github.com/pyrycode/pyrycode/issues/513)'s eventual `Session.Events()` collapse minimal — it gets to delete the wrapper without re-shaping its inputs first.

- **Zero-tick fallback moved from consumer to library.** Pre-#510 the function opened with `if tick <= 0 { tick = defaultWatchdogTick }`. Post-#510 that branch is gone — `tuidriver.RunWatchdog` performs the same `tick <= 0 → DefaultWatchdogTick` fallback internally against the same 1 Hz constant. Behavioural envelope is unchanged; the bookkeeping just lives one delegation deeper. `Config.WatchdogTick`'s doc-comment ("zero defaults to 1 second") stays accurate without any edit — the spec explicitly cautioned against "tightening" it to name `tuidriver.DefaultWatchdogTick` because that would expand the refactor's blast radius for no behavioural gain.

- **Class B/C/D freeze-arm invariant preserved by upstream contract, not by an explicit AND.** Pre-#510 the local loop passed `tr.ObserveSpinner(ok && tuidriver.IsThinking(stripped), total)` — the explicit `IsThinking` AND was the guard against class B (`✻ Channeling…`) / class C (`✻ Actualizing… (2s · ↓N tokens)`) renderings engaging the spinner-freeze arm (they have no `for Ns` counter, so there's no progress to track and a stuck render would false-positive). Post-#510 `tuidriver.RunWatchdog` calls `tr.ObserveSpinner(ok, total)` where `ok` is `tuidriver.ParseSpinner`'s second return. The invariant is now structural: `ParseSpinner`'s class-A regex (`✻\s+\S+(?:\s+\S+)?\s+for\s+(?:(\d+)m\s+)?(\d+)s`) requires the literal `✻` glyph AND the literal `for` keyword, so `ok == true` already implies `IsThinking == true`, and class B/C/D return `ok = false` (no `for` keyword), which `ObserveSpinner` treats as "spinner not visible" — same outcome the explicit AND produced, achieved by a tighter library contract. AC #3 satisfied without any pyrycode-side test.

- **Side-effect ordering is load-bearing for e2e-realclaude byte-equivalence.** `SetExitReason` MUST fire **before** `cancel()` — matches the pre-#510 order at the old watchdog.go:82-83 and the order the byte-equivalence assertion (#506) pins on the trailer shape (`Subtype = "error_during_execution"`, `IsError = true`, empty `TerminalReason`). The code-review pass explicitly spot-checked this — if a future contributor reorders to `cancel(); SetExitReason(…)`, the emitter may close on `cancel()`'s effect before `SetExitReason` runs, and the trailer composition flips to `ExitReasonCompletion` based on EOT-not-observed instead of `ExitReasonError`. The spec calls this out under § Testing strategy as the most likely regression vector for `make e2e-realclaude` drift.

- **`TestParseSpinnerSeconds` deleted; no shim test added on the pyrycode side.** The local table test covered class-A single/two-word verb, minutes+seconds, class B ellipsis, class C parenthesised, no-glyph, glyph-only — same fixture set covered upstream by `tui-driver` `TestParseSpinnerClassAMatches` + `TestParseSpinnerNonClassA`. The spec explicitly forbade a shim test calling `tuidriver.ParseSpinner` from pyrycode — that would be re-testing library code from the consumer side, which the upstream tests exist to prevent. `TestRun_WatchdogFires` (runner_test.go:464-497) stays verbatim and is the single integration test that asserts the glue's three side effects end-to-end (`cfg.WatchdogTick = 50ms` + `WatchdogTrackerOpts.{PTYQuietLimit,SpinnerFreezeLimit} = 200ms` → trailer `Subtype = "error_during_execution"` + `IsError = true` + wall-clock < 5s). The tick-loop mechanics (cancellation, default-tick application, PTY-quiet arm, spinner-freeze arm) are owned upstream by `tui-driver` `TestRunWatchdog*` (four tests, named in the spec's "Files to read first" section).

- **No new tests added.** Coverage migrates upstream. The pyrycode-side test surface contracts by one function (`TestParseSpinnerSeconds`); the contracted-down ratio (one integration test on the glue, four library tests on the loop, two library tests on the parse) replaces a 1:1 mirror of library mechanics with a one-way "glue asserts its own side effects, library asserts its own internals" split.

- **No `go.mod` bump.** `go.mod` already pinned `github.com/pyrycode/tui-driver v0.0.0-20260523181457-c2dcd1e49992` from [#509](509.md)'s `go get` (2026-05-23) — same version that exports `RunWatchdog` / `ParseSpinner` / `WatchdogOpts` / `DefaultWatchdogTick` from tui-driver #89. Pre-pickup verification was `go doc github.com/pyrycode/tui-driver/pkg/tuidriver RunWatchdog`; non-empty output confirmed the spec's premise. #510 is the second consumer of the 2026-05-23 pin; #511 / #512 (the remaining JSONL-side migrations) will be the next.

- **No edits to `runner.go`.** `runner.go:380-387` (the `wg.Add(1)` / `defer wg.Done()` / `defer wg.Wait()` / `defer cancel()` spawn shape) is unchanged. The defer-LIFO cleanup chain documented at runner.go:206-216 (`cancel() → wg.Wait() → counter.Stop() → emitter.Close() → sess.Close()`) is load-bearing and untouched. `Config.WatchdogTick` and `Config.WatchdogTrackerOpts` doc-comments (runner.go:155-165) stay accurate without edits — the zero-default behaviour is preserved end-to-end.

## Patterns established

- **Library-helper delegation with a preserved function signature shrinks the consumer to a glue layer without rippling into the caller.** Pre-#510 `watchdog.go` defined four package symbols (`defaultWatchdogTick`, `spinnerSecondsRe`, `parseSpinnerSeconds`, `runWatchdog`); post-#510 only `runWatchdog` remains, and its signature is byte-identical to before. The caller (`runner.go:384`) is structurally unaware of the migration. Generalises whenever you're delegating an internal mechanism to a library that ships a strict superset of the local behaviour: **keep the consumer's function as a typed wrapper around the library call**, even when "the wrapper is now one line." The wrapper is still load-bearing for two reasons: (1) it carries the consumer-specific side effects the library doesn't and can't know about (here: `streamjson.ExitReasonError` + the run-ctx `cancel()`); (2) it isolates the call site from future library-signature changes (`WatchdogOpts` may grow fields; the wrapper absorbs that without touching `runner.go`). Trading the library-side win (deleted regex + deleted loop) for a consumer-side cost (preserved wrapper signature) is the right balance whenever a downstream capstone (#513) is going to collapse the wrapper anyway — landing the cleanup in two strokes is safer than coupling the delegation to the spawn-shape rewrite.

- **An explicit AND in a consumer can become structurally redundant when the library's contract tightens.** Pre-#510 the consumer wrote `tr.ObserveSpinner(ok && tuidriver.IsThinking(stripped), total)` because the library's old `ParseSpinner` (or its pre-promotion equivalent in the spike binary) didn't make the `IsThinking` guarantee — the consumer had to bolt it on. Post-#510 `tuidriver.ParseSpinner` requires the `✻` glyph by literal regex match, so `ok == true → IsThinking == true` is a contract, not a hope; the explicit AND becomes redundant. Generalises whenever a library promotes a previously-stateless helper into one with a stricter input contract: audit consumer-side AND-guards against the new contract — if the guard becomes redundant, **delete it** rather than keeping it "for safety". The redundant guard imposes a documentation burden (why is this AND here?) and a maintenance hazard (a future contributor strengthening the guard to "fix" something might inadvertently re-introduce a class B/C false-negative). The spec's Technical Notes section explicitly mapped the equivalence — the right place to record "this is intentional, the AND is gone because the library now guarantees it" is in the codebase doc, not in a leftover comment.

- **A stale rationale comment is its own deletion target during a refactor.** The deleted `spinnerSecondsRe` var carried a multi-line comment ("Kept here rather than in tuidriver because the regex is consumer-policy (which spinner classes to parse) not driver-policy") that was correct at the time of #479 but became false the moment tui-driver #89 promoted the regex into `ParseSpinner`. The right move was to delete the comment along with the symbol it justified — not to migrate the rationale into the new doc-comment or leave a "deprecated" trail behind. Generalises whenever a delegation makes a "we did it this way because of X" comment obsolete: **delete the comment**, don't preserve it as historical context. The historical context lives in git blame and in the per-ticket codebase doc (this file); the live doc-comment should only describe the present-day contract. Pre-existing comments that document why a choice was made over an alternative are valuable; comments that document why the now-deleted code existed are noise.

## Out of scope (follow-ups)

- **Capstone [#513](https://github.com/pyrycode/pyrycode/issues/513) — collapse the watchdog goroutine into `Session.Events()`.** The `wg.Add(1)` / `defer wg.Wait()` spawn shape at `runner.go:380-387` is deliberately preserved by #510 so #513's diff stays minimal. After #513, the entire `runWatchdog` wrapper function may be deletable (or shrink further), depending on what shape `Session.Events()` exposes upstream. #510 makes that future change a single-file edit by keeping the boundary unchanged.

- **Class B/C "does NOT engage the freeze arm" integration test on the pyrycode side.** The spec explicitly forbade this — that's library-contract testing, not pyrycode-glue testing, and `tui-driver` `TestRunWatchdogSpinnerFreezeWedge` (fires the freeze arm on a class-A sequence with a stuck counter) + `TestParseSpinnerNonClassA` (confirms class B/C return `ok=false`) already cover it upstream. Recording the explicit prohibition here so a future contributor tempted to "tighten the integration coverage" sees that the answer is no.

- **JSONL-side migrations [#511](https://github.com/pyrycode/pyrycode/issues/511) / [#512](https://github.com/pyrycode/pyrycode/issues/512).** #511 / #512 (whichever lands next) finish the migration of `internal/agentrun/jsonl/tail` onto `tuidriver.TailJSONL` and delete the package; that's independent of #510 (different file). The shared dependency is the 2026-05-23 pin already in `go.mod`.
