# Ticket #700 — Permission request/response seam (`internal/turnevent`)

Extends the neutral internal turn-event model ([#606]) with the permission
**request/response** pair — the one Phase 3 ([EPIC #597]) internal type that is
genuinely cross-cutting: both the mobile wire adapter (#597) and the future
`pyry acp` adapter ([#600]) map onto the same daemon-owned, **ACP-neutral**
shape, so ACP's `session/request_permission` lands on this exact
`PermissionRequest` later instead of each adapter inventing its own. Pure value
types + one constructor — no transport, no wire, no consumers wired; mirrors
#606's footprint and conventions exactly. Full package reference:
[features/turnevent-package.md](../features/turnevent-package.md).

This slice also **establishes the inbound seam**: `turnevent` previously sealed
only the *outbound* `Event` family, and `PermissionResponse` is the package's
first *inbound* type. The architect's one design call (AC bullet 3) was where
the inbound home lives — resolved in-spec as **(B) establish the `Inbound`
sealed sum type now**, rather than a standalone struct.

## Implementation

One production file created, one modified; pure additive value types (`permission.go`
needs **zero imports** — the stdlib-only `boundary_test.go` covers it for free):

- **`permission.go`** (new) — three things:
  - **`PermissionRequest`** joins the existing sealed `Event` sum via the same
    value-receiver `isTurnEvent()` marker + a `var _ Event = PermissionRequest{}`
    assertion, identical to how `Stall` graduated in [#638]. Fields: `RequestID`
    (correlates request ↔ response), `ToolCallID` (the gating tool call by id —
    same by-id reference style as `ToolUpdate`, empty when prompt-only),
    `Title` (always-present human prompt text), `Options []PermissionOption`
    (ordered; the consumer renders + selects from these).
  - **`PermissionOption`** — one selectable answer: `ID` (referenced by the
    response), `Label`, `Kind PermissionOptionKind`.
  - **`NewPermissionRequest(requestID, toolCallID, title, options)`** — the
    AC-mandated positional, **non-validating** assembler (order pinned by the
    test). Constructing an invalid one is allowed and caught downstream via
    `Kind.Valid()` — the package's construct-then-validate-downstream stance.
  - **`Inbound interface{ isInbound() }`** — the new sealed inbound sum type, the
    same 4-line marker idiom as `Event` / `ToolContent`.
  - **`PermissionResponse`** — first `Inbound` member (value-receiver
    `isInbound()` + `var _ Inbound = PermissionResponse{}`). Fields: `RequestID`
    (the request it answers), `OptionID` (the selected `PermissionOption.ID` — the
    **option id, not the kind enum**; empty when cancelled), `Cancelled` (true ⇒
    dismissed without selecting). Exactly one of {`OptionID` set, `Cancelled`} is
    meaningful; the package does **not** enforce the exclusivity — validation is
    the inbound parser's job.
- **`taxonomy.go`** (modified) — `PermissionOptionKind` added as the **fourth**
  ACP enum, shape-identical to `ToolKind` / `ToolStatus` / `TurnEndReason`:
  string-backed type, `<Type><Value>` const block, an unexported canonical
  `permissionOptionKinds` SSOT slice, and a `Valid()` that linear-scans it. The
  four values are **exactly** the four ACP `session/request_permission` kinds —
  `allow_once`, `allow_always`, `reject_once`, `reject_always` — no more, no
  fewer. Co-located with its three siblings so all four ACP enums + their SSOT
  slices stay visible together (and the exactness test sits in `taxonomy_test.go`
  beside its siblings, as the AC asks).

Tests (`permission_test.go` new, `taxonomy_test.go` extended): construct/read-back
through the constructor (option order + each `ID`/`Label`/`Kind`), `Event`
membership of `PermissionRequest` via `[]Event` type switch, `Inbound` membership
of `PermissionResponse` via `[]Inbound` type switch, the selected vs. cancelled
response cases, an option carrying a fabricated kind reporting `Kind.Valid() ==
false`, and the `PermissionOptionKind` exactness test (independent raw-string
`want`, `len == 4` guard, `reflect.DeepEqual`, every-canonical-`Valid()`, +
in-taxonomy/fabricated/empty validity table). `make check` green;
code review **PASS**, no MUST/SHOULD/NIT findings.

## Patterns established

- **The inbound seam mirrors the outbound seam — establish the home with the
  first member.** `Inbound interface{ isInbound() }` is the same sealed-marker
  idiom the package already applies twice (`Event`, `ToolContent`), not
  speculative abstraction: this package is a forward-looking neutral contract
  with no wired consumers (#606 shipped it that way deliberately), so defining
  the inbound contract ahead of its consumer is the package's job. The payoff is
  that the deferred inbound-commands ticket (`Prompt` / `Cancel` / `DropQueued`)
  becomes a **purely additive** "add variants to an existing home" — no retrofit,
  no one-type-outside-the-sum inconsistency. (Option A — a standalone struct —
  was the loser precisely because it would force that retrofit.)
- **A cross-cutting type is shaped adapter-neutral now, not mobile-specific.**
  `PermissionRequest` references the gating tool by `ToolCallID` rather than
  re-embedding tool details (the consumer already saw the matching `ToolStart`),
  and carries `Title` as the always-present context spanning both the
  tool-triggered and prompt-only cases. Shaping it ACP-neutral keeps each adapter
  a thin pass-through and keeps external ACP-spec churn out of the daemon core —
  the same containment logic the rest of `turnevent` provides.
- **The "no constructors" stance gains exactly one AC-mandated, non-validating
  exception.** `NewPermissionRequest` is a 4-field value assembler, not a
  long-lived component (no `Config` struct — that pattern is for `Supervisor`-style
  wiring). It does **not** validate; the package's no-errors / validate-downstream
  contract is otherwise unchanged.
- **The option id, not the kind enum, correlates a response to its option.**
  `PermissionResponse.OptionID` references a `PermissionOption.ID` — the kind is
  request-side metadata. An easy subtlety to get wrong in the downstream wire
  mapping.

## Size note (S kept — at the exported-type cap)

Two production files (`taxonomy.go` modified, `permission.go` created), five new
exported types (`PermissionRequest`, `PermissionResponse`, `PermissionOption`,
`PermissionOptionKind`, `Inbound`) — at the architect's `>5 exported types` red
line, not over it. Every type is a ~4-line declaration with no logic, no
untrusted input, no concurrency, no consumer in scope — the same `S` shape #606
holds. The body's S→XS hedge ("if the inbound home turns out trivial") was
declined: establishing the `Inbound` sum + its member + the fourth enum + two
new test scenarios spans more than an XS.

### Why not `security-sensitive`

Pure value types + one assembler: no I/O, no wire parse, no dispatch / gate /
route policy — the identical shape to #606 (unlabelled). The security review
belongs on the **downstream** slices that (a) parse the inbound
`PermissionResponse` frame from the phone (untrusted input) and (b) gate
answering behind `--allow-remote-permissions` + deny-on-timeout + the one-time
nonce ([ADR 025] § "Security model — remote permission granting", flagged
`security-sensitive` in the Phase 3 decomposition). The label rides there, not
here — a slice earns it only when it adds/changes a gate, route, or
trust-boundary parse; this adds none.

## Out of scope (deferred to siblings)

- **The v2 wire types + the `PermissionRequest` → wire mapping** — `internal/protocol`
  is the wire SSOT ([#607]'s territory); this ticket defines no wire types and
  pushes nothing to the wire. #607's mapping may add an exported `PermissionOptionKind`
  accessor when it needs to enumerate (same deferral the existing three enums use).
- **The broader modal wire shape** — `modal_shown` with `class` / `multi_select`
  / `default_option_id` ([ADR 025] wire section) and the mapping of
  `PermissionRequest` onto it are separate downstream tickets. Scope here is the
  permission modal specifically (ACP `session/request_permission`), not the
  general modal.
- **Parsing the inbound `PermissionResponse` frame + the gate / nonce / modal
  control loop** — the security-sensitive downstream slices (see above).
- **The remaining inbound commands** — `Prompt`, `Cancel`, `DropQueued` join
  `Inbound` in the deferred inbound-commands ticket (now an additive change; that
  ticket may relocate the `Inbound` marker to its own `inbound.go` when it adds
  siblings — a forward note, not this ticket's work).
- **The `pyry acp` adapter** — re-derives ACP's `session/request_permission` onto
  this `PermissionRequest` ([#600]).
