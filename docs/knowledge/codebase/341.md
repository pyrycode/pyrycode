## Patterns established

- **Sibling lock file for atomic-rename writers.** Lock `~/.claude.json.lock`, NOT `~/.claude.json`. The atomic-write recipe replaces the data file's inode on every write, so a second writer that opens the data file *after* the first writer's rename gets a different inode and acquires its own independent lock — locks would not serialize. The sibling lock file's inode never changes. This is the third file-locking site in pyrycode (after `internal/devices` and `internal/sessions`); any future helper that takes a file-lock around an atomic-rename writer must follow the same pattern.
- **`json.Decoder.UseNumber()` is mandatory for pass-through preservation.** Round-tripping JSON we do not own (`~/.claude.json` is claude's, not pyry's) through `map[string]any` requires `UseNumber` to avoid silent corruption of int64-sized values above the 2^53 float64 mantissa. The dedicated test row (`TestMarkWorkdirTrusted_PreservesNumericPrecision`) pins this — without `UseNumber` the row fails immediately. Convention for any future "preserve unknown fields" registry merge.
- **`syscall.Flock` is stdlib-only on darwin + linux.** No `golang.org/x/sys/unix` dependency; no build tag. BSD flock semantics on both platforms. Kernel releases the lock on fd close (including process crash), so no stale-lock cleanup is needed and the lock file is intentionally never deleted (deletion races acquisition).
- **Explicit `homeDir` parameter over `os.UserHomeDir` + `t.Setenv("HOME", ...)`.** `t.Setenv` blocks `t.Parallel`; the explicit-`homeDir` shape lets every test row run in parallel. Same shape already used by `internal/install.ResolveWorkDir`. Production callers pass `os.UserHomeDir()` at the call site.
- **Pre-existing file mode is preserved.** `os.Stat` the data file once before the lock (mode is informational, not a security boundary). If claude wrote at `0o644`, pyry continues writing at `0o644`. Lock file is always `0o600`. If a future security review wants `0o600` enforced regardless, that's a separate ticket — re-tightening a permission the user may have intentionally widened is not a silent decision to make in a helper slice.
- **Malformed-input failures leave the file byte-identical.** The atomic-rename pattern means a write either happens fully or not at all; combined with refusing to proceed past parse / type-assertion errors, the helper guarantees the on-disk state is unchanged on every error path that isn't an I/O failure mid-rename. The three loud-failure test rows assert this directly via `bytes.Equal` against the pre-write snapshot.

## Lessons learned

- **`encoding/json` defaults silently corrupt unknown JSON.** Without `UseNumber()` a `map[string]any` round-trip through `json.Decode` → `json.Encode` will mangle any number above 2^53 even if the input was a string-formatted integer that fits in int64. The "preserved verbatim" AC was the surface signal; the actual lesson is broader — any time pyry round-trips JSON it does not own (claude's state, third-party config files, MCP-server-emitted JSON), `UseNumber` belongs in the decoder configuration by default.
- **Atomic-rename invalidates "lock the file you're writing" intuition.** A reflexive first design was to flock the data file; security-review caught it during architect self-review. Generalisation: when the write strategy replaces the inode, the lock target must be a separate stable inode. Worth pinning here because the trap is invisible — a "lock the file" implementation passes single-process tests trivially and only fails under genuine multi-process contention, which unit tests cannot reach.
- **Spec self-review caught two MUST FIXes (numeric precision + logging discipline) before code review.** Both made the codified architecture stronger in-place rather than leaving them as code-review pushback. Pattern worth repeating on any helper that touches shared state pyry does not own.
