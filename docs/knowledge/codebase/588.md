# Ticket #588 — `internal/relay`: enumerate open v2 Noise session conn IDs for server-initiated fan-out

The enumeration half of server-initiated delivery to v2 (Noise) phones — the snapshot primitive [#571](571.md) deliberately deferred. `(*V2SessionManager).Push` ([#571](571.md)) can address *one* phone by `conn_id` but cannot discover *which* phones are connected; fanning an unsolicited frame out to **all** connected phones needs a concurrency-safe list of currently-open `conn_id`s. `(*V2SessionManager).ActiveConnIDs(ctx) []string` is that list: it returns a snapshot of the conn IDs of every session in `V2StateOpen`, safe to call from a producer goroutine other than the dispatch goroutine. The consumer is #572's assistant-turn bridge, which calls `ActiveConnIDs`, then `Push` on each returned id to fan one `message` envelope out to every connected phone. It is the v2 analog of v1's `dispatch.Dispatcher.ActiveConns()`. Purely additive: one new public method + its funnel plumbing in a single production file, no new wire shapes, no change to the inbound dispatch path, no change to v1.

## Implementation

**One production file: `internal/relay/v2session.go` (+~70 LOC).** The whole change is the fourth instance of the single-writer funnel idiom — the structural twin of [`Push`](571.md) / [`Rekey`](462.md), with the seal/marshal steps dropped (a snapshot reads the map, it doesn't touch any CipherState).

- **`snapshotReq` struct** — `{reply chan []string}`. Mirrors `pushReq` minus the `connID`/`env` inputs (a snapshot takes no addressed-conn argument). `reply` is `cap=1` so `Run`'s reply send is non-blocking even if the caller's ctx fires between enqueue and reply.
- **`snapshot chan snapshotReq` field on `V2SessionManager`** — unbuffered, allocated in `NewV2SessionManager` alongside `push` (`snapshot: make(chan snapshotReq)`). Doc-comment mirrors `push`'s: unbuffered backpressure is correct (Run busy ⇒ caller waits); not closed on `Run` exit; in-flight callers unblock via `ctx.Done`.
- **One new `Run` select arm** — `case req := <-m.snapshot: req.reply <- m.handleActiveConnIDs()`, sitting next to the existing `m.push` arm.
- **Public `ActiveConnIDs(ctx) []string` method** — does only channel I/O on the **caller's** goroutine: a `select` send onto `m.snapshot` (or `ctx.Done` → return `nil`), then a `select` receive on `req.reply` (or `ctx.Done` → return `nil`). Byte-for-byte the `Push` shape with the `ErrConnNotFound` return swapped for `nil`. No session-state access happens here.
- **Private `handleActiveConnIDs() []string` handler** — runs on `Run`'s dispatch goroutine, the only site that reads `m.sessions` for the snapshot:
  ```go
  out := make([]string, 0, len(m.sessions))
  for connID, s := range m.sessions {
      if s.state == V2StateOpen { out = append(out, connID) }
  }
  return out
  ```

### Return type `[]string`, not `([]string, error)`

A snapshot has no failure mode the caller can act on. The only non-completion is ctx cancellation or `Run` already exited (Frames closed) — both return `nil`, equivalent to "no open sessions" for the broadcast consumer (it fans out to nobody this round; the next assistant turn re-enumerates). `nil` and an empty non-nil slice are both `len 0` and interchangeable to callers. This avoids forcing #572 to handle an error it cannot meaningfully act on. (Divergence from v1's `ActiveConns()`, which takes no `ctx` because it reads `d.conns` under a mutex; v2 has no mutex, so the read must funnel through `Run` and the public method needs a `ctx` escape arm or it blocks forever once `Run` exits — same reason `Push`/`Rekey` take `ctx`.)

### Unordered set, documented

`handleActiveConnIDs` returns conn IDs in Go's randomized map-iteration order. The AC requires no ordering and the broadcast consumer fans out order-independently, so the handler does **not** sort — keeping Run's per-call work minimal (a sort would add O(n log n) on the single dispatch goroutine for no functional benefit). The method's doc-comment states the result is an unordered set; tests sort-then-compare, never assert positional order.

## Patterns established

- **The single-writer funnel, reused a fourth time, is the package's concurrency idiom — not a mutex.** `m.sessions` is owned exclusively by `Run`; there is no mutex (the loop is the lock, because flynn/noise `CipherState`s are not safe for concurrent use). Reading `m.sessions` from any other goroutine is a data race. The snapshot read executes on `Run`'s existing dispatch goroutine when it services the `m.snapshot` arm — serialised by `Run`'s `select` against every map write (lazy-create in `handleFrame`, `delete` in `closeWith`, state transitions in the handshake handlers). No read can observe a half-updated map or a torn `s.state`. The unbuffered `m.snapshot` channel forces a cross-goroutine caller to *wait its turn* rather than race. No new goroutine, no new lock, no atomic.
- **The `V2StateOpen` filter is the load-bearing security gate (`security-sensitive`).** Only an open session has had its token validated (in `handleNoiseInit`'s accept branch); a `V2StateHandshakeComplete` session holds CipherStates but never passed the token check. The filter excludes it — identical to the gate `handlePush` enforces — so a server push never reaches an un-authenticated peer. **Belt-and-suspenders, different fabric:** even if this filter regressed, the consumer calls `Push(ctx, id, env)` per returned id, and `Push` independently gates on `V2StateOpen`. Two deterministic, independent code-level checks in two different methods (enumeration filter + `handlePush` gate); neither is a stochastic agent rule. The enumeration filter is the primary; `Push`'s gate is the net.
- **Composition with re-key and teardown is free.** A `handleRekeyInit` CipherState swap, a `closeWith` delete, and a `handleActiveConnIDs` read all run on `Run`; a snapshot reflects either the pre-event or post-event map, never an inconsistent intermediate. A `V2StateClosed` session cannot appear — `closeWith` already removed it from the map.
- **No new reader of `s.state` off the Run goroutine.** `V2Session.State()`'s doc-comment flags that a cross-goroutine reader of `s.state` would need `atomic.Int32`/mutex. This snapshot does **not** create such a reader — `handleActiveConnIDs` reads `s.state` *on* `Run`, behind the `m.snapshot` funnel. `State()` still needs no atomic/mutex. The "broadcast/enumeration layer added in a later slice" that the `State()` comment anticipated is this ticket — and it lands without disturbing that invariant.
- **No log line, no secret-bearing field.** `handleActiveConnIDs` emits nothing (a pure read, consistent with `handlePush`, which logs only on its transport path), so no conn-id-of-an-unauthenticated-session can leak. The return holds only conn-id strings (non-secret routing identifiers), never a `*V2Session` or any key/plaintext bytes, and is handed only to the in-process consumer — never emitted on a wire.

## Lessons learned

- **Pre-existing `gofmt` drift in `v2session.go` / `v2session_test.go` was left untouched as out-of-scope (continuation of [#571](571.md)'s call).** Both files carry gofmt drift (an `openSession` struct alignment, an envelope-literal alignment, an EOF trailing blank line) inherited from [#581](581.md)'s `code-review: auto-commit` commit. This slice's additions are gofmt-clean; `gofmt -l` is **not** a CI gate (CI runs `go vet` / `staticcheck` / `go test -race`), so the drift does not block. Worth a dedicated `gofmt -w` sweep, but folding an unrelated whitespace reformat into a feature PR would muddy the diff — left for a cleanup ticket. (Lifted from the PR body's Lessons-learned section; same finding as #571.)

## Testing

Five white-box tests in `internal/relay/v2session_test.go` (same package), stdlib `testing` only, all `-race`-clean. Built on the existing harness (`startManager`, `driveToOpen`, `v2Recorder`, `genV2Keypair`, `v2PairedRegistry`, `waitForEnvelopes`, `sealAppFrame`, `decryptAppFrame`) and the white-box `mgr.sessions[id] = &V2Session{…}` injection idiom. No existing test touched.

- `TestV2Session_ActiveConnIDs_OpenOnly` (AC#1) — inject `{a: Open, b: Open, c: HandshakeComplete, d: AwaitingInit}` into a started manager, then `ActiveConnIDs`; assert the sorted result equals `{a, b}` exactly. Pins both halves of AC#1 (every open id present, every non-open id absent). `-race` clean via the same happens-before edge as `TestV2Session_Push_NotOpen`: `ActiveConnIDs` reads `s.state` only (never `s.send`), so the nil CipherStates on injected sessions are safe, and the snapshot channel-send publishes the map writes to `Run` (no frames fed, no timers armed; Run touches the map only when it services the snapshot).
- `TestV2Session_ActiveConnIDs_TornDownSessionAbsent` (AC#2) — drive a real open via `driveToOpen`, confirm its id is enumerated, then drive an AEAD-failure 4421 teardown (flip a ciphertext byte — the #571 `Push_ClosedSession` recipe) so `closeWith` deletes it. Observing the 4421 close envelope guarantees the `delete` ran on `Run` before the subsequent snapshot (also serviced on `Run`), so the re-enumeration deterministically sees the post-delete map and no longer contains the id.
- `TestV2Session_ActiveConnIDs_ConcurrentWithDispatch_RaceClean` (AC#3 — the `-race` proof) — a separate goroutine hammers `ActiveConnIDs` in a tight loop while M=16 inbound sealed request frames drive `dispatchAppFrame` replies on the same open session, so the snapshot funnel and the reply path contend for the single `Run` goroutine. Every snapshot is either `{v2TestConnID}` or empty (never garbage); afterwards the solicited replies still decrypt in capture (= seal) order, proving the concurrent reads never corrupted session state (nonce integrity intact).
- `TestV2Session_ActiveConnIDs_EmptyManager` — a started manager with zero sessions returns a `len 0` slice without blocking.
- `TestV2Session_ActiveConnIDs_CtxCancelled_ReturnsNil` (deterministic) — construct the manager but do **not** start `Run`; an `ActiveConnIDs` call with an already-cancelled ctx returns `nil` without blocking. With no `Run` goroutine draining `m.snapshot`, the `ctx.Done` send arm is the only ready case — no flakiness from `select` picking the send.

AC#4 (no regressions) is covered by the existing `internal/relay` suite passing unmodified — the change is additive. `go test -race ./...` and `go vet ./...` pass; `go build ./cmd/pyry` OK.

> **Real-claude e2e not evaluated (environmental, not a code regression).** Code review recorded a PASS on judgment grounds but flagged that `make e2e-realclaude` could not run: every test died at the harness's internal `go build pyry` step before any test logic, because the sandbox cannot fetch the private module `github.com/pyrycode/tui-driver v1.0.1` (no git credentials). This is a sandbox-credential limitation untouched by #588 (the diff touches only `v2session.go` + its test; `go.mod`/`tui-driver`/`agentrun` are unchanged) — escalated to the operator to run the real-claude e2e in a credentialed environment before merge, rather than routing `needs-rework:developer` (unfixable by the developer, would loop).

## Out of scope (deferred — owned by #572)

- **Broadcast wiring / the assistant-turn bridge.** `ActiveConnIDs` is the enumeration *primitive*; the bridge that taps the assistant/PTY output stream and fans `message` envelopes to every connected phone (call `ActiveConnIDs`, then `Push` per id) is #572's deliverable, on the v2 leg. With #588, both pieces #571 deferred to #572 — the enumeration snapshot **and** the addressable push — now exist; only the bridge wire-up and outbound envelope-ID policy remain.
- **Production wiring.** Like `Push` and `Rekey` before their wire-ups, `ActiveConnIDs` is reachable only from `internal/relay` tests until #572 wires the bridge. `cmd/pyry/relay.go:startRelayV2` is where #572 obtains the `*V2SessionManager` handle.
- **Per-session richer snapshot.** #572 needs only conn IDs (it builds the envelope itself and calls `Push`). If a later consumer needs device name / open-since timestamp per conn, the return type evolves to a small struct slice — out of scope here; `[]string` is the minimal surface the AC asks for.
- **E2E oracle extension.** No production caller of `ActiveConnIDs` exists until #572, so the real `send_message → message` fan-out round-trip is #572's e2e deliverable (extending `internal/e2e/relay_v2_daemon_test.go`). No e2e change in this slice.
