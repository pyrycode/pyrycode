# Ticket #382 — `e2e/realclaude`: protocol resilience tests

Fifth consumer of the `internal/e2e/realclaude` fixture trio (#372 + #373 + #409). Adds four named tests — one per production failure mode that pyry's supervisor must interpret as a structured signal rather than a silent hang: a Bash tool returns non-zero, claude's stdin closes before processing finishes, claude receives malformed stream-json, and a prompt approaches the context window.

Motivated by the gap the prior consumers don't cover: every existing realclaude test asserts on the happy path. Production dispatch fails in deviation shapes that pyry can recover from ONLY if claude surfaces them as named exit codes / JSONL events / stderr diagnostics. Each test pins a **structured exit signal** under a bounded timeout — a hang IS the failure mode being guarded against, so deadline expiry is itself an assertion target. No production-code change; the only non-test edit is a two-field extension of the file-local `contentBlock` struct in `tool_loop_test.go`.

See [features/e2e-realclaude.md](../features/e2e-realclaude.md) for the suite contract.

## Implementation

- **`internal/e2e/realclaude/resilience_test.go`** (~380 LoC, new). `//go:build e2e_realclaude`, `package realclaude`. Four top-level `Test…` functions + three file-local helpers (`resolveClaudeBin`, `directClaudeArgs`, `runClaudeDirect`) + one envelope builder (`buildUserTurnEnvelope`). No new fixture-package surface; no edits to `fixtures.go`. Two tests (`BashTool_NonZeroExit`, `LargePromptNearContextWindow`) drive claude through pyry via `RunPyryAgentRun`; two (`PrematureStdinClose`, `MalformedStreamJSON`) bypass pyry and exercise `claude` directly via `exec.CommandContext` so the test can control raw stdin bytes and observe claude's untransformed exit shape.
- **`TestRealClaude_BashTool_NonZeroExit`** (`resilience_test.go:35-142`). `RunPyryAgentRun` + `AllowedTools: []string{"Bash"}` + prompt `"Use the Bash tool to run \"ls /nonexistent-path-for-pyrycode-382\". Report what you see."`. **Three independent JSONL observations tracked separately** so a failure message names the actual missing event rather than a downstream consequence: (1) capture `bashToolUseID` from the first `assistant` `{Type:"tool_use", Name:"Bash"}` content block; (2) on `user` entries, scan for `{Type:"tool_result", ToolUseID==bashToolUseID}` and record `toolResultIsErr` + `toolResultContent`; (3) on subsequent `assistant` entries, set `sawFinalText` if any `{Type:"text", Text:non-empty}` block lands. Then five assertions: `bashToolUseID != ""`, `sawToolResult`, `toolResultIsErr == true`, `len(toolResultContent) > 0`, `sawFinalText`. Finally `parseResultTrailer(result.Stdout)` → assert `trailer.Subtype == "success"`. **Do NOT assert on `StopReason`** — claude may end on `end_turn` (recovered gracefully) or `tool_use_error` (named structured signal) and both satisfy the AC; the spec contract is "structured signal, not pipe break" and `Subtype == "success"` IS that contract.
- **`TestRealClaude_PrematureStdinClose`** (`resilience_test.go:149-171`). `WithWorktreeAuthenticated(t)` (real API call — the property is "claude exits cleanly after our envelope and stdin-close", which requires claude to actually accept and process the envelope). `resolveClaudeBin(t)` + `buildUserTurnEnvelope("Reply with a single short word.")` + 60 s `context.WithTimeout`. Writes the single newline-terminated JSON envelope `{"type":"user","message":{"role":"user","content":[{"type":"text","text":"…"}]}}` — bit-identical to `streamrunner.marshalEnvelope`'s production shape — then closes stdin. Two assertions: `ctx.Err() != context.DeadlineExceeded` (hang regression guard) and `exitCode == 0` (clean exit per AC). No assertion on stdout content; the property is liveness, not response.
- **`TestRealClaude_MalformedStreamJSON`** (`resilience_test.go:176-214`). `WithWorktree(t)` is sufficient (no API call expected — claude rejects before reaching the model). `resolveClaudeBin(t)` + garbage bytes `"this is not stream-json\n}}{{not json either\n"` + 60 s timeout. Three assertions: no deadline expiry, `exitCode != 0` (rejected), `len(stderr) > 0` (diagnostic emitted). **Stderr predicate is an OR-of-keywords match**, not a hard-coded substring: `{"json", "parse", "input", "format", "envelope"}` checked case-insensitively. The AC asked for "a diagnostic substring"; pinning a specific phrase couples the test to claude's diagnostic copy which can change without notice. The keyword set is a parse-shape fingerprint that catches "claude crashed silently" without ratcheting on prose. If a future claude version emits zero matches, expand the set with whatever it actually prints — keep the OR-shape, never pin a phrase.
- **`TestRealClaude_LargePromptNearContextWindow`** (`resilience_test.go:222-269`). `WithWorktreeAuthenticated(t)` + `strings.Repeat("All work and no play makes Jack a dull boy. ", 1200)` → ~52 KB prompt. `RunPyryAgentRun` with `MaxTurns: 3`, `Effort: "low"`, `Model: "claude-haiku-4-5"`, `AllowedTools: []string{"Read"}`. **Disjunctive assertion structured as a single `if !branchA && !branchB`** so the failure message lists both branches' requirements when "neither" trips. Branch A (claude processed it): `trailer.Subtype == "success"` AND `trailer.StopReason NOT IN {"context_overflow", "max_context_length"}` — an unknown new stop_reason is treated as the structured-error branch by virtue of the membership check (overflow-shaped reasons are known and explicit; everything else passes). Branch B (claude rejected via structured stream-json): `trailer.Subtype != "success"` AND `result.ExitCode != 0` (do NOT pin a specific exit code — `!= 0` is sufficient when paired with the non-success subtype). The point is: a result trailer IS present (no crash mid-stream), and either claude processed the prompt cleanly or it rejected with a named subtype.
- **File-local helpers (`resilience_test.go:271-378`).** `resolveClaudeBin(t) string` honours `PYRY_CLAUDE_BIN` UNLESS it points at the current test binary — **fork-bomb defense pattern from 2026-05-16** that `RunOpts.UseTestBinaryAsFakePyry` (#409 era) also encodes; resolves the absolute path of both `PYRY_CLAUDE_BIN` and `os.Args[0]` and `t.Fatalf`s if they match (NOT `t.Skipf` — this is a misconfigured run, not an absent claude binary). Falls back to `exec.LookPath("claude")` and `t.Skipf`s on miss (matches `smoke_test.go:12-14`'s skip-with-context shape). `directClaudeArgs() []string` returns the eight-flag argv mirroring `cmd/pyry/agent_run.go:buildClaudeArgs` minus `--append-system-prompt-file` (the direct-exec tests don't need a system prompt for the property they probe). `runClaudeDirect(t, ctx, bin, workdir, stdinBytes) (int, []byte, error)` wraps `exec.CommandContext` + `StdinPipe` + `Start` + `Write` + `Close` + `Wait` and returns `(exitCode, stderr, waitErr)`. Stdout is captured but not returned — neither caller asserts on response content, and capturing it (vs. discarding) ensures the child's pipe doesn't backpressure. **Stdin-write errors are `t.Logf`-not-`t.Fatalf`** because Test 3's fast-reject path produces an EPIPE-wrapped `*os.PathError` from claude having already closed its end; the child's exit status is the authoritative outcome, mirroring streamrunner's log-and-continue posture. `buildUserTurnEnvelope(prompt) ([]byte, error)` constructs the single-line `{"type":"user","message":…}` envelope locally — kept file-private rather than importing `internal/agentrun/streamrunner` from a test helper to keep the dependency direction one-way (e2e tests don't depend on production wiring packages they're meant to probe).
- **`internal/e2e/realclaude/tool_loop_test.go`** — two-field extension to `contentBlock` (`tool_loop_test.go:155-163`): added `Content json.RawMessage \`json:"content,omitempty"\`` and `IsError bool \`json:"is_error,omitempty"\``. **One-line struct field additions; no behavior change to existing tests** — the new fields are `omitempty`-tagged and the existing `TestRealClaude_ToolLoopIntegrity` (#376) does not read either. Both fields are read only by `TestRealClaude_BashTool_NonZeroExit`. Doc-comment on the struct gained a four-line note explaining why `tool_result` content uses `json.RawMessage`: the body is a string OR an array of nested content blocks per claude's stream-json contract, and `RawMessage` holds either shape so callers can check non-emptiness without binding to one form. **Reuse over duplication**: both `resilience_test.go` and `tool_loop_test.go` live in the same package under the same build tag, so the import dependency works without re-declaration. The caveat that a structural refactor excluding one file from the build set would break the other is acceptable today — both files share the same tag.
- **Bounded timeouts everywhere.** Tests 2 and 3 use 60 s `context.WithTimeout` — tight, because the property is "no hang". Tests 1 and 4 inherit the 5-minute `RunPyryAgentRun` default — the runaway guard, not an SLO (haiku at 50 KB resolves in well under that). Deadline expiry is itself a structured failure mode that the assertions check via `ctx.Err()`.

## Patterns established

- **Track independent observations independently when the goal is failure-message attribution.** Test 1 sets `bashToolUseID`, `sawToolResult`, and `sawFinalText` as three separate variables, then asserts each in order with a distinct `t.Fatalf` message. The first-pass implementation gated `sawFinalText` on a derived `toolResultErr` flag; review caught this as a SHOULD FIX because a failure in the tool_result observation would have surfaced as "no subsequent assistant text" — pointing the operator at the wrong root cause. When a test has N causally-ordered observations, track them as N independent flags so the first-missing observation is named explicitly. Generalises to any state-machine assertion walking a stream of events.
- **For raw-byte stdin tests against a CLI, bypass the helper and call `exec.CommandContext` directly.** Tests 2 and 3 do NOT go through `RunPyryAgentRun` — the helper's contract assumes well-formed stream-json on stdin (built from `opts.Prompt`/`opts.SystemPrompt` via prompt files, not raw bytes). Tests that need to drive a CLI with arbitrary stdin bytes should drop down to `exec.Cmd` and own the argv directly; expanding `RunOpts` with a raw-stdin escape hatch would expose a footgun for the helper's normal callers. The spec's architectural call to keep `runClaudeDirect` file-local rather than promoting it to `fixtures.go` (single-file consumer today) follows the same YAGNI bar the suite uses everywhere.
- **Disjunctive assertions belong in a single `if !A && !B` guard with both branches in the failure message.** Test 4's "either processed cleanly OR rejected with named subtype" structure produces actionable diagnostics on the "neither" failure mode by listing both branches' requirements in the `t.Fatalf` string. Splitting into nested `if/else` chains hides the second branch's expectation from the operator. Generalises to any "either A or B" property where A and B are observable from the same data.
- **Keyword-OR predicates beat hard-coded substrings for CLI diagnostic assertions.** Test 3's stderr check matches against `{"json", "parse", "input", "format", "envelope"}` case-insensitively; pinning a specific phrase ratchets the test to claude's prose, which Anthropic can change without notice. The keyword set is the diagnostic's *shape* fingerprint, not its literal content. Generalises to any test asserting on a third-party CLI's stderr / log output where the operator-readable message is allowed to evolve.
- **Fork-bomb defense for any helper that reads `PYRY_*_BIN`.** `resolveClaudeBin` compares the absolute path of `PYRY_CLAUDE_BIN` against `os.Args[0]` and `t.Fatalf`s on match. This is the second occurrence of the pattern (`RunOpts.UseTestBinaryAsFakePyry`'s opt-in was the first, after the 2026-05-16 fork-bomb incident). Any future helper that resolves a binary path from an env var should check whether the env var has been pointed at the test binary itself and refuse rather than recurse. Generalises to any test infrastructure that allows external override of a child-process binary.

## Lessons learned

- **`tool_result` content blocks carry their body in `content`, not `text`.** First-pass implementation read `b.Text` for the `tool_result` content non-emptiness check; `b.Text` is always `""` for `tool_result` blocks (the field is populated only for `text` blocks), so the empty-content assertion would have fired on every nightly run regardless of claude's actual output. Code review caught this as a MUST FIX (commit `0e8e364`); the fix adds a `Content json.RawMessage` field to `contentBlock` because the body can be either a string OR an array of nested blocks per claude's stream-json contract, and `json.RawMessage` holds either shape. Generalises: when extracting a field from a foreign CLI's structured output, verify the field name and shape against real captured output (e.g. `internal/agentrun/jsonl/testdata/*.jsonl`) before writing the assertion. Don't infer the field name from a similar-but-different block type.
- **Verify documented stream-json fields against real captured testdata, not just specs.** The architect's spec asserted `is_error == true` would appear on tool_result blocks for non-zero Bash exits and flagged "developer should verify with one manual run". The implementing developer verified against `internal/agentrun/jsonl/testdata/{clean,no_end_turn,double_end_turn}.jsonl` which contain 19 `"is_error":...` occurrences from real claude runs — confirming the field's presence without spending a real API call on a single manual probe. The testdata directory is a free verification surface for stream-json shape questions; consult it before scheduling exploratory API calls.
- **`io.ErrClosedPipe` does NOT match EPIPE-wrapped `*os.PathError` from `*exec.Cmd.StdinPipe().Write`.** First-pass `runClaudeDirect` had `if err != nil && !errors.Is(err, io.ErrClosedPipe)` to swallow expected broken-pipe writes silently; review caught this as a NIT — broken-pipe writes to a closed `*os.PathError` chain produce an `*os.PathError` wrapping a `syscall.Errno(EPIPE)`, which is not `io.ErrClosedPipe`. The predicate matched nothing, so the original intent (silent broken-pipe handling) failed open and EPIPE errors logged as warnings. The fix drops the predicate entirely — always `t.Logf` non-nil errors, mirroring streamrunner's log-and-continue posture. Generalises: when guarding against a "specific expected error" sentinel, verify the sentinel actually matches the error you receive (read the implementation, check the wrapping chain) rather than guessing at the right one.
- **Couple sub-observations only when they share a causal premise, not when they share a chronological position.** First-pass Test 1 derived `sawFinalText` from a `toolResultErr` flag (final text only counts after we saw an error tool_result); review caught this as a SHOULD FIX. The chronological coupling (final text comes after tool_result) is real, but the *causal* premise is just "tool_use happened" — the final text doesn't depend on whether the result was an error. Coupling them obscured failure attribution: a missing tool_result would surface as "no subsequent assistant text". The fix tracks `sawToolResult` independently and gates `sawFinalText` on it, so the assertion that fails is the assertion at the missing observation. Generalises: when a test walks a stream and the assertions are ordered, gate later flags on the *minimum prerequisite* observation (existence), not on derived properties of it (like `IsError`).

## Verification

- `go vet -tags=e2e_realclaude ./internal/e2e/realclaude/...` — passes.
- `go build -tags=e2e_realclaude ./internal/e2e/realclaude/...` — compiles.
- `go test -tags e2e_realclaude -run 'TestRealClaude_(BashTool_NonZeroExit|PrematureStdinClose|MalformedStreamJSON|LargePromptNearContextWindow)' -v ./internal/e2e/realclaude/` — all four pass against a working `claude` binary and live API key. **Three real haiku calls per run** (Tests 1, 2, 4; Test 3 rejects before reaching the model and incurs zero API cost). Target ~$0.08 per `make e2e-realclaude` run with prompt cache warm; per-test wall time well under the 60 s / 5 min bounded timeouts.
- `make test` (no tag) — unchanged; the new file is dropped at the build stage by tag exclusion.

## Out of scope (deferred)

- **Promoting `runClaudeDirect` / `directClaudeArgs` / `buildUserTurnEnvelope` to `fixtures.go`.** Single-file consumer in #382; if a future ticket adds a third raw-exec test, promote then. The spec explicitly forbade touching `fixtures.go` in this ticket.
- **Promoting `parseContentBlocks` / `contentBlock` / `parseResultTrailer` / `resultTrailer` / `jsonlPathFor` / `truncate` to exported fixture helpers.** All five are now multi-consumer (`parseContentBlocks` + `contentBlock`: #376 + #382 = two; `parseResultTrailer` + `resultTrailer`: #376 + #381 + #382 = three; `jsonlPathFor`: six consumers; `truncate`: two consumers). Package-private access still suffices because every consumer lives in the same `realclaude` package under the same build tag. Promote when a sibling-package test (or a separate-package operator-facing fixture) needs the same primitive.
- **Per-version diagnostic-keyword maintenance for Test 3.** If a future claude version emits stderr with zero matches against `{"json", "parse", "input", "format", "envelope"}`, expand the set with whatever it actually prints. Keep the OR-shape; never pin a specific phrase. A pull request that adds a keyword and cites the claude version is the right shape.
- **Resilience tests for other failure modes.** The four picked here are the highest-value production-dispatch failure modes today (per the ticket body). Additional modes that might warrant future tests: API timeout / partial-stream-truncation, rate-limit (429) handling, expired-credential mid-stream, max_tokens exceeded under tool-loop. Each is a separate ticket and a separate test file when the operational need surfaces.
