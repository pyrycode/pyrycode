# Ticket #421 — `e2e/realclaude`: long-running session JSONL append integrity (≥10 turns)

New `internal/e2e/realclaude/long_session_test.go` — eighth consumer of the realclaude fixture trio and the **first ≥10-turn test in the suite**. Drives a real `claude` session through ten single-command Bash operations against a seeded `numbers.txt`, then pins three append-discipline contracts on the result (trailer `num_turns >= 10`, ≥10 assistant entries with `EndOfTurn=true` in the on-disk JSONL, last assistant entry is itself a well-formed end-of-turn with non-empty text) plus a forward-defensive negative stderr tripwire for `bufio.Scanner: token too long`. Closes the long-turn-count gap left by the existing suite (peer tests cap at `MaxTurns ∈ {1,2,3,4}`): a subtle regression in the multi-turn append path — Scanner buffer downgrade, off-by-one trailer write, trailing-newline drift, scanner reset across turns, buffer flush gap swallowing the last event — would not have been caught by any test that runs today.

## Implementation

- **One new file, zero edits elsewhere.** `internal/e2e/realclaude/long_session_test.go`, `//go:build e2e_realclaude`, ~140 LoC including the two prompt constants and inline comments. No edits to `fixtures.go`, `tool_loop_test.go`, or any peer test file — no new exported types, no new helpers, no new struct fields. Purely additive.
- **Composition: `WithWorktreeAuthenticated` → seed `numbers.txt` → `RunPyryAgentRun` → `parseResultTrailer` + `ReadJSONL`.** Mirrors the canonical realclaude pattern from `budget_test.go` ([`#385`](385.md)). The seed file is ten lines of one-digit-per-line content (`"1\n2\n3\n4\n5\n6\n7\n8\n9\n10\n"`, `0o600`) written under `workdir` before `RunPyryAgentRun`. Content is incidental — the file exists solely to give Bash one-per-turn things to inspect.
- **`RunOpts`: `MaxTurns=12, Effort="low", Model="claude-haiku-4-5", AllowedTools=["Bash"], Timeout=10 * time.Minute`.** `MaxTurns=12` is a deliberate 2-turn headroom above the ≥10 floor — caught early if haiku starts collapsing turns despite the anti-chain steering, but never trips on the natural completion path. `Timeout` bumped from the 5-minute `RunPyryAgentRun` default to 10 minutes: a 10-turn haiku-low run is typically 1–3 minutes wall-clock, but cold network and queue variance can push it; doubling the default keeps the test from flaking on tail latency without inflating the success-path budget.
- **Two file-scope prompt constants** mirroring `budget_test.go`'s `maxTurnsSystemPrompt` / `maxTurnsPrompt` shape. `longSessionSystemPrompt` is the one-paragraph regression-guard steering ("use the Bash tool one command at a time, wait for each result before continuing, do NOT combine commands, and do NOT chain commands with && or ;"); `longSessionUserPrompt` enumerates the ten distinct single-command Bash operations against `numbers.txt` (numbered list: `wc -l`, `head -n 3`, `tail -n 3`, `sort`, `uniq`, `cat`, `grep 5`, `wc -c`, `awk '{s+=$1} END {print s}'`, `ls -l`) and ends with the explicit anti-collapse line plus a "summarize what you saw in one short sentence" tail — the summary nudges the model to emit one more `assistant`-kinded `end_turn` text block, giving the last-event-EndOfTurn assertion something to land on without depending on the model spontaneously producing a closing turn.
- **Six sequential assertions, in evaluation order** (each `t.Fatalf` with truncated stderr/stdout + JSONL path where relevant):
  1. `result.ExitCode == 0` (truncated stderr on failure).
  2. `result.SessionID != ""` (truncated stdout on failure).
  3. `parseResultTrailer(result.Stdout)` succeeds (truncated stdout on failure).
  4. `trailer.NumTurns >= 10` — failure message names the observed value AND the prompt-design recourse: "if the model is collapsing turns, expand the prompt or strengthen anti-chain steering — do NOT lower the threshold". The in-message recourse is load-bearing; without it the next maintainer is one search-and-replace away from defeating the test.
  5. JSONL walk via `ReadJSONL`: single pass counting assistant entries with `EndOfTurn=true` and tracking the last assistant entry. Asserts `endOfTurnCount >= 10` (failure includes `jsonlPath`, the observed count, and the total assistant-event count) and that the last assistant entry satisfies `EndOfTurn && TextChars > 0` (failure includes `jsonlPath`, the last entry's `EndOfTurn`/`TextChars`, and the total assistant count — same shape as `per_agent_test.go:128-132`).
  6. `!bytes.Contains(result.Stderr, []byte("bufio.Scanner: token too long"))` — the literal stdlib error text, never paraphrased. The substring is what `bufio.Scanner` emits when a single token exceeds its buffer; reformulating ("buffer overflow", "scanner error") would silently disable the tripwire.
- **Forward-defensive `bufio.Scanner` tripwire.** Today's `parseResultTrailer` / `parseInitSessionID` use the default 64 KiB `bufio.Scanner` buffer but never see lines large enough for the regression to bite; `ReadJSONL` parses through `jsonl.NewReader` which has a 16 MiB cap, so the read path is structurally safe. The assertion exists to fire **the day someone wires a stdlib `bufio.Scanner` into pyry's production stdout/stderr path without bumping its buffer**, hits a long claude line in a multi-turn run, and the regression would otherwise land silently. The substring check is cheap (one `bytes.Contains` per run), expected to pass trivially today, and the failure message points the next maintainer directly at the fix (`tool_loop_test.go:210` precedent: bump to 1 MiB).
- **Helper reuse, zero new helpers.** `jsonlPathFor` (from #364), `truncate` (from #381), `parseResultTrailer` + `resultTrailer` (from #376), `ReadJSONL` + `JSONLEntry` (from #372), `WithWorktreeAuthenticated` (from #409). The test is **eighth consumer of the fixture trio**, **sixth consumer of `parseResultTrailer`**, **eighth consumer of `jsonlPathFor`**, **fourth consumer of `truncate`**.
- **Concurrency.** N/A — single test, single subprocess, synchronous orchestration via `RunPyryAgentRun`. `t.Parallel()` is NOT called — matches the existing realclaude convention.
- **Cost.** One real haiku call per run, ~$0.05–$0.10 per `make e2e-realclaude` invocation. Higher per-test than the existing realclaude tests because of the turn count, but bounded — same order of magnitude as `budget_test.go`'s cache-hit + max-turns pair.

## Patterns established

- **In-failure-message recourse is load-bearing on threshold assertions whose right fix is "tighten the prompt, not lower the bar".** When an assertion guards against a regression class that COULD be papered over by relaxing the threshold itself, the failure message must name the alternative explicitly. Here `trailer.NumTurns < 10` could be silenced by lowering the constant to `>= 5` — making the test green at the cost of deleting its purpose. The failure message names the right move ("expand the prompt or strengthen anti-chain steering") so the next maintainer reads the correct recourse at the point of failure, not in a doc that's been buried under newer tickets. Generalises to any threshold assertion where the assertion value itself is part of the contract being tested.
- **Forward-defensive negative substring assertions on stdlib error text are cheap and worth landing pre-incident.** The `bufio.Scanner: token too long` assertion is a tripwire for a failure mode that doesn't exist today: no production `bufio.Scanner` is in the relevant path, so the assertion is expected to pass trivially. The cost is one `bytes.Contains` per run; the value is catching the day someone introduces a Scanner and the regression would otherwise land silently in a long-turn-count run. Two design notes that matter for similar tripwires:
  1. **Pin the literal error text, do NOT paraphrase.** The substring `bufio.Scanner: token too long` is what the stdlib actually emits. Paraphrasing ("scanner overflow", "buffer too small") would silently disable the trip.
  2. **Anchor on the long-running multi-turn test, not a short one.** A 1-turn test would not produce a long-enough line to trigger the regression. The assertion has to live where the symptomatic input shape exists, even if the assertion itself is forward-defensive.
- **One ≥10-turn test is the right coverage cap for the multi-turn append regression class.** The existing realclaude suite caps at `MaxTurns ∈ {1,2,3,4}`. Going from 4 to 10 closes the long-turn-count gap; going from 10 to 100 buys diminishing returns at multiplying cost. The architect's call (echoed in the ticket scope) is to land one well-bounded ≥10-turn test, not a 10/25/50/100 ladder. Same shape as the budget test's two-call cache-hit verification (#385): one call captures the contract; two would muddy the diagnostic without adding coverage.
- **`Timeout` bumps belong to the test, not the fixture default.** `RunPyryAgentRun`'s 5-minute default is right for the ≤4-turn tests it was sized for; a long-running test bumps `Timeout` in its own `RunOpts` rather than mutating the shared default. Keeps the success-path budget tight on the short tests (where a 10-minute default would hide a hang) and explicit on the long ones (where the wall-clock variance is real). Generalises to any per-test budget that diverges from the suite norm.

## Lessons learned

- **`ReadJSONL`'s 16 MiB cap is structural, not test concern — but the test's stderr tripwire still belongs in the long-running test.** First instinct on a "long lines could break the read path" worry is to add an assertion that the read succeeded. Wrong — `ReadJSONL` already delegates to `jsonl.NewReader` which has a 16 MiB cap (`internal/agentrun/jsonl/reader.go:27-30`), so a long-line read failure surfaces as a `t.Fatalf` inside `ReadJSONL` itself with the resolved path embedded. The remaining regression class is a future `bufio.Scanner` being introduced ELSEWHERE in pyry's stdout/stderr path WITHOUT a buffer bump, where the symptom shape is the literal stdlib error text in stderr. The assertion shape matches that symptom shape (stderr substring), not the (already-safe) read-path failure mode. Worth reading `jsonl/reader.go:27-30` once before designing a "long line" guard so the assertion lands on the actual regression class.
- **The "summarize what you saw" tail on the prompt is what makes the last-event-EndOfTurn assertion robust.** Without it, the model may close the conversation on the final `tool_result` rather than a fresh assistant text block — leaving the last `assistant`-kinded JSONL entry as a `tool_use`-only message with `TextChars == 0` and the EndOfTurn flag unset on the `assistant` side (the run still completes successfully, the trailer still reports `num_turns >= 10`, but the assertion's "the last assistant entry is itself a well-formed end-of-turn" guard fires on a non-regression). The summary nudge converts the model's natural closing behaviour into the exact shape the assertion expects, AT NO ASSERTION COST — the per-tool steps are unchanged, the summary is one extra sentence. Generalises to any test that asserts on the shape of the LAST event of a multi-turn run: nudge the model to emit the shape you're asserting on.
- **The seeded-Bash variant outperforms the text-only fallback for turn pacing.** The ticket allowed a text-only fallback ("list 10 facts about Helsinki, one per turn") if the seeded-Bash path proved flaky. The seeded-Bash variant won the rehearsal: ten distinct shell commands against a file give the model ten concrete, separable tasks (`wc -l` ≠ `head -n 3` ≠ `tail -n 3` etc.) that resist collapsing into a single combined turn even when the model is being smart about brevity. A text-only "list ten facts" prompt is more vulnerable to "here are all ten in one turn" collapse because there's no per-fact tool round-trip to enforce the turn boundary. The architect's preference for seeded-Bash held up; the fallback stays documented as recourse if a future haiku revision collapses the Bash prompt too.

## Out of scope (deferred)

- **Turn-by-turn content fidelity.** The prompt design guides turn count; the content of any individual turn is incidental to the contract. Asserting on `cat numbers.txt` returning the exact ten digits would couple the test to model behaviour that has no bearing on the append-integrity contract being guarded.
- **Session-rotation mid-turn.** Different regression class, different test surface.
- **100+ turn sessions.** Diminishing returns past 10 — see Patterns established.
- **A separate `make e2e-realclaude-long` target.** Flagged as a follow-up in the spec's Open Questions if the realclaude suite expands further. Not in scope for #421.
- **A second long-running test exercising the text-only fallback variant.** Documented as fallback in this file; only land if the seeded-Bash variant flakes in practice.
