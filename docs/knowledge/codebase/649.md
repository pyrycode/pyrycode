# Ticket #649 ‚Äî Surface the durable per-conversation event id on the live structured stream

The **producer half** of mid-turn reconnect ([EPIC #596] Phase 2, [ADR 025]
¬ß Backpressure / replay). #646 (merged) built the per-conversation event ring and
already assigns a durable, connection-independent id to every structured event the
interactive emitter fans out ‚Äî but `emit()` **discarded** that id. This slice
captures it and stamps it on the **outbound** wire via a new additive
`EventID *uint64 \`json:"event_id,omitempty"\`` field on `protocol.Envelope`, so a
reconnecting phone finally has a stable position to advertise as `last_event_id`.
The reconnect **consumer** (per-conn `last_event_id` in `hello`, ring replay,
resync marker) is sibling [#647] (`security-sensitive`), **blocked on this slice**:
a phone cannot advertise a `last_event_id` it was never sent. **Not
`security-sensitive`** ‚Äî outbound, additive, non-secret monotonic id; no inbound
trust boundary, no dispatch policy, does not touch the Noise send-nonce
single-writer invariant (the id is stamped on the envelope struct in the emitter,
before the Run-goroutine seal). Split from [#611] (sibling of #646 / #647).
Architect-sized **XS** (the ticket pre-authorized the S‚ÜíXS downgrade for the
additive-field path).

## Implementation

Two production edits, both additive. **0 new exported types**, **1 changed
construction site** (the interactive emitter); every other
`protocol.Envelope{...}` site is untouched.

| File | Change |
|---|---|
| `internal/protocol/envelope.go` | **+1 field** ‚Äî `EventID *uint64 \`json:"event_id,omitempty"\`` on `Envelope`, placed after `InReplyTo` with a doc comment. Mirrors the existing `InReplyTo *uint64` precedent exactly. |
| `cmd/pyry/interactive_turn_v2.go` | **~+6 LOC** in `emit()` ‚Äî capture `eventID := e.ring.Append(...)` (was discarded), stamp `EventID: &eventID` on each per-conn envelope, correct the stale `#647 surfaces it on the wire` comment to `#649`. No new helper, no signature change. |
| `internal/protocol/envelope_test.go` | **+1 test** ‚Äî `TestEnvelope_EventIDOmitempty` (nil ‚Üí absent; set ‚Üí `"event_id":<n>` + round-trips). |
| `cmd/pyry/interactive_turn_v2_test.go` | **additions only** ‚Äî `pushEventIDs` helper + three emitter tests extending the `// --- #646 durable event ring ---` block (existing tests unchanged). |

See [features/protocol-package.md](../features/protocol-package.md) ¬ß `Envelope`
for the evergreen envelope reference (now carrying `EventID`) and
[features/eventring-package.md](../features/eventring-package.md) for the id
source. Highlights specific to how it was built:

- **The id source is unchanged.** `eventring.Ring.Append` already returns the
  durable per-conversation id (starts at 1, strictly increasing per conversation,
  never reset). #649 only stops discarding it. No new id source, no ring change.
- **One capture, fanned to every conn by reference.** `eventID` is captured once
  per `emit()` call, before the fan-out loop; every per-conn envelope in the loop
  shares `&eventID`. So all interactive conns observe the **identical** durable id
  for a logical event (AC-2), while the per-conn `env.ID` (`e.nextID`) still
  increments per-conn-per-envelope and resets each reconnect ‚Äî distinct cursors,
  same envelope.
- **`docs/protocol-mobile.md` is the wire-spec home for the new field.** Per the
  architect constraint the developer's worktree touched only code/tests/spec; the
  wire-spec doc update was deferred to this documentation phase. `event_id` is now
  documented on the Interactive-events section of `docs/protocol-mobile.md` and on
  the application-envelope shape (envelope-level, optional, present only on
  interactive structured-stream frames).

## Patterns established

- **Additive `omitempty` pointer field keeps every other construction site
  byte-identical for free.** Choosing a field on `protocol.Envelope` (over a
  separate replay frame) means the ~13 production + ~18 test
  `protocol.Envelope{...}` sites (`internal/dispatch`, `internal/relay`,
  `cmd/pyry/assistant_turn*.go`, ‚Ä¶) leave `EventID` nil ‚Üí omitted ‚Üí byte-identical.
  AC-4's "v1 frame and non-interactive frame round-trip byte-identically" falls out
  with **zero edits** to those sites. A pointer (not a plain `uint64`) is the lever:
  ring ids are always ‚â•1, so a non-nil pointer never encodes `0` and a nil pointer
  is omitted ‚Äî "absent, not null/0". Same shape as `InReplyTo` and
  `RoutingEnvelope.{Token,CloseCode}`.
- **The load-bearing safety fact behind the shared `&eventID` is the mutex
  happens-before edge + pointee immutability, not "Push reads only".** Code review
  verified the aliasing against the **real** `V2SessionManager.Push` ‚Üí
  `drainOnce` ‚Üí `forwardEnvelope` path, not just the test double: `Push` enqueues
  the envelope under `m.pushMu` and it is drained/marshalled on a *different*
  goroutine, retaining `&eventID` across the handoff. The race-freedom holds
  because the write to `eventID` is sequenced-before the enqueue under `m.pushMu`
  (a happens-before edge the drain reads under the same mutex) and the pointee is a
  fresh per-`emit()` local written once and never mutated ‚Äî **not** merely because
  Push "reads only". The emitter still takes no lock; the single-`Run`-goroutine /
  unguarded-counter invariant ([codebase/632.md](632.md)) is unchanged.
- **Comment-as-contract: the wire-surface attribution moves with the producer.**
  `emit()`'s comment misattributed the wire surface to #647 (a pre-split
  artifact). The producer/consumer split made **#649** the producer, so the comment
  is corrected `#647` ‚Üí `#649` in the same commit that adds the behaviour ‚Äî merged
  code stops lying about who owns the surface. Same discipline applied to the
  [eventring-package.md](../features/eventring-package.md) feature doc here.
- **Durable `event_id` ‚âÝ per-conn envelope `id` ‚Äî surfaced, not merged.** The wire
  now carries both: `id` (per-conn counter, resets each reconnect) and `event_id`
  (durable per-conversation replay cursor). Keeping them as two distinct wire fields
  preserves the single-writer envelope-id invariant #646 established
  ([codebase/646.md](646.md) ¬ß Patterns) and is exactly what lets a phone use the
  latest `event_id` as a reconnect-stable `last_event_id`.

## Lessons learned

- **A commented field inserted mid-struct splits the gofmt alignment group.**
  Placing `EventID` (with its doc comment) between `InReplyTo` and
  `PayloadEncrypted` breaks the gofmt-aligned struct block into separate alignment
  runs: the new field + the field after it align independently from the run above.
  This is correct and intentional ‚Äî it mirrors the existing `RoutingEnvelope`
  `Token`/`CloseCode` shape, and `gofmt -l` confirms the file is clean. Don't
  "fix" the visual misalignment by re-tabbing across the comment; that fights
  gofmt. (From the PR body's Lessons learned.)

## Testing

`internal/protocol/envelope_test.go` ‚Äî `TestEnvelope_EventIDOmitempty` (the
omitempty unit test, AC-4): a nil `EventID` is absent from the marshalled bytes
(no `"event_id"`); a non-nil `EventID` encodes as `"event_id":<n>` and round-trips
to a pointer to the same value. The pre-existing `TestEnvelope_RoundTrip_Full/_Minimal`
and every `testdata/` fixture round-trip **byte-identically unchanged** ‚Äî they are
the other half of the AC-4 "absent on existing frames" proof (no fixture edits).

`cmd/pyry/interactive_turn_v2_test.go` (additions only) ‚Äî three emitter tests
extending the #646 ring block, reusing the established doubles
(`fakeInteractiveBcast`, `recordedPush`, `stubCursor`, `pushesFor`,
`e.ring.After`) plus a new `pushEventIDs` helper (fails on any nil wire id):
- **AC-1** `‚Ä¶_WireCarriesDurableEventID` ‚Äî every push carries a non-nil
  `env.EventID`, and the wire id sequence equals the ring id sequence from
  `e.ring.After(convID, 0)` (same order, same values).
- **AC-2** `‚Ä¶_WireEventIDIdenticalAcrossConns` ‚Äî with two interactive conns, for
  each logical event the `EventID` on conn "a" equals conn "b" (one ring id fanned
  to both), while the per-conn `env.ID` differs.
- **AC-3** `‚Ä¶_WireEventIDsStrictlyIncreasing` ‚Äî the `EventID` sequence on one
  conn's stream is strictly increasing and is exactly the `1..N` ring sequence, so
  the latest is a valid `last_event_id`.

`make check` (vet + `-race` + staticcheck) and `make build` green. **real-claude
e2e** clean pass (`internal/e2e/realclaude` ~150s, no flake). **Code review PASS** ‚Äî
no blocking findings; two NITs (mid-struct field placement and a comment that
slightly understated the real concurrency guarantee), both confirmed correct, no
change required (see Patterns established for the concurrency detail).

## Out of scope (deferred)

- **The reconnect consumer ‚Äî sibling [#647]** (`security-sensitive`, blocked on
  this slice): per-conn `last_event_id` in the phone's `hello`, the on-reconnect
  ring replay via `eventring.After`, and the resync marker. This slice only gives
  the phone an id to advertise; accepting it back and acting on it is the inbound
  trust boundary #647 owns.
- **The mobile consumer** ‚Äî pyrycode-mobile#369 (also drives #647's reconnect
  surface), which records `event_id` as its stream position and advertises it on
  reconnect.
