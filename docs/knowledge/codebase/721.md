# Ticket #721 ‚Äî route `send_message` through the inbound queue (daemon wiring)

The **consumer/wiring slice** that makes the [#704](704.md) inbound message queue
live ([EPIC #597] Phase 3 interactive, [ADR 025]). #704 shipped
`internal/msgqueue.Queue` **unwired** ‚Äî `msgqueue.New` was never called and
`send_message` still delivered synchronously via `Route ‚Üí Activate ‚Üí
WriteUserTurn`, blocking the per-conn goroutine while claude was mid-turn and
**failing** (retryable `server.binary_offline`) once the #594
`sendMessageDeliverTimeout` (30s) elapsed. #721 constructs one `msgqueue.Queue`
in the daemon bootstrap, runs it under the daemon lifecycle, and swaps the
handler from sync-deliver to **enqueue-and-ack**. A message typed mid-turn is now
queued and drained in order when the turn ends, instead of blocking or being lost
on a long turn. `security-sensitive` ‚Äî it wires untrusted phone-originated
`send_message` payloads into a daemon-resident backlog and changes the inbound
dispatch policy on an internet-exposed surface. 3 production files, ~70 net-new
production LOC, 1 new exported type (`handlers.Enqueuer`).

Key seams / contracts:

- **`handlers.Enqueuer`** ‚Äî `Enqueue(conversationID, text string) uint64`, satisfied
  verbatim by `*msgqueue.Queue`. The handler appends and acks; it no longer drives
  a `TurnWriter` directly.
- **`sessionRouter.resolve` (new) / `Route` (now a thin wrapper).** `resolve` is the
  old `Route` body **minus** the `r.active.set` stamp ‚Äî the single resolution
  authority, carrying the #678 AC#4 empty-`CurrentSessionID` guard (rejects before
  any `Pool.Lookup("")` bootstrap fallthrough). `Route` calls `resolve` then stamps
  the active-conversation cursor on success only. `SessionRouter` interface
  **unchanged** ‚Üí zero ripple to #678 tests / the e2e `routeTo` helper.
- **`newInboundDeliver(resolve func(string)(TurnWriter,error)) msgqueue.DeliverFunc`** ‚Äî
  the delivery seam, built over `router.resolve` (a func value, not the struct, so it
  is unit-testable with a fake). Per attempt it: re-`resolve`s the bound session
  (binding may change between enqueue and drain ‚Äî error ‚Üí drain retries the head);
  `Activate`s under `inboundActivateTimeout` (30s, a wedged respawn ‚Üí retryable
  error, not a permanent block); `WriteUserTurn`s with the **raw lifecycle ctx** ‚Äî no
  deliver timeout, because that block IS the drain's turn-end pacing.

Data flow (the handler is now fully synchronous and non-blocking; all delivery is
on the engine's per-conversation drain goroutine):

## The ack contract (AC#3) ‚Äî asymmetric by design

**The ack now means "accepted into the backlog," not "delivered/committed."** Which
failures still produce an error reply versus are absorbed by the drain is
deliberately asymmetric:

| When | Failure | Wire reply | Retryable | Enqueued? |
|---|---|---|---|---|
| **Sync (handler)** | payload won't decode | `protocol.malformed` | no | no |
| **Sync (handler)** | `Route` ‚Üí `ErrConversationNotFound` (unknown conv) | `conversation.not_found` | no | no |
| **Sync (handler)** | `Route` ‚Üí `errNoBoundSession` / `Pool.Lookup` miss (**unbound**, #678) | `server.binary_offline` | yes | no |
| **Sync (handler)** | `Route` OK ‚Üí `Enqueue` | **`ack`** | ‚Äî | yes |
| **Async (drain)** | resolve / Activate / WriteUserTurn error post-ack | none (retry head after 1s, warn) | ‚Äî | held |
| **Async (drain)** | ctx cancelled (daemon shutdown) | none (leave head queued) | ‚Äî | held, lost on daemon restart |

The rationale: **at enqueue we have a live phone to tell "retry"** (so malformed /
unknown / unbound all reject synchronously, preserving #678's "unbound ‚Üí error, not
bootstrap"); **once enqueued, the ack promised delivery**, so a transient unbind /
respawn / PTY error is held and retried by the drain rather than dropped. The #594
30s deliver cap is **gone** ‚Äî it existed only because the phone was blocked awaiting
an ack; the enqueue-ack is now immediate and delivery is async.

## Patterns established

- **Factor the side-effecting core out of a method when a second caller must skip the
  side effect.** `Route`'s `r.active.set` is the #679/#687 follow-active cursor stamp ‚Äî
  the phone-interaction signal driving the structured turn stream. Re-resolving via
  `Route` inside the drain would re-stamp the cursor at *drain* time (drain order, not
  phone-interaction order), corrupting the cursor. Splitting a stamp-free `resolve`
  core out (and making `Route` a one-line wrapper that layers the stamp on) keeps the
  cursor **single-writer** (the routing-path goroutine only) while giving the drain a
  bypass through the *same* resolution authority ‚Äî so neither path can skip the #678
  AC#4 isolation guard. Guarded by `TestSessionRouter_ResolveDoesNotStamp`.
- **The blocking delivery seam gets the raw lifecycle ctx, the bounded retry guard gets
  a budget.** `WriteUserTurn` is allowed to block for a whole claude turn (that block IS
  the pacing ‚Äî `DeliverFunc` must return nil only on a confirmed commit); only `Activate`
  is wrapped in a timeout, because a wedged respawn must become a *retryable error* the
  drain can re-attempt, not a permanent block. The two budgets the old synchronous
  handler shared (`sendMessageActivateTimeout` + `sendMessageDeliverTimeout`) collapse to
  one (`inboundActivateTimeout`).
- **Join the engine's lifecycle goroutine with the existing `ctrlDone` channel pattern.**
  `qDone := make(chan error, 1); go func() { qDone <- queue.Run(ctx) }()`, joined with
  `<-qDone` after `pool.Run` returns. The buffered channel + `defer cancel()` (every
  return path, including the early `startRelay`-error return before the join) lets the
  goroutine finish its send and exit cleanly ‚Äî no leak. On ctx-cancel `Run` stops
  spawning drains and `wg.Wait`s the in-flight ones.
- **Construction order:** build `router` (named) ‚Üí `msgqueue.New(... newInboundDeliver(
  router.resolve) ...)` ‚Üí spawn `Queue.Run` ‚Üí `startRelay(‚Ä¶, router, queue, ‚Ä¶)`. The
  queue must exist before the handler is registered; `Run` may start before or after
  registration (it sweeps any pre-existing backlog at Run time and spawns drains lazily
  on first enqueue ‚Äî no lost wakeup).

## Lessons learned

- **`sessionRouter.Route` had a non-obvious side effect that made the naive drain wrong.**
  The synchronous handler called `Route` exactly once per send, at delivery time (which
  was also send time), so the cursor stamp coincided with the phone interaction. Deferring
  delivery to a drain decouples those two moments: the *validation* must still stamp at
  send time (phone interaction), but the *delivery re-resolution* must NOT. Anyone wiring a
  deferred consumer onto a `Route`-style helper should check what the resolver mutates
  before reusing it on the deferred path. (Not flagged in the PR body; surfaced by the
  spec's "single non-obvious design decision" and confirmed in code review.)
- **Removing the deliver timeout is a feature, not a regression.** The #594
  `sendMessageDeliverTimeout` looks like a safety net but only existed to bound a
  *blocked phone*. Once delivery is async the phone is never blocked, so the cap is pure
  liability (it would convert a long-but-healthy turn into a spurious failure). The drain
  is *meant* to block for a whole turn ‚Äî `WaitReady`/`DeliverPrompt`/transcript-growth are
  all ctx-bounded, so shutdown still unblocks promptly.

## Out of scope (deferred)

- **Inbound queue bound / backpressure** ‚Äî the wiring makes the **unbounded inbound-queue
  growth** vector live: a paired phone flooding `send_message` during a long turn grows the
  per-conversation FIFO without bound (frames are ‚â§1 MiB by the WS read cap, but the
  *count* is uncapped). ADR 025 ¬ßBackpressure (line 130) bounds only the **outbound** push
  queue. Surfaced, **not pre-built** (evidence-based): the threat actor is a
  trusted-but-malicious/buggy *paired* device, blast radius is daemon memory, no deployed
  users, failure unobserved. The single insertion point is pre-specified at
  `msgqueue.Queue.Enqueue` (`queue.go:34-35`): add `MaxQueuedPerConversation` to `Config`,
  change `Enqueue` to `(uint64, error)`, return an `ErrQueueFull` sentinel before appending
  (**reject, never drop** ‚Äî preserves losslessness). **Recommend a PO follow-up ticket**
  (the code-review NIT also asks that this gap be tracked, not lost).
- **`queue_state` / `dequeue_message` wire types** ‚Äî #705 siblings (the
  reporting/removal surface; #720 landed the wire types). This slice is the prerequisite ‚Äî
  there is no live backlog to report or manage until inbound messages route through the
  queue.
- **Drain-time permanent unbind** ‚Äî if a conversation were deleted while messages are
  queued, the head would retry every 1s for the daemon's life. There is no
  conversation-delete verb today, so this is unreachable; the daemon-restart boundary
  bounds it. Note for whoever adds conversation deletion.

## Restart semantics (pinned)

The in-memory backlog **survives a claude child respawn** (the drain retries the head ‚Äî
`ErrNoLiveSession` is the retry trigger ‚Äî until the new child is live, then drains into
it) but **not a full daemon-process restart** (purely in-memory, same loss boundary as
`eventring`; resync covers it). No on-disk persistence in this slice.
