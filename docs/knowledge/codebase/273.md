# Ticket #273 — `internal/protocol`: v1 conversations-read payload structs

Phase 3 Track C — third slice of the #256 per-type payload catalog. Two pure DTOs (plus one row type) that slot into `Envelope.Payload (json.RawMessage)` once the future dispatcher reads `Envelope.Type`: `ListConversationsPayload` (empty body), `ConversationsPayload` (slice of `ConversationSummary`). Shapes are fixed verbatim by `docs/protocol-mobile.md` § Message types → `list_conversations` / `conversations`; the example payload in each subsection is the golden-file fixture. Sibling slices to #275 (`RegisterPushTokenPayload`) and #272 (messaging + backfill payloads).

## Implementation

- **One production file, one test file, two fixtures, stdlib-only.** `internal/protocol/conversations_read.go` (~35 LOC, three struct declarations + doc comments) + `internal/protocol/conversations_read_test.go` (~79 LOC, two round-trip tests) + two fixtures under `internal/protocol/testdata/` (`list_conversations.json`, `conversations.json`). The `TypeListConversations` / `TypeConversations` constants the tests consume already exist in `codes.go` from #255 — no constants added.
- **Three types in one file, by spec-grouping precedent.** `list_conversations` is a request, `conversations` is its paired response, and `ConversationSummary` is only referenced from `ConversationsPayload.Conversations`. Splitting across files would either duplicate the row type or force a cross-slice dependency for a single request/response pair. Same rationale used in #272's `messaging.go`.
- **`ConversationSummary.Name *string` WITHOUT `omitempty` — deliberate deviation from the AC body's literal wording.** Spec example for `conversations` shows literal `"name": null` on the wire (an unnamed scratch conversation). `*string` distinguishes "null on wire" (nil pointer) from "key absent" and "empty string"; `omitempty` is DROPPED because with it a nil pointer marshals as absent (key dropped), and without it a nil pointer marshals as literal `null` — byte-identical to the spec fixture. The AC body said "`*T` + `omitempty`"; reading the AC literally would silently break the byte-equivalent round-trip the *same AC* also requires. Spec wire shape wins over AC literal wording (see [lessons.md § Preserving JSON `null` on the wire](../../lessons.md#preserving-json-null-on-the-wire-t-without-omitempty-with-a-mandatory-why-comment), already pinned by sibling #272's `BackfillSincePayload.ConversationID`). A multi-line WHY comment on the field is mandatory and present — the only comment block on a field in `conversations_read.go` under the project's "default to no comments" rule — so a future contributor doesn't "fix" the missing `omitempty` and silently break the wire. `TestConversationsPayload_RoundTrip`'s byte-equal check is the regression detector.
- **Fixture carries both `Name` branches — the load-bearing test invariant.** `conversations.json` has two rows: row 0 with `"name": "kitchen-claw refactor"` (pointer-to-string round-trip), row 1 with `"name": null` (nil-pointer round-trip). Without the two-row fixture the `*string`-without-`omitempty` invariant would be unobservable. The test pins both branches explicitly: row 0 asserts `c0.Name != nil && *c0.Name == "kitchen-claw refactor"`; row 1 asserts `c1.Name == nil` (NOT `*c1.Name == ""` — would panic on nil deref AND be the wrong check). The byte-equal envelope check then proves both branches survive marshal: row 0 stays a JSON string, row 1 stays literal `null`.
- **`ConversationSummary` field declaration order matches fixture key order for byte-stable round-trip.** Go's `encoding/json` marshals struct fields in declaration order. Declaration order is `ID, Name, IsPromoted, Cwd, LastMessageTS, LastUsedAt` to match the fixture's per-row key order (`id, name, is_promoted, cwd, last_message_ts, last_used_at`). A future reorderer must update the fixture in lockstep.
- **`time.Time` round-trip needs no custom marshaller.** Spec uses `"2026-05-08T10:31:02Z"` — RFC 3339, no fractional seconds. `time.Time.MarshalJSON` uses `RFC3339Nano`, which omits a fractional component when none is present, so a parsed `2026-05-08T10:31:02Z` re-marshals byte-identically. The AC's mention of "RFC 3339 nano" is the *upper bound* of precision the wire carries, not a requirement to always emit fractional seconds. Same discipline as `Envelope.TS`, `BackfillSincePayload.SinceTS`.
- **`ListConversationsPayload` is an empty struct (`struct{}`).** Spec shows `{}` on the wire. The type exists so the dispatcher can decode into a concrete value rather than `json.RawMessage`; no fields. Round-trip is `{}` → `ListConversationsPayload{}` → `{}` byte-identical.
- **All other required fields are non-pointer, no `omitempty`.** `ConversationSummary.{ID, Cwd}` are required `string`; `IsPromoted` is required `bool` (the spec example shows both `true` and `false` values on the two rows — the canonical inverse-coverage); `LastMessageTS` / `LastUsedAt` are required `time.Time`. Empty / zero-valued fields surface as `""` / `false` / RFC 3339 zero-time on the wire; the dispatcher rejects malformed frames via shape validation.
- **Pure DTOs: no methods, no constructors, no `Validate()`.** Identical posture to #275 and #272. Required-field validation, ID uniqueness, ordering invariants (e.g. "MRU-first") — all dispatcher concerns. `ConversationsPayload` doc explicitly notes "Order of the Conversations slice is preserved from the wire — the binary is the source of truth for ordering; this type does not reorder."
- **Test reuses `canonical(t, b)` and `readFixture(t, name)` from `envelope_test.go`.** Same package, no import, no duplication. Two test functions (`TestListConversationsPayload_RoundTrip`, `TestConversationsPayload_RoundTrip`) follow the same shape as `TestEnvelope_RoundTrip_Full` and the #272 / #275 sibling-slice tests: unmarshal full envelope → assert `Type == Type<X>` (uses the constant, not a hardcoded string) → second-pass unmarshal `Envelope.Payload` → spot-assert fields → re-marshal envelope → `bytes.Equal(canonical(out), canonical(raw))`.
- **`conversations.json` envelope carries `in_reply_to: 3` (matching spec example "response references the request ID 3").** First fixture in the protocol corpus where `in_reply_to` rides alongside an array-carrying payload; the byte-equal check pins both the envelope omitempty discipline and the payload shape in one assertion. `list_conversations.json` has no `in_reply_to` (it IS the request).

## Out of scope (deferred)

- The other 10 v1 payload structs (`hello` / `hello_ack` / `error` / `ack` for #271 handshake-and-control; `create_conversation` / `conversation_created` / `promote_conversation` / `conversation_updated` for the conversations-write slice). Each lands in its own ticket with its own file in this package.
- The future `list_conversations` dispatch handler — reads `internal/conversations.Registry`, builds the response slice (likely MRU-first by `LastUsedAt`), maps each `Conversation` row to a `ConversationSummary`, sends a `conversations` envelope with `in_reply_to` set to the request id. Future ticket; registry-to-payload mapping is not this package's concern.
- A `Conversations.Sort()` / `SortMRU()` helper or any ordering predicate — `ConversationsPayload` is wire-typed, not behaviour-typed; ordering is the producer's (binary) responsibility.
- Pagination / cursor fields on `ConversationsPayload` — spec v1 does not paginate; the binary sends the full list in one envelope. Pagination would be an additive v1 change (new optional fields) or a v2 envelope, both reserved.
- A `ConversationSummary.IsArchived` / `ArchivedAt` field — auto-archive (#219/#237) removes archived rows from the registry rather than tombstoning them, so they never appear in this response. If a future feature retains tombstones, the field is additive.
- Validation that `IsPromoted == true` implies `Name != nil` — semantic invariant the dispatcher / registry enforces (`internal/conversations.Registry.Promote` already gates on non-empty name and uniqueness), not a wire shape concern.
- A `json.Marshaler` / `json.Unmarshaler` custom implementation for `ConversationSummary.Name` — the default struct-tag-driven implementation correctly round-trips `null` ↔ nil-pointer with `*string` + no `omitempty`. Custom marshallers would re-open the subtlety inside Go code.
