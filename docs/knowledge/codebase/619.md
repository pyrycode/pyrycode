# Ticket #619 — bump tui-driver to v1.3.0 (`Events` gains a `Tracker` param)

Pins `github.com/pyrycode/tui-driver` from **v1.2.0 → v1.3.0** to unpark [#615](https://github.com/pyrycode/pyrycode/issues/615) (Phase 2 event-stream bridge, EPIC #596 / [ADR-025](../decisions/025-mobile-remote-head-interactive-session.md)), which maps tui-driver `JSONLEntry` tool payloads into the neutral internal turn-event model via `ParseToolUse` (tui-driver#140) and `ParseToolResult` (tui-driver#142) — helpers that exist only from v1.3.0. #615 could not compile against the v1.2.0 pin, which is why it was parked. This ticket does the minimum to unpark it: the version pin plus the one adaptation v1.3.0's lone breaking change forces. Behaviour-preserving — not `security-sensitive` (the `Events` stream parses the locally-supervised claude child's own JSONL, trusted local input; the internet-exposed surface lives downstream in #615 and [#618](618.md)).

## Implementation

- **Pin bump (2 lines).** `go.mod:11` moves `github.com/pyrycode/tui-driver` to `v1.3.0`; `go.sum` rewrites the two matching lines. Only the `h1:` zip hash changed — the `/go.mod` hash is **unchanged** across v1.2.0→v1.3.0 because tui-driver's own `go.mod` didn't change between the tags. Clean released tag: no `replace` directive, no pseudo-version, no local checkout (the spec forbade all three).

- **`Events` call adaptation (1 line + comment).** v1.3.0's only breaking change (tui-driver#141): `(*Session).Events` gained a required trailing `tr *Tracker` parameter for the `stall_detected` degrade marker —
  ```go
  func (s *Session) Events(ctx, jsonlPath, startOffset int64, tr *Tracker) (<-chan Event, error)
  ```
  A `nil` tracker **panics** on first dereference inside tui-driver. The repo's **only** `Session.Events` call site is `internal/agentrun/ptyrunner/runner.go:505`; the fix passes the watchdog `tracker` already constructed at `runner.go:467` (`tuidriver.NewTracker(cfg.WatchdogTrackerOpts)`, always non-nil), not a fresh one and not `nil`:
  - before: `sess.Events(runCtx, jsonlPath, 0)`
  - after:  `sess.Events(runCtx, jsonlPath, 0, tracker)`

  A half-line inline comment at the call site notes the tracker is the same one the watchdog uses and that v1.3.0 requires it non-nil.

- **No new concurrency.** The `tracker` was *already* shared across three goroutines in the v1.2.0 code: the main `Run` goroutine (`RecordTransition("prompt-written")`, 468), the budget `Terminate` hook (`RecordTransition("budget-hit")`, 474, fired from the budget Counter's timer goroutine), and the watchdog goroutine (`runWatchdog(...)`, 492). `*tuidriver.Tracker` is therefore already required to be concurrency-safe. Passing it to `Events` adds tui-driver's internal Events goroutine as one more accessor of an already-concurrent-safe type — no new sharing pattern. The existing `go test -race` covers the new accessor for free.

- **No behaviour change — `EventKindStallDetected` falls through.** v1.3.0 also adds an additive `EventKindStallDetected` constant. ptyrunner's `switch ev.Kind` (`runner.go:516`) has **no `default` arm**, so the new kind is silently dropped — identical to how any unhandled kind behaves today. This is the load-bearing structure behind the "no behaviour change" AC; the no-`default`-arm shape was a deliberate choice in [#513](513.md) (explicit no-op over `default: continue`). Adopting stall handling is explicitly deferred (evidence-based — no agent-run stall has been observed; see Out of scope).

- **No new test.** Verified by the existing suite unchanged — `go build ./...`, `go vet ./...`, `go test -race ./...` all green (per PR [#620](https://github.com/pyrycode/pyrycode/pull/620)). The ptyrunner tests drive `Run`, which constructs the real tracker and calls `Events` on the live path, so the new signature and the non-nil tracker are both exercised end-to-end with no test edit. No `_test.go` references `Session.Events` directly.

Net production diff: **6 insertions / 4 deletions** across `go.mod` + `go.sum` + `runner.go`. No new files, no new exported symbols, no new imports.

## Patterns established

- **Established `chore(deps): bump tui-driver to vX` ladder, this rung.** Follows #601 → v1.2.0 (`Session.Resize`), #599 → v1.1.0 (sealed mirror), #564 → v1.0.1 (short-prompt stall fix). The recurring shape: bump the pin via `go get …@vX && go mod tidy`, then absorb whatever lone breaking change the new tag forces at the single in-tree call site — never work around an unresolvable tag with `replace`/pseudo-version/local checkout (that's a release/environment blocker to escalate, not a code workaround). This is the first bump ticket to also land a `codebase/<N>.md` (the prior three predate the per-ticket doc phase).

- **Adding a required parameter is wire-compatible when the only value in scope is the correct one.** The breaking signature change was absorbed in one line precisely because the watchdog `tracker` was already live and in lexical scope at the call site — the AC's "never `nil`" constraint had exactly one satisfying value to pass. Same shape as [#513](513.md)'s observation that a strict expansion is migration-free when the consumer only depends on identity, not context.

## Lessons learned

- **A private-org dep-version bump cannot have its real-claude e2e certified in a credential-less sandbox — the e2e harness must build `pyry` with the real `HOME`.** Code review's diff judgment passed cleanly (zero MUST/SHOULD/NIT findings), but `make e2e-realclaude` **could not run**: every test died at the harness's internal `go build pyry` step with `git ls-remote … github.com/pyrycode/tui-driver … exit status 128` — the sandbox could not fetch the newly-pinned **private** `tui-driver v1.3.0` (no git credentials / no `GOPRIVATE` resolution in that build environment). Per the escalation contract this was routed `error:code-review` (operator review), **not** `needs-rework:developer` — the developer cannot fix an environment blocker, and routing rework would loop. The blocker was resolved out-of-band by **PR #623 / `8da261a` ("fix(e2e): build pyry with the real HOME so private deps resolve")**, after which this branch merged main and advanced to documentation. **Why it matters:** the same private-org-fetch failure first surfaced in [#471](471.md) (the `GOPRIVATE=github.com/pyrycode/*` lesson) and again in [#588](588.md) (sandbox couldn't fetch `tui-driver v1.0.1`); #623 is the structural fix on the harness side. **How to apply:** any `tui-driver` (or other pyrycode-org private dep) version bump will re-trip a credential-less e2e build until the build inherits the real `HOME`/`GOPRIVATE`; expect the e2e column to need a credentialed environment and don't mistake the fetch failure for a code defect.

## Out of scope (deferred)

- **Stall adoption in ptyrunner.** v1.3.0's `EventKindStallDetected` (the `stall_detected` degrade marker) is intentionally unhandled — it falls through the `switch ev.Kind` with no `default` arm. Adopting a stall response is deferred until a real agent-run stall is observed (evidence-based-fix principle). When a ticket does adopt it, the natural home is a new arm in the `runner.go:516` switch, mirroring the existing `*Shown` sentinel arms from [#513](513.md).
- **The #615 producer caller.** The supervisor / Phase 2 producer does not yet call `Events` — that arrives with #615, which writes its new caller directly against the v1.3.0 signature (it does not reuse ptyrunner's call site).
