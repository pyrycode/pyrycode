# Ticket #687 — structured turn stream stamps and follows the active-conversation cursor

The first half of the **outbound** per-conversation routing that [#678](678.md) deferred (peeled prerequisite of [#679](https://github.com/pyrycode/pyrycode/issues/679); part of the EPIC [#672](https://github.com/pyrycode/pyrycode/issues/672) decomposition). Since #678 routes an inbound `send_message` to the conversation's **bound-session** supervisor via `sessionRouter.Route`, routed turns commit on bound-session supervisors and never touch the **bootstrap** supervisor's conversation cursor. But the outbound structured turn stream still read that bootstrap cursor (`convID := e.sup.CurrentConversation()`, `interactive_turn_v2.go:131`) and **drops every event on an empty cursor** — so after the first per-conversation route the bootstrap cursor stayed empty and the structured reply stream went silent. #687 introduces the missing signal ("the conversation currently being interacted with") and re-keys the structured stream's two cursor readers to it, so the stream emits again and each envelope carries the routed conversation's `conversation_id`.

This ticket fixes **attribution + the drop**. It does **not** change which transcript the producer tails (`resolveLatestSessionJSONL` stays recency-resolved) — that is #679, which `blocked-by` this ticket.

See the evergreen feature doc: [features/conversation-session-binding.md § Edge cases & limitations](../features/conversation-session-binding.md#edge-cases--limitations) (the outbound asymmetry this partially closes).

## Implementation

5 files (3 production, 2 test); +213/-6. No new exported type; no `internal/relay` change.

- **New `activeConversation` holder** (`cmd/pyry/main.go`, beside `sessionRouter`/`boundSession`/`sessionMinter`/`poolResolver` — the `cmd/pyry` adapter family). A `sync.Mutex`-guarded `string` with `set(id)` and `CurrentConversation() string`. It holds the id of the conversation the operator is currently interacting with — the one most recently resolved by `sessionRouter.Route`. Zero value `""` is the well-defined "no conversation routed yet" state the emitter drops on (AC#4). The developer chose the mutex over `atomic.Value` (both were sanctioned by the spec); it mirrors the supervisor's own `convMu`+`currentConvID` cursor verbatim.
- **`Route` stamps on success only** (`main.go:691`). After `pool.Lookup` resolves the bound session and before `return boundSession{…}`, `Route` calls `r.active.set(conversationID)`. No reject path stamps it — unknown conversation (`ErrConversationNotFound`), empty binding (`errNoBoundSession`), or dangling id (`ErrSessionNotFound`) — so a failed route never moves the cursor.
- **Pointer field on the value-receiver router.** `sessionRouter` gained an `active *activeConversation` field. `Route` has a **value receiver** behind the `handlers.SessionRouter` interface, so the field must be a pointer — the copy `Route` operates on must write the *one* holder the emitter reads. (Same constraint shape as #678's adapter-by-value seam.)
- **One holder, two references** (`runSupervisor`, `main.go:583`). `runSupervisor` constructs a single `active := &activeConversation{}` and shares it two ways: (1) writer side — stored on the `sessionRouter{…, active: active}` literal; (2) reader side — passed as a new `*activeConversation` param down `startRelay` → `startRelayV2` → `startInteractiveTurnStreamV2`. Explicit param threading (not extracting the holder off the `handlers.SessionRouter` interface, which only exposes `Route`) keeps the dependency visible and avoids a type assertion. Each of the three signatures gained one arg; each has exactly one caller.
- **Both structured-stream readers re-keyed** (`interactive_turn_stream_v2.go`). The live emitter is now built with the holder as its cursor — `newInteractiveTurnEmitterV2(active, mgr, logger)` (was `sup`) — and the #647 reconnect-replay source follows the same signal — `mgr.SetReplaySource(emitter.ring, active.CurrentConversation)` (was `sup.CurrentConversation`). `sup *supervisor.Supervisor` stays a param: `NewSessionSubscriber(sup, …)` still subscribes over the supervised session. The `:131` read site does **not** change — it reads through the `cursorReader` interface (`assistant_turn.go:24`), so re-keying is a pure injection swap at the constructor call.
- **Tests.** New `cmd/pyry/active_conversation_test.go`: zero-value→`""`, `set` overwrites (last-writer-wins), an AC#3 `-race` concurrent set/get probe (every observed value is one that was set or `""`, never a torn read), and an emitter-stamps-cursor injection test (a `set` holder emits with that `conversation_id`; an unstamped holder drops every event). `session_router_test.go` extended: each subtest gets a fresh router with its own holder (order-independence), the success subtest asserts the cursor is stamped, every reject subtest asserts it stays `""`.

## Patterns established

- **Re-key a cursor by injection, not by editing the read site.** The emitter already read its cursor through the narrow `cursorReader` interface (`interface { CurrentConversation() string }`), injected at the constructor. So swapping *which* cursor it reads is a one-arg change at the construction site (`newInteractiveTurnEmitterV2(active, …)`) — `interactive_turn_v2.go:131` is untouched, and the 30+ emitter unit tests that inject a `cursorReader` stub are unaffected. The PO/architect verified this seam up front; it is what dropped the size from S to XS.
- **A `cursorReader` cursor has more than one reader — re-key all of them.** The structured stream reads the conversation cursor in two places: the live emitter (`interactive_turn_v2.go:131`) and the #647 reconnect-replay source (`interactive_turn_stream_v2.go:60`, `SetReplaySource`). Re-keying only the live reader would re-introduce the exact empty-cursor drop this ticket fixes — on the reconnect-replay path instead of the live path, so replayed envelopes would drop or carry the wrong attribution. Both were re-keyed to the same signal so live-emit and replay attribution stay consistent. (Foreseen by [[po-cursor-rekey-has-second-replay-reader]].)
- **A guarded holder absorbs a cross-goroutine hand-off so the rest of the emitter stays lock-free.** The emitter runs on the producer's single Run goroutine and keeps deliberately-unguarded counters (`interactive_turn_v2.go:42-44`); the routing path that writes the cursor is a different goroutine. Rather than add synchronization throughout the emitter, the *one* shared field lives behind a leaf mutex in the holder — it is the only new synchronization, and nothing else in the emitter has to change. This is the cost that held the ticket at a real (if small) increment rather than a trivial field swap.

## Lessons learned

- **The holder is read on three distinct goroutines, not two — and the mutex is right precisely because of that.** The code comment (and the spec) framed the readers as "the producer's single Run goroutine (live emit + replay)". The code-review NIT corrected the detail: the live emitter reads on the producer's Run goroutine, but the #647 replay reader (`cursor()` in `V2SessionManager.replayMissed`, `internal/relay/v2session.go`) runs on the **manager's** Run goroutine — a third goroutine (writer = dispatch path, live reader = producer Run, replay reader = manager Run). The mutex makes the holder safe regardless of how many goroutines touch it, so this was a doc-accuracy nit, not a correctness bug — if anything it *strengthens* the case for a mutex over an unsynchronized field. The general lesson: when a cursor's `func() string` is handed to another package's manager, count the manager's own goroutine as a reader too; "single Run goroutine" reasoning that holds for the emitter does not transfer to a cursor shared across package boundaries.
- **Re-keying the cursor un-breaks delivery without changing the broadcast model.** The structured stream still broadcasts to all interactive conns by capability with no conn→conversation binding for output, keyed off one global cursor. #687 restores a single global cursor that behaves for attribution exactly as the pre-#678 bootstrap cursor did (every turn stamped one cursor); #678 had emptied that cursor (breaking delivery, not leaking). Multi-operator cross-conversation isolation remains undesigned and is owned by #679 (content follows the bound transcript) + a future conn→conversation subscription — not regressed here.

## Out of scope (deferred)

- **Which transcript the producer tails** — still recency-resolved `resolveLatestSessionJSONL`. In the single-operator case the active bound session is also the most-recent JSONL writer, so attribution (this ticket) and content already agree; the cross-conversation hazard (a *different* session writing more recently) is what #679 closes. #679 is `blocked-by` this ticket.
- **The v1/v2 coarse bridges** (`assistant_turn.go:102`, `assistant_turn_v2.go:110`) read the same empty bootstrap cursor post-#678 and would also go silent for a routed conversation. They serve the *coarse* / non-interactive surface (#589/#634), not the operator-facing structured stream this ticket targets, and are left unchanged. The `activeConversation` holder is directly reusable as their cursor if that surface needs the same fix — file a follow-up.
