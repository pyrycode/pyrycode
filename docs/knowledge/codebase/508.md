# Ticket #508 ‚Äî delete `agentrun.EncodeProjectDir`; drop the redundant `EvalSymlinks` from the JSONL hot path

Foundational cleanup that completes the [#501](501.md) delegation: the `agentrun.EncodeProjectDir` wrapper is removed and the `internal/agentrun/jsonl/tail/watcher.go` `New` constructor stops calling `agentrun.ResolveWorkdir` upstream of `tuidriver.SessionJSONLPath`. Net: one canonicalisation per call, single source of truth inside [`tui-driver/pkg/tuidriver.EncodeCwd`](https://github.com/pyrycode/tui-driver) (darwin: `F_GETPATH`; linux: `EvalSymlinks`). All seven call sites (one production, six tests) mechanically switch to `tuidriver.SessionJSONLPath(home, workdir, sessionID)` (or `tuidriver.EncodeCwd(workdir)` at the ptyrunner test seam). `ResolveWorkdir` stays ‚Äî its sole remaining caller is [`internal/agentrun/trust`](../features/agentrun-trust-subpackage.md), tracked for follow-up removal in [#516](https://github.com/pyrycode/pyrycode/issues/516). Diff: +206/‚àí163 LOC across 8 files; per-byte the production change is one block in `watcher.go` and one deleted wrapper in `workdir.go`. Sibling [#509](509.md) landed first by chance ‚Äî its delete of the same `watcher.go:115` line is now a no-op from #508's perspective, exactly as the architect's "land in any order" framing predicted.

## Patterns established

- **When a library helper's internal-canonicalisation contract is documented as "silent fallback on resolution failure," that fallback IS the spec ‚Äî don't bolt your own fail-closed wrapper on top.** [#509](509.md)'s codebase doc argued the opposite: it identified `tuidriver.EncodeCwd`'s silent-fallback behaviour as a reason to **keep** `agentrun.ResolveWorkdir` upstream so a typo'd workdir would surface as `tail: resolve workdir: <fs.ErrNotExist>` instead of silently watching a wrong dir. Reading that doc made the silent-fallback look like a hazard; reading the actual call site (`watcher.go` ‚Üí `ptyrunner/runner.go:361`) revealed the upstream caller had already guaranteed the workdir exists: `ptyrunner` invokes `cmd.Process` with `Dir: cfg.WorkDir` before `tail.New(cfg)` is reached, and `cmd.Start` fails before the watcher constructor runs on a non-existent dir. The "fail closed" arm in `ResolveWorkdir` was defending against a scenario the upstream caller already prevented. Per CLAUDE.md "Evidence-Based Fix Selection ‚Äî Don't ship a defense for a failure mode that hasn't been observed": no observed bug warranted the dual layer; the per-CLAUDE.md threshold for keeping the eager check was not met. Generalises: when a sibling doc identifies "library does X silently on failure, our wrapper does Y loudly" as a reason to keep the wrapper, **trace the actual failure surface upstream** ‚Äî if the failure cannot reach the library call site, the loud wrapper is dead defence and its survival costs you the second canonicalisation forever.

- **Two-source-of-truth canonicalisers are an invisible drift trap until they aren't.** Pre-#501 there were two encoders (`agentrun` and `tuidriver`) ‚Äî #501 made `agentrun` delegate. But the delegation chain `EncodeProjectDir ‚Üí ResolveWorkdir ‚Üí EncodeCwd ‚Üí canonicalisePath` resolved symlinks twice on linux and did `EvalSymlinks`-then-`F_GETPATH` on darwin. Different OS, different perf cost, same shape of waste. The drift was invisible because both layers happened to return the same string for any input the test suite covered ‚Äî until the day someone profiled the JSONL hot path or audited the call graph for `EvalSymlinks`-coupling. #508 collapses both. Generalises: when a delegation lands (#501-shape) but retains the upstream caller as a "safety net" (the [#509](509.md)-shape rationale), schedule a follow-up to audit whether the upstream call is still semantically distinct from the delegated call. If they collapse to the same operation, the safety net is the cost of a syscall and the inversion of the "single source of truth" property the delegation was meant to establish.

- **Per-ticket codebase doc lessons are recommendations to the next contributor, not load-bearing decisions.** [#509](509.md)'s ¬ß Patterns established argued strongly for keeping `ResolveWorkdir`: "Two reasons to keep a call upstream of a delegation ‚Äî only one was in the spec; the second emerged in code review and is the more durable one. Generalises whenever a downstream library helper trades silent fallback for ergonomics: audit *both* the semantic invariant and the failure surface before deleting the upstream caller." #508 inverts that recommendation a week later, based on a deeper audit (upstream-caller pre-checks the existence anyway). Both docs are correct at the moment they were written; the cumulative truth is "audit the failure surface AND the upstream pre-checks." Recording this because the per-ticket convention's value is high-fidelity moment-in-time context, NOT a stable rule set ‚Äî future contributors should treat earlier per-ticket recommendations as inputs to their own audit, not as proscriptions.

## Lessons learned

- **The "load-bearing `ResolveWorkdir`" finding in [#509](509.md) was incomplete.** [#509](509.md) recorded that retaining `ResolveWorkdir` upstream of `tuidriver.SessionJSONLPath` was load-bearing for two reasons: (1) the macOS realpath invariant (`/var/folders/...` ‚Üí `/private/var/folders/...`), and (2) the fail-closed error contract on missing workdirs. #508's spec re-audited and found both reasons collapsed: (1) `tuidriver.EncodeCwd` already canonicalises internally via `canonicalisePath` (darwin `F_GETPATH` / linux `EvalSymlinks`), so the realpath invariant is preserved by the library; (2) the workdir's existence is structurally guaranteed by the upstream `ptyrunner` caller (which `cmd.Start`s with `Dir: cfg.WorkDir` before `tail.New` is reached), so the "fail closed on missing workdir" check was defending against an impossible state. The lesson is recorded here, not in the spec, because the architect's spec already noted the contract correctly; the **gap** was between [#509](509.md)'s doc-author and the actual `EncodeCwd` source. Read the source of the delegated function before recording that the upstream wrapper is load-bearing; doc-comment cross-references are not a substitute.

- **The "stable test name" guidance also applied: `TestEncodeProjectDir_*` deletion was lossless because `tui-driver/pkg/tuidriver/cwd_test.go` owns the same coverage.** When deleting tests across a refactor, audit whether the deleted assertions have an equivalent in the downstream package. Here the darwin-realpath, literal-substitution, non-alnum-bytes, dot-in-path, and missing-path cases each have a counterpart inside `tuidriver.canonicalisePath` + `tuidriver.EncodeCwd`'s own test files. No coverage gap from the deletion; re-asserting in pyrycode would have been double-bookkeeping prone to drift. Generalises whenever delegating: count the downstream coverage before deleting upstream tests, not after.

- **Sibling-ticket ordering "land in any order" turned out to be exactly right.** PO's framing of #508 and [#509](509.md) as overlapping at `watcher.go:115` predicted that whichever landed second would see the line already deleted by the first. [#509](509.md) landed first; #508's diff at that line was a no-op (line was already gone). The other side ‚Äî #508's deletion of the explicit `ResolveWorkdir` call upstream ‚Äî is similarly invisible to [#509](509.md), because [#509](509.md)'s test seam (`expectedEncodedDir`) was already updated to use `tuidriver.SessionJSONLPath`. No coordination overhead, no merge conflict, exactly as the PO framing predicted. Recording for the next sibling-ticket pair: when two tickets touch the same line and both reach the same end state (deletion or identical rewrite), the natural answer is "land in any order" rather than scheduling sequential.
