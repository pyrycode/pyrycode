# Ticket #680 — per-discussion claude sessions participate in idle-eviction + the active cap

The capstone slice of EPIC #672 ("per-conversation sessions") that closes **Phase 2.0**: *first-message lazy bind makes idle eviction load-bearing — daemon RAM scales with the number of **active** discussions, not the total ever created.* #677 made `create_conversation` eagerly mint + bind a dedicated session; #678 routed `send_message` through that bound session's cap-enforcing `Pool.Activate`. #680 **proves** the consequence: a phone-created discussion's bound session is a full citizen of the pre-existing `internal/sessions` idle-evict / active-cap machinery — it idle-evicts, reactivates on the next turn (resumed from its own `--session-id` JSONL), obeys `ActiveCap` with LRU victim selection, and never bleeds across discussions.

**Zero production diff.** The mechanism was already wired end-to-end; the only adapter-level gap was that no test drove the *phone path* (the `sessionMinter`/`sessionRouter`/`boundSession` adapters in `cmd/pyry`) through eviction. That gap is closed with tests, not code.

See the evergreen feature docs: [features/idle-eviction.md](../features/idle-eviction.md) (§ Status, § Testing) and [features/conversation-session-binding.md](../features/conversation-session-binding.md).

## Implementation

Single new file `internal/e2e/per_conversation_eviction_test.go` (build tag `e2e`, package `e2e`), composing existing package helpers (`waitForSessionState`, `assertActive`, `waitForBootstrap`, `shortHome`, `encodeWorkdir`, the fakephone/fakerelay/protocol builders) plus a few small file-local helpers. Two tests:

- **`TestE2E_PerConversation_IdleEvictsAndReactivates`** (AC#1, AC#2, AC#4 no-bleed). Harness: fakeclaude-TUI, `-pyry-idle-timeout=2s`, uncapped. Two `create_conversation`s bind **distinct** dedicated sessions (`boundA != boundB`, both non-empty — AC#4 binding distinctness); each idle-evicts (`waitForSessionState(... "evicted")`, AC#1); a `send_message(convA)` reactivates `boundA` and acks within 15s, then `waitForSessionState(boundA, "active")` (AC#2); `assertEvicted(boundB)` immediately after proves A's reactivation left B untouched (AC#4 — "churn in one discussion leaves another's untouched", "A's turn landed in A's session not B's").
- **`TestE2E_PerConversation_CapEvictsCrossDiscussion`** (AC#3, AC#4 bystander). Harness: fakeclaude-TUI, `-pyry-active-cap=2`, **no** idle timeout (so the only transitions are cap-driven → deterministic). Driven purely by `create_conversation` (each is itself a spawning `Pool.Activate` — the cleanest way to push past the cap). Create A → `{bootstrap, A}` at cap, no evict; create B → cap-evicts LRU peer = **bootstrap**; create C → cap-evicts LRU peer = **`boundA`, a per-conversation session** (the security-sensitive cross-discussion eviction the PO flagged). At each checkpoint the active count never exceeds 2; `assertDistinctIDs` + a re-read of each `current_session_id` pins that only the deliberate LRU victim transitioned and no binding was rebound.

New file-local helpers (PR notes them as reusable for future per-conversation phone-path e2e):

- `startPerConvHarness(t, home, sessionsDir, initialUUID, relayURL, extraFlags...)` — a **variadic-flag generalization** of `respawn_after_eviction_test.go`'s `startEvictionHarness` (which hardcodes `-pyry-idle-timeout`), so one caller passes `-pyry-idle-timeout=2s` and the other `-pyry-active-cap=2`. Same `spawnWith` + fakeclaude-TUI env + `waitForReady`. Kept local; the sibling `respawn_after_eviction_test.go` is **not** edited.
- `createConversationViaPhone(t, phone, reqID) → convID` — sends an all-null `CreateConversationPayload` (server defaults), drains racing `message`/spinner envelopes to `TypeConversationCreated`, returns the server-minted `.ID`. Returns only after the daemon has minted + bound + eagerly persisted the session (the reply is sent after the handler's `reg.Save`). **First e2e to drive `create_conversation` end-to-end over the v1 fakephone path.**
- `boundSessionID(t, convPath, convID) → string` — reads `conversations.json` via the real `conversations.Load`/`Registry.Get` types, returns `CurrentSessionID`, failing on empty (short poll absorbs cross-process visibility lag).
- `dialHelloPhone`, `assertEvicted` (the one-shot evicted-side counterpart of `assertActive`), `assertDistinctIDs`.

**Verification, not coverage of new behavior.** Each AC maps to a real observable that would fail on regression: AC#1 idle-evict (`waitForSessionState … "evicted"` would time out), AC#2 reactivation ack + `waitForSessionState(boundA,"active")` (pins the turn reactivated *A's* session, not the bootstrap — so AC#4's no-bleed isn't papered over), AC#3 cap-evict of a cross-discussion peer (would time out). `go test -race -tags e2e ./internal/e2e/ -run TestE2E_PerConversation -count=3` is green, no flakes.

## Patterns established

- **A pure verification slice can be the entire deliverable.** When the architect walks every AC against live code and finds the path already wired (#677 mint + #678 route + the pre-existing idle-evict/cap primitive), the ticket's value is *pinning* the invariants with tests, with zero production diff. "If you find yourself editing a production file to make an AC pass, stop" — the wiring isn't missing.
- **Generalize a single-flag harness helper into a variadic one for the second caller; don't mint a third public variant.** `startEvictionHarness` hardcoded the idle flag (#398's deliberate "next caller pays the refactor tax"). #680 is that caller: a file-local `startPerConvHarness(..., extraFlags ...string)` serves both the idle and cap tests without touching `harness.go` or the sibling test.
- **Pin isolation by *identity*, not content.** The AC#4 no-bleed / no-cross-delivery guarantee is proven with distinct-bound-UUID assertions + "only the LRU victim transitions" registry checks + reactivation-target checks — observables the harness *can* faithfully witness — rather than asserting per-file JSONL content the fakeclaude harness cannot produce (see Lessons).

## Lessons learned

- **fakeclaude derives its JSONL stem from `PYRY_FAKE_CLAUDE_INITIAL_UUID` (env), not the `--session-id` argv**, so every supervised child shares one JSONL file. Consequence: the e2e faithfully proves **lifecycle/routing scoping** (which session's *registry entry* transitions, which session *reactivates* on the next turn) but **not** per-file JSONL *content* scoping. The content-recall half of AC#2/#4 ("prior conversation intact (resumed)", "turns land in its own JSONL" *as content*) is realclaude's domain and is already covered by the existing #677/#678 realclaude round-trip e2e. No new realclaude/two-phone test was built — that harness does not exist (the structured-stream harness gap, [#603](603.md)). The lifecycle/routing observable is the faithful and sufficient one for what #680 asserts.
- **The reactivation ack is gated on `WaitReady` + `DeliverPrompt` even on the nil-resolver delivery path** (`supervisor.go` `deliverViaSession`, which every per-conversation session takes). Only a TUI-emitting child reaches idle, so the send_message test **must** use fakeclaude with `PYRY_FAKE_CLAUDE_TUI=1` — `sleep` would never produce the ack. Used for both tests for uniformity.
- **`create_conversation` is itself a spawning `Pool.Activate` operation**, so a sequence of bare creates is the cleanest deterministic way to push past `ActiveCap` and observe LRU victim selection — no need for the WaitReady/ack dance a send-driven cap test would add for no extra cap-policy coverage. Space the creates ~50ms so `lastActiveAt` ordering is deterministic for `pickLRUVictim`.
- **Bootstrap is cap-evictable at runtime** (Test B evicts it as the LRU peer). The bootstrap warm-start carve-out applies only at `Pool.New` load time ([ADR 016](../decisions/016-bootstrap-ignores-persisted-lifecycle-state.md)), not to runtime cap pressure — consistent with the design, not a bug.
- **Code-review NIT (non-blocking, left as-is):** `createConversationViaPhone` drains to `TypeConversationCreated`; if the daemon were to *reject* the create instead, `recvEnvelope` skips the reject and the test fails at the 15s timeout with a generic "did not receive" message rather than surfacing the reject reason. Diagnostic-only — the test still fails correctly.

## Security (label: `security-sensitive`)

Test-only, **no new attack surface / trust boundary / dispatch policy**. The label is correct because #680 makes phone-driven per-conversation sessions full cap citizens *in practice*, so the **cross-conversation cap eviction** interaction becomes observable (remote create/send activity in discussion C can force-evict discussion A's active session — the cap policy ignores `attached`). This is by-design and confidentiality-preserving: eviction acts on exactly the victim's own `--session-id`/JSONL (per-UUID registry transition), preserves the on-disk JSONL, and the victim re-activates intact on its next `send_message` — no session reads another's data. The three load-bearing isolation guards the argument rests on are confirmed in live code and **pinned** by the new tests (distinct bound UUIDs, only-the-LRU-victim transitions, reactivation targets the bound session) rather than papered over:

1. empty-`CurrentSessionID` rejection **before** any `Pool.Lookup` (`Lookup("")` returns the bootstrap — the confused-deputy hazard),
2. `boundSession.Activate` → `Pool.Activate` (full cap citizen),
3. Pool idle-timeout inheritance from `Config.IdleTimeout`.

No code-level defense was prescribed for the unobserved resource-exhaustion mode (a per-operator create quota remains a named #672-family follow-up); `ActiveCap` is the existing bound and #680 demonstrates it works. Review verdict: **PASS** (one NIT).
