# Ticket #318 — dispatch: per-conn auth slot on `*dispatch.Conn`

Adds the seam that downstream verb handlers will use to read the
authenticated device snapshot without re-validating the phone token on
every frame. The first-frame gate now threads the matched
`*devices.Device` from `relay.AuthenticateFirstFrame` through
`dispatch.FirstFrameOutcome` into a per-conn auth slot on
`*dispatch.Conn`, set exactly once on the accept-and-continue branch and
never on reject / `Err` / close-intent paths. No verb handler is
registered here; slice B (`register_push_token` rewrite — split from
#305) is the first consumer.

## Implementation

### `relay.AuthOutcome` (`internal/relay/auth.go`)

One new field, populated only on accept:

```go
type AuthOutcome struct {
    Response  protocol.RoutingEnvelope
    CloseConn bool
    Device    *devices.Device // accept-only; nil on reject / malformed
}
```

`AuthenticateFirstFrame`'s accept branch takes the `Device` value
returned by `reg.Validate`, copies it into a local `snapshot`, and
returns `&snapshot` in `AuthOutcome.Device`. The explicit copy keeps the
contract honest: `reg.Validate` already returned a value copy under
`reg.mu`, so the pointer the dispatcher holds is to a stable, point-in-
time view of the registry row — token revocation propagation to live
conns remains a separate concern (see "Out of scope" below). Reject
branches and the `ErrMalformedHelloFrame` early return leave `Device`
nil structurally — the literals never set it.

`device.Name` continues to be logged on accept; the new field does not
change log policy. The matched device is never logged on reject (name-
enumeration probes via binary logs stay defended).

### `dispatch.FirstFrameOutcome` (`internal/dispatch/dispatch.go`)

Symmetric one-field extension carrying the device from the gate closure
into the dispatcher:

```go
type FirstFrameOutcome struct {
    Response  protocol.RoutingEnvelope
    CloseConn bool
    Code      uint16
    Err       error
    Device    *devices.Device // populated iff Err == nil && !CloseConn
}
```

The contract is doc-pinned: `Device` is only consumed when both
`Err == nil` and `!CloseConn`. The dispatcher **MUST NOT** propagate
`Device` on the `Err` or close-intent branches — even a buggy gate that
fills the field there leaves `Conn.Auth()` nil. Defence-in-depth lives
on the consumer (dispatcher), not the producer (gate).

### `*dispatch.Conn` — the slot

```go
type Conn struct {
    id       string
    nextID   atomic.Uint64
    outbound chan<- protocol.RoutingEnvelope
    auth     *devices.Device // set once by setAuth on accept
}

func (c *Conn) Auth() *devices.Device { return c.auth }
func (c *Conn) setAuth(d *devices.Device) { c.auth = d }
```

No mutex, no atomic — the same single-writer-per-conn invariant that
justifies the existing `gateRun bool` field. The writer is `runGate` on
the per-conn goroutine, exactly once per conn, before any handler-table
dispatch. Readers are handlers running on the same per-conn goroutine,
sequentially after the write. Cross-goroutine reads (handler-spawned
worker goroutines) are happens-before-safe under Go's memory model:
goroutine start synchronises with prior writes on the spawning
goroutine. `go test -race ./...` is clean.

`setAuth` is **unexported** so verb handler closures cannot mutate auth
state from inside a handler — handlers can read the device through
`Auth()`, but the type system forbids them from forging one. This is
the threat-model-aligned design from the architect's security review.

### Dispatcher integration — `runGate` (`internal/dispatch/dispatch.go`)

The slot-write slots into the existing accept-and-continue branch,
alongside the per-conn id-counter advance from #308:

```go
resp := outcome.Response
if outcome.CloseConn {
    resp.CloseCode = outcome.Code
} else {
    // Accept-and-continue: populate the per-conn auth slot before any
    // handler runs. Close-intent and Err branches do not reach here.
    st.conn.setAuth(outcome.Device)
}
```

The `Err` branch returns earlier in `runGate` without reaching the
`setAuth` call site at all, so the slot stays nil there by construction.
The close-intent branch reaches the publish but skips the `else`.

### Gate-closure wiring (`cmd/pyry/relay.go`)

One field threaded through the existing literal:

```go
out := dispatch.FirstFrameOutcome{
    Response: outcome.Response,
    Device:   outcome.Device, // nil on reject; populated on accept
}
```

`relay.AuthOutcome.Device` is structurally nil on the reject path
(`AuthenticateFirstFrame` only sets it inside the accept branch), so
the close-intent `FirstFrameOutcome` carries a nil `Device` even though
the field is filled unconditionally here. The dispatcher's
"ignore-Device-when-CloseConn" guarantee makes the producer-side
posture safe either way.

## Test surface (`internal/dispatch/gate_test.go`)

Three new in-package tests, all under `-race`:

- **`TestConnAuth_NilBeforeGate`** — zero-value contract. Constructs
  `c := &Conn{id: "c-1"}` and asserts `c.Auth() == nil`. One-liner that
  pins the documented pre-accept state.
- **`TestFirstFrameGate_AcceptPopulatesAuth`** — accept-path device
  propagation. Builds a sentinel `&devices.Device`, returns it from a
  fake gate, registers a handler for `protocol.TypeSendMessage` that
  records `c.Auth()` into a channel, drives hello + sentinel-typed
  frame. Asserts the handler observes `c.Auth() == sentinel` (pointer
  equality — handlers see the same snapshot the gate handed in, not a
  copy).
- **`TestFirstFrameGate_CloseConnDoesNotPopulateAuth`** — close-intent
  invariant. Fake gate returns
  `FirstFrameOutcome{CloseConn: true, Code: 4401, Device: sentinel}` —
  a deliberately misbehaving gate that supplies a device on the close
  path. Asserts (a) the reject envelope is published with `CloseCode ==
  4401`, (b) the second frame is dropped (existing close-intent demux
  behaviour from #308), (c) no handler observes the sentinel device.

One assertion added to `internal/relay/auth_test.go`:

- `TestAuthenticateFirstFrame_ValidToken` — asserts `outcome.Device != nil`
  and `outcome.Device.Name == testDeviceName` on accept.
- `assertRejectOutcome` helper — asserts `outcome.Device == nil` on
  every reject-shape branch.
- `TestAuthenticateFirstFrame_MalformedHelloFrame` — asserts
  `outcome.Device == nil` on the malformed early-return.

## Patterns established / reinforced

- **Trusted-pointer carriers in the dispatcher's import graph.** The
  dispatcher gains an import edge to `internal/devices`. The
  type-erased alternative (`any` on `FirstFrameOutcome.Device` and
  `Conn.Auth() any`) was rejected: every verb handler would need a
  type-assertion on read, and the first handler author to forget the
  nil-check after the assertion would have a runtime panic on a
  non-obvious code path. Returning concrete `*devices.Device` lets the
  compiler enforce the contract and makes "trusted device snapshot"
  self-documenting at the call site. `internal/devices` is a leaf
  package (stdlib-only imports), so the new edge introduces no cycle
  risk — verified at build time.
- **Unexported seams for mutate-once-per-lifetime state.** `setAuth` is
  the second unexported `*Conn` mutator (after the implicit
  `outbound`/`nextID` initialisation in `getOrCreateConn`); like them,
  it relies on the same single-writer-per-conn invariant and lives in
  the same package as its sole caller (`runGate`). Verb handler
  closures can never reach it. This is the dispatcher's third instance
  of the "single-writer-per-conn invariant carries the field without
  synchronisation" pattern (after `gateRun`, `nextID`'s atomic-as-
  cheap-insurance).
- **Producer fills unconditionally, consumer guards by contract.** The
  gate closure in `cmd/pyry/relay.go` sets `Device: outcome.Device` on
  every `FirstFrameOutcome` it builds, without nil-checking
  `outcome.CloseConn` first — the structural guarantee that
  `relay.AuthOutcome.Device` is nil on reject paths makes that safe.
  The dispatcher's `runGate` then guards the consumption explicitly by
  only calling `setAuth` inside the accept-and-continue branch. Two
  defences, one structural and one explicit, for the same invariant.

## Lessons learned

### `&snapshot := device; &snapshot` over `&device`

`reg.Validate` returns `(Device, bool)` — a value copy. In the
`AuthenticateFirstFrame` accept branch, `device` is the local copy. The
implementation introduces a named `snapshot := device` and returns
`&snapshot` rather than `&device` directly. The rename is the entire
documentation: future readers see a variable called `snapshot` and
understand the pointer's lifetime is decoupled from registry mutations,
without needing to chase the `reg.Validate` signature. The compiled
output is identical; the readability win is worth the one extra line.

### The slot doesn't need atomic, but the next one might

This slice's slot is mutate-once-then-read-many: the writer is the per-
conn goroutine, the readers are handlers on the same goroutine. Future
state that crosses goroutines — e.g. a per-conn rate-limit counter,
last-seen timestamp updated on every outbound — would need `atomic.*`
or `sync.Mutex`. The doc-comment on `Conn.auth` calls out exactly which
invariant the field relies on, so a future field author who copy-
pastes the pattern without thinking will at least be primed to check.

### No dispatcher-side panic on "accept with nil device"

A hostile-by-bug gate closure that forgets to set `Device` on accept
would leave `c.Auth()` returning nil even after the gate has run. The
dispatcher does NOT defend against this with a panic or guard — verb
handlers must nil-check `c.Auth()` before dereferencing, and the
handler reviewer enforces the rule. Per pipeline principle "evidence-
based fix selection": no such buggy gate has shipped, so a defensive
check would be a defence for an unobserved failure mode. Slice B's
`register_push_token` handler is the first place the nil-check
discipline takes effect.

## Out of scope (deferred)

- **`register_push_token` dispatch.Handler rewrite + wiring** (slice B,
  split from #305). The first consumer of the seam.
- **Token revocation propagation to live conns.** A paired device whose
  token is revoked via `pyry pair revoke` mid-session still holds its
  `*devices.Device` pointer on `Conn.Auth()` until the WS closes. This
  is documented v1 semantic; an in-process reload primitive is a
  future hardening pass (same posture as #308's registry hot-reload
  deferral).
- **Handler-side mutation of the auth slot.** Forbidden by design
  (`setAuth` is unexported in the dispatcher package). If a future
  feature needs to update auth state mid-conn — e.g. resume a session
  on token rotation — the design will add a new exported method with
  its own contract, not relax the existing one.
