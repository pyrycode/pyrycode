# Ticket #647 — Inbound mid-turn replay: `hello.last_event_id` → ring replay / resync marker

The reconnect **consumer** of mid-turn replay ([EPIC #596] Phase 2, [ADR 025]
§ Backpressure / replay). A phone advertises where it left off via
`hello.last_event_id`; the daemon, on that connection, replays the current
conversation's events with `id > last_event_id` from the in-memory event ring
([#646]) **before** the live stream resumes — or, when the requested position
has aged out of the bounded ring, emits an honest **resync marker** telling the
phone to do a full reload. `last_event_id` is untrusted remote input:
range/shape-validated by the `*uint64` decode, ring-bounded, and scoped to the
daemon-resolved conversation (never one the phone names). `security-sensitive`
(inbound trust boundary that returns buffered conversation content). Split from
[#611]; consumes the [#646] ring + the [#649] outbound `event_id`. Mobile
consumer: pyrycode-mobile#369.

## ⚠️ Known issue — unresolved code-review MUST FIX (do NOT merge as-is)

**Status at the time this doc was written:** PR [#651] is **OPEN / not merged**
(`git merge-base --is-ancestor HEAD origin/main` → not an ancestor); the
code-review phase **never posted a PASS**. Its last comment (2026-06-08 16:21Z)
ended at "Ready for human review" with one **MUST FIX** outstanding, which the
sole post-implementation commit (`c8c98ba`, a testdata-JSON re-alignment) did
**not** address. The bug is present in the tree at HEAD. This documentation
records what shipped on the branch; it is **not** a sign-off.

**The defect — caught-up branch trusts the remote `last_event_id` as a
watermark, silently suppressing the live stream.** `replayMissed`
(`internal/relay/v2session.go:1484`) sets the per-conn dedup watermark from the
*untrusted* `afterID` even when the ring reports **caught-up** (no events to
replay):

```go
events, gap := ring.After(convID, afterID)
if gap { m.emitResync(ctx, s, convID); return }
s.replayThrough = afterID            // ← set even when events == nil (caught-up)
for _, ev := range events { … s.replayThrough = ev.ID }
```

paired with the guard in `forwardEnvelope` (`:1949`):

```go
if env.EventID != nil && *env.EventID <= s.replayThrough { return nil } // drops the live frame
```

Two silent-gap failure modes — the exact failure the feature exists to prevent:

1. **`/clear` + reconnect.** Phone saw `event_id=100` in conversation A. `/clear`
   rotates to conversation B; B's per-conversation ring counter restarts at 1.
   The phone reconnects advertising `last_event_id=100`. `cursor()` resolves to
   B, so `After(B, 100)` → **caught-up** (B's newest id ≪ 100) → no replay, and
   `replayThrough = 100`. Every live event in B (ids 1, 2, 3 … ≤ 100) is then
   **dropped by the guard** until B's counter passes 100. The architect's Open
   Question #1 assumed the phone would notice the rotation "via the
   `conversation_id` on the resumed live stream and resync itself" — but the
   watermark **drops those very frames**, defeating its own escape hatch.

2. **Hostile / absurd `last_event_id`.** Same conversation, no `/clear`. Phone
   advertises `last_event_id = 2^64-1` → caught-up → `replayThrough = 2^64-1` →
   **all** future live frames on that conn are dropped permanently. Blast radius
   is the conn's own stream (a paired-but-misbehaving phone silences itself), but
   it still contradicts AC-5's bounded-effect / "never left with a silent gap"
   guarantee.

The same-conversation *normal* caught-up case is fine (live ids are always
`> afterID`), which is why the unit suite is green — the defect only bites when
`afterID` is outside the current conversation's id space.

**Fix direction (for the developer + re-review; not applied here).** Do not set
the watermark from the untrusted `afterID`. The dedup race the watermark exists
for only involves events that are actually in the ring (the replay branch
already sets `replayThrough = ev.ID` per frame). The robust shape is to cap the
caught-up watermark to the conversation's **newest retained id**,
`min(afterID, newestRetained(convID))` — which requires the ring to expose its
newest id for a conversation (a small additive `eventring` accessor, or having
`After` also return it). That keeps the legitimate same-conversation dedup
(newestRetained == afterID there) while letting a rotated-away or hostile
`afterID` fall back to the live stream. This is a developer change + a fresh
security/code review, not a documentation edit. **Recommend: route #647 back to
`needs-rework:developer` before merge.**

Highlights specific to how it was built:

- **The late-bound `SetReplaySource` breaks the emitter↔manager construction
  cycle.** The emitter takes the manager as its `interactiveBroadcaster`; the
  replay path needs the emitter-owned `eventring.Ring` (`emitter.ring`,
  unexported, created **inside** the emitter constructor — [#646] made it
  emitter-owned to dodge the 20-call-site constructor cascade). Construction
  order is `NewV2SessionManager` → `go mgr.Run` → emitter built, so the ring does
  not exist when the manager is constructed. The ticket's literal hint ("wire the
  ring into `V2SessionConfig` like `Snapshotter`/`KnownConversation`") is
  therefore **not buildable** as a construction-time field without re-triggering
  the cascade #646 avoided. A setter called once during wiring is the only seam
  that keeps the change at 4 production files / S. (See [Patterns established].)
- **Replay/resync run inline on the manager's single `Run` goroutine.** The hook
  is at the very tail of `handleNoiseInit`'s success path — after `noise_resp` is
  sent (so the phone's recv CipherState exists), after `state == V2StateOpen` (so
  `forwardEnvelope`'s gate passes), after the push queue is created. Frames seal
  via `forwardEnvelope` (the established `handleRequestSnapshot` **inline control
  reply**, bypassing the buffered push stream). Because `Run` is single-threaded,
  the whole replay completes **before** `Run` returns to its select to service
  `drainCh` (live events) — the structural guarantee behind AC-2's "before the
  live stream resumes".
- **`After` is the [#646] 3-way classifier, consumed unchanged.** caught-up
  `(nil,false)` → set watermark, no frames; gap `(nil,true)` → one resync marker,
  **never** a partial gap-ful replay (AC-4); replay `(events,false)` → forward
  each ascending, each carrying `EventID: &id` (a per-iteration local, never
  `&ev.ID` of the range variable). The ring is self-synchronised by its own
  mutex, so the cross-goroutine read is safe off the emitter goroutine
  ([codebase/646.md] § Concurrency); the emitter keeps its lock-free
  single-`Run`-goroutine invariant.
- **The resync marker carries no named payload type.** `emitResync` marshals an
  inline `struct{ ConversationID string \`json:"conversation_id"\` }`, mirroring
  `emitRekeyRequest`'s payload-less inline-struct precedent — deliberately
  avoiding a third `protocol` file. It carries **no** `EventID` (it is not a
  structured event), so the watermark guard never touches it. `convID` is the
  daemon's own resolved id, never attacker-derived.

## Patterns established

- **Break a constructor cycle with a late-bound, leaf-lock-guarded setter, not a
  config field.** When B must be published to A but B is only constructed *after*
  A (and after `go A.Run`), a `V2SessionConfig`-style field is unbuildable; a
  one-shot setter (`SetReplaySource`) called during wiring, storing under the
  existing `pushMu` leaf lock (write off `Run` at wiring, read on `Run` at
  reconnect), is the seam — no new lock, no new ordering, no constructor cascade.
  The narrow window where a reconnect lands between `go mgr.Run` and the setter
  reads nil → that one reconnect gets the live stream only (benign), never a torn
  read. This is the inbound mirror of [#646]'s "emitter-owns-the-ring retires the
  constructor cascade".
- **Inline control reply via `forwardEnvelope`, on the `Run` goroutine, is the
  reusable "respond now, ahead of the buffered stream" primitive.**
  `handleRequestSnapshot` ([codebase/618]), `emitRekeyRequest`, and now
  `replayMissed`/`emitResync` all seal immediately on `Run` and bypass the
  `Push`/`drainOnce` queue. Use it whenever ordering against the live stream
  matters (here: replay must precede live).
- **A per-conn watermark is the deterministic fabric for replay/live dedup —
  *when computed from trusted data*.** The intended design (a Run-owned
  `replayThrough` + a pre-seal guard in `forwardEnvelope`) is the right "different
  fabric" complement to the phone's own `event_id` dedup ([ADR 025]
  belt-and-suspenders). **The shipped implementation mis-sources the watermark
  from the untrusted `afterID` in the caught-up branch — see [⚠️ Known issue].**
  The pattern is sound; the input is not. The lesson for future watermark/guard
  pairs: a dedup bound derived from remote input must be clamped to server-known
  reality (here, the ring's newest retained id), or the guard becomes a
  remote-triggerable mute switch.

## Lessons learned

- **A replay watermark sourced from untrusted client position can silently mute
  the live stream.** The caught-up branch trusting `afterID` is the [⚠️ Known
  issue] above. The general shape: an inbound cursor used as a *drop-everything-
  at-or-below* threshold must be range-bounded against server state, not just
  shape-validated by its type. `*uint64` decode stops non-integers, but a
  syntactically valid out-of-conversation-range id is just as dangerous as a
  malformed one for a "≤ watermark ⇒ drop" guard. Verify the guard against the
  *cross-conversation* and *hostile-large* cases, not only the happy
  same-conversation path (the green unit suite covered only the latter for the
  caught-up branch).
- **The repo is gofmt-dirty at HEAD under Go 1.26.2** (trailing-blank-at-EOF +
  alignment skew across many untouched files, e.g. `auth_test.go`,
  `v2session_test.go`). New code conforms to the **repo baseline**; do not
  reformat to the newer local gofmt or you spray spurious diffs across files you
  didn't touch and fight CI's pinned Go. (From the PR body's Lessons learned;
  matches the standing developer note.)

## Concurrency model

- **No new goroutine.** Replay + resync run inline on the manager's single `Run`
  goroutine inside `handleNoiseInit`, via `forwardEnvelope`. Run is
  single-threaded, so the replay completes before Run services `drainCh` /
  `m.snapshot` — the AC-2 ordering guarantee.
- **Cross-goroutine ring read is safe by fabric separation.** `ring.After` (on
  `Run`) and the emitter's `ring.Append` (on the producer goroutine) are
  serialised by the ring's own `sync.Mutex` ([codebase/646.md] § Concurrency).
  No new lock on the ring; the emitter's lock-free invariant is unchanged.
- **The replay-source publish is race-clean.** `SetReplaySource` writes
  `replayRing`/`replayCursor` under `pushMu`; `replayMissed` reads them under
  `pushMu`. `pushMu` is the existing leaf lock — no new ordering. Publish happens
  once during wiring; read happens much later at reconnect.
- **`replayThrough` is Run-owned.** Written in `replayMissed`, read in
  `forwardEnvelope`, both on `Run` — same single-writer regime as `s.state`. No
  atomic, no lock. (The *value* it is set to in the caught-up branch is the
  defect; the *concurrency* of the field is correct.)

`internal/relay/v2session_replay_test.go` (new) — drives a real Noise handshake
whose hello carries `last_event_id` against a manager whose `SetReplaySource` was
given a hand-populated `eventring.Ring` + a stub cursor, then decrypts the
forwarded frames and asserts: in-ring replay (3,4,5 ascending before any live
frame), caught-up + hostile-large (no frames), gap → single `resync` with
`conversation_id`, absent id (no replay), cursor-scoped (cursor→B, ring holds
A → zero A events — AC-5), replay-disabled (nil ring), other conns unaffected,
and the `forwardEnvelope` watermark guard (drops `EventID ≤ replayThrough`,
forwards above, never drops `EventID == nil`).

**Coverage gap that let the defect through:** the caught-up/scoped tests assert
*no replay frames*, but **not** that the conn's subsequent **live** stream still
flows after a caught-up/out-of-range `last_event_id`. A regression test for the
fix should reconnect with an out-of-conversation-range `last_event_id`, then push
a live event with a low `event_id` and assert it is **delivered** (not dropped by
the watermark). See [⚠️ Known issue].

`make check` (vet + `-race` + staticcheck + substrate-guard) and `make build`
reported green by QA; real-claude e2e's only red was the documented
`TestRealClaude_SigtermMidToolUse` timing flake (passed in isolation; no causal
path from this diff to signal/PTY/reaping). **Green tests are not the same as a
passing review here:** the judgment review found the
MUST FIX above, which no test exercises.

## Out of scope (deferred)

- **The `backfill_since` full-reload handler** a phone calls in response to the
  resync marker. Needs a message-history store that does not exist
  (`internal/conversations` is metadata-only); `backfill_since` is a phone→binary
  *type* with no handler. This slice only **emits** the resync signal. File a
  follow-up when a message store lands.
- **Cross-`/clear` reconnect catch-up** (stale `last_event_id` from a
  rotated-away conversation). Replay is scoped to the *current* conversation
  only; cross-conversation catch-up is deferred `backfill_since` territory. The
  phone is meant to detect the change via `conversation_id` on the live stream
  and resync itself — **but the watermark defect currently suppresses that live
  stream; see [⚠️ Known issue]** (architect Open Question #1).
- **A live two-phone reconnect-replay e2e** — blocked on the structured-receive
  harness gap (#603; see [#642]). The `internal/relay` unit tests driving a real
  Noise handshake against a hand-populated ring are the behaviour oracle here
  (the inherited #589/#615/#632/#633 caveat).
- **`MaxEventsPerConversation` calibration (1024)** — inherited from [#646]; the
  reconnect e2e or a Phase 2 load test is the right place to tune it.
