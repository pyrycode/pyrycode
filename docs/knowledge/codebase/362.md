# Ticket #362 — e2e/realclaude nightly GitHub workflow

Schedules the real-`claude`-binary suite from #361 to run once per day so an upstream `claude` regression surfaces within 24h rather than at the next production dispatch. One new YAML file, no production or test code.

See [features/e2e-realclaude.md](../features/e2e-realclaude.md) for the suite contract and the workflow's full CI-cadence section.

## Implementation

- **`.github/workflows/e2e-realclaude-nightly.yml`** (~40 lines, new). Triggers: `schedule: cron: "0 4 * * *"` (04:00 UTC daily) + `workflow_dispatch: {}`. Single job `e2e-realclaude` on `ubuntu-latest` with `timeout-minutes: 15`. Steps: `actions/checkout@v6` → `actions/setup-go@v6` with `go-version: "1.26.x"` → `npm install -g @anthropic-ai/claude-code` → `make e2e-realclaude` with `ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}` exported in the step's `env:` block.
- **Leading comment block** documents the operator prerequisite (`ANTHROPIC_API_KEY` repo secret), the badge-only monitoring contract (mirrors `self-check-daily.yml` #336), the cost note (every run consumes API credits), and why a more aggressive cadence is not enabled by default.

## Patterns established

- **Action versions / Go pin in nightly workflows must match `self-check-daily.yml` exactly.** Both this workflow and #336's daily self-check use `actions/checkout@v6`, `actions/setup-go@v6`, `go-version: "1.26.x"`, and the same `npm install -g @anthropic-ai/claude-code` line verbatim. Divergence between the two creates two CI footprints to keep in sync; future nightlies should copy from these two and bump them in lockstep.
- **The make target is the only contract between the workflow and the test suite.** The workflow invokes `make e2e-realclaude`, not `go test -tags e2e_realclaude ...` directly. New tests added under `internal/e2e/realclaude/` are picked up without touching CI. Generalisable: when a CI workflow's whole job is "run our test suite for area X", invoke the make target, not the raw `go test` invocation — keeps the test surface evolvable from one place.
- **No `go build` when running `go test` via a make target.** `self-check-daily.yml` builds `pyry` first because it then exec's the binary (`./pyry agent-run --self-check`); this workflow runs Go tests directly, and `go test` handles its own compilation. Don't add a redundant build step for the visual parity — the rule is "build only what you'll exec, not what `go test` will rebuild for you".
- **Schedule offset between sibling daily real-claude workflows.** `self-check-daily.yml` runs at 06:13 UTC; this one at 04:00 UTC. Avoids any shared rate-limit window between two cron jobs that both spawn `claude` against the Anthropic API at the same minute. Future scheduled real-claude workflows should pick a non-overlapping slot rather than colliding on round hours.
- **Badge-only monitoring as the default contract.** Same posture established by #336: red workflow badge IS the failure signal. No auto-issue, no webhook. Per the project's evidence-based fix selection principle, escalating to notification channels is deferred until a missed-badge failure is actually observed.

## Lessons learned

- **`timeout-minutes` is the load-bearing safety net against a hung upstream binary.** The realistic failure mode this workflow protects against is not "test logic deadlocks" — it's "the freshly-installed `claude` binary hangs, and without a per-job timeout we'd burn the full 6h GitHub Actions ceiling silently on every cron tick". 15 minutes is conservative for today's #361 smoke test (~seconds) but accommodates the longer #363–#368 scenarios already on the roadmap. Revisit when those land. Generalisable: any CI job that spawns an external binary as a black box needs an explicit `timeout-minutes` cap; don't rely on the platform default.
- **`continue-on-error: true` would silently neuter the entire workflow.** Adding it on a "but the green badge looks nicer" instinct would mean a real-claude regression flips no signal anywhere. A red nightly IS the intended behaviour — the workflow's whole purpose is to surface failure. This is worth a code-review pattern check on any future nightly: if the job's job is to detect failure, `continue-on-error` must NOT be present.

## Out of scope (deferred)

- **Notification channels (Discord webhook, Slack, auto-issue).** Deferred to a follow-up ticket if badge-only proves insufficient in practice.
- **Aggressive cadence (hourly, per-PR).** Cost-prohibitive — every run consumes API credits. Daily is the right default for catching upstream regressions within 24h; tighter cadence requires a cost/coverage revisit.
- **`act`-based local validation.** Documented in the spec but not pursued — `act` has known limitations with secrets and `npm install -g`. Author validation is via `workflow_dispatch` after merge, or pushing to a personal branch with the secret set on a fork.

## Verification

- `make test` locally is unaffected — build-tag exclusion enforced by #361's scaffold; `make test 2>&1 | grep realclaude` is empty.
- YAML syntax validated by GitHub on push; no separate linter step needed in CI.
- First real-binary execution happens at the next 04:00 UTC tick or on the first `workflow_dispatch` trigger.
