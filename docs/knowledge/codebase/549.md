# Ticket #549 — `cmd/pyry`: wire `V2SessionManager` into the daemon (Mobile Protocol v2 cutover)

The cutover that the entire v2 binary-side series (#430–436 / #450 / #453 / #462) was building toward. Mobile Protocol v2 (Noise_IK E2E) was fully implemented and e2e-tested as [`V2SessionManager`](../features/v2-session-manager.md), but `relay.NewV2SessionManager` had **no caller under `cmd/`** — `cmd/pyry/relay.go:startRelay` built only the v1 path (`dispatch.New` + `authGate`), so a real daemon decoded a phone's `noise_init` as an unknown v1 envelope and the phone could never complete its handshake. This slice flips that with **one explicit operator switch** (`PYRY_MOBILE_V2=1`): with the switch on, the daemon runs the v2 Noise manager over `conn.Frames()` registering the **same three handlers**; with the switch off (the default) the v1 path is byte-for-byte unchanged. Pure wiring — no new exported types, ~60 lines of production code. The cutover-behaviour documentation it owns lives below (§ Cutover behaviour) per AC#5.

## Implementation

- **Two production files modified.** `cmd/pyry/main.go` (+6 LOC) reads the switch in `runSupervisor` next to `allowInsecure` and threads it into the sole `startRelay` call site; `cmd/pyry/relay.go` (+~140/−49 LOC net) gains the `v2Enabled bool` parameter, the v1/v2 branch, and the new `startRelayV2` helper. One new e2e file, `internal/e2e/relay_v2_daemon_test.go` (+259 LOC). No `internal/**` production code changed — every primitive (`V2SessionManager`, the handlers, `keys.LoadOrCreate`, the `conn.Wait()` classifier) already existed.

- **The switch — `PYRY_MOBILE_V2=1`, an env var.** `v2Enabled := os.Getenv("PYRY_MOBILE_V2") == "1"`. Mirrors `PYRY_ALLOW_INSECURE_RELAY=1` exactly: env-only, no `config.Config` field, no CLI flag. Cutover is a one-time operator decision that persists naturally in the systemd unit / launchd plist `Environment=`. Default-unset selects v1 (AC#2). See § Cutover behaviour for the load-bearing reason it is *not* driven by device count.

- **`startRelay` branches after `conn` is established; the prologue is shared.** The empty-`relayURL` short-circuit, `identity.LoadOrCreate`, `devices.Load`, the `allowInsecure` log, and `relay.Connect → conn` run identically on both paths. After `conn`:
  - **`v2Enabled` true** → log `relay: PYRY_MOBILE_V2=1 — Mobile Protocol v2 (Noise_IK) cutover enabled` at Info, call `startRelayV2`, and on its error `conn.Close()` + return (fail fast). No `dispatch.New`, no `authGate`, no assistant-turn bridge, no separate outbound forwarder.
  - **else** → the existing v1 body verbatim (dispatcher + auth gate + `Register`×3 + assistant-turn bridge + dispatcher/forwarder goroutines).

- **`legCleanup` indirection unifies the two teardown shapes.** A `var legCleanup func()` captures the protocol-specific cleanup (v1: `bridgeCleanup` → `conn.Close()` → `<-dispatcherDone` → `<-forwarderDone`; v2: `conn.Close()` → `drain()`). The outer `cleanup` is then just `legCleanup(); <-waitDone`. This is the structural device that lets the `conn.Wait()` classifier goroutine (`waitDone`) be **shared after the branch** — the v2 leg inherits the 4409 server-id-conflict → `shutdown()` contract "for free" without duplicating the classifier.

- **`startRelayV2` helper** (the new leg, ~45 LOC). Returns a `drain func()`:
  1. `keys.LoadOrCreate(resolveStaticKeyBaseDir(), sanitizeName(instanceName))` — the **same `(baseDir, name)` pair `pyry pair` uses** (`pair.go:188`), so the loaded private key derives the public key the phone pinned at pairing. On error: `fmt.Errorf("load static key: %w", err)` → daemon startup fails fast, mirroring the `identity.LoadOrCreate` / `devices.Load` posture.
  2. `priv := staticKey.PrivateKey()` (`[32]byte`); pass `priv[:]` as `StaticPriv` (`noise.KeyLen == 32`). The `[32]byte` backing array escapes to heap via the manager's retained slice — no premature GC.
  3. Build the handler table with the **same `dispatch.Handler` constructor values as v1** (AC#3): `protocol.TypeListConversations: handlers.ListConversations(convReg)`, `protocol.TypeRegisterPushToken: handlers.RegisterPushToken(registry, resolveDevicesPath(instanceName), logger)`, `protocol.TypeSendMessage: handlers.SendMessage(sess, logger)`.
  4. `relay.NewV2SessionManager(relay.V2SessionConfig{Frames: conn.Frames(), Outbound: conn.Send, StaticPriv: priv[:], Devices: registry, ServerID: string(serverID), Logger: logger, Handlers: <table>})`. On error: `fmt.Errorf("build v2 session manager: %w", err)` (unreachable in practice — all fields populated).
  5. One goroutine: `go func(){ defer close(mgrDone); if err := mgr.Run(ctx); err != nil { logger.Debug("relay: v2 manager run returned", "err", err) } }()`.
  6. `drain` is `func(){ <-mgrDone }`. The caller `Close()`s `conn` *before* calling drain.

- **No separate outbound forwarder.** v1 spins a goroutine to drain `d.Outbound()` into `conn.Send`. v2 passes `Outbound: conn.Send` directly — the manager calls it synchronously from its single dispatch goroutine, so the forwarder goroutine simply doesn't exist on the v2 path.

- **Shutdown ordering (v2 leg).** `cleanup` runs `legCleanup()` = `conn.Close()` → `drain()` (`<-mgrDone`), then `<-waitDone`. This is leak-free because `Connection.run`'s `defer close(c.frames)` fires before `defer close(c.done)` (LIFO): closing `conn` closes `frames` → `mgr.Run`'s `<-Frames` sees the closed channel → `Run` returns → `close(mgrDone)` unblocks `drain`; `conn.Wait()` returns once `conn.run` completes, unblocking `waitDone`. The early-return-on-error path (`startRelayV2` fails → `conn.Close()` → return) unwinds `conn.run` via its `<-c.closed` arm with no leak.

## Cutover behaviour (AC#5 — the durable knowledge)

- **How the switch is set.** Export `PYRY_MOBILE_V2=1` in the daemon's environment (systemd `Environment=` / launchd `EnvironmentVariables`) and restart `pyry`. Unset (or any value other than `1`) keeps the v1 dispatch path. The relay route (`/v1/server` vs `/v2/server`) is **not** synthesised by the switch — it is part of the operator-configured `RelayURL` (see `resolveRelayURL`); the operator points the daemon at the v2 route independently.

- **`pyry pair preflight` is the operator's pre-flip safety check — NOT the runtime selector.** Run it first; exit 0 means **zero** v1 devices are paired, so flipping the switch breaks no existing v1 pairing. **The daemon's runtime v1-vs-v2 selection MUST NOT be driven by `preflightVerdict(len(registry.List()))`** — this is the load-bearing constraint of the whole ticket. The preflight gate passes only on an *empty* registry, but a v2 phone must have a paired device in that *same* registry to authenticate (`Devices.Validate(token)` inside the handshake). Driving the daemon off the device count would make a v2 phone impossible to ever authenticate. Devices carry no protocol-version field today, so the registry cannot distinguish a v1 pairing from a v2 one — hence the switch is an explicit operator knob, and preflight is advisory.

- **Simultaneous v1+v2 operation is out of scope.** Per [ADR 024](../decisions/024-noise-ik-mobile-e2e.md) and `docs/protocol-mobile.md` § Security model, v2 is a **hard cutover, no soft fallback**. The switch is binary and exclusive: with it on, *only* the v2 manager consumes `conn.Frames()` (a stray v1 frame fails `decodeInnerFrameV2` and closes at 4421); with it off, *only* the v1 dispatcher runs (a `noise_init` is handled as an ordinary v1 first frame — see Lessons learned). There is no mixed-mode path and therefore **no protocol-downgrade attack surface** — the choice cannot be coerced at the wire by either party. Device-version-tagged v1↔v2 coexistence (a registry schema change) is a deliberate non-goal.

## Patterns established

- **Operator-set relay knobs are env vars, not config/flags.** `PYRY_MOBILE_V2` joins `PYRY_ALLOW_INSECURE_RELAY` and `PYRY_RELAY_URL` as an env-only switch read in `runSupervisor`. For a one-time, persists-in-the-service-unit operator decision, an env var is the right surface — adding a `config.Config` field or a `-pyry-*` flag is a wider surface than the decision warrants (Simplicity First). The precedent: relay-leg behaviour that an operator sets once and forgets lives in the environment, not in `config.Config`.

- **`var legCleanup func()` to share a downstream goroutine across two mutually-exclusive branches.** When two branches each own a distinct teardown sequence but both feed a common trailing step (here the shared `waitDone` classifier), capture the branch-specific teardown in a closure assigned inside each arm, then run `legCleanup(); <-shared` once after the branch. The shared goroutine is wired exactly once and both legs inherit its contract — no duplication, no risk of the two legs drifting on the shared step.

- **A new param threaded through one call site is the minimal cutover wiring.** The switch is read at the daemon boundary (`main.go`) and threaded as a single `bool` into `startRelay`; everything downstream branches on it. No new package, no new exported type, no new interface — the v2 primitives all pre-existed. A protocol cutover does not need a cutover *framework*.

## Lessons learned

- **The v1 first-frame auth gate decodes *any* first frame — including a v2 `noise_init` — as a generic `protocol.Envelope`, and that is what makes the switch-off test clean.** `relay.AuthenticateFirstFrame` ignores unknown `v`/`data` fields and replies `hello_ack` for a paired token. So the AC#2 assertion ("with the switch unset, a `noise_init` is rejected exactly as today") becomes the cleaner, more deterministic "the phone gets a v1 `hello_ack`, never a `noise_resp`" — the distinguishing signal is purely *which protocol's reply comes back*, with no need for a malformed-frame path or a close-code assertion. When proving a protocol-selection switch is off, assert the *positive presence of the other protocol's response* rather than the *absence/rejection of this one* — the former is unambiguous, the latter can be satisfied by many failure modes.

- **A signature change is the natural moment to refresh the doc comment — and the easiest to forget.** Code review filed a [NIT] that `startRelay`'s doc comment still described only the shared `waitDone` classifier and never mentioned the new `v2Enabled` branch (staleness that partly predated this PR, surfaced because the signature changed here). It never blocks merge, but the lesson stands: when you change a function's signature, re-read its doc comment in the same edit — the comment's drift is invisible in the diff unless you look for it.

## Out of scope (deferred — each a candidate follow-up)

- **Assistant-turn `message` fan-out to v2 phones.** The v1 path wires `startAssistantTurnBridge(ctx, sup, bridge, d, logger)` against the v1 dispatcher; `V2SessionManager` exposes no broadcast surface. The v2 branch does not call it — `sup` and `bridge` go unused on that branch (they stay in `startRelay`'s signature for the v1 path). PTY-output streaming (#311) is silently absent on v2; the AC require only inbound `list_conversations`.
- **Per-conn fan-out for handler dispatch.** v2 runs handlers synchronously on the manager's single dispatch goroutine — a slow handler (worst case `send_message`'s 30 s `Activate` timeout) stalls *all* `conn_id`s. This is the documented priority concern before a production cutover at scale; the follow-up spawns one goroutine per `conn_id` with a per-session mutex guarding `s.send` / `s.recv`. See [`features/v2-session-manager.md`](../features/v2-session-manager.md) § Concurrency.
- **`(*V2SessionManager).Rekey` → control socket.** The manager-side manual-rekey trigger shipped in [#462](462.md), but wiring it to `control.Server.SetRekeyer` (so `pyry rekey <conn_id>` reaches the live manager) is still not done. `NewV2SessionManager` now has its first production caller (this ticket), but the `Rekey` method remains reachable from tests only until the control-socket wire-up lands.
- **Device-version-tagged v1↔v2 coexistence** — a `devices` registry schema change; always-split. See § Cutover behaviour.
- **`relay.Connection.handshake`'s `ProtocolVersions: ["v1"]`** (`connection.go:268`) is left unchanged. The phone-side Noise leg is proven independent of the binary↔relay hello's advertised version (the inline v2 e2e harness connects through the unchanged `["v1"]` hello and the handshake still succeeds). Evidence-based open question: if the production relay ever routes or rejects the binary leg on advertised version, a follow-up makes `ProtocolVersions` switch-aware. No observed failure → not changed here.
- **`pyry pair preflight` as a hard startup gate.** Today it is advisory (operator runs it manually before flipping). Making the daemon refuse to start with v2 enabled + non-empty v1 pairings cannot be precise yet — the registry has no protocol-version field to tell a v1 pairing from a v2 one.

## Files

- `cmd/pyry/main.go` — read `PYRY_MOBILE_V2` in `runSupervisor`; thread `v2Enabled` into the sole `startRelay` call.
- `cmd/pyry/relay.go` — `startRelay` gains `v2Enabled bool`; v1/v2 branch via `legCleanup`; shared `waitDone` classifier after the branch; new `startRelayV2` helper (static-key load, handler table, manager construction + run, drain). New import `internal/keys`.
- `internal/e2e/relay_v2_daemon_test.go` (`//go:build e2e`) — `TestRelayV2_Daemon` with two subtests:
  - `v2_enabled_list_conversations_round_trip` (AC#1, AC#4) — `RunBareIn "pair"` → `decodePairPayload` (`.Token` + base64-decoded `.ServerStaticPubkey`) → seed `conversations.json` → `StartInWithEnv(…, PYRY_ALLOW_INSECURE_RELAY=1, PYRY_MOBILE_V2=1, -pyry-relay=…/v2/server)` → wait binary↔relay hello → `fakephone.Dial` → `driveHandshakeToOpenDaemon` (mirrors the inline-harness `driveHandshakeToOpen` but takes `pubKey []byte` directly) → seal `list_conversations`, send `noise_msg`, decrypt the reply, assert `protocol.TypeConversations` with `InReplyTo` → request id and the seeded conversation present. The round-trip also structurally exercises AC#3's handler table.
  - `v2_disabled_does_not_engage_v2` (AC#2) — pair a device, start the daemon *without* `PYRY_MOBILE_V2` (relay `/v1/server`), phone sends a raw `noise_init` `InnerFrameV2` via `SendBytes`; assert a v1 `hello_ack` envelope comes back and *never* a `noise_resp`.
