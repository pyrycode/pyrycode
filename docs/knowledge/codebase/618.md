# Ticket #618 — Screen-snapshot daemon handler (`request_snapshot` → render → `screen_snapshot`)

Wires the daemon side of ADR 025's always-available, parser-independent **screen
snapshot** (the floor of the safe-degradation strategy). A paired phone in an open
v2 session sends `request_snapshot`; the daemon renders the current claude screen to
plain text **inside the tui-driver seal** and pushes a `screen_snapshot` back to the
requesting connection. Consumes the wire vocabulary [#617](617.md) merged
(`RequestSnapshotPayload` / `ScreenSnapshotPayload` + `TypeRequestSnapshot` /
`TypeScreenSnapshot`); adds the three missing pieces — a supervisor render seam, the
inbound interception, and the push. **`security-sensitive`** (label present): the
handler accepts an inbound frame from a non-trusted party over an internet-exposed
relay and returns rendered screen content, so the spec-stage security review was
mandatory (verdict PASS — see the spec). Part of [EPIC #596] (Phase 2 structured
streaming). Split from #612.

## Implementation

Three production files, ~110 LOC, no new files, one new exported interface, **zero**
forced edits to existing call sites (the two new config fields are optional).

- **`internal/supervisor/supervisor.go`** — new `(*Supervisor).ScreenSnapshot()
  (text string, live bool)`. Mirrors `WriteUserTurn`'s `sessMu` capture (lock →
  read `s.sess` pointer → unlock), then renders the live session to text:
  `tuidriver.Render(sess.Snapshot(), 0, 0)`. `sess == nil` → `("", false)` (no child
  attached: between restarts, mid-spawn, idle-evicted). `0,0` selects tui-driver's
  120×40 default grid, which matches the daemon PTY's allocation in headless mode
  (`resizeOnce` only fires for a TTY stdin), so the render is 1:1. **Total — no error
  path.** Non-blocking: a pointer read + a bounded in-memory VT100 render (no I/O, no
  channel ops), so it cannot wedge the relay's dispatch goroutine.

- **`internal/relay/v2session.go`** — interception + handler:
  - New consumer-declared interface `ScreenSnapshotter { ScreenSnapshot() (text
    string, live bool) }`. `*supervisor.Supervisor` satisfies it; declared in the
    relay so `internal/relay` depends on neither `internal/supervisor` nor tui-driver
    (CODING-STYLE: interfaces where consumed).
  - Two **optional** `V2SessionConfig` fields: `Snapshotter ScreenSnapshotter` and
    `KnownConversation func(conversationID string) bool`. `NewV2SessionManager` does
    **not** validate them (nil ⇒ defined fallback behaviour). Optionality is the
    load-bearing sizing decision — see § Scope.
  - `dispatchAppFrame`'s single `if probeEnv.Type == TypeRekeyRequest` probe becomes
    a `switch probeEnv.Type` with a `TypeRequestSnapshot` arm next to the
    `TypeRekeyRequest` arm — intercepted **before** `dispatch.Route`, same boundary
    [#454](454.md) established. The JSON-decode-failure fall-through to `dispatch.Route`
    (which emits the `protocol.malformed` reply) is preserved.
  - `handleRequestSnapshot(ctx, s, env)` runs on the single `Run` dispatch goroutine.
    Deterministic, every branch pushes exactly one reply and returns (AC #3 — never
    panics, hangs, or silently drops): decode `RequestSnapshotPayload` (a decode
    failure is tolerated → empty `ConversationID`, collapses into not-found); **AC
    #4** reject unknown/foreign/empty/`nil-seam` `conversation_id` →
    `error{conversation.not_found, retryable=false}` *before any render*; **AC #3**
    `nil Snapshotter` or `!live` → `error{server.binary_offline, retryable=true}`;
    happy path → `ScreenSnapshotPayload{ConversationID, Text, TS: time.Now().UTC()}`
    wrapped as `Envelope{Type: screen_snapshot, InReplyTo: &env.ID}`.
  - `snapshotReplyError(...)` — thin helper marshalling a `protocol.ErrorPayload` into
    a `TypeError` envelope correlated by `InReplyTo`. Both success and error replies
    go through `m.handlePush` — the **single existing seal-and-forward path** ([#571](571.md)).
    The public `Push` is deliberately NOT used: it funnels onto `m.push` and waits for
    `Run`, which is the goroutine executing the handler → guaranteed self-deadlock.

- **`cmd/pyry/relay.go`** — `startRelayV2` adds `Snapshotter: sup` and a
  `KnownConversation` closure over `convReg.Get(conversations.ConversationID(id))`
  returning a `bool` (mirrors the `ValidateConversation` registry-membership pattern,
  but `bool` so the relay needs no `conversations` import or `errors.Is` coupling).
  `sup` and `convReg` are already `startRelayV2` parameters; both packages already
  imported.

## Patterns established

- **Consumer-declared narrow capability seam keeps the relay zero-knowledge.** Rather
  than the relay importing `internal/supervisor`/tui-driver, the one-method
  `ScreenSnapshotter` is declared where it is consumed and the supervisor satisfies it
  structurally. Same shape as `Config.ValidateConversation`'s closure-over-registry
  ([#312] / `internal/sessions/pool.go`): a behaviour passed in, not a package
  dependency taken on.
- **Optional config field over required → zero call-site cascade.** `V2SessionConfig`
  is constructed at ~42 sites (1 production + ~40 in `v2session_test.go` + 1 in
  `relay_v2_handshake_test.go`). Leaving the two new fields unvalidated means all
  existing sites compile unchanged with the fields nil; only the new unit tests set
  them. A *required* field would have triggered a 40-site cascade and forced a split
  past the 10-call-site red line. The nil behaviours are first-class, not accidents:
  `nil KnownConversation` rejects everything not-found, `nil Snapshotter` reports
  `server.binary_offline` — the feature is simply unavailable, never a crash.
- **Reuse `handlePush`, never the public `Push`, on the dispatch goroutine.** The
  rekey ([#454](454.md)) and `dispatchAppFrame` reply paths already seal-and-forward
  inline via `handlePush`; this handler does the same. Flagged in three code comments
  and in the security review: code-review should confirm no `m.Push(` appears inside
  `handleRequestSnapshot`. This is the canonical inline-reply choreography for any
  future intercepted-control handler.
- **No-log-of-screen-text discipline, pinned by a sentinel test.** The rendered text
  is sensitive and is NEVER logged (mirrors `assistant_turn_v2.go`'s chunk-bytes
  discipline); error replies carry only a **static** message constant
  (`msgSnapshotConvNotFound` / `msgSnapshotOffline`), never the decode-error text or
  the attacker-controlled raw `conversation_id`. The handler logs only `conn_id`,
  `conversation_id` (non-secret UUID), and event names. A unit scenario captures slog
  output to a buffer and asserts a benign non-substrate sentinel never appears.
- **Render inside the seal; pyrycode forwards opaque text.** `sess.Snapshot()`'s raw
  VT100 bytes are consumed by `tuidriver.Render` in the same expression and never
  named or stored in pyrycode — no claude-screen literal enters the package, so
  `cmd/substrate-guard` stays green. Tests assert `Text` is *a string* and use a
  benign non-substrate sentinel; none hardcode a claude screen glyph/anchor.

## Test surface

- **`internal/relay/v2session_test.go`** — `TestV2Session_OpenState_RequestSnapshot`
  (table-driven over: happy render, empty-screen → empty `screen_snapshot`, foreign
  conv → `conversation.not_found`, `nil KnownConversation` rejects all, empty
  `conversation_id`, malformed payload, no-live-session → `server.binary_offline`,
  `nil Snapshotter` → `server.binary_offline`; each asserts `Type`, `InReplyTo ==
  reqID`, and code/retryable or text/`ConversationID`/recent `TS`).
  `..._Repeat` asserts two consecutive requests each get a fresh `screen_snapshot`
  with a freshly stamped `TS`. `..._NeverLogsScreenText` pins the no-log invariant
  with a sentinel. A `fakeSnapshotter{text, live}` is the test double.
- **`internal/e2e/relay_v2_daemon_test.go`** (AC #5) —
  `testV2DaemonRequestSnapshotRoundTrip`, wired into `TestRelayV2_Daemon`'s `t.Run`
  list, modelled on `testV2DaemonListConversationsRoundTrip`. The default `sleep`
  child gives a **live** tui-driver session whose buffer renders to (typically empty)
  text, so the render → `screen_snapshot` round-trip exercises the real daemon seam
  without `send_message`/`WaitReady`. Asserts `screen_snapshot`, `InReplyTo ==
  reqID`, payload `ConversationID == knownConvID`, `Text` **is a string** (no content
  coupling — would risk the substrate guard), `TS` non-zero; a second request naming
  a foreign uuid → `error{conversation.not_found}` (AC #4 at the daemon seam).
- `make check` (vet + race + staticcheck + **substrate-guard**) green.

## Out of scope (deferred)

- **Per-conversation screen routing.** This phase hosts a single bootstrap
  conversation; `KnownConversation` validates *registry membership*, so any registered
  `conversation_id` renders the one live screen. The multi-conversation phase ([#596]+)
  MUST tighten the render to the named conversation's session — recorded so a future
  implementer does not mistake registry-membership for per-conversation authorization.
  No new exposure today: the rendered screen is the same content the phone already
  receives via the [#589](589.md) assistant-turn `message` fan-out.
- **Resized-foreground render dimensions.** `ScreenSnapshot` renders at the 120×40
  default; in headless/daemon mode the PTY stays at that default so the render is 1:1.
  A foreground operator who resized via SIGWINCH could see different word-wrap.
  Acceptable for a degrade-floor view; threading the live size through (tui-driver has
  `Resize` but no size getter) is deferred.
- **Eager push on `stall_detected`.** ADR 025 flags "also push the snapshot on stall"
  as a Phase 2/3 sub-decision; this ticket is on-demand (`request_snapshot`) only.
- **Request-flood DoS.** An authenticated phone could loop `request_snapshot`,
  serialising v2 traffic behind per-render CPU on the single dispatch goroutine. Not
  gated: the peer is token-authenticated, each render is bounded (~ms), inbound is
  capped at `maxNoisePayloadBytes` (65535), output by the 120×40 grid — parity with
  every other frame type (no per-frame rate limit exists in the v2 manager). Deferred
  to a future cross-cutting rate-limit ticket if observed.
