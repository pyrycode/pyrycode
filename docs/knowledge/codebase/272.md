# Ticket #272 — `internal/protocol`: v1 messaging + backfill payload structs

Phase 3 Track C — second slice of the #256 per-type payload catalog. Five pure DTOs that slot into `Envelope.Payload (json.RawMessage)` once the future dispatcher reads `Envelope.Type`: `SendMessagePayload`, `MessagePayload`, `BackfillSincePayload`, `MessageChunkPayload`, `BackfillDonePayload`. Shapes are fixed verbatim by `docs/protocol-mobile.md` § Message types; the example payload in each subsection is the golden-file fixture. Sibling slice to #275 (`RegisterPushTokenPayload`).

## Implementation

- **One production file, one test file, five fixtures, stdlib-only.** `internal/protocol/messaging.go` (~50 LOC, five struct declarations + doc comments) + `internal/protocol/messaging_test.go` (~187 LOC, one round-trip test per type) + five fixtures under `internal/protocol/testdata/` (`send_message.json`, `message.json`, `backfill_since.json`, `message_chunk.json`, `backfill_done.json`). All five `Type*` constants the tests consume (`TypeSendMessage`, `TypeMessage`, `TypeBackfillSince`, `TypeMessageChunk`, `TypeBackfillDone`) already exist in `codes.go` from #255 — no constants added.
- **Five types in one file, by spec-grouping precedent.** Spec § Message types presents these as a continuous block; the sibling-slice convention is "one file per spec-section group" (#275 → `push.go`). Splitting five DTOs across five files would cross the architect's "more than 3 new files" red line for no readability gain. Total production LOC well under 100.
- **`MessageChunkPayload.Messages []MessagePayload` reuses the row type directly — no `MessageChunkRow` duplicate.** The spec says "same shape as `message.payload`, multiple." A duplicate row struct would silently drift from `MessagePayload` over time, which is precisely the bug the spec text forbids. Reuse is an explicit AC. The `message_chunk` fixture contains exactly 2 messages with distinct roles (`assistant`, `user`) and distinct IDs (`m12`, `m13`) so the slice round-trip is non-trivially covered (`TestMessageChunkPayload_RoundTrip` asserts both).
- **`BackfillSincePayload.ConversationID` is `*string` WITHOUT `omitempty` — the single subtle interaction.** Spec example shows literal `"conversation_id": null` on the wire (meaning "all conversations"). `*string` distinguishes "null on wire" (nil pointer) from "empty-string conversation id" at the boundary. Critically, `omitempty` is DROPPED for this field: with `omitempty` a nil pointer marshals as absent (key dropped); without it, a nil pointer marshals as `null` — byte-identical to the fixture. The AC reads "*T + omitempty for optional fields," but the spec's `null`-on-wire requirement overrides for this field. A one-line WHY comment is mandatory on the field (the file's only comment under the project's "default to no comments" rule) so a future contributor doesn't "fix" the missing `omitempty` and silently break the round-trip. The byte-equal check in `TestBackfillSincePayload_RoundTrip` is the regression detector — adding `omitempty` back drops the `conversation_id` key from the output and fails the test.
- **`MessagePayload.Role` stays `string`, not a named `Role` enum type.** Spec defines a closed set (`"user"`, `"assistant"`, `"system"`) but the binary already treats role-strings as `string`-typed elsewhere; a typed `Role` would force a converter at every internal call site for no observable wire-format gain. Validation belongs at the dispatcher. Matches the rationale in #275 for `Platform`.
- **All required-field gates are absent — pure DTO, dispatcher validates.** Missing required fields (`text` absent from `send_message`, etc.) decode to zero values per stdlib's permissive contract; required-field validation is the dispatcher's job (#248), not this package's. No `Validate()` methods, no constructors. Same boundary `IsV1Compatible` draws: framing-layer types are "structurally well-formed JSON" but not "semantically validated."
- **Fixtures are envelope-wrapped, decoded through `Envelope.Payload`.** Each test reads the fixture → `json.Unmarshal` into `Envelope` → asserts `env.Type == Type<X>` (uses the constant, not a hardcoded string) → second-pass `json.Unmarshal(env.Payload, &payload)` into the per-type struct → spot-asserts fields → re-marshals envelope → `bytes.Equal(canonical(out), canonical(raw))`. This exercises the exact composition the dispatcher will use, and the byte-equal check pins wire-bytes against any silent struct-tag drift.
- **Tests reuse `canonical(t, b)` and `readFixture(t, name)` from `envelope_test.go`.** Same package, no import, no duplication. Five test functions (`TestSendMessagePayload_RoundTrip`, `TestMessagePayload_RoundTrip`, `TestBackfillSincePayload_RoundTrip`, `TestMessageChunkPayload_RoundTrip`, `TestBackfillDonePayload_RoundTrip`) follow the same shape as `TestEnvelope_RoundTrip_Full`.
- **Specific assertions worth pinning:** `TestBackfillSincePayload_RoundTrip` asserts `payload.ConversationID == nil` explicitly (NOT `*payload.ConversationID == ""` — would panic on nil deref AND be the wrong check) and `payload.SinceTS.Equal(time.Date(...))` via `time.Time.Equal` not `==` (monotonic-clock strip on marshal, per the existing `docs/PROJECT-MEMORY.md` discipline). `TestBackfillDonePayload_RoundTrip` asserts `env.InReplyTo != nil && *env.InReplyTo == 6` (the only fixture in this slice that pins `in_reply_to` since `backfill_done` is always a response).
- **Field declaration order matches fixture key order for byte-stable round-trip.** Go's `encoding/json` marshals struct fields in declaration order; `BackfillSincePayload`'s declaration order is `SinceTS, ConversationID, MaxMessages` to match the fixture's payload key order (`since_ts, conversation_id, max_messages`). A future reorderer must update the fixture in lockstep.

## Out of scope (deferred)

- The other 10 v1 payload structs (`hello` / `hello_ack` / `error` / `ack` for #271 handshake-and-control; `list_conversations` / `conversations` / `create_conversation` / `conversation_created` / `promote_conversation` / `conversation_updated` for the TBD conversations slice). Each lands in its own ticket with its own file in this package.
- A named `Role` type or any role-validation predicate. Rationale: see § Implementation above; identical to #275's `Platform` decision.
- Token-by-token streaming. `message_chunk` is the wire shape only; v1 emits one finished message per `message` envelope per spec. Streaming is a reserved-for-v1.1 use of the same envelope shape.
- Backfill orchestration: chunk-size enforcement (spec's "≤ 50 messages per envelope" default), ordering by `(conversation_id, ts)`, deciding when to send `backfill_done`. All dispatcher concerns, not framing.
- `BackfillSincePayload.IsAllConversations() bool` predicate, `MessageChunkPayload.Validate() error`, any constructor for the five DTOs. None have a use site; dispatcher is the only legitimate consumer.
- A `json.Marshaler` / `json.Unmarshaler` custom implementation. The default struct-tag-driven implementation suffices; a custom marshaler would re-open the `*string` round-trip subtlety inside Go code rather than declaring it via tags.
