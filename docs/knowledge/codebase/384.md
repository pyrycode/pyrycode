# Ticket #384 — `e2e/realclaude`: MCP server smoke tests (qmd, context7, codegraph, figma)

Sixth consumer of the `internal/e2e/realclaude` suite. Adds four named tests — one per MCP server the dispatcher agents depend on (`qmd`, `plugin:context7:context7`, `codegraph`, `plugin:figma:figma`) — that drive a haiku turn through `pyry agent-run` and assert the named MCP tool fires with a non-empty `tool_result`.

Motivated by the failure mode the existing suite does not cover: MCP protocol drift on the server side. If a server renames a tool, changes a parameter shape, or breaks its stdio handshake at startup, an affected dispatcher agent silently fails (model never calls the tool) or hangs to `max_turns`. The four tests surface that drift as a structured failure under bounded cost, four independent pass/fail signals so a regression in any single server is attributable on sight. No production-code change.

See [features/e2e-realclaude.md](../features/e2e-realclaude.md) for the suite contract.

## Implementation

- **`internal/e2e/realclaude/mcp_smoke_test.go`** (~290 LoC, new). `//go:build e2e_realclaude`, `package realclaude`. Four top-level `Test…` functions + one shared `assertMCPToolUsed(t, workdir, result, toolName)` helper + one file-local `mcpServerHealthy(t, displayName) (bool, string)` pre-flight probe + two file-level constants (`figmaTestFileURL`, `mcpSmokeSystemPrompt`). No edits to `fixtures.go`, no new exported fixture surface, no production-source changes.
- **Tool-name pinning.** Each test passes the **claude-reported** MCP tool name (the form printed by `claude mcp list` and emitted in JSONL `tool_use.name`) on `--allowed-tools` and asserts the same name in the JSONL walk:
  | Server display name (in `claude mcp list`) | Tool name pinned |
  | --- | --- |
  | `qmd` | `mcp__qmd__query` |
  | `plugin:context7:context7` | `mcp__plugin_context7_context7__resolve-library-id` |
  | `codegraph` | `mcp__codegraph__codegraph_search` |
  | `plugin:figma:figma` | `mcp__plugin_figma_figma__get_metadata` |

  The plugin-prefixed names (`plugin_context7_context7`, `plugin_figma_figma`) differ from the unprefixed `mcp__context7__*` form the dispatcher's `dispatcherBaseTools` (#381) currently lists — that mismatch is invisible there because the role smoke prompts never compel a context7 call, but is binding here because the prompt explicitly drives the named tool. A follow-up to align the dispatcher list is noted in Out of scope.
- **HOME is deliberately NOT pinned.** The four tests do NOT call `WithWorktree(t)` or `WithWorktreeAuthenticated(t)`; both pin `$HOME` to a tempdir, which disables claude's MCP server discovery (claude reads its server registry from `$HOME/.claude.json` + `$HOME/.claude/plugins/...`). The tests allocate a workdir via `t.TempDir()` and leave `$HOME` pointing at the outer operator's home so claude resolves the same MCP server set the operator has configured. Trade-off: each test leaves a JSONL session file under the outer `~/.claude/projects/<encoded-tempdir>/`. Acceptable because (a) the encoded path is unique per run, (b) `WithWorktreeAuthenticated` already leaks `ANTHROPIC_API_KEY` from the outer environment so the suite is not fully hermetic today, (c) CI runners have a controlled HOME. The file-header doc-comment names this trade explicitly so a future ticket doesn't "fix" it by reintroducing HOME isolation and breaking MCP discovery.
- **Inline `ANTHROPIC_API_KEY` gate per test.** Each test starts with `if os.Getenv("ANTHROPIC_API_KEY") == "" { t.Skipf(...) }` — mirrors `WithWorktreeAuthenticated`'s skip-message shape but inlined because the tests can't use the helper (the helper pins HOME). Skip message names the variable and the action ("Export it (or rely on a CI secret) to run") so the operator can act on the test log alone.
- **`mcpServerHealthy(t, displayName) (bool, string)` pre-flight probe.** File-local, package-private helper at the bottom of `mcp_smoke_test.go`. Runs `claude mcp list` once via `exec.CommandContext` under a 10 s `context.WithTimeout`, scans each line for the prefix `<displayName>:`, and checks whether the suffix contains `✓ Connected`. Returns `(false, "")` when healthy; `(true, reason)` when the server is absent / failed / needs auth — caller invokes `t.Skipf` with the reason. The probe uses the PATH-resolved `claude` binary; **`PYRY_CLAUDE_BIN` is intentionally NOT honored** (that variable is reserved for stubbed pyry tests and carries the 2026-05-16 fork-bomb defense from `resolveClaudeBin` in `resilience_test.go`). When `claude mcp list` itself times out (e.g. one unrelated MCP server is hung on startup), the helper returns `(true, "claude mcp list timed out after 10s")` and the test skips — hung-on-list is operator-environment trouble, not a #384 regression to surface. The reason string is the **suffix after the final ` - `** when the server line is found-but-unhealthy, so the operator sees the literal `! Needs authentication` / `✗ Failed to connect` text from claude rather than a synthesised "unhealthy".
- **`TestRealClaude_MCP_QMD`** (`mcp_smoke_test.go:47-68`). Prompts claude to call `mcp__qmd__query` against the `second-brain` collection for the literal `pyrycode`. `AllowedTools: []string{"mcp__qmd__query", "Bash"}`, `MaxTurns: 3`, `Effort: "low"`, `Model: "claude-haiku-4-5"`. Asserts the standard MCP-tool-used shape (see `assertMCPToolUsed` below). `Bash` in `--allowed-tools` is a passive declaration per AC — the model may or may not use it for verification; the test does not assert on Bash invocation.
- **`TestRealClaude_MCP_Context7`** (`mcp_smoke_test.go:70-91`). Prompts claude to call `mcp__plugin_context7_context7__resolve-library-id` for `react`. Same `RunOpts` shape, `AllowedTools: []string{"mcp__plugin_context7_context7__resolve-library-id", "Bash"}`. The prompt names the plugin-prefixed tool form explicitly so the model passes it through without re-mapping.
- **`TestRealClaude_MCP_CodeGraph`** (`mcp_smoke_test.go:97-136`). Seeds the workdir with a one-symbol Go source file (`func PyrycodeSeedSentinel384() {}`) and runs `codegraph index .` via `exec.CommandContext` under a 30 s timeout, **then** drives `pyry agent-run` with a prompt that asks for that symbol. An MCP server with no index returns empty results regardless of protocol health, so the index seed is the prerequisite that closes the false-negative window. Three skip gates in order: `ANTHROPIC_API_KEY` unset → `mcpServerHealthy("codegraph")` reports unavailable → `exec.LookPath("codegraph")` fails → `codegraph index .` returns non-zero (combined output captured for the skip reason). The prompt names the sentinel literal so the search has a deterministic single-match target.
- **`TestRealClaude_MCP_Figma`** (`mcp_smoke_test.go:144-165`). Pins a public Figma Community file URL (`figmaTestFileURL = "https://www.figma.com/community/file/1035203688168086460/material-3-design-kit"`) and prompts claude to call `mcp__plugin_figma_figma__get_metadata` on it. A top-of-file comment names the URL choice as **swappable**: "If this URL stops resolving, replace it with another stable public Figma file — the test asserts the protocol round-trip, not any one file's content". `! Needs authentication` or `✗ Failed to connect` from `mcpServerHealthy("plugin:figma:figma")` skips cleanly per AC.
- **`assertMCPToolUsed(t, workdir, result, toolName)` shared helper** (`mcp_smoke_test.go:178-244`). Consolidates the six identical assertions every MCP smoke test demands: (1) `result.ExitCode == 0`, (2) `result.SessionID != ""`, (3) walk JSONL events and find the first `assistant` entry with a `tool_use` block whose `name == toolName` and whose `id` is non-empty (capture `toolUseID`), (4) walk forward and find a `user` entry with a `tool_result` block whose `tool_use_id == toolUseID` (capture `toolResultContentLen`), (5) `toolResultContentLen > 0` — non-emptiness is sufficient; content text is server- and version-dependent (qmd snippets, figma file names, codegraph match shapes all vary across runs) and pinning literal content would couple the test to one server's output shape, (6) `parseResultTrailer(result.Stdout)` succeeds and `trailer.PermissionDenials` is nil-or-empty. The shared helper is NOT a table loop — each `Test…` function calls the helper directly with one role's parameters so per-server failure attribution surfaces on the nightly board.
- **Tight system prompt.** `mcpSmokeSystemPrompt = "You are an e2e regression-guard test. When asked to use an MCP tool, call exactly that tool once with the requested parameters, then briefly report the result."` Keeps the model on a single-call rail rather than drifting into exploratory tool-chaining at higher max-turns budgets.
- **`RunOpts` shape (identical across all four tests).** `MaxTurns: 3`, `Effort: "low"`, `Model: "claude-haiku-4-5"`, no `Timeout` override (5-minute `RunPyryAgentRun` default is the runaway guard). Expected aggregate cost ~$0.08 with cache warm, matching the AC estimate.
- **`t.Parallel()` is NOT called.** Matches the existing realclaude convention. Keeps the nightly cost predictable and avoids two concurrent `claude mcp list` invocations contending on the same MCP server stdio sockets (e.g. `npx -y @upstash/context7-mcp` two-process collision).

## Reuses (no re-implementation)

- `realclaude.ReadJSONL` + `realclaude.JSONLEntry` alias (#372) — JSONL-walk path for `assertMCPToolUsed`.
- `realclaude.parseContentBlocks` + `contentBlock` (#376) — content-block decoder. **Fourth consumer** (#376 originated; #382 added `Content` + `IsError` fields; this consumer uses both). The `Content json.RawMessage` field added in #382 is what makes the non-emptiness check on `tool_result` content correct — the lesson from #382 (don't read `b.Text` for `tool_result` blocks) directly applies here and was honoured from first-pass.
- `realclaude.parseResultTrailer` + `resultTrailer` (#376) — stdout-trailer parser. **Fourth consumer.** Used to assert no permission denials.
- `realclaude.jsonlPathFor` (#364) — file-local diagnostic helper for resolved JSONL path strings in failure messages. **Seventh consumer.**
- `realclaude.truncate` (#381) — 1 KiB byte-slice cap formatter for failure messages. **Third consumer.**

Deliberately NOT reused:
- `realclaude.WithWorktree` / `realclaude.WithWorktreeAuthenticated` (#372/#409) — both pin HOME, which disables MCP discovery. See § "HOME is deliberately NOT pinned" above.

## Patterns established

- **Pre-flight probe + clean skip for cross-process prerequisites.** When a test depends on external state that is not part of the test's contract (an MCP server's connection status, a third-party binary's presence, an index built in the workdir), the test should probe that state up front and `t.Skipf` with an operator-actionable reason rather than fail. The reason string must name the probed prerequisite (variable, binary, server) so the operator can fix it without re-running. Skip-not-fail keeps the suite green when the prerequisite is locally absent (figma unauthenticated, codegraph binary not installed) and reserves red signals for **protocol drift in something we expected to work** — the only failure mode this ticket exists to catch. Generalises to any e2e test whose precondition depends on operator environment.
- **Bound the pre-flight probe and treat its timeout as a skip.** `mcpServerHealthy` wraps `claude mcp list` in a 10 s `context.WithTimeout`. If a single MCP server hangs on startup, `claude mcp list` itself can wedge. Treating the probe timeout as a skip (with the timeout reason in the message) keeps the suite resilient to operator-environment trouble unrelated to the property under test. Generalises: any probe whose failure is environmental rather than functional should bound itself and surface the timeout as a skip-reason, not as a test failure.
- **Pin the tool name the CLI emits, not the name an upstream config registers.** The plugin-prefixed `mcp__plugin_context7_context7__resolve-library-id` is what claude actually emits in JSONL; pinning the unprefixed `mcp__context7__*` form (as `dispatcherBaseTools` does) would make the test never compile a `tool_use` match because the names diverge. When a test asserts on a tool name that crosses a CLI boundary, verify the name against the CLI's actual output (e.g. `claude mcp list` for server display names, JSONL `testdata/*.jsonl` for emitted tool names) before pinning. Generalises to any test asserting on identifiers crossing a process boundary.
- **Const-pinned external URLs with a swap-license comment.** Hard-coding `figmaTestFileURL` to a public Figma Community file is fragile; renaming or deleting the file from Figma's side breaks the test. The fix is not to remove the literal but to name it `const figmaTestFileURL = …` and document in a top-of-file comment that **swapping the URL for another stable public file is the correct response** to a 404. The test's contract is the protocol round-trip, not any one file's content; the comment lets a future maintainer fix the test without first reading the design. Generalises to any test pinning an externally-controlled identifier (URL, public dataset, community-file id).
- **Inline an auth gate when the centralised helper imposes incompatible side effects.** The `ANTHROPIC_API_KEY`-unset skip mirrors `WithWorktreeAuthenticated`'s shape — but inlined, because the helper pins HOME and these tests need outer HOME for MCP discovery. When a centralised fixture's side effects conflict with a test's needs, copy the relevant gate inline and document why; do NOT widen the fixture's surface with a flag the other consumers don't need. Generalises: surface drift between a centralised helper and a single divergent caller is better solved by inline duplication-with-a-comment than by parameter creep on the helper.

## Lessons learned

- **Workdir-side prerequisites need explicit seeding even when the MCP server is healthy.** First-pass mental model assumed `mcp__codegraph__codegraph_search` would return results against an empty workdir as long as the codegraph server was `✓ Connected`. It does not — the server returns empty results against an absent or empty `.codegraph/` index, which would have made the assertion fail for the wrong reason (false positive on protocol drift). The fix is to seed the index in the workdir via `codegraph index .` as a test setup step, gated on `exec.LookPath("codegraph")` so the test skips cleanly when the binary is absent rather than failing on a misleading "no tool_result" assertion. Generalises: when an MCP server's tool operates on workdir-resident state (index, cache, config), verify the state exists before driving the call; the server's healthy handshake is necessary but not sufficient for a meaningful response.
- **MCP server discovery and HOME isolation are mutually exclusive.** Naive composition with `WithWorktree(t)` would have pinned HOME to a tempdir, causing claude to see zero MCP servers and refuse every `mcp__*` entry in `--allowed-tools` as unknown — the prompt would produce no `tool_use` and the test would fail for "no protocol round-trip" rather than the actual cause (config absence). The right answer is to let HOME point at the outer operator's configured `~/.claude.json`. The cost (per-test JSONL artifacts under the operator's home) is bounded by `t.TempDir()` uniqueness; the alternative (synthesising a temp MCP config and passing `--mcp-config`) would require a flag-surface change to `cmd/pyry/agent_run.go` for one test family. Generalises: a hermetic-tempdir-HOME pattern can mask the very capability the test means to probe; isolate only what the test owns, not what the system reads.
- **Match the assertion granularity to the protocol's variation surface.** Asserting `tool_result.content == "<specific text>"` would couple the test to one server's output text, which varies across runs and versions (qmd's snippet ranking, figma's file metadata, codegraph's match formatter all evolve independently). `len(content) > 0` is the right granularity: it pins "the protocol round-trip completed and produced a non-empty result" without ratcheting on prose. The same shape as #382's keyword-OR predicate (assert on structural shape, not on literal content). Generalises: when probing protocol health against a third-party producer, assert on structural invariants (non-empty, well-formed, named-field-present) and leave content open.

## Verification

- `go vet -tags=e2e_realclaude ./internal/e2e/realclaude/...` — passes.
- `go build -tags=e2e_realclaude ./internal/e2e/realclaude/...` — compiles.
- `go test -tags e2e_realclaude -run 'TestRealClaude_MCP_' -v ./internal/e2e/realclaude/` — all four pass against a working `claude` binary, live API key, and the four MCP servers configured + connected on the operator's HOME. **Four real haiku calls per run** (~$0.08 with cache warm; per-test wall time well under the 5-minute `RunPyryAgentRun` default).
- Skip-path matrix verified: with `ANTHROPIC_API_KEY` unset all four skip without API spend; with figma unauthenticated the figma test skips while the other three pass; with `codegraph` not on PATH the codegraph test skips while the other three pass.
- `make test` (no tag) — unchanged; the new file is dropped at the build stage by tag exclusion.

## Out of scope (deferred)

- **Aligning `dispatcherBaseTools` (#381) with the plugin-prefixed names this ticket pins.** `agents/dispatcher/src/dispatch.ts` declares `mcp__context7__*` and the mirrored `dispatcherBaseTools` follows that unprefixed form. The actual claude tool names are `mcp__plugin_context7_context7__*`. The mismatch is currently invisible because the per-role smoke tests don't compel context7 calls. Now that #384 has pinned the real names, file a follow-up ticket to align the dispatcher list; out of scope for this developer turn per the spec.
- **Promoting `mcpServerHealthy` to a fixture helper.** Single-file consumer in #384; the helper is one-purpose (MCP-smoke pre-flight). Minimising `fixtures.go` surface keeps it stable for parallel branches. Promote when a second MCP-bearing test file needs the same probe.
- **Enforcing the cost ceiling in code.** AC bounds nightly cost at ~$0.08 (haiku, max-turns 3, four tests). If a future change pushes turns past 3 silently (e.g. a model refusal driving max_turns hits), the cost climbs. Acceptable for now; revisit if the suite grows past 10 MCP-bearing tests.
- **Resilience tests for MCP server failure modes** (server crash mid-call, partial-response truncation, malformed tool_result payload). Each is a separate ticket when the operational need surfaces. The four named tests here pin the **healthy-path** protocol round-trip; protocol-resilience for MCP-side failures is the next layer.
