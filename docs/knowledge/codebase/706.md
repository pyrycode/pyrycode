# Ticket #706 — two-heads modal ownership: first-answer-wins, local attach dismisses

The **local resolution arm** of the daemon-side modal bridge ([EPIC #597] Phase 3,
[ADR 025] § Security model #4 — *two-heads coexistence*). A surfaced permission/trust
modal can be answered by **two heads**: the local `pyry attach` TTY (operator types
`1`/Esc directly into claude) or a paired phone (`modal_answer`/`modal_cancel`). The
remote arm (#717), the safe-answer keystroke seam (#726), the cancel/dismiss/broadcast
seam (#727), the registry + `modal_id` nonce (#716), the deny-on-timeout (#725), and
the `modal_dismissed{source}` wire type (#701) had all landed. This slice adds the
**local** arm and makes resolution **single-shot across both heads**: whichever head
answers first owns the resolution; the other head's prompt clears and any later answer
for that `modal_id` is rejected — so a single modal is never double-answered and an
internet-sourced answer can never act on a modal the operator already handled locally.

It introduces **no new wire type, no new package, no new exported type, no new
crypto, no constructor/wiring change** — only one additive `audit.Outcome` const. The
first-answer-wins guarantee is **structural in machinery that already existed**:
`modalbridge.Registry.Resolve` is a mutex-guarded one-shot consume, and this slice
makes the local arm one more `Resolve` consumer (joining the three remote arms). The
live producer that feeds the surfacer real tui-driver events is **#708** (the Phase 3
capstone, blocked-by this ticket); this slice is unit-testable against the surfacer and
the registry directly, no live claude.

## Implementation

Two production files change (~57 LOC production + ~150 LOC tests), additive — no new
files, no consumer cascade. `security-sensitive` (a cross-goroutine first-answer-wins
race on the permission surface; spec-stage security review verdict **PASS**, 9
categories; code-review verified the race against live code, no findings).

| File | Change |
|---|---|
| `cmd/pyry/interactive_modal_v2.go` | **The whole behavioural change.** The single-event `Handle` body splits into a `Kind` switch dispatching to two arms. **`handleModalShown`** is the old body (gate → `Record` → arm timeout → marshal → fan-out), unchanged except it now **tracks the just-surfaced modal** in two new single-goroutine fields (`outstandingID string`, `outstandingClass tuidriver.ModalClass`), set immediately after `Record`/`ArmModalTimeout` (consistent with the registry entry + armed timeout, even on the defensive marshal-fail return). **`handleModalHidden`** is the new local arm. The Shown/Hidden fan-out loop is factored into a private **`broadcastInteractive(ctx, envType, payloadJSON, pushErrEvent)`** helper (shared timestamp, #607 capability gate, per-conn monotonic `nextID`, `EventID==nil` control envelope, Push-error-tolerant). |
| `internal/audit/audit.go` | **One new const:** `OutcomeDismissedLocal Outcome = "dismissed_local"` — "resolved at the desktop TTY; the picked choice is not observable by the daemon." `SourceLocal` (= `"local"`) already existed (reserved for #706 by #712). One vocabulary feeds both the wire dismissal and the audit entry. |

### `handleModalHidden` — the local arm

`EventKindPtyModalHidden` carries the just-hidden modal **class, not the `modal_id`**.
The id was minted in `Registry.Record` on the preceding `Shown`. The correlation
mechanism (architect's call, Technical Notes left it open) is **emitter-tracked
outstanding id + class** — *not* a registry "resolve-the-current" method (which would
force the registry to track recency/insertion-order, muddying a type whose whole value
is being a precise keyed one-shot). tui-driver's single-modal + `Hidden(old)`-before-
`Shown(new)` invariants guarantee the tracked id always names the showing modal when a
`Hidden` arrives, and `Handle` is single-goroutine so the tracking needs no lock.

```
LOCAL DISMISSAL (producer Run goroutine, cmd/pyry — live in #708)
  handleModalHidden(ctx, ev):
    if outstandingID == ""        → return  // nothing we surfaced is outstanding (AC1 gate)
    if ev.Modal != outstandingClass → warn + return WITHOUT clearing  // defensive, unreachable
    id := outstandingID; clear outstandingID/Class   // modal gone from screen regardless of who consumes
    out, ok := reg.Resolve(id)
    if !ok → return              // first-answer-wins LOSER: remote (#717/#727) or timeout (#725)
                                 //   already consumed it. No audit, no broadcast (AC2 / AC3-b)
    // WINNER:
    audit.Log({ModalID: id, ModalClass: out.Class, Outcome: dismissed_local, Source: local})  // empty device
    broadcastInteractive(ctx, TypeModalDismissed, {id, dismissed_local, local}, …)             // one envelope
```

The daemon **cannot observe which option the operator picked** locally — only that the
modal vanished — so `modal_dismissed{local}` carries a **producer-defined sentinel**
`outcome` (`dismissed_local`), not a real `option_id`; the phone uses it only to clear
its prompt.

## Patterns established

- **First-answer-wins across heads is structural, not added here.** All four resolution
  paths — remote answer (#717), remote cancel (#727), deny-on-timeout (#725), and this
  local arm — route through the one mutex-guarded `Registry.Resolve` (a delete-and-
  report-presence under a leaf lock). Whichever goroutine's `Resolve(id)` runs first
  deletes the entry and gets `ok`; the other gets `!ok`. Each arm broadcasts/audits
  **only on `ok`**, so the same `modal_id` is resolved, broadcast, and audited **at most
  once**, regardless of interleaving. This ticket added a *consumer*, not an arbitration
  mechanism. (Mirror of #725's "exactly-once is structural" pattern, but across **two
  real goroutines** — producer Run vs relay dispatch — rather than two arms of one
  `Run` select.)
- **No keystroke-after-resolution window — verified against live code.** The remote
  answer path (`modalResolverV2.ResolveAnswer`) consumes via `reg.Resolve` (`modal_resolve_v2.go:206`)
  **strictly before** routing any keystroke (`:215`). So if the local `Hidden`'s
  `Resolve` won first, the remote's `Resolve` misses and the remote path **returns
  before any keystroke** reaches claude — an internet-sourced answer can never drive a
  keystroke into a modal the operator already resolved locally (the exact ADR 025 §4
  threat). **The local arm routes no keystroke of its own** (the operator already
  pressed the key at the TTY; the modal is already gone) — it only consumes + notifies,
  a strictly *narrower* actuation surface than every remote arm.
- **Two broadcast sites, accepted cross-package duplication.** The remote/cancel/timeout
  dismissal fans out via `internal/relay`'s `broadcastModalDismissed` (iterates
  `m.sessions` directly **on the Run goroutine**). The local dismissal originates on the
  **producer goroutine** and reuses the surfacer's existing `interactiveBroadcaster`
  (`ActiveConns` + `Push`) — the path `modal_shown` already proves safe from that
  goroutine. The two loops are structurally different (different envelope-`ID`
  conventions: per-conn monotonic `nextID` vs the relay's `ID:1`); `ID` is
  **non-load-bearing for modal events** (the phone correlates on `modal_id`). The shared
  contract is the `protocol.ModalDismissedPayload{ModalID, Outcome, Source}` *shape*,
  already a single type pinned by `testdata/modal_dismissed.json`. A cross-package helper
  would force an awkward dependency for ~3 saved lines — deliberately **not** extracted
  (the project's "resist over-DRY on duplicated broadcast primitives" guidance). The
  emitter **must not** call `broadcastModalDismissed` from the producer goroutine (it is
  Run-goroutine-only).
- **Within-package DRY: one `broadcastInteractive` helper for both arms.** The `Shown`
  and `Hidden` arms otherwise duplicate the `ts := now; for ActiveConns { capability
  gate; nextID++; build Envelope; Push }` loop. Factored into one private helper
  parameterised by `envType`/`payloadJSON`/`pushErrEvent`. The existing `Shown` tests
  (`TestModalEmitter_PermissionFanout`, `_PushErrorContinues`, the nextID-monotonicity /
  non-interactive-filter assertions) guard the refactor; the only behavioural loss is the
  `Shown`-specific `class` field on the push-error debug log (now the generic
  `pushErrEvent`/`conn_id`/`env_id`/`err` set) — acceptable, class is non-essential debug
  context.
- **One source vocabulary feeds both the wire dismissal and the audit** (the
  `ResolveCancel`/`ResolveTimeout` precedent). `audit.OutcomeDismissedLocal` /
  `audit.SourceLocal` produce both `protocol.ModalDismissedPayload{Outcome, Source}` and
  `audit.Entry{Outcome, Source}` — no second, divergent set of strings.
- **Audit the local arm, on the winner only.** ADR 025 §6 requires every modal
  resolution to be logged; a local resolution **retires a one-time nonce and pre-empts a
  remote answer**, so it is exactly as security-relevant as a remote cancel. It records
  with **no answering device** (a local TTY resolution has none → empty `DeviceHash`/
  `DeviceLabel`, the no-device case `ResolveTimeout` uses); `ModalClass` comes from the
  resolved `Outstanding.Class`. Auditing only on the `ok` (winner) path avoids a phantom
  `dismissed_local` record for a modal the phone actually answered.
- **Emitter-local tracking stays single-goroutine.** `nextID`, `outstandingID`,
  `outstandingClass` are read/written **only** inside `Handle` (one goroutine), so they
  need no synchronisation — preserving the surfacer's documented single-goroutine
  invariant (the struct doc comment was extended to name the two new fields). The only
  field a second goroutine touches remains `reg`, which carries its own mutex. No new
  lock, no new goroutine — the surfacer stays a passive state machine.
- **Clear tracking unconditionally once the class matches.** `handleModalHidden` captures
  `id`, then clears `outstandingID`/`outstandingClass` **before** calling `Resolve` — so
  no return path (winner *or* loser) leaks a stale id. The modal is gone from the local
  screen regardless of which head consumes the registry entry.

## Lessons learned

- **Production-inert until #708.** Nothing feeds `Handle` real tui-driver events in
  production until #708 live-wires the surfacer into the PTY event stream, so the local
  arm never fires in production today (the same harmless state as #716/#725/#727). The
  scripted tests are the correctness proof; the live two-phone e2e (operator answers at
  TTY → phone prompt clears) is **#708's capstone**, which this slice blocks.
- **The cross-head race is proven against the *real* remote resolver, not a stub.** The
  two-ordering test matrix drives the genuine `newModalResolverV2.ResolveAnswer` against
  the **same** `*modalbridge.Registry` the emitter uses — so local-first-then-remote and
  remote-first-then-local exercise the actual first-answer-wins arbiter, reusing the
  keystroker/device fakes from `modal_resolve_v2_test.go`. This is the daemon-side proof;
  it does not need live claude.
- **The class-mismatch branch is deliberately unreachable.** Under tui-driver's
  single-modal invariant the showing modal's class always matches the tracked one. The
  branch warn-logs and returns **without clearing** the tracking — so if tui-driver ever
  drifts, the correct `Hidden` can still resolve the modal. Marked defensive in-code.

## Out of scope (siblings / deferred)

- **Live producer wiring — [#708]** (this slice blocks it). #708 constructs the surfacer
  with the daemon-singleton `*modalbridge.Registry` (the same instance #717/#725 wire
  into the relay), feeds it `EventKindPtyModalShown`/`Hidden` from the follow-active
  `Session.Events()` stream, supplies the bound session's rendered `screenText`, and
  provides the two-phone e2e capstone. The shared registry instance is what makes the
  cross-head `Resolve` arbitration real in production.
- **The outcome sentinel spelling is non-load-bearing.** The phone treats any
  unrecognised `outcome` as "clear the prompt," so `"dismissed_local"` is the producer's
  choice (chosen for unambiguous standalone audit-log filtering). If #708's live e2e or a
  mobile-side contract pins a different spelling, it is a one-line change in
  `internal/audit`.

## Testing

Scripted, deterministic, `-race`-clean; no live e2e (deferred to #708). All in
`cmd/pyry/interactive_modal_v2_test.go`, reusing `newModalEmitterTestDeps`,
`fakeInteractiveBcast` (its `ActiveConns` reuses the last snapshot across the
Shown→Hidden call sequence), `fakeArmer`, `pushesFor`, plus a `decodeModalDismissed`
helper mirroring `decodeModalShown`.

- **AC3-(a) local-first wins, then remote rejected.** `Handle(Shown, permission)` →
  one `modal_shown` (capture its `modal_id`); `Handle(Hidden, permission)` → exactly one
  `modal_dismissed{outcome: dismissed_local, source: local, modal_id: <same>}`, and
  `reg.Lookup(id)` now misses. Then drive the **real** `ResolveAnswer(id, …)` against the
  same registry → `ok == false`, **no** keystroke recorded, **no** second
  `modal_dismissed`.
- **AC3-(b) remote-first wins, then local emits nothing.** `Handle(Shown)`; capture `id`;
  drive remote `ResolveAnswer`/`ResolveCancel` → consumes; then `Handle(Hidden)` → **no**
  `modal_dismissed` from the emitter, **no** `source=local` audit record.
- **No-outstanding no-op** (`Handle(Hidden)` with no preceding `Shown` → zero pushes, no
  audit); **non-permission no-op** (`Shown`+`Hidden` for a slash-picker class →
  `outstandingID==""` → zero pushes); **defensive class-mismatch** (keeps tracking,
  warn-logs); **#607 capability gate** (non-interactive conns receive zero
  `modal_dismissed`); **audit-on-winner-only** (local-first emits exactly one
  `dismissed_local`/`local` audit line with empty device; remote-first emits none from
  the emitter).

**Code review: PASS** ([PR #734]), **no findings** (no MUST FIX / SHOULD FIX / NIT). The
load-bearing concurrency claim was verified against live code (`ResolveAnswer`'s
consume-before-keystroke ordering), not just the diff. Real-claude e2e green
(`make e2e-realclaude` → ok 159s). `go test -race ./...` + `go vet ./...` pass, `gofmt -l`
clean.
