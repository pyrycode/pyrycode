# Ticket #723 — `dequeue_message` inbound v2 control handler (`QueueRemover` seam → `msgqueue.Remove`)

The **capstone of the queued-backlog wire side** ([EPIC #597] Phase 3, [ADR 025];
split from [#705]). Lets a phone with an interactive session **cancel a queued
`send_message` before it drains to claude**: a paired phone sends a
`dequeue_message` v2 control frame `{conversation_id, queued_msg_id}`; the daemon
removes that not-yet-drained message from the live inbound queue, and the existing
change seam refreshes the phone's `queue_state` view.

Everything this handler stands on had already landed, so it is **~70 lines across
two production files**: the wire types ([#720]/[#705] — `TypeDequeueMessage` /
`DequeueMessagePayload`), the engine op ([#719]'s `msgqueue.Remove`, with the
in-flight-head/unknown-id no-op semantics AC-1/AC-2 rely on), and the `queue_state`
producer over the `OnChange` seam ([#722]). The handler adds **no new exported
types beyond one consumer-declared interface** and wires the one-line engine call.
Held at **S**, no split.

The one **genuinely new** surface: the **inbound `interactive` capability gate**.
Until now `s.interactive` gated only *outbound* fan-out; `dequeue_message` is the
**second** inbound frame (after `interrupt`, [#707]) whose *authorization* is the
interactive capability. `security-sensitive` (an inbound handler that mutates
daemon queue state from an untrusted, internet-exposed phone frame — mirror of
`send_message`); architect spec-stage security review verdict **PASS**.

## Implementation

Two production files; one new package-exported interface; two new test files. All
changes **additive** — the new `V2SessionConfig.QueueRemover` is an optional
nil-safe field (the `Interrupter`/#707, `ModalResolver`/#727, `Snapshotter`/#618
precedent), so every existing construction site compiles unchanged.

| File | Change |
|---|---|
| `internal/relay/v2session.go` | **The only new logic.** The consumer-declared `QueueRemover interface{ Remove(conversationID string, queuedMsgID uint64) bool }` (new sibling of `Interrupter` / `ModalResolver`, so `internal/relay` imports neither `internal/msgqueue` nor `cmd/pyry`), the optional `V2SessionConfig.QueueRemover` field (nil ⇒ inert), the `case protocol.TypeDequeueMessage: m.handleDequeueMessage(s, probeEnv)` arm in `dispatchAppFrame`, `handleDequeueMessage` itself, **and** the reconciled `handleInterrupt` doc comment (see Design note). |
| `cmd/pyry/relay.go` | The wiring. Widen the `queue` parameter of `startRelay` (line 98) and `startRelayV2` (line 283) from `handlers.Enqueuer` to the concrete `*msgqueue.Queue` (+1 import); add `QueueRemover: queue` to the `V2SessionConfig{…}` literal. The concrete `*msgqueue.Queue` (built at `main.go:788`) already satisfies both `handlers.Enqueuer` (unchanged `SendMessage` call) and the new `relay.QueueRemover` (it has `Remove(string, uint64) bool`). **No edit fan-out** — each `startRelay*` has exactly one caller and `main.go` already passes the concrete type. |

### `handleDequeueMessage(s, env)` — the new logic

Intercepted in `dispatchAppFrame`'s v2-control discriminator **before**
`dispatch.Route` (the `interrupt` / `modal_cancel` / `request_snapshot` boundary —
**no** `dispatch.Route` handler). Runs on the manager's **single Run dispatch
goroutine**, so the `s.interactive` read is lock-free under the package's
single-owner invariant. The signature takes **`(s, env)` — no `ctx`** — a
documented deviation from the `(ctx, s, env)` siblings, mirroring `handleInterrupt`:
`Remove` takes no context, there is no `forwardEnvelope`, and `queue_state`
convergence is decoupled onto the #722 emitter goroutine, so the handler owns no
cancellable work. Order is **load-bearing — capability gate first**:

1. **`if !s.interactive` → return** (no `Remove`, no mutation, no panic). The new
   inbound capability gate (AC-3 negative path). `s.interactive` is
   server-authoritative (#626) — set fail-closed from the daemon's
   `negotiateCapabilities`, never from the phone's raw advertisement.
2. **`if m.cfg.QueueRemover == nil`** → debug-log `v2.dequeue.inert`, return
   (foreground / pre-wire; mirrors `handleInterrupt`'s nil-`Interrupter` guard).
3. **Decode tolerantly:** `_ = json.Unmarshal(env.Payload, &p)`. A decode failure
   leaves zero-value fields, which `Remove("", 0)` no-ops on. The decode error and
   the payload bytes are **never** echoed back to the phone or into a log
   (`encoding/json` can quote attacker bytes into its error string — the never-echo
   discipline of `handleRequestSnapshot` / `handleModalCancel`).
4. **`removed := m.cfg.QueueRemover.Remove(p.ConversationID, p.QueuedMsgID)`** — the
   one-line engine call.
5. **Log only content-free discriminants:** `removed` → INFO `v2.dequeue.removed`;
   `!removed` → DEBUG `v2.dequeue.noop`. Both carry only `conn_id`,
   `conversation_id` (a routing id), `queued_msg_id` — never `env.Payload`, never
   queued text (the handler never holds text; `Remove` takes only an id).
6. **No reply, no broadcast.** `dequeue_message` is fire-and-(automatic-
   `queue_state`), like `modal_cancel` / `interrupt`. AC-4's convergence is the
   `OnChange` → #722-producer path that `Remove`'s `notify` fires **on success**; on
   a `false` no-op nothing changed, so emitting nothing is correct. The handler MUST
   NOT `Push` / `forwardEnvelope` / re-emit `queue_state` directly.

**AC-2 framing — `false` is success of a valid request, not an error.** The handler
treats unknown-id, already-delivered, and the in-flight (draining) head
identically: a quiet no-op, never an error reply. `msgqueue.Remove` already encodes
all four no-op cases ([#719], `queue.go:244`); the handler adds no second-guessing.

### Data flow

```
phone ── dequeue_message (noise_msg, AEAD, {conversation_id, queued_msg_id}) ──> relay ──> V2SessionManager.Run
  └─ handleNoiseMsg → dispatchAppFrame (probe decode)
       └─ case TypeDequeueMessage → handleDequeueMessage(s, env)
            ├─ !s.interactive          → return (inert)                 [AC-3 negative]
            ├─ cfg.QueueRemover == nil → debug log, return              [foreground/unwired]
            └─ cfg.QueueRemover.Remove(convID, id)
                 ├─ false → DEBUG v2.dequeue.noop (no reply/broadcast)  [AC-2]
                 └─ true  → INFO  v2.dequeue.removed
                              └─ msgqueue notify → OnChange seam → #722 queueStateEmitterV2
                                   └─ updated queue_state pushed to interactive phones   [AC-1, AC-4]
```

### Design note — the stale `handleInterrupt` comment was reconciled

`handleInterrupt`'s doc previously read "(dequeue is ungated, modal_answer uses the
per-device gate, modal_cancel a nonce)". Since #723 `dequeue_message` **does** gate
on the interactive capability (it just carries no per-device / nonce gate), so the
parenthetical was updated to say so — leaving it would have contradicted the new
code. The two `if !s.interactive { return }` guards stay **inline**: a one-line
bare-capability check shared by two consumers (`interrupt`, `dequeue_message`) does
not warrant a helper abstraction (CODING-STYLE; over-DRY).

## Patterns established

- **The second inbound v2 control frame whose authorization IS a capability — still
  a one-line gate, not an abstraction.** [#707] was the first; #723 is the second.
  Each existing inbound control authorizes differently (`modal_cancel` consumes a
  nonce, `modal_answer` uses the per-device gate #702), and now two frames share the
  bare `if !s.interactive` check. **Two consumers still do not justify a shared
  inbound-gate helper** (CODING-STYLE over-DRY); the architect's spec explicitly kept
  both checks inline. This is the deliberate counterpoint to the YAGNI note #707 left
  ("interrupt is its only consumer") — a second consumer arriving did *not* flip the
  decision.
- **Per-payload `convID` scoping is the security boundary — deterministic code, not a
  stochastic gate.** `msgqueue.Remove(convID, id)` looks up `c := q.convs[convID]`
  and only ever mutates `c.items`; it never iterates other map entries, so
  `Remove(A,…)` provably cannot touch conversation `B`'s backlog. A FIFO for `A`
  exists only because a `send_message` for `A` passed `router.Route` validation at
  enqueue time. The handler passes the untrusted `conversation_id` **verbatim** to
  `Remove` and relies on that scoping — no second membership check. (See § Security.)
- **The optional nil-safe `V2SessionConfig` seam is what holds the slice at S.** A
  fourth instance of the `Snapshotter`(#618) / `ModalResolver`(#727) /
  `Interrupter`(#707) precedent: the relay declares the one-method interface it
  consumes (consumer-side, CODING-STYLE "define interfaces where consumed"), the
  composition root supplies the concrete impl, and the optional field (nil ⇒ inert)
  keeps every existing construction site compiling unchanged.
- **The interface is named for its relay-domain role; the method keeps the engine
  op's name.** `QueueRemover` (role) with method `Remove` (the existing
  `msgqueue.Queue.Remove`, #719) — so `*msgqueue.Queue` satisfies it with **zero new
  msgqueue code**, mirroring `Interrupter.SendEsc` / `ScreenSnapshotter.ScreenSnapshot`.
- **Convergence is automatic, never re-emitted.** AC-4 is satisfied entirely by the
  existing `Remove → notify → OnChange → #722 producer` path; the handler emits
  nothing. Pinned by a unit test (`Remove` that fires no `OnChange` ⇒ handler pushes
  nothing). The `queue_state` producer remains the **sole** `queue_state` emitter.

## Security (`security-sensitive` — architect spec-stage review, verdict PASS)

**Trust boundary.** `dequeue_message` is an AEAD-decrypted frame from a paired,
Noise-authenticated phone; `conversation_id` and `queued_msg_id` are **untrusted
phone input**.

- **Cross-conversation mutation — mitigated structurally.** A hostile
  `conversation_id` cannot reorder or drop another conversation's backlog:
  `Remove`'s `convID`-scoping confines the effect to that one FIFO (which only exists
  from a `Route`-validated `send_message`). The worst case is naming one of the
  **operator's own** conversations (all paired devices for a daemon share one
  conversation space — no foreign tenant) → dropping one of the operator's own queued
  messages. No privilege boundary crossed.
- **No `KnownConversation`/`Route` gate on `dequeue_message` — a deliberate, reviewed
  choice.** Its siblings validate `conversation_id` for reasons this frame
  structurally lacks: `request_snapshot` *renders content* (leak risk),
  `send_message` *routes a delivery* (mis-route risk). `dequeue_message` only mutates
  the named conversation's **own existing** backlog and returns **nothing** — the
  `convID`-scoping is self-validating. Adding a membership gate would defend an
  unobserved failure mode (**Evidence-Based Fix Selection**) **and** risk a
  false-negative (a conversation can hold a backlog while momentarily absent from a
  registry snapshot → would silently drop a legitimate dequeue). The deterministic
  `Remove` scoping is the belt; a stochastic membership check would be a second cloth
  of the same weave (**Belt-and-Suspenders → different fabric**).
- **In-flight cancellation.** A `dequeue_message` naming the draining head must not
  cancel an in-flight delivery; `Remove` no-ops on `idx == 0 && c.draining`, atomic
  w.r.t. the drain under `q.mu` (the [#719] guarantee; the handler adds nothing).
- **Capability bypass.** `if !s.interactive { return }` is the **first** statement,
  before any decode or `Remove`, so a non-interactive conn cannot mutate the queue
  even with a well-formed payload. Pinned by a unit test.
- **Information disclosure.** The handler never returns content and never logs
  `env.Payload` or any queued text (it never holds text). Logs carry only `conn_id`,
  `conversation_id` (routing id), `queued_msg_id`.

## Lessons learned

*(Lifted from the PR #737 Lessons-learned note.)*

- **The AC-5 "no live claude" deterministic backlog needs a specific harness shape.**
  It is realised by `StartInWithEnv` (default `/bin/sleep 99999` child — never
  commits a turn, so the backlog persists) **+ pre-creating `<uuid>.jsonl` in the
  daemon's *computed* sessions dir** (`<HOME>/.claude/projects/encode(workdir)`,
  no env override) so `reconcileBootstrapOnNew` pins the bootstrap session id for
  `seedBoundConversation` — `send_message` rejects pre-enqueue without a bound
  conversation.
- **Dequeue a NON-head, or AC-5 silently no-ops.** Under sleep-claude the **first**
  enqueued message is the perpetually in-flight (draining) head and is an un-removable
  no-op (`Remove → false`). A test that enqueues one message and dequeues it sees a
  silent no-op and fails. Enqueue ≥ 2 and dequeue a non-head (`msg2`); the surviving
  backlog is `[msg1]` in order.

## Testing

**Unit (white-box, `package relay`) — `internal/relay/v2session_dequeue_test.go`
(new).** A `fakeQueueRemover` (mutex-guarded `(convID, id)` call recorder +
programmable bool return, mirroring `fakeInterrupter`), driving everything through
the **real `Frames`/`Run` loop** with the #727 helpers (`startManager`,
`openModalConn`, `sealAppFrameConn`). The fire-and-no-reply handler is synced with a
**barrier conn opened after the dequeue frame** — the single FIFO `Frames` + single
`Run` goroutine guarantees the dequeue is fully handled before the barrier opens
(the reusable idiom #707 established):

- `TestV2Session_DequeueMessage_RemovesByCapability` — table-driven: interactive +
  `Remove → true` (AC-1) and interactive + `Remove → false` (AC-2) both call `Remove`
  **exactly once** with the decoded `(conversation_id, queued_msg_id)`; non-interactive
  (`caps=nil`) calls `Remove` **zero** times (AC-3). Every case asserts the handler
  pushes **no** app frame to the conn (no reply / no broadcast, AC-2/AC-4).
- `TestV2Session_DequeueMessage_NilRemoverInert` — nil `QueueRemover`, interactive
  dequeue → no panic, manager keeps serving (barrier-open is the proof).
- `TestV2Session_DequeueMessage_DecodeTolerant` — a payload malformed as a
  `DequeueMessagePayload` (`queued_msg_id` a string) but valid JSON (so the type
  switch routes) → no panic, `Remove` reached with zero-value fields, and a unique
  marker in the raw payload **never appears** in a captured `bytes.Buffer` logger
  (the never-echo assertion).
- `TestV2Session_DequeueMessage_InterceptedNotRouted` — a sentinel `dispatch.Handler`
  registered under `TypeDequeueMessage` **never fires**, proving interception in
  `dispatchAppFrame` happens **before** `dispatch.Route`.

**E2E (`package e2e`, build tag `e2e`) — `internal/e2e/relay_v2_dequeue_test.go`
(new).** `TestRelayV2_DequeueMessage_RemovesQueuedBeforeDrain` is the AC-5 capstone
(the original #705 AC): spawned-daemon v2 Noise harness +
`driveHandshakeToOpenDaemonInteractive`, sleep-claude (no live claude). Pair → seed a
**bound** conversation → enqueue **two** `send_message` frames → observe a
`queue_state` with both in FIFO order → `dequeue_message{queued_msg_id: msg2.id}`
(the **non-head**) → observe an updated `queue_state` of `[msg1]` only (msg2 removed,
order preserved) — and **no error envelope** (AC-2 forbids it).

## Out of scope (flagged, not designed here)

- **No `dequeue` ack.** The slice sends the phone no positive acknowledgement of a
  successful dequeue beyond the `queue_state` it will receive — matching
  `modal_cancel`'s fire-and-broadcast posture. A correlated ack, if the mobile client
  later needs one, is a follow-up wire addition.
- **Reconnect snapshot of `queue_state`.** `queue_state` carries no `event_id` and is
  off the #647 replay ring; a phone reconnecting after a missed dequeue sees the
  backlog only on the next change (cf. [#722] Out of scope).
- **Per-connection conversation isolation.** The `queue_state` fan-out reaches every
  interactive conn (each payload stamped with its `conversation_id`); there is no
  connection→conversation output binding (cf. [#679]), consistent with ADR 025's
  ungated-for-any-paired-phone model.
- **Inbound queue bound / backpressure.** The unbounded inbound-queue growth vector
  is the [#704]/[#721] deferred concern (chokepoint pre-specified at
  `msgqueue.Enqueue`); this handler only *removes*, adding no growth surface.
