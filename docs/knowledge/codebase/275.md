# Ticket #275 — `internal/protocol`: `RegisterPushTokenPayload` (first slice of #256 catalog)

Phase 3 Track C — first per-type payload struct slotting into `Envelope.Payload (json.RawMessage)`. Pure DTO matching `docs/protocol-mobile.md` § Message types → `register_push_token` verbatim: three required `string` fields (`platform`, `token`, `device_name`) with snake_case JSON tags, no methods, no constructors, no validation. Split from the umbrella payload-catalog ticket #256; sibling slices land separately for the remaining 15 v1 type discriminators.

## Implementation

- **One production file, one fixture, one test, stdlib-only.** `internal/protocol/push.go` (~18 LOC, struct + doc comment) + `internal/protocol/testdata/register_push_token.json` (single-line compact JSON, copied verbatim from the spec example) + `internal/protocol/push_test.go` (~33 LOC, one round-trip test). Production diff is the entire `RegisterPushTokenPayload` struct, no helpers, no `Validate()`, no `String()` — the ticket explicitly forbids them, and per the [feature doc](../features/protocol-package.md) the dispatcher (#248) is the validation point.
- **`Platform` stays `string`, not an enum type.** Spec defines two values (`"fcm"` Android, `"apns"` iOS); a Go enum would force a converter at every internal call site for no observable wire-format gain, and validation belongs at the dispatcher. The convention matches the rest of `internal/protocol` (wire-string `Code*` / `Type*` constants are `string`-typed, not custom enums).
- **All three fields required: no `omitempty`, no `*T`.** Mirrors `Envelope`'s discipline for its required fields (`id`, `type`, `ts`, `payload`). The struct is for decode-side use; encode-side absence of a field would surface as zero-value `""` on the wire, which the dispatcher rejects via shape validation. No defensive optional shape for fields the spec marks required.
- **Test reuses `canonical(t, b)` and `readFixture(t, name)` helpers from `envelope_test.go`.** Same package, no import, no duplication. The test shape mirrors `TestEnvelope_RoundTrip_Full`: unmarshal full envelope → assert `Type == TypeRegisterPushToken` (uses the constant, not a hardcoded string) → unmarshal `Envelope.Payload` into `RegisterPushTokenPayload` → field-by-field assertions against the spec example → re-marshal envelope → `bytes.Equal(canonical(out), canonical(raw))`. Decode-from-`Envelope.Payload` (not decode-from-raw-payload-bytes) is the AC-mandated path because it exercises the exact composition the dispatcher will use: read `Envelope.Type` from the outer parse, then second-pass `json.Unmarshal` against the per-type struct.
- **Fixture `ts` reuses the same `2026-05-08T10:33:14.012Z` as sibling fixtures.** No semantic constraint on `ts` for this payload, and reusing keeps the `testdata/` corpus internally consistent. `payload.token` is the literal `"f0r..."` from the spec — placeholder in the spec, placeholder here; the wire type does not interpret tokens. No `in_reply_to`, no `payload_encrypted` — neither appears in the spec example, and adding either would over-specify what this fixture proves.
- **One golden round-trip test, no unknown-field or missing-field tests.** stdlib `encoding/json` is permissive by default (unknown fields ignored, missing fields zero-valued); ticket scopes those concerns to the dispatcher. The golden round-trip is the contract.
- **File layout: own file (`push.go`), not folded into an existing file.** Architect's discretion call. Keeps the file-to-message-family mapping legible as more payload types land: each future slice from the #256 catalog will own its own `*.go` file in the same package, mirroring the spec's table-of-message-types grouping.

## Out of scope (deferred)

- The other 15 v1 payload structs (#256 umbrella + sibling slices) — `hello` / `hello_ack` / `error` / `ack` / `send_message` / `message` / `list_conversations` / `conversations` / `create_conversation` / `conversation_created` / `promote_conversation` / `conversation_updated` / `backfill_since` / `message_chunk` / `backfill_done`. Each is a separate ticket with its own fixture.
- The `register_push_token` dispatch handler — validates `Platform ∈ {"fcm", "apns"}`, persists `(platform, token, device_name)` to `devices.json`, de-duplicates against the stored triple, returns `ack`. Future ticket.
- APNs/FCM client integration — explicitly out of protocol scope per spec.
- `devices.json` schema changes to persist the push-registration triple — handler's concern, not the wire-type's.
- Enum type for `Platform` — rationale recorded in the architect's spec; converter cost at every call site outweighs the type-safety gain when validation already lives at the dispatcher.
- A `Validate()` method, `NewRegisterPushTokenPayload(...)` constructor, `String()` for logging — none have a use site; logging `Payload` is explicitly forbidden by the [feature doc](../features/protocol-package.md) security posture (may contain tokens).
