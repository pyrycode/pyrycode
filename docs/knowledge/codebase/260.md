# Ticket #260 — E2E happy-path for `pyry update` against fake release server

First e2e coverage of the `pyry update` subcommand (#189) and its #190 daemon-restart wiring. Locks the v0.10.1-shaped supervisor-startup-hang regression class at the release-acceptance tier — the precise evicted-bootstrap reproducer for that incident lives in `internal/e2e/bootstrap_warm_start_test.go` (#253); this ticket complements it by exercising the full fetch → verify → atomic-replace → restart → smoke chain end-to-end against an in-process fake release server. Test-only change: +420 LOC `cmd/pyry/update_e2e_test.go` (new, `//go:build (darwin || linux) && e2e_update`). Production diff: zero.

## Implementation

- **`TestUpdate_HappyPath_E2E` drives `doUpdate` directly with a real running daemon stapled onto the `runRestart` seam.** Phase 1: `buildPyryBinE2E` (honours `PYRY_E2E_BIN` for CI prebuild short-circuit) → `os.MkdirTemp("", "pyry-up-")` (sun_path-safe; `t.TempDir()` overflows macOS APFS's 104-byte limit when extended with `/pyry.sock` — same workaround as `internal/e2e/restart_test.go:newRegistryHome`) → install srcBin into `<home>/bin/pyry` → spawn daemon 1 with `cmd.Stdin = nil` so Go's `os/exec` wires `/dev/null` automatically (structural satisfaction of AC#2's "stdin closed" requirement) → `waitForSocketE2E` until the control socket dials. Phase 2: `fakeRelease(t, "v999.0.0", runtime.GOOS, runtime.GOARCH, srcBytes)` + `newFakeReleaseServer(t, ...)` → call `doUpdate` with `Fetcher.BaseURL` / `releaseBaseURL` pointed at the test server, `replace = update.AtomicReplace`, `executablePath = func() string { return targetPath }`, a `RestartProbe` whose platform-discriminant flag (`LaunchdPlistExists` on darwin, `SystemdUnitExists` on linux) makes `DetectRestartCommand` emit non-nil argv so `runRestart` fires. Phase 3: `runRestart` ignores the launchctl/systemctl argv it's handed and stands in for what supervisor-restart would do — `stopDaemonE2E(cmd1)` (SIGTERM → 3s grace → SIGKILL → 1s grace) → `spawnDaemonE2E` from the same `targetPath` (now holding the new bytes via `AtomicReplace`'s rename) → `waitForSocketE2E` for the new socket. Phase 4: assert inode changed (`syscall.Stat_t.Ino` before vs after — `os.Rename` swaps in a fresh inode), PID changed (`cmd1.Process.Pid != cmd2.Process.Pid`), `pyry status` Phase advances past `starting` within 3s, `pyry sessions list` exits 0 within 10s.
- **`runRestart` is a kill-and-respawn stand-in, NOT a launchctl/systemctl exec.** The test never touches the operator's real launchd/systemd — see `docs/lessons.md` § "E2E against the operator's real systemd `--user` / launchd `gui/<uid>`". The probe-makes-argv-non-nil shape is what proves the wiring, not the argv content; the kill-and-respawn shape is what proves the daemon survives a binary swap, not which CLI executable triggers it.
- **Drive `doUpdate` directly from `package main` over a `--release-base-url` flag or `PYRY_RELEASE_BASE_URL` env var.** Adding either to production code would be the "hidden env var added 'just for the test'" pattern lessons.md rejects. Importing the package from `internal/e2e` is structurally impossible because `cmd/pyry` is `package main`. Same-package e2e is the only seam that satisfies both the AC and the env-var-discipline rule. Cost: the `internal/e2e` harness is unreachable, so the spawn/socket-dial/teardown scaffolding is replicated inline (~120 LOC of helpers, all `_E2E`-suffixed to avoid collision with `update_test.go`); same trade `update_test.go` already accepted for `buildTarGzForTest` / `fakeRelease` / `newFakeReleaseServer`.
- **`waitForPhasePastStartingE2E` polls, doesn't single-shot.** `supervisor.Run` sets Phase to Running asynchronously after `onSpawn` fires; the control socket can be dialable a few ms before that. Single-shot `pyry status` after socket-ready would race the Phase update and flake. The poll loop runs `runVerbE2E(... "status")` every 50ms up to a 3s deadline, parses the `Phase:` line via `parsePhaseE2E` (returns `(string, bool)` — distinguishes "no Phase: line" from "Phase: line present but empty"), returns on first non-`starting` value, and `t.Fatalf`s on timeout with the captured stdout. A v0.10.1-shaped hang would never clear within 3s; healthy startup clears in 50–200ms.
- **`spawnDaemonE2E` returns stdout AND stderr buffers; both surface in readiness-failure diagnostics.** `bytes.Buffer` is safe here — the test goroutine reads only after `waitForSocketE2E` returns (or after `cmd.Wait` closes the doneCh). A v0.10.1-shaped hang dumps both streams in the failure message so the operator-style log comes through verbatim.
- **`cmd1Stopped` flag avoids double-stop in `t.Cleanup`.** The closure-side stop in `runRestart` and the `t.Cleanup`-side stop both target the same `cmd1`; the boolean ensures only one fires. Pairs with the `cmd2 == nil` guard in the post-test cleanup that handles the "runRestart never fired" case (where `doUpdate` errored before the restart step). `stopDaemonE2E` is also internally idempotent (short-circuits on `<-doneCh`), so the guards are belt-and-suspenders.
- **No new harness infrastructure.** Pre-existing `update_test.go` helpers (`fakeRelease`, `newFakeReleaseServer`, `buildTarGzForTest`) are reused verbatim — they live in the same `package main` with no build tag, so the tagged file picks them up automatically. The AC's "extracted into a reusable helper" criterion is met structurally; no textual extraction is required. Zero new dependencies, zero CLI surface, zero wire-format change.
- **No new ADR.** Architect's spec § "Open questions" is "None"; every design choice is anchored in an existing precedent (`update_test.go`'s helper-reuse, `internal/e2e/restart_test.go`'s sun_path-safe HOME, `internal/e2e/harness.go`'s childEnv shape, lessons.md's env-var-discipline + don't-touch-real-launchd rules).

## Out of scope, by design

Failure-path tests (separate ticket — checksum mismatch, fetch failure, restart failure, permission-denied `AtomicReplace`, etc., all leaning on this ticket's `fakeRelease` / `newFakeReleaseServer`). Cross-architecture updates. A different `Version` string for the served binary (the happy path needs only "different inode + working binary," not behavioural drift). Refactoring `update_test.go`'s helpers into a new file or package (the AC's reusability requirement is met structurally — same package, no build tag — without textual extraction). Touching the operator's real launchd/systemd. Adding a `--release-base-url` flag or `PYRY_RELEASE_BASE_URL` env var to production code. CI tag wiring (existing `e2e_install` job pattern is the template; ops adds an `e2e_update` invocation when ready).
