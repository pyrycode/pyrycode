# Ticket #478 — agent-run: ptyrunner JSONL tail + stream-json emit + end-of-turn

Extends `internal/agentrun/ptyrunner/runner.go` in-place to wire the per-session JSONL tail and stream-json re-emit on top of [#471](https://github.com/pyrycode/pyrycode/issues/471)'s spawn + idle-wait + WritePrompt scaffolding. Sub-issue of [#329](https://github.com/pyrycode/pyrycode/issues/329) (tracking). Siblings: [#469](https://github.com/pyrycode/pyrycode/issues/469) (trust + settings helpers), [#475](https://github.com/pyrycode/pyrycode/issues/475) (slimmed workdir trust pre-write), [#470](https://github.com/pyrycode/pyrycode/issues/470) (`cmd/pyry/agent_run.go` cutover, blocked on this + the follow-up budget slice), and the follow-up budget + watchdog slice (no GitHub number assigned yet at the time this doc is written). The slice's scope is narrow: no new files, no new exported types or funcs, one new field on `Config` (`HomeDir` test seam) plus one promotion (`Stdout` now required). The dispatcher contract is unchanged downstream of `streamjson.Emitter` — zero edits in `cmd/pyry/`, `internal/dispatch`, or anywhere else that reads stream-json.

## Implementation

- **`runner.go` package doc narrows the forbidden-imports list.** The 471-era invariant forbade importing `internal/supervisor` AND every sibling agentrun subpackage (`jsonl`, `streamjson`, `budget`). This slice consumes three of the four siblings (`jsonl`, `jsonl/tail`, `streamjson`), so the doc now forbids only `internal/supervisor`. The verifying command becomes a plain `grep pyrycode/internal/supervisor` (no `-E` regex; one target). A new sentence in the logging-discipline paragraph notes that the wired `jsonl/tail` watcher and `streamjson` emitter inherit the same "never log Event content" discipline. The `MaxTurns`/`Stdout` forward-compat enumeration sentence is gone (this slice consumes `Stdout`); the field comment on `MaxTurns` retains its forward-compat marker but now points at "the follow-up slice on top of #478" rather than the now-superseded #472.

- **`Config` gets one new field (`HomeDir`) and one promoted field (`Stdout`).** `HomeDir` is a test seam — when non-empty it overrides the home dir the `jsonl/tail` watcher uses to compute `~/.claude/projects/<encoded>/<sid>.jsonl`. Production callers leave it empty; the watcher falls back to `os.UserHomeDir()`. `Stdout` flips from "declared but not consumed" to required — `streamjson.Emitter` writes per-event stream-json lines and the trailing `result` trailer there. `Stdout required` joins the nine other validation messages at the top of `Run`. Field positions: `HomeDir` slots between `Stderr` and `Env`, matching the spec; no positional reordering of existing fields, so struct-literal callers in tests stay valid.

- **`Run` extends in-place after `sess.WritePrompt`.** Order of operations:
  1. `streamjson.New(streamjson.Config{Writer: cfg.Stdout, SessionID: cfg.SessionID, Logger: logger})` — wrap a `streamjson.New` failure as `fmt.Errorf("ptyrunner: emitter: %w", err)`. The error is defensive (`Writer` is non-nil post-validation, `SessionID` is non-empty post-validation), but the wrap is cheap insurance.
  2. `defer func() { _ = emitter.Close() }()` — registered AFTER the existing `defer sess.Close()`, so defer LIFO runs emitter.Close FIRST. emitter.Close's return is discarded; if Emit was failing, the dispatcher already sees a broken stream and there's nothing operator-actionable to add. The sess.Close defer keeps its Warn-level log on non-nil return (unchanged from #471).
  3. `var emitErr error` declared in the enclosing scope — captures the first sticky Emit failure from inside the OnEvent closure. Single writer (the watcher goroutine), single reader (the post-Run-return composition step); no mutex needed because the read happens AFTER `watcher.Run` has returned.
  4. `tail.New(tail.Config{Workdir: cfg.WorkDir, SessionID: cfg.SessionID, HomeDir: cfg.HomeDir, OnEvent: ..., OnEndOfTurn: ..., Logger: logger})` — wrap a `tail.New` failure as `fmt.Errorf("ptyrunner: tail: %w", err)`. The watcher owns `agentrun.EncodeProjectDir`, the project-dir MkdirAll, the fsnotify arming, the bounded existence-wait (`probeRetryDelays`), and the drain loop; ptyrunner provides callbacks only.
  5. `watcher.Run(ctx)` — blocks synchronously in `Run`'s goroutine until EOT or ctx-cancel.
  6. Return composition: `ctx.Err() != nil → nil` (collapses watcher ctx.Err, in-flight Emit failures during teardown, broken-pipe writes from a dispatcher closing its end); else `emitErr != nil → wrap as "ptyrunner: emit: %w"` (prioritised over a non-ctx `runErr` because emit failure is operator-actionable while the watcher likely returned cleanly at EOT regardless); else `runErr != nil → wrap as "ptyrunner: tail: %w"`; else nil.

- **Cleanup ordering — emitter.Close BEFORE sess.Close — is enforced by defer LIFO, not by an explicit close call.** Go's defer fires in reverse-registration order, so the implementation reads top-to-bottom as "register sess.Close defer first, register emitter.Close defer second" and the runtime correctly runs emitter.Close first. The load-bearing property is that the `result` trailer reaches `cfg.Stdout` before the claude child is SIGTERMed: the dispatcher reads the trailer as the "this turn ended" signal, and flushing it before the child takes the SIGTERM grace window means the dispatcher sees a complete record even if the child takes the full 3s grace to exit. If a future refactor re-registers the emitter.Close defer earlier (or moves the emitter construction higher), it will silently reverse this ordering — leave the comment block in `runner.go` lines 277-282 intact.

- **`OnEndOfTurn` is a no-op closure.** The emitter's `EndOfTurnSeen` state is set inside `Emit` whenever `ev.EndOfTurn == true`, so the default-classification path in `Emitter.Close` produces `ExitReasonCompletion` automatically — no explicit `SetExitReason(ExitReasonCompletion)` call is needed (per the AC's "Rely on streamjson.Emitter's default exit-reason without an explicit SetExitReason call"). The callback exists only because `tail.Config.OnEndOfTurn` is required (will fail validation if nil). The follow-up budget slice may wire `OnEndOfTurn` to signal an early-return path for turn-count tallying; nothing in this slice precludes that.

- **`OnEvent` captures the first Emit error in a closure-bound `emitErr`.** `tail.Config.OnEvent` is `func(jsonl.Event)` (no error return), but `Emitter.Emit` returns an error. The wrapper closure stores the FIRST non-nil Emit return into `emitErr` and lets subsequent Emit calls no-op naturally (Emit is sticky internally — once an underlying write fails, subsequent Emit calls return nil so the watcher can drain to EOT without thrashing on a broken pipe). The closure does not need a mutex because the watcher invokes `OnEvent` synchronously from its `Run` goroutine; the test goroutine never touches `emitErr` while watching.

- **Helper `jsonl` mode + stdin-first-byte gate.** The fake-claude grows a `jsonl` mode in `helper_test.go`: write the `❯ ` idle glyph (same as `idle` mode), then on first byte from stdin write `GO_PTYRUNNER_JSONL_BODY` to `GO_PTYRUNNER_JSONL_PATH` (mode 0600). The stdin-first-byte gate is implemented by replacing the existing single-purpose `io.Copy(io.Discard, os.Stdin)` drain with a per-byte read loop that signals a `chan struct{}` on the first successful read. The body-writer goroutine selects on `stdinSeen` (with a 20s safety timeout) before opening the file. The helper also `os.MkdirAll`s the encoded project directory at 0700 — required because `tail.New`'s own MkdirAll has not necessarily run by the time the helper's body-writer fires (the helper's stdin-seen signal can outrace the parent's `tail.New` call). The watcher handles both orderings transparently via its initial-stat + CREATE-event paths.

- **`helperRunCfg` threads `HomeDir` + the JSONL path env vars.** The builder now computes the encoded JSONL path from a per-test `t.TempDir()` home and `agentrun.EncodeProjectDir(workdir)`, sets `cfg.HomeDir` to the same home, and adds `GO_PTYRUNNER_JSONL_PATH` + `GO_PTYRUNNER_JSONL_BODY` env vars so the watcher and helper agree on the path. The `jsonlBody` parameter is empty-by-default for non-jsonl-mode tests; the helper's `mode == "jsonl"` guard means modal/banner/slow_spawn tests never enter the body-writer branch even though the env vars are present.

- **Four new test cases, one row added to `TestRun_MissingRequiredFields`.** New tests: `TestRun_HappyPath_EmitsAndEndOfTurn` (replaces the bare `TestRun_HappyPath` from #471; asserts the verbatim JSONL prefix appears in stdout AND the trailer carries `subtype:"success"` / `terminal_reason:"completed"` / `is_error:false` / `num_turns:1` / `stop_reason:"end_turn"` / `session_id` matches), `TestRun_CtxCancelDuringStream` (uses `noEotBody`, polls stdout until the first emitted line appears to prove the watcher surfaced the event, then cancels ctx and asserts the trailer carries `subtype:"error_during_execution"` / `terminal_reason:""` / `is_error:true`), `TestRun_EmitErrorPropagation` (uses a one-line `failingWriter` as cfg.Stdout, asserts Run returns an error containing both `ptyrunner: emit:` and `simulated pipe broken`). New row in `TestRun_MissingRequiredFields`: `{"no Stdout", func(c *Config) { c.Stdout = nil }, "Stdout required"}` plus a non-nil `Stdout: &bytes.Buffer{}` in the base config so existing rows still validate past the new check.

- **`syncBuffer` is a tiny concurrency-safe `bytes.Buffer` wrapper used only by the ctx-cancel test.** The test goroutine polls `stdout.Bytes()` while the emitter writes from the watcher goroutine; the race detector would flag a bare `*bytes.Buffer` here. Defined inline in `runner_test.go` (no separate file); ~12 lines total. The other tests pass `*bytes.Buffer` directly because Run is synchronous from the test's perspective — the polling pattern is only needed for the ctx-cancel-during-stream case.

## Patterns established

- **Wire a sibling-package side effect into a spawn primitive via defer-LIFO Close ordering.** When a spawn primitive (`ptyrunner.Run`) composes a write-side primitive (`streamjson.Emitter`) with a process-lifecycle primitive (`tuidriver.Session`) and the write-side primitive's trailer needs to flush BEFORE the process is SIGTERMed, register the process-lifecycle Close defer FIRST and the write-side Close defer SECOND. Go's LIFO firing order means the write-side closes first at any return path (clean, ctx-cancel, error), so the trailer reaches the dispatcher before the child gets SIGKILLed at the end of the grace window. The follow-up budget slice will compose a Counter + watchdog into this same primitive — the pattern says the Counter's Close (if any) should slot above the emitter's, and the watchdog goroutine's shutdown signal should be sent before any of the Close defers fire. Pinned here so the next slice doesn't reinvent it.

- **Capture Emit-errors in a closure when the producer's signature has no error return.** When ptyrunner composes a write-side primitive whose Write/Emit method returns an error with a watcher whose callback signature has NO error return (e.g. `tail.Config.OnEvent func(jsonl.Event)`), wrap the write-side primitive in a closure that stores the FIRST non-nil error into an enclosing-scope variable and surfaces it AFTER the watcher's Run returns. The watcher continues draining (which is correct — the write-side primitive is sticky-error and no-ops subsequent calls), and ptyrunner sees a deterministic "did Emit ever fail?" answer at the composition site. Generalises whenever ptyrunner composes a write-side primitive with a watcher-side primitive whose callbacks are error-less.

- **Helper stdin-first-byte gate decouples helper-write timing from parent-write timing.** When a `TestHelperProcess`-style fake-child needs to perform a side effect (here: writing a JSONL file) AFTER the parent has reached a specific lifecycle point (here: `WritePrompt`), gate the side effect on the helper's stdin first-byte signal rather than a `time.Sleep`. The PTY's stdin/master pipeline guarantees the helper sees bytes only after the parent's WritePrompt fires, which is the lifecycle point we actually want to observe. Sleeps are fragile; first-byte gates are deterministic. The body-writer also MkdirAlls the parent directory before opening, so the helper is robust against the ordering race with the parent's own MkdirAll inside `tail.New`.

## Lessons learned

- **`tail.New`'s MkdirAll races with helper-side file open under the PTY-driven WritePrompt → stdin → helper-write pipeline.** First test run failed because the helper's body-writer goroutine opened the JSONL path BEFORE the parent's `tail.New` had MkdirAll'd the encoded project directory. The error surfaced as "no such file or directory" from `os.OpenFile`. Resolved by adding `os.MkdirAll(filepath.Dir(path), 0o700)` in the helper before the OpenFile call. Both orderings are now valid — if the helper makes the dir first, `tail.New`'s MkdirAll is a no-op; if `tail.New` makes the dir first, the helper's MkdirAll is a no-op. The watcher's initial-stat + CREATE-event paths handle both file-existence orderings transparently, so the test is order-insensitive. Pattern reinforced: when two goroutines/processes share a directory tree, both creators should MkdirAll defensively rather than rely on the other's ordering.
