# Ticket #381 — `e2e/realclaude`: per-dispatcher-role smoke tests

Fourth consumer of the `internal/e2e/realclaude` fixture trio (#372 + #373). Adds five named smoke tests — one per dispatcher role (`po`, `architect`, `developer`, `code-review`, `documentation`) — that exercise each role's `(allowedTools, system-prompt shape)` combination end-to-end against the real claude binary.

Motivated by the failure mode the existing suite does NOT cover: per-role drift. The 2026-05-14 `/doctor` prompt-poisoning bug failed all five roles together; the next failure of this class will likely hit one role only (a single role's allowed-tools list grows or shrinks, a single role's system prompt changes shape, a single role's effort tier flips). Five separate top-level functions give five independent pass/fail signals on the nightly board, so any single-role regression is attributable on sight. No production-code change.

See [features/e2e-realclaude.md](../features/e2e-realclaude.md) for the suite contract.

## Implementation

- **`internal/e2e/realclaude/per_agent_test.go`** (~174 LoC, new). `//go:build e2e_realclaude`, `package realclaude`. One file-private `dispatcherBaseTools` `[]string` constant + one `dispatcherAllowedToolsForRole(role string) []string` helper + one shared `runRoleSmokeTest(t, role, systemPrompt, userPrompt)` body + one `truncate([]byte) string` formatter + five top-level tests (`TestRealClaude_PO_RoleLoop`, `TestRealClaude_Architect_RoleLoop`, `TestRealClaude_Developer_RoleLoop`, `TestRealClaude_CodeReview_RoleLoop`, `TestRealClaude_Documentation_RoleLoop`). No new fixture surface, no new exports.
  - **Five named functions, not a table loop.** The "do NOT factor into a table-driven loop" AC is binding. A table loop collapses five rows on the nightly board into one (with sub-test names buried in `-v` output); five top-level functions give five independent rows. The five system prompts and five user prompts also genuinely differ per role — table-driving them would obscure that.
  - **Shared `runRoleSmokeTest` helper is NOT the same as a table loop.** Each `Test…_RoleLoop` is its own top-level function calling the helper with role-specific arguments; failure attribution still surfaces per-role. The helper exists so a single bug in the **shape** of the five identical assertion blocks (zero exit, non-empty session id, trailer, last assistant event) is fixed once rather than five times. The spec was silent on whether to extract this; the developer chose extraction and the code-review accepted it.
  - **`dispatcherBaseTools` mirrors `agents/dispatcher/src/dispatch.ts:1183`** from the sibling `agent-dispatcher` repo. Twenty-one entries: `Bash`, `Read`, `Write`, `Edit`, `Glob`, `Grep`, `TodoWrite`, four `mcp__qmd__*` (query/get/multi_get/status), two `mcp__context7__*` (resolve-library-id/query-docs), seven `mcp__codegraph__*` (search/callers/callees/impact/node/context/files/status). The cite comment above the constant is the **renumber-detection breadcrumb** per the AC — a future renumber of `dispatch.ts` lands a reviewer at this line via `grep dispatcherBaseTools`.
  - **No figma entries in `dispatcherBaseTools`.** The ticket-body parenthetical mentioned "figma read-only" but the dispatcher source literal at `dispatch.ts:1183` has no figma entries; the source of truth named by the spec wins over the ticket-body parenthetical. Shipping pre-drifted defeats the suite's purpose (see "Lessons learned" below).
  - **`dispatcherAllowedToolsForRole(role)`** returns `dispatcherBaseTools` for `"po"`/`"developer"`/`"documentation"` and `dispatcherBaseTools + "Agent"` for `"architect"`/`"code-review"`. Mirrors `dispatch.ts:1184-1186` (`needsAgent = ["architect", "code-review"]` membership check + conditional append). **Defensive copy** in both branches — `make+copy` for the no-`Agent` case, `make([]string, 0, n+1) + append+append` for the `Agent` case — so a caller mutating the returned slice cannot corrupt the package-level `dispatcherBaseTools`. Unknown role panics with `"dispatcherAllowedToolsForRole: unknown role: <role>"` — typos at the five call sites are programmer errors, not runtime conditions, so the panic is correct (cf. spec's "Alternative considered: `(tools, ok)` return; rejected as ceremony for a 5-call-site helper").
  - **System prompts are deliberate stand-ins, NOT verbatim from `agents/<role>/CLAUDE.md`.** Each is 1-2 lines naming the role and pointing at the tool surface ("You are a Pyrycode product-owner agent. You refine issue bodies. Use Read/Edit/qmd/codegraph for research."). The goal is to exercise pyry's agent-run wiring with dispatcher-shaped inputs, not to validate operator prompts — those are large, frequently revised, and would make this suite both expensive and falsely sensitive to prompt-edit churn. A file-header comment names this contract so a future ticket doesn't "fix" the test by pulling in the real prompts.
  - **User prompts are role-appropriate one-shot tasks** that resolve in ≤4 turns with haiku at low effort, with the response being a single end-of-turn assistant text block (no `tool_use`). Each prompt explicitly requests a constrained reply shape ("Reply with just the rewritten title", "Reply with the function only, no commentary") so haiku doesn't drift into exploratory tool-use that would inflate turn count and break the `last assistant event has EndOfTurn=true` assertion.
  - **`RunOpts` shape (identical across all five tests).** `MaxTurns: 4`, `Effort: "low"`, `Model: "claude-haiku-4-5"`, no `Timeout` override (the 5-minute `RunOpts` default is the runaway guard, not the SLO; haiku + low + single-shot typically resolves in under 30 seconds). Production dispatch uses `model: "opus"` and `effort: "high"`; this suite deliberately downgrades to haiku/low for cost (~$0.10 total/run with cache warm). The downgrade is the cost/coverage trade explicitly accepted in the ticket — if a future ticket adds nightly opus coverage it should be a separate file/suite, not an additive flag here.
  - **Assertions per test (five, in order).** (1) `result.ExitCode == 0`. (2) `result.SessionID != ""`. (3) `parseResultTrailer(result.Stdout)` succeeds. (4) `trailer.PermissionDenials == nil || len(*trailer.PermissionDenials) == 0` — the pointer-vs-empty-slice distinction from #376 is preserved; do NOT collapse to `len(deref) == 0` without the nil-check (panics on absent-field). (5) `trailer.NumTurns >= 1` — the realistic minimum for a single-shot no-tool prompt; #376's `>= 2` is tool-loop-specific. (6) Walk JSONL events, find the LAST `assistant`-kinded event, assert `EndOfTurn == true && TextChars > 0`. Each failure embeds truncated stdout/stderr (1 KiB cap, suffixed `... (truncated)`) or the resolved JSONL path so the operator can diagnose without re-running.
  - **`truncate([]byte) string` is file-local.** A 1 KiB cap formatter used in every failure message. The pattern (cap at 1024, suffix `... (truncated)`) is already established in `tool_loop_test.go` and `allowed_tools_enforcement_test.go`; the helper exists here because the five assertion blocks each invoke it 2-3 times and the in-line `if len(b) > 1024 …` ternary would clutter the body. Not promoted to `fixtures.go` — single-file consumer, YAGNI bar not met.
  - **`t.Parallel()` is NOT called.** Matches the existing realclaude tests. Each test spawns a real claude subprocess and incurs API cost; serialising keeps cost predictable and avoids API-side rate-limit interactions. Five tests × ~30 s each × one warm haiku call each = ~150 s total wall time per `make e2e-realclaude` run, well under any operator's patience threshold.

The spec explicitly says "Both are package-private but live in the SAME test package (`realclaude`) — you can call them directly from `per_agent_test.go`. Do NOT duplicate them." Honoured; this file calls `parseResultTrailer` directly. No fixture-export widening.
- `realclaude.jsonlPathFor` (#364) — file-package-private path-recompute helper for failure-message diagnostics. **Fourth consumer.** The YAGNI bar for promoting to an exported fixture helper rose another notch but is still package-private; promote only when a sibling-package test needs the same lookup.
- `realclaude.JSONLEntry` alias (#372) — used to typed-iterate `events []JSONLEntry` from `ReadJSONL`.

## Patterns established

- **Per-axis test for cross-cutting regression classes.** When a bug affects multiple roles/configurations *together* (single shared code path), the next-failure of that class is one role/configuration *in isolation*. Add one named test per axis value, not a single shared test. Generalises to any multi-role/multi-config surface that has a shared code path today but might drift to per-axis tomorrow.
- **Mirror dispatcher constants by-hand with a cite comment as the renumber breadcrumb.** When a Go test needs to know a TypeScript dispatcher's allowed-tools list (cross-repo, cross-language), the right call is a hand-mirrored constant with a `// mirrors agents/dispatcher/src/dispatch.ts:NNN` cite. The cite is the seam a `grep <constName>` lands a reviewer at after a renumber. Cross-repo CI cross-check is out of scope (see Out of scope) — the cite is the deliberate review breadcrumb in the meantime.
- **Defensive-copy returned slices when the source is package-level shared state.** Helpers like `dispatcherAllowedToolsForRole` that read a package-level `[]string` and may return it directly must `make+copy` before returning. Without this, a caller's `append` past capacity is fine, but a caller's index-mutation corrupts every subsequent call's result. Generalises to any helper returning a slice derived from package-level constants.
- **Shared sub-helper for assertion-block shape is NOT a table loop.** When N top-level tests have identical assertion bodies with N varying inputs, extracting the body into a `runXSmokeTest(t, ...)` helper called by N distinct `Test…` functions preserves per-test pass/fail signal AND consolidates the assertion shape. Distinct from a `for _, tc := range cases { t.Run(tc.name, …) }` table loop, which collapses N signals into one (or buries them in `-v` output). Generalises to any per-axis suite with identical assertions and varying inputs.

## Lessons learned

- **Treat ticket-body parentheticals as suggestive, not authoritative; the spec's named "source of truth" wins.** First-pass implementation pulled three figma read-only entries (`get_design_context`, `get_screenshot`, `get_metadata`) into `dispatcherBaseTools` based on the ticket body's "figma read-only" parenthetical. The spec named `agent-dispatcher/src/dispatch.ts:1183` as the source of truth; the literal at that line has zero figma entries. Code review caught this as a MUST FIX (commit `be2d1a0`); shipping a pre-drifted mirrored constant would have defeated the suite's purpose — the whole point is to catch drift. Generalises: when mirroring a constant from another file/repo, verify against the source literal, not against the ticket's prose summary.
- **Cite comments must point at the actual source-of-truth line, not at the ticket body's stated line.** First-pass implementation cited `dispatch.ts:1190` / `:1191-1193` from the ticket body; the actual lines are `:1183` and `:1184-1186`. Code review fixed both cites in the same MUST FIX (commit `be2d1a0`). The cite is the renumber-detection breadcrumb — landing it on the wrong line means a future `grep` for `baseTools` near the cited line returns nothing and the breadcrumb is broken from day one. Ticket-body line numbers can drift between spec authoring and ticket dispatch; always verify the cited line against the current source before committing.

## Verification

- `go vet -tags=e2e_realclaude ./internal/e2e/realclaude/...` — passes.
- `go build -tags=e2e_realclaude ./internal/e2e/realclaude/...` — compiles.
- `go test -tags e2e_realclaude -run 'TestRealClaude_.*_RoleLoop' -v ./internal/e2e/realclaude/` — all five pass against a working `claude` binary and live API key. **Five real haiku calls per run** (~$0.10 with `--effort=low --max-turns=4`, prompt cache warm; ~30 s per test wall time, ~150 s aggregate).
- `make test` (no tag) — unchanged; the new file is dropped at the build stage by tag exclusion.

## Out of scope (deferred)

- **Cross-repo CI check that `dispatcherBaseTools` matches `agent-dispatcher/src/dispatch.ts:baseTools`.** Today the mirroring is manual and drift goes silent until a role test starts producing surprising tool denials. A future ticket could add a CI cross-repo check (either pyrycode's CI pulling `agent-dispatcher` and diffing, or `agent-dispatcher`'s CI emitting a canonical JSON consumed here). Out of scope per the spec; the cite-comment is the deliberate review breadcrumb in the meantime.
- **Nightly opus coverage.** Production dispatch uses opus/high; this suite uses haiku/low for cost. The cost/coverage trade is explicit. If a future protocol divergence shows up only on opus, a separate file/suite (not an additive flag here) is the right shape — keeps the cheap nightly cheap and the expensive nightly opt-in.
- **Verbatim operator-prompt regression tests.** This suite intentionally uses stand-in system prompts, not verbatim copies of `agents/<role>/CLAUDE.md`. If a future ticket adds operator-prompt regression coverage, it should be a separate file/suite — the cost and churn-sensitivity profile is different.
- **Promoting `parseResultTrailer`/`resultTrailer`/`jsonlPathFor` to exported fixture helpers.** Two consumers each (per ticket) — #376 + #381 for `parseResultTrailer`; #364 + #365 + #376 + #381 for `jsonlPathFor`. Package-private access still suffices. Promote when a sibling-package test needs the same primitive.
- **Promoting `truncate` to a fixture helper.** Single-file consumer in #381; the analogous truncation inline in #376 and #365 is unchanged. Promote when a third or fourth file needs the same cap.
