# Ticket #296 — `net/e2e`: fakephone harness package

Phase 3 Track C, e2e tooling. New `internal/e2e/internal/fakephone` package — a thin in-process WebSocket **client** that speaks the phone half of the mobile↔relay protocol (`docs/protocol-mobile.md` § Authentication, § Message envelope). Sibling of #295 (fake relay server); together they are the two halves of the seam consumed by the still-to-come daemon-side appendix-flow roundtrip test (third split from #254). Ships in isolation — no consumers wired.

## Implementation

- **One production file (`fakephone.go`, ~150 LOC) + one test file (`fakephone_test.go`, ~290 LOC).** Mirrors `fakerelay`'s single-file shape; the architect's ~70 LOC production budget was overrun (every method has a paired Close-race recheck plus the Send/Receive contract docstrings), but no second file would have clarified anything.
- **Surface:** `func Dial(ctx, baseURL, serverID, token, deviceName) (*Client, error)`, `(*Client).Send(env protocol.Envelope) error`, `(*Client).Receive(timeout time.Duration) (protocol.Envelope, error)`, `(*Client).Close() error`. Two sentinel errors (`ErrReceiveTimeout`, `ErrClosed`) — both `errors.Is`-matchable. No `Config` struct, no logger field; failures surface via returned errors.
- **`coder/websocket` matching #247 / #295.** `websocket.Dial` on the client side, `conn.SetReadLimit(maxFrameBytes = 1<<20)` mirrors the 1 MiB inbound cap. The headers (`x-pyrycode-server` / `x-pyrycode-token` / `x-pyrycode-device-name`) are set verbatim via `http.Header.Set` — Go canonicalises on the wire, and `r.Header.Get` is the symmetric reader on the relay side.
- **State.** Two fields: `conn *websocket.Conn` and `mu sync.Mutex` + `closed bool`. No pumps, no channels, no per-conn ctx tree. Each method blocks on its own I/O.
- **Send/Receive race recheck pattern.** Both methods pre-check `closed` under `mu`, release before I/O, and **re-check `closed` under `mu` on the I/O-error path**. This is what maps a "Close races with in-flight call" into `ErrClosed` rather than a wrapped low-level WS error — the I/O fails because Close pulled the conn out, but the caller wants the sentinel.
- **Receive timeout via `context.WithTimeout`.** AC pins `Receive(timeout time.Duration)` (not `ctx context.Context`); the implementation derives a context internally so external cancellation is via `Close()`. Deadline classification checks both `errors.Is(err, context.DeadlineExceeded)` AND `ctx.Err() == context.DeadlineExceeded` since `coder/websocket` may wrap or substitute the error depending on which I/O path tripped.
- **`Close` writes `closed=true` BEFORE force-closing the conn** so an in-flight Send/Receive that observes the I/O error post-Close-flag-write maps to `ErrClosed` via the recheck. Idempotent via the `closed` flag (no `sync.Once` — the bool doubles as the matchable flag plus the read-from-other-methods seam, which `Once` doesn't cover).
- **No `recvCh` / `sendCh` / pump goroutines.** Unlike `internal/transport.Client` (which needs pumps to bridge async lifecycle with sync send/receive across reconnects), this harness is synchronous end-to-end: tests run one goroutine per direction, and `Close` unblocks any in-flight call via the conn-force-close.
- **Tests.** Seven `Test*` functions, all `t.Parallel()`-able, stdlib only. A single `newWSServer(t, behavior, hdrCapture, send)` factory stands up an `httptest.NewServer` that upgrades via `websocket.Accept` and runs one of three scripted server-side behaviours (`behaviorEcho` / `behaviorIdle` / `behaviorSendOne`); `hdrCapture` lets the headers-forwarded test snapshot the upgrade request. Coverage: headers forwarded verbatim (`TestDial_ForwardsHeaders`), envelope round-trips through echo (`TestSend_RoundTripsThroughEcho`, with `time.Time.Equal` comparison per the monotonic-clock-strip rule), single-frame receive decodes (`TestReceive_DecodesEnvelope`), timeout returns sentinel (`TestReceive_TimeoutReturnsSentinel`), Close unblocks in-flight Receive within 500ms + subsequent Send/Receive return `ErrClosed` (`TestClose_UnblocksReceive`), idempotency (`TestClose_IsIdempotent`), handshake error surfaces as Dial error (`TestDial_FailsOnHandshakeError`).

## Spec deviation

- **`TestReceive_TimeoutReturnsSentinel` is single-shot, not the spec's two-shot.** The architect's spec called for "Then call Receive again with a longer deadline after server sends a frame; assert it succeeds (proves the conn is still usable after a timeout)." This was dropped. `coder/websocket` closes the underlying conn when the Read context is cancelled (a documented library invariant: cancelled-Read → unrecoverable conn), so a timed-out Client cannot be reused. The AC ("on deadline exceeded returns a typed error the caller can match against") is still covered; the "conn remains usable" extra clause is library-incorrect. Receive's docstring records the constraint so consumer tests know to construct a fresh Client per Receive attempt if a timeout is plausible.

## Patterns established (or reinforced)

- **`internal/e2e/internal/<harness>/` for in-process test peers** — now three: `fakeclaude` (#122, supervised-child stand-in), `fakerelay` (#295, WS-server stand-in), `fakephone` (this ticket, WS-client stand-in). Double-`internal/` visibility-fences the package to e2e-package consumers; one prod `.go` + one `_test.go` is the standard shape.
- **Pre-check + post-I/O recheck of `closed`** for harness methods that race a `Close` call. Cheap pre-check fast-path rejects calls on an already-closed client; recheck after I/O error maps Close-mid-call to `ErrClosed` rather than the wrapped library error. Two mutex acquisitions per method, both extremely short, neither held across I/O.
- **Timeout-as-duration over timeout-as-ctx** is fine for a harness that owns its own conn — internal `context.WithTimeout` recovers the expressive power, and external cancellation rides `Close()` instead. The spec's AC pinned the duration form; this ticket validates the ergonomics work in practice.
- **Sentinel errors over typed error structs** for harness-test ergonomics. `errors.Is(err, ErrReceiveTimeout)` reads cleaner in tests than a switch-on-type, and the two outcomes (timeout / closed) are flat enough that no additional context fields would help.

## Out of scope (deferred)

- **Roundtrip e2e test** — third ticket. Stands up fakerelay + fakephone, points the daemon's WSS client at the fakerelay, asserts the appendix-flow envelope sequence end-to-end.
- **`hello` / `hello_ack` handshake handling** — envelope sequencing is the consumer test's concern. The harness ships raw Send/Receive.
- **Reconnect, ping pump, backfill state, push-token registration** — application-layer concerns owned by `internal/transport` (heartbeat) and the future mobile-client code.
- **Routing-envelope wrap/unwrap** — the phone speaks raw envelopes; the relay does the wrap (covered in #295).
- **TLS** — `ws://` only.
- **`Config` struct, dial-timeout knobs, ctx-for-Send/Receive, logger field** — add when a consumer ticket needs it.

## Lessons learned

- **`coder/websocket` cancels the underlying conn on Read-context cancellation.** A `Receive(timeout)` that hits the deadline cannot be retried on the same Client. This surfaced in implementation when the spec's "conn remains usable after timeout" sub-assertion couldn't be made to pass without injecting fresh contexts per Read — but `coder/websocket`'s `conn.Read(ctx)` ties conn lifecycle to ctx lifetime by design. The fix was to drop the sub-assertion and document the constraint on `Receive`'s godoc; consumer tests construct a fresh Client per timeout-plausible Receive. This is the same library trade-off `internal/transport` works around with pump goroutines and a long-lived parent ctx; the harness deliberately doesn't pay that complexity.
