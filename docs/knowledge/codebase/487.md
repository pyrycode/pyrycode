# Ticket #487 — agent-run/settings: replace invalid `permissions.defaultMode` literal "deny" with "dontAsk"

Single-string-literal behaviour change in [`internal/agentrun/settings.WriteSettings`](../features/agentrun-settings-subpackage.md): the per-spawn permissions JSON now emits `"defaultMode":"dontAsk"` instead of the invalid `"deny"`. Restores the deny-by-default permission gate that the [Phase A spike](https://github.com/pyrycode/pyrycode/issues/329) accidentally dissolved — the prior literal was guessed rather than read from Anthropic's [agent-sdk/permissions docs](https://code.claude.com/docs/en/agent-sdk/permissions), and claude 2.1.145 silently fell back to `"default"` mode (which happened to refuse the spike's Bash prompt, masquerading as a passing contract). Closes the same-incident `/doctor`-template prompt-injection vector on the [ptyrunner](../features/ptyrunner-package.md) path: when claude rejects the settings JSON at startup, it prepopulates a `/doctor` repair template into the input buffer and processes that instead of pyry's prompt — the [`Session.WritePrompt`](https://github.com/pyrycode/tui-driver) bracketed-paste delivery does NOT preempt this. New build-tagged real-claude regression test asserts the first `user` JSONL entry is the operator's prompt, not the `/doctor` template.

## Implementation

- **Single-literal flip at [`internal/agentrun/settings/settings.go:72`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/settings/settings.go) — `"deny"` → `"dontAsk"`.** Nothing else in the package's structure, validation, error handling, tempfile naming (`pyry-agent-run-settings-*.json` in `os.TempDir()`), or cleanup contract moves. The Go struct tag `json:"defaultMode"` stays — the key name is not affected, only the value. The encoded byte sequence is shape-identical except for the literal: `{"permissions":{"allow":[<tools>],"defaultMode":"dontAsk"}}\n`. Field order (`Allow` then `DefaultMode`) is preserved.

- **`"dontAsk"` preserves deny-by-default semantics per Anthropic's documented contract.** From [`code.claude.com/docs/en/agent-sdk/permissions`](https://code.claude.com/docs/en/agent-sdk/permissions): *"Don't ask mode (`dontAsk`) converts any permission prompt into a denial. Tools that are pre-approved by `allowed_tools`, `settings.json` allow rules, or a hook will still run as normal. Everything else is denied without calling `canUseTool`."* The semantic property ([`agentrun-settings-subpackage.md`](../features/agentrun-settings-subpackage.md) describes it as a "deny-default whitelist") is unchanged; only the literal JSON value moves. Phrases like "per-spawn deny-default permissions JSON", "deny-default whitelist held", and the variable name `canonicalAllow` remain accurate and unchanged.

No structural changes to any test; the empty-input / path-shape / `t.Parallel()` discipline / other pinned substrings (`["Read"]`, `PTY`, `#329`, `#336`, `#470`, `#473`) all stay.

- **New regression test [`internal/e2e/realclaude/doctor_poisoning_regression_test.go`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/doctor_poisoning_regression_test.go) guards against silent recurrence.** Build tag `//go:build e2e_realclaude` (same as siblings); test name `TestRealClaude_DoctorPoisoningRegression`. Runs `pyry agent-run --allowed-tools=Read --max-turns=1 --effort=low --model=claude-haiku-4-5` with prompt `"Reply with the single word: pong"` against real claude via `RunPyryAgentRun(t, RunOpts{...})`, reads the JSONL via `ReadJSONL(t, workdir, sessionID)`, walks events to find the first `Kind == "user"` entry, decodes its `Raw` into a content string (coercing both observed claude shapes: string literal AND `[{type:"text", text:"..."}]` array), and asserts the content does NOT contain the verbatim opening of the `/doctor` template: `"Help me fix the issues reported by /doctor below."` (the substring observed in the ticket's reproduction `out.jsonl`). Defence-in-depth: requires at least one `assistant` event in the JSONL — empty assistant set under a non-poisoned session indicates a different upstream failure that masks the regression-guard's signal.

- **Reuses the existing realclaude fixtures verbatim.** `WithWorktreeAuthenticated(t)` (re-pins HOME, skips on missing `ANTHROPIC_API_KEY` with a named-variable diagnostic), `RunPyryAgentRun` / `RunOpts` / `RunResult`, `ReadJSONL` / `JSONLEntry`, `jsonlPathFor`, the PATH gate inherited suite-wide from [`TestClaudeBinaryAvailable`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/smoke_test.go) — no fixture-package surface change. The Max-auth-only operator scenario falls through the API-key skip (the fixture-wide auth-skip cleanup to add Max-auth detection is explicit OOS, see "Out of scope" below).

- **Malformed-line policy mirrors [`bashInvokedInRaw`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/allowed_tools_enforcement_test.go) at `allowed_tools_enforcement_test.go:74-76`.** A JSONL line that fails to parse into the minimal `{Message: {Content: ...}}` shape is skipped silently — one malformed line must not turn a PASS into an inconclusive. The `decodeUserContent` helper returns `(text, true)` on success and `(zero, false)` on failure or unrecognised shape; on failure the walker simply continues looking for the next `user` entry.

- **Failure diagnostic includes the verbatim user-entry content + JSONL path + actionable hint.** `t.Fatalf` formats: the operator-visible content (truncated to ~512 bytes with `"... (truncated)"` suffix), `jsonlPathFor(workdir, result.SessionID)`, and the direction `"claude is rejecting the per-spawn settings JSON at startup — see #487"`. The content is operator-supplied test data ("Reply with the single word: pong") under a tempdir-pinned HOME, so the diagnostic dump is safe to print.

- **`stop_reason: end_turn` within ~30s is satisfied by the manual reproduction, not the automated assertion.** AC #6 (the reproduction emits a clean `system/init` envelope, delivers the intended prompt, and receives `assistant` event with `stop_reason: end_turn` within ~30s) is verified by the operator running the ticket's reproduction script on their Mac. The regression test deliberately does NOT pin `stop_reason` — that would couple the test to an upstream model-behaviour detail beyond the contract we're guarding (the contract is "first user entry is pyry's prompt", not "model produces end_turn").

## Patterns established

- **Document-the-docs over guess-the-docs for upstream config values.** The Phase A spike (#329) chose `"deny"` by guessing what the value should be named for the desired semantic, then "verified" empirically with a prompt that happened to refuse under the fallback mode — a confirmation that masked the real failure. The fix is to cite the upstream docs explicitly in the package comment AND in the rationale note, so future readers see both the value AND why it was chosen. Generalises: any external system's config literal in pyry source should carry a citation to the upstream documentation that defines the value's behaviour. When the value is wrong, the citation is also wrong, and the discrepancy surfaces during review rather than under a poisoned smoke six months later.

- **Empirical "verified" findings need a positive-control: reproduce the failure mode first.** The spike's "Bash blocked, agent used Read instead" was observed under an invalid config that triggered fallback-to-`"default"` mode — the test couldn't distinguish "the gate held because the contract works" from "the gate held because the fallback happened to refuse this prompt." A positive-control would be: deliberately misconfigure the value, see the gate FAIL, then fix the value and see the gate hold. Generalises: when verifying a deny-default contract empirically, include a deliberately-misconfigured run in the same matrix so the test distinguishes "contract held" from "fallback masked the test."

- **Real-claude regression tests for prompt-injection class bugs go to the same fixture family the other realclaude tests live in.** The ticket spec offered choices between extending the existing `allowed_tools_enforcement_test.go` and writing a sibling. Sibling won because the contract guarded is different (first-user-entry-shape, not gate-held-on-Bash) — the regression-class is "claude rejected pyry's settings JSON and processed `/doctor` instead of the prompt," not "claude executed a forbidden tool." The shared fixtures (`WithWorktreeAuthenticated` / `RunPyryAgentRun` / `ReadJSONL`) carry both tests; each test's failure diagnostic stays narrowly focused on its own contract.

- **Defence-in-depth via "assistant event present" soft-floor.** The `assistantCount == 0` check after the negative-substring assertion catches the case where the test misses the regression because something else upstream broke (auth failure, network error, model refusal of the prompt itself). On a poisoned session there's still a `user` entry (the `/doctor` template) AND likely an `assistant` entry (claude's reply to the `/doctor` template) — so `assistantCount > 0` is a weak sanity check that pre-empts a silent false-pass when the test setup itself is broken. Treated as `t.Fatalf` with diagnostic context, not a silent pass, so a regression class outside #487's intent surfaces with operator-actionable text rather than masquerading as PASS.

- **One-line literal-flip changes are size-XS even under a security-sensitive label.** The architect override of PO's `size:s` to `XS` was correct: the production behaviour change is one character (`y` → `tAsk`), the doc-comment updates are mechanical, the test updates re-pin the new literal, and the new realclaude regression test fits the existing fixtures unchanged. Generalises: a security-sensitive label does NOT inflate scope on its own — what inflates scope is when the security review uncovers a NEW failure mode that needs a new defence. Here the security review was PASS without scope additions, so the size estimate held.

## Lessons learned

- **`/doctor` poisoning is structurally enabled by invalid claude config at startup — NOT specifically by streamrunner OR ptyrunner.** The 2026-05-14 PROJECT-MEMORY claim that the regression was "structurally impossible after Phase B's stream-json pivot (#391)" was wrong in scope: the pivot to stream-json removed the PTY-blind-typing surface AND happened to also remove the invalid-settings entry point for `/doctor` prepopulation (because the streamrunner path uses `--allowed-tools` + `--dangerously-skip-permissions`, not a settings file). The ptyrunner cutover (#470) reopened the door because it reintroduced a per-spawn settings file — and the invalid-`"deny"` literal made the door swing on every spawn. **The correct invariant is: any invalid claude config that survives to startup-time reopens this class of failure.** The defence is to never write a config claude won't accept; the architectural defence (ptyrunner startup-modal detection) is OOS for #487 but tracked as a follow-up.

- **`ptyrunner_byte_equivalence_test.go` (#482) did NOT catch this.** The byte-equivalence test exists to detect drift between the two pipelines' wire-shape outputs and runs the production settings file through real claude — but it was guarded by `WithWorktreeAuthenticated`, which `t.Skip`s when `ANTHROPIC_API_KEY` is unset. The operator's validation Mac uses Max-plan auth (no `ANTHROPIC_API_KEY`), so the test silently skipped during #482's validation gate and the literal-flip was never exercised against real claude. **The fixture-wide auth-skip cleanup** (extending `WithWorktreeAuthenticated` to accept Max-plan credentials, or pairing the API-key check with a Max-auth probe) is the follow-up that closes this gap; it is OOS for #487 and tracked separately. Post-mortem call-out: every test gated by `WithWorktreeAuthenticated` has the same masquerade risk under Max-only environments.

- **The "verified" status in a spike comment can be a `default`-mode mirage.** The #329 Phase A spike's "Bash blocked, agent used Read instead — VERIFIED" finding was published before claude documented its fallback behaviour on invalid `defaultMode`. The finding survived three rewrites of `agentrun/selfcheck` (#336 / #375 / #473) — each rewrite preserved the empirical contract verbatim, none probed whether the underlying literal was valid. Reinforces: empirical contracts pinned by a single prompt's behaviour are fragile in the presence of upstream silent-fallback semantics; pair them with a contract-test that probes the config layer directly (which is what the new regression test does for `/doctor` poisoning).

## Out of scope (follow-ups)

- **Fixture-wide auth-skip cleanup**: extend `WithWorktreeAuthenticated` (and the 9 other realclaude tests that use it) to accept Max-plan credentials, so the byte-equivalence test (#482) and all other gated tests stop silently skipping on the operator's Mac. This is the structural fix for the post-mortem in "Lessons learned" above.
- **ptyrunner startup-modal detection**: even with the dontAsk fix, any future startup-time settings validation error from claude would re-open the `/doctor` poisoning path. ptyrunner should detect "claude is showing a `/doctor` prompt at startup" (PTY signal — UTF-8 anchors in the rendered buffer) and fail fast rather than paste over it. The [`tui-driver`](https://github.com/pyrycode/tui-driver) library may need a startup-modal-class for `/doctor`.
- **Post-mortem doc on the masqueraded test gate**: standalone lesson-doc would name the "test silently skipped because of a credential type the harness doesn't recognise" anti-pattern. Captured here in "Lessons learned" for the per-ticket file, but a cross-cutting note in the realclaude feature doc may be the right home.
