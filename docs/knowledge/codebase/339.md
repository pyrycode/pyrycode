# Ticket #339 — `agentrun.WriteSettings` + `pyry agent-run` settings-file emit

Phase B-1 of #329 (split from #331). Lands the deny-default settings JSON writer and wires `pyry agent-run` to call it after flag parse, replacing the scaffold's "no spawn yet" confirmation line with the dispatcher-scrapable marker `settings-file: <path>`. The file at `<workdir>/.pyry-agent-run-settings.json` is the load-bearing mechanism that gives interactive `claude` the same deny-default permission semantics `-p` mode has natively — `--allowedTools` alone is additive (spike-verified in #329) and does NOT block tools outside the list.

## Implementation

- `internal/agentrun/settings.go` (new, ~80 LOC) — `WriteSettings(workdir, allowed []string) (string, error)` + exported `SettingsFilename = ".pyry-agent-run-settings.json"`. Internal `settingsFile` / `permissions` structs whose field order is load-bearing (must serialize as `{"permissions":{"allow":[...],"defaultMode":"deny"}}`). Nil-slice normalisation to `[]string{}` at entry so the byte-for-byte output uses `"allow":[]` not `"allow":null`. Compact `json.NewEncoder` write (no `SetIndent`) — the dispatcher and `claude --settings` don't care about indentation, but the byte-for-byte test pins something stable. Atomic-write mirrors `internal/devices/registry.go:Save`: `os.CreateTemp(workdir, ".pyry-agent-run-settings-*.tmp")` → `os.Chmod(0o600)` → encode → `Sync` → `Close` → `Rename`, with `defer os.Remove(tmpName)` as best-effort cleanup. Stdlib only.
- `internal/agentrun/settings_test.go` (new, ~140 LOC) — table-driven byte-for-byte shape test (two-tool, single-tool, empty-slice, nil-slice rows), mode pin (`0o600`), overwrite-of-existing-file (asserts post-state bytes + no stray `.tmp` leftover in the workdir), and the missing-workdir failure path (asserts the `agentrun: write settings:` wrap prefix without pinning the OS-specific tail).
- `cmd/pyry/agent_run.go` — `runAgentRun` now calls `agentrun.WriteSettings(parsed.workdir, parsed.allowedTools)` and prints `settings-file: <path>\n` on a line of its own. The scaffold confirmation `fmt.Println` is **replaced**, not augmented — the marker line is the sole stdout contract.
- `cmd/pyry/agent_run_test.go` — appends `TestRunAgentRun_WritesSettingsFile` which redirects stdout via `os.Pipe()`, drives `runAgentRun` with `newValidArgsFixture(t).argv`, asserts (a) the file at `<workdir>/.pyry-agent-run-settings.json` exists with the canonical byte sequence, (b) stdout equals exactly `settings-file: <path>\n`.

## Patterns established

- **Stdout marker contract.** `settings-file: <path>` on a line of its own, single space after the literal prefix. Sibling #332 will scrape this with `^settings-file: (.+)$`. The format is documented both in `WriteSettings`'s package doc-comment (path policy) and inline next to the `fmt.Printf` in `runAgentRun` (format) — the two anchors are deliberate redundancy so neither half drifts without the other being noticed.
- **Compact JSON for machine-consumed config.** No `SetIndent` on `json.NewEncoder`. The trailing `\n` from `json.Encoder.Encode` is part of the golden bytes and IS pinned by the test. If a future operator wants pretty-printed settings for human debugging, the change is one line — but the AC test would need updating; recommendation in spec is to keep compact.
- **Nil-to-empty normalisation at the entry boundary.** A one-line `if allowed == nil { allowed = []string{} }` at the top of `WriteSettings` guarantees the `"allow":[]` (not `"allow":null`) shape regardless of whether the caller passed `nil` or `[]string{}`. The dedicated "nil slice normalises to empty" test row pins this — without the normalisation, `nil` serialises to `null` and the row fails. Pattern: any time a JSON-serialising helper takes a slice from a CLI-flag parse path, normalise at entry rather than trusting the caller to have constructed `[]string{}` explicitly.
- **Path-policy lives in the helper, not the caller.** Filename literal `SettingsFilename = ".pyry-agent-run-settings.json"` is exported as a package-level constant so test code (and any future inspector) can reference the canonical name without re-typing the string. The caller passes only the workdir; the helper joins.
- **Single-package, multi-helper layout.** `internal/agentrun` now holds two orthogonal helpers (`trust.go` for `MarkWorkdirTrusted` + `ResolveWorkdir`; `settings.go` for `WriteSettings`). They share no state; they cohabit because both are "primitives `pyry agent-run` uses to set up claude's environment before spawn." Package doc-comment lives only in `trust.go` (the helper that demands the logging-discipline directive); `settings.go` opens with a bare `package agentrun` so Go doesn't render two paragraphs of package-level docs. Convention: in a multi-file package, whichever file lands second adopts the bare `package` form.

## Lessons learned

- **`encoding/json` serialises `nil` slices as `null`.** A `[]string` field set to `nil` becomes `"key": null`, not `"key": []`. The byte-for-byte AC ("empty slice produces `\"allow\": []` cleanly") would silently fail on a caller passing `nil` if the helper did not normalise. Generalisation: any time the on-disk shape must distinguish "absent" from "empty," construct the empty value at the marshal boundary rather than relying on every caller to have done it.
- **Replacing a stdout line — don't augment.** The scaffold (#337) printed `pyry agent-run: flag set valid; scaffold-only ticket #337 — no spawn yet`. The single most likely regression in this slice was leaving that line in place alongside the new marker, which would have given the dispatcher's regex two candidate matches and the wrong one wins under naive matching. The spec called this out explicitly ("the pre-existing 'flag set valid…' `fmt.Println` is removed") because it's an easy thing to add-instead-of-replace. Pattern: when a slice introduces a new stdout contract on top of a placeholder, the placeholder MUST go in the same commit.
- **Compact vs. pretty-printed JSON: pick one and pin it.** Either is correct; neither breaks consumers. The byte-for-byte test only has value because the chosen shape is pinned. The spec went with compact because the file is machine-consumed (`claude --settings`); the trade-off is documented in "Open questions" so a future "let me pretty-print this for debugging" PR knows the test will fail and that's the test doing its job.

## Out of scope (deferred to siblings)

- `--settings <path>` argument wiring on the supervised `claude` subprocess (#332). This ticket emits the file and prints its path; consumption is the consumer-wiring ticket.
- Boot-time self-check that the schema still enforces deny-default against a live `claude` (#336). Independent of this file's writer.
- `.gitignore` propagation for `.pyry-agent-run-settings.json`. The helper doc-comment notes the filename so the dispatcher's repo can gitignore it; this ticket does not modify any `.gitignore` in `pyrycode/pyrycode`.
- Per-spawn filename uniqueness (PID / nonce suffix). The filename is fixed; concurrent `pyry agent-run` against the same workdir would clobber, but the dispatcher serialises per-workdir. If that changes, a `.pyry-agent-run-settings-<nonce>.json` variant is the obvious follow-up.
