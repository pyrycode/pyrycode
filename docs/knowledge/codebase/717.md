# Ticket #717 — gated remote `modal_answer` resolution (validate, gate, route, dismiss, audit)

The **security-critical** leg of the daemon-side modal bridge ([EPIC #597]
Phase 3, [ADR 025] § Security model): it routes an internet-sourced
`modal_answer` into claude's permission prompt **only** when the answering device
is gated. #727 introduced the relay-side `ModalResolver` seam and shipped its
answer arm as a deliberate **deferred no-op** so the escalating ALLOW path stayed
dead until the per-device gate existed. This ticket *is* that gate: it replaces
the `ResolveAnswer` body and extends one interface — **nothing else**.

## Implementation

**One production file edited: `cmd/pyry/modal_resolve_v2.go`.** No new file, no
new exported type, no new sentinel, no manager change. The relay's
`handleModalAnswer` already broadcasts `modal_dismissed` iff `ResolveAnswer`
returns `ok=true`, so filling the resolver impl was sufficient — that "needs no
manager change" property is the whole point of the seam #727 delivered.

### Interface extension (no consumer cascade)

`modalKeystroker` grew from `SendEsc() error` to the full safe-answer surface —
`SendEsc` / `Answer(choice string)` / `AcceptTrust()`. `*supervisor.Supervisor`
already implements all three (#726, `internal/supervisor/modal.go:52-66`), so the
production wiring `newModalResolverV2(modalReg, sup, logger)`
(`cmd/pyry/relay.go`) keeps compiling unchanged. The only other implementer is the
test `fakeKeystroker`. **Zero production call-site cascade.**

### `ResolveAnswer` decision matrix — gate **before** consume

The hard invariant: *nothing but a fully-authorized, valid answer may consume
(`Resolve`) the modal or route a keystroke.* The order is the load-bearing
security property:

1. **`Lookup(modalID)`** (read, no consume). Stale / unknown / **already-resolved
   replay** all miss here → `(zero, false)`, no side effect. **This is AC-2's
   idempotency / first-answer-wins** (see `answer_token` below).
2. **Fail-closed eligibility gate** `!dev.MayAnswerRemotePermission()` (#702) —
   nil device / unauthenticated / opt-in bit unset → audit `denied_unauthorized`,
   `(zero, false)`. **`Lookup`-only, so the modal is left outstanding** for a
   legitimate local answer or the #725 deny-on-timeout. Runs *before*
   classification and *before* consume.
3. **`classifyAnswer(out, optionID)`** — maps `option_id` → `(outcome, verb,
   choice)` against **this** modal's surfaced options. A forged / wrong-class id
   (not in `o.Options`) → no keystroke, no consume, **no audit** (a malformed
   client frame, not a security decision); `Warn`-logged with the
   length-bounded `option_id`.
4. **`Resolve(modalID)`** (consume first — commits idempotency), then route the
   keystroke **best-effort**, then `AuthorizeRemotePermission(dev, outcome)` (#702)
   splits audit `allowed` (true) from `denied` (false). Returns
   `({option_id, remote}, true)`.

### `classifyAnswer` — positional keystroke is the single source of truth

| `option_id` | class | outcome | verb | `choice` |
|---|---|---|---|---|
| `allow_once` / `allow_always` | permission | `OutcomeAllow` | `Answer` | `strconv.Itoa(idx+1)` |
| `reject_once` / `reject_always` | permission | `OutcomeDeny` | `Answer` | `strconv.Itoa(idx+1)` |
| `proceed` | trust | `OutcomeAllow` | `AcceptTrust` | — |
| `exit` | trust | `OutcomeDeny` | `SendEsc` | — |
| anything else / id not in `o.Options` | — | — | — | `ok=false` |

A permission option's keystroke is its **1-based position in `Outstanding.Options`**
(`slices.IndexFunc` on `ModalOption.ID`), which the producer (#716) builds in
claude's display/selection order. Deriving the digit from the index keeps **one**
source of truth for the option ordering and gives **free membership validation** —
hardcoding `allow_once→"1"` would duplicate the producer's invariant in a second
place that silently breaks on reorder. `answerVerb` is a small unexported enum
(`verbAnswer`/`verbAcceptTrust`/`verbEsc`) mirroring supervisor's `modalKey`;
`routeAnswerKeystroke` switches on it (the same shape as
`supervisor.sendModalKeystroke`).

### Two distinct "outcome" vocabularies — do not conflate

- **Wire** `ModalDismissal.Outcome` = the **`option_id`** (e.g. `"allow_once"`),
  echoed to every interactive phone (`ModalDismissedPayload` contract).
- **Audit** `Entry.Outcome` = the **classification** (`allowed` / `denied` /
  `denied_unauthorized`).

`Source` is `remote` on every row. The audit entry carries only non-secret
identity (`DeviceHash` = `TokenHash`, `DeviceLabel` = `Name`); nil device → empty
fields (the `ResolveCancel` pattern).

### `answer_token` — decoded, not used server-side (by design)

`answer_token` is the **client's** idempotency key (uniqueness matters, secrecy
does not — it is **not** authorization). The daemon's dedup is the `modal_id`
one-shot `Resolve`, which already collapses a replay to step 1 — a silent no-op
with **no second dismissal**, exactly what AC-2 demands. A server-side token store
that *re-broadcast* the prior result would *violate* "no second dismissal" and add
unbounded state, so it is deliberately not built. The token is decoded (it arrives
as a param), otherwise unused, and **never logged**. Documented inline so a
reviewer doesn't read its absence as a bug.

## Patterns established

- **Gate-before-consume is the security ordering, not an implementation detail.**
  The eligibility gate (step 2) runs **before** option classification and
  **before** the registry consume, so an internet-sourced answer can route an
  ALLOW keystroke only from a gated device, and an ungated/forged probe never
  mutates the modal (it stays outstanding for the local answer / #725 timeout).
- **No attacker byte reaches claude's PTY.** The routed keystroke is
  `strconv.Itoa(idx+1)` — a digit derived from the *trusted* option **index** — or
  a fixed verb (`AcceptTrust`/`SendEsc`). The raw `option_id` is validated for
  membership in the trusted `Outstanding.Options` **and** matched against the
  known-const switch before any use; it is never passed to `Answer`. The attacker
  selects *among surfaced options*, it cannot inject a keystroke.
- **Both #702 primitives, deliberately.** `MayAnswerRemotePermission` (eligibility,
  step 2) distinguishes `denied_unauthorized` from a real decision;
  `AuthorizeRemotePermission(dev, outcome)` then splits `allowed` from `denied`.
  For an eligible device the latter reduces to `outcome==OutcomeAllow`, but calling
  the primitive keeps the fail-closed conjunction in #702's single unit-tested
  place (it re-checks eligibility — defense in depth).
- **Keystroke is best-effort, like `ResolveCancel`.** Consume *first* (commit
  idempotency), then route; a keystroke error (no live session / teardown) is
  `Warn`-logged with the supervisor sentinel and tolerated — the modal is already
  consumed and moot, so the dismissal must still broadcast and the audit must still
  be written. Aborting would orphan a consumed modal.
- **`truncateForLog` bounds attacker-controlled log fields rune-safely.** The
  row-3 `Warn` logs the attacker-controlled `option_id` truncated to 64 bytes,
  backing up to a rune boundary so a multi-byte rune is never split (mirrors
  `modalbridge.boundPrompt`). slog JSON-escapes the value (no log-injection) and
  the payload is already AEAD-frame-capped (#446); the truncation only stops a
  hostile gated device padding the field to bloat the log. (The spec's one
  SHOULD-FIX, implemented.)
- **`(zero, false)` is the "nothing to do" signal, not an error.** Mirrors
  `ResolveCancel` / `ScreenSnapshotter` — no new sentinel.

## Concurrency

No new goroutines, no new locks. `ResolveAnswer` runs synchronously on the V2
manager's single `Run` dispatch goroutine (`dispatchAppFrame → handleModalAnswer →
ResolveAnswer`), so all resolutions are serialized — no `Lookup`/`Resolve` TOCTOU.
The producer goroutine (`Record`, #716) only **adds** fresh `modal_id`s under the
registry's leaf mutex; it can neither delete nor collide with an in-flight
resolution. The defensive `Resolve`-returns-`false` branch (modal vanished between
`Lookup` and `Resolve`) is therefore unreachable in practice but handled as a
step-1 no-op rather than asserted.

## Testing

`go test -race ./...` + `go vet ./...` pass (gofmt clean); real-claude e2e green
(`ok internal/e2e/realclaude 143.5s`).

- **`cmd/pyry/modal_resolve_v2_test.go`** (extends the #727 harness:
  `fakeKeystroker` now records `Answer` choices + `AcceptTrust` count;
  `recordTrustModal` mirrors `recordPermissionModal`; `eligibleDevice` adds the
  opt-in bit, `testDevice` stays the ineligible baseline). Scenarios assert
  keystroke verb+arg, consume/no-consume (follow-up `Lookup`), audit count+fields,
  returned dismissal, and `ok`:
  - **`Answer_Authorized`** (AC-1, table): `allow_once`→`Answer("1")`,
    `reject_once`→`Answer("3")` (3rd option), `proceed`→`AcceptTrust()`,
    `exit`→`SendEsc()` — each consumed, one audit (`allowed`/`denied`), dismissal
    `{option_id, remote}`, `ok=true`.
  - **`Answer_UngatedDevice`** / **`Answer_NilDevice`** (AC-2 fail-closed): no
    keystroke, modal **NOT consumed**, audit `denied_unauthorized` (empty identity
    for nil), `ok=false`.
  - **`Answer_StaleModalID`** (AC-2): unknown id → no keystroke, no audit,
    `ok=false`.
  - **`Answer_ReplayIdempotent`** (AC-2, the idempotency assertion): eligible
    `allow_once` twice → first `ok=true` (1 keystroke/audit, consumed); second
    `ok=false` (no 2nd keystroke/audit/dismissal).
  - **`Answer_ForgedOption`** (AC-2 defense): `option_id` not in `o.Options` (or
    wrong-class) → no keystroke, no consume, no audit, `Warn`-logged, `ok=false`.
  - **`Answer_KeystrokeError`**: `Answer` returns `ErrNoLiveSession` → modal still
    consumed, audit `allowed` still written, `Warn` logged, `ok=true` + dismissal.
  - **`Answer_NoBodyLeak`** (security): across allow / deny / `denied_unauthorized`,
    the modal body and title never appear in any log line.
  - `TestModalResolverV2_Answer_NoOp` (#727) was **replaced** — answer is no longer
    a no-op.
- **`internal/relay/v2session_modal_test.go`** — `TestV2Session_ModalAnswer_FanOut`
  drives a `modal_answer` through the real `Frames`/`Run` loop (fake resolver
  returns `({option_id, remote}, true)`) and asserts `modal_dismissed{outcome:
  option_id, source: remote}` decrypts at **both** interactive heads and **none**
  at a non-interactive one (mirrors `TestV2Session_ModalCancel_FanOut`). The relay
  seam can't import supervisor/audit/registry, so keystroke+audit live on the
  cmd/pyry side; together they cover AC-3 end-to-end.
  `TestV2Session_ModalAnswer_NoOp` (fake returns `false`) stays.

## Lessons learned

- **The stale `handleModalAnswer` doc comment in the relay is intentionally left
  for #708.** Code review flagged (NIT) that `internal/relay/v2session.go:1496-1499`
  still says "ResolveAnswer is a deferred no-op (always ok=false) … until #717" —
  now stale once this ticket lands. It was **correctly not touched**: the spec and
  the #727 hand-off note explicitly scope the relay manager *out* ("needs no
  change"), so editing it here would have widened the diff past the seam contract.
  Swept into #708's live-wiring pass. (The evergreen *feature* docs that described
  the answer arm as a deferred no-op — `v2-session-manager.md`,
  `modalbridge-package.md` — **are** updated here, since those track product state,
  not the seam diff.)

## Out of scope (named)

- **Live producer wiring — #708.** Nothing `Record`s into the daemon-singleton
  `modalReg` in production yet, so every production `modal_answer` takes the
  unknown-`modal_id` no-op path until #708 lands. This ticket is fully exercised in
  unit tests against a real `modalbridge.Registry` with `Record`ed modals — the
  intended seam-first phased rollout.
- **Deny-on-timeout — #725.** Reuses `broadcastModalDismissed` + the audit path
  with `{denied_timeout, timeout}`, driven by the daemon's own timer.
- **Local-TTY answer arm + cross-source first-answer-wins — #706.**
- **Verifying positional keystroke vs real claude rendering.** The design trusts
  the producer's contract that `Outstanding.Options[i]` is selectable by pressing
  key `i+1`. If a future producer surfaces an option set that doesn't match
  claude's on-screen numbering, the positional keystroke would mis-select — the
  `classifyAnswer` membership check bounds the blast radius (an id not in
  `o.Options` is rejected, never mis-keyed). Flagged for the #708 live-wiring run to
  verify against real claude. Not a blocker for this ticket.
