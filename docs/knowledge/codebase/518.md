# Ticket #518 — fix flaky `TestBridge_OutputObserver_NilSkipped` (and one sibling) via `io.Pipe` input

Test-only fix in `internal/supervisor/bridge_test.go`. Two tests (`TestBridge_OutputObserver_NilSkipped`, `TestBridge_OutputForwardsWhenAttached`) attached an immediate-EOF reader (`strings.NewReader("")`) to `Bridge.Attach`, then synchronously called `b.Write([]byte(...))` and asserted on the destination buffer. When the Attach goroutine's EOF-driven cleanup (acquires `b.mu`, sets `b.output = nil`, exits the input-pump loop) won the race against `b.Write` (acquires `b.mu`, snapshots `out := b.output`, releases, writes to `out`), the snapshot was nil and the write was discarded per the documented "EOF on input ⇒ detach, drop subsequent output bytes" contract that `pyry attach` relies on (`bridge.go:159-167`). Production semantics are correct; the bug was test-side — the assertion expected output forwarding after handing `Attach` a reader that was already at EOF. Fix: swap both tests to `io.Pipe()`, matching the existing pattern in `TestBridge_WriteSwallowsAttachedWriteErrors` (`bridge_test.go:38-44`). The input pump parks in `pr.Read(buf)` through the full assertion window; explicit `pw.Close()` after assertions triggers the deterministic EOF, and `<-done` confirms the detach completed. +21/−5 LOC, single file, zero production change. Surfaced during QA of PR #517 (touched only `internal/agentrun/*` — confirmed pre-existing flake, not a regression); recorded in #508's "Out of scope" follow-ups before this ticket was filed.

## Implementation

- **Two tests fixed, both with the same swap.** `TestBridge_OutputObserver_NilSkipped` (`bridge_test.go:97-135`) and `TestBridge_OutputForwardsWhenAttached` (`bridge_test.go:153-184`). Each replaces `strings.NewReader("")` with `pr, pw := io.Pipe(); defer pr.Close()` and passes `pr` to `b.Attach`. Existing assertions on `out.String()` are unchanged — they just stop racing the cleanup. After the final assertion, both tests `pw.Close()` to deliver EOF on the input side, then either `<-done` directly (in `_NilSkipped`) or `select { case <-done: case <-time.After(time.Second): }` (in `_OutputForwardsWhenAttached`, which keeps the pre-existing deadlock-guard timeout from the original test shape). The `time.After(time.Second)` is a deadlock guard, NOT a wait-for-flush — its semantic role flipped from "we expect EOF to have fired already" to "we just delivered EOF; assert it propagated within 1 s" — but the line itself didn't move.

- **No new helper, no shared fixture.** The architect spec considered an extracted helper and rejected it: the file already uses the same `io.Pipe()` shape in `TestBridge_WriteSwallowsAttachedWriteErrors`, and three uses with two-line setup is not the threshold for extraction. Each test remains readable as a standalone case.

- **The `defer func() { <-done }()` from the original `_NilSkipped` was dropped.** Replaced with explicit `pw.Close(); <-done` at the bottom of the test. Reason: with `defer`, the wait happened after `t.Cleanup` ran in unspecified order; explicit teardown after the close makes the synchronisation control flow visible at the call site. (`_OutputForwardsWhenAttached` already had no `defer <-done` — its original code relied on the empty-reader EOF firing `done` "promptly," with a 1 s deadlock guard; that contract is now load-bearing on `pw.Close()` rather than on `Attach`'s input being already-EOF.)

- **Audit per AC#4 — walked all 9 bridge tests for the same `Attach(immediate-EOF-reader, &out)` + synchronous `out.String()` pattern.** Exactly the two named tests had the bug. The full audit table is reproduced in the spec at `docs/specs/architecture/518-bridge-output-test-attach-eof-race.md`; summary: `TestBridge_WriteSwallowsAttachedWriteErrors` already used `io.Pipe()`; `TestBridge_OutputObserver_InvokedOnWrite` and `TestBridge_DiscardsWhenUnattached` never call `Attach`; `TestBridge_InputFlowsToReader` has a benign race (only reads `b.in`, doesn't depend on `b.output`); `TestBridge_RejectsConcurrentAttach`, `TestBridge_BlocksReadUntilAttached`, and the three Resize tests don't have the assertion-after-Write shape. No third-test fix needed.

- **No `bridge.go` change.** The Attach goroutine's "EOF on input ⇒ clear `b.output`" cleanup at `bridge.go:241-249` is the production detach path that `pyry attach` depends on for daemon-side cleanup when the client disconnects. Re-architecting it to expose a "wait until output flushed" sync hook for tests would change production semantics to paper over a bug that only manifests when the test feeds an already-EOF reader. The right shape was to fix the test inputs.

## Patterns established

- **When testing a seam whose goroutine treats input-EOF as a teardown signal, never hand it an already-EOF reader and then expect post-Attach state to remain bound.** This is the test-side mirror of the bridge's actual production contract: `Bridge.Attach`'s goroutine reads from the supplied input, and on EOF clears the output binding so subsequent `Write`s are discarded (this is correct — `pyry attach` client disconnect IS represented by input-side EOF). The architect's spec re-diagnosed PO's "wait for the copy goroutine to flush" framing: `Bridge.Write` is synchronous, there is no async output fan-out to wait for — the racing goroutine was the input pump cleaning up after EOF, not an output copier. The deterministic test signal isn't a wait-for-flush primitive; it's *control over input-side EOF*. `io.Pipe()` provides that control: the pump parks in `pr.Read` until the test explicitly calls `pw.Close()`. Generalises to any goroutine-owning seam whose lifecycle is tied to an input source — give the test code the ability to keep the source alive across the assertion window, then close deterministically.

- **Same-file pattern reuse beats a new test helper when the pattern is already 3 lines and used once.** `TestBridge_WriteSwallowsAttachedWriteErrors` had used `io.Pipe()` for `Attach` since the file was written; the two flaky tests were the outliers. Normalising them to the existing pattern is one line of comment + two lines of pipe setup per test — no helper warranted. Generalises to test refactors: when a fix means "make N tests look like the one test that already does it right," the bar for extracting a helper is "would the helper read better than three lines of inline setup?" Here it would not; pumping every io.Pipe call through a helper would obscure the per-test reason for needing it.

- **QA's single-sample compare gate magnifies test-side races into PR-routing decisions.** PR #517's QA observed this flake once, retried, all green; the issue body documents this as a "false-positive routing to rework" risk. The dispatcher's convention for flake observations is "record in tracking issue, don't re-route the PR being QA'd" — #508 already cited #518 in its own "Out of scope" section before this ticket was picked up, so the convention worked here. The structural fix is the deterministic test, not weakening the QA gate; recording the convention because future contributors should expect QA to surface latent flakes that don't reproduce easily on the affected PR's branch.

## Lessons learned

- **PO's diagnosis was directionally right, mechanically wrong; the architect spec's re-diagnosis is the durable record.** The PO body framed the race as the test reading `out.String()` before "Bridge.Attach's copy goroutine has flushed the write into the user-supplied `&out` buffer" — but `Bridge.Write` does not spawn a copy goroutine; it acquires `b.mu`, snapshots `b.output`, and writes synchronously inside `Write`. The racing goroutine was the Attach-spawned input pump, and its racy operation was clearing `b.output = nil` on EOF, not failing to flush a queued write. The fix-shape PO implied ("deterministic signal instead of relying on scheduler order") was correct; the deterministic signal turned out to be input-side EOF control, not output-side flush observation. Lesson: when a PO body describes the race in code-author-friendly terms, treat the framing as a hypothesis to verify against `bridge.go` — re-read the source before encoding the framing in test code. The architect catching this in the spec is the system working as intended; the cost would have been a "wait for output observer to fire" hook in production code, defending against a problem that only existed in test inputs.

- **`time.After(t)` as a deadlock guard is allowed by the no-`time.Sleep` rule; the distinction matters.** AC#2 explicitly bans time-based waits and fixed sleeps, but `TestBridge_OutputForwardsWhenAttached`'s retained `time.After(time.Second)` is a deadlock fence around a `select` that expects `<-done` to be the first arm. The semantic is "the deterministic signal we just sent (`pw.Close()`) must propagate within 1 s or the test fails loudly" — not "let's wait a second and hope the data arrived." Pre-existing similar guards exist elsewhere in the same file (lines 306, 315). Lesson for future test-flake fixes: when the spec or AC says "no `time.Sleep`," interpret as "no time-based correctness arguments." A bounded `select`-with-timeout where the timeout arm is a `t.Fatal` is a different primitive — it asserts a bound on the propagation latency of a deterministic signal, rather than substituting for one.

- **The "QA flake observed, recorded once, don't re-route" convention worked end-to-end across two cycles.** #508's PR-author noticed the flake during QA, recorded it in the "Out of scope" section of their own ticket's codebase doc (this ticket's #518 entry), filed the tracking issue, and shipped #508 without re-running QA on the affected PR. #518 then picked up cleanly from the tracking issue, with the QA observation and re-run results already documented. Net cost: zero blocked PRs, one accurate flake-tracking ticket. Lesson: the dispatcher's "QA-flake → tracking issue, not re-route" convention only works if the cycle that observes the flake actually files (or appends to) the tracking issue with the failure shape and the re-run evidence — silent observations leave the next cycle to re-diagnose. #508's per-ticket doc made the convention visible; that visibility is what made this ticket trivial to pick up.
