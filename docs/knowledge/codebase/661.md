# Ticket #661 — Mobile live e2e: documented daemon-side setup so the turn bridge resolves a real claude transcript

Rung 3 of the mobile e2e ladder (the first **live emulator** run, 2026-06-17) failed
because the daemon's v2 structured turn bridge could not resolve a claude session
transcript: started with a throwaway `-pyry-workdir=/tmp` and no real conversation, the
producer spun on `WARN turnbridge: resolve session jsonl, retrying error="no session
jsonl found in ~/.claude/projects/-tmp"` instead of opening its events stream. This
ticket documents the minimal daemon-side setup (real workdir + seeded conversation +
a real claude session) so the bridge resolves a **real** (non-scripted) transcript on
its own. Deliverable is documentation; **zero production code**.

## Implementation

- **One new doc:** [`features/mobile-live-e2e-runbook.md`](../features/mobile-live-e2e-runbook.md)
  — the manual operator runbook for the live emulator + real-claude + relay stack (the
  manual sibling of the automated `e2e-harness.md` / `e2e-realclaude.md` suites). It
  covers: the resolution machinery as SSOT (`resolveClaudeSessionsDir`,
  `resolveLatestSessionJSONL`, `encodeWorkdir`'s `/`-and-`.`→`-` rule), why `/tmp` with
  no seed spins forever, the four-step setup (real workdir `W` → real claude session in
  `W` → seed `conversations.json` → start daemon with `-pyry-workdir=W PYRY_MOBILE_V2=1`),
  and the two operator-verified daemon-log observables (the retry warning stops; the
  producer opens its events stream on the resolved transcript).
- **Size XS** (architect downgrade from S): one doc file, zero `.go`, zero new types,
  zero tests.
- Cross-links the new doc to [`codebase/642.md`](642.md) (the SSOT seeding pattern +
  cold-start race), the two resolution functions by file:line, and [ADR 025](../decisions/025-mobile-remote-head-interactive-session.md)
  § Phase 2 / [`turnbridge-package.md`](../features/turnbridge-package.md).

## Patterns established

- **The doc-vs-code fork resolved to documentation by evidence, not a code split.** The
  ticket left a fork: document the setup, or — if a phone-created conversation genuinely
  can't be mapped to a real transcript with existing features — route back `needs-rework:po`
  for a code split. The architect resolved it to documentation-only on four existing-feature
  facts: (1) the sessions dir is computed purely from `-pyry-workdir`
  (`resolveClaudeSessionsDir`); (2) `resolveLatestSessionJSONL` is **conversation-agnostic**
  — it returns the newest `<uuid>.jsonl` in the one sessions dir, so a conversation id never
  participates in transcript resolution (there is one supervised claude, one workdir, one
  sessions dir); (3) seeding `conversations.json` only satisfies `WriteUserTurn`'s
  `ValidateConversation` gate (#312), an already-shipped feature; (4) #642 already proves the
  full producer path against a real-claude-*format* transcript — the only live delta is *who
  writes the transcript* (a real `claude` vs a scripted append). No code to add ⇒ no split.
  (Matches `[[po-document-or-support-fork-resolve-by-evidence]]`.)
- **Operator-only live-stack AC = runbook-verified daemon-side observable, not a CI gate.**
  The full live stack (emulator + real claude + relay) runs only on an operator machine,
  never under `-tags e2e` CI. AC#3 ("the bridge resolves a real transcript") is verified by
  executing the runbook and observing the two daemon-log observables, recorded on the ticket
  — the developer cannot and need not run the stack. #642's deterministic two-phone proof is
  the automated stand-in for resolution correctness; this runbook is the real-claude
  confirmation that a real claude (not fakeclaude) populates the same aligned dir.
- **Documentation-deliverable ticket exception.** The developer worktree mutated exactly one
  doc file (the runbook). The usual "developer worktree touches only code/tests/spec" rule
  guards against smuggling a fixed-cost knowledge-doc AC onto a *code* ticket; here the doc
  **is** the work, so the rule's rationale doesn't fire. The `codebase/<N>.md` + `INDEX.md`
  entries remain the documentation phase's job (this file).

## Lessons learned

- **The cold-start producer-subscribe race is why step 2 (pre-existing transcript) matters.**
  The producer captures its tail offset at the **first successful resolve**; a resolve that
  lands before any transcript exists Warn-retries, and a later retry can capture an EOF
  offset *past* the events the phone expects. The runbook cites this from
  [`codebase/642.md`](642.md) "Lessons learned" (the SSOT) rather than re-deriving it.
- **A single transient `resolve session jsonl, retrying` at startup is benign; the bug is the
  *permanent* loop** against an empty sessions dir (throwaway `/tmp`, no real claude). The doc
  documents both states so an operator doesn't mistake a transient startup warning for the
  failure.
- **Incidental realclaude golden churn (not feature work).** The merged diff also touched
  `internal/e2e/realclaude/testdata/permission_protocol_v2.1.158.json` (run-specific values —
  temp-dir paths, `session_id`, `uuid` — regenerated and auto-committed during code-review's
  `make e2e-realclaude` verification run). Same noise documented in [642.md](642.md) /
  [603.md](603.md); not part of #661's diff-of-interest.
