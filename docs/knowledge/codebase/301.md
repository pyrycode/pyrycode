# Ticket #301 — net: wire `relay.Connect` into the pyry daemon startup

First consumer of `internal/relay` from the binary side. Until this ticket
the package shipped pure-library and `cmd/pyry`/`internal/supervisor`
never imported it; `cfg.RelayURL` was consumed only by `pyry pair` for the
QR payload. This slice wires `relay.Connect` into `runSupervisor`'s
startup path, adds a tightly-scoped test seam so e2e suites can point the
daemon at a plain-`ws://` fakerelay, and brings the fakerelay harness up
to a real binary↔relay handshake (hello/hello_ack + opt-in WS 4409 close).
No phone-frame handling, no per-conn dispatcher — `Connection.Frames()`
is drained-and-discarded for now; the dispatcher slice consumes it.

## Implementation

### `internal/relay` — `AllowInsecureScheme` test seam

- **`Config.AllowInsecureScheme bool`** field added (`connection.go:69-74`).
  Field godoc names the seam explicitly as test-only and points at the
  env-gate (`PYRY_ALLOW_INSECURE_RELAY=1`) that cmd/pyry uses to flip it.
- **`Connect` scheme check** relaxed at `connection.go:120` from
  `parsedURL.Scheme != "wss"` to
  `parsedURL.Scheme != "wss" && !(cfg.AllowInsecureScheme && parsedURL.Scheme == "ws")`.
  Production default behaviour is unchanged: `AllowInsecureScheme=false`
  → `ws://` still rejected with `ErrInvalidConfig`.
- **Pre-existing `connectWithClient` test seam stays.** It bypasses
  header/transport build entirely; the new field is for production-shaped
  Connect paths that just need ws://. Two seams, two purposes.

### `cmd/pyry/relay.go` — new file (~120 LOC)

Package-internal helpers; mirrors `cmd/pyry/pair.go`'s split out of
`main.go`.

- **`resolveRelayURL(flagValue, envValue, cfg) string`** — pure
  precedence helper: flag > env > `cfg.RelayURL`. `cfg.RelayURL` itself
  is already overlay-defaulted by `config.Load` per
  [ADR 018](../decisions/018-config-overlay-decode.md), so the empty
  return is only reachable when the operator has deliberately blanked
  every leg. Returning `""` is the disabled-relay branch.
- **`startRelay(ctx, logger, instanceName, relayURL, version, allowInsecure, shutdown)`**
  → `(cleanup func(), err error)`. Five behavioural arms:
  1. `relayURL == ""` → log info `relay: disabled (no URL configured)`,
     return no-op cleanup and nil err. The daemon must boot without a
     relay for local-only `claude` supervision.
  2. `identity.LoadOrCreate` against `resolveServerIDPath(instanceName)`
     (reuses `pyry pair`'s on-disk file via the resolver in `pair.go`).
     Synchronous failure → wrapped error, no goroutine spawned.
  3. If `allowInsecure`, log info `relay: PYRY_ALLOW_INSECURE_RELAY=1 —
     accepting ws:// scheme` so the boot log carries the downgrade.
  4. `relay.Connect` — sync validation errors (invalid scheme, missing
     identity) wrap and surface; lifecycle goroutine spawns on success.
  5. Goroutine drains `conn.Frames()` until close, then reads
     `conn.Wait()`, classifies via `errors.Is`:
     - `ErrServerIDConflict` → log error + `shutdown()` (cancels the
       supervisor ctx; `pool.Run` unwinds; daemon exits cleanly).
     - `context.Canceled` / `context.DeadlineExceeded` → log debug.
     - any other terminal error → log warn, **goroutine exits without
       restarting** (the supervisor's claude child stays running).
     - nil → log debug `relay: lifecycle ended cleanly`.
- **Cleanup is idempotent:** `_ = conn.Close()` then `<-done`. The
  goroutine drain is awaited so the daemon process does not exit while a
  WS handle is still in flight.
- **`Frames()` drained-and-discarded:** `for range conn.Frames() {}`.
  `Connection.run` defers `close(c.done)` then `close(c.frames)`, so the
  range exit is the safe signal that `Wait` is about to return.

### `cmd/pyry/main.go` — wiring (~16 LOC)

In `runSupervisor`'s flagset:
- `relayFlag := fs.String("pyry-relay", "", "...")` (line 404).
- `"pyry-relay"` added to `pyryFlagValues` so `splitArgs` recognises it
  (the `--pyry-relay=<url>` form keeps the value out of `claudeArgs`).

After `signal.NotifyContext`/`cancel` exist but before `pool.Run`:

```go
cfg, err := config.Load(resolveConfigPath())
if err != nil { return fmt.Errorf("loading config: %w", err) }
relayURL := resolveRelayURL(*relayFlag, os.Getenv("PYRY_RELAY_URL"), cfg)
allowInsecure := os.Getenv("PYRY_ALLOW_INSECURE_RELAY") == "1"
relayCleanup, err := startRelay(ctx, logger, *name, relayURL, Version, allowInsecure, cancel)
if err != nil { return fmt.Errorf("relay start: %w", err) }
defer relayCleanup()
```

The `cancel` value is the single shutdown lever already wired by
`signal.NotifyContext`; the relay goroutine reuses it for the 4409 exit
path. No new resolver — `resolveServerIDPath(*name)` lives in `pair.go`.

### `internal/e2e/internal/fakerelay` — handshake + WS-4409 support

Two additive surfaces on the harness; existing tests stay green.

- **Binary-direct hello → hello_ack dispatch.** `binaryRecvPump` now
  branches on `env.ConnID == ""` (binary-direct vs routed). The direct
  path decodes a second time as `protocol.Envelope`, captures it under
  `s.mu` keyed by `serverID` (`lastBinaryHello map[string]protocol.Envelope`),
  and replies with a routing-wrapped `Envelope{Type:TypeHelloAck,
  InReplyTo:&helloID, Payload:HelloAckPayload{ProtocolVersion:"v1",
  ServerID, ConnID:"-"}}`. Non-hello binary-direct envelopes log at debug
  and drop (the dispatcher slice will replace this stub). The routing
  path is untouched, so existing `TestPhoneToBinary_FrameWrappedWithConnID`
  still passes.
- **`Server.RejectNextBinaryWith4409()`** — one-shot opt-in. The next
  `/v1/server` upgrade accepts the WS handshake and immediately closes
  with `websocket.StatusCode(4409)` and reason
  `"server-id already claimed"`. The flag clears after one use; existing
  HTTP-409 first-claim-wins path is unchanged.
- **`Server.LastBinaryHello(serverID) (protocol.Envelope, bool)`** — read
  accessor for e2e introspection.
- **`Server.ForceCloseBinary(serverID) bool`** — closes the WS conn
  currently bound to `serverID` with `StatusInternalError` (1011). Used
  by `TestRelay_1011` to drive a non-fatal close and assert the daemon
  absorbs it via transport-internal reconnect.

### `internal/e2e/relay_test.go` — three new tests (build tag `e2e`)

- **`TestRelay_Hello`** — happy-path: spin up fakerelay, spawn pyry with
  `-pyry-relay=<fr.URL>/v1/server` + `PYRY_ALLOW_INSECURE_RELAY=1`, poll
  `fr.LastBinaryHello(serverID)`, assert `role=server` and `server_id`
  match the daemon's persisted server-id file.
- **`TestRelay_4409`** — arms `RejectNextBinaryWith4409()` before
  spawning, spawns the daemon without the readiness gate (it may exit
  before the control socket binds), asserts exit within 5s with exit
  code 0 and stderr contains `"server-id conflict"`.
- **`TestRelay_1011`** — drives `ForceCloseBinary` after the first hello,
  asserts the daemon process is still alive 2s later (transport
  reconnect absorbed the close) and `pyry status` exits 0.

### `internal/e2e/harness.go` — `StartInWithEnv`

Adds a sibling helper to `StartIn` that takes `extraEnv []string` in
addition to `extraFlags`. Internally calls `spawnWith(t, home,
spawnOpts{extraEnv, extraFlags})` (the shared core landed in #123),
mirrors `StartIn`'s Harness construction + readiness wait. Used by the
two e2e tests that need `PYRY_ALLOW_INSECURE_RELAY=1` without disturbing
every existing `StartIn` call site.

### Concurrency model

One supervisor-owned goroutine for the daemon lifetime:

```
runSupervisor              startRelay              Connection.run
─────────────              ──────────              ──────────────
conn = Connect(ctx, ...) ► spawn drain+wait    ──► transport.Connect loop
pool.Run(ctx) (blocks)     for range Frames(){}    (reconnect internally)
                           err := conn.Wait()
                             ErrServerIDConflict:
                               shutdown() ──► cancel ──► pool.Run unwinds
                             ctx.Err / other:
                               log; return
relayCleanup()
  conn.Close()
  <-done
```

The shutdown decision (4409 → exit) lives in this goroutine, not in
`Connection`. The daemon owns the policy; `Connection` owns the
classification.

## Error / recovery table

| Source                              | Classification                       | Goroutine action     | Daemon impact             |
|-------------------------------------|--------------------------------------|----------------------|---------------------------|
| Transient network drop              | Handled inside `transport.Connect`   | Never observed       | None (reconnect)          |
| Non-fatal WS close (e.g. 1011)      | Handled inside `transport.Connect`   | Never observed       | None (reconnect)          |
| WS close 4409                       | `ErrServerIDConflict` from `Wait`    | Log + `shutdown()`   | Daemon exits cleanly      |
| `ctx.Err` (SIGTERM/SIGINT)          | `ctx.Err()` from `Wait`              | Log debug; return    | None (already shutting)   |
| Other terminal transport error      | Wrapped `transport.ErrFatalClose` etc| Log warn; return     | None (claude stays up)    |

The "exit, don't restart" choice on the last row is the architect's
explicit decision (AC#4: "exit vs. schedule reconnect — architect's call,
covered by an explicit test"). The transport already retries every
transient failure; reaching the terminal arm means a genuinely
unrecoverable classification surfaced, and a restart loop at this layer
would mask the underlying fault. The supervisor's claude child stays
running because the goroutine exit does not propagate to `pool.Run` —
only the explicit `shutdown()` call does.

## Security posture

- **`AllowInsecureScheme` is env-gated only.** No `-pyry-allow-insecure-relay`
  flag, no `allow_insecure_relay` config-file key. The env name says
  "insecure"; the boot log records the downgrade when active. A future
  flag/config plumb is a deliberate decision, not a drift.
- **No new secret surface.** `x-pyrycode-server` / `x-pyrycode-version`
  / `user-agent` are non-secret; `ServerID` is a UUIDv4, not a credential.
  Tokens are phone-side; the dispatcher slice introduces them.
- **4409 reconnect-storm DoS** mitigated by AC#3: 4409 is terminal at
  both the transport layer (`FatalCloseCodes: [4409]`) and the daemon
  layer (goroutine triggers shutdown, no reconnect).
- **Resource leak on shutdown:** `relayCleanup()` is `defer`red after
  `pool.Run` returns and blocks on the goroutine drain.

## Patterns established / reinforced

- **Operator-facing env-gate over config-file key for security downgrades.**
  Same shape will apply to any future "I know this is dangerous"
  knob: env var with the word `INSECURE` (or equivalent) in the name,
  logged at info on activation, no flag/config-file plumb until an
  operator actually needs it.
- **Daemon-side helpers split out of `main.go` mirror the `pair.go`
  pattern.** Three files now (`main.go`, `pair.go`, `relay.go`); the next
  daemon-side subsystem (dispatcher / handler wiring) gets its own
  `cmd/pyry/<topic>.go`. `main.go` stays the wiring orchestrator.
- **`StartInWithEnv` sibling helper over a variadic option on `StartIn`.**
  Additive surface, no churn at existing call sites. Future env-injection
  needs reuse the same shape.
- **Lifecycle goroutine drains `Frames()` even when discarding.** Ranging
  the channel is the cheapest correct way to follow the lifecycle and
  catch the moment `Wait` will not block; alternative (calling `Wait`
  first) races the channel close.
- **Single shutdown lever.** `signal.NotifyContext`'s `cancel` is the
  one process-wide unwind. The relay goroutine reuses it rather than
  introducing a sibling. Future subsystems with "must exit" classifications
  do the same.

## Lessons learned

### "Goroutine-side reconnect logic" is the wrong layer

The AC explicitly left "exit vs. reconnect" open for the architect. The
right answer was a one-liner: the transport already reconnects on every
non-fatal close (1s/2s/4s backoff with jitter and reset-after-stability);
the relay package classifies terminal errors; the daemon goroutine
treats classified-terminal as terminal. Anything else duplicates state
machines. Reconnect storms come from layer confusion; the fix is
strictly less code, not more.

### `Wait()`-return ordering vs `Frames()` close

The architect's spec flagged this as an open question. `Connection.run`
defers `close(c.done)` then `close(c.frames)`, which means `Wait()` (which
reads `c.done`) may unblock before the `Frames()` channel close
propagates to a concurrent ranger. The implementation order — drain
`Frames()` first, then call `Wait()` — sidesteps the race entirely: the
range exit is the natural signal that `Wait` is about to return, and a
single-goroutine drain has no reorder hazard. Calling `Wait` in a sibling
goroutine while ranging would have been racy; calling `Wait` first and
then expecting `Frames` to have closed would have been racy in the other
direction. The single-drain shape is correct regardless of which `defer`
ordering `Connection.run` uses, which is the desirable property for code
that integrates with a library type whose internal `defer` order isn't
part of its public contract.

### Implicit env-injection through `spawnWith`

`StartInWithEnv` is a 25-line helper, but discovering that the
underlying `spawnWith` already accepted `extraEnv` saved a refactor
across every existing `StartIn` call site. The pattern is: when adding
a new harness option, look first for an unexposed parameter on the
shared core (the `spawnOpts` struct that #123 introduced) rather than
adding it to every signature. The cost of a sibling helper is one new
function; the cost of a variadic option is every call site reading
slightly differently. For a single test consumer, the helper wins.

## Out of scope (deferred)

- **Per-conn dispatcher** consuming `Frames()` with per-type handlers
  (the next slice of #297). The drain-and-discard loop here is a
  placeholder.
- **Outbound `Send` API on `*Connection`.** The relay package surfaces
  `Frames()` only; binary-initiated frames (conversation updates,
  message echoes, hello_ack replies on phone conns) wait for the
  dispatcher.
- **First-frame token validation wiring.** `relay.AuthenticateFirstFrame`
  exists (#249) but the per-phone-conn caller that actually delivers a
  token to it is the dispatcher's job.
- **`-pyry-no-relay` / `-pyry-relay=""` semantics.** Today an empty
  `-pyry-relay` falls through to env then config; an operator who wants
  to explicitly *disable* a configured relay has no first-class verb.
  Architect's note: "defer to a follow-up unless an operator hits this."
