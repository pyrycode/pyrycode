# Ticket #342 — `pyry agent-run`: invoke `MarkWorkdirTrusted` after flag parse

Consumer wire-up for the trust-state helper shipped in [#341](341.md). `runAgentRun` now resolves `os.UserHomeDir()` and calls `agentrun.MarkWorkdirTrusted(home, parsed.workdir)` between flag validation and `WriteSettings` (#339), so the supervised claude landing in #332 does not block on the workspace-trust TUI dialog at startup.

## Implementation

- `cmd/pyry/agent_run.go` — six net production lines inside `runAgentRun`, slotted between the `parseAgentRunArgs` success and the `agentrun.WriteSettings` call:
  1. `home, err := os.UserHomeDir()` → wrap as `agent-run: resolving home directory: %w` on error.
  2. `agentrun.MarkWorkdirTrusted(home, parsed.workdir)` → wrap as `agent-run: pre-populating workspace trust: %w` on error.
- Order is **mark-trust → settings**. Both writes are atomic in isolation, but `MarkWorkdirTrusted` modifies a shared file under a lock (idempotent re-run cheap) while `WriteSettings` creates a per-spawn artefact inside the user's `workdir`. Putting trust first means a trust-mark failure short-circuits before any per-spawn cruft lands in `workdir` — satisfies the AC's "no partial state writes after a failed mark" rule directly.
- No CWD-fallback on `os.UserHomeDir()` failure (the way `resolveSocketPath` / `resolveRegistryPath` do). Those helpers degrade gracefully because they only need a writable path; `MarkWorkdirTrusted` needs the *actual* `~/.claude.json` claude itself reads. A guessed-empty `homeDir` would write `./.claude.json` and the trust mark would have no effect at spawn time. Fail loud.
- `cmd/pyry/agent_run_test.go` — `newValidArgsFixture` adds `t.Setenv("HOME", t.TempDir())` as its first statement so every `runAgentRun`-driving test (today: `TestRunAgentRun_EmitsSettingsFile` plus the new test below) is hermetic and cannot mutate the developer's real `~/.claude.json`. Pure-`parseAgentRunArgs` table tests share the fixture harmlessly.
- New `TestRunAgentRun_MarksWorkdirTrusted`: drives `runAgentRun(fx.argv)` with stdout discarded via `os.Pipe()`, then reads `<HOME>/.claude.json`, decodes to `map[string]any`, computes the expected key with `filepath.EvalSymlinks(fx.workdir)` (the macOS `/var → /private/var` rule applies because `t.TempDir()` lives under `/var/folders/...`), and asserts `projects[wantKey].hasTrustDialogAccepted == true`. Asserts only the observable contract — the helper's lock semantics, atomic-rename, `json.Number` preservation, and idempotency are pinned by `internal/agentrun/trust_test.go` per [#341](341.md).

## Patterns established

- **Helper-explicit `homeDir` + caller-side `os.UserHomeDir()` is the production shape, full-stop.** [#341](341.md) made `homeDir` an explicit parameter so tests run `t.Parallel` without `t.Setenv`. The corollary on the caller side: production callers resolve `os.UserHomeDir()` at the call site (matching `runInstallService` in `cmd/pyry/main.go:1203-1210`), and the call site fails loud on `UserHomeDir` error rather than falling through. Mirror this shape for any future verb that pre-populates files under `$HOME` before claude spawn.
- **Test-fixture HOME isolation is a correctness fix, not a stylistic one.** Once a wire-up calls `os.UserHomeDir()`, every existing test that drives the same entry point will start writing to the developer's real `~/.claude.json` unless the shared fixture redirects HOME. The redirection belongs in the fixture (single point of truth) rather than per-test, so future tests added to the same fixture inherit isolation by default. Pin the new behaviour in the fixture in the same commit that introduces the wire-up.
- **`pyry: <verb>: <operation>: <err>` is the canonical error wrap shape for verb-level initialisation failures.** Two distinct wraps in this slice (`agent-run: resolving home directory: %w` and `agent-run: pre-populating workspace trust: %w`) name the operation that failed, so the stderr line tells the operator *what* the verb was doing, not just *which verb* failed. The double `agent-run:` / `agentrun:` (verb namespace + helper package wrap-prefix from [#341](341.md)) is intentional layering, not duplication — mirrors how `install-service` errors render today.

## Lessons learned

- **Once a wire-up calls `os.UserHomeDir()`, audit every existing test on the same code path before shipping.** The single biggest correctness risk in this slice was leaving `TestRunAgentRun_EmitsSettingsFile` (#339) un-redirected — it would have silently mutated the developer's real `~/.claude.json` on every `go test ./...` run, with no visible failure signal. The architecture doc called this out explicitly ("This is a correctness requirement, not a stylistic one"); the fix went into the shared fixture so the inheritance is automatic. Generalisation: any time a wire-up introduces an environment-touching dependency (HOME, XDG vars, well-known files), grep for callers and decide where the test isolation lives *before* writing the new test.
