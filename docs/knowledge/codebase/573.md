# Ticket #573 — ptyrunner `num_turns` reports claude's logical-turn count, not raw assistant-entry count

ptyrunner's synthesised `type:"result"` trailer incremented `num_turns` once per
assistant JSONL entry, but claude 2.1.158 serialises one logical reply as multiple
consecutive assistant entries sharing a `message.id` (a `thinking` line, a `tool_use`
line, a `text` line). The emitter counted each → `num_turns` roughly doubled for trivial
replies versus **streamrunner**, which forwards claude's native count from claude's own
result envelope. The dispatcher logs and acts on `num_turns`, so the signal changed
silently after the ptyrunner cutover. The fix counts **logical turns** — one per distinct
assistant `message.id` — so the two runners agree. Reporting path only; the **enforcement**
path (the `internal/agentrun/budget` Counter, which double-counts identically) is the
out-of-scope dependent sibling. Split from [#567](https://github.com/pyrycode/pyrycode/issues/567).

## Implementation

One production file: `internal/agentrun/streamjson/emitter.go` (~12 LOC, no signature
change, no new exported symbols, no new package).

- **New unexported field** `lastAssistantMsgID string` on `Emitter`, alongside `numTurns`
  under the pre-existing `mu`. Holds the previous assistant entry's `message.id`; `""` =
  none seen yet.
- **Transition-counted increment** in the `if entry.Type == "assistant"` block of `Emit`,
  replacing the unconditional `e.numTurns++` (was `emitter.go:189`):
  ```go
  id := ""
  if entry.Message != nil {
      id = entry.Message.ID
  }
  if id == "" || id != e.lastAssistantMsgID {
      e.numTurns++
  }
  e.lastAssistantMsgID = id
  ```
  A new turn begins only when the `message.id` changes. The second/third line of a split
  reply hits `id == e.lastAssistantMsgID` → no increment. Intervening non-assistant
  entries (`user`/`tool_result`) fall outside the block, so they never touch
  `lastAssistantMsgID` and never split a turn whose lines straddle them.
- **empty-id floor.** `id == ""` short-circuits to "its own turn." Real claude always
  emits a non-empty id; empty ids appear only in synthetic/malformed entries, so this both
  stays correct and preserves the pre-fix per-entry count those entries had (keeps the
  `TestReadUsage_*` family green).
- **AC#5 — byte shape unchanged apart from the value.** `Close` and the `trailer` struct
  are untouched; only the `num_turns` *value* changes. Field set, key order, and every
  other field the dispatcher parses stay identical.
- Concurrency model unchanged (leaf state under `e.mu`, mutated only in `Emit`). The
  package's `MUST NOT log entry content` discipline is preserved — the change reads
  `entry.Message.ID` into a count decision and never logs it.

## Patterns established

- **Count logical turns by `message.id`, not raw assistant entries.** This is the
  canonical signal for "how many turns did claude take," and it's empirically grounded
  (`internal/e2e/realclaude/testdata/permission_protocol_v2.1.158.json`: 3 assistant events,
  2 distinct `message.id`s, claude's native `num_turns: 2`). The alternatives the ticket
  floated both fail on that capture: *"count entries carrying text"* undercounts to 1
  (only the final `[text]` entry has text); *"thinking-only vs text entry"* can't classify
  the `[tool_use]`-only entry. Any future count or budget keyed on "assistant entries"
  must group by `message.id`.
- **Transition-counting == distinct-id-counting under the no-interleaving invariant.**
  Comparing each assistant entry's id to the *immediately-previous* one equals counting
  distinct ids **iff** one message's lines are never interleaved with another's (no
  `A,B,A`). claude completes one assistant message before starting the next, so this holds.
  Documented as a one-line code comment, **not** defended with a seen-set — per
  Evidence-Based Fix Selection, no interleaving has been observed.
- **tuidriver deliberately leaves `msg_id` grouping to consumers.** Its `IsEndTurn` /
  `AssistantText` are per-entry; the library docstring (`pkg/tuidriver/jsonl.go`) states a
  multi-line turn "needs msg_id grouping on top — out of scope for the library; consumers
  compose if needed." This ticket is exactly that consumer-side composition.

  rewritten table-driven with four subtests:
  - *split thinking+text reply* (`msg_C` `[thinking]`, `msg_C` `[text]`) → **1** (the

    floor).

## Lessons learned

- **claude 2.1.158 emits one logical assistant turn as several consecutive JSONL entries
  sharing a `message.id`** (thinking / tool_use / text on separate lines). tuidriver
  deliberately leaves `msg_id` grouping to consumers (`IsEndTurn`/`AssistantText` are
  per-entry); any count or budget keyed off "assistant entries" must group by `message.id`
  to match claude's native `num_turns`. Transition-counting (compare to the previous
  entry's id) equals distinct-id counting because claude finishes one message before
  starting the next — no `A,B,A` interleaving.
- **A synthesised metric that shadows a native one diverges silently.** ptyrunner's
  hand-rolled `num_turns` was "close enough" to claude's native count until 2.1.158's
  split-reply serialisation roughly doubled it, and the dispatcher saw the change with no
  signal. Pin cross-runner equivalence with a *strict* assertion (`streamResult ==
  ptyResult`), not a permissive `both >= 1` floor — the floor would have masked this exact
  regression.

## Out of scope / known same-shape gaps

- **The budget Counter double-counts identically.** `internal/agentrun/budget` increments
  per assistant entry via `OnEvent`, so the `--max-turns` enforcement path has the same
  bug. It is the dependent **sibling** ticket. The precise rule above (transition-count on
  `message.id`, empty-id floor) is documented here for the sibling to mirror verbatim.
- **No shared turn-detection primitive was extracted.** The change is ~6 lines of
  consumer-local state; the two consumers (the emitter's running count vs. the budget
  Counter's per-turn gate) have different shapes, and the only common dependency
  (tuidriver) declines to own `msg_id` grouping. Per "Don't define interfaces
  preemptively," the sibling decides whether to extract once both call shapes are visible.
