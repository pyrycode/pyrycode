# Ticket #539 — `agent-run --self-check`: move the probe tool off `dontAsk`'s read-only-Bash carveout (`Bash` → `Write`)

The pre-#539 `--self-check` exhibit prompted claude with `"Use Bash to echo hello. Be brief."` against a `permissions.allow: ["Read"]` settings file. Per the [permission-modes reference](https://code.claude.com/docs/en/permission-modes) — *"`--permission-mode dontAsk` … auto-denies all tool calls except those matching allow rules **and read-only Bash commands**"* — `echo hello` rode the read-only-Bash carveout and was permitted regardless of allow-list state. The boundary the selfcheck claimed to verify ("claude refuses tools NOT in `permissions.allow`") was therefore tested on the wrong side of a permanent claude exemption: PASS/FAIL did not track that boundary 1:1, and a future re-scoping of "read-only Bash" would shift the test result without the boundary itself changing. The fix moves the probe tool to `Write` — a tool with no analogous documented carveout — and introduces `canonicalProbeTool` as the single source of truth so the prompt, the detector, and the wrap-error site all latch onto the same const. Test-credibility half of [#537](https://github.com/pyrycode/pyrycode/issues/537)'s two-bug pair; the production-impact half (argv `--permission-mode dontAsk` in ptyrunner `buildArgs`) landed in [#538](538.md) one commit earlier, and the real-claude PASS predicted by the Phase A spike's allow-list-pre-emption mechanism is operator-verifiable only — the developer agent has no real claude in dispatch.

## Implementation

- **Single source of truth via const concat.** `internal/agentrun/selfcheck/selfcheck.go` introduces `const canonicalProbeTool = "Write"` and reformulates `canonicalPrompt` as `"Use " + canonicalProbeTool + " to create a file named probe.txt with content 'hello'. Be brief."`. Go evaluates two-const concatenation at compile time, so no `var`/`init` is needed — the prompt remains a `const` string and the probe-tool name is a single literal in the file. The detector compares `c.Name == canonicalProbeTool` rather than a literal; flipping the probe tool is now a one-line change at the const. The exact-case discipline (claude emits capitalised tool names; lowercased `"write"` MUST NOT match) is preserved verbatim from the prior `"Bash"` detector — the discipline is byte-stable, only the matched value is parameterised.

- **`probe.txt` is relative, sandbox-cleaned, and the canary itself.** The new prompt names a filename, not a path; under PASS (the deny-default holds) the file never lands. Under a boundary regression where claude attempts and succeeds the `Write`, the file lands inside `os.MkdirTemp("", "pyry-self-check-*")` (the workdir `runAgentRunSelfCheck` materialises at `cmd/pyry/agent_run_selfcheck.go:42`) and is reaped by the existing `defer os.RemoveAll(workdir)`. No new defer; no fixed `/tmp` path; the write **is** the canary the FAIL signal alerts on, not a separate hazard. Path-traversal surface is nil — the literal is compile-time `const` with no caller substitution and contains no `..` / leading `/` / env reference.

- **Three coupled renames, one new exported sentinel.** The contract identifiers move from Bash-specific to probe-agnostic: `Result.BashInvoked` → `Result.ProbeToolInvoked`; `ErrBashInvoked` → `ErrProbeToolInvoked` (message reworded to `"agentrun: self-check: probe tool invoked despite deny-default settings"`); `bashInvokedInRaw` → `probeToolInvokedInRaw`. The wrap-error site at `selfcheck.go:323` formats `canonicalProbeTool` via `%q` rather than a bare `"Bash"` literal, so the operator's FAIL diagnostic always names the current probe. `ErrTimeout` is unchanged (tool-agnostic by name). All five wrapper-error namespaces (`"mark workdir trusted"`, `"write settings"`, `"mint session id"`, `"self-check: jsonl read"`, `"terminated without end-of-turn or probe-tool signal"`) are preserved verbatim; only the last gets a one-word swap (`bash` → `probe-tool`).

- **Package doc grows ~30 → ~60 lines, documents the full deny-default contract.** The expanded comment at `internal/agentrun/selfcheck/selfcheck.go:1-57` names the three coupled halves end-to-end: (1) **argv half** — `--permission-mode dontAsk` (from [#538](538.md)), CLI reference cited for argv-overrides-settings precedence; (2) **settings-file half** — `permissions.defaultMode: "dontAsk"` written by [`internal/agentrun/settings`](../features/agentrun-settings-subpackage.md), the belt-and-suspenders pair with the argv half (different deterministic substrates per CLAUDE.md "Belt-and-Suspenders Means Different Fabric"); (3) **allow-list half** — `permissions.allow: ["Read"]`, with the explicit pin that `canonicalProbeTool` MUST NOT appear in `canonicalAllow`. The **read-only-Bash carveout** gets its own paragraph naming the [permission-modes reference](https://code.claude.com/docs/en/permission-modes) and the one-sentence rationale ("Write sits off the carveout; Bash does not") — this is the load-bearing why-not-Bash a future contributor will look for. SECURITY block is preserved verbatim.

- **Convention → deterministic-fail test.** `TestProbeToolIsNotInAllowList` (4 LOC) pins the invariant via `slices.Contains(canonicalAllow, canonicalProbeTool)`. `canonicalAllow` is `[]string` (not const-able for slice types in Go), so a future PR that appends `"Write"` to widen the allow list would silently break the invariant — PASS would become structurally unreachable but no compile-time signal would surface. The doc-comment convention at both definition sites is the soft belt; the test is the hard one. Surfaced by the security-review pass on the architect spec and folded into the AC list rather than left as a SHOULD-FIX recommendation, because the disjointness IS the load-bearing rationale for picking `Write` as the probe.

- **Operator FAIL banner rewritten + reference chain extended.** `cmd/pyry/agent_run_selfcheck.go:writeSelfCheckFailMessage` flips the quoted prompt (`"Use Bash to echo hello…"` → `"Use Write to create a file named probe.txt with content 'hello'."`) and the observation line (`name "Bash"` → `name "Write"`). The historical reference chain on the last two lines grows from `#329 / #336 / #470 / #473` to `#329 / #336 / #470 / #473 / #538 / #539`, so the operator reading a FAIL years from now can trace what the contract was at each cutover point. The PASS line at line 60 (`"… Bash refused."`) and the INCONCLUSIVE line at line 70 (`"…Bash invocation was observed…"`) get the same one-word swap to `"Write"`. The consumer doesn't import `canonicalProbeTool` — the operator-facing prose names the current probe tool directly, which is the right shape for a diagnostic message (no indirection through a const a developer can't see).

- **Test cascade is mechanical.** `bashLine` → `writeLine` (JSON shape: `name:"Write"`, `input:{"file_path":"probe.txt","content":"hello"}`); `selfCheckBashLine` → `selfCheckWriteLine` (same fixture, duplicated to keep the CLI test surface self-contained); `TestSelfCheck_BashInvoked` → `TestSelfCheck_ProbeToolInvoked`; `TestBashInvokedInRaw` → `TestProbeToolInvokedInRaw` (table cases rewritten: "Bash tool_use" → "Write tool_use", "Read tool_use is not Bash" → "Read tool_use is not Write", "lowercase bash does not match" → "lowercase write does not match"; the malformed-JSON and missing-name negatives are tool-agnostic and unchanged). The `TestRunAgentRunSelfCheck_FAIL` `required` substring list adds `"Use Write to create a file named probe.txt"` (matches the new banner prompt as a substring rather than verbatim, so small prose changes don't over-pin) plus `"#538"` and `"#539"` (additive — the existing `#329`/`#336`/`#470`/`#473` substrings still assert).

- **Out of scope, with rationale (so a developer doesn't accidentally pull them in).** `internal/e2e/realclaude/allowed_tools_enforcement_test.go`'s local `bashInvokedInRaw` helper (line 187, callsite line 70, `name "Bash"` fatal line 78) stays as-is — that file tests a DIFFERENT contract ("with `--allowed-tools=Read` the runtime gate refuses Bash specifically"); renaming its helper would lose the Bash-specific intent. The docstring at line 184 ("mirrors selfcheck.go:284 exactly") describes a **policy** mirror (skip on decode error), not a **name** mirror; the policy is preserved across the rename. Same applies to `tool_loop_test.go:77` and `doctor_poisoning_regression_test.go:73,108` — comments-only mirrors of the same policy, line numbers drift by a handful, leave alone. The streamrunner path (`PYRY_USE_STREAMJSON=1`) is a separate enforcement model (`--dangerously-skip-permissions` + `--allowed-tools`, explicitly bans `--permission-mode`) and not exercised by `--self-check`; out of scope per [#538](538.md)'s no-touch list, same applies here.

## Patterns established

- **Single source of truth via const concatenation, not via `fmt.Sprintf` / `var` / `init`.** The probe-tool name lives in `canonicalProbeTool` and is consumed by two physically distinct site-shapes: (a) string-embedded in `canonicalPrompt` (the prompt claude sees), and (b) identifier-compared in `probeToolInvokedInRaw` (the detector). Go's compile-time evaluation of `const A = "x"; const B = "use " + A + " here"` keeps both consts immutable and zero-allocation while preserving the single-source property. `fmt.Sprintf` would force `var`, breaking the const-ness; an `init()` would scatter the rationale across two file regions. **Generalises: when a sentinel value must appear in both a prompt-style human-readable string and an identifier-compared exact-case match, prefer const concatenation — Go evaluates it at compile time, the value remains immutable, and grep-for-the-literal still finds both call sites because both sites textually reference the const. The shape applies anywhere two coupled call sites need to agree on a value but live in different lexical forms (URL template + regex match, log format string + log-parse pattern, error-message-substring + test assertion). The const-concat is one line; the `var` + `init` alternative is three lines and a synchronisation point a future contributor can desynchronise without warning.**

- **Convention-to-deterministic-fail-test promotion: one-line `slices.Contains` for a two-identifier disjointness invariant.** `canonicalProbeTool MUST NOT appear in canonicalAllow` started as a doc-comment "MUST NOT" at both definition sites. The architect spec promoted it to `TestProbeToolIsNotInAllowList` (4 LOC) during the security-review pass. Cost is one test that runs every `go test ./...`; payoff is the bug class "two package-level identifiers must remain disjoint" can never silently regress (a future PR that appends `"Write"` to `canonicalAllow` to widen the gate would fail the test the same second the change lands). **Generalises: when a code invariant takes the shape "value A MUST NOT appear in collection B" (or its inverse) and both A and B are package-level, write the one-line `slices.Contains` (or `maps.HasKey`, or whatever the right deterministic check is) instead of relying on a doc-comment convention. The test costs nothing structurally — no fixtures, no mocks, runs in microseconds — and converts an invariant that was previously enforceable only by human attention into an invariant the compiler-adjacent toolchain enforces on every save. The pattern recurs whenever two package-level identifiers are required to be disjoint (allow-list / deny-list pairs, claimed-paths / forbidden-paths, accepted-modes / rejected-modes); the boilerplate is identical.**

- **Tool-agnostic identifiers when the matched value is parameterised.** Pre-#539: `ErrBashInvoked`, `Result.BashInvoked`, `bashInvokedInRaw` — three identifiers that named the matched VALUE, not the role. Post-#539: `ErrProbeToolInvoked`, `Result.ProbeToolInvoked`, `probeToolInvokedInRaw` — three identifiers that name the ROLE, with the matched value flowing through `canonicalProbeTool`. The rename costs one PR; the alternative compounds every time the probe tool rotates (a future ticket flips to `Edit`, the identifier still says `Bash`, the comment-rot bug class shipped here is shipped again). **Generalises: when the public name of a contract identifier (sentinel, struct field, helper function) bakes in a specific matched value, the identifier signals "this value will never change" — and the moment that signal becomes false, the codebase grows a fleet of comment-rot mismatches that drift over years. Prefer naming the ROLE (`ProbeTool`, `CanonicalGate`, `RefusalSignal`) over the VALUE (`Bash`, `Read`, `Write`); the value lives in a const next door. The cost of getting this right at first-write is zero; the cost of retro-fitting it (as #539 had to) is the entire test suite plus every consumer site plus a CLI banner rewrite. Apply this when the matched value is plausibly rotatable on a multi-year horizon — which most are.**

- **Test-credibility tickets pair structurally with production-impact tickets — the order matters.** [#537](https://github.com/pyrycode/pyrycode/issues/537) identified a two-bug pair: argv `default` shadowed the settings deny-default (production impact), and the test exhibit rode the read-only-Bash carveout (test credibility). The split delivered #538 first ([commit `09efe9f`](https://github.com/pyrycode/pyrycode/commit/09efe9f)), #539 second; the order is load-bearing because the new exhibit's real-claude PASS only proves the contract once the contract is actually being enforced. Had #539 landed first, the new exhibit would have FAILed on a healthy production binary (argv `default` would let claude attempt `Write` regardless of settings), and the operator would have spent a debugging cycle distinguishing "test exhibit broken" from "production boundary broken." **Generalises: when a single bug report splits into a "production silently broken" half and a "test silently passing" half, land the production fix first — the empirical signal from the test fix is only credible against a fixed production. Document the dependency in both ticket bodies (the production-impact half names the test-credibility half as a deferred follow-up; the test-credibility half cites the production-impact half as a prerequisite landed in the predecessor commit). The split-and-order discipline is cheap; the alternative is a debugging cycle on a freshly-deployed selfcheck that's structurally unable to PASS.**

## Lessons learned

- **A diagnostic test on the wrong side of a documented carveout can PASS-for-the-wrong-reason for years.** The Bash echo exhibit landed in #336 (original PTY selfcheck) and survived #375 (streamrunner rewrite) and #473 (ptyrunner rewrite) before the carveout was noticed in [#537](https://github.com/pyrycode/pyrycode/issues/537). Three rewrites copied the exhibit forward without anyone re-checking whether `echo hello` was on the right side of the deny-default contract — the assumption was "Bash is not in `["Read"]`, therefore Bash will be refused." The truth was "Bash is not in `["Read"]` AND Bash is on the carveout list, therefore Bash will be permitted via the carveout regardless of `["Read"]`." The two conditions are independent; the test exhibit needed BOTH "absent from allow list" AND "outside any documented carveout" — only the first was checked. **Generalises: when picking a diagnostic exhibit for a deny-default test, the necessary condition is "the probe action would be denied under the contract being tested AND only under that contract." Audit the test exhibit against the full set of documented exemptions, not just the obvious one. The audit costs one careful reading of the policy docs; the alternative ships a test that PASSes structurally regardless of whether the contract holds. The bug class is "the failure-mode probe is on the wrong axis" and the cheapest counter is to write down what the test would distinguish — if "X holds" and "X is bypassed by a carveout we forgot about" both produce PASS, the test isn't actually testing X. The Phase A spike (#329) explicitly verified Bash refusal empirically against a real binary, but didn't notice the empirical verification ran on a different machine than CI; the production binary on CI's headless runner under `-p` mode behaved differently than the operator's interactive binary, and the carveout-eligible probe masked the asymmetry. (Same shape as [#538](538.md)'s headless-vs-interactive lesson — both are reminders that the verification environment must match the production-failure environment.)**

- **The "three coupled invariants" specification for a deny-default probe.** Picking a probe tool that actually tests what the contract claims requires THREE conditions, not two: (a) absent from the allow list; (b) outside every documented carveout; (c) reliably attempted by the model rather than refused pre-emptively due to training. (a) alone gets you Bash-with-carveout (pre-#539); (a)+(b) alone would have allowed `Edit`, but `Edit` requires the target file to exist and claude will ask a clarifying question rather than attempt the tool — violating (c). The architect's spec lists all three explicitly and pins the empirical reasoning for each. The Phase A spike (#329) verified (c) for `Bash`; the (c) verification for `Write` is operator-AC and pending. **Generalises: when specifying the requirements for a deny-default probe (or any "this action should be refused" exhibit), enumerate the conditions independently and check each. The most common bug is conflating (a) + (c) into "the model will attempt it and it'll be refused"; the second most common is omitting (b). When the test fixture is a const, comment the three conditions at the definition site so a future re-pick keeps them coupled. `internal/agentrun/selfcheck/selfcheck.go:canonicalProbeTool` does exactly this post-#539.**

- **A diagnostic that produces a side-effect (a file write) is acceptable when the side-effect IS the canary AND is sandbox-cleaned.** The Bash echo exhibit had no file-write surface (stdout only). The new Write exhibit asks claude to create `probe.txt`. The file landing IS the FAIL signal — the operator's red badge surfaces on the same event that produces the write. The `os.MkdirTemp` + `defer RemoveAll(workdir)` wrapper at `cmd/pyry/agent_run_selfcheck.go:42` reaps the file via the existing teardown; no new defer needed. **Generalises: diagnostic tools whose stated job is to detect a specific failure mode are entitled to produce side-effects bounded by the failure they're testing — the side-effect cleanup discipline is the same one the failure imposes anyway, and the additional surface is "what's the blast radius if the test mis-fires." Audit: is the cleanup guaranteed by the test scaffolding regardless of where the failure happens? Is the path discipline tight enough that the side-effect can't escape the sandbox even under an adversarial model? Pre-#539's Bash echo passed both (no side-effect at all); post-#539's Write passes both (sandbox-cleaned, relative path, no caller substitution). The pattern fails when the side-effect can escape the test's cleanup or land on a path outside the test's control — at that point the diagnostic IS the hazard and you need a different probe shape (or a different test layer).**

## Out of scope (follow-ups)

- **Real-claude PASS verification (AC #4).** The fourth AC — `pyry agent-run --self-check` returning PASS against a real `claude` (≥ 2.1.150) — is operator-verifiable only; the developer agent has no real claude in dispatch. The Phase A spike (#329) verified the allow-list-pre-emption mechanism is tool-agnostic in claude's implementation; the architect spec predicts PASS reachable for `Write`. Empirical confirmation is the operator's next manual drill, flagged in the PR body. If claude empirically attempts `Write` (emits `tool_use name:"Write"` even though the gate would deny it), the selfcheck will FAIL despite the boundary actually holding — and the architect must revisit ("try `Edit` with a pre-existing seed file"; or accept a detector contract that distinguishes "attempted-and-denied" from "attempted-and-allowed" via `permission_denials` envelope inspection). The risk surfaces only on the integration AC and is not detectable by `go vet ./... && go test -race ./... && go build ./...`.
- **Narrow-probe coverage.** The selfcheck verifies that ONE tool (`Write`) is refused. A regression that allows `Edit` but not `Write` would PASS this test. The AC explicitly asked for a single probe with a single source of truth; broader coverage (rotating probes; multi-tool batteries; rotating-on-cron probe-tool selection) is a follow-up ticket and not motivated until a regression of that exact shape is observed (per CLAUDE.md "Evidence-Based Fix Selection").
- **Model-rename risk on exact-case detector.** If a future claude release renames `"Write"` to `"WriteFile"`, the exact-case detector will report no `tool_use` and PASS will fire on every run — including under a real boundary failure. This is a structural property of any exact-case detector and applied pre-#539 to the Bash probe as well; net change zero. Mitigations (integrate against a tool-name registry published by claude; fuzzy-match a known tool set) are out of scope for this ticket; revisit only if claude begins renaming tools in published releases.
- **Compromised-claude affordance.** A compromised claude binary can emit any `tool_use` shape it wants and trigger a false FAIL. That adversary already controls the operator's environment; the selfcheck is a diagnostic for the healthy-binary case. Not a defense-in-depth surface, out of scope per the security-review pass.
- **Trust-config cleanup (carried forward from #473).** `trust.MarkWorkdirTrusted` writes the selfcheck's throwaway workdir into `~/.claude.json :: projects[...]`; the CLI's `defer os.RemoveAll(workdir)` removes the directory but not the trust entry. Across many pyry boots the operator's `~/.claude.json` accumulates stale `/tmp/pyry-self-check-*` entries (~50 bytes each). Same `trust.UnmarkWorkdir` follow-up as #473's; not motivated by this ticket.
