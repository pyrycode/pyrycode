# Ticket #576 — ptyrunner unit tests hermetic against an ambient `PYRY_RECORD_DIR`

`internal/agentrun/ptyrunner`'s `make check` was non-deterministically red. The
non-recording `TestRun_*` tests build their `Config` via `helperRunCfg`, which never
sets `PYRY_RECORD_DIR`, so they **inherited whatever was exported in the ambient shell**.
When that var was set, the opt-out-by-default [flight recorder](../features/ptyrunner-package.md#session-flight-recorder-pyry_record_dir)
turned on for those `t.Parallel()` tests, which all share `const testSessionID` and a
second-granularity `<UTC-stamp>-<sessionID>.cast` path in the same ambient dir. Two
parallel tests starting in the same wall-clock second computed an **identical** path;
tui-driver's `O_EXCL` open failed with `file exists`, cascading into prune/rename
`no such file or directory` warnings and (for `TestRun_CtxCancelDuringStream`) a `did not
observe first emitted line within 5s` timeout. Purely a test-isolation defect — no
production bug; the recording scheme is correct and frozen by AC #4.

## Implementation

One test file, six lines: `internal/agentrun/ptyrunner/helper_test.go` (`ef2c129`). The
package already had exactly one `TestMain` (dispatching to the fake-claude helper on
`GO_PTYRUNNER_HELPER=1`, otherwise running the suite). The fix adds an
`os.Unsetenv("PYRY_RECORD_DIR")` to the **parent branch only**, immediately before
`os.Exit(m.Run())`:

```go
// #576: drop any ambient PYRY_RECORD_DIR so tests that don't explicitly
// opt into recording never inherit it. The recording gate (runner.go) reads
// os.Getenv in THIS process; clearing it once here makes every non-opt-in
// TestRun_* deterministic regardless of the caller's shell. Recording tests
// re-set it per-test via t.Setenv (which restores to unset on cleanup).
_ = os.Unsetenv("PYRY_RECORD_DIR")
```

- **Why the parent branch, not before the dispatch.** The recording gate
  (`runner.go:316` — `if dir := os.Getenv("PYRY_RECORD_DIR"); dir != ""`) runs inside
  `Run()` in **this** (parent) process, not in the fake-claude child. The
  `GO_PTYRUNNER_HELPER=1` child re-execs into `runHelper` and never reaches `m.Run()`; it
  is fake-claude and never records. Unsetting there would be pointless; the parent branch
  is both correct and minimal.
- **`os` was already imported** (`helper_test.go:6`) — no new import. The discarded
  return (`_ =`) follows the project's "document a deliberately-ignored error" convention;
  the inline comment is the justification (`os.Unsetenv` on a key returns nil in practice).
- **No production code touched.** `recordingPath` (`runner.go:585`), `const testSessionID`
  (`runner_test.go:40`), and `castOkNameRe` (`runner_test.go:756`) are all byte-identical.
  There is no session-id change for the matcher to track.

## Patterns established

- **Hermeticity is the root-cause lever; pick it over per-test uniqueness.** QA surfaced
  three independent levers, any one of which stops the collision — (a) unset the ambient
  var, (b) per-test unique `testSessionID`, (c) sub-second timestamp. Only **(a)**
  satisfies the AC's "neither dependent on **nor perturbed by** an inherited
  `PYRY_RECORD_DIR`": (b) leaves the env-coupling intact (tests still write stray `.cast`
  files into the inherited dir) *and* forces `castOkNameRe` to track a new id; (c) is
  forbidden by AC #4 (recording scheme frozen). A per-test session id would also be a
  defense for an **already-impossible** failure mode — the six opt-in recording tests each
  use a distinct `t.TempDir()`, so their paths are unique by construction regardless of the
  shared session id. Per Evidence-Based Fix Selection, don't add it. The fix is
  deterministic code in `TestMain`, not a stochastic per-test rule.
- **AC #1 ("recording paths unique per test") holds by construction post-fix.** Non-recording
  tests now generate *no* recording path at all; the only path-generating tests are the six
  opt-in recorders, each writing into its own `t.TempDir()`. No two `TestRun_*` can produce
  the same `.cast` path.
- **The one-time `TestMain` mutation doesn't race the per-test `t.Setenv`.** Go runs the
  serial `t.Setenv` recording tests to completion before the `t.Parallel()` batch. The
  unset is a single process-global mutation before any test runs; each recording test's
  `t.Setenv` overrides it for that test's duration and restores to the unset baseline on
  cleanup. No `t.Setenv` is added to any parallel test, so there is no
  `t.Setenv`-after-`t.Parallel()` panic risk and the env never leaks into the parallel batch.

## Testing

No new test functions — verification is by running the existing suite under the conditions
the ACs name (developer-reported green):

- **Hermetic under an exported ambient var (AC #2/#3):**
  `PYRY_RECORD_DIR=$(mktemp -d) go test -race ./internal/agentrun/ptyrunner/ -count=20` →
  `ok`, zero `file exists`, zero prune/rename `no such file or directory`, zero `did not
  observe first emitted line within 5s`, and **no stray `.cast` files** left in the scratch
  dir (non-recording tests no longer write there).
- **Hermetic with the var unset (AC #3):** `go test -race ./...-count=20` → `ok`.
- **Recorders still green:** `go test -race -run '^TestRun_Record' ./internal/agentrun/ptyrunner/ -count=5`
  (assert `.cast` creation, `-ok`/`-err` tagging, `0600`, valid asciinema v2, prune scoping).
- **`TestRun_RecordOff_NoFileCreated`'s explicit `t.Setenv("PYRY_RECORD_DIR", "")` is now
  redundant** (the baseline is already unset) but left in place — harmless, self-documenting
  the explicit-off contract, and keeps the diff to a single file.

## Lessons learned

- **A test that reads an env var it doesn't set is coupled to the operator's shell.** The
  `TestRun_*` tests never set `PYRY_RECORD_DIR`, so for a long time the flake was invisible —
  it only manifested on machines/CI where that var happened to be exported, and the failing
  *set* varied run-to-run with the same-second timing. The fix is to make the package binary
  hermetic at `TestMain` time, not to chase the symptom per-test. Any package whose
  production code reads an env var should clear/normalise it once in `TestMain` so the suite
  result never depends on the caller's environment.
- **Confirmed pre-existing, not a regression.** The collision reproduced on merge-base
  `a958670` with a *different* failing set — QA's PR #575 baseline comparison is what
  distinguished a test-isolation flake from a PR-introduced bug. Same family was flagged as a
  secondary observation while documenting [#573](573.md) (the two PR-unique #575 failures
  passed in isolation, proving collision artifacts).
