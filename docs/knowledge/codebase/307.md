# Ticket #307 — net: per-phone-conn dispatcher scaffold + empty handler table + id counter

First consumer of `Connection.Frames()` from the binary side. Before this
ticket, `cmd/pyry/relay.go` drained-and-discarded the routing-envelope
stream while no other package imported it. This slice installs the
demultiplexer + handler-table seam that downstream verb slices (#303 /
#304 / #305) plug into, plus the per-conn outbound id counter and
`Connection.Send` outbound surface. Auth gating and the WS 4401 close
path remain deferred to #308.

## Implementation

### New package `internal/dispatch` (~250 LOC, 2 files)

Pure: imports `internal/protocol` only. No I/O, no transport, no
`internal/relay` import — the dispatcher takes a generic
`<-chan protocol.RoutingEnvelope` and exposes outbound frames on a
channel the caller forwards. Carrier-agnostic by design; this is the
same shape as the #249 primitives.

- **`Handler`** — `func(ctx, *Conn, protocol.Envelope) error`. Non-nil
  return logged at WARN; no close-conn semantics in v1 (the auth slice
  #308 introduces the first real "close one conn" surface).
- **`Conn`** — per-`conn_id` state exposed to handlers.
  - `ConnID() string`
  - `NextID() uint64` — monotonic, starts at 1, `sync/atomic.Uint64`.
  - `Send(ctx, env protocol.Envelope) error` — marshals env, wraps in
    `RoutingEnvelope{ConnID: c.id, …}`, pushes on dispatcher outbound;
    selects on ctx.Done so shutdown unblocks a backpressured send.
  - `Reply(ctx, req, respType, payload) error` — convenience that sets
    `ID = c.NextID()`, `InReplyTo = &req.ID`, `TS = time.Now().UTC()`
    then calls `Send`. This is the AC-load-bearing helper for "
    `in_reply_to` on responses matches the request's `id`".
- **`Config`** — `Frames` (required), `OutboundBuffer` (defaults to 32),
  `Logger` (required). Nil `Frames` or `Logger` panics in `New`
  (programmer error; same posture as `transport.New`).
- **`Dispatcher`** — `New` / `Register` / `Run` / `Outbound`. `Register`
  panics on duplicate type so downstream slices fail loudly if they
  collide.
- **`Run(ctx)`** — blocks until `Frames` closes (returns nil) or `ctx`
  cancels (returns `ctx.Err()`). On return, every per-conn goroutine has
  exited and `Outbound()` is closed; the caller can `range` over it
  cleanly.

### Concurrency model

```
Run(ctx)  (demux goroutine)
  │
  ├─ select { ctx.Done | Frames close | env <- Frames }
  │      env: getOrCreateConn(env.ConnID) → send to per-conn input
  │
  └─ on ctx done / Frames close:
        close every per-conn input
        wg.Wait()
        close(Outbound)
        return
```

- **One demux goroutine.** `Run` itself.
- **N per-conn goroutines.** One per active `conn_id`, each reading a
  buffered `chan protocol.RoutingEnvelope` (size 8) and serially calling
  `handleOne` for each frame. Arrival order preserved within a conn; no
  cross-conn ordering guarantee.
- **Shared bounded outbound channel.** Per-conn goroutines push; the
  cmd/pyry forwarder pulls. Backpressure on a slow consumer pauses the
  per-conn goroutine, which is the intended flow control.
- **WaitGroup teardown.** `Run` waits for all per-conn goroutines to
  exit before closing Outbound — that's what makes
  `for env := range d.Outbound()` a complete drain signal in
  `cmd/pyry/relay.go`.

### Per-frame routing inside the per-conn goroutine

`handleOne` runs four branches against each `RoutingEnvelope`:

| Inbound shape | Wire code | `in_reply_to` |
|---|---|---|
| `Frame` not JSON-decodable | `protocol.malformed` | absent (no request id) |
| `PayloadEncrypted=true` | `protocol.unsupported` | `&req.ID` |
| `Type` not in v1 set | `protocol.unknown_type` | `&req.ID` |
| v1 `Type` with no handler registered | `protocol.unsupported` | `&req.ID` |
| v1 `Type` with handler | handler invoked | — |

Sentinel-to-code mapping happens inside the dispatcher (the consumer's
job per `docs/PROJECT-MEMORY.md:20`), not inside `protocol`. The
`IsV1Compatible` sentinels (`ErrUnsupported`, `ErrUnknownType`) are
distinguished via `errors.Is` and mapped 1:1 to the `Code*` constants.

### `internal/relay` — new `Connection.Send`

~10-line method (`connection.go:170-185`): marshals
`protocol.RoutingEnvelope` to JSON and forwards to `transport.Client.Send`.
Returns `transport.ErrDisconnected` (or `ErrNotConnected` / `ErrClosed`)
verbatim — the caller decides whether to drop or retry. Frames sent
while disconnected are lost; that's consistent with v1 protocol
expectations (reconnect runs a fresh `hello/hello_ack`, so per-conn
state on the relay is implicitly the wrong frame of reference for retry).

### `cmd/pyry/relay.go` — dispatcher wiring

Replaces the previous single drain-and-discard goroutine with three
goroutines plus a deterministic cleanup chain:

1. **Dispatcher goroutine.** `d := dispatch.New(...)`; `d.Run(ctx)` —
   the dispatcher does the actual `for env := range conn.Frames()` loop
   internally.
2. **Outbound forwarder.** `for env := range d.Outbound() { conn.Send(env) }`.
   Send errors logged at DEBUG (transport-internal reconnect already
   handles transient drops; the dispatcher's job ends at "frame queued").
3. **Wait classifier.** `conn.Wait()` arms the 4409 / ctx-cancel /
   terminal-error switch (unchanged from #301).

Cleanup waits in lifecycle order: `conn.Close()` triggers
`Connection.run` to close `c.frames` → `dispatcher.Run` observes the
channel close and returns → dispatcher closes `Outbound()` → forwarder
exits → `Wait` returns. `cleanup` waits on `dispatcherDone`,
`forwarderDone`, `waitDone` in that order so the daemon process never
exits with an in-flight WS handle.

The handler table stays empty in this slice. Downstream verb slices
(#303 `list_conversations`, #304 `send_message`, #305
`register_push_token`) each add one `d.Register(protocol.TypeXxx, …)`
call before the goroutines are spawned. Until then, every inbound frame
from a phone falls through to `protocol.unsupported`.

## Test coverage (`internal/dispatch/dispatch_test.go`)

All tests use stdlib only; same-package; pass under `go test -race`.

- **`TestEmptyTable_UnsupportedType`** — known v1 type
  (`TypeSendMessage`) with no handler returns
  `code=protocol.unsupported`, `in_reply_to=&req.ID`, `ConnID` echoed.
- **`TestUnknownType`** — `Type="bogus"` returns
  `code=protocol.unknown_type`.
- **`TestEncryptedRefusal`** — `PayloadEncrypted=true` returns
  `code=protocol.unsupported` (stricter rejection per
  `IsV1Compatible` check order).
- **`TestMalformedInnerFrame`** — `Frame: []byte("not json")` returns
  `code=protocol.malformed`, `InReplyTo == nil`.
- **`TestIDCounter_MonotonicPerConn`** — a stub handler calls
  `c.NextID()` four times for two distinct `conn_id`s; asserts each
  returns `[1, 2, 3, 4]` (per-conn ownership).
- **`TestReply_InReplyToMatchesRequest`** — `c.Reply` builds an envelope
  with `InReplyTo == &req.ID`, `Type == respType`, `ID == 1`.
- **`TestCtxCancel_Teardown`** — long-lived input; cancel ctx; assert
  `Run` returns within 1s and `Outbound` closes (no leaked goroutines).
- **`TestFramesClose_Teardown`** — close input; assert `Run` returns nil
  and `Outbound` closes.
- **`TestTwoConns_ArrivalOrderPreservedPerConn`** — interleaved frames
  for `conn-a` / `conn-b`; per-conn order honoured, no cross-conn
  ordering claim made.
- **`TestRegister_DuplicatePanics`** / **`TestNew_NilFrames/LoggerPanics`** —
  pin the programmer-error posture.

The e2e suite continues to pass because the new wiring is a strict
extension: `conn.Send` is additive, and the dispatcher's outbound stays
empty until a verb slice registers a handler. No e2e tests in this
slice — manual smoke at first verb-slice integration is enough.

## Patterns established / reinforced

- **Carrier-agnostic dispatcher.** `internal/dispatch` imports
  `internal/protocol` only. The relay package and the cmd/pyry wiring
  are the only things that know "this is a WS conn". Future transports
  (e.g. a local-loopback test harness) plug the same `Frames` /
  `Outbound` channels in.
- **Sentinel-to-wire-code mapping at the call site.** Reinforces the
  rule pinned in `docs/PROJECT-MEMORY.md:20`. Primitives
  (`protocol.IsV1Compatible`) return Go sentinels; the dispatcher
  does `errors.Is` → `Code*` mapping when constructing the error
  envelope. New code-paths in #303/#304/#305 follow the same shape.
- **Atomic per-conn id counter.** `sync/atomic.Uint64` Add returns the
  new value, which makes `NextID` a trivial `c.nextID.Add(1)` starting
  at 1 with no zero-vs-one fence-post bug. Future per-conn counters
  (`req_id` for outgoing requests, etc.) use the same shape.
- **Three-goroutine cleanup chain via channel-close lifecycle.**
  `Connection.run` → `dispatcher.Run` → `Outbound` close → forwarder
  exit. Each goroutine signals the next via channel close; cleanup
  waits in the same order. No extra synchronisation primitives.
- **Bounded backpressure as flow control, not as buffer.** Per-conn
  inputs (8) and outbound (32) are sized to absorb bursts, not to
  decouple producers from consumers. A slow consumer is supposed to
  pause the producer; that's how phone-side back-off propagates.

## Lessons learned

### `IsV1Compatible` check order pins the stricter rejection

The protocol primitive checks `PayloadEncrypted` before `Type`. An
encrypted frame with an unknown type therefore reports as
`ErrUnsupported` rather than `ErrUnknownType`. The dispatcher inherits
that ordering by using a single `errors.Is` chain. Reversing the order
in the dispatcher would silently downgrade the encrypted-refusal to
"unknown type", which is a privacy regression (the relay would tell a
v2-encrypted phone that the type was the problem, not the encryption).
Documenting the check-order invariant in the primitive's godoc
(`envelope.go:62`) was load-bearing; the dispatcher just composes it.

### Channel-close as lifecycle signal beats sibling done-channels

Three goroutines in `cmd/pyry/relay.go` (dispatcher, forwarder, wait
classifier) chain entirely via existing channel closes:
`Frames` close → dispatcher exit → `Outbound` close → forwarder exit.
No `done` channel needed for the dispatcher↔forwarder seam because
`range d.Outbound()` is itself the wait. Adding a sibling `dispatcherDone`
*only* to await the cleanup → fine; using it as a barrier *between* the
dispatcher and forwarder would have been a layer mistake. The right
shape: one channel per goroutine, closed when that goroutine exits, used
only by code that waits on that specific goroutine.

### `Register` before `Run`, not during

The `map[string]Handler` is read by the per-conn goroutine without a
lock. That's only safe if no `Register` call happens after `Run` starts.
Documenting that contract on `Register`'s godoc and panicking on
duplicate insertion is what makes the lock-free read defensible.
Downstream verb slices register at startup, then call `Run` — same
shape as `http.ServeMux.Handle` before `http.Server.Serve`.

## Out of scope (deferred)

- **Auth gating the first frame.** `relay.AuthenticateFirstFrame`
  exists (#249) but the per-phone-conn caller that delivers a token is
  #308's job. Until then, the first phone frame after `hello` hits the
  empty handler table and gets `protocol.unsupported` back — intentional
  per the architect's spec.
- **Per-conn close intent on handler error.** Handlers return `error`
  in v1 but the dispatcher only logs at WARN. #308's `AuthOutcome.CloseConn`
  introduces the first real close-conn intent; the architect's note is
  to extend `Handler`'s contract (or add a sentinel like `ErrCloseConn`)
  when that lands.
- **Per-conn close signal from relay.** The wire protocol does not
  currently emit a `connection_closed` envelope per `conn_id`. Per-conn
  goroutines live until daemon shutdown or whole-stream lifecycle end.
  Follow-up paired with #308 either extends the wire spec or adds a
  per-conn inactivity timeout.
- **Verb handlers.** `list_conversations` (#303), `send_message` (#304),
  `register_push_token` (#305) each register one route into the
  dispatcher's table; this slice ships the seam, not the routes.
