# Ticket #271 — `internal/protocol`: v1 handshake/control payload structs

Phase 3 Track C — the handshake-and-control slice of the v1 payload catalog (split from #256). Adds five typed Go DTOs for the `hello` (server + client roles), `hello_ack`, `error`, and `ack` payloads that slot into the framing primitive `Envelope.Payload (json.RawMessage)` once the dispatcher reads `Envelope.Type`. Pure data — no methods, no constructors, no behaviour. Sibling slices (messaging, conversations, backfill, push) are out of scope and follow in their own tickets.

## Implementation

- **One new production file, `internal/protocol/handshake.go` (~48 LOC including doc comments).** Five exported structs in spec-table-grouping order: `HelloServerPayload`, `HelloClientPayload`, `HelloAckPayload`, `ErrorPayload`, `AckPayload`. Single `import "time"` (the `json` tag mechanism is string literals — no `encoding/json` import needed in this file). Zero new package dependencies. No changes to `envelope.go` or `codes.go`.
- **Two `Hello*Payload` structs, not a union.** Spec models the binary's hello and the phone's hello as two structurally distinct frames sharing only the envelope type name (`"hello"`) and the dispatch site. `role` is the discriminator (`"server"` / `"client"`); field sets diverge — server has `server_id` + `binary_version`; client has `device_name` + `client_version` + optional `last_seen_ts`. A union (single struct, mostly-optional fields) would lose type-level encoding of which fields belong with which role and force every consumer to validate role-field consistency by hand. Two structs lets the dispatcher (#248–#250) decode based on `role` and pass a strongly-typed payload to each handler.
- **Optional fields are `*T` + `omitempty`; required fields are non-pointer.** Only `LastSeenTS *time.Time` (on `HelloClientPayload`) and `RetryAfterS *int` (on `ErrorPayload`) carry `omitempty`. All other fields — `Role`, `ServerID`, `BinaryVersion`, `ProtocolVersions`, `DeviceName`, `ClientVersion`, `ProtocolVersion`, `ConnID`, `Code`, `Message`, `Retryable` — are non-pointer and always emitted. Rejected `time.Time` zero-value as sentinel for `LastSeenTS` because `time.Time{}` marshals as `"0001-01-01T00:00:00Z"`, which would pollute the wire; pointer + `omitempty` is the only correct encoding for an optional timestamp.
- **`AckPayload` is `struct{}`, not `map[string]any` or `json.RawMessage`.** `json.Marshal(AckPayload{})` emits `{}` byte-for-byte, matching the spec's `"payload": {}`. Pinned by `TestAckPayload_RoundTrip` asserting `string(payloadBytes) == "{}"` on the marshalled output.
- **Field order matches the spec example order, not the markdown table order.** The JSON encoder emits fields in struct-declaration order; that's what the round-trip byte-equivalence check verifies. Reordering struct fields would break tests. The header comment on each struct names this so a future reader can't innocently reorder.
- **Five fixture files under `internal/protocol/testdata/` — one per type, each a complete `Envelope` with the payload inlined.** `hello_server.json`, `hello_client.json`, `hello_ack.json`, `error.json`, `ack.json`. Envelope-level field order matches `envelope.go`'s struct order (`id, type, ts, payload, in_reply_to`) — `omitempty` strips `payload_encrypted:false` and absent `in_reply_to`. `ts` is normalised to `2026-05-08T10:33:14.012Z` (the existing `envelope_full.json` value) so all fixtures share one canonical timestamp. Identifier strings shortened from the spec's illustrative `"8f7e..."` ellipsis form to concrete bytes (`"8f7e"`, `"c-7f3a"`); the spec's `"..."` is illustrative, not literal.
- **`hello_client.json` uses `last_seen_ts: "2026-05-08T08:14:02Z"` (no fractional seconds) to pin `*time.Time` round-trip behaviour.** Verifies that `time.RFC3339Nano` re-emits the same Z-suffixed string without fractional padding for the no-fractional case. Documented Go stdlib behaviour, but pinning it as a fixture means a future stdlib regression (or a refactor that swaps the `time.Time` codec) would surface here, not in production.
- **Five per-type `*_RoundTrip` tests in `internal/protocol/handshake_test.go` (~213 LOC).** Each test decodes the fixture as an `Envelope`, asserts `Type` against the matching `Type*` constant from `codes.go` (referenced by symbol, not literal), unmarshals `env.Payload` into the typed struct, asserts the discriminating field(s), then re-marshals payload → splices into envelope → re-marshals envelope → byte-equivalence check via `canonical(t, b)` (a `json.Compact` wrapper). Reuses `readFixture(t, name)` and `canonical(t, b)` helpers verbatim from `envelope_test.go` (same `package protocol` test binary, no copy-paste, no new helpers). Per-type discriminating asserts: `Role` value, identifier presence, `LastSeenTS != nil` for client, `InReplyTo != nil` for hello_ack/error, `RetryAfterS == nil` (omitempty-with-nil contract) for the error path, `string(payloadBytes) == "{}"` for ack.
- **Field-value asserts exist to localise failure messages, not to gate correctness.** The byte-equivalence check is the load-bearing assertion — when round-trip diff fires, it's faster to see "ServerID was wrong" than to eyeball two JSON blobs. Removing the field asserts would not weaken the contract; removing the byte-equivalence check would.
- **Test references `CodeAuthInvalidToken` constant in `error.json`.** Concrete `code` value matters less than the round-trip; `CodeAuthInvalidToken` is the spec's example value, used as a representative non-retryable error so the `Retryable == false` and `RetryAfterS == nil` assertions ride a plausible wire payload.
- **Architect's stdlib-behaviour assumption (`time.RFC3339Nano` no-fractional round-trip) confirmed empirically.** Spec § Open questions noted "if the test fails on `last_seen_ts`, normalise the fixture to include `.000Z`." Test passed on first run; assumption holds; fixture stays at `"2026-05-08T08:14:02Z"`.
- **No new ADR.** Architect's spec § Open questions is "None." Two implementer-choice points (single-file vs per-type-file layout; whether to add a second `error_with_retry.json` fixture) — chose single-file (matches the package's spec-table-grouping convention) and one fixture per type (AC stipulation, the present-path for `RetryAfterS` is structurally identical to `LastSeenTS` which IS exercised). Both stylistic.

## Out of scope (deferred)

- Sibling payload-catalog slices: messaging (`send_message` / `message`), conversations (`list_conversations` / `conversations` / `create_conversation` / `conversation_created` / `promote_conversation` / `conversation_updated`), backfill (`backfill_since` / `message_chunk` / `backfill_done`), push (`register_push_token`). Each lands in its own ticket with its own fixtures.
- `RetryAfterS` *present* path on `ErrorPayload`. AC stipulates one fixture per type and the spec's example omits this field; the omitempty-with-nil path covers half the contract; the present-path is structurally identical to `HelloClientPayload.LastSeenTS` which IS exercised. If a future ticket needs to pin the present-path, add the fixture then.
- Decode errors (malformed payload, wrong type, cross-type swaps). Not the AC's concern; the dispatcher (#248) owns that boundary.
- Runtime validation of `Role == "server"` / `Role == "client"`. Documented in struct comments; enforcement is the dispatcher's concern.
- Constructors, methods, factory helpers. AC explicit: pure DTOs.
- Auth/dispatch wiring (which struct to decode into, role-based type restriction, `hello_ack`-on-connect) — #248–#250.
