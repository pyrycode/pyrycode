# Ticket #290 — transport: harden FatalCloseCodes against in-flight Read cancellation

Production fix for the `TestServerIDConflict_FatalNoReconnect` stress flake (`ConnCount = N, want 1` under `go test -race -count=N` against `./internal/relay/`). The flake was visible: variable `ConnCount` (3/5/9/10) ruled out a flat counting bug — the transport WAS reconnecting on a fraction of iterations before one reconnect's close eventually got classified as 4409 (so `Wait()`'s `ErrServerIDConflict` assertion still passed, only the count assertion failed). Root cause: `coder/websocket`'s `prepareRead.done()` deferred override replaces an inbound `CloseError` with `ctx.Err()` when ctx fires while a `Read` is suspended; when `sendPump` (or `pingLoop`) errors first into `serve`'s `errCh` and `serve` cancels immediately, `recvPump`'s pending `Read` returns wrapped `context.Canceled` instead of the peer's 4409 `CloseError`, so no slot of `errs[]` carries the close status and the post-serve `FatalCloseCodes` check misses. Fix: in `serve`, if the first errCh arrival has no recognizable WS close status, grant a bounded 50ms grace for one more arrival BEFORE `cancel()` — preserves the close status in `errs[]` for the existing preference loop (#248) to pick. Sibling pin to #288's regression test, addressing the same close-frame-race surface from the opposite angle (#288 verified the preference loop works when both errors carry meaningful info; #290 ensures the close-status error is actually IN `errs[]` to be preferred).

## Implementation

### `internal/transport/wssclient.go` (+ ~40 LOC, modifies `serve`)

- **`closeFrameGrace = 50 * time.Millisecond`** — new package-level constant near the existing cadence block. Bounds the wait in `serve` for `recvPump`'s close-frame observation when the first pump error has no recognizable WS close status. NOT exposed via `Config` — wire-internal timing constant; only the test seam (`testOpts.closeFrameGrace`, parallel to existing cadence overrides) substitutes a shorter value.
- **`Client.closeFrameGrace time.Duration`** — per-Client field initialised from the package constant in `New`. Tests substitute via `newClientForTest` (`opts.closeFrameGrace > 0` branch in the constructor, identical shape to `pingInterval` / `pongTimeout` / `reconnectInitial` / `stabilityReset`).
- **`awaitCloseStatus(errCh, grace) []error`** — new package-private helper extracting the grace-wait logic. Reads one error from `errCh`; if its `websocket.CloseStatus` is `-1`, blocks up to `grace` for one more arrival; returns the 1- or 2-element slice. Caller (`serve`) is responsible for draining remaining slots AFTER calling `cancel()`. Extraction is deterministic-testability-driven: orderings A/B/B' (close first / non-close-then-close / non-close-then-timeout) are exercised against `awaitCloseStatus` directly with channel-fed errors, no goroutines / no relay / no real `Read` — eliminates the scheduler dependency that made the original symptom flaky to begin with.
- **`serve` rewire** (the two-line change at the previous `errs = append(errs, <-errCh)` / `errs = append(errs, <-errCh, <-errCh)` pair): now calls `errs := awaitCloseStatus(errCh, c.closeFrameGrace)` → `cancel()` → loop-drains the remaining 1 or 2 slots into `errs`. Preference loop (added by #248, refined by #288's regression test) is unchanged — `serve`'s tail still walks `errs[]` and returns the first error with a recognizable `websocket.CloseStatus`, falling back to `errs[0]`.

### Why this is correct under all orderings

- **Order A (recvPump first, `CloseError` arrives):** `awaitCloseStatus` sees `websocket.CloseStatus(first) != -1`, skips the grace, returns immediately. Behaviour identical to pre-fix. Pass.
- **Order B (sendPump/pingLoop first, generic net error):** grace fires. `recvPump`'s pending `conn.Read` typically returns within microseconds because the close frame is already buffered. `errCh` receives `recvPump`'s `CloseError`, preference loop picks it. **This is the flaky case the ticket exists for.** Pass.
- **Order B' (true black-hole drop — TCP RST, NIC drop, `recvPump` returns no close status either):** grace expires after 50ms, `cancel()` proceeds as before, no slot carries a close status, `Connect` reconnects. **Identical to pre-fix behaviour** — the grace-wait does not invent a new failure mode; it only widens the existing window for `recvPump` to surface a close frame that's actually inbound.

### Why this is a complementary fix to #288's preference loop

The preference loop pins "if `errs[]` contains a close-status error, return it." This ticket pins "the close-status error IS in `errs[]` when the peer sent a close frame, even when `serve` would otherwise cancel before `recvPump` surfaces it." Together: when the peer sends a fatal close, `Connect` classifies it as fatal regardless of (a) which pump returns first OR (b) whether the close frame had arrived before the first pump return.

### `internal/transport/wssclient_test.go` (+89 LOC)

- **`TestAwaitCloseStatus_GraceBranchPreservesCloseError`** — three flat sub-tests against the extracted helper:
  - `close_first_skips_grace` — pre-loaded `CloseError`, asserts <20ms elapsed and `CloseStatus(errs[0]) == 4409`. Order A.
  - `non_close_first_grace_catches_close` — pre-loaded non-close error, second goroutine pushes `CloseError` after a 5ms sleep, 200ms grace, asserts `len(errs) == 2` and one slot carries 4409. Order B — the regression-catching case.
  - `non_close_first_grace_expires` — pre-loaded non-close error, no second pusher, 10ms grace, asserts elapsed ≥ 10ms and `len(errs) == 1`. Order B'.
- **Determinism vs #288's busy-Send approach:** #288 documented the explicit tradeoff (probabilistic regression detection via `-count=10` busy-Send vs. a production test seam). #290 lands on the seam: `awaitCloseStatus` is a non-trivial helper extracted to a function boundary, NOT a hook into `serve`'s flow, and is *not* exported. The unit-tests-against-the-helper approach is the cheap right answer here precisely BECAUSE the grace branch's correctness is determined by `awaitCloseStatus`'s logic alone — the timing-bound concern in #288 was the goroutine ordering inside `serve`, which we deliberately don't try to exercise here.
- **No new integration-shape test was added** (per spec § "Regression test", Approach A "busy-Send with `closeFrameGrace = 5ms`" was the architect's preferred shape). The unit tests against `awaitCloseStatus` plus the existing `TestServerIDConflict_FatalNoReconnect` (which was failing pre-fix under `-race -count=N` and passes post-fix) cover the contract. Rationale: pinning the helper's behavioural contract is strictly stronger than pinning an integration shape that's still scheduler-dependent on goroutine arrival order.

## Patterns established

- **Extract for testability when a timing-bound branch can be expressed as a pure function over inputs.** `serve`'s grace branch is inherently racy at the goroutine level but deterministic at the channel level. Lifting `awaitCloseStatus(errCh, grace)` out into a non-exported helper turned a `-race -count=N` flake into a flat unit test. The decision rule from #288's "Probabilistic regression detection" lesson (avoid production test seams for test determinism) still holds — `awaitCloseStatus` is *not* a test seam, it's a real factoring of `serve`'s logic that happens to also be easier to test.
- **Wire-internal timing constants live as `package const`, not `Config` fields.** `closeFrameGrace` follows the precedent of `pingInterval` / `reconnectInitial` / `stabilityResetMin` — wire-internal cadence, not operator-facing. Tests substitute via `testOpts`; production is fixed. Operators shouldn't be tuning this.

## Lessons learned

### `coder/websocket`'s `prepareRead.done()` deferred override is the kind of library-internal behaviour you can only learn the hard way

`read.go:226-255` (in `coder/websocket@v1.8.13`) installs a deferred function that replaces the in-flight `Read`'s pending error with `ctx.Err()` if ctx fired during the read. From the outside, the library looks like it does the right thing — a `conn.Read` parked on a `CloseError{Code: 4409}` should return that `CloseError`. It does, UNLESS ctx fired first; then the close error is silently substituted by `ctx.Err()`. The implication for any concurrent-pump pattern (sendPump/recvPump/pingLoop sharing a child ctx, the consumer cancelling the ctx on the first pump return) is that cancelling before `recvPump` has surfaced the close frame turns a peer-initiated close-with-status into an indistinguishable `context.Canceled`. The fix is necessarily "wait before cancelling," not "look at the error harder." When debugging concurrent-pump tear-down code against a library, suspect the library's deferred error overrides BEFORE suspecting your own select.

### When variable counts in a flaky test ("3, 5, 9, 10") sit alongside a passing terminal-classification assertion, you're seeing a partial-recovery race, not a counting bug

The ticket-time symptom looked like the test was miscounting connections; the architect's spec correctly read this as "the transport IS reconnecting some fraction of the time, then eventually one reconnect's close IS classified as 4409 (terminal), which is why `Wait()` passes." The lesson generalises: a counting-style assertion that flakes with variable counts is rarely a counting bug — it's a recovery / retry path that fires some fraction of the time before the recovery happens to succeed in the way the test expected. The fix lives upstream of the counter, not at the counter.

### "Investigate before applying" worked here precisely because the architect ranked the three candidates first

The spec's pre-investigation read of the three candidates (test-side counting / transport-side close-status miss / connectedCh race) ruled out candidates 1 and 3 by inspection and named candidate 2 as "most likely — by inheritance from #288's spec." The developer's first task was confirmation via instrumented logging, not coding. This is the right cadence for transport-layer flakes where the candidate fixes have wildly different blast radii. Documenting it so the next stress-flake ticket follows the same protocol rather than diving into a fix.

## Out of scope

- **Semantic widening of `FatalCloseCodes` to non-close-frame disconnects** (TCP RST, NIC drop, where `recvPump` returns a generic IO error and no close frame ever reaches the client). The spec called this a "flag back to PO" follow-up — it's a protocol-level decision (does observing a TCP-level drop on a server-id-bound conn count as a 4409 conflict?), not a close-frame-visibility fix. No evidence yet that this surfaces in production.
- **Refactoring `serve`'s pump-error model to a structured errgroup or similar.** Single-function change. The grace branch is the minimum hardening.
- **`serveHooks` / `beforeCancel` callback** (architect's Approach B fallback for the regression test). Approach A's `closeFrameGrace = 5ms` + busy-Send was the preferred shape; the actual landed shape uses the function-extraction approach instead, which sidesteps the need for either.
