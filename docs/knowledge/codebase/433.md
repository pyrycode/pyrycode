# Ticket #433 — `internal/noise` flynn/noise IK wrapper for handshake + AEAD transport

Lands the narrow Go wrapper around [`github.com/flynn/noise`](https://github.com/flynn/noise) that owns the `Noise_IK_25519_ChaChaPoly_BLAKE2s` cipher-suite pin for [Mobile Protocol v2](../../protocol-mobile.md) and exposes `Responder` / `Initiator` state machines plus a `CipherState` whose `Encrypt`/`Decrypt` take no AD parameter — the empty-associated-data invariant from `docs/protocol-mobile.md` § *Transport* is enforced at the type system. No production consumer in this PR: the per-`conn_id` handshake routing lives in #434 and the re-key state machine in #435; both will hold pointers to `Responder` / `CipherState` values rather than reaching into `flynn/noise` directly. Closes the v2 e2e-encryption pre-requisite chain that started with [#430](../decisions/024-noise-ik-mobile-e2e.md), [#438](438.md) (keypair), [#439](439.md) (filesystem hardening), and [#432](432.md) (QR `server_static_pubkey` field).

## Implementation

- **New package `internal/noise`** lives one level under `internal/` (sibling to `keys`, `pair`, `identity`, `devices`). The ticket body offered `internal/encryption/noise` as an alternative — rejected by the architect spec because every other internal-package occupant is one level deep and an `encryption/` intermediate for a single occupant breaks the layout.
- **Single production file `internal/noise/noise.go`** holds the cipher-suite constant, `Responder`, `Initiator`, `CipherState`, the four state-machine methods, and the two encrypt/decrypt methods (~215 LOC). Splitting into `responder.go` / `initiator.go` / `cipher.go` was considered and rejected — each "section" is ~30 LOC and three-file overhead buys no readability.
- **Cipher suite pinned in exactly one place.** `var cipherSuite = flynnNoise.NewCipherSuite(flynnNoise.DH25519, flynnNoise.CipherChaChaPoly, flynnNoise.HashBLAKE2s)`. The suite name `Noise_IK_25519_ChaChaPoly_BLAKE2s` appears as a string only in the package doc-comment; a future suite migration is one edit, not a grep-and-replace across `internal/relay`. The `flynnNoise` import alias avoids collision with our own package name.
- **Public-key derivation via `crypto/ecdh`, not flynn.** `NewResponder` / `NewInitiator` accept a 32-byte private key and derive the matching public via `ecdh.X25519().NewPrivateKey(staticPriv).PublicKey().Bytes()`. flynn's `DHKey{Private, Public}` then receives the full pair. This matches the wire-compatibility of `internal/keys.StaticKey.PrivateKey()`'s output (`internal/keys` already uses `crypto/ecdh` for generation, so the bytes round-trip bit-for-bit).
- **Defensive slice-copy of every key argument.** Both constructors `append([]byte(nil), staticPriv...)` (and `peerStaticPub` on initiator) before handing to flynn's `DHKey` struct. A caller-side mutation after construction — or a hypothetical future `StaticKey` zeroisation — cannot corrupt the live handshake state mid-flight. The cost is two 32-byte copies once per handshake.
- **`Random: rand.Reader` is explicit** in both `Config` literals. flynn falls back to a default when `Random` is nil, but the explicit wire removes one class of "did the default change?" question and pins the entropy source visibly. `crypto/rand` is the only RNG path in production.
- **The responder swaps cs1/cs2; the initiator does not.** flynn's `WriteMessage` / `ReadMessage` return `(cs1, cs2)` where `cs1` carries initiator→responder traffic and `cs2` carries responder→initiator traffic. For the responder, that means cs1 is *recv* and cs2 is *send* (the swap lives in `WriteResp` at `internal/noise/noise.go:120`); for the initiator, cs1 is *send* and cs2 is *recv* (no swap in `ReadResp` at `internal/noise/noise.go:181`). Wrapping the asymmetry inside the package lets every caller use `send` / `recv` symmetrically. The swap convention is pinned structurally by `TestRoundTrip_BothDirections` — if `WriteResp` returned `(cs1, cs2)` the very first cross-side `Decrypt` fails with a MAC error.
- **Empty AD enforcement at the type system.** `CipherState.Encrypt(plaintext)` and `Decrypt(ciphertext)` have no AD parameter. They call flynn's `Encrypt(nil, nil, plaintext)` / `Decrypt(nil, nil, ciphertext)` internally — the first `nil` is the output buffer (allocate fresh), the second is the AD. There is no way for a caller to pass non-empty AD without editing `noise.go`, which would itself be a spec amendment. This is the wrapper's enforcement of `docs/protocol-mobile.md:197` ("Implementations MUST NOT pass a non-empty AD without a corresponding spec amendment").
- **`KeyLen = 32` const + `ErrInvalidKeyLength` sentinel.** Both constructors length-check before any flynn call; mismatched length → wrapped `ErrInvalidKeyLength` (`errors.Is`-matchable) and `nil` receiver. Wrapping prefix names which key tripped (`"noise: responder static key:"`, `"noise: initiator peer static pub:"`) so the caller can fix the call site. All other errors are bare wrapped `fmt.Errorf` with no sentinel — caller (#434) closes the WS with one close code per surface (4426 for handshake, 4421 for transport) regardless of the underlying flynn-noise reason. Adding finer-grained sentinels is deferred until a caller actually needs to branch.
- **No `HandshakeState` re-export.** The wrapper deliberately exposes only `Responder`, `Initiator`, `CipherState`, the four state-machine methods, the two encrypt/decrypt methods, plus `KeyLen` and `ErrInvalidKeyLength`. No flynn type leaks — no `HandshakeState`, no `Config`, no `HandshakePattern`, no `CipherSuite`, no `DHKey`. A future migration off flynn/noise touches only `noise.go`.
- **Zero logging, zero key bytes in error messages.** The package emits no `slog` calls and every error path wraps flynn's error with a `"noise: <op>:"` prefix that adds no input bytes, no key material, no plaintext. Pinned by `TestErrorMessages_DoNotLeakPlaintextOrKey` which encrypts a 16-byte high-entropy probe, tampers the ciphertext, asserts the resulting error string contains neither the hex nor the base64 encoding of the probe.
- **Concurrency contract: not safe for concurrent use.** Each `Responder`, `Initiator`, and `CipherState` is owned by one goroutine. flynn's `CipherState` carries a mutable 64-bit nonce counter; concurrent access would corrupt it. The wrapper adds no locks — adding them would mask programming errors at #434/#435 where the per-`conn_id` dispatch goroutine is the structural owner. Doc-comment on each type pins the contract.
- **`github.com/flynn/noise v1.1.0`** is the only new direct dependency (the latest stable, released Feb 2024). `go mod tidy` promoted `golang.org/x/crypto v0.51.0` from indirect to direct (flynn imports `golang.org/x/crypto/chacha20poly1305` and `golang.org/x/crypto/blake2s`); already present indirectly for #432's BLAKE2s fingerprint, so no new transitive concern. No CGo.

## Patterns established

- **Empty-AD enforcement belongs at the wrapper API, not in caller-side discipline.** The spec mandate (`docs/protocol-mobile.md:197`) bans non-empty AD in v2 transport; encoding that as "all callers must pass `nil`" would require code-review enforcement every time a new caller wires up encryption. Putting the rule in the function signature (`Encrypt(plaintext)` takes no AD parameter at all) makes the mandate impossible to violate without editing the wrapper itself. The pattern generalises: spec invariants that hold across every caller are the wrapper's responsibility to enforce structurally; per-caller policy belongs in caller-side validation.
- **One cipher-suite pin per package — string appears exactly once.** The literal `flynnNoise.NewCipherSuite(DH25519, CipherChaChaPoly, HashBLAKE2s)` lives in `var cipherSuite = …` at package scope and the human-readable name appears only in the package doc-comment. Future suite migration is a one-line edit to the var + a doc-comment update; consumers do not need to change. Compare with [#432](432.md)'s `Fingerprint([32]byte) string` exported once and reused — both shapes follow the "one-place-to-edit" discipline for cross-cutting cryptographic constants.
- **Asymmetry between sides is collapsed at the wrapper boundary.** flynn's `(cs1, cs2)` is "first key carries initiator→responder, second key carries responder→initiator" — true for both sides, asymmetric per role. The wrapper swaps inside `Responder.WriteResp` so every caller, regardless of role, receives `(send, recv)` in the same shape and uses `send.Encrypt(...)` / `recv.Decrypt(...)` symmetrically. The swap is pinned structurally by `TestRoundTrip_BothDirections`: if the swap is removed, the very first cross-side `Decrypt` fails the MAC check. The pattern: when a third-party library exposes an asymmetric API that maps naturally to a symmetric concept on this side, the wrapper is the place to do the collapse — not the per-callsite call.
- **Defensive `append([]byte(nil), s...)` slice-copy at trust boundaries with untrusted-lifetime input.** Callers hand the wrapper a `[]byte` whose backing array may live arbitrarily long after the call (or be mutated, or even zeroed in a hypothetical future `StaticKey` hardening). The constructor copies into flynn's `DHKey` struct so the handshake state is structurally isolated from any caller-side change. The cost is 32 bytes per key per constructor call — paid once per handshake, irrelevant. The pattern recurs every time a wrapper holds onto a caller-supplied slice past the function return; preferring the copy over documenting "callers must not mutate" eliminates a class of latent bugs.
- **Sentinel granularity matches caller-action granularity, not flynn-error granularity.** `ErrInvalidKeyLength` is the sole sentinel because that is the only error class the caller can branch on — for everything else (MAC failure, out-of-order, AEAD failure, counter exhaustion, oversize message) the caller closes the WS with one close code per surface regardless of the underlying flynn reason. Adding `ErrMACFailure` / `ErrCounterMismatch` would require the caller to branch on them; it does not. The same shape as [#439](439.md)'s `ErrInsecureKeyDirMode` / `ErrInsecureKeyFileMode` split — distinct sentinels there because the *operator* fix differs (chmod the dir vs. the file). Both are instances of "sentinel-distinction-equals-caller-action-distinction."

## Lessons learned

- **The AC's "wrong-peer-static-pubkey → ReadResp errors on the initiator side" wording does not match the IK message flow.** The acceptance criterion says *"initiator with wrong responder static → `ReadResp` errors on the initiator side after MAC failure."* The natural failure surface for a wrong peer-static is the responder's `ReadInit`, because that is the first DH-derived decryption in the IK pattern: the responder reads with `R_priv` but the initiator's `es`/`ss` were derived against `F_pub`, so DH outputs disagree and tag verification fails when flynn decrypts the encrypted-`s` field on message 1. The responder never produces a `respMsg`, so the initiator's `ReadResp` is never reached. The architect spec called this out explicitly under § *Wrong-key rejection*; the implemented test (`TestHandshake_RejectsWrongResponderStatic`) meets the spirit of the AC (handshake rejected, MAC failure observed) while accurately reflecting the message flow. The lesson generalises: when an AC's surface-of-failure description contradicts the protocol's message ordering, the architect spec should reconcile and the implementation should follow the spec, not the AC text. Document the reconciliation in the test's doc-comment so the next reader does not re-trip the same review.
- **`go mod tidy` does promote `golang.org/x/crypto` from indirect to direct when a new direct dep transitively imports it.** Before this ticket `x/crypto` was indirect (pulled in via [#432](432.md)'s BLAKE2s fingerprint helper); after `go get github.com/flynn/noise`, `tidy` rewrote it to direct because the file-level imports now reference it through `flynn/noise`'s exported APIs (`flynn/noise` itself imports `x/crypto/chacha20poly1305` and `x/crypto/blake2s`). No manual `go.mod` edit needed; the promotion is automatic and idempotent. Worth knowing for the next dep that pulls a previously-indirect into the direct set — the diff looks like a manual edit but is `tidy`'s doing.
- **flynn/noise's `WriteMessage` and `ReadMessage` accept a payload argument that becomes IK message early-data — no caller-side framing.** The spec calls for `hello` / `hello_ack` to "piggyback on the IK handshake" but does not pin the mechanism; flynn's payload-arg shape is exactly the natural fit. `WriteResp(earlyData)` passes `earlyData` straight through to `hs.WriteMessage(nil, earlyData)` and the responder side surfaces it as the return value of `ReadResp`. No separate frame, no length prefix, no extra round-trip. The same shape works for re-key (#435) when the early-data is empty — flynn handles zero-length payloads correctly. Worth pinning because the wrapper API surface is shaped around this fit; if a future Noise library lacks the payload arg, the migration becomes a bigger refactor.

## Out of scope (filed elsewhere or deferred)

- **Per-`conn_id` state machine + handshake routing in `internal/relay`** — **#434**. First production consumer of this wrapper.
- **Re-key state machine + `rekey_request` envelope** — **#435**. Second production consumer; creates new `Responder` / `Initiator` per re-key, atomically swaps the `(send, recv)` `CipherState` pair on the same dispatch goroutine, no lock needed.
- **JSON encoding of `hello` / `hello_ack` payloads into early-data bytes** — caller's concern (likely #434). The wrapper consumes and returns raw `[]byte`; it does not parse JSON.
- **Base64 encoding of `data` field on the wire** — `internal/relay` (#434). The wrapper consumes and produces raw bytes; base64 lives at the routing-envelope JSON boundary.
- **Migration off `flynn/noise`** — the narrow wrapper surface is the migration affordance, not a plan. Not blocked-by anything.
- **PSK support** (Noise `IKpsk2` et al.) — not in v2's threat model per [ADR 024](../decisions/024-noise-ik-mobile-e2e.md). `Config.PresharedKey` and friends intentionally not exposed.
- **Multi-suite negotiation** — v2 is one suite, hard cutover. [ADR 024](../decisions/024-noise-ik-mobile-e2e.md) § *Why hard cutover*.
- **`CipherState` in-memory zeroisation** — Go has no reliable zeroisation primitive; the per-handshake `CipherState`s live on the per-`conn_id` state for the conn's lifetime and are dropped by GC on disconnect. Same posture as [#438](438.md) for the static private key.
- **Finer-grained transport sentinels** (e.g. `ErrMACFailure`, `ErrCounterMismatch`) — deferred until a caller actually needs to branch. Today's caller (#434) closes the WS on any error with one close code per surface.
