# Ticket #506 — fail-on-additive-drift assertions in ptyrunner ↔ streamrunner byte-equivalence test

Extends `internal/e2e/realclaude/ptyrunner_byte_equivalence_test.go` (the #482 baseline) with two package-level allowlist tables (`expectedStreamRunnerOnly`, `expectedPtyRunnerOnly`) and an `additiveDriftViolations` helper that fails the byte-equivalence test when either runner emits a top-level event type or `result`-trailer field outside the cross-runner intersection plus its per-runner allowlist. The pre-#506 shape comparison (`reflect.DeepEqual` over `[]envelopeShape{Type, Subtype}`) only caught ptyrunner SHRINKING relative to streamrunner; either side GROWING a new event type or trailer field could still land silently. #506 closes the additive-drift half of that hole; sibling #505 (blocked on the [#503](https://github.com/pyrycode/pyrycode/issues/503) audit) tunes the allowlist contents per the per-field "must close" vs "document as omitted" decisions. Test-only change, ~290 LOC into a single file, no production-code edits.

## Implementation

- **Two package-level vars, named per the AC literally so failure messages are grep-able.** `additiveDriftAllowlist{Events, ResultTrailerFields}` struct (both fields `map[string]struct{}`, the canonical Go set type) groups the two facets per runner. `expectedStreamRunnerOnly` is populated initially; `expectedPtyRunnerOnly` is `map[string]struct{}{}`-empty (NOT nil — the architect spec calls out the explicit "intentionally empty" signal so a future reader doesn't confuse it for "not yet wired"). Both vars sit alongside the existing `ptyRunnerArgvFlags` table near the top of the file. Each populated entry carries a trailing `// #503` line comment per AC #5; the `uuid` entry adds the disambiguating "claude's own session UUID, distinct from session_id" because the field name otherwise reads like a duplicate of `session_id`.

- **Initial allowlist contents follow the architect's corrected count, NOT the AC's "8 fields + 4 events" paraphrase.** Final populated set is 7 trailer fields + 1 event, with two corrections vs the existing `envelopeShape` doc-comment enumeration the AC pointed at: (1) `result` is in the intersection (ptyrunner emits it as `""` from `emitter.go:258` / `:320`, no `omitempty`), so it is NOT streamrunner-only — omitting it means a real divergence on `result` surfaces as a real bug; (2) `usage.server_tool_use` is nested under the `usage` sub-object, but the AC scopes to top-level field names, so it does not belong here — top-level `usage` itself is in the intersection. The corrected list: `api_error_status`, `duration_api_ms`, `fast_mode_state`, `modelUsage`, `permission_denials`, `ttft_ms`, `uuid` (7 fields) + `rate_limit_event` (1 event). Recording the corrected count in this doc so a future contributor reconciling against the AC body doesn't waste turns chasing "8".

- **Three new helpers placed next to `extractShapes`.** `extractEventTypeSet([]byte) (map[string]struct{}, error)` walks `bytes.Split(stream, '\n')`, unmarshals each non-empty line into `struct{ Type string }`, and returns the set of distinct `type` values (empty string included — an envelope missing `type` is itself drift worth surfacing). `extractResultTrailerFields([]byte) (map[string]struct{}, error)` mirrors the same loop, filters for the FIRST line whose `type == "result"`, then re-unmarshals that line into `map[string]json.RawMessage` and returns the keys; errors with `"no result line found"` if the loop exhausts. `additiveDriftViolations(streamRaw, ptyRaw []byte) []string` orchestrates: calls both extractors on both streams (helper errors become violation strings, not panics — a malformed stream is itself a drift signal), then runs four subset checks (`streamEvents − ptyEvents` against `expectedStreamRunnerOnly.Events`; mirror for `ptyEvents`; same for the field sets) and appends one violation message per failure.

- **Returning `[]string` instead of taking `*testing.T` is load-bearing for the self-check.** The self-check sub-tests feed hand-crafted byte fixtures directly to `additiveDriftViolations` and assert on the returned slice — no fake `*testing.T` shim needed. The real test (`TestPtyRunnerVsStreamRunner_StructuralEquivalence`) loops the returned slice with `t.Error(v)` (not `t.Fatal`) so the field-level invariants 7-10 still run after a drift failure and all signals surface in one test output. Spec § "Why return []string" pins the rationale.

- **Failure-message shapes name (a) the unknown identifier AND (b) the allowlist table to update.** Per AC #1 — failure messages quote the offending event-type or field name with `%q`, then reference the table identifier (`expectedStreamRunnerOnly.Events`, `expectedPtyRunnerOnly.ResultTrailerFields`, etc.) literally so a future contributor reading a CI failure has a single edit point. The streamrunner-side messages suggest "either add to the table OR treat as a real bug" (streamrunner forwards claude verbatim — an unknown emission from claude is news either way); the ptyrunner-side messages suggest "either add to the table OR file a follow-up ticket to align ptyrunner's emitter" (ptyrunner's `streamjson.Emitter` is owned by this repo, so the actionable next step is different).

- **Wired into the existing test as one block right after the `reflect.DeepEqual` shape check.** Around `ptyrunner_byte_equivalence_test.go:328` (post-#506: ~445), the call is `for _, v := range additiveDriftViolations(streamOut.Bytes(), ptyOut.Bytes()) { t.Error(v) }`. Placed BEFORE the existing field-level invariant block (`checkInit` / `checkUserContains` / `decodeResultTrailer`) so a drift failure surfaces alongside any concurrent invariant failure in the same test output. The block comment above the loop calls out the SHRINKING-vs-GROWING asymmetry so a future reader understands why both checks coexist.

- **`TestAdditiveDriftAssertion_SelfCheck` is fully hermetic — does NOT call `claude` or hit the network.** Inherits the `e2e_realclaude` build tag from the file but runs against an inline `const intersectionBaseline` JSONL string. Six sub-tests cover the (event-type, result-field) × (streamrunner-side, ptyrunner-side) × (drift, no-drift) cross-product plus an allowlist-absorbs-asymmetry sanity check using the literal `api_error_status` entry: `intersection_match_no_drift`, `streamrunner_extra_event_type`, `ptyrunner_extra_event_type`, `streamrunner_extra_result_field`, `ptyrunner_extra_result_field`, `allowlisted_streamrunner_field_does_not_fire`. Each drift sub-test asserts exactly one violation, that the violation message names the synthetic identifier, AND that it names the appropriate allowlist table identifier — protecting both the drift detection AND the "single edit point" promise.

- **Fixture injection is a tiny in-test `inject` closure, not a fixture-library import.** `strings.Replace(stream, target, replacement, 1)` with a fatal-on-no-match guard; `inject(t, intersectionBaseline, ` `"usage":{}` `, ` `"usage":{},"synthetic_stream_field":"x"` `)` is enough to splice a new top-level key into the result line. Hand-rolled per the architect's recommendation — the realclaude fixture helpers (`RunPyryAgentRun`, `WithWorktreeAuthenticated`) wrap real-claude invocations and aren't reusable for byte-level fixture construction.

## Patterns established

- **Symmetric subset assertions over unidirectional ones, for cross-implementation wire contracts.** When two implementations of the same wire contract must agree, a shape-equality check (`reflect.DeepEqual` over normalised tuples) catches one side SHRINKING but misses either side GROWING — both sequences grow the same new element and stay equal length, OR the new element is in a field-set the shape comparison doesn't visit. The symmetric subset check (`A − B ⊆ allowlistA`, `B − A ⊆ allowlistB`) catches both directions independently; the allowlist tables turn the "is it documented" question into a per-identifier source-tracked decision rather than a generic "strict subset by design" justification that admits anything. Generalises whenever a wire-shape comparison spans two implementations with disjoint-by-design surfaces.

- **Per-identifier allowlists with citation comments over generic-tolerance prose.** The pre-#506 `envelopeShape` doc-comment described the field-set delta in English ("streamrunner forwards 8 fields ptyrunner doesn't emit"). That worked while the delta was the audit's working assumption — but it failed open: anything *new* that ptyrunner or streamrunner emitted slid in under the same generic justification. Replacing the prose with a named `map[string]struct{}` per side, each entry carrying a citation comment to the audit ticket or doc, makes the tolerance per-identifier and fail-closed. Failure messages naming the table identifier give a contributor reading the failure the single edit point. Generalises whenever a contract has "documented exceptions" — replace prose with named sets, fail closed by default.

- **Self-check sub-tests for assertion logic, gated by the same build tag as the production test.** `additiveDriftViolations`'s correctness is itself a regression target: a future refactor that breaks the subset logic must not silently pass the byte-equivalence test. The fix is a sibling `TestAdditiveDriftAssertion_SelfCheck` that feeds hand-crafted byte fixtures through the helper and asserts on the returned violation slice. Hermetic (no `claude`, no network), runs in microseconds, but inherits the build tag so it co-locates with the contract it guards. The helper signature returning `[]string` (not taking `*testing.T`) is what makes this possible — the assertion logic and the assertion-test logic are decoupled. Generalises whenever an assertion helper has multiple failure modes that warrant their own regression coverage.

- **Mirror cmd-package-main argv shapes in test-local helpers with a source-of-truth pin comment.** Inherited from #482 and reaffirmed here: `cmd/pyry/agent_run.go:buildStreamRunnerClaudeArgs` is `package main` and not importable; the test-local `streamRunnerArgs` mirror keeps the byte-equivalence test pinned to the production argv shape without exporting the helper. Drift in either direction (cmd-side rename, test-side stale-bit-rot) is caught by the next byte-equivalence run because the streamrunner side stops emitting compatible argv. Generalises whenever a `package main` argv builder needs to be pinned from a test in a sibling package.

## Lessons learned

- **AC numerics paraphrased the source catalogue inaccurately — defer to the cited source, not the paraphrase.** The ticket body AC #1 said "4 events + 8 fields per #503's catalogue." The architect's spec § "Initial allowlist contents" + § "Note: corrections to the existing doc-comment enumeration" identified that two of the 8 listed fields were misclassified (`result` is in the intersection; `usage.server_tool_use` is nested), and the catalogue's "4 events" was a paraphrase that the per-source-of-truth audit (#503's table) doesn't actually back. Final populated set is 7 + 1, with the divergence documented inline in this codebase doc so the next contributor reconciling against the AC body has the answer. Generalises: when an AC paraphrases a cited source, the source wins; record the divergence with the rationale rather than fighting to match the paraphrase.

- **Returning `[]string` instead of taking `*testing.T` is what unlocked the self-check sub-tests.** First-draft mental model was "the helper calls `t.Errorf` directly per violation" — natural for a test-internal helper. But that shape forces the self-check sub-test to construct a fake `*testing.T` (or a `*testing.T` whose `Errorf` writes into a buffer the sub-test then reads), both of which are awkward and fragile. Flipping to `[]string` made the assertion logic pure-functional, the real-test call site one line (`for _, v := range ... { t.Error(v) }`), and the self-check sub-tests trivial. Generalises: when a test-internal assertion helper might want regression coverage of its own, return data the caller acts on rather than mutating an external sink — the test-of-the-test is then a plain unit test, not a fixture exercise.

- **The pre-existing `envelopeShape` doc-comment is now stale relative to the allowlist.** Code review's NIT #1 (`ptyrunner_byte_equivalence_test.go:128-141`) called out that the doc-comment still enumerates the original "8 streamrunner-only fields" list, including `result` and `usage.server_tool_use` — both of which the new allowlist (correctly) excludes. The doc-comment was left as-is per the ticket scope ("ship the mechanism, not a rewrite of the audit prose"), but the allowlist is now the authoritative source. Recorded as a follow-up under § Out of scope below; the next contributor touching the doc-comment can replace the field enumeration with a pointer to `expectedStreamRunnerOnly.ResultTrailerFields` so the two stop competing for ground truth.

- **Open question #1 (assistant vs message event-type spelling) self-resolved before landing.** The architect spec flagged the empirical question "does the minimal-prompt run produce `type:"assistant"` on the streamrunner side and `type:"message"` on the ptyrunner side?" with the recommendation "populate only `rate_limit_event` initially, run the test, add entries if drift fires." On the merge run, the existing `reflect.DeepEqual` shape comparison was still passing on main (both runners emit matching `(type, subtype)` sequences for the minimal `--max-turns=1 "Reply with the single word OK"` prompt) — meaning either both runners emit `assistant`, or both emit `message`, but they agree. The new assertion adds no fresh signal on this axis until a future run surfaces a divergence; if it ever does, sibling #505 is the right place to land the entry per the audit's per-identifier decision.

## Out of scope (follow-ups)

- **Sibling ticket [#505](https://github.com/pyrycode/pyrycode/issues/505)** — align the test's tolerance scope with the [#503](https://github.com/pyrycode/pyrycode/issues/503) audit decisions. Blocked on #503. The mechanism #506 ships is correct regardless of how the audit lands; #505 tunes the allowlist contents per the per-field "must close" vs "document as omitted" verdicts. If #503 decides an item must close, #505 removes it from the allowlist and a real fix to the offending emitter follows; if #503 decides an item is a documented omission, #505 confirms the existing entry (or adds it to `expectedPtyRunnerOnly` if the asymmetry runs the other way) and the citation comment is updated to point at the audit doc instead of the bare `#503` placeholder.

- **`envelopeShape` doc-comment vs allowlist source-of-truth reconciliation.** Code review NIT #1: the doc-comment's "8 streamrunner-only fields" enumeration is now stale (includes `result` and `usage.server_tool_use` which the allowlist correctly excludes). Replace the enumeration with a single pointer to `expectedStreamRunnerOnly.ResultTrailerFields` so the two stop racing each other. Cheap edit; deliberately deferred to keep #506's diff scoped to the new mechanism.

- **Cascade-noise reduction in `additiveDriftViolations` when one extractor errors.** Code review NIT #2: if `extractResultTrailerFields(ptyRaw)` errors (e.g. ptyrunner emitted no result line), the field-set loops still iterate stream-side fields against a nil `ptyFields`, and every stream-side field outside the allowlist surfaces as a separate cascade violation after the underlying "no result line found" message that already explains the failure. Behaviour is correct, just noisy. Short-circuiting the field-set comparison when either extract errored would cut the noise; would only matter if a real run produces a malformed stream.

- **Sub-object structural divergence checks (`usage.server_tool_use` and the rest).** The current assertion is scoped to TOP-LEVEL identifiers per AC; nested-object delta is out of scope. If #505's audit decides nested deltas matter, a separate helper that walks the trailer's sub-objects (specifically `usage`) and asserts on its key set is the right shape.
