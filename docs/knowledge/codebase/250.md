# Ticket #250 — relay: `register_push_token` handler

Phase 3 Track C — the binary's inbound handler for the phone's `register_push_token` frame. Composes A4 (devices.json registry, #209), A5 (`Validate`, #210), C1 (envelope types + `RegisterPushTokenPayload`, #275), C4 (#249 — first-frame auth returning a matched `Device`), and #282 (Device gains `Platform` / `PushToken`). Per `docs/protocol-mobile.md` § Phone background behaviour, the persisted `(platform, push_token)` triple is load-bearing for the offline-phone UX: a backgrounded phone closes its WS and is woken via APNs/FCM using the stored token. This ticket lands the pure inbound handler — dispatcher-side wiring (per-conn `Device` cache, per-conn id counter, type-dispatch) is a future sibling. Out of scope: actually calling APNs/FCM.

## Implementation

### New sub-package `internal/relay/handlers/`

Sibling to `internal/relay/auth.go` and `internal/relay/connection.go`, not a child. Depends on `internal/devices` + `internal/protocol` only — **no import of `internal/relay`**, so the future dispatcher (in `internal/relay`) can import `handlers` without a cycle. `auth.go` stays in `internal/relay` because it is the gate into the dispatcher, not a per-type handler — the dispatcher calls it directly during conn setup before any frame dispatch.

### `internal/relay/handlers/register_push_token.go` (~202 LOC)

- **Single exported function `Handle(routing, device, reg, registryPath, nextID, logger) (RoutingEnvelope, error)`** — pure: routing envelope in, routing envelope out, plus the optional registry side-effect. No goroutines, no channels owned. The dispatcher (future) is responsible for routing the inbound frame here only after type-dispatch on `TypeRegisterPushToken`; `Handle` does not re-verify the type.
- **Five behavioural branches, one error sentinel.**
  | Branch | Trigger | Response | Side effect |
  |---|---|---|---|
  | Malformed frame | `json.Unmarshal(routing.Frame, &Envelope)` fails OR `json.Unmarshal(inner.Payload, &RegisterPushTokenPayload)` fails | `(RoutingEnvelope{}, ErrMalformedFrame)` — dispatcher owns the `protocol.malformed` response | none |
  | Unauth | `device == nil` (dispatcher bug — auth was supposed to close the conn) | `error` envelope: `Code=auth.invalid_token`, `Retryable=false`, `Message=msgUnauthorized` | none |
  | Dedupe | `(payload.Platform, payload.Token, payload.DeviceName) == (device.Platform, device.PushToken, device.Name)` | `ack` envelope | **none — does NOT call `UpdatePushRegistration`, does NOT call `Save`** |
  | Gone mid-conn | `UpdatePushRegistration` returns `false` (concurrent revoke between auth-accept and frame arrival) | `error` envelope: `Code=auth.invalid_token`, `Retryable=false` (same UX as unauth) | none |
  | Save failure | `reg.Save(registryPath)` returns non-nil | `error` envelope: `Code=server.binary_busy`, `Retryable=true`, `RetryAfterS=nil` | **in-memory IS mutated; disk is not** |
  | Write success | triple differs and Save succeeds | `ack` envelope | in-memory + disk both updated |
- **Dedupe is the load-bearing optimisation.** Without it, the binary rewrites `devices.json` on every WS connect — ~100 bytes × N reconnects/day per phone of disk churn on a registry that almost never actually changes. The AC's "test must assert no write occurred" maps to `errors.Is(os.Stat(path), fs.ErrNotExist)` on the dedupe path; the test deliberately does NOT pre-`Save` the file so the spy is structural.
- **`Name` is part of the dedupe triple.** Per the registry mutator's doc-comment, the phone is the source of truth for its self-reported name (a user renaming the device in iOS Settings propagates). So `(Platform, PushToken, Name)` is the comparison, not just `(Platform, PushToken)`. The handler's "re-register with different `device_name` only" case is implicit in the "changed triple" test — any of the three fields differing takes the write path.
- **Save-failure leaves in-memory mutated.** Documented post-condition, mirroring `Validate`'s `LastSeenAt` pattern: in-memory is the runtime source of truth; the next successful Save catches disk up. The phone retries per the retryable contract; on the retry, dedupe will succeed (in-memory already matches) and no further write attempt occurs. `TestHandle_SaveFailure_EmitsServerBinaryBusy` pins this by asserting `reg.FindByTokenHash(...).PushToken == "new-fcm"` after the failed call.
- **`nextID` is the dispatcher seam.** Auth's `hello_ack` used envelope id 1; the dispatcher's per-conn counter allocates `2..N` for subsequent outbound frames. The handler stamps `nextID` onto every response envelope (ack and error alike) and never increments it.
- **`wrap(connID, inReplyTo, nextID, envType, payload) (RoutingEnvelope, error)` is a file-local helper** that marshals the payload, builds an `Envelope{ID:nextID, Type:envType, TS:time.Now().UTC(), Payload:…, InReplyTo:&inReplyTo}`, marshals it, wraps in `RoutingEnvelope{ConnID:connID, Frame:…}`. **Deliberately duplicates `relay.buildResponse`** rather than importing across the new sub-package boundary — premature to lift into `handlers/internal/wire` for one handler. Lift if/when a second handler lands.
- **File-local messages, not exported constants.** `msgUnauthorized = "not authenticated; handshake required before register_push_token"` and `msgBinaryBusy = "registry save in progress; retry"` are unexported. The AC tests assert on `Code` + `Retryable`, never on `Message`, so spec-sentence pinning is not required here.
- **`ErrMalformedFrame` sentinel.** Mirrors `relay.ErrMalformedHelloFrame`: structural JSON errors are dispatcher concerns. The handler returns `(RoutingEnvelope{}, ErrMalformedFrame)` with no synthesized response envelope. The dispatcher maps it to its existing `protocol.malformed` handling (sibling future ticket owns the response shape).

### `internal/devices.Registry.UpdatePushRegistration` (+15 LOC in `registry.go`)

```go
func (r *Registry) UpdatePushRegistration(tokenHash, platform, pushToken, name string) bool
```

- **Identifies the target row by `TokenHash`.** The only stable key the handler already has from the auth-time `Device` snapshot — `Registry.Validate` returns a value-copy, not a pointer into the slice, so the handler cannot mutate-in-place from outside the package.
- **Mutates three fields together** (`Platform`, `PushToken`, `Name`) under `r.mu`. Linear scan, byte-exact `TokenHash` compare, in-place assignment, returns `true` on match / `false` on miss. No logging (the package never logs).
- **Caller persists via `Save`.** The mutator is the in-memory primitive; the handler chains `UpdatePushRegistration` + `Save` as two distinct mutex acquisitions (each method takes/releases `r.mu` independently, which is intentional — `Save` snapshots under the lock then writes outside, per [ADR 020](../decisions/020-devices-registry-snapshot-then-write.md), keeping the auth-path read predicate `Validate` unblocked during the file I/O window).
- **`Name` overwrite is intentional.** The protocol's `register_push_token.device_name` is the phone's current self-reported name; pyrycode treats the phone as the source of truth (iOS Settings rename should propagate). The original pairing-time `Name` carries no protocol-level invariants. Documented in the doc-comment so a future PR doesn't "fix" the overwrite.

## Tests

### `internal/relay/handlers/register_push_token_test.go` (~311 LOC, six tests)

Package `handlers` (not `handlers_test`) so the test could read file-local consts if needed — though in practice it asserts on `Code` + `Retryable` only and never touches `msgUnauthorized` / `msgBinaryBusy`. Stdlib only. Each test is its own function (test bodies diverge on which assertions run — table-driven would obscure the shape).

- **`TestHandle_FirstTimeRegister_WritesAndAcks`** — Device with empty `Platform`/`PushToken`, name `"old-name"`. Payload `(fcm, fcm-token-abc, Juhana's Pixel 8)`. Asserts ack envelope shape + `os.Stat(path)` succeeds + reload via `devices.Load` shows all three fields updated. Pins the happy-path write.
- **`TestHandle_ReregisterIdentical_NoWriteAndAcks`** — Device already carries the incoming triple. **Crucially, does NOT pre-`Save`** — so if dedupe were broken and Save fired, the file would be created. Asserts ack + `errors.Is(os.Stat(path), fs.ErrNotExist)`. This is the structural spy on the dedupe contract; the AC's "no write occurred" assertion turns on this.
- **`TestHandle_ReregisterChanged_WritesAndAcks`** — Device with old triple `(fcm, old-fcm, phone)`. Pre-`Save` so the file exists. Payload changes only `Token` → `"new-fcm"`. Asserts ack + reload shows `PushToken == "new-fcm"`. **Uses content-based equality, not mtime** — sidesteps CI filesystem mtime-resolution flakes.
- **`TestHandle_SaveFailure_EmitsServerBinaryBusy`** — write a regular file at `<tempdir>/blocker`, point `registryPath` at `<tempdir>/blocker/devices.json`. `Save`'s `MkdirAll` fails because a regular file blocks the parent path. Asserts `error` envelope with `Code == "server.binary_busy"`, `Retryable == true`, `RetryAfterS == nil`. **Plus pins the post-condition that in-memory IS mutated** despite the disk failure (`reg.FindByTokenHash(...).PushToken == "new-fcm"`).
- **`TestHandle_UnauthenticatedConn_EmitsAuthInvalidTokenNoWrite`** — `device = nil`, registry seeded with an unrelated device. Asserts `error` envelope with `Code == "auth.invalid_token"` / `Retryable == false`, no file written, in-memory device count unchanged.
- **`TestHandle_MalformedFrame_ReturnsSentinel`** — `routing.Frame = []byte("not-json")`. Asserts `errors.Is(err, ErrMalformedFrame)` and zero-value `RoutingEnvelope` response. Off-AC but pins the sentinel.

`assertEnvelopeShape(t, resp, wantType)` is the file-local helper: asserts `ConnID` echo + `Envelope.Type` + `ID == testNextID` + `*InReplyTo == testRequestID`. Returns the decoded envelope so the test body can pull the payload out for further assertions (the error tests do this).

### `internal/devices/registry_test.go` — `TestRegistry_UpdatePushRegistration` (+80 LOC)

Table-driven, four cases: hit (returns `true`, target row mutated, sibling row untouched), unknown hash (returns `false`, both rows unchanged), idempotent overwrite (calling with the same values twice still returns `true` both times), name-only update (`Platform` and `PushToken` unchanged, `Name` flipped). Sits alongside the existing `TestRegistry_FindByTokenHash` in the same file.

## Security discipline

- **Push token is opaque infrastructure data** (FCM/APNs registration id), not a secret on par with the device token. Logged at INFO when a write happens (operationally useful for "did the phone actually re-register?"); not logged on the dedupe path (no useful signal). The phone-side **device token** (used at auth) is NEVER read or logged by this handler — it isn't even in scope; the handler operates on a post-auth `Device` snapshot.
- **The Device snapshot's `Name` IS logged on every path** (write, dedupe, gone-mid-conn, save-failure). This is the inverse of #249's reject-path discipline: there, name-logging on reject would enable enumeration of paired-device names from binary logs by an attacker brute-forcing tokens. Here, the only path that reaches the handler is post-auth — the caller has already cleared the auth gate, so there is nothing to enumerate. The **unauth** branch (`device == nil`) by definition has no name to log; it logs `code=auth.invalid_token` only.
- `Handle` synthesizes no error envelope for `ErrMalformedFrame` and so carries no parser detail that could leak inputs; the dispatcher's `protocol.malformed` path owns that response.

## Concurrency model

No goroutines spawned. No channels owned. The handler is a synchronous call. Shared state it touches:

- `reg.UpdatePushRegistration` — single critical section under `reg.mu` (linear scan + three-field assignment).
- `reg.Save` — snapshot under `reg.mu`, release the lock, then file I/O (existing #209 primitive, unchanged).

Two concurrent `Handle` calls for the same conn (a misbehaving phone re-sending) would interleave at the registry mutex. Each performs an independent `UpdatePushRegistration` + `Save`; last writer wins in-memory (whichever lock acquisition was later) and on disk (Save's atomic rename is the commit point). The dispatcher serialises per-conn frame processing in practice (single inbound frame loop per conn), so this case is largely theoretical.

## Out of scope (deferred)

- **Dispatcher seam.** The relay-conn dispatcher (per-conn id allocation, Device cache, type-dispatch table, the actual call to `Handle`) is a separate, larger future ticket. `Handle`'s `nextID` parameter and `*devices.Device` parameter are the seams.
- **`server.binary_busy` retry-after advisory.** `RetryAfterS` is `nil`. If we later observe phones retrying in tight loops on save failures, a sibling ticket can add a Config-driven default; no evidence of this failure yet, deferred per "Evidence-Based Fix Selection".
- **Distinguishing revoked from never-paired** at the gone-mid-conn branch. Same v1 equivalence as #249's auth path — `Validate` is two-state.
- **APNs/FCM dispatch.** Emphatically out of scope. The push-delivery trigger ("binary detects offline phone → wakes via stored token") is its own ticket once the message-dispatch chain exists.
