# Ticket #379 — Remove `e2e-realclaude-nightly` workflow

Symmetric CI-side removal of the workflow introduced hours earlier in #362. The real-`claude`-binary suite no longer runs on a 04:00 UTC cron on GitHub Actions; it now runs locally during the code-review phase of every dispatched ticket via the pipeline (companion ticket: code-review agent `CLAUDE.md` update, tracked separately). Pure deletion + documentation correction — zero Go code touched.

## Implementation

- **`.github/workflows/e2e-realclaude-nightly.yml`** — deleted (`git rm`).
- **`docs/knowledge/features/e2e-realclaude.md`** — the `## CI cadence: nightly workflow` section was rewritten as `## CI cadence: code-review phase, no nightly workflow`. New section names the code-review agent's `CLAUDE.md` as the invocation contract surface, lists the CI-side rationale (Max-plan tokens free vs. paid `ANTHROPIC_API_KEY` secret; per-run $0.10–$0.50 buys nothing local doesn't already cover; failure surface synced to dispatch cadence beats 04:00 UTC; one fewer CI file in lockstep with `self-check-daily.yml`), and pins that `make e2e-realclaude` is unchanged. `## Related` section's #362 reference was rephrased as "the now-removed nightly workflow" with a #379 pointer.
- **`docs/knowledge/codebase/362.md`** — end-of-life banner inserted immediately after the H1, before the body. Body is preserved as historical record of how the workflow was shaped; banner warns readers not to infer current behaviour from the notes below it.
- **`docs/knowledge/INDEX.md`** — the `e2e-realclaude.md` row's `**CI cadence:**` clause was stripped of all #362 workflow internals (cron string, action versions, Go pin, env wiring, badge contract, etc.) and replaced with a short statement: no GitHub Actions workflow, runs in code-review phase, with a one-paragraph rationale and a pointer to #379 for the removal.

## Patterns established

- **Same-day reversal IS allowed, and the codebase note is the right surface for the EOL pointer.** The original spec (`docs/specs/architecture/362-e2e-realclaude-nightly-workflow.md`) is a frozen build artifact — editing it would rewrite history. The codebase note is evergreen: it's the "current view" of what #362 produced, and a banner at the top reframes the body as historical without losing it. Generalisable: when a future ticket reverses a recent decision, point the EOL banner at the codebase note, not the spec.
- **Documentation phase owns the INDEX.md revision, even when the substantive content change happens in the developer phase.** The architect's spec explicitly carved `INDEX.md` out of the developer's scope (per `agents/architect/CLAUDE.md` § "Never Update" and `agents/developer/CLAUDE.md`); the developer flagged the stale row in the PR description and this phase rewrote it. The single-writer rule combined with `serial: true` is what keeps `INDEX.md` conflict-free — the developer touching it would have broken that invariant for stale-branch concurrency.
- **Grep-cleanliness ACs should explicitly allow historical references.** The AC for this ticket permitted `e2e-realclaude-nightly` matches in `codebase/362.md` (the EOL note) and implicitly in the two spec files (`362-…`, `379-…`) which are immutable build artifacts. Generalisable: any "no matches except…" AC for a removal should pre-name the historical surfaces; otherwise code-review chases ghosts.

## Lessons learned

- **A "remove what we just added" ticket is not pure subtraction in the docs layer.** The feature doc, the codebase note, and the INDEX row each need a different treatment: feature doc gets a wholesale section rewrite (evergreen, current state only), codebase note gets a top-banner without body edits (historical record preserved), INDEX row gets a re-summary (single hot line, owned by this phase). Three different surfaces, three different edit shapes — naming them upfront in the spec prevented under-editing.
- **The "free Max tokens vs. paid API secret" trade-off is worth re-stating in the doc, not just the ticket.** The rationale for moving real-`claude` coverage off GitHub Actions is the load-bearing why for the entire arrangement; if a future contributor wonders why there's no nightly workflow, the feature doc has the answer without needing to dig up #379. Generalisable: when removing a thing, the doc that describes the surviving alternative should also explain why the removed thing isn't there.

## Out of scope (deferred)

- **Code-review agent `CLAUDE.md` update to invoke `make e2e-realclaude` on every PR.** Named in the issue body as a separate companion ticket. Until that lands, the suite has neither nightly CI nor automated per-PR coverage — operator-acceptable for a few days given the existing local `make e2e-realclaude` entry point.
