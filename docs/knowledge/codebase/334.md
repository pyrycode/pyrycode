# Ticket #334 — `agent-run/budget`: pyry-side `--max-turns` enforcement + SIGTERM-then-SIGKILL teardown

New sibling sub-package `internal/agentrun/budget`. `Counter` counts assistant JSONL entries via a `tail.Config`-shaped callback pair, fires an injected `Terminate` at the budget, then escalates to an injected `Kill` after a 5s grace period. Reports the terminal outcome via `Reason()` — `ReasonMaxTurns` vs `ReasonCompletion`. Leaf unit only; no call-site changes in this PR — the `Drive` integration is downstream once the session-ID discovery mechanism lands. ~190 production LOC, 12 tests. See [features/budget-package.md](../features/budget-package.md) for the full surface.

## Implementation

- `internal/agentrun/budget/budget.go` — one production file. Package doc-comment explains why a leaf unit exists ahead of the integration (claude-side: `claude -p` self-enforces `--max-turns`; interactive `claude` does not, so pyry has to). Imports: `errors`, `log/slog`, `sync`, `time`, and `internal/agentrun/jsonl` (for the `Event` shape only — no fs or process deps).
- **`OnEvent` filters on `Kind == "assistant"` defensively.** Post-#353 the upstream `tail.Watcher.OnEvent` callback fires for every well-formed JSONL line, not just assistant entries; the Counter cannot trust upstream filtering. Non-assistant kinds (`user`, `tool_use`, `tool_result`, `system`, `attachment`, `""`) return immediately without incrementing.
- **`OnEvent` increments-then-branches under the lock.** Branches in order: `EndOfTurn=true` returns (lets `OnEndOfTurn` classify completion); `fired=true` returns (idempotent — only one SIGTERM per Counter); `count < MaxTurns` returns; else set `fired=true`, set `reason=ReasonMaxTurns`, arm `time.AfterFunc(GracePeriod, killAfterGrace)`, unlock, then invoke `cfg.Terminate()` (logging-on-error). The Terminate callback runs unlocked so a caller that holds the mutex indirectly cannot deadlock.
- **`OnEndOfTurn` is first-terminal-wins.** Sets `reason = ReasonCompletion` only if `reason == ""`. If `ReasonMaxTurns` already fired on a prior `OnEvent`, completion does not overwrite it. Pinned by `TestOnEndOfTurn_DoesNotOverwriteMaxTurns`.
- **Budget-boundary natural completion classifies as `ReasonCompletion`.** When the `MaxTurns`-th event itself carries `EndOfTurn=true`, the `OnEvent` "return on EndOfTurn" branch fires before the budget check — Terminate does NOT fire, and the subsequent `OnEndOfTurn` sets `ReasonCompletion`. Pinned by `TestOnEvent_BudgetBoundaryEndOfTurnIsCompletion`. The spec called out this as an open question; default is "completion is a stronger signal than turn count", flip is one line if the future stream-json emitter needs the inverse semantic.
- **`Stop()` cancels the pending SIGKILL.** Captures `killTimer` under the lock, nils it, releases the lock, then calls `t.Stop()`. Idempotent. `killAfterGrace` re-checks `killTimer == nil` under the lock so a `Stop` that races a firing timer leaves the system consistent (Kill may or may not have fired; both outcomes are safe).
- **`defaultGracePeriod = 5 * time.Second` is copied, not imported.** Mirrors `supervisor.spawnWaitDelay` per the spec ("the units are independent and we should not couple the budget package to the supervisor package").
- **`Terminate` / `Kill` are callbacks, not `*exec.Cmd` ownership.** Production wires them to `cmd.Process.Signal(syscall.SIGTERM)` / `syscall.SIGKILL` in the integration ticket; tests inject `signalRecorder` stubs. The Counter does not import `os/exec` or `syscall` — keeps unit tests fast (no real processes) and keeps process lifecycle owned by the supervisor / driver layers.
- `internal/agentrun/budget/budget_test.go` — 12 tests, all `t.Parallel`. `signalRecorder` (mutex + counts + first-call timestamps) is the workhorse stub; `assistantEvent(endOfTurn)` and `mustNew(t, cfg)` are the one-liner constructors. Grace-period tests compress to 50-80ms so the suite stays sub-second. `TestKillError_IsLogged` uses a `syncWriter`-wrapped `strings.Builder` because slog handlers may write concurrently from `time.AfterFunc` and the test goroutine.

## Patterns established

- **Leaf-unit-then-integration split for non-trivial wiring.** When integration needs a precondition this ticket can't satisfy (here: session-ID discovery), land the testable leaf with the eventual integration's callback shape, defer wiring. The leaf merges fast, integration ticket reads against a known surface. Sibling pattern to #347/#348/#349/#332 — each landed independently with the integration deferred to the next ticket.
- **Callback-shaped public API + recording-stub tests.** When a unit's job is to call into the OS (signals, syscalls, exec), inject the OS call as a `func() error` field. Lets table-driven tests pin "fires exactly at X, not at X-1, not twice" without spawning real processes. Same shape as the existing `tail.Config{OnEvent, OnEndOfTurn}` callbacks (which this package wires into).
- **First-terminal-event-wins classification.** When multiple inputs may set a terminal outcome (`reason`), set it once and never overwrite. The "if reason == \"\" then set" check in `OnEndOfTurn` and the `fired` guard in `OnEvent` together implement this. Avoids state-machine confusion when the budget-th event coincides with natural completion.
- **Copy-don't-import for cross-package constants where coupling would be a tax.** `defaultGracePeriod = 5 * time.Second` matches `supervisor.spawnWaitDelay` numerically but is duplicated, with a one-line comment explaining why. Lets the budget package compile without `supervisor` in its import graph; if the supervisor's grace ever changes, the budget package decides for itself whether to follow. Same shape as how `internal/agentrun` declines to import `internal/supervisor` for the atomic-write recipe (it copies it from `internal/devices/registry.go`).
- **External callbacks invoked unlocked.** Acquire the mutex, snapshot the state needed for the callback, release the mutex, then invoke. Avoids deadlocking a caller that holds the mutex indirectly via a recursive callback chain. Both `OnEvent`'s Terminate path and `killAfterGrace`'s Kill path follow this shape.

## Lessons learned

- **`time.AfterFunc` callbacks need defensive re-checks against `Stop`.** `time.Timer.Stop()` returns `false` if the timer has already fired and its callback has been scheduled (but may not have run yet). The `killAfterGrace` body therefore re-checks `killTimer == nil` under the lock before doing anything destructive. Without this guard, a `Stop()` that races a firing timer can let Kill fire after the caller believed it was cancelled. The repo doesn't have a generic `safeTimer` helper because there are only two timer call sites total (this one and `internal/sessions/rotation/watcher.go`), and inlining is clearer than abstracting two uses.
- **slog handlers serialise their own output, but a `strings.Builder` doesn't.** `TestKillError_IsLogged` first crashed with `concurrent map write`-style panics because `slog.NewTextHandler(buf, …)` calls `buf.Write` from whichever goroutine fires the log record — here, both `time.AfterFunc`'s goroutine (when Kill fails) and the test goroutine (during teardown via `defer c.Stop()`). The fix is the `syncWriter` wrapper — a mutex around `Write`. Any test that asserts on slog output collected concurrently needs the same wrapper. (`io.Discard` from `discardLogger()` is fine because Discard is documented as safe for concurrent use; only buffered sinks need wrapping.)
- **Defensive filter on `OnEvent` was load-bearing once #353 landed.** The original architect spec said "the watcher fires only on assistant entries", which was true at #349's merge but became false at #353 (which now surfaces every `Kind` to the callback so a future stream-json emitter can re-emit byte-equivalent output). If the Counter had trusted the upstream contract literally, every `user` / `tool_use` / `tool_result` line would have counted against the budget. The `Kind != "assistant"` early-return at the top of `OnEvent` survived the upstream change without modification — the spec's "defensive even though…" comment was right to call this out.
- **The 5s grace literal is a soft contract with two owners.** `supervisor.spawnWaitDelay` and `budget.defaultGracePeriod` are independent literals that happen to be equal today. If one changes without the other, the integration (when it lands) may issue SIGTERM via the Counter, arm both timers, and have one fire before the other. That's harmless (both signals are sent eventually) but worth noting if anyone ever tunes the budget grace separately from the supervisor grace.

## Out of scope (deferred to siblings / future)

- **Drive integration.** Wiring the Counter to a live `tail.Watcher` + `cmd.Process.Signal` belongs to a downstream ticket. That ticket also has to solve session-ID discovery (claude's session UUID isn't known at spawn time — the spec calls this out as the reason for the leaf-vs-integration split).
- **`exit_reason` on-the-wire trailer.** Surfacing `Reason()` over stream-json is the future emitter ticket's job (split from #335). The Counter's `Reason()` is the source of truth; the emitter decides the wire shape.
- **End-to-end signal verification.** Real claude + real signals → integration ticket's tests, not unit tests.
- **Two-deadline clock injection.** Tests use `time.Sleep` with millisecond `GracePeriod` values today. If flakiness is observed in CI, a Clock interface follow-up is cheap; per the project's evidence-based-fix-selection principle, do not pre-build it.
- **`exec.Cmd.WaitDelay` vs custom timer reconciliation.** The integration ticket may choose to wire `Terminate` to direct `cmd.Process.Signal(SIGTERM)` and leave `cmd.Cancel` unset, OR keep the cancel mechanism and treat the Counter's Kill as a redundant safety. That decision belongs to integration, not here.
