# Ticket #534 — release-tag gate: goreleaser waits on per-agent self-check passing

`release.yml` previously ran goreleaser on every `v*` tag push independent of the per-agent self-check ([#375](https://github.com/pyrycode/pyrycode/issues/375)) state. During the 2026-05-15 → 2026-05-23 unread-badge window ([#526](526.md) regression on main; safety net red the workflow's entire 9-day life — see [#533](533.md)), a `v0.14.0` tag would have published binaries with the per-agent security boundary broken. [#533](533.md) closed the visibility gap for cron runs (active Discord push); this ticket closes the publishing gap for tag pushes (active fail-closed gate). Split from #532.

## Implementation

- **Reusable-workflow shape over duplicated steps (+26 LOC net across 2 files; no Go).** Converted `self-check-daily.yml` into a dual-purpose workflow by adding `workflow_call: {}` to the `on:` block alongside the existing `schedule:` and `workflow_dispatch:`. `release.yml` invokes it as a new `self-check:` job (`uses: ./.github/workflows/self-check-daily.yml` + `secrets: inherit`); the existing `goreleaser` job declares `needs: self-check`. Future edits to the self-check body (npm package version, `timeout-minutes: 5`, new env vars, new steps) land in one file and propagate to both call sites automatically — the duplicate-the-steps alternative (a sibling `self-check` job inside `release.yml`) is one missed sync away from the daily cron drifting from the release gate, exactly the failure mode the AC2 "defined once" wording is structured to prevent.

- **`secrets: inherit` over an explicit `workflow_call.secrets:` enumeration.** Reusable workflows cannot read repo secrets directly — they must come via the caller. `secrets: inherit` passes the entire repo-secrets bundle through; the called workflow's existing job bodies pick up `ANTHROPIC_API_KEY` (self-check) and `DISCORD_WEBHOOK_URL` (notify-failure) by the same `${{ secrets.* }}` references that already worked on the cron trigger. The explicit-list alternative (declare both under `workflow_call.secrets:` with `required: true`/`false`, have the caller pass each by name) was rejected: it adds a second source of truth for "which secrets does self-check need" that already exists implicitly in the job bodies, and the caller and called workflow live in the same repo so no cross-repo trust-boundary argument applies. `secrets: inherit` is the documented same-repo idiom.

- **Fail-closed gate via `needs:` without an `if:` override.** GitHub Actions treats a `needs:` of a `workflow_call` job as "the entire called workflow concluded `success`" — so `goreleaser` is `skipped` (not `failed`) on every non-success conclusion of the called workflow: `self-check` failed, `self-check` timed out (`timeout-minutes: 5` exhausted), reusable-workflow invocation itself errored out (malformed `uses:`, missing file). The asymmetry is intentional: `skipped` keeps the tag in the repo but produces no GitHub Release, no binaries, no homebrew formula push. No `if: always()` / `if: failure()` is set on `goreleaser`, so there is no design path where it runs without `self-check` having concluded `success`. The fail-closed property holds across "self-check ran and failed" and "self-check couldn't even start" alike.

- **`notify-failure` (the Discord alert from #533) travels for free.** A release-time self-check failure invokes the same `notify-failure` job that the daily cron uses — the operator gets the same Discord ping (in addition to the goreleaser job being skipped and the tag's overall workflow going red). The reusable-workflow design is what makes this automatic; the alternative shape would have required wiring a parallel webhook POST inside `release.yml` to preserve symmetry, doubling the surface area of the alerting plumbing.

- **Header comments on both files document the dual trigger surface (AC3).** `release.yml` gets a leading block explaining that `v*` pushes invoke the reusable workflow at `self-check-daily.yml` and that goreleaser fails closed on regression, including the operator recovery dance (`git tag -d vX.Y.Z && git push origin :refs/tags/vX.Y.Z`). `self-check-daily.yml` gets a new "Triggers" paragraph listing all three call sites (cron + dispatch + workflow_call from release.yml) and noting that a failure surfaces via Discord on every trigger AND blocks goreleaser when invoked from release.yml. The existing "Belt-and-suspenders safety net" + "Alerting contract (#533)" paragraphs stay intact — additive only.

- **Operator recovery contract documented in `release.yml` header and PR body.** On a self-check failure, the `v*` tag stays in the repo (GitHub Actions does not unwind the tag-push that triggered the workflow). Operator dance: delete the tag, fix the regression on main, retag from the new main HEAD. No new operator burden vs. the pre-#534 status quo — a goreleaser-only workflow that failed at goreleaser would have left the same orphaned tag — but worth surfacing once at the header because the failure mode is now more likely (any per-agent regression on main blocks every tag push, where before only goreleaser-side issues did).

- **Out of scope, preserved.** No change to the `self-check` job body or the `notify-failure` job body (the ticket's "do not alter the existing self-check body" constraint, mirrored from #533). No change to `.goreleaser.yaml`. No `concurrency:` block on the called workflow (deferred — see Patterns established). No explicit `permissions: contents: read` on `self-check-daily.yml` workflow level (defense-in-depth deferred — default `GITHUB_TOKEN` permissions are sufficient for the existing job bodies and `contents: write` on the caller is not implicitly granted to the called workflow's jobs). No Go code touched.

- **No automated tests.** Workflow YAML; the test harness is the workflow itself. Developer-side verification: `actionlint` clean on both files; `go vet ./...` + `go build ./cmd/pyry` clean (defensive — no Go code touched). End-to-end "the gate actually fires" verification deferred to operator at merge time — see Lessons learned.

## Patterns established

- **`workflow_call: {}` + `secrets: inherit` is the minimal idiomatic shape for "this workflow is also reusable from a sibling caller in the same repo, with the same secret needs as the existing trigger".** No `inputs:` block needed when behaviour does not vary by caller. No `workflow_call.secrets:` enumeration needed when caller and called are same-repo — `secrets: inherit` is the documented mechanism and avoids duplicating a list that already exists implicitly in the called workflow's job bodies. Reach for the explicit-secrets shape only when the caller needs to remap secret names or when the called workflow lives in a different repo where the trust boundary matters. **Generalises: when factoring a CI workflow body for reuse across multiple call sites in the same repo, prefer the minimal `workflow_call: {}` shape + `secrets: inherit` on the caller — adding inputs/secrets blocks is YAGNI until the caller actually needs to override something.**

- **`needs: <reusable-workflow-job>` without an `if:` override is the fail-closed-by-default shape for security gates.** GitHub Actions computes a called workflow's overall conclusion from all its jobs; a needed job's conclusion must be `success` (not `failure`, not `skipped`, not `cancelled`) for the dependent to run. The dependent is `skipped` (not `failed`) on every non-success terminal state — which is the right contract for a publishing gate: the tag is recorded but no artifacts are produced, and the overall workflow run is red so the operator sees the gate fired. Step- or job-level `if: always()` / `if: failure()` overrides on the dependent would silently break the fail-closed property; their absence here is load-bearing, not accidental. **Generalises: when wiring a security or publishing gate via `needs:`, do not add any `if:` modifier on the dependent — the default "conclude `success` or dependent is skipped" semantics is exactly the fail-closed contract you want, and an `if:` override is a foot-gun that can rescue the dependent past a failed gate.**

- **Same-repo reusable-workflow `uses:` resolves at the caller's SHA — no `@ref` pinning needed, and no TOCTOU on the workflow body.** `uses: ./.github/workflows/self-check-daily.yml` (relative path, no `@ref`) is the documented same-repo form; GitHub resolves the called workflow at the same SHA as the caller (i.e. the `v*` tag's commit). An attacker who could push a malicious tag could also rewrite the self-check body in the same push — but they would still be subject to the gate evaluating the body they pushed, which would either pass (no regression) or fail (gate fires). No version-skew or "the gate ran the wrong workflow body" attack is possible. **Generalises: for same-repo `workflow_call` invocations, use the relative-path form without `@ref` — pinning to a SHA would defeat the "called workflow stays in sync with the caller" property, and pinning to a branch would introduce a real TOCTOU window. The unpinned same-SHA resolution is the secure default.**

## Lessons learned

- **A dispatcher-driven developer agent cannot safely complete `v*`-tag-push verification ACs against the upstream repo when upstream has publish-side secrets set.** AC4 asked for an end-to-end test — push `v0.0.0-rc.test`, observe `skipped` on the failure path and `ran` on the success path, delete the tag. The developer correctly deferred this to the operator at merge time: the dispatcher runs non-interactively and cannot safely push `v*` tags to upstream, because upstream `pyrycode/pyrycode` has `HOMEBREW_TAP_TOKEN` set — on the passing-self-check path goreleaser would publish a real GitHub Release at `v0.0.0-rc.test` and push to `homebrew-tap`, undesired side effects without operator approval. No developer-side fork was configured for the alternative no-side-effects path. Code review flagged this as `[SHOULD FIX]` on the PR body (verification was not performed as AC4 required) but PASSed the diff itself; the human reviewer must complete the throwaway-tag walkthrough before merging to avoid shipping the gate untested against a real tag-push trigger. **Generalises: when a spec gates on an AC that requires pushing to a publish-capable upstream trigger (`v*` tags, production deploy webhooks, billing-side webhooks), either (a) call for the verification to be operator-performed at merge time in the AC text itself, or (b) provide a no-side-effects equivalent — e.g. invoke the reusable workflow via `workflow_dispatch` against a non-`v*` ref, push to a fork, or run the gate against a synthetic trigger that bypasses the publish-side secret. The dispatcher's headless contract makes "just do the manual test" infeasible when upstream is publish-capable.**

- **Code review caught a stale-path reference in the PR body (`docs/specs/architecture/534-release-tag-gate-self-check.md` vs the actual `534-release-self-check-gate.md`).** `[NIT]`-classified — the actual spec exists at the correct path, the PR-body link is just broken. The mismatch arose from a working-title vs. final-filename drift during spec drafting; the developer chose the developer-friendly working title when writing the PR body without re-checking. **Generalises: when referencing spec paths or other artifact paths in PR bodies, copy the path from the filesystem (or `git status`/`git log` output), not from the ticket/spec working title — these can drift. Cheap to fix at PR-body-edit time, expensive to chase down a year later when the spec gets renamed again.**

- **The "duplicate the steps" alternative was tempting for the wrong reason and the spec correctly rejected it.** AC2 explicitly listed both shapes as acceptable ("refactor into a reusable `workflow_call` OR add a sibling `self-check` job to release.yml"). The duplicate-the-steps shape would have been simpler in line count (zero `secrets: inherit` ceremony, no cross-workflow indirection in the Actions UI) and the AC permitted it. The architect's spec rejected it anyway on the "single source of truth, structurally enforced" axis: any future edit to the self-check body — npm package version, timeout, env vars, new steps — would have to be made in two places, and the daily cron drifting from the release gate is exactly the failure mode the AC's "defined once" wording is structured to prevent. **Generalises: when a spec offers two acceptable shapes and one is locally simpler (fewer lines, less indirection) but the other is structurally simpler (one source of truth, drift impossible by construction), the structural shape is almost always correct on a CI workflow — workflow YAML is rarely edited and lives forever, so the cost of the once-per-feature drift is paid by every future contributor and the cost of the structural shape is paid once.**

## Out of scope (follow-ups)

- **Operator action required at merge.** AC4 manual verification (throwaway `v0.0.0-rc.test` push against a failing-self-check HEAD then a passing-self-check HEAD, observe `skipped` vs `ran` on `goreleaser`, delete the tag) was deferred from the dispatcher run to the operator at merge time — see Lessons learned. Code review flagged as `[SHOULD FIX]` on the PR body.
- **`concurrency:` group on the called workflow.** Two `v*` tag pushes within the ~3-5 minute self-check window would today queue two parallel runs of the reusable workflow. A `concurrency: { group: self-check, cancel-in-progress: false }` block would serialize them. Deferred — no operator pain observed; the cost is one extra YAML block hiding a potential surprise (cancelled cron run if a tag push arrives mid-cron). Revisit only if observed.
- **Explicit `permissions: contents: read` at workflow level on `self-check-daily.yml`.** Defense-in-depth. Default `GITHUB_TOKEN` permissions are already sufficient for the existing job bodies, and the reusable-workflow invocation does not implicitly grant the caller's `contents: write` to the called workflow's jobs. Deferred — outside AC.
- **Explicit `name: Per-agent self-check` on the `self-check:` job in `release.yml`.** A bare `uses:` job displays the reusable workflow's own job names in the Actions UI ("self-check / self-check", "self-check / notify-failure"). Acceptable as-is; add the explicit name only if the rolled-up label reads poorly in practice.
