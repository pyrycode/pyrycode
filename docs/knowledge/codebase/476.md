# Ticket #476 — `internal/agentrun/settings/` subpackage: per-spawn deny-default permissions JSON

Resurrects the deny-default settings writer that [#392](https://github.com/pyrycode/pyrycode/issues/392) deleted when stream-json subprocess mode replaced PTY drive (stream-json used `--dangerously-skip-permissions`; no settings file). The 2026-05-19 pivot back to PTY drive (#329 tracking; [`ptyrunner`](../features/ptyrunner-package.md) in #471/#472; cutover in #470) re-introduces the helper because Phase A (2026-05-14) found `claude --allowedTools <list>` is **additive** in interactive mode — tools omitted from the flag still run when the model asks for them. Only a per-spawn `--settings <path>` JSON of shape `{"permissions":{"allow":[...],"defaultMode":"deny"}}` replicates the `-p` enforcement contract.

The new helper is a **strict subset** of the [#339](https://github.com/pyrycode/pyrycode/issues/339) original (see [codebase/339.md](339.md) — that helper's #392 deletion is what this ticket reverses). Sub-issue of #329; blocks #470.

## Implementation

- **Subpackage, not sibling file.** Lives at `internal/agentrun/settings/settings.go` (86 LOC) rather than `internal/agentrun/settings.go` as #339 did. Mirrors the [`internal/agentrun/trust/`](../features/agentrun-trust-subpackage.md) layout from #475 — sibling subpackages under `internal/agentrun/` for the spawn / trust / settings primitives, parent package retains only workdir helpers (`ResolveWorkdir`, `EncodeProjectDir`). New feature doc at [`docs/knowledge/features/agentrun-settings-subpackage.md`](../features/agentrun-settings-subpackage.md).

- **API delta from #339.** Signature changed from `WriteSettings(workdir, allowed []string) (string, error)` to `WriteSettings(allowedTools []string) (string, error)`. The caller (#470's `runAgentRun`) does not need to plumb `--workdir` here; the file lives in `os.TempDir()` and the caller owns `defer os.Remove(path)`. Returning the path is the only surface — ptyrunner's `Config.SettingsPath` consumes it verbatim.

- **Tempfile in `os.TempDir()` with random suffix.** `os.CreateTemp("", "pyry-agent-run-settings-*.json")` — empty `dir` argument resolves to `os.TempDir()` per stdlib contract; the `*` is replaced with OS-generated random bytes (`O_CREAT|O_EXCL` under the hood). Concurrent invocations write to distinct paths by construction. Replaces #339's `<workdir>/.pyry-agent-run-settings.json` fixed-name + atomic-rename pair.

- **Empty input is an error.** `len(allowedTools) == 0` returns `errors.New("agentrun/settings: allowedTools required")` immediately, before any file is created. Defence-in-depth — #470's CLI parse boundary already validates non-emptiness. The "nil-to-empty normalisation at the entry boundary" pattern from #339 (one-liner converting `nil` to `[]string{}` so the JSON shape was `"allow":[]` not `"allow":null`) is no longer applicable because both `nil` and `[]string{}` hit the empty-input early return.

- **Atomic-rename DROPPED — random suffix is the per-spawn uniqueness primitive.** #339 mirrored `internal/devices/registry.go:Save` (`os.CreateTemp` → `Chmod` → encode → `Sync` → `Close` → `Rename`) because it owned the path-overwrite semantics. The slim helper has no pre-existing file to overwrite, no second reader during writes, and no use case where a partial file would be observable to anything outside the helper. `f.Sync()` was dropped too — the consuming `claude --settings` invocation happens within the same operator's process tree after the helper returns; a crash between write and read also loses the claude spawn that would have read the file.

- **`0o600` chmod DROPPED.** `os.CreateTemp`'s Unix default is already `0o600`. The file lives in operator-controlled `$TMPDIR`; defence-in-depth is satisfied by the OS default. No `os.Chmod` call.

- **Lifecycle owner inverted.** #339's helper owned overwrite/rename semantics; the slim helper owns cleanup **only on the error path**. On Encode / Close failure: `_ = f.Close(); _ = os.Remove(tmpName)`. Caller (`runAgentRun`, #470) owns success-path cleanup via `defer os.Remove(path)`. The contract is pinned in the doc-comment and the empty-input leak test: callers get `(path, nil)` on success and `("", err)` on failure — never a leaked path on the error path.

- **Stdout marker line NOT owned here.** #339 owned `settings-file: <path>` on stdout for dispatcher scraping. The slim helper writes nothing to stdout; #470 owns whatever operator-visible signal the verb needs.

- **Logging discipline pinned at package doc-comment** — mirrors [`internal/agentrun/trust`](../features/agentrun-trust-subpackage.md)'s discipline note. The helper makes no `slog` calls; error wraps name a step (`create temp` / `encode` / `close`) and the path only — never the `allowedTools` slice contents or the JSON payload. Even though tool names are operational config (not credentials), the parent `agentrun` package family's no-content-logging rule applies uniformly to every subpackage so the surface stays minimal.

- **Internal payload shape unchanged from #339.** Two unexported types whose field order produces the canonical byte sequence under default Go marshalling:

  ```go
  type settingsFile struct {
      Permissions permissions `json:"permissions"`
  }
  type permissions struct {
      Allow       []string `json:"allow"`
      DefaultMode string   `json:"defaultMode"`
  }
  ```

  `DefaultMode` is plain `string` (not a typed enum — explicit non-feature per #339's spec). Compact JSON via `json.NewEncoder(f).Encode(...)` — no `SetIndent`. The trailing `\n` from `Encode` is part of the on-disk bytes and the test pins it.

- **Order and duplicates preserved.** The helper round-trips `allowedTools` verbatim — no sort, no dedup, no canonicalisation. Test row `TestWriteSettings_PreservesOrderAndDuplicates` pins both invariants with `[]string{"Bash", "Read", "Bash", "Edit"}`.

- **Tests: six cases, all parallel.** Tests target the public `WriteSettings` directly (no test seam needed — there's no `os.UserHomeDir`-style indirection like #475 had). Empty-input case uses a `filepath.Glob` pre/post diff (snapshot to a set) rather than a count, because parallel sibling tests create and clean up files in the same `os.TempDir()` during the window. Each successful-path test owns `defer os.Remove(path)`. Test cases cover the AC matrix: empty/nil input → no file + error message + empty path; single-tool exact bytes; multi-tool order + duplicate preservation; round-trip JSON parse; path location (dir + prefix + suffix); path absoluteness.

- **No e2e test in `go test`.** Schema validity against the current claude binary is #470's responsibility (issue body § Out of scope). Phase A's empirical verification on 2026-05-14 (claude 2.1.143) established the schema; re-validation is the cutover ticket's e2e smoke.

## Patterns established

- **"Slim resurrection" pattern continues** — second instance after #475 (`internal/agentrun/trust`). When a helper deleted alongside a major-mode change is resurrected because the original conditions return, the resurrection enumerates what the old version did, identifies what the new system covers, and descopes the now-redundant parts. Here: atomic-rename + `Sync` + `0o600` chmod + fixed-name overwrite + stdout-marker + nil-to-empty normalisation are all descoped because the random-suffix tempfile + same-process-tree consumer + `os.CreateTemp` defaults + empty-input rejection + caller-owned lifecycle make them unnecessary. The deltas are documented in a table both in the per-ticket file and in the feature doc so a future reviewer can see what each line of descope is buying.

- **Random tempfile location for per-spawn config that doesn't need overwrite semantics.** When a helper writes a single-use file consumed once by a sibling process in the same tree, `os.CreateTemp("", "<prefix>-*.<ext>")` + caller-owned `defer os.Remove(path)` is simpler than the atomic-write recipe. The decision rule: if there's no pre-existing file at the destination AND no second reader before the writer closes, you don't need the atomic-rename recipe. If either condition is false (e.g. `internal/devices/registry.go:Save`, `internal/agentrun/trust`), use the recipe.

- **`filepath.Glob` pre/post set-diff for no-leak assertions under parallel tests.** When tests must verify "no tempfile was created" in a shared directory (`os.TempDir()`) and `t.Parallel` sibling tests create / clean up their own files in the same directory during the window, a count-based check is racy. Snapshotting the pre-call set, asserting no new entry appears in `after \ before`, is the parallel-safe shape. Useful any time a "negative tempfile" assertion meets parallel tests.

## Lessons learned

- **Spec said `t.Cleanup`-pattern wasn't needed; this slice used explicit `defer os.Remove(path)` per test.** Architect spec's § Testing strategy specified per-test `defer os.Remove` rather than `t.Cleanup`. The reason: `defer` runs in LIFO order at function return, while `t.Cleanup` runs post-subtest at the outer `t.Run` boundary; for tests that read the file inside the same function, the `defer` shape is one fewer indirection and the lifetime is identical. Worth keeping the pattern wherever a single-test-function reads-then-cleans-up its own file.

- **Don't follow the atomic-write recipe mechanically — check whether the failure modes it defends are even present.** PROJECT-MEMORY.md mentions the canonical recipe (and the trust subpackage uses it for `~/.claude.json`); the temptation when writing a sibling subpackage is to copy the pattern by default. Spec called this out explicitly under § Files to read first: "this helper does NOT need the atomic rename." The discipline: pause before importing the recipe and enumerate the failure modes you're defending. No pre-existing file + no second reader + no orphan-reader-after-crash = no recipe needed. Stated as a positive rule above under § Patterns established; called out here as a lesson because the default of "be consistent with the sibling subpackage" would have produced the wrong answer.
