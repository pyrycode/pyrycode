# Ticket #332 — `pyry agent-run`: spawn interactive claude + PTY-drive single user-turn

Closes the verb's primary behaviour gap left by [#337](#) (flag scaffold) / [#339](339.md) (settings file) / [#342](342.md) (workspace-trust mark): after the `settings-file:` marker prints, `runAgentRun` now spawns `claude` in a PTY, drives one user-turn (defensive trust-dialog Enter → typed prompt), background-drains PTY output, and tears down on operator SIGTERM with a 5s SIGKILL grace. No JSONL parsing yet — that lands in #333.

## Implementation

- **`internal/supervisor/spawn.go` (new, ~50 LOC).** Exports `SpawnConfig{Bin, Args, WorkDir, Env, Logger}` and `SpawnPTY(ctx, cfg) (*exec.Cmd, *os.File, error)`. Thin primitive over `exec.CommandContext` + `pty.Start`: sets `cmd.Cancel = SIGTERM` (not the stdlib default Kill) and `cmd.WaitDelay = 5s`. The existing `supervisor.runOnce` is **not** refactored to consume it in this slice — duplication is one line (`pty.Start(cmd)`) and `runOnce`'s I/O bridge teardown shape doesn't fit the one-shot caller. Follow-up to consolidate is deferred to #329's tracking comment.
- **`internal/agentrun/drive.go` (new, ~130 LOC).** Exports `DriveConfig{ClaudeBin, WorkDir, Args, Env, Logger, TrustDialogDelay, PromptDelay, PromptBytes}` and `Drive(ctx, cfg) error`. Sequence:
  1. `supervisor.SpawnPTY` to launch claude.
  2. `defer ptmx.Close()`.
  3. Spawn one background goroutine running `io.Copy(io.Discard, ptmx)` — load-bearing: without it claude blocks once its kernel PTY buffer fills (~16-64 KiB; the `blast` helper writes 256 KiB to pin the regression).
  4. `sleepOrCancel(ctx, TrustDialogDelay)` then write `"\r"`. If ctx cancels first, skip and go to wait.
  5. `sleepOrCancel(ctx, PromptDelay)` then write `PromptBytes + "\r"` as a single combined write (byte-identical to the spike).
  6. `waitAndMap(ctx, cmd)` — collapses the child's exit error to nil when `ctx.Err() != nil` (operator-driven SIGTERM is success at this verb).
  Defaults: `TrustDialogDelay = 2500ms`, `PromptDelay = 3500ms`, both from the Phase A spike.
- **`cmd/pyry/agent_run.go` (modified, ~90 net LOC).** After the `settings-file:` print:
  1. `os.ReadFile(parsed.promptFile)` → wrapped as `agent-run: read prompt-file: %w`.
  2. Resolve the claude binary: `PYRY_CLAUDE_BIN` env override → fallback `"claude"` (test-only knob; production never sets it).
  3. `signal.NotifyContext(context.Background(), SIGTERM, SIGINT)` + `defer stop()`.
  4. `agentrun.Drive` with argv from the new `buildClaudeArgs` helper and optional sleep overrides from `PYRY_AGENT_RUN_TRUST_DELAY` / `PYRY_AGENT_RUN_PROMPT_DELAY` (test knobs, parsed via `parseDurationEnv`; empty/unparseable = zero = defaults).
- **`buildClaudeArgs(parsed, settingsPath) []string` (new helper).** Emits exactly:
  ```
  --settings <path> --permission-mode default --model <m>
  --append-system-prompt-file <sp> --effort <e>
  ```
  Two security invariants pinned by `TestBuildClaudeArgs_Shape`: `--permission-mode default` is **present** (the deny-default settings file requires it; `acceptEdits` would silently override) and `--allowedTools` is **absent** (the settings file is the sole allow-list authority; the flag is additive and would broaden it). `--max-turns` and `--output-format` are accepted at the pyry CLI surface but **not** propagated — claude's interactive mode doesn't honour `--max-turns`, and `stream-json` is `-p`-mode only.

## Test surface

- **`internal/supervisor/spawn_test.go`** — three `TestHelperProcess`-style tests. `exit0` happy path, `sleep_default` (Go's default-handler shape) pins SIGTERM-on-cancel ≤500ms, `ignore_sigterm` pins SIGKILL after `WaitDelay` (overridden to ~200ms post-return in the test so we don't burn 5s per run).
- **`internal/agentrun/drive_test.go`** — six tests against a `TestDriveHelperProcess` fake with four modes (`capture` / `exit1` / `blast` / `fast_exit`). Covers happy-path drive sequence, drain-prevents-block (with a 4s deadlock guard), ctx-cancel-during-trust-sleep (zero bytes captured), ctx-cancel-between-writes (single CR captured), child-exits-non-zero (`*exec.ExitError` returned), and PTY-write-error-tolerated (`fast_exit` race-loses to the prompt write; Drive logs Warn and returns the child's clean exit).
- **`cmd/pyry/agent_run_test.go`** — `TestBuildClaudeArgs_Shape` table over four argv permutations pinning order and the two security invariants. `TestRunAgentRun_DrivesFakeClaude` drives the full verb against a `TestRunHelperProcess` fake injected via `PYRY_CLAUDE_BIN`, with `PYRY_AGENT_RUN_TRUST_DELAY` / `PYRY_AGENT_RUN_PROMPT_DELAY` compressed to ~50ms.
- **`internal/agentrun/drive_e2e_test.go`** — gated by `!testing.Short()` and `exec.LookPath("claude")`; drives the real claude binary in a tmpdir workdir, polls `~/.claude/projects/<encoded>/` for a `.jsonl` (30s budget), then `cancel()`s and asserts Drive returns within 7s. Skipped under `go test -short` and on machines without `claude` on PATH.

## Patterns established

- **One-shot spawn primitive separate from the restart-loop supervisor.** `SpawnPTY` is `Config`-less by design — agent-run does not need backoff, bridge, lifecycle state, or any of the other `supervisor.Config` fields. Any future "fire one PTY child, scripted I/O, exit" verb (think: `pyry doctor` running a probe binary) should consume `SpawnPTY` directly rather than constructing a `Supervisor`. The runOnce/SpawnPTY duplication is deliberate; consolidation needs a third caller to justify lifting.
- **Operator-cancel maps to nil at the verb boundary.** `waitAndMap` checks `ctx.Err() != nil` first and returns nil even if the child died from SIGTERM. Rationale: at this verb, the operator (dispatcher) issued the cancel; the resulting signal-exit is by-design, not a failure. The next verb up (dispatcher exit code) gets to decide what "operator cancel" means at its layer; we don't pre-classify it as an error here. Mirror this for any future verb where `signal.NotifyContext` is the shutdown trigger.
- **Env-knob overrides for test-only flag wiring.** `PYRY_CLAUDE_BIN`, `PYRY_AGENT_RUN_TRUST_DELAY`, `PYRY_AGENT_RUN_PROMPT_DELAY` are all test-only — production never sets them. They live as env knobs rather than new CLI flags so the surface area visible to operators stays the same. Documented in code with "production never sets this" comments. The alternative (factor `runAgentRun` to take a `Driver` interface) was heavier and rejected here as premature.
- **Background-drain is mandatory whenever you script PTY I/O.** `io.Copy(io.Discard, ptmx)` in a goroutine is the load-bearing piece — without it the child blocks on its first ~16-64 KiB of output and the parent deadlocks on `cmd.Wait()`. The drain goroutine's `io.Copy` unblocks via the deferred `ptmx.Close()` after Wait returns. No mutex needed (PTY masters tolerate concurrent reader + writer on different sides of the line discipline).

## Lessons learned

- **PTY input line discipline rewrites CR→LF before the child reads it.** The spike's `pty_drive.py` wrote `"\r"` and assumed `claude` would see `"\r"`. The Go port's capture test (`TestDrive_HappyPathDriveSequence`) revealed that the kernel translates input CR to LF by default (`ONLCR` / `ICRNL` flags on the slave): bytes written as `"\rping\r"` arrive at the child as `"\nping\n"`. The TUI accepts both, so the production drive sequence is unaffected — but the *test assertions* had to be written against `"\nping\n"`, not the literal CR bytes. Generalisation: when porting a PTY-driving test from a high-level wrapper (Python's `pty.fork`) to Go's lower-level `creack/pty`, do not assume byte-identity between the parent's `Write` and the child's `Read` — write a capture-and-diff test first.
- **Claude's `~/.claude/projects/<encoded>/` directory encoder rewrites `_` too, not just `/` and `.`.** `docs/lessons.md`'s "Claude session storage on disk" entry mentions only `/` and `.`; the e2e test for #332 surfaced that underscores in a tmpdir-derived workdir (e.g. `/var/folders/xx/.../T/_test`) also become dashes. The e2e test handles all three (`for _, ch := range []string{"/", ".", "_"}`). If the encoded-cwd rule is ever centralised in `internal/agentrun` (planned consumer in #333), include the third character.
- **`cmd.WaitDelay` is the only knob that bounds SIGKILL latency; we hard-code 5s.** A child that ignores SIGTERM extends shutdown by exactly `WaitDelay`. `SpawnConfig` does not expose this today — if production tuning is ever needed, surface it via `SpawnConfig.WaitDelay`. Tests override it post-return on the `*exec.Cmd` so they don't burn 5s per run.
- **Pgid-kill is NOT done.** SIGTERM/SIGKILL go to the single child PID via `cmd.Process.Signal`. A claude that forks before refusing SIGTERM would leak its forks. Consistent with existing `supervisor.runOnce`; not a regression. File a follow-up if the dispatcher observes orphaned claude processes in practice (would require `cmd.SysProcAttr.Setpgid: true` + `syscall.Kill(-pid, SIGKILL)` on the grace path).
