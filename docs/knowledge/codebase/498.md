# Ticket #498 — `streamjson.New` synthesises the leading `system/init` envelope so ptyrunner's stdout matches streamrunner's wire shape

[`internal/agentrun/streamjson.New`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/streamjson/emitter.go) now writes a `{"type":"system","subtype":"init","cwd":...,"tools":[...],"model":...,"session_id":...}\n` envelope to `cfg.Writer` before returning. The six required fields are sourced from three new `Config` fields (`Cwd`, `Tools`, `Model`) plus the pre-existing `SessionID`; constants `"system"` / `"init"` fill `type` / `subtype`. [`internal/agentrun/ptyrunner.Config`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/ptyrunner/runner.go) gains a required `AllowedTools []string` field, threaded by `runAgentRunPty` ([`cmd/pyry/agent_run.go:317`](https://github.com/pyrycode/pyrycode/blob/main/cmd/pyry/agent_run.go)) from `parsed.allowedTools`. Together these restore the [[Drop-In Contract]] for the agent-run migration — `parseInitSessionID` ([`internal/e2e/realclaude/fixtures.go:342-358`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures.go)) anchors session-id discovery on `type=system, subtype=init`, and any dispatcher parser keyed on the leading envelope (test fixtures, future tooling) now works against ptyrunner's output without runner-specific branching.

## Implementation

- **Synthesis lives in `streamjson`, not `ptyrunner`.** The `streamjson` package owns the wire shape — `Close` already composes the trailing `result` envelope, so `New` symmetrically composes the leading `init` envelope. One package, both ends of the wire, mirroring the trailer's `tagged-struct + JSON-key-order` pattern. The alternative (ptyrunner writes the init line itself, then constructs the emitter) would split the wire-shape contract across two packages and force every test that uses `streamjson` outside ptyrunner to re-implement the synthesis.

- **`initLine` struct ([`emitter.go:296-309`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/streamjson/emitter.go)) is sibling to `trailer` and pins JSON key order by struct-field declaration order.** Six fields in fixture order: `Type` / `Subtype` / `Cwd` / `Tools` / `Model` / `SessionID`. Reordering breaks byte-equivalence with `testdata/captured_run.jsonl:1` even though the resulting JSON stays semantically valid — same convention `trailer` follows.

- **`Tools []string` validated non-nil, accepts empty slice.** `nil` rejected (`errors.New("streamjson: nil Tools")`); `[]string{}` accepted and marshals as `[]` (Go's `encoding/json` already does this for non-nil empty slices). `TestNew_EmptyToolsMarshalsAsEmptyArray` pins both the present-`"tools":[]` and absent-`"tools":null` halves of the contract, because a consumer anchoring on field-presence-as-array would break if a future contributor accidentally allowed nil-marshalling-as-null.

- **On init-write failure, `New` returns `(nil, err)`.** Recommended shape from the architect's spec open-question. The `Emitter` is unusable when its first write fails, so returning a non-nil emitter would force callers to write defensive code with negligible benefit. `TestNew_InitWriteFailureReturnsError` pins both the wrap (`fmt.Errorf("streamjson: emit init: %w", err)`) and the nil-emitter contract.

- **No new goroutines, no new locks.** `New` writes the init synchronously before any other goroutine can observe the Emitter — `Emit` and `Close` are unreachable until `New` returns. The Emitter's `mu` is not held during init synthesis because no other path can race. The init line is structurally the first bytes on the writer by construction.

- **`ptyrunner.Config.AllowedTools` required (non-nil; `[]string{}` accepted).** Validated in the same shape as the other required-field checks ([`runner.go:236-241`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/ptyrunner/runner.go)). Doc-comment cross-references the deny-default settings file: the field is the wire-shape mirror of the names enforced at runtime by `--settings`, kept caller-synchronised at the `runAgentRunPty` wiring site. The two surfaces (envelope `tools`, settings file allow-list) are populated from the same `parsed.allowedTools` slice at the cmd-level call, so they cannot drift.

- **`streamjson.New` call site in `ptyrunner.Run` ([`runner.go:322-329`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/ptyrunner/runner.go)) grows three field assignments.** `Cwd: cfg.WorkDir`, `Tools: cfg.AllowedTools`, `Model: cfg.Model`. No reordering of construction or defer-LIFO needed — `streamjson.New` already runs before the tail watcher and watchdog goroutines start, so the init line precedes every other event by construction.

- **`runAgentRunPty` ([`cmd/pyry/agent_run.go:317`](https://github.com/pyrycode/pyrycode/blob/main/cmd/pyry/agent_run.go)) threads `parsed.allowedTools` in one line.** `parsed.allowedTools` is already required at parse time and non-nil for every code path that reaches `runAgentRunPty`.

- **`failingWriter` test helpers grew a `failAfter int` counter.** Once `streamjson.New` performs I/O, the pre-#498 "fail every write" `failingWriter` would cause `New` itself to fail and `Emit`'s sticky-write-error contract would never be exercised. The fix is a `failAfter` counter that discards the first N writes (passing them through with `len(p), nil`) and fails the rest — `TestEmit_WriteErrorIsSticky` uses `failAfter: 1` so init succeeds and the first `Emit` hits the failure. The same pattern lives in `internal/agentrun/streamjson/emitter_test.go` and `internal/agentrun/ptyrunner/runner_test.go`; the ptyrunner-side variant gets a `sync.Mutex` because the watcher's goroutine can race the run goroutine.

- **`TestCapturedFixture_ByteEquivalence` ([`emitter_test.go:541-619`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/streamjson/emitter_test.go)) feeds `nonResult[1:]` through the reader, not `nonResult[:]`.** The fixture's first line is now produced by `New` itself, so feeding it through `Emit` would duplicate it. The test's load-bearing assertion — `out[0]` is byte-equivalent to `nonResult[0]` (the fixture's first line) — is preserved unchanged, just now satisfied by the producer-side synthesis rather than by pass-through. `Config.SessionID` / `Cwd` / `Tools` / `Model` are set to the fixture's exact values (`"28b6666c-aaaa-4aaa-baaa-aaaaaaaaaaaa"`, `"/tmp/example"`, `["Read","Bash"]`, `"claude-haiku-4-5"`) so the synthesised line matches byte-for-byte.

- **`TestNew_InitLineKeyOrderMatchesFixture` ([`emitter_test.go:677-708`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/streamjson/emitter_test.go)) byte-compares the synthesised init against `testdata/captured_run.jsonl:1` directly.** Parsing both lines into `map[string]json.RawMessage` would lose key order; the byte-compare pins the struct's tag declaration order to the fixture as source of truth. A future contributor reordering `initLine`'s fields will break this test with a diagnostic pointing at the fixture — the unit-test pin catches drift on every `go test ./internal/agentrun/streamjson/...` run, instead of only under `make e2e-realclaude` CI.

- **realclaude `checkInitModel` → `checkInit` ([`ptyrunner_byte_equivalence_test.go:349-385`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/ptyrunner_byte_equivalence_test.go)) extends the field-level assertion set.** Pre-#498 it decoded only `type` / `subtype` / `model`. Post-#498 it asserts the full load-bearing set: `Cwd == wantCwd`, `Model == wantModel`, `SessionID != ""` (the `parseInitSessionID` contract), `reflect.DeepEqual(Tools, wantTools)`. Each pipeline passes its own workdir (`workdirStream` for streamrunner; `realpath` — the post-`MarkWorkdirTrusted` realpath — for ptyrunner) and the shared `allowedTools` slice. The comment block that documented the pre-#498 "normalize away the leading envelope" rationale is trimmed; the trailer-side field-set differences are unchanged and the rationale for those stays. The `extractShapes` helper still reads only `type` + `subtype` for the cross-runner shape match.

- **`parseInitSessionID` ([`fixtures.go:342-358`](https://github.com/pyrycode/pyrycode/blob/main/internal/e2e/realclaude/fixtures.go)) is unchanged.** The producer-side fix makes the existing three-field decode (`type`, `subtype`, `session_id`) return non-empty on a successful ptyrunner run. The AC required no fixture changes; preserved.

- **`docs/knowledge/features/streamjson-package.md` and `ptyrunner-package.md` updated in-place.** New `Config` fields, the leading-envelope wire shape, and the init-envelope test list extend the streamjson feature doc; the ptyrunner feature doc gets the new `AllowedTools` required field and a forward-reference to the streamjson init synthesis.

## Patterns established

- **Producer-side wire-shape fixes over consumer-side relaxations.** When a downstream consumer breaks because a producer's wire shape diverges from a contract, the chosen fix is to make the producer emit the contract — not to loosen the consumer. The rejected alternative for #498 was loosening `parseInitSessionID` to accept whichever envelope comes first; that admits the divergence and forces every future consumer (test fixture, dispatcher tool, fixture-compare assertion) to special-case ptyrunner. Generalises: when N consumers anchor on a producer's output shape, fixing the producer is O(1); relaxing each consumer is O(N), and the relaxations rot independently.

- **JSON key order is part of the wire contract; pin it with byte-level fixture comparison, not value-only round-trips.** A test that decodes both sides into a struct or `map[string]any` and checks equality will silently accept a struct-field reorder, because both representations decode to the same logical content. The byte-compare against a captured fixture is the only assertion that catches the drift, and it does so on every unit-test run — without it, the drift would be invisible until the next `e2e_realclaude` run (network + token required). Concretely here: `TestNew_InitLineKeyOrderMatchesFixture` byte-compares against `testdata/captured_run.jsonl:1`; combined with the existing `TestCapturedFixture_ByteEquivalence` for the trailer, the captured fixture is the source of truth for both ends of the wire shape.

- **`failAfter` counter on a test `failingWriter` is the canonical retrofit when a constructor grows I/O.** When a constructor previously did no I/O but now does (e.g. an emitter that emits a leading envelope on construction), every existing test that uses a "fails every write" `failingWriter` to exercise downstream failure modes breaks at construction. The simple fix is to add a `failAfter int` field: discard the first N writes successfully (`return len(p), nil`), fail the rest. The counter makes the test's intent explicit (e.g. `failAfter: 1` = "let init succeed, fail the first Emit") and survives future I/O additions to the constructor with one field adjustment. Generalises to any test double whose contract changes when a previously-pure operation grows side effects.

- **Wire-shape sibling structs pin the contract at the type system level.** `initLine` and `trailer` in `streamjson/emitter.go` are unexported structs whose only purpose is to carry the JSON tag declarations for the two synthesised lines. Field order in the struct IS the field order on the wire. A casual contributor refactoring "tidy" struct fields will break the byte-equivalence test with a diagnostic that names the fixture — making the wire contract auditable without separate documentation that can rot. Same pattern `streamjson.trailer` established; #498 extended it to the init line.

- **Doc-comment cross-references between wire-shape fields and their runtime enforcement.** `ptyrunner.Config.AllowedTools`'s doc-comment names the deny-default settings file at `SettingsPath` as the runtime enforcement, and notes that the two surfaces are caller-synchronised at the `runAgentRunPty` wiring site. The next reader who wonders "what if these drift?" finds the answer in the field comment without having to grep across packages.

## Lessons learned

- **The PR body's `## Lessons learned` section was load-bearing for this ticket and lifted verbatim.** PR [#500](https://github.com/pyrycode/pyrycode/pull/500)'s body included two genuine lessons — the key-order byte-compare against the fixture, and the `failAfter` retrofit on `failingWriter`. Both are reflected in the "Patterns established" section above. Same shape as the convention: when the work surfaces a non-obvious surprise the spec did not predict, the PR body's optional Lessons section is where the developer flags it; the documentation agent then lifts those bullets into the per-ticket file (verbatim where clear, paraphrased where terse).

- **Spec-level open questions get a recommended shape AND a flag, not silent resolution.** The architect's spec named two open questions: (1) `New` return shape on init-write failure (recommended `(nil, err)`; not awaiting input; decision documented inline), and (2) whether `Effort` belongs in the init envelope (skip; not in fixture or AC; file follow-up if future migration needs it). Both were resolved in the spec body with explicit recommendation + rationale; the implementation followed the recommendation; code review confirmed. Generalises: an open question with a recommended shape and rationale is decided-but-flagged; the architect-developer hand-off is unambiguous without forcing a synchronous decision pass. Anti-pattern: an open question left genuinely open at spec time would block the developer mid-implementation.

- **The code-review NIT on `_ = em` was correctly a no-fix.** Reviewer flagged that `TestNew_WritesInitLineFirst`'s `_ = em` line is unnecessary; the test only inspects `buf`, so the constructor could discard the emitter inline (`if _, err := New(...); err != nil`), matching `TestNew_InitLineKeyOrderMatchesFixture`'s shape. The fix is mechanically obvious but the reviewer explicitly noted "Not worth a re-review" — the test behaviour is correct, the line is stylistic, the cost of a re-review cycle exceeds the cost of one stale `_ = em` line. Same pattern as #496's `os.Getenv("HOME") == ""` NIT: spec-acknowledged or reviewer-flagged stylistic items resolve via the natural future-edit path, without forcing a re-review.

- **Code review's codegraph callers audit confirmed `streamjson.New` has exactly one production caller.** `ptyrunner.Run` is the sole non-test caller; the other 10 are tests, all routed through `newTestEmitter` / `helperRunCfg` helpers, which were updated centrally. No missed call sites means the new Config required-field set rolls out atomically with no partial-validation hazard. Generalises: when extending a constructor's required-field surface, an explicit codegraph-callers audit before merge is the simplest proof that no caller was missed — same shape as #496's `WithWorktreeAuthenticated` 19-caller audit.

## Out of scope (follow-ups)

- **Effort in the init envelope.** Spec open-question 2 explicitly resolved as skip: the captured fixture (`testdata/captured_run.jsonl:1`) does not carry `effort`, and the AC's required-field list does not name it. If a future migration target needs `effort` for symmetry with claude's own envelope, file a follow-up; do not extend the contract here.

- **Richer init fields (`mcp_servers`, `permissionMode`, `apiKeySource`, `claude_code_version`).** streamrunner's live output may carry these but the captured fixture does not, and the load-bearing required set is the six fields. The ticket's Technical Notes flagged them as optional-if-cheap: `claude_code_version` would require a `claude --version` shell-out (cache-it-once), `permissionMode` mirrors `--permission-mode default`, `apiKeySource` is `"none"` when token-via-env. None are gating; defer until a consumer requires one.

- **`error_during_execution` / `duration_ms:55002` failures (ticket [#499](https://github.com/pyrycode/pyrycode/issues/499)).** Some realclaude tests still fail with the `error_during_execution, num_turns:0` pattern even after #496's `.claude.json` seeding and #498's init synthesis. The ticket explicitly notes these are claude-side (no JSONL events at all), not parser-side; tracked separately as #499.
