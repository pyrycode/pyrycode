# Ticket #248 — `net`: outbound dial with server-id handshake

Phase 3 Track C handshake layer. New `internal/relay` package wraps the generic `internal/transport.Client` (#247) with the binary-side wire-protocol semantics: builds the `x-pyrycode-server` / `x-pyrycode-version` / `user-agent` upgrade headers, runs the one-shot `hello` → `hello_ack` handshake on every fresh conn, classifies WS close-code `4409` as terminal (server-id already claimed), and forwards inbound `RoutingEnvelope` frames to consumers via `Frames()`. Composes the foundations from #246 / #255 / #256 / #271 (envelope + v1 payload types), #247 (transport), and #207 (server-id store). Wire-spec source-of-truth is `docs/protocol-mobile.md` § Authentication / § Connection lifecycle / § Worked example.

## Implementation

### `internal/relay/connection.go` (~300 LOC)

- **`Connect(ctx, cfg) (*Connection, error)`** — synchronous-validate, asynchronous-run. Validates `Config{ServerID, RelayURL, BinaryVersion, Logger}` (all required), `url.Parse`-rejects non-`wss` schemes as `ErrInvalidConfig` (defense-in-depth against operator `ws://` misconfig leaking the server-id header in cleartext — server-id is a public routing key per `docs/protocol-mobile.md` § Security model Threat 2, but the check is cheap and structural), builds headers, constructs the transport client with `FatalCloseCodes: []websocket.StatusCode{4409}`, kicks off the `run` goroutine, returns.
- **`Frames() <-chan protocol.RoutingEnvelope`** — channel of post-handshake inbound frames. Closes when the lifecycle terminates. Reconnects are transparent: a fresh `hello`/`hello_ack` runs first on each new conn, then frames resume — frames are NEVER buffered across the gap (consumers see a contiguous in-order stream punctuated by no-op reconnects). The unbuffered channel means a slow consumer applies back-pressure all the way to the relay's send buffer.
- **`Wait() error` / `Close() error`** — `Wait` blocks until `run` exits and returns the terminal classification (`ErrServerIDConflict` for fatal 4409, `ctx.Err()` for graceful shutdown, or a wrapped transport error for unexpected halts); `Close` is idempotent via `sync.Once`, closes the internal `closed` channel and the transport client. Both `frames` and `done` close exactly once via deferred closes in `run`.
- **`run(ctx)` goroutine** — single owner. Spins a trampoline goroutine that calls `transport.Client.Connect(ctx)` and pipes its return into a 1-buffered `transportErrCh`. Main `select` has four arms: `ctx.Done()`, `c.closed`, `transportErrCh`, and `c.client.Connected()`. The `Connected()` arm calls `handshake(ctx)`; on success runs `forwardFrames(ctx)` to completion (which returns when the underlying conn drops), then loops back; on failure logs WARN, calls `c.client.DropConn()` to recycle the conn, and loops back. The transport's reconnect-with-backoff is the recovery mechanism for both transport drops and application-layer handshake failures.
- **`handshake(ctx)`** — marshals `protocol.HelloServerPayload{Role: "server", ServerID, BinaryVersion, ProtocolVersions: []string{"v1"}}` into the Payload of `protocol.Envelope{ID: 1, Type: TypeHello, TS: time.Now().UTC(), Payload: payloadJSON}`, sends, then awaits the response under a 5-second `context.WithTimeout` (the wire-spec deadline). **Two-pass JSON decode at the trust boundary**: relay-to-binary frames are ALWAYS wrapped in `RoutingEnvelope` (`conn_id: "-"` for handshake responses per `docs/protocol-mobile.md` § Worked example), so the handshake decoder first unmarshals to `RoutingEnvelope`, then unmarshals `routing.Frame` to the inner `Envelope`, then asserts `env.Type == protocol.TypeHelloAck`. Mismatch → error → `DropConn` → reconnect.
- **`forwardFrames(ctx)`** — pump loop reading `client.Receive(ctx)`, unmarshalling to `RoutingEnvelope`, sending to `c.frames` (or returning on `ctx.Done()` / `c.closed`). Receive returning ANY error exits the loop with a Debug breadcrumb log — the three expected errors are `transport.ErrDisconnected` (conn dropped; run will re-handshake on next `Connected`), `transport.ErrClosed` (Close called), and `ctx.Err()`. Malformed inbound JSON is logged at WARN and dropped at the trust boundary; a single bad frame does not tear the conn.
- **`classifyTransportErr`** — terminal-error mapping. If the transport returned `ErrFatalClose` AND `websocket.CloseStatus(err) == 4409`, surface `ErrServerIDConflict`; otherwise pass the transport error through unchanged.
- **Sentinels** — `ErrServerIDConflict` (terminal 4409), `ErrInvalidConfig` (bad config from `Connect`). Callers branch via `errors.Is`. `ErrServerIDConflict`'s string contains no dynamic content beyond the fixed close-code reference (no token / server-id leakage via error messages).
- **`statusServerIDConflict websocket.StatusCode = 4409`** — typed package-local constant. The `websocket` import is the only one in `internal/relay`; consumers in `cmd/pyry` never need to import `coder/websocket` transitively.
- **`handshakeTimeout = 5 * time.Second`** — exposed as a package `var` (lowercase) so tests can substitute a 200ms value via `t.Cleanup`. Same idiom as `internal/transport`'s per-Client cadence fields (#247).
- **`connectWithClient(ctx, cfg, client) *Connection`** — unexported test seam that wraps a supplied `*transport.Client` (typically wired to a `httptest.NewServer` via a custom `dialFn`) and bypasses `Connect`'s `wss://` validation. Production callers always use `Connect`.

### `internal/transport/wssclient.go` (additions, ~100 LOC)

The deferred-decision note in #247's spec (`docs/specs/architecture/247-wssclient-with-auto-reconnect-backoff.md:488-495`) assigned four additions to this ticket. All are purely additive — zero-value `FatalCloseCodes` preserves #247's "reconnect on every drop" behaviour, and there are no in-repo callers of the transport package outside `internal/relay`.

- **`Connected() <-chan struct{}`** — buffer-1 channel with drop-on-full emit. Emitted in `serve` AFTER `setConn(conn)` (consumers waking on `Connected` must find `Send`/`Receive` already wired against the live conn — same precedent as `docs/lessons.md:290`). Non-blocking send: a slow observer sees the most recent connect, not every connect since boot. Single-observer (the relay's `run` goroutine) — multiple observers can miss events by design.
- **`Config.FatalCloseCodes []websocket.StatusCode`** + **`ErrFatalClose` sentinel** — close codes in this list terminate the reconnect loop with `errors.Is(err, ErrFatalClose) == true` and `websocket.CloseStatus(err)` recoverable for caller branching. Check fires in BOTH the post-`serve` path (peer-initiated close during a live conn) AND the post-`dialFn` path (peer closing during/immediately after the WS upgrade — `coder/websocket` surfaces the close error from `Dial` directly in this case rather than from a subsequent `Read`, so the same check sits on both paths so 4409 halts regardless of which path observed it). The post-`serve` path also runs an `errs[]` walk over the three pump-return errors and PREFERS one with a recognizable close status (recvPump observes the `CloseError` but sendPump's mid-write `conn.Write` and pingLoop's `conn.Ping` can return generic `use of closed network connection` errors; without preferring the close-status error, fatal-close classification would silently fall through to reconnect when sendPump or pingLoop happened to return first).
- **`ErrDisconnected` sentinel + per-conn `connDone` channel** — names from #247's spec, implementations landed here. `connDone` is a per-conn signal: pre-closed at construction (so `Receive`/`Send` before `Connect` return `ErrDisconnected` immediately rather than blocking forever — matches `ErrNotConnected`'s "you can't do I/O yet" semantics), replaced with a fresh open channel inside `serve` before pumps install, closed inside `serve`'s deferred teardown. `Receive`'s select reads the current `connDone` once under `connDoneMu` THEN selects (a serve iteration that replaces `c.connDone` between the load and the select cannot accidentally make us wake on the new channel — the captured value is the one we wanted to observe). `Send` does the same. Without `ErrDisconnected`, a `Receive` blocked when the conn drops would stay blocked until the next conn delivers a frame — but the relay won't send anything until our handshake completes, and our handshake won't run because we're wedged in the previous-conn `Receive`. Pyry would silently stop processing inbound traffic after the first transport drop.
- **`DropConn()`** — force-closes the live conn (if any) via `conn.CloseNow()` (abrupt 1006-equivalent, no close-frame round-trip). The serve loop observes the closed conn, returns to the dial loop, reconnects via backoff. Does NOT halt the dial loop. Idempotent. `CloseNow` over `Close(status, reason)` because the caller is not blocked for up to 10s waiting on a close handshake when the only purpose is to recycle the conn. The relay package uses this on application-layer handshake failures (`hello_ack` timeout, wrong type, malformed JSON) — the transport-layer conn is healthy but the application-layer protocol is stuck.

The post-`serve` "prefer close-status error" walk was discovered during testing — `TestServerIDConflict_FatalNoReconnect` flakes (race-detector) without it, because under load sendPump or pingLoop can return their generic `use of closed network connection` error before recvPump returns the actual `CloseError{Code: 4409}`. The `errs[]` walk pins the right error into `Connect`'s return regardless of pump completion order.

### Concurrency model

```
Connect(ctx) returns synchronously after config validation
    │
    └── run(ctx) goroutine [single owner]
          │
          ├── trampoline goroutine: transportErrCh <- client.Connect(ctx)
          │       │
          │       └── transport.Client owns 4 goroutines per live conn:
          │             dial loop + recvPump + sendPump + pingLoop
          │
          ├── select on:
          │     ├── ctx.Done()       → result = ctx.Err(); return
          │     ├── c.closed         → result = nil; return
          │     ├── transportErrCh   → result = classifyTransportErr; return
          │     └── client.Connected() → handshake → forwardFrames; loop
          │
          └── forwardFrames(ctx) reads client.Receive → c.frames
```

Exactly two goroutines owned by the relay package per `Connection`: the `run` lifecycle goroutine, plus the trampoline that calls `client.Connect(ctx)`. The trampoline exits when transport returns; `run` exits when any of its four select cases fires. `c.frames` and `c.done` are closed via deferred closes in `run` (each closed exactly once). `c.closed` is closed via `sync.Once` in `Close`. No double-close.

Lock order: relay package holds zero locks (only the `sync.Once`). Transport gained one new leaf-only mutex (`connDoneMu`) with no edges to existing transport mutexes (`mu`, `rngMu`).

### Failure modes

| Failure | Surface | Recovery |
|---|---|---|
| WS close `4409` (server-id conflict) | `Wait() → ErrServerIDConflict` | None. Fatal. Operator escalation. |
| WS close `1011` / `1006` / transport drop | invisible to caller; transport reconnects | `run` loops on `Connected()`, re-sends `hello`, awaits `hello_ack`. |
| `hello_ack` not received within 5s | logged WARN; `DropConn` fires; transport reconnects | Same loop. Persistent failure → backoff saturates at 30s. |
| First post-`hello` frame is not `hello_ack` | logged WARN; `DropConn` fires; transport reconnects | Same. |
| Malformed JSON on inbound frame (post-handshake) | logged WARN; frame dropped; loop continues | Single bad frame does not tear the conn. |
| `ctx` cancelled | `run` returns `ctx.Err()` via `Wait()` | Graceful shutdown. |
| `Close()` called | `run` returns nil via `Wait()` | Graceful shutdown. |
| `Send(hello)` returns `ErrNotConnected` (race: conn dropped between Connected signal and Send) | handshake returns error → `DropConn` is no-op → transport reconnects | Same loop. |
| Conn drops mid-handshake | handshake's `Receive` returns `ErrDisconnected` immediately (before the 5s expires); `DropConn` is no-op; `run` loops and catches the next `Connected` | Same loop. Tested via `TestTransportDropDuringHandshake`. |

### Logging

All structured slog with strict field discipline (security-review § Error messages):

- `relay: handshake complete` → `server_id`
- `relay: handshake failed; recycling conn` → `err`
- `relay: malformed routing envelope; dropping` → `err`
- `relay: forwardFrames exiting` (Debug) → `err`

Forbidden fields:

- never `token` (no tokens flow through this layer; token validation is per-phone-conn, future ticket)
- never `payload` (frame bodies can contain user message text in the future; even pre-encryption v1, log routing metadata only)
- never raw `frame` bytes
- never full `Headers` map (would leak `x-pyrycode-server` to every line; the `server_id` field is the operator-actionable subset)

## Lessons learned

### Three pump-return errors, only one carries the close status — prefer the one with a recognizable WS close

When a peer closes with status 4409, `coder/websocket` surfaces a `CloseError{Code: 4409}` from `recvPump`'s `conn.Read`. But the same close races against `sendPump` mid-`conn.Write` and `pingLoop` mid-`conn.Ping`, both of which return generic `use of closed network connection` errors. `serve` waits on a 3-buffered `errCh` then returns the FIRST received error — which, under `-race` load, is often NOT the recvPump's close-status error. `websocket.CloseStatus()` returns `-1` on the generic net errors, the `FatalCloseCodes` check in `Connect` falls through, the dial loop reconnects, and 4409 gets handled as a transient drop. `TestServerIDConflict_FatalNoReconnect` flakes under `-race -count=N`, passes single-run.

The fix is to drain all three pump returns and walk the list preferring an error whose `websocket.CloseStatus` is not `-1`. Captured here because it's a non-obvious symptom (intermittent flake on a deterministic test) with a non-obvious cause (Go select non-determinism crossed with a library API that puts the close-code on only one of three goroutines' return paths). Future package additions that fan-out pump-style goroutines should adopt the same "drain and prefer" pattern when downstream code branches on a property only one goroutine can observe.

### `Receive` blocked mid-conn-drop must surface a per-conn signal, not just a global closed-channel

#247's spec named `ErrDisconnected` but the implementation never wired it up — `Receive`'s select only fired on `recvCh`, caller `ctx`, or `closeCh`. A caller blocked in `Receive` when the underlying conn dropped would stay blocked until the next conn delivered a frame. For the relay's `forwardFrames` loop, the failure mode is silent-and-permanent: a reconnect arrives, but the wedged Receive prevents `forwardFrames` from returning, which prevents `run` from observing the next `Connected`, which prevents the re-handshake — and without the binary's `hello`, the relay won't send anything. Pyry silently stops processing inbound traffic after the first transport drop.

The fix is a per-conn `connDone chan struct{}` that's closed at serve teardown, replaced with a fresh open channel before the next dial. `Receive` (and `Send`) read it once under a mutex then select. The captured value is the one the call wanted to observe; a serve iteration that replaces `c.connDone` between the load and the select cannot accidentally wake the caller on the new channel. Initial state is a pre-closed channel — a `Receive`/`Send` call before `Connect` has produced any live conn returns the sentinel immediately rather than blocking forever (matches the `ErrNotConnected` discipline).

Caught during the spec's concurrency walk (the "wedged Receive after disconnect" line in security review § Concurrency) and the corresponding `TestTransportDropPostHandshake_ReHandshakes` is the pin. The lesson generalizes: when adding a sentinel error to a select-driven API, audit every existing caller in the same package for "what happens if this caller is blocked when the new condition triggers?" — silent-and-permanent wedges are the worst failure mode because no test for the new sentinel exercises the old caller.

### `DropConn` over per-conn context plumbing for application-layer recycle

When the application layer (handshake state machine here) needs to force a fresh transport conn — `hello_ack` timeout, unexpected frame type, malformed inbound JSON — the question is how to signal "the wire is fine, but the conversation on this wire is broken." Two options: (a) plumb a per-conn context through `Receive`/`Send` so the application can cancel it, or (b) expose `DropConn()` that calls `conn.CloseNow()` and lets the transport's existing dial loop handle the recycle.

The deciding factor is API surface: (a) requires `Receive(ctx, perConnCtx)` or wraps both into a single passed-down context, and `Send` needs the same; either coupling layers more than necessary or changing every existing caller. (b) is one method on the Client that's idempotent and does nothing when no conn is live. `DropConn` is a strict subset of "per-conn context" in capability — anything (a) would let you do, (b) also lets you do for the recycle case. Picked (b) for the smallest API increment achieving the same end.

The corollary lesson: when transport boundaries already have a "force a fresh primitive" path (here, the dial loop's reconnect-on-drop), application-layer escape hatches should ride that path rather than introducing a parallel cancellation surface.
