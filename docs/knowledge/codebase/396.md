# Ticket #396 — `send_message` triggers lazy respawn after idle eviction

After `--pyry-idle-timeout` SIGKILLs the bootstrap session, the next inbound `send_message` from a relay-routed phone/plugin (Discord/Telegram) used to disappear into `Supervisor.WriteUserTurn`'s `ptmx == nil` discard branch — silent ack, no respawn, no log line. Companion bug to [#395](395.md): #395 flipped the *default* idle-timeout to 0, this ticket closes the *opt-in path* so an operator who explicitly enables eviction still gets respawn-on-inbound. Observed on pyrybox 2026-05-15 as a 7.5h silent outage between `signal: killed` and a manual `systemctl --user restart`.

## Implementation

Path A from the issue body: make `send_message` an attach event that drives `Session.Activate` the same way CLI `pyry attach` does. Three production files, ~50 prod-line diff.

- **`Supervisor.WaitForPTY(ctx)` + `ptmxReadyCh`** (`internal/supervisor/supervisor.go`). New `chan struct{}` field initialised open in `New`; `setPTY(non-nil)` closes it (idempotent), `setPTY(nil)` allocates a fresh open channel (idempotent). New method captures the chan reference under `ptmxMu`, awaits unlocked. Same chan-snapshot-then-await pattern as `Session.activeCh`. Locking invariant unchanged: `ptmxMu` stays leaf-only.
- **`Session.Activate` waits for PTY readiness at the tail.** After the existing `select { case <-ch: case <-ctx.Done(): }` block, the function now returns `s.sup.WaitForPTY(ctx)` instead of `nil`. Closes the hundreds-of-ms gap between `transitionTo(stateActive)` closing `activeCh` and `runOnce → setPTY(ptmx)` actually binding the master. The doc comment now promises "WriteUserTurn/Resize will reach the live claude" after Activate returns. State-flip choreography (`activateCh`/`evictedCh`, the `#202` bootstrap warm-start invariant) is unchanged — the strengthening is one extra wait at the tail. See [ADR 023](../decisions/023-activate-waits-pty-readiness.md).
- **`runActive` logs `session.idle_eviction` at the SIGKILL moment** (`internal/sessions/session.go:346-358`). WARN-level with `session_id`, `idle_timeout`, `bootstrap` fields, emitted just before `cancelSup()`. Pairs with the existing supervisor-level `"claude exited" err="signal: killed"` line at `internal/supervisor/supervisor.go:265` — operators no longer need to correlate the generic `signal: killed` against the configured idle window. AC#3.
- **`handlers.SendMessage` runs Activate before WriteUserTurn** (`internal/relay/handlers/send_message.go`). `TurnWriter` gained `Activate(ctx context.Context) error` (`*sessions.Session` satisfies it adapter-free at `cmd/pyry/relay.go:139`). The handler runs Activate with a 30s budget — `sendMessageActivateTimeout`, package-private const matching `internal/control/server.go:676` so CLI attach and inbound `send_message` use the same window. On Activate timeout: WARN `send_message.activate_failed` and a wire reply with `protocol.CodeServerBinaryOffline` + `Retryable=true`. On `ctx.Canceled` (conn closing), propagate to the dispatcher's per-conn unwind.

### Data flow after the change

```
phone/plugin send_message → handlers.SendMessage
                              ↓
                       w.Activate(activateCtx)         ← new (30s budget)
                              ↓ (blocks 0…30s)
                       supervisor has bound PTY        ← new guarantee
                              ↓
                       w.WriteUserTurn(convID, text)
                              ↓
                       ptmx.Write(payload) succeeds
                              ↓
                       handler emits send_message.ack
```

On an already-active session: two non-blocking channel receives (`activeCh` already closed, `ptmxReadyCh` already closed). Negligible steady-state cost.

## Tests

- **`internal/supervisor/supervisor_test.go`** — `WaitForPTY` contract: already-set returns immediately; not-yet-set blocks until `setPTY(ptmx)` unblocks; `setPTY(nil)` after a prior set freshens the chan and the next `WaitForPTY` blocks again until the next non-nil `setPTY`; `ctx` cancel returns `ctx.Err()`.
- **`internal/sessions/session_test.go`** — Activate's PTY-readiness guarantee end-to-end against `/bin/sleep` as the supervised child (same harness as `TestSession_IdleEvictionDeferredWhileAttached`); `ctx` cancel during the PTY-wait portion returns `ctx.Err()` and does NOT roll back the state flip; new test asserts the `session.idle_eviction` log record fires at SIGKILL with the expected fields by attaching a `slog` recording handler.
- **`internal/relay/handlers/send_message_test.go`** — `stubTurnWriter` gained an `Activate` method that records call order against `WriteUserTurn`; happy path asserts Activate-before-WriteUserTurn and ack emission; Activate timeout asserts WriteUserTurn is NOT called, the WARN log fires, and the wire reply uses `CodeServerBinaryOffline` with `Retryable=true`.

E2E coverage is intentionally NOT added here — that's [#398](https://github.com/pyrycode/pyrycode/issues/398)'s job, and #398 is unblocked by this ticket landing.

## Patterns established

- **Activate is the named "ensure session is ready" seam.** Path A's wait belongs at the Activate seam — not internally inside `WriteUserTurn` (would have required a `ctx` parameter, breaking the established `TurnWriter` shape) and not by restructuring `transitionTo` to defer the `activeCh` close (would have entangled the `#202` bootstrap warm-start invariant). Future readiness gates that supervisors add (graceful-stop quiescence, JSONL flush, etc.) should layer onto Activate the same way `WaitForPTY` does — strengthen the Activate contract with a tail wait, leave the channel choreography alone.
- **Per-handler attach budgets converge on the CLI attach budget.** `sendMessageActivateTimeout = 30 * time.Second` is the same constant `handleAttach` has used since the lazy-respawn primitive (#40). Any future inbound carrier (e.g. an HTTP webhook handler) should reuse the 30s budget rather than picking a fresh number, so operator mental models for "how long before the binary gives up" stay uniform across attach paths.
- **Eviction call sites must emit a structured cause line.** The `session.idle_eviction` WARN at the SIGKILL moment is the operator-facing signal that #395's "Lessons learned" flagged. The cap-policy eviction path (`Session.Evict` driven by `Pool.Activate`) does not yet emit a sibling line — when that path next gets touched, mirror this pattern.

## Lessons learned

- **`WriteUserTurn`'s discard-on-nil-ptmx is fail-closed, not fail-loud.** The `ptmx == nil → return nil` branch was added to mirror `Bridge.Write`'s discard-on-unattached semantics — sane for the foreground-CLI shape it was designed for, where "no terminal attached" really means "drop the write." For relay-routed inbound it was the wrong default: there's no human watching the wire to notice the silent drop, and the dispatcher cheerfully sent `send_message.ack` regardless. The fix lifted the readiness check up to the Activate seam where it can fail loudly with a `CodeServerBinaryOffline` reply; the discard branch in `WriteUserTurn` stays as a defence-in-depth no-op for already-evicted sessions that race past Activate.
- **State-flip wakeup is not the same as "ready."** `Session.Activate` returning used to mean "the lifecycle goroutine has flipped to `stateActive` and persisted." Callers reasonably read that as "the supervisor is up and writeable." It wasn't — `runActive` schedules `runOnce` on a goroutine, which then walks `pty.Start → setPTY(ptmx)`. That ~hundreds-of-ms gap was masked on the CLI attach path because `Bridge.Attach` queues bytes via `b.in` until `Bridge.SetPTY` is called; the relay path writes to `ptmx` directly with no queue, so the race surfaced. Lesson: when documenting an "ensure X is ready" primitive, name the *observable post-condition* (PTY bound, write reaches the live process), not the internal trigger that kicks off the readiness work.

## Out of scope (deferred)

- **`pyry status` exposing "evicted" lifecycle state.** AC#1 from the issue body offered Path B's status field as part of the loud-failure trio. Path A's automatic respawn satisfies AC#1 without it, so no `status` payload change here. A future ticket can extend the status payload if operator visibility into the lifecycle state becomes valuable.
- **Metric for activate-on-`send_message` wait time.** No metrics infrastructure exists yet; the `send_message.activate_failed` WARN log on Activate timeout is enough for the current observability bar.
- **Cap-policy eviction structured log line.** `Session.Evict` driven by `Pool.Activate`'s cap policy still produces only the supervisor-level `signal: killed` line. Mirror the `session.idle_eviction` shape there next time the cap path gets touched.
- **End-to-end test for the relay-routed respawn.** Owned by [#398](https://github.com/pyrycode/pyrycode/issues/398).
