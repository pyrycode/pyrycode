# Ticket #666 — `create_conversation` handler on the v2 phone-protocol dispatch

Adds the missing `create_conversation` handler to the daemon's phone-protocol
dispatch so tapping **"New discussion"** on mobile creates a conversation end to
end. Before this, `cmd/pyry/relay.go` registered only `list_conversations` /
`register_push_token` / `send_message` on both dispatch paths; a
`create_conversation` frame hit `dispatch.Route`'s "no handler registered"
branch → `protocol.unsupported`, the mobile `createDiscussion` await received an
error instead of `conversation_created`, and the app never navigated into the
new thread (diagnosed live on 2026-06-18 against the pyrycode-mobile#421 rung-3
e2e). The wire contract already existed in `internal/protocol`
([protocol-package.md] § conversations) — this ticket is **purely the daemon
handler and its registration on both paths**. `security-sensitive` (inbound
handler on the internet-exposed v2 Noise dispatch that creates persistent server
state from an untrusted remote frame).

## Implementation

| File | Change |
|---|---|
| `internal/relay/handlers/create_conversation.go` | **New.** `CreateConversation(reg ConversationCreator, registryPath, defaultCwd string, logger *slog.Logger) dispatch.Handler` — a sibling of `ListConversations` / `SendMessage`. Decodes `protocol.CreateConversationPayload`, resolves the three `*`-nullable fields to effective values (server defaults for `null`), mints a server-side id via `conversations.NewID()`, `reg.Create`s the row, **eagerly** `reg.Save`s, and replies `conversation_created` via `c.Reply` (dispatcher stamps `id`/`ts`/`in_reply_to`). Declares its own minimal `ConversationCreator` interface (`Create` + `Save`) that `*conversations.Registry` satisfies structurally — same consumer-declared-interface discipline as `ConversationLister`. |
| `cmd/pyry/relay.go` | Register `protocol.TypeCreateConversation` on **both** dispatch paths: the v1 `d.Register(...)` block and the v2 `Handlers` map, alongside the existing three. `startRelay` / `startRelayV2` grow a `defaultCwd string` parameter; the registry path reuses the existing `resolveConversationsRegistryPath(instanceName)`. |
| `cmd/pyry/main.go` | New `resolveDefaultCwd(workdir string) string` helper mirroring `resolveClaudeSessionsDir`: absolute form of `*workdir`, or `os.Getwd()` when empty, falling back to the raw value on `Abs`/`Getwd` error so the call site always gets a usable string. Computed once in `runSupervisor` and threaded into the single `startRelay` call (`main.go:489`). Records the conversation's `Cwd` to match where the bootstrap session actually runs. |
| `internal/relay/handlers/create_conversation_test.go` | **New.** Five flat `t.Parallel()` tests via the light `dispatch.NewTestConn` idiom (real `*conversations.Registry` + `t.TempDir()` path, no mocks). |

Field resolution: `promoted := p.IsPromoted != nil && *p.IsPromoted` (default
`false`); `cwd := defaultCwd; if p.Cwd != nil { cwd = *p.Cwd }`; `name` is a
pointer passthrough (`nil` stays `nil` — an unnamed scratch discussion). The
all-null `{"is_promoted":false,"name":null,"cwd":null}` request is the common
"scratch discussion" path (matches `testdata/create_conversation.json`).

**No new protocol types, no registry edits.** `TypeCreateConversation` /
`TypeConversationCreated`, `CreateConversationPayload` /
`ConversationCreatedPayload`, round-trip tests, and fixtures were already in the
tree ([protocol-package.md]); `Registry.Create` / `Save` / `NewID` already
existed ([conversations-registry.md]). The handler is the same `dispatch.Handler`
type on both paths — the v2 manager runs it through the identical
`dispatch.Conn`/`Reply` machinery, proven by `ListConversations` working on both
paths today.

## Key decision — eager persistence (divergence from the lazy sweep)

`Registry.Create` is in-memory only, and the sweep loop
([conversations-auto-archive.md]) Saves **lazily** — only on a non-zero archive
tick. A freshly created conversation would therefore be **non-durable**: absent
from the registry on the next `pyry start`, which a crash-recovery supervisor
performs routinely. For a product whose defining promise is surviving restarts,
a create path that silently drops user-created state on restart is a
self-contradiction. So the handler **Saves eagerly after Create**, matching the
established `pyry pair` precedent (Save immediately after `Add`,
`cmd/pyry/pair.go`) — both are user-initiated genesis-of-state events. The Save
is **best-effort / non-fatal**: on failure it logs `Error`
(`event=create_conversation.persist_failed`) and still sends the success reply,
because the row is live in-memory and immediately usable; durability is
best-effort exactly as `RunSweepLoop` treats its own Save.

## Key decision — no session spawn at create time (no split)

The original ticket flagged a possible session-binding split. Evidence said no:
`send_message` writes to the bootstrap session via `TurnWriter.WriteUserTurn`,
and conversation validity is just `convReg.Get` (the pool's `ValidateConversation`
closure, `internal/sessions/pool.go`). A `Registry.Create`d row validates
**immediately**, so a created conversation accepts a `send_message` on the same
id with **no claude session spawned at create time** — lazy bind on first turn
already works. AC#3's "immediately accepts a send_message" is satisfied by the
in-memory Create alone; proven at unit level by
`TestCreateConversation_CreatedID_ValidatesForSendMessage`.

## Patterns established

- **A user-genesis create path Saves eagerly; the lazy sweep is for background
  bulk archive only.** When an interactive handler creates durable user state,
  persist it inline (best-effort, non-fatal) rather than relying on the
  archive-tick sweep — otherwise a routine daemon restart drops it. Mirror the
  `pyry pair` Add→Save precedent, not `RunSweepLoop`'s laziness. The two Saves
  can race on the same path; both snapshot under the registry lock with the new
  row already appended and commit via atomic temp+rename, so it is byte-level
  last-writer-wins with no lost row.
- **Phone-influenced path metadata stored verbatim is acceptable while it is
  inert; the future spawn-consumer owns canonicalisation.** The payload's `cwd`
  is the only phone-influenced field that could touch the filesystem, but **no
  code path in the current tree** spawns a process, `chdir`s, or joins a path
  from `conversation.Cwd` — its sole reader is `list_conversations`'s echo back
  to the operator's own paired devices. Storing it verbatim is the evidence-based
  call (no observed exploit, no consumer to exploit it). A code comment on the
  handler states the cwd is inert stored metadata and that any future ticket
  spawning a per-conversation session at `conversation.Cwd` MUST canonicalise +
  boundary-check the path first — the same deferred-enforcement discipline #594's
  spec applied to a phone-influenced `SessionJSONLPath`.
- **Server-minted id closes the inbound-create trust surface.** The id comes from
  `conversations.NewID()` (`crypto/rand`), never the phone, so a phone cannot
  choose, collide, or overwrite a row (`Create` appends, never upserts) nor forge
  the reply correlator (`in_reply_to` is the dispatcher-stamped `env.ID`).

## Lessons learned

- **The create-path eager `Save` intentionally diverges from `RunSweepLoop`'s
  lazy (archive-tick-only) persistence: an interactive user-genesis event must
  be durable across the routine daemon restarts a crash-recovery supervisor
  performs, matching the `pyry pair` precedent (Save immediately after `Add`).**
  (From the PR body's Lessons learned.)
- **A missing dispatch handler was invisible because the no-handler path was not
  logged on the daemon.** `dispatch.Route` returns `protocol.unsupported` for an
  unregistered type silently, so the gap stayed hidden until the 2026-06-18 live
  run surfaced it phone-side. Registration parity across the v1 `d.Register`
  block and the v2 `Handlers` map is the structural guard — both lists must add
  every new verb together (this ticket adds one line to each).

## Testing

`internal/relay/handlers/create_conversation_test.go` — five `t.Parallel()`
tests, stdlib only, `dispatch.NewTestConn` (NextID not pre-advanced, so the
first reply lands at `id=1`):

- **`…_AllNull_CreatesRowAndReplies`** — all-null payload (the scratch-discussion
  fixture shape): reply `Type == conversation_created`, `InReplyTo` → request id,
  `id == 1`, payload `Cwd == defaultCwd` / `IsPromoted == false` / `Name == nil` /
  `LastUsedAt` non-zero / `ID` is `conversations.ValidID`-shaped, and the registry
  holds exactly one row matching the reply (`LastUsedAt` compared via `.Equal`).
- **`…_ExplicitFields_RecordedVerbatim`** — `is_promoted=true`, `name="proj"`,
  `cwd="/work/proj"` stored and echoed verbatim; the explicit cwd is **not**
  overridden by `defaultCwd`.
- **`…_EagerPersist_SurvivesReload`** — after the call, `conversations.Load(path)`
  from disk finds the created row (proves the eager Save, i.e. survives a daemon
  restart — the load-bearing assertion for the eager-persist decision).
- **`…_Malformed_EmitsProtocolMalformed`** — `Payload = []byte("{")` → `Type ==
  error`, `Code == protocol.CodeProtocolMalformed`, `Retryable == false`, message
  is the static `msgCreateConversationMalformed` (decode-error text not echoed),
  registry left empty (no partial row).
- **`…_CreatedID_ValidatesForSendMessage`** — `reg.Get` resolves the new id, the
  same predicate `ValidateConversation` uses to admit a `send_message` (AC#3 at
  unit level).

`go test -race ./...` + `go vet ./...` pass. AC#3's full end-to-end leg (real
phone / pyrycode-mobile#421 rung-3) is **operator-verified**, not a Go test — it
exercises the live Noise v2 channel, the mobile client, and navigation; the unit
suite proves the daemon-side contract.

## Concurrency model

No new goroutines. The handler runs synchronously on the dispatcher's per-conn
goroutine (the same context every handler runs in). `Registry.Create` / `Get` /
`Save` are all mutex-guarded. The eager `Save` can race the sweep loop's `Save`
on the same path: each snapshots under `r.mu` (both snapshots include the
freshly-appended row), writes a private temp file, and `os.Rename`s into place —
byte-level last-writer-wins, no lost row, no new lock ordering.

## Out of scope (named)

- **`conversation_updated` broadcast to the operator's other phones.** A
  `ConversationUpdatedPayload` exists; multi-phone create-sync is deferred to a
  future ticket. The AC asks only for the `in_reply_to` reply to the creator.
- **Per-conversation session-spawn cwd confinement.** The deferred SHOULD FIX
  above: the future spawn-consumer canonicalises + boundary-checks
  `conversation.Cwd` before use.
