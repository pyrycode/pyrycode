# Ticket #422 — `e2e/realclaude`: SIGTERM-mid-tool_use cleanup regression guard

Ninth consumer of the `internal/e2e/realclaude` fixture trio (#372 + #373 + #409). Adds one named test — `TestRealClaude_SigtermMidToolUse` — that pins pyry's behaviour when SIGTERM lands while real claude has a Bash subprocess in flight. Three production invariants are pinned together in one cell of the resilience matrix `resilience_test.go` (#382) started: (a) no orphan Bash subprocess survives pyry's exit, (b) the on-disk session JSONL ends at a complete envelope boundary (no half-written trailing line a future `--continue` would choke on), and (c) pyry exits within the 5 s `streamrunner.killGrace` window (a hang IS the regression). Same family as the 2026-05-16 fork-bomb incident (`ca8b688`) — different shape (test binary self-execing via `PYRY_E2E_BIN` inheritance, not signal handling), same class (cleanup paths leaking subprocesses or inconsistent state).

See [features/e2e-realclaude.md](../features/e2e-realclaude.md) for the suite contract.

## Implementation

- **`internal/e2e/realclaude/sigterm_mid_tool_use_test.go`** (~280 LoC including the package-level comment block and inline rationale, new). `//go:build e2e_realclaude`, `package realclaude`. Single top-level `TestRealClaude_SigtermMidToolUse` + two file-local helpers (`spawnPyryAgentRun`, `processesInProcessGroup`) + two file-scope prompt constants. No edits to `fixtures.go`, `resilience_test.go`, `tool_loop_test.go`, or any other peer file — purely additive.
- **Flow.** `WithWorktreeAuthenticated(t)` (opt-in real-API fixture) → write `prompt.txt` / `system.txt` at `0o600` → `ensurePyryBuilt(t)` (build-once-per-process, reused from `RunPyryAgentRun`) → `spawnPyryAgentRun(t, …)` returns `*exec.Cmd` started under its own process group → `time.Sleep(3 * time.Second)` → `cmd.Process.Signal(syscall.SIGTERM)` → bounded `cmd.Wait` in a goroutine raced against `time.After(5 * time.Second)` → orphan check via `pgrep -g <pgid>` → JSONL byte-tail check → `ReadJSONL` walk for tool_use-present / tool_result-absent.
- **Spawn helper `spawnPyryAgentRun(t, bin, workdir, promptPath, systemPath, stdoutBuf, stderrBuf) *exec.Cmd`** (~20 LoC). Constructs the same eight-flag argv as `RunPyryAgentRun` (`agent-run --prompt-file=… --system-prompt-file=… --allowed-tools=Bash --max-turns=2 --effort=low --model=claude-haiku-4-5 --workdir=… --output-format=stream-json`) but does NOT wait for completion — the test needs PID access to send SIGTERM mid-run. `cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}` makes pyry its own process-group leader so `pgid == cmd.Process.Pid` and the orphan check can walk the group. `cmd.Env = os.Environ()` inherits `HOME` and `ANTHROPIC_API_KEY` set by `WithWorktreeAuthenticated`. **`RunPyryAgentRun` is unusable here** — it runs to completion synchronously and provides no PID access during the run. Per the ticket's "Technical Notes" guidance and `resilience_test.go`'s precedent (`resolveClaudeBin` / `runClaudeDirect` are also file-local), the helper stays file-local; promote to `fixtures.go` only when a second test needs the same shape.
- **Orphan check `processesInProcessGroup(t, pgid int) []int`** (~15 LoC). Shells out to `pgrep -g <pgid>`. Exit code 1 with empty output → no matches (success). Exit code 0 with PID lines → orphans; the test filters out `pgid` itself (defense against transient zombie visibility) and returns the rest. Any other error → `t.Fatalf` naming `pgrep -g` as required on both macOS and Linux. **`pgrep -g` is the architect-picked shape**, not `pgrep -P`: after pyry exits, claude (and its Bash descendant) are reparented to init/launchd, so `-P <pyry-pid>` (direct-children only) would falsely report "no orphans" regardless of whether the orphan is still alive. Process-group membership is preserved across reparenting, so `-g <pgid>` catches the actual leak. The AC explicitly invited the architect to pick the shape "most reliable on macOS + Linux"; `pgrep -g` is that shape — present on macOS since 10.8 and on Linux via procps-ng.
- **`t.Cleanup(func() { _ = syscall.Kill(-pgid, syscall.SIGKILL) })` defense-in-depth.** Beyond the orphan check's salvage `syscall.Kill(-pgid, SIGKILL)` on a positive hit, the test registers an unconditional cleanup that reaps the process group at test end. ESRCH on the success path is harmless; the cleanup guards the path where an early `t.Fatalf` (e.g. `Process.Signal` returning unexpectedly) trips before the orphan check's salvage runs. The orphan assertion still has teeth — the test only fails if leaked processes are present at the moment of check, not just at test exit.
- **Pre-SIGTERM window: 3 s `time.Sleep`.** Balances "long enough for claude to write the system/init envelope to stdout, write the assistant `tool_use` envelope to disk, and fork the Bash subprocess" against "short enough not to dominate wall-clock". Per the architect's empirical reasoning, 3 s is reasonable for haiku-low; if the "no Bash tool_use observed" failure fires reproducibly, raise to 5 s — do NOT go below 2 s (the tool_use envelope write is the timing-critical event) and do NOT lower `--max-turns` or shorten the prompt (those don't help and may break the test in other ways).
- **Bounded exit assertion: 5 s `time.After`.** Matches `internal/agentrun/streamrunner/runner.go:40`'s `killGrace = 5 * time.Second` constant. Pyry's production teardown is structurally bounded by it: SIGTERM → ctx cancel from `signal.NotifyContext(ctx, syscall.SIGTERM, syscall.SIGINT)` in `cmd/pyry/agent_run.go:runAgentRun` → `streamrunner.Run`'s `cmd.Cancel = SIGTERM` + `cmd.WaitDelay = killGrace` → `cmd.Wait` returns within `killGrace` (SIGKILL fallback via `WaitDelay`). Deadline expiry IS the regression being guarded against — a hang at this point is not a flake. On timeout the test calls `cmd.Process.Kill()` + bounded drain of the wait channel (2 s, so a wedged child does not hang the suite) before `t.Fatalf`-ing with the stderr capped via `truncate`.
- **JSONL terminal-shape pin: branch B.** The architect picked branch B (clean stream truncation at a complete envelope boundary) over branch A (structured trailer with non-success subtype) on the reasoning that the on-disk JSONL is claude's session-state file for `--continue`, NOT a stream-json result stream — there is no evidence claude flushes a structured trailer line to disk on signal (the stream-json result trailer is a stdout feature, not an on-disk JSONL feature). Assertion sequence: (1) `parseInitSessionID` over captured stdout, (2) `os.ReadFile` of `jsonlPathFor(workdir, sessionID)`, (3) **byte-tail check** — `len(jsonlBytes) > 0` AND `jsonlBytes[len-1] == '\n'` (failure message names trailing-byte count + last-newline index, the ONLY way to surface a half-written tail since `ReadJSONL` silently retains it — see `internal/agentrun/jsonl/reader.go:188-262`), (4) `ReadJSONL` walk for the first `assistant`-kinded event whose content blocks include `{Type:"tool_use", Name:"Bash", ID:!=""}` (record `bashToolUseID`), (5) walk subsequent `user`-kinded events and assert NONE contain a `{Type:"tool_result", ToolUseID==bashToolUseID}` content block. Each failure message gives a remediation hint distinguishing "test setup wrong" from "production broken" (`no Bash tool_use observed` → raise pre-SIGTERM sleep; `matching tool_result on disk` → reduce sleep or extend Bash duration; `pyry did not exit within 5s` → real hang regression; `does not end with a newline` → real append-discipline regression). If a future probe reveals branch A actually holds (a structured trailer IS present), flip assertion (5) to "find a result envelope with `subtype != success`" — same surface, opposite sign; the spec contract that the shape is **pinned** in one direction or the other is unchanged.
- **Prompts.** Two file-scope constants. `sigtermSystemPrompt` is the one-paragraph regression-guard steering (`"You are an e2e regression-guard test. When asked to run a shell command, use the Bash tool exactly once, run the command verbatim, do NOT chain commands with && or ;, do NOT comment, and do NOT do anything else."` — mirrors `longSessionSystemPrompt`'s anti-chain wording). `sigtermPrompt` is the user task (`"Use the Bash tool to run \`sleep 30\`. Do nothing else."`). `sleep 30` is far longer than the combined 3 s pre-SIGTERM + 5 s post-SIGTERM window so the Bash subprocess is guaranteed to be alive (and the relevant in-flight `tool_use`) when the signal lands. `--max-turns=2` is the minimum that lets claude reach the `tool_use` event — turn 1 is the assistant `tool_use`; `1` would not give claude a budget to call a tool. Turn 2 never completes because SIGTERM tears claude down before the Bash subprocess returns.
- **Cost.** One real haiku call per run, ~$0.005 with cache warm (lower than the suite's other haiku-low tests because the run is truncated at SIGTERM before claude generates the final response turn — only the `tool_use` envelope is produced and billed before pyry tears the call down). `t.Parallel()` is NOT called — matches the existing realclaude convention.

The architect chose this over plain `WithWorktree` because the assertions depend on claude actually invoking Bash (a synthetic-auth envelope would skip the `tool_use` event entirely and the "no Bash tool_use observed" assertion would fire on a non-regression).

- `realclaude.parseContentBlocks` + `contentBlock` (#376, extended #382) — content-block decoder; **fifth consumer**. The struct's existing fields (`Type`, `Name`, `ID`, `ToolUseID`) cover exactly what assertions (4) and (5) need; no struct extension required.

- `realclaude.truncate` (#381) — 1 KiB-cap byte-slice formatter for failure messages; **fifth consumer**.

## Patterns established

- **Process-group cleanup, not direct-children, for orphan checks after parent exit.** When a test asserts "the parent's subtree is fully reaped" after the parent exits, `pgrep -P <parent-pid>` is structurally wrong: the child is reparented to init/launchd at the moment the parent exits, so the parent-pid linkage is gone before the test can observe it. Process-group membership is preserved across reparenting — set `cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true}` on the parent and `pgrep -g <pgid>` (or `syscall.Kill(-pgid, 0)`) catches the actual orphan. The same trick is what makes `syscall.Kill(-pgid, SIGKILL)` a one-call subtree reap for salvage-on-failure. Generalises to any test that asserts on the post-mortem state of a process subtree.
- **Defense-in-depth cleanup belongs in `t.Cleanup`, not at the end of the assertion sequence.** The orphan check's positive-hit salvage (`syscall.Kill(-pgid, SIGKILL)` before `t.Fatalf`) and the unconditional `t.Cleanup(func() { _ = syscall.Kill(-pgid, SIGKILL) })` registered immediately after `cmd.Start` serve different purposes: the salvage is part of the contract diagnostic (clean up the leak this run detected), the cleanup is the safety net for any early `t.Fatalf` that trips before the salvage runs. Registering the cleanup at the top of the test (right after the spawn) means it covers every assertion path including the bounded-wait timeout case where the salvage block doesn't execute. Generalises to any test that spawns long-lived subprocesses: register the subtree-reap cleanup at spawn time, not at success time.
- **Bounded-wait assertions need a bounded post-`t.Fatalf` drain too.** When a `select { case <-waitDone: ; case <-time.After(D): }` block fails the deadline branch and then calls `cmd.Process.Kill()`, the goroutine still running `cmd.Wait()` may still be writing to the captured stdout/stderr buffers; reading those buffers in the `t.Fatalf` message racing the kill produces undefined behaviour under `-race`. The fix is a bounded second `select` after `Kill` that drains `waitDone` or times out (2 s ceiling so a truly wedged child does not hang the suite). The buffers are then read after the drain, where the stdlib's internal forwarder writes are happens-before. Generalises to any test that uses a `time.After` race against a goroutine-driven `cmd.Wait` and then needs to read child-managed buffers in the timeout branch.
- **Bytes-tail JSONL check is the ONLY way to surface a half-written line.** `ReadJSONL` parses through `jsonl.NewReader` which silently retains trailing partial bytes internally and NEVER returns them — by design, because the production reader is streaming over an open log that's being appended to. Asserting "the file ends at a complete envelope boundary" therefore CANNOT route through `ReadJSONL`'s event slice; it has to read raw bytes via `os.ReadFile` and check `jsonlBytes[len-1] == '\n'`. The check is one line; missing it would silently green-light a regression that leaves a half-written `tool_result` line on disk and breaks the next `--continue`. Generalises to any assertion on JSONL append discipline where "no half-written tail" is the contract.

## Lessons learned

- **The 5 s SIGTERM-to-exit deadline IS the production contract, not a test-tuning budget.** First instinct on writing the bounded-wait assertion was "pick a deadline that comfortably covers haiku's teardown latency". Wrong — the 5 s number is not arbitrary; it's `internal/agentrun/streamrunner/runner.go:40`'s `killGrace = 5 * time.Second`, which sets pyry's own `cmd.WaitDelay = killGrace` on the claude child. Production behaviour is exactly: SIGTERM lands on pyry, ctx cancels, streamrunner forwards SIGTERM to claude, and SIGKILL fires `killGrace` later if claude hasn't exited. The test deadline pins the SAME constant from the outside. A future PR that bumps `killGrace` to 10 s would (correctly) break this test, forcing the maintainer to read both sides together. Generalises: when a test deadline is structurally tied to a production constant, name the constant in a comment so the linkage is discoverable from the failure message.
- **`Process.Pid == pgid` only when `Setpgid: true` is set BEFORE `Start`.** The orphan check needs the process-group ID, not the pid; on Linux/macOS a fresh process inherits its parent's pgid unless `setpgid(0, 0)` is called (or `SysProcAttr.Setpgid = true` is set, which the stdlib turns into a `setpgid(0, 0)` immediately after fork). The test captures `pgid := cmd.Process.Pid` AFTER `cmd.Start` returns, relying on the `Setpgid: true` invariant that the child placed itself in a new group whose pgid equals its pid. Forgetting the `SysProcAttr` field would silently make `pgrep -g <pgid>` walk the TEST's own group (the Go test binary's group), pulling in unrelated processes and either failing-loudly with confusing diagnostics or — worse — false-positiving the orphan check on a benign sibling. Generalises: when relying on `pgid == pid`, the `SysProcAttr.Setpgid` field is load-bearing and not a stylistic choice.
- **A `tool_use` envelope on disk does NOT imply the Bash subprocess has started.** The architect's pre-SIGTERM sleep budget covers two events that happen in order: (a) claude writes the `tool_use` envelope to the on-disk JSONL, (b) claude forks the Bash subprocess. The assertion that the orphan check is meaningful — i.e. that there's actually something to leak — depends on (b) having happened. The 3 s sleep is sized to clear both; raising it to 5 s on flakes is the documented recourse precisely because the (a)-only state would still satisfy the "no Bash tool_use observed" assertion but fail to exercise the cleanup path the test is meant to pin. If a future probe ever observes "tool_use present, no orphan check failure, but the Bash subprocess never actually started", the pre-SIGTERM sleep is the lever — the assertion is structurally fine. Generalises to any test that asserts on a multi-stage state machine: name the stage you actually depend on, and budget for it explicitly, not just the first observable transition.
- **Test-side `t.Cleanup` runs AFTER the test goroutine returns, not after `t.Fatalf` immediately.** The `t.Cleanup(func() { _ = syscall.Kill(-pgid, syscall.SIGKILL) })` registered after `cmd.Start` does NOT defeat the orphan-check assertion: `t.Fatalf` marks the test as failed and stops the test goroutine, but registered cleanups run AFTERWARDS. So the orphan check runs first, sees the leak, reports it via `t.Fatalf`, and only then does the cleanup salvage-kill the group. The test correctly fails. Worth verifying once on any test that mixes `t.Cleanup` with assertions on resources the cleanup would tear down.

## Out of scope (deferred)

- **SIGKILL handling.** Forced kill is a different contract — partial state IS the contract on SIGKILL — and a separate ticket if/when needed. The 5 s assertion deliberately doesn't extend to "after a forced kill"; it pins the graceful-shutdown contract only.
- **PTY-driven sessions.** Post-stream-json migration this surface is not in scope; the test exercises `pyry agent-run` (stream-json), not the legacy PTY supervisor path.
- **Multi-session Pool SIGTERM.** The Pool wrapper has its own cleanup paths (`internal/sessions`); a SIGTERM-mid-tool_use cell that drives the Pool is a separate ticket with a separate fixture surface.
- **Promoting `spawnPyryAgentRun` / `processesInProcessGroup` to `fixtures.go`.** Single-file consumer in #422; the architect's spec deliberately keeps both helpers file-local to match `resilience_test.go`'s precedent for raw-exec helpers. Promote when a second test needs the same shape.
- **Cross-platform alternative to `pgrep -g`.** If a future CI runner ships without `pgrep` (procps-ng), the test fails loudly with "pgrep -g is required" — the architect's recourse is then either replace the shell-out with `syscall.Kill(-pgid, 0)` + `errors.Is(err, syscall.ESRCH)` (no diagnostic PID enumeration) or install pgrep in the CI image. Not in scope for #422.
