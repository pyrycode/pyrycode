# Ticket #631 — daemon appends `/v1/server` to a base `relay_url` (or fails loudly on a non-101)

The daemon dialed its configured `relay_url` **verbatim**, so the shipped default
(`DefaultConfig().RelayURL = "wss://relay.pyrycode.dev"` — a **base** URL with no
path) made it dial relay-root `/`, which the live relay 404s on the WS upgrade.
That produced an **endless silent INFO backoff loop** (`"transport: dial failed,
backing off" attempt=39 …`) — no hint the path was wrong, so it read as a phone
problem when it was a daemon-config problem. Caught live 2026-06-08 on the first
real phone↔daemon test; the env-override workaround was
`PYRY_RELAY_URL=…/v1/server`.

The root cause is an **asymmetry** in how the single `~/.pyry/config.json`
`relay_url` is consumed: `pyry pair` (`cmd/pyry/pair.go`) puts the URL in the QR
as a **base** and the mobile client appends `/v1/client` itself, so a base config
is correct there; the daemon dialed verbatim, so it needed `/v1/server` baked in.
The natural single value (the base) silently broke the daemon. The e2e harness
hid the gap by always passing `fr.URL()+"/v1/server"` explicitly.

This slice implements the spec's recommended **Option A + Option B** combination:
append the path **and** keep a loud error if the upgrade still fails. Package
references updated this ticket: [features/relay-package.md](../features/relay-package.md)
(Option A) and [features/transport-package.md](../features/transport-package.md)
(Option B).

## Implementation

Two production files, no exported-signature changes anywhere → no consumer
cascade (`relay.Connect`'s only production caller, `cmd/pyry/relay.go:122`,
passes `RelayURL` unchanged and now gets the append for free).

### Option A — `internal/relay/connection.go` (`resolveDialURL`)

`Connect`'s inline `url.Parse` + scheme-validate block was extracted into a new
unexported pure helper that becomes the **single home for all relay-URL
handling** (parse → scheme → path-append):

```go
func resolveDialURL(raw string, allowInsecure bool) (string, error)
```

- **Parse failure** → `fmt.Errorf("%w: RelayURL parse: %v", ErrInvalidConfig, err)`
  — byte-identical to the pre-extraction message, so `TestConfig_Validation_TableDriven`
  keeps passing through `Connect`.
- **Scheme not `wss`** (and not `ws` under `allowInsecure`) → `"%w: RelayURL
  scheme must be wss (got %q)"` (unchanged).
- **Path append (the only new logic):** if `u.Path == "" || u.Path == "/"` → set
  `u.Path = "/v1/server"`; otherwise leave it untouched. Returns `u.String()`,
  whose stdlib reconstruction preserves host/port/query/userinfo
  (`wss://h/?x=1` → `wss://h/v1/server?x=1`).

`Connect` keeps its non-empty `RelayURL` guard (so the precise `"RelayURL is
required"` message survives), then calls `resolveDialURL` and assigns the result
to `tcfg.URL`.

| `relay_url` (config or `-pyry-relay`) | Dialed URL |
|---|---|
| `wss://relay.pyrycode.dev` | `wss://relay.pyrycode.dev/v1/server` |
| `wss://relay.pyrycode.dev/` | `wss://relay.pyrycode.dev/v1/server` |
| `wss://relay.pyrycode.dev/v1/server` | passthrough |
| `wss://relay.pyrycode.dev/v2/server` | passthrough (AC#4 — an operator path is preserved) |
| `ws://127.0.0.1:54321` (e2e, insecure) | `ws://127.0.0.1:54321/v1/server` |

This mirrors the phone's `/v1/client` convention (`fakephone.Dial` does
`baseURL+"/v1/client"`), so a single base `relay_url` now serves **both**
`pyry pair` and the daemon with no `PYRY_RELAY_URL` override (AC#2). The
`/v1/server` literal lives only in `relay`, keeping `transport` protocol-agnostic.

### Option B — `internal/transport/wssclient.go` (`ErrUpgradeRejected`)

The safety net for the cases A cannot prevent (operator typed an explicit wrong
path, relay misroute): a remaining non-101 upgrade now **fails loudly on the
first failure** instead of being buried in silent backoff (AC#1's diagnostic
branch).

- **New sentinel** `ErrUpgradeRejected` (in the existing `var (…)` block).
- **`realDial`** captures the response: `conn, resp, err := websocket.Dial(...)`.
  coder/websocket returns a **non-nil `*http.Response` iff** the relay answered
  the upgrade with a non-101 status; a network-level failure (DNS, refused, TLS)
  leaves `resp == nil`. So `err != nil && resp != nil` →
  `fmt.Errorf("%w (HTTP %d): %w", ErrUpgradeRejected, resp.StatusCode, err)`
  (double-`%w` preserves both the sentinel and the underlying error chain);
  `resp == nil` → the existing `"dial: %w"` (unchanged).
- **`Connect` dial-fail branch** gates a single WARN per outage with a loop-local
  `warnedUpgrade := false`: first `errors.Is(err, ErrUpgradeRejected)` →
  `Logger.Warn("transport: relay rejected the WebSocket upgrade; verify the relay
  URL path is correct", …)` and set the flag; subsequent rejects fall to the
  existing INFO backoff line. `warnedUpgrade` is **re-armed** (`= false`) on the
  successful-connect path, so a fresh outage after a healthy connection gets its
  own first-failure WARN.

`ErrUpgradeRejected` is deliberately **not** in `FatalCloseCodes` and does not
unwind the daemon — it is a log-classification signal only. The supervisor's
retry-forever posture is preserved (the relay or config may be fixed live and the
next dial succeeds, re-arming the warning). Network failures (relay down) stay at
INFO — they are transient, not a config error.

`make check` (vet + `-race` + staticcheck + `cmd/substrate-guard`) green;
implemented test-first (RED → GREEN). Code review **PASS, no findings**.

## Patterns established

- **Mirror the phone's base-URL convention on the daemon side.** Both consumers
  read the same base `relay_url`; the phone appends `/v1/client`, the daemon now
  appends `/v1/server`. A single config value is the right shape — the fix was to
  make the daemon honour it the same way the phone already does, not to demand two
  values or an env override. The endpoint knowledge lives in the layer that owns
  the protocol (`relay`), never in the generic `transport`.
- **`resp != nil` is the string-free signal for "server answered" vs "never
  reached".** coder/websocket's `Dial` returns a non-nil `*http.Response` **iff**
  the peer responded with a non-101 status; a network-level failure leaves it nil.
  Classifying on the response pointer (not on matching `"404"`/`"handshake"` in
  the error string) is the clean way to split "wrong path / misroute" (loud,
  actionable) from "relay down" (quiet, transient) — worth reusing for any future
  dial-error classification.
- **Loud-once-per-outage, re-armed on recovery.** A config-actionable failure
  warrants a WARN, but repeating it every backoff attempt (capped at 30s) is spam.
  A loop-local boolean gated on the first occurrence and reset on the next
  successful connect gives "loud on the first failed upgrade, not buried" (the AC
  wording) without a spam stream — and a *new* outage still gets its own WARN.
- **Extract-to-one-home preserves byte-identical error messages.** Folding
  `Connect`'s inline parse/scheme block into `resolveDialURL` kept the exact
  `ErrInvalidConfig` wrapping and message strings (`"RelayURL parse"`, `"scheme
  must be wss"`), so the existing `Connect`-level validation table stayed green
  with no edit — the refactor is invisible to callers and tests.
- **slog level encodes operator-actionability.** WARN for the config-actionable
  upgrade rejection (the operator can fix the path); INFO for transient network
  backoff (nothing to do but wait). The recording-handler test pins exactly this
  split.

## Lessons learned

- **coder/websocket's `Dial` returns a non-nil `*http.Response` iff the relay
  answered with a non-101 status; a network-level failure leaves it nil.** That
  `resp != nil` check is the clean, string-free signal for distinguishing "wrong
  path / misroute" from "relay down" — worth knowing for any future dial-error
  classification. (From the PR body's Lessons-learned section.)
- **A single config value consumed by two clients with different path conventions
  is a latent footgun.** The base `relay_url` was correct for `pyry pair` and
  silently wrong for the daemon, and the shipped *default* tripped it out of the
  box. The e2e harness masked the gap for months by pre-appending `/v1/server` at
  every call site — a test fixture that "helpfully" normalises an input is exactly
  how a real-world default path goes untested. The new e2e
  (`TestRelay_BaseURLAppendsServerPath`) deliberately dials the **bare** base URL
  to exercise the path the harness used to hide.

## Scope (S confirmed)

Two production files, both single-purpose: `resolveDialURL` (~8 lines relocated +
3 new) and the `ErrUpgradeRejected` classify/warn (~6 lines). No exported-signature
changes, no consumer cascade, no new dependencies, no screen-byte surface (so
`substrate-guard` is unaffected). Well under all split red lines.

## Out of scope (deferred)

- **WARN-every-N-attempts during a sustained outage** — the spec demotes
  post-first rejections to INFO and re-arms after a successful connect. If
  operators want a periodic WARN during a long outage that's a trivial follow-up;
  left out to avoid log spam (backoff caps at 30s, so even the first WARN is the
  actionable one). Evidence-based: no observed need.
- **`/v1/server` as a shared constant** — the path is a single string literal in
  `resolveDialURL`; the phone-side literal lives in a separate repo. Not worth a
  shared constant for one daemon-side occurrence.
