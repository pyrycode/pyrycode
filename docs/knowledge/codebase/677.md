# Ticket #677 — `create_conversation` mints & binds a dedicated claude session per conversation

The foundational slice of EPIC #672 ("per-conversation sessions"). `create_conversation` previously only wrote a registry row with `CurrentSessionID` empty, so every discussion shared the single bootstrap claude. #677 wires the create path onto the existing `sessions.Pool`: each conversation now eagerly mints + binds its own dedicated session, recorded via the existing `Conversation.CurrentSessionID` field, so discussions are isolated. The session spawns in the daemon's already-trust-marked shared workdir — the phone-influenced `Cwd` is **not** yet a spawn input (deferred to the per-conversation-workdir follow-up).

See the evergreen feature doc: [features/conversation-session-binding.md](../features/conversation-session-binding.md).

## Implementation

- **Substantive change concentrated in `internal/relay/handlers/create_conversation.go`.** The handler now mints a session before recording the row:
  - New `SessionCreator` interface (`Create(ctx, label) (string, error)`) — a consumer-declared seam mirroring the sibling `TurnWriter`, keeping `handlers/` free of any `internal/sessions` import.
  - `creator.Create(mintCtx, string(id))` wrapped in a 30s `context.WithTimeout` (`createConversationMintTimeout`). The label is the server-minted conversation id — a session↔conversation breadcrumb in the session registry; it never reaches claude's argv.
  - The one substantive line: `reg.Create(Conversation{… CurrentSessionID: sessionID …})` before the existing eager `Save`. AC#3 (persistence across restart) is then automatic — the registry round-trips all fields.
  - Mint failure → `Warn` log (`create_conversation.session_mint_failed`) + reply `protocol.CodeServerBinaryOffline` retryable, returning **before** `reg.Create` (no half-bound orphan row). Success log gains a `session_id` field.
  - The SECURITY comment was rewritten: the handler is now a spawn-consumer, but spawns into `buildSession`'s `tpl.WorkDir` (trusted shared workdir), never `conversation.Cwd`.
- **Threading (mechanical, compiler-checked).** A narrow `creator handlers.SessionCreator` param added to `startRelay` + `startRelayV2` (`cmd/pyry/relay.go`), threaded into both `handlers.CreateConversation(...)` registration sites (v1 dispatcher, v2 manager handler map). `*sessions.Pool` is adapted to the interface by a new `sessionMinter{p}` wrapper in `cmd/pyry/main.go` (mirrors the existing `poolResolver` — `Pool.Create` returns `SessionID`, the interface speaks `string`).
- **Tests** (`create_conversation_test.go`): new `stubSessionCreator` test double (mirrors `stubTurnWriter`); returns a fresh distinct id per call so AC#2 distinctness is observable. New/rewritten cases — `BindsDedicatedSession` (AC#1, replaces the inverted `CreatedID_ValidatesForSendMessage`), `DistinctSessionPerConversation` (AC#2), `EagerPersist_SurvivesReload` extended to assert the binding round-trips (AC#3), `CwdNeverReachesMint` (AC#4), `MintFailure_NoRowAndBinaryOffline`, plus a "malformed short-circuits before mint" assertion. All `t.Parallel()`.
- **Out of scope (deferred to #672 follow-ups):** distinct per-conversation working directory + its trust/boundary validation; per-operator create quota / rate-limit (dispatch policy). See the feature doc.

## Patterns established

- **Eager bind forced by an AC, not by the architect's preference.** The ticket delegated eager-vs-lazy to the architect, but AC#1 ("the row carries a non-empty `CurrentSessionID` referring to a pool session *after the frame is handled*") settles it: a lazy first-message bind would leave the field empty until a later `send_message`. Reuse `Pool.Create` as-is — no new "register-without-spawn" primitive (that would be a new exported method + scope creep). General lesson: read an AC's *timing clause* before choosing a mechanism the AC may already constrain.
- **Covariant adapter at the one package that knows both types.** `*sessions.Pool.Create` returns `SessionID`; the consumer interface speaks `string`. The adapter (`sessionMinter`) lives in `cmd/pyry/main.go` — the only package importing both — beside the precedent `poolResolver` / `transitionObserverSink`. Two distinct capabilities (mint vs transition-observe) get two distinct narrow interfaces, not one conflated sink (per CODING-STYLE).
- **AC#4 enforced structurally, not by validation.** `Cwd` cannot reach the spawn path because the handler passes only `(mintCtx, string(id))` and the `SessionCreator` interface has no cwd parameter; `buildSession` puts only the server-minted `--session-id` on the argv. The code-review independently verified this against `buildSession` (`internal/sessions/pool.go:949`) rather than trusting the spec — structural exclusion beats a runtime check.

## Lessons learned

- **"Exists in the Pool" ≠ "claude is running."** A bound session satisfies AC#1 while it has a registry entry + `p.sessions` map entry; that holds even after idle-eviction moves the process to disk. Evicted is a lifecycle *state*, not removal — the binding is durable through eviction. Don't conflate pool membership with a live process.
- **Accepted residue: `Pool.Create` can return a non-empty id *with* an error.** Per its docstring, the mint may persist and the lifecycle goroutine may still bring the session up against the pool ctx after the handler's 30s timeout fires. The handler treats *any* error as a clean mint failure and never binds it, leaving a session registered in the Pool with no conversation pointing at it. This is benign (same shape as a session that ran and idled out, recoverable by the Pool's own lifecycle) and the race is **unobserved**, so per evidence-based fix selection **no** cleanup logic was added. If this residue is ever observed accumulating, that's the trigger to add reconciliation — not before.
- **Eager binding turns a metadata write into a process spawn — and a new resource-exhaustion surface.** `create_conversation` now spawns a real claude per frame, so an authenticated phone spamming creates can exhaust host processes/memory. The existing bound is `Pool.ActiveCap` (LRU eviction), but it **defaults to uncapped**. A per-operator create quota is new dispatch policy → deferred follow-up; production should set `-pyry-active-cap` and/or `-pyry-idle-timeout`. Flagged honestly in the spec's Open questions + security review rather than papered over with a half-measure.
- **A wedged spawn must be bounded at the consumer.** `Pool.Activate` blocks until the PTY is ready or ctx-cancel; without the 30s `context.WithTimeout` an unbounded wait would pin the per-conn goroutine on a stuck spawn. The bound converts it into a retryable `server.binary_offline` (same idiom as `send_message`'s activate budget).
