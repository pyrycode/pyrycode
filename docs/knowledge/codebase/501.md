# Ticket #501 — `agentrun.EncodeProjectDir` delegates to `tuidriver.EncodeCwd` (single source of truth for the projects-dir encoder)

[`internal/agentrun.EncodeProjectDir`](https://github.com/pyrycode/pyrycode/blob/main/internal/agentrun/workdir.go) now calls `tuidriver.EncodeCwd` instead of the local `strings.NewReplacer("/", "-", ".", "-")`. The old encoder mapped only `/` and `.` to `-`; claude's real on-disk encoder maps **every byte outside `[a-zA-Z0-9]`** to `-` (per-byte, no run collapse — verified empirically and pinned by [`tui-driver/pkg/tuidriver.EncodeCwd`](https://github.com/pyrycode/tui-driver) as of 2026-05-19). The divergence wedged every real-claude e2e test whose workdir contained `_` or ` ` (Go test temp dirs whose names include underscores, e.g. `Test_With_Underscores`): ptyrunner's JSONL tail watched the pyry-encoded path while claude wrote at the per-byte-encoded sibling, the 30 s PTY-quiet watchdog fired, and the run wedged at ~55 s with `error_during_execution num_turns:0`. This ticket is the root-cause fix for that wedge pattern reported in [#499](https://github.com/pyrycode/pyrycode/issues/499).

## Implementation

- **Option A from the ticket — re-export the tui-driver encoder.** `workdir.go` drops `projectDirReplacer` and the `strings` import, adds `github.com/pyrycode/tui-driver/pkg/tuidriver`, and the body of `EncodeProjectDir` becomes `tuidriver.EncodeCwd(resolved)`. The exported signature `(string, error)` is unchanged. `ResolveWorkdir` is untouched. The `tui-driver` module was already a direct dependency of `internal/agentrun/ptyrunner` siblings, so no new module-graph surface — Option B (inline byte-loop) was rejected for two-source-of-truth drift risk.

- **Doc-comment on `EncodeProjectDir` updated.** Now reads "maps every byte outside `[a-zA-Z0-9]` to '-' (matching how claude derives the on-disk projects-dir name; see tuidriver.EncodeCwd)" instead of the old "maps '/' and '.' to '-'". The cross-reference to `tuidriver.EncodeCwd` is the auditable source-of-truth pointer; future contributors who wonder which rule wins find it in the field comment.

- **Six call sites untouched.** Consumers (`internal/agentrun/jsonl/tail/watcher.go:115`, `internal/e2e/realclaude/fixtures.go:366`, `prompt_fidelity_test.go:84`, `fixtures_test.go:296/:559`, `watcher_test.go:94/:260`, `ptyrunner/runner_test.go:35`) treat the returned encoded string as opaque and compose it into a path. They were *already* calling `EncodeProjectDir` expecting it to match claude's on-disk path — they were just getting the wrong answer when the workdir contained non-alnum bytes. The fix makes them right; grep confirmed no test hardcodes an expected encoded string that contained `_` or space under the old rule.

- **`TestEncodeProjectDir_LiteralSubstitution` oracle flipped.** Pre-#501 it computed its expected value with `strings.NewReplacer("/", "-", ".", "-").Replace(resolved)` — that was *the implementation under test*, asserting trivially. Post-#501 the oracle is `tuidriver.EncodeCwd(resolved)`, making this case the parity-gate AC #1 demanded. On macOS, `t.TempDir()` emits a path containing the running test's name, so this case differentially exercises the underscore mapping (`TestEncodeProjectDir_LiteralSubstitution` itself contains `_`) every run.

- **New `TestEncodeProjectDir_NonAlnumBytes` table.** Four sub-cases pin the per-byte rule end-to-end through `ResolveWorkdir`: underscores (`Test_With_Underscores` → `-Test-With-Underscores`), space (`with space` → `-with-space`), idempotent pre-existing dash (`already-dashed` → `-already-dashed`), and mixed specials (`a_b-c.d e` → `-a-b-c-d-e`). Each constructs the workdir with `filepath.Join(t.TempDir(), <literal>) + os.MkdirAll`, then asserts `strings.HasSuffix(got, want)` rather than full equality — the `t.TempDir()` prefix is variable and not the assertion target. `t.Parallel()` on outer and on each subtest, stdlib `testing` only.

- **Existing tests unchanged.** `TestEncodeProjectDir_DarwinRealpath` (prefix check on `-private-var-folders-`) and `TestEncodeProjectDir_DotInPathSegment` (`.hidden` → `--hidden` suffix) pass either encoder; both `/` and `.` are non-alnum so the new rule agrees on those inputs. `TestEncodeProjectDir_MissingPath` is a pure error-path test and unaffected. All `TestResolveWorkdir_*` cases unaffected — `ResolveWorkdir` is not modified.

- **Out-of-band regression gate.** `make e2e-realclaude` (AC #3) is the third acceptance signal — no new e2e test was added because the existing real-claude tests in `internal/e2e/realclaude/` already encode their `t.TempDir()` workdirs through `_`-bearing test names. The moment the encoder is right, the on-disk JSONL tail finds the file claude is writing. Per AC #3, remaining failures of *other* root causes are out of scope; the goal is to remove the `54-55 s error_during_execution num_turns:0` failure mode that #499 surfaced.

## Patterns established

- **Single source of truth for cross-module wire-format rules — re-export, don't reimplement.** When two packages both need to compute "the same encoding rule that claude uses on disk," the source of truth lives in the package closest to the rule (here, `tui-driver/pkg/tuidriver` because tui-driver's job *is* speaking claude's protocol). Other packages delegate. Reimplementation — even of a one-line `strings.NewReplacer` — is a future-drift trap: the original `internal/agentrun` encoder shipped #347-era based on a partial sample of claude's directory names (alnum + `/` + `.` only); when later evidence showed `_` and ` ` also map to `-`, the rule moved to `tuidriver` but the duplicate in `internal/agentrun` quietly diverged. The general rule: **one encoder per wire-format contract**, located at the package that owns the protocol.

- **Test oracles must not be the implementation under test.** `TestEncodeProjectDir_LiteralSubstitution`'s pre-#501 oracle was `strings.NewReplacer("/", "-", ".", "-").Replace(resolved)` — copy-pasted from the implementation. The test passed trivially regardless of what the rule *should* have been, because both sides computed the same wrong answer. The post-#501 oracle is `tuidriver.EncodeCwd(resolved)`, which is the *independent* canonical encoder. Generalises: when a test's oracle reads "I'll compute the expected value by re-doing the operation locally," check whether the oracle and the implementation are tracking the same source of truth. If yes, the test guards encoding *consistency* but not *correctness*; pair it with at least one literal-expected-value case (`{input: "...", want: "..."}`) that pins the actual rule.

- **Per-byte loops over `strings.NewReplacer` for character-class encoders.** `tuidriver.EncodeCwd` is a per-byte loop with `unicode.IsLetter || unicode.IsDigit`-equivalent check, not a `NewReplacer` with an enumerated character list. The replacer pattern only works when the input character set is small and fully known; the moment the rule becomes "every char outside class X," `NewReplacer` is the wrong tool — adding a character to the rule requires editing the call site, while a byte-loop with a class predicate is closed under future class refinements. `EncodeProjectDir` cleanly absorbs this: the package no longer enumerates which bytes map to `-`; it asks tui-driver, which uses the class predicate.

## Lessons learned

- **The PR body had no `## Lessons learned` section.** Nothing surprising surfaced beyond what the spec already captured. The work was a clean Option A delegation; the spec's recommendation, the test plan, and the implementation matched one-for-one. Recorded for the per-ticket convention: when a ticket lands exactly as scoped with no surprises, the codebase doc still gets written (this file) but the "Lessons learned" section can be empty or note the absence — a deliberate "nothing notable" beats silence.

- **The code-review NIT is a follow-up, not a deferred fix.** Reviewer found that `internal/sessions/reconcile.go:32` and `internal/e2e/rotation_test.go:23` still carry the same narrow `strings.NewReplacer("/", "-", ".", "-")` encoder with stale doc comments. The ticket and spec are explicit about `internal/agentrun/` only, so it's correctly out of scope. But the AC #2 claim "single source of truth" is only partially true today — `tuidriver.EncodeCwd` is the only encoder *in `internal/agentrun/`*, not project-wide. The right move is a follow-up ticket that delegates those two remaining call sites; recorded under § Out of scope below.

- **`tuidriver` was already a direct dependency, so the coupling argument for Option B had no weight.** The architect noted this in the spec; the developer didn't have to weigh the cost. Generalises: when proposing two implementation options where one adds a dependency, check `go.mod` and the package's existing import graph *before* presenting the trade-off. If the dependency already exists, the "added coupling" cost is zero.
