# Ticket #288 ‚Äî transport: regression test for close-status preference under sendPump race

Pure test addition. Pins the close-status preference loop in `transport.serve` (landed in #248, see [`248.md`](248.md) ¬ß "Three pump-return errors, only one carries the close status") under the racing-error shape that the existing `TestFatalCloseCodes_HaltsReconnect` happy-path does not exercise: when `recvPump` and `sendPump` both error from the same peer-close event, the `CloseError` must be selected from `errCh` regardless of arrival order so `Connect`'s `FatalCloseCodes` check halts reconnect. Production code unchanged.

## Implementation

### `internal/transport/wssclient_test.go` (+163 LOC)

- **`racingCloseRelay`** ‚Äî new test relay adjacent to `closeCodeRelay`. `httptest.NewServer` that:
  - CAS-gates `connCount` so only the first dial accepts (`http.StatusGone` on subsequent dials ‚Äî a silent reconnect surfaces as the test's `connCount == 1` assertion failing).
  - After `websocket.Accept`, signals `connectedCh` (non-blocking), then spawns an inner **echo loop** reading client frames and writing them back. The echo loop is load-bearing: without it, sendPump's writes pile up in the OS TCP send buffer and `Write` doesn't actually block, shrinking the in-flight-Write window at close time and making the regression-catching property statistically weaker.
  - Blocks on `<-triggerClose`, then calls `conn.Close(status, reason)`; waits for the echo loop to drain before returning.
  - Exposes `URL()` (the `"ws" + strings.TrimPrefix(...)` shape used by every other transport test relay) and idempotent `TriggerClose()` (non-blocking send on a buffered channel).

- **`TestFatalCloseCodes_HaltsReconnect_RacingSendError`** ‚Äî the regression pin:
  1. Stand up `racingCloseRelay` with status `4409`. `Config.FatalCloseCodes = {4409}`, `WriteTimeout = 50ms`.
  2. `newClientForTest` with short cadence (10ms initial reconnect, 50ms max) so a regression would actually re-dial within the 5s ctx budget rather than parking on backoff.
  3. Spawn `c.Connect(ctx)` in a goroutine; gate the rest of the test on BOTH `<-c.Connected()` (client saw a live conn) AND `<-relay.connectedCh` (relay's accept ran).
  4. **Busy-Send loop** ‚Äî tight `c.Send([]byte("x"))` loop in a goroutine, exiting on `Send` error or `stopSend` close. Maximizes the rate of in-flight `sendPump` Writes at close time.
  5. `relay.TriggerClose()` ‚Üí relay sends 4409 close frame ‚Üí recvPump observes `CloseError{Code: 4409}` and sendPump's mid-flight `Write` returns a generic "websocket closed" error. Both push to the 3-buffered `errCh` near-atomically.
  6. Assert `Connect` returns wrapping `ErrFatalClose`, `websocket.CloseStatus(err) == 4409`, `connCount == 1`.

### Why this is deterministic with the fix

The close-status preference loop at `internal/transport/wssclient.go:405-410` walks all three pump returns and selects the first with a recognizable `websocket.CloseStatus`. The test's outcome is INVARIANT across goroutine orderings:

- `recvPump`-first ‚Üí `errs[0]` already carries the close status, preference loop returns it on iteration 0.
- `sendPump`-first ‚Üí `errs[0]` is the generic net error; preference loop walks and finds the `CloseError` at `errs[1]` or `errs[2]`.

Both orderings produce the same observable outcome. No `time.Sleep` ordering hacks ‚Äî channel synchronization throughout.

### Why this catches a regression of the fix

Reverting to `return errs[0]` makes the test outcome scheduler-dependent. Under `-count=10` with busy-Send priming, the `sendPump`-first ordering hits on a meaningful fraction of iterations; any iteration that hits it without the fix sees a generic error in `errs[0]`, `websocket.CloseStatus(err) == -1`, `Connect` reconnects, and the relay's single-shot accept produces `connCount > 1` (or the second dial fails into the dial-error path with no fatal-close classification). Empirically verified: with the fix temporarily reverted, the test fails 10/10 under `-race`. With the fix intact, it passes 20/20 in isolation and across `-count=10` runs of the whole transport-package test suite.

## Lessons learned

### Coverage-of-a-loop tests must exercise the slots, not just the entry

`TestFatalCloseCodes_HaltsReconnect` (pre-#288) accepted the close from a relay that closes IMMEDIATELY on accept ‚Äî recvPump's `CloseError` was effectively guaranteed to arrive first because sendPump and pingLoop never had a frame in flight to fail on. The test passed even if the preference loop was deleted entirely (would-be regression silently uncovered). The fix to the test isn't a more aggressive assertion ‚Äî it's arranging the input so a different slot of the preference loop is the one carrying the close-status error. Generalises: when a defensive-loop fix is added because slot N was the answer in a flaky test, the regression test for the fix must DELIBERATELY put the answer in slot N.

### Concurrent-push tests need draining-counterparties to keep the push-causing operation blocking

The echo loop in `racingCloseRelay` is not for assertion ‚Äî it exists purely so the client's `sendPump.Write` actually blocks on the wire instead of returning immediately into the kernel send buffer. Without the echo, the busy-Send goroutine produces no in-flight `Write` at close time, the sendPump-first ordering essentially never fires, and the test loses its regression-catching property even though it still passes deterministically with the fix. The corollary lesson: when a test relies on a specific failure-mode for a specific goroutine, audit whether the OS / kernel / library is silently buffering its way out of that failure mode.

### Probabilistic regression detection is an honest compromise when the alternative is a production test seam

The strictest interpretation of "deterministic regression coverage" would expose a package-private `selectPumpError(errs []error) error` helper that the test could call directly with hand-crafted error slices. That would catch a regression on every run, not statistically. The cost is a production-code test seam serving no in-prod caller ‚Äî a permanent shape-of-package concession to make a test deterministic. The spec for #288 weighed this and chose the busy-Send + `-count=10` approach, with the documented caveat that a future Go scheduler change could in principle reduce the bug-trigger rate below the detection threshold. Documented here rather than buried in the spec so the next person who notices the probabilism doesn't "fix" it by adding the seam without knowing it was deliberately rejected.
