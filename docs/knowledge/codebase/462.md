# Ticket #462 â€” `internal/relay`: `V2SessionManager.Rekey` + `emitRekeyRequest(reason)` refactor (manual re-key, manager-side)

Slice B1 of the [#460](https://github.com/pyrycode/pyrycode/issues/460) split (itself a re-split of #451). [#459](459.md) shipped the wire contract and the `control.Rekeyer` interface; this slice ships the manager-side implementer. `*V2SessionManager` now satisfies `control.Rekeyer` by funnelling `Rekey(ctx, connID)` requests onto Run's single dispatch goroutine via a new `manualRekey` channel, reusing the existing [#450](450.md) emit machinery with `payload.reason = "manual"` instead of `"scheduled"`. The sibling slice B2 ships the `pyry rekey <conn_id>` operator verb in `cmd/pyry`; production wire-up of `*V2SessionManager` into the daemon (currently no production caller of `NewV2SessionManager`) lands in a separate ticket. Until that wire-up arrives the method is reachable from `internal/relay` tests only â€” same "callable surface, no production caller" posture #459's client helper deliberately ships.

## Implementation

- **One production source file modified** â€” `internal/relay/v2session.go` (+~115 LOC): two new package-var sentinels, one new private request type, one new field on `V2SessionManager`, one constructor delta, one new `Run`-loop select arm, the public `Rekey` method, a private `handleManualRekey` method, a compile-time `var _ control.Rekeyer = (*V2SessionManager)(nil)` assertion, the signature change on `emitRekeyRequest` (now `(ctx, s, reason string)`), and the one existing call site update inside `handleWake`'s `wakeRekeyEmit` arm. Tests +332 LOC in `internal/relay/v2session_test.go` (four new tests under a `// --- re-key manual-trigger tests (#462) ---` header). New imports on the production side: `errors` (for `ErrSessionNotOpen`) and `github.com/pyrycode/pyrycode/internal/control` (for the sentinel-wrap target). No new exported symbols beyond the two sentinels.

- **Method name is `Rekey`, not `TriggerRekey`.** The AC text said `TriggerRekey(connID string) error`; the slice A `control.Rekeyer` interface declared `Rekey(ctx context.Context, connID string) error`. These are not assignment-compatible â€” different name, different signature. The architect's review picked the interface signature so `*V2SessionManager` satisfies `control.Rekeyer` directly without an adapter. The AC's `TriggerRekey` reads as an informal label for the manual-rekey entry point; the wire-contract interface method name wins. A compile-time `var _ control.Rekeyer = (*V2SessionManager)(nil)` pins the contract so a future refactor cannot drift the signature silently.

- **Two new sentinels in `internal/relay`.** `ErrConnNotFound = fmt.Errorf("relay: conn not found: %w", control.ErrConnNotFound)` (wraps the slice A sentinel) and `ErrSessionNotOpen = errors.New("relay: session not open")`. The first is load-bearing: slice A's dispatcher uses `errors.Is(err, control.ErrConnNotFound)` to map to `ErrCodeConnNotFound` on the wire, and `%w` keeps that mapping firing without any further plumbing. The second has no wire-code analogue today â€” slice A defined no `ErrCodeSessionNotOpen`; the control dispatcher surfaces it verbatim through `Response.Error` with no `ErrorCode`. Defining sentinels in `internal/relay` (rather than reusing slice A's) lets the wire-mapping layer route the relay's not-open distinction through the dispatcher whenever a future wire code is added, without dragging relay-internal state-machine vocabulary into `internal/control`. The `relay â†’ control` import is non-cyclic â€” verified at spec time; `internal/control` imports `internal/sessions` and `internal/supervisor`, neither of which imports `internal/relay`.

- **`ErrSessionNotOpen` collapses two distinct internal states.** Both `s.state != V2StateOpen` (still handshaking, or already torn down) and `s.awaitingRekeyReply` (a prior emit is in flight) return the same sentinel. From the operator's perspective both states mean "the conn is not in a state where a manual rekey can be initiated"; distinguishing them externally would leak internal state-machine vocabulary and add a wire code with no operator-actionable consequence. The collapse also imposes a natural per-conn rate limit: at most one manual rekey per `rekeyReplyTimeout` (30 s in production) â€” a second `Rekey` arriving within that window hits the awaiting-reply branch.

- **`manualRekey chan manualRekeyReq` is unbuffered.** Backpressure is the correct semantics here: if `Run` is busy processing a frame, `Rekey` waits. A buffer would mislead the caller into thinking the request was accepted when in reality `Run` had not yet observed it; the caller's `ctx` is the escape hatch in either select arm. The per-request reply channel `req.reply chan error` is `cap=1` so the manager's `req.reply <- err` send is non-blocking even if the caller's `ctx` fires between enqueue and reply. The channel is **not** closed by the manager on `Run` exit â€” in-flight callers unblock via `ctx.Done()` instead. Matches the existing posture for `m.cfg.Frames` (also never closed by the manager).

- **`Rekey` does no session-state access.** The public method runs on the caller's goroutine and performs only channel sends/receives:
  ```go
  func (m *V2SessionManager) Rekey(ctx context.Context, connID string) error {
      req := manualRekeyReq{connID: connID, reply: make(chan error, 1)}
      select {
      case m.manualRekey <- req:
      case <-ctx.Done():
          return ctx.Err()
      }
      select {
      case err := <-req.reply:
          return err
      case <-ctx.Done():
          return ctx.Err()
      }
  }
  ```
  All session-state reads and writes happen in `handleManualRekey`, which runs on `Run`'s goroutine â€” the same goroutine that owns `m.sessions` / `s.state` / `s.send` / `s.rekeyTimer`. No new lock, no new atomic, no new goroutine. Five reject branches total across `Rekey` + `handleManualRekey` (caller-ctx-cancelled before enqueue, caller-ctx-cancelled after enqueue, unknown conn, state â‰  open, awaiting reply); none log on the failure path â€” emission belongs at the operator surface (slice B2) or the wire layer.

- **The `Run` select grows a fourth arm**, placed directly below the existing `case w := <-m.wake:` arm:
  ```go
  case req := <-m.manualRekey:
      req.reply <- m.handleManualRekey(runCtx, req.connID)
  ```
  Single-line dispatch â€” the actual work lives in `handleManualRekey`. The `runCtx` derivation from [#450](450.md) flows through unchanged; the manual path benefits from the same "Frames-channel-close cancels callback goroutines" property via the same mechanism.

- **`handleManualRekey` is the lookup + emit dispatch site.** Looks up `m.sessions[connID]`; not found â†’ `ErrConnNotFound`. State guard: `s.state != V2StateOpen` â†’ `ErrSessionNotOpen`. Awaiting-reply guard: `s.awaitingRekeyReply` â†’ `ErrSessionNotOpen`. If all checks pass: stop + nil `s.rekeyTimer` (re-base the scheduled cadence â€” see below), then call `m.emitRekeyRequest(ctx, s, "manual")`, return nil. The defensive `awaitingRekeyReply`-skip inside `emitRekeyRequest` is now structurally unreachable from the manual path (the explicit check above fires first); it stays because the scheduled path still relies on it for the stale-wake race (Â§ "Stale-wake benign race" below).

- **`emitRekeyRequest` signature gains a `reason string` parameter.** Body deltas are mechanically minimal: the `reqPayload`'s `Reason:` field literal becomes `reason`, and the final `m.cfg.Logger.Info("relay: v2 rekey emit", ...)` carries `"reason", reason` instead of the hardcoded `"reason", "scheduled"`. Every other line â€” the seal-failure WARN, the awaiting-defensive skip, the marshal-failure WARN, the `s.awaitingRekeyReply = true` set, the `armRekeyReplyTimer` call â€” is byte-identical. Call sites: `handleWake`'s `wakeRekeyEmit` arm passes `"scheduled"`; `handleManualRekey` passes `"manual"`. **One emit function, two callers â€” no parallel emit path.** The reviewer's "no parallel emit machinery" AC is satisfied by sharing the function body, not by sharing the wake-channel routing (the manual path skips the wake channel entirely; it runs `emitRekeyRequest` synchronously inside the `manualRekey` select arm on the same goroutine).

- **Timer re-base: stop + nil `s.rekeyTimer` BEFORE the emit.** The AC asks that a manual emit at T+45min not be followed by a stale scheduled emit at the original T+60min boundary. Mechanism is structurally the same as the post-swap re-arm in [#450](450.md)'s `rekeyComplete`: `s.rekeyTimer.Stop()` cancels the pending 1-hour `AfterFunc`, then `s.rekeyTimer = nil` so a future `rekeyComplete` knows to arm a fresh one. `Stop()`'s bool return is intentionally ignored â€” the fired-but-not-yet-delivered race is benign (see Â§ "Stale-wake benign race"). After the emit, `s.awaitingRekeyReply = true` and `s.rekeyReplyTimer` arms (inside `emitRekeyRequest`'s tail). On the phone's reply, `handleRekeyInit` â†’ `rekeyComplete` clears the bool, stops the reply timer, and arms a **fresh** `rekeyTimer` from the swap moment via `armRekeyTimer` â€” the next scheduled emit lands at T_swap + `rekeyInterval`, never at the original boundary. On reply-timeout the conn closes at WS 4426 and the session is removed entirely.

- **Stale-wake benign race.** Sequence: scheduled `rekeyTimer` fires at T=0, the `AfterFunc` callback pushes a `wakeSignal{s, wakeRekeyEmit}` onto `m.wake` (cap 16), `Run` picks up the `manualRekey` arm first (Go's `select` is fair-random), `handleManualRekey` runs `Stop()` (returns false â€” fired), nils the timer, runs the manual emit (sets `awaitingRekeyReply = true`), and then on a subsequent `Run` iteration the stale wake is processed: `handleWake` â†’ `wakeRekeyEmit` arm â†’ `emitRekeyRequest(ctx, s, "scheduled")` â†’ the defensive `awaitingRekeyReply` check catches it and logs `v2.rekey.emit.skipped_already_awaiting`. **No double emit; one spurious WARN.** The architect's security review accepts the noise as benign diagnostic â€” the load-bearing safety property holds (no second outbound `rekey_request` envelope). A more concerning sub-race (the phone responds and `rekeyComplete` clears the bool before the stale wake is processed â†’ a second emit could fire under correct guards) is documented in the spec but rejected as practically unreachable: `m.wake`'s cap-16 buffer drains every `Run` iteration in microseconds, and the harm even if reached is one extra Noise handshake on the phone side (no security impact).

- **AEAD-seal-failure posture: pause-not-close.** `emitRekeyRequest`'s existing Warn-and-drop branch (inherited from [#450](450.md): an internal seal failure is not a wire-protocol failure) means a sub-emit failure on the manual path leaves the conn with `rekeyTimer = nil` and `awaitingRekeyReply = false` â€” automatic scheduled re-keying is paused indefinitely until a phone-initiated rekey rebases the timer via `rekeyComplete`. Acceptable because seal failures are realistically unreachable under correct flynn/noise (same posture as `sealError`), the remediation is operator-visible (the `v2.rekey.emit.seal_failed` log line is the signal, and the operator can re-run `pyry rekey <conn_id>` once slice B2 lands), and re-arming the timer in the seal-failure branch would mask the underlying error AND introduce a code path the scheduled emit doesn't have. The architect's security review documents the behaviour shift; no new sentinel is introduced for seal failure.

- **The control dispatcher's `errors.Is(err, control.ErrConnNotFound)` mapping fires unchanged.** Slice A's `handleRekey` uses `errors.Is` (not `==`), so the relay's `%w`-wrapped sentinel maps to `ErrCodeConnNotFound = "conn_not_found"` on the wire automatically. No additional plumbing on the slice A side. `ErrSessionNotOpen` flows through as `Response{Error: err.Error()}` with no `ErrorCode` â€” the operator client sees `"relay: session not open"` verbatim.

## Patterns established

- **Public method does only channel I/O; private method does state work; one `Run` arm bridges them.** The shape generalises whenever a single-owner-goroutine type needs to expose an externally-callable operation that mutates owned state. The caller-side method runs on the caller's goroutine and only enqueues a request (typed struct with a per-request reply channel `cap=1`); a new `Run` select arm dequeues and dispatches to a private handler that runs on the owner goroutine and does the actual work; the per-request reply channel returns the typed error. No new lock, no new atomic, no new long-lived goroutine. The pattern preserves the single-owner invariant *structurally* â€” there is no syntactic path from the public method to the owned state, so a future refactor that adds a second public method cannot accidentally race against the owner goroutine's handlers. Generalises whenever a Go type needs an external trigger surface but the owned state (here: flynn/noise's mutable nonce-counter `CipherState`) cannot tolerate concurrent access.

- **Wrap a wire-layer sentinel with `%w` to inherit its `errors.Is` mapping.** Slice A's dispatcher uses `errors.Is(err, control.ErrConnNotFound)` to map to `ErrCodeConnNotFound`. This slice's `ErrConnNotFound = fmt.Errorf("relay: conn not found: %w", control.ErrConnNotFound)` inherits that mapping without any change at the dispatcher â€” the producer adds its own package context to the message, the consumer's existing `errors.Is` check still fires. The pattern generalises: when a producer package needs to surface a sentinel through a consumer's existing wire-mapping check, wrap with `%w` rather than re-defining a parallel sentinel at the producer or stripping the package prefix at the consumer. Inverse direction works too (slice A's consumer-side wrap of `sessions.ErrSessionNotFound`); pick by which package owns the canonical contract.

- **Collapse multiple internal states into one external sentinel when the external distinction is operator-irrelevant.** `handleManualRekey` distinguishes three reject states internally (`!ok`, `state != V2StateOpen`, `awaitingRekeyReply`), but the second and third collapse into the same `ErrSessionNotOpen` because from the operator's perspective both mean "the conn is not in a state where a manual rekey can be initiated." Distinguishing them externally would leak internal state-machine vocabulary into the operator surface with no actionable consequence. The pattern generalises: when an internal state machine has multiple "not-eligible" states for the same operation, the external API should expose one "not-eligible" sentinel and a structured log on the producer side (deferred here â€” operator-debug-grade detail in a future ticket if observed). Per-state external sentinels are appropriate only when the operator's response differs per state.

- **Refactor a private method's signature to add an axis of variation; share the body across N callers.** `emitRekeyRequest` gained a `reason string` parameter; the two body deltas are mechanically minimal (struct literal field + log field); two call sites pass distinct literals. The shape generalises: when a private method has two natural callers that need a single axis of behavioural variation, prefer a parameter on the existing method over a second method, a wrapper helper, or a parallel emit path. The "one function body, multiple call sites with distinct literals" shape preserves the AEAD-seal posture, the awaiting-defensive skip, the marshal-failure WARN, and every other branch under one identifier â€” a future refactor of any branch lands at exactly one site, and the security-review surface is one function not two.

## Lessons learned

- **AC-vs-interface naming drift: the interface signature wins.** The AC said `TriggerRekey(connID string) error`; the slice A `control.Rekeyer` interface declared `Rekey(ctx context.Context, connID string) error`. These are not assignment-compatible â€” different name, different parameter set. The architect's review picked the interface signature so `*V2SessionManager` satisfies `control.Rekeyer` directly with no adapter. Slice A's docstring at `internal/control/server.go:138` carried the same drift ("Slice B's *relay.V2SessionManager satisfies it via TriggerRekey") â€” the doc-comment lagged the canonical method name. The lesson generalises: when AC text and an interface declaration disagree on method naming, the interface wins because it carries the type-system check. The compile-time `var _ control.Rekeyer = (*V2SessionManager)(nil)` assertion is the load-bearing artefact â€” without it, a future refactor could drift the name back to the AC's informal label and the interface satisfaction would silently break.

- **A test that needs a long-lived in-flight state asks for the package-var substitution idiom.** `TestV2Session_RekeyManual_AlreadyAwaitingReply_ReturnsErrSessionNotOpen` needed the first emit's awaiting-reply window to stay open across the second `Rekey` call. The natural shape â€” substitute `rekeyReplyTimeout = 10 * time.Second` via the `prev := â€¦; t.Cleanup(restore)` idiom â€” is the same pattern [#450](450.md) established and the same posture #453's tests use. The test is explicitly NOT `t.Parallel()` because the substitution is process-global; a parallel-sibling test reading `rekeyReplyTimeout` during the substitution window would observe the wrong value and flake. The lesson: package-var test substitution and `t.Parallel()` are structurally incompatible whenever a goroutine spawned by the test reads the substituted value; the no-parallel posture is declarative, not a performance regression â€” flag it in a top-of-test comment so a future contributor doesn't add `t.Parallel()` for "faster CI" and re-introduce the race.

- **The timer-rebase test's load-bearing assertion is the negative one, not the positive one.** `TestV2Session_RekeyManual_RebasesScheduledTimer` makes two assertions: (1) past the original scheduled boundary the envelope count is still 3 (manual + responder + nothing else), and (2) at T_rekeyComplete + rekeyInterval a fourth envelope arrives with `reason == "scheduled"`. The positive assertion (2) proves the re-armed timer is real; the negative assertion (1) proves the original timer was actually stopped. A future refactor that accidentally re-arms the timer without first stopping the old one â€” e.g. setting `s.rekeyTimer = m.armRekeyTimer(...)` without calling `Stop()` on the old reference first â€” would pass assertion (2) (the new timer fires) but fail assertion (1) (the old timer also fires). The asymmetry is structural: positive-only assertions cannot catch a "did the right thing AND ALSO the wrong thing" bug class. Generalises whenever a state-machine refactor replaces one timer/handler/callback with another â€” assert the *absence* of the old behaviour in the window where it would have fired, not just the *presence* of the new behaviour at the new boundary.

- **Slice naming convention: "B1" for the manager-side producer, "B2" for the operator-side CLI consumer.** This slice is documented as "slice B1 of the [#460](https://github.com/pyrycode/pyrycode/issues/460) split (itself a re-split of #451)." The two-letter-plus-digit convention captures the dependency chain: slice A (#459) ships the wire contract, B1 (this slice, #462) ships the producer, B2 ships the consumer CLI. Once both B1 and B2 land + the production wire-up (a separate, unnumbered ticket) lands, the operator-facing `pyry rekey <conn_id>` is end-to-end functional. The lesson: when a single user-visible feature splits across N tickets bound by a strict dependency chain, the naming convention (letters for top-level phases, digits for parallelisable sub-slices) captures the chain in the ticket vocabulary itself â€” a reader who finds B1 knows there's also a B2, and the existence of B1+B2 implies an A predecessor. Future readers benefit from the structural clarity; ticket titles alone don't carry it.
