# Ticket #395 — `-pyry-idle-timeout` defaults to `0` (idle eviction opt-in)

Flips the CLI default for `-pyry-idle-timeout` from `15m` to `0`. The `15m` default fired deterministically 15 minutes after every `systemctl --user restart pyry` and — compounded by a companion respawn-on-attach bug (separate ticket) — produced a silent bot outage on daemon-mode installs (`pyry install-service`, #202/#190) until manual restart. Failure mode was confirmed on pyrybox 2026-05-15 with two back-to-back cycles showing `claude exited "signal: killed"` at `uptime=15m0.130s` / `15m0.050s`.

## Implementation

- **One-line production change.** `cmd/pyry/main.go:405` — `fs.Duration("pyry-idle-timeout", 0, …)`. The short help string and the long `printHelp` block (`main.go:1329`) both rephrase to "default 0 / disabled; pass e.g. 15m to enable". The "respawn latency 2-15s on next attach" rider was dropped from the long help — it described post-eviction behaviour that's only meaningful when eviction is on, and the literal "2-15s" claim is wrong while the companion respawn-on-attach bug is open.
- **No plumbing change downstream of `cmd/pyry`.** `sessions.Pool` / `Session.runActive` already treat `IdleTimeout == 0` as "never arm the idle timer" via the never-armed `time.Timer` / nil-channel pattern (see [ADR 005](../decisions/005-idle-eviction-state-machine.md) and the lesson at `docs/lessons.md:88`). The wiring at `cmd/pyry/main.go:462` is direct field assignment (`IdleTimeout: *idleTimeout`), so a zero value flows through unchanged.
- **No refactor.** The flag construction wasn't extracted into a helper struct or `registerPyryFlags(fs)`. A literal-flip regression deserves a literal-flip fix; the rest of `runSupervisor`'s 60-line flag block is unrelated surface area.
- **Operator opt-in.** Anyone who wants the prior behaviour passes `-pyry-idle-timeout=15m` (or any non-zero duration) — pool semantics on the opt-in path are identical to pre-#395.

## Tests

- **New regression guard:** `cmd/pyry/args_test.go::TestPyryIdleTimeoutDefault` registers `-pyry-idle-timeout` on a fresh `flag.FlagSet` with default `0`, parses empty argv, and asserts the parsed value is `time.Duration(0)`. The test deliberately doesn't invoke `runSupervisor` or assemble a real `sessions.Pool` — the "0 disables eviction" invariant is already pinned by `internal/sessions.TestPool_ParityWhenIdleDisabled`. This test only guards the CLI-seam literal.
- **No existing test required modification.** Audit at write time: `internal/sessions/{session,session_persist,pool}_test.go`, `internal/e2e/idle_test.go`, `internal/e2e/{harness,auto_attach,attach_pty,install_*}.go`, `internal/e2e/cap_test.go`, and `cmd/pyry/update_e2e_test.go:271` all either set `IdleTimeout` explicitly on `SessionConfig` / `Config`, or pass `-pyry-idle-timeout=…` explicitly to `StartIn`. No test depended on the implicit `15m` default.

## Patterns established

- **Defaults follow the dominant deployment.** `pyry`'s deployment shifted from "operator runs `pyry claude …` interactively" to "operator installs a long-lived service via `pyry install-service`" over Phase 2. A default chosen for the older shape (#40 / ADR 005, amortised memory cost matters on short sessions) became a silent foot-gun on the newer shape (warm claude is the product). When the dominant deployment changes, revisit the defaults — don't add a daemon-aware default special-case inside `install-service`, which proliferates code paths. The rejected alternative is recorded in the ticket and in the ADR footer.
- **CLI-seam regression guards are sufficient at the CLI seam.** The new test asserts only the literal `0`; downstream zero-handling is the pool's coverage to maintain. Splitting responsibilities this way keeps the cmd-level test from re-proving invariants that already have dedicated tests, and keeps the test fast (no goroutines, no pool construction).
- **ADR amendments append, never edit.** [ADR 005](../decisions/005-idle-eviction-state-machine.md) gained an `## Update — 2026-05-15` footer noting the default reversal and the daemon-mode rationale. The original `Consequences › Positive` line still cites `15m` as the production default — that captures the reasoning at the time of the ADR and stays intact. Readers see both the original decision and the later reversal in one file.

## Lessons learned

- **"0 = disabled" semantics must be re-verified at every flag flip.** The fix relied on the pool already treating `IdleTimeout == 0` as "never arm the timer" — that's true today (the never-armed `time.Timer` pattern at `docs/lessons.md:88`), but it's an invariant that lives in the consumer, not at the CLI. The new test guards the CLI literal; `TestPool_ParityWhenIdleDisabled` guards the consumer. Both must exist for the fix to be safe to reason about — if anyone refactors the pool to special-case `IdleTimeout < someThreshold` instead of `== 0`, the CLI test alone wouldn't catch it.
- **Silent eviction is the worst failure mode for a supervisor.** The eviction emitted no log line naming it as an eviction; the only on-the-ground signal was `claude exited "signal: killed"` 15 minutes after the supervisor came up. The companion respawn-on-attach bug then prevented recovery. Even after this default flip, operators who explicitly enable eviction will hit the same opacity if they don't tail logs — a follow-up ticket on the eviction call site to emit an explicit "evicting idle claude" log line at info level would close the residual hazard. Not in scope for this ticket; flagged here for the next time someone touches `Session.runActive`'s eviction branch.

## Out of scope (deferred)

- Companion respawn-on-attach bug — separate ticket. The default flip *avoids* the bug for daemon-mode installs by ensuring eviction never fires by default; it does not fix the underlying respawn failure on the opt-in path.
- Explicit "evicting idle claude" log line at the eviction call site — see "Lessons learned" above.
- Daemon-aware default injected by `install-service` — rejected in the ticket body. The rejection rationale is recorded in the ADR 005 footer.
