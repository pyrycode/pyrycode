# Ticket #574 — ptyrunner `--max-turns` budget counts claude's logical turns, not raw assistant events

The `budget.Counter` charged `c.count` for **every** assistant JSONL entry, but claude
2.1.158 serialises one logical reply as multiple consecutive assistant entries sharing a
`message.id` (a `thinking` line, a `tool_use` line, a `text` line). The Counter charged
each, so one logical turn cost 2–3 and a `--max-turns=N` cap was reached ~twice as fast on
ptyrunner as on the **streamrunner** baseline. [#561](https://github.com/pyrycode/pyrycode/issues/561)
worked around the symptom by bumping test `maxTurns` ceilings; this is the root cause. This
is the **enforcement** sibling of [#573](573.md) (the **reporting** path, which fixed the
identical over-count in `streamjson/emitter.go`'s `num_turns`). The fix applies the same
predicate to the budget Counter and — because the two counters must agree forever on "what
is a turn" — extracts that predicate into a single shared function both call, so the
reported `num_turns` and the enforced budget cannot drift. Split from
[#567](https://github.com/pyrycode/pyrycode/issues/567).

## Implementation

Three production files: one new ~1-LOC-body helper plus two ~6-line call-site changes. No
signature changes to existing symbols; no `go.mod`/`go.sum` change.

- **New `internal/agentrun/turncount.go` — the shared predicate.** One exported pure
  function:
  ```go
  func IsNewLogicalTurn(currentID, lastID string) bool {
      return currentID == "" || currentID != lastID
  }
  ```
  Reports whether an assistant entry with `message.id == currentID` begins a new logical
  turn, given `lastID` (the previous assistant entry's id; `""` before any). `currentID ==
  ""` → its own turn (the empty-id floor); a changed id → new turn; same id → same turn.
  This is the **single source of truth** for the turn boundary, byte-identical to the
  predicate #573 had inline in the emitter.
- **`budget.Counter` (`budget.go`).** New unexported field `lastAssistantMsgID string`
  alongside `count` under the pre-existing `mu` (`""` = no assistant entry seen yet). In
  `OnEvent`, after the `entry.Type != "assistant"` early-return, `id` is extracted from
  `entry.Message` (nil-guarded) **before** the lock, then under `c.mu` the unconditional
  `c.count++` becomes `if agentrun.IsNewLogicalTurn(id, c.lastAssistantMsgID) { c.count++ }`
  followed by `c.lastAssistantMsgID = id` — all before the existing `fired` short-circuit
  and `count < MaxTurns` gate. `OnEndOfTurn`, `Reason`, `Stop`, and the SIGTERM→SIGKILL
  escalation are **untouched** — only what increments `c.count` changes.
- **`streamjson.Emitter` (`emitter.go`) — behaviour-preserving refactor.** The inline
  boundary test `if id == "" || id != e.lastAssistantMsgID` is replaced by
  `if agentrun.IsNewLogicalTurn(id, e.lastAssistantMsgID)`; the `id`-extraction lines above
  and the `e.lastAssistantMsgID = id` line below stay. Adds the `internal/agentrun` import.
  After this change there are **exactly two callers** of the helper and **no inline
  turn-boundary predicate remains** anywhere — the drift surface is closed.

The id-extraction (`entry.Message != nil` nil-guard) deliberately stays **inline** in each
consumer — it is trivial nil-safety, orthogonal to the turn definition, and keeping it out
of the helper is what lets `IsNewLogicalTurn` operate over two `string`s rather than a
`JSONLEntry` (see Patterns below).

## Patterns established

- **Extract a shared function — not a cross-reference comment — when divergence is *always*
  a bug.** The drift-prevention AC was explicit and the ticket left the mechanism to the
  architect (extract vs. replicate-with-comment). Extraction won because the entire purpose
  of the ticket is that reported `num_turns` and the enforced budget agree on "what is a
  turn" — divergence is never a desired future state. This is the **inverse** of the
  atomic-write-recipe duplication the codebase deliberately tolerates (`PROJECT-MEMORY.md`
  § "Atomic-write recipe duplicated until a fifth registry forces extraction"), where each
  registry's copy is expected to diverge over time. The distinguishing test for "extract or
  duplicate": *is future divergence always a bug, or a likely future need?* A shared
  function is a deterministic drift guarantee; a comment is advisory.
- **When "no shared home" blocks extraction, express the predicate over primitives, not the
  domain type.** #573's sole objection to extracting was "no natural shared home below both
  packages — tuidriver declines `msg_id` grouping." That dissolved by making the helper take
  two `string`s (`currentID, lastID`) instead of a `tuidriver.JSONLEntry`: the home becomes
  the common parent package `internal/agentrun` (already imported by `budget`, the parent of
  `streamjson`), which stays free of any `tuidriver` dependency, so adding `streamjson →
  agentrun` introduces no import cycle (`agentrun` imports neither subpackage). Lifting the
  domain-specific extraction out and keeping it inline is the move that unblocks a clean
  shared home — and it makes the predicate trivially unit-testable without constructing
  entries. Generalisable beyond this ticket.
- **Reported and enforced turn counts now share one definition.** `num_turns` (streamjson
  reporting) and `--max-turns` (budget enforcement) both flow through `IsNewLogicalTurn`;
  any future change to "what counts as a turn" lands in exactly one place. See
  [features/budget-package.md](../features/budget-package.md) and
  [features/streamjson-package.md](../features/streamjson-package.md), both updated to the
  shared-predicate framing.

## Testing

- **New `internal/agentrun/turncount_test.go`** — table-driven, `t.Parallel`, stdlib only.
  Five cases pin the pure predicate: first entry (`"",""`)→true, empty id after non-empty
  (`"","msg_A"`)→true, first non-empty (`"msg_A",""`)→true, id changed
  (`"msg_B","msg_A"`)→true, same id (`"msg_A","msg_A"`)→false.
- **`budget_test.go` — new `TestOnEvent_CountsLogicalTurns`** (table-driven, reuses
  `signalRecorder`/`mustNew`). A new `assistantEntryID(id string)` helper builds an
  assistant entry carrying `message.id == id` (sibling of the existing `assistantEntry()`,
  which has an empty id). Cases:
  - **Split reply = 1 turn (AC#1).** `MaxTurns=2`; feed `msg_A`, `msg_A`, an interleaved
    `user` (`tool_result`) entry, then `msg_B`. Terminate fires only after `msg_B` (index
    3) — the interleaved non-assistant entry hits the early-return and does **not** split
    the straddled turn.
  - **K turns span >K entries (AC#2).** `MaxTurns=3`; six entries (`msg_1`×2, `msg_2`×2,
    `msg_3`×2). Terminate fires when the 3rd distinct id first appears (index 4), **not** at
    `MaxTurns` raw entries (old per-entry counting would have fired at index 2, mid-turn-2).
  - **One distinct id per entry (AC#5).** `msg_A`,`msg_B`,`msg_C`, `MaxTurns=3` → fires at
    index 2.
  - **Empty-id floor (AC#5).** Three `assistantEntry()` (empty id), `MaxTurns=3` → each is
    its own turn, fires at index 2.
- **No-regression.** All pre-existing budget tests use `assistantEntry()` (empty id) and
  stay green unchanged under the empty-id floor — e.g. `TestOnEvent_SIGTERMFiresExactlyAtBudget`
  still fires at the 3rd of 3. The emitter's 21 existing tests (incl.
  `TestEmit_NumTurnsCountsLogicalTurns`, `TestCapturedFixture_ByteEquivalence`) pin the
  refactor as behaviour-preserving with **no test edits**.
- **Verification.** `go build ./cmd/pyry`, `go test -race ./internal/agentrun/...`, and
  `go vet ./...` all pass.
- **Code review: judgment review PASS.** Implementation matches the spec exactly; no
  MUST/SHOULD/NIT findings. The drift surface is closed (exactly two callers, no remaining
  inline predicate). **The mandated real-claude e2e gate could not execute** in the review
  sandbox: all 18 `realclaude` tests fail uniformly at the harness `go build` step because
  it fetches the **private** `github.com/pyrycode/tui-driver` dep into a hermetic temp
  module cache with no GitHub credentials. This is a pre-existing import at `budget.go:21`
  unchanged by this PR (this PR adds only an internal `internal/agentrun` import and touches
  no `go.mod`/`go.sum`); `go build ./cmd/pyry` succeeds against the normal cache. Reviewer
  declined to route back to the developer (not a developer-fixable regression — it recurs
  every review in an uncredentialed sandbox) and escalated to **operator review** to run the
  gate in a credentialed environment. Merged as PR #580.
- **Fixture churn (incidental).** `internal/e2e/realclaude/testdata/permission_protocol_v2.1.158.json`
  changed in the code-review auto-commit, but identifier-only (`session_id`, `message.id`s,
  `toolu` ids); the structure that #573's e2e relies on is preserved — 3 assistant events,
  2 distinct `message.id`s, native `num_turns: 2`. This fixture belongs to #573's e2e test,
  not to any test #574 adds; the churn is not a #574 behaviour change.

## Lessons learned

- **Extract vs. duplicate is decided by whether future divergence is a bug or a need.** Two
  copies that must stay bit-identical forever (a metric and its enforcement) should be one
  shared function; two copies expected to diverge as each grows its own needs (the
  per-registry atomic-write recipe) should stay duplicated. A cross-reference comment is the
  wrong tool for the first case — it is advisory, and the AC asked for a guarantee.
- **"No shared home" is often an artefact of extracting the wrong thing.** #573 couldn't
  find a home because it pictured a helper over `JSONLEntry`, which drags in `tuidriver`.
  Extracting the *pure boundary rule* over two `string`s — and leaving the entry-specific
  nil-guard inline — moved the home up to the dependency-free parent package and removed the
  cycle/home objection entirely. When extraction seems blocked by dependencies, check
  whether the genuinely-shared core can be expressed over primitives.
- **A synthesised count that shadows claude's native one must use claude's own grouping.**
  Both this ticket and #573 trace to the same root: ptyrunner hand-rolls counts that claude
  emits natively (`num_turns`), and claude 2.1.158 splits one logical reply across several
  assistant JSONL entries sharing a `message.id`. Any count or budget keyed on "assistant
  entries" must group by `message.id` or it silently doubles. #561's test-ceiling bumps were
  treating the symptom.

## Out of scope

- **Reverting #561's test `maxTurns` ceiling bumps.** Now that the per-entry over-count is
  fixed on both the reporting (#573) and enforcement (#574) paths, those bumped ceilings are
  redundant — but the cleanup is an explicitly-deferred separate follow-up, not part of this
  ticket.
- **Defending `A,B,A` `message.id` interleaving with a seen-set.** Transition-counting
  equals distinct-id counting only if one message's entries are never interleaved with
  another's; claude completes one assistant message before starting the next, and no `A,B,A`
  has been observed. Per Evidence-Based Fix Selection, no seen-set is added. `IsNewLogicalTurn`
  is the single place such a change would land if it ever becomes necessary.
