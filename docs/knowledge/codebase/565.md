# Ticket #565 — ptyrunner: reap claude's descendant process groups on operator SIGTERM

On `pyry agent-run` (ptyrunner, the default path), SIGTERM while claude has a Bash command in
flight used to leave claude's Bash subprocess orphaned. claude (2.1.158) runs **every** Bash command
in its own detached process group two levels below pyry — `pyry → claude → zsh -c eval '<cmd>' →
<cmd>` (e.g. `tail -f /dev/null`), where the `zsh`+`cmd` live in a NEW group claude creates. pyry
reaped claude (its direct child) but not that group, so on operator SIGTERM the group survived,
reparented to init. For a command that never returns the orphan was **unbounded**. This ticket makes
pyry walk claude's descendant tree at operator-SIGTERM teardown and SIGKILL the orphan-able groups
itself. Surfaced while recalibrating the SIGTERM e2e for claude 2.1.158 (#563); the e2e origin is
[#422](422.md).

## Implementation

Two parts, both confined to `internal/agentrun/ptyrunner`. No new exported symbols, no new deps
(`go.mod`/`go.sum` untouched).

- **New file `reap.go`** (~125 LoC), two unexported functions:
  - **`descendantPGIDs(ctx, rootPid) (map[int]struct{}, error)`** — takes a single
    `ps -axo pid=,ppid=,pgid=` snapshot (one portable enumeration across Linux + macOS; the trailing
    `=` on each column suppresses the header so every line is three integers), BFS-walks `rootPid`'s
    transitive descendants over the `ppid→children` map, and returns their distinct `pgid`s. A `seen`
    set guards against a pid-reuse cycle in a stale snapshot. Bounded by a 2s ctx timeout
    (`reapPSTimeout`) so a hung `ps` cannot wedge teardown; empty on enumeration failure.
  - **`reapDescendantGroups(rootPid, logger)`** — `syscall.Kill(-pgid, SIGKILL)` each descendant
    group (negative pid = whole group), except three **load-bearing guards**: `pgid <= 1`
    (init/invalid), `pgid == syscall.Getpgrp()` (pyry's own group — never suicide), and
    `pgid == rootPid` (claude is a `setsid`/`pty.Start` group leader, so `pgid == pid`; `sess.Close()`
    owns claude's own teardown). SIGKILL, not SIGTERM-then-grace: these are abandoned tool commands
    whose output is already discarded (no `tool_result` will be sent), and the AC wants a hard "no
    longer running" guarantee. `ESRCH` is benign (group already exited in the window). Best-effort and
    content-blind: enumeration/kill failures log at Warn (pgids/counts only — never command strings)
    and don't propagate; claude still gets its SIGTERM regardless.

- **Wiring in `runner.go`** (~20 lines incl. comments), set after `tuidriver.EnsureClaudeEnv(cmd)`
  and **before** `tuidriver.Spawn` so the hook is installed when os/exec's ctx watcher is armed at
  `Start`:
  ```go
  cmd.Cancel = func() error {
      reapDescendantGroups(cmd.Process.Pid, logger) // SIGKILL claude's detached Bash group(s)
      return cmd.Process.Signal(syscall.SIGTERM)     // then graceful SIGTERM (lets claude flush JSONL)
  }
  cmd.WaitDelay = killGrace // local const = 5s; mirrors streamrunner.killGrace / supervisor.spawnWaitDelay
  ```

### Why direction (b), not (a) — measured

The ticket offered (a) mirror streamrunner's SIGTERM grace so claude reaps its own child, or (b)
track + kill claude's descendant groups. The architect ran the repro against a throwaway build with
(a) applied: claude given a graceful SIGTERM + grace exits cleanly (`num_turns:2`) but **still leaves
the Bash group it deliberately isolated orphaned**, 3/3 runs. Relying on claude to reap a grandchild
it intentionally detached is a stochastic dependency that does not hold, so the guarantee must come
from deterministic pyry code — *belt-and-suspenders means different fabric*. Direction (a)'s graceful
SIGTERM is **still adopted**, not for reaping but so claude flushes its session JSONL cleanly
(invariant 2) instead of dying mid-write under the old immediate SIGKILL. Direction (b) (the
`reap.go` walk) provides the actual no-orphan guarantee.

### Why `cmd.Cancel` is the right chokepoint

`runAgentRun` builds `ctx` via `signal.NotifyContext(ctx, SIGTERM, SIGINT)`
(`cmd/pyry/agent_run.go`) and threads it to `exec.CommandContext`. os/exec's default ctx-cancel is an
immediate `Kill()` (SIGKILL) of claude — which both skips any reap and denies claude grace to flush.
Overriding `cmd.Cancel` restores the graceful path **and** gives the one teardown moment where claude
and its whole descendant tree are guaranteed alive and not-yet-signalled, so the `ps` snapshot sees
the live tree — the walk is **race-free**, not a hopeful defer-time scan. `cmd.Cancel` fires **only**
on operator ctx-cancel; normal completion, budget-hit, and watchdog-fire don't cancel the parent ctx,
so those paths are byte-for-byte unchanged (minimal blast radius).

## Patterns established

- **Reap at `cmd.Cancel`, the one race-free teardown moment.** Overriding `exec.Cmd.Cancel` (vs a
  deferred best-effort scan) is the chokepoint: it fires once, while the whole child tree is alive and
  unsignalled. Reuse for any future "walk the live child tree before tearing it down" need.
- **Content-blind process enumeration via a single `ps` snapshot.** `ps -axo pid=,ppid=,pgid=` is one
  portable command across Linux + macOS (no `//go:build` split, no cgo, no new dep), consistent with
  the codebase's shell-out precedent (`internal/sessions/rotation/probe_darwin.go` shells to `lsof`;
  the e2e shells to `ps`/`pgrep`). Reads pid/ppid/pgid only — never command lines — so nothing
  sensitive crosses into a log line.
- **The three guards are load-bearing safety, and the unit test asserts each.** `pgid<=1`,
  `pgid==Getpgrp()`, `pgid==rootPid` are the difference between "kill the orphan" and "SIGKILL pyry or
  init." A process-group reaper must explicitly test the self-group and root-group exclusions.
- **`cmd.WaitDelay` is non-binding; the binding exit bound is tui-driver's `Session.Close` grace.**
  Two SIGKILL backstops follow the SIGTERM — `Session.Close`'s 3s grace and `cmd.WaitDelay` (5s). The
  3s fires first, so pyry exits well within the e2e's 5s window (~700 ms happy path). `WaitDelay` only
  has to stay ≥ the `Close` grace so os/exec doesn't preempt the graceful path. (Same spirit as
  [#332](332.md): "`WaitDelay` is the only knob that bounds SIGKILL latency; hard-code 5s.")

## Testing

- **Unit — `reap_test.go` (`TestReapDescendantGroups`, CI-runnable, `go test -race`).** Builds real
  process trees via new `helper_test.go` reap-helper modes, keyed by `GO_PTYRUNNER_REAP_MODE` so the
  claude-stand-in switch stays untouched: `leaf` (block), `parent_fresh` (spawn a `leaf` grandchild in
  a FRESH group via `SysProcAttr{Setpgid:true}`, report its pid, block), `parent_same` (same but SAME
  group). Four subtests, **sequential (no `t.Parallel`)** because two reap at `os.Getpid()` and would
  sweep a concurrent subtest's fresh-group helper:
  - `ReapsDescendantGroupSparesCaller` — fresh-group child reaped; same-group sibling + caller survive
    (suicide guard).
  - `NoDescendantsIsNoOp` — root with no children: nothing killed.
  - `ExcludesRootOwnGroup` — same-group grandchild spared (the `pgid==rootPid` guard; this is what
    leaves claude's own group to `sess.Close`).
  - `ReapsGrandchildGroupSparesRoot` — faithful `pyry→claude→zsh+tail` mirror: grandchild's fresh
    group reaped, root spared.
  - Liveness probed via `kill(pid,0)` / `kill(-pgid,0)` polled to a deadline
    (`processAlive`/`waitGroupGone`). Helpers background-`Wait` their grandchild so a SIGKILL'd group
    truly empties (see Lessons).
- **e2e — `sigterm_mid_tool_use_test.go` (AC #3/#4, build-tagged out of normal CI).** Invariant 1
  became **full-subtree cleanup**: a new `waitForGroupGone(bashPGID)` assertion (sibling of
  `waitForProcessGone`, probes `kill(-pgid,0)→ESRCH`) runs in the test body **before** the demoted
  self-reap `t.Cleanup`, so a production regression is caught, not masked. The old "out of scope /
  tracked by #565" carve-out doc-comment is removed; the self-reap kill is reframed as a
  belt-and-suspenders safety net. Invariants 2 (JSONL ends at a complete envelope boundary), 4a (Bash
  `tool_use` present), 4b (no matching `tool_result`) unchanged — the graceful SIGTERM makes invariant
  2 *more* robust (claude flushes before exit). Code-review ran the live-claude suite green:
  `ok internal/e2e/realclaude 146.340s`.
- **Review verdict: PASS**, no findings. (QA flagged a RED `make check` traced to a *pre-existing*
  flaky `.cast` filename collision in unrelated `TestRun_*` tests — shared `testSessionID` +
  wall-clock-second timestamp; filed as #576, not caused by this PR.)

## Lessons learned

- **A graceful SIGTERM does not make claude reap its own Bash subprocess.** Measured 3/3: claude given
  a graceful SIGTERM + grace exits cleanly but leaves the Bash group it deliberately detached
  orphaned. The cheap fix (mirror streamrunner's grace) is necessary for a clean JSONL flush but
  **insufficient** for reaping. Don't assume a child reaps grandchildren it intentionally isolated
  into a new process group.
- **Same shape as the 2026-05-23 "`detached:true` creates a teardown contract" lesson.** Isolating a
  subprocess from a shared context (claude putting every Bash command in its own group) obligates
  migrating every teardown that depended on that context. The pre-#563 e2e orphan check used
  `pgrep -g <pyry-pgid>`, which assumed the Bash subprocess lived in pyry's group — it never has, so
  the check scanned an always-empty group and could never observe this leak.
- **A SIGKILL'd child you spawned becomes a zombie that still answers `kill(-pgid,0)`.** The
  reap-helpers must background-`Wait` their grandchild, else a "group gone" liveness probe sees the
  zombie and the test wedges. Worth knowing for any process-liveness test that polls a group it just
  killed.

## Out of scope / known same-shape gaps

- **Budget-hit and watchdog-fire teardown paths retain the same structural descendant leak.** They
  tear claude down without cancelling the parent ctx (budget `Terminate` signals claude directly;
  watchdog cancels only `runCtx`), so `cmd.Cancel` does not fire. Out of scope per the AC (operator
  SIGTERM only) and *evidence-based fix selection* (unobserved). If a future ticket observes a
  budget/watchdog-time orphan, the same `reapDescendantGroups(cmd.Process.Pid, logger)` call drops
  into the budget `Terminate` hook (`runner.go`) — a one-line addition reusing this helper. A **known
  gap, not a regression.**
- **The streamrunner (stream-json) path is unverified.** It already grants claude a SIGTERM grace,
  which may already avoid the leak; not in scope here. This fix is ptyrunner-only.
- **`ps` availability.** Chosen for portability; a `/proc` Linux fast path isn't worth the platform
  split for a once-per-shutdown call. Revisit only if a hardened-PATH environment makes `ps`
  unavailable.
