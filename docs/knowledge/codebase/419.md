# Ticket #419 — UTF-8 prompt fidelity round-trip test

A sibling to `prompt_fidelity_test.go` (#364) that extends the ASCII-only round-trip pin to non-ASCII UTF-8. Closes the gap left by [#364](364.md), which deliberately picked an ASCII literal so the substring search would be JSON-escape-safe — leaving JSON escape handling, JSON re-encoding, and buffer slicing on UTF-8 byte boundaries unpinned. A regression on any of those would land silently for Finnish-speaking users; this test makes it loud.

## Implementation

- **`internal/e2e/realclaude/prompt_fidelity_unicode_test.go` — new file**, build tag `//go:build e2e_realclaude`, package `realclaude`, single exported test `TestRealClaude_PromptFidelity_Unicode`. ~75 LoC. Structurally a 1:1 clone of `prompt_fidelity_test.go`'s body — same `WithWorktree` → `RunPyryAgentRun` → `ReadJSONL` composition, same `RunOpts` (`Model: "claude-haiku-4-5"`, `MaxTurns: 1`, `Effort: "low"`, `AllowedTools: ["Read"]`), same `SystemPrompt` shape, same three early-return failure modes, same `bytes.Contains(e.Raw, []byte(literal))` assertion against the first `user`-kinded JSONL entry, same trailing "no user entry found" diagnostic. The ONLY substantive differences are the prompt literal and the test function name.
- **Prompt literal — `unicodePrompt`.** Single Go string written with non-ASCII characters literal in source (source file is UTF-8 by Go default; no `\uXXXX` escapes — the point is to assert the bytes that were *typed* survive end-to-end, not a normalized form). Three byte-width classes are present to canary the boundary cases:
  - Finnish `ä` / `ö` / `å` (U+00E4 / U+00F6 / U+00E5 — 2-byte UTF-8)
  - em-dash `—` (U+2014 — 3-byte UTF-8)
  - `🐍` (U+1F40D — 4-byte UTF-8, outside the BMP; this is the byte-slicing canary)

  Wrapped in a distinctive ASCII frame mirroring [#364](364.md)'s naming convention: `INTEGRATION_TEST_PROMPT_UNICODEFIDELITY_K3M9P2X7 … INTEGRATION_TEST_PROMPT_UNICODEFIDELITY_END_R5T8V1Z4`. The frame doubles as a sanity bound — if the prompt is dropped entirely the frame match also fails, distinguishing "prompt missing" from "prompt corrupted on a UTF-8 boundary" via the failure-message contents.
- **Single `bytes.Contains` call against the full framed literal** is the contract — the AC's "frame + payload" diagnostic shape is satisfied by what the failure message prints (the literal and the raw event), not by splitting into multiple assertions. Resist factoring into "frame check" + "payload check".
- **Reuses `jsonlPathFor` directly** from `prompt_fidelity_test.go` (unexported, package-scoped helper, same package + same build tag). No duplication.
- **No fixture-helper changes.** `WithWorktree` / `RunPyryAgentRun` / `ReadJSONL` from `fixtures.go` were sufficient as-is; the spec confirmed `RunOpts` already exposes every field needed. The Makefile target `make e2e-realclaude` runs `go test -tags e2e_realclaude ./internal/e2e/realclaude/...`, so the new `*_test.go` is picked up automatically.

## Patterns established

- **Sibling-file expansion for prompt-handoff coverage.** The spec offered both a sibling-test-in-same-file option and a new-file option; picked the new file because it isolates the diff, keeps `git blame` on the ASCII test clean, and lets the two literals stay greppable and visually parallel (both named `INTEGRATION_TEST_PROMPT_<KIND>FIDELITY_<suffix>`). The same shape is the right play for any future prompt-fidelity dimension (control characters, very-long prompts, mixed encodings if a binary path ever opens): one file per dimension, single-assertion body, shared helpers.
- **Source-literal bytes, not normalized form.** The prompt is written with the characters literal in source (not `\uXXXX`) and is sent without any NFC/NFD massaging. The assertion is `bytes.Contains` on the on-disk JSONL — byte-identical preservation, not "decodes to the same code-point sequence". This is the right contract because the regression class (JSON escape / re-encoding / buffer slicing on multi-byte boundaries) only surfaces against a byte-level assertion. Normalization-form differences are out of scope (called out in the ticket body).
- **One `bytes.Contains` against the framed literal, not split assertions.** The diagnostic shape "prompt missing vs. prompt corrupted" is communicated by the failure message printing the literal + raw event, not by structuring two assertions. Splitting would invite future maintainers to weaken individual checks; keeping it one call preserves the byte-strict contract.

## Lessons learned

- **The JSON-escape risk is the test's whole point — don't weaken on first red.** Spec called this out explicitly and worth restating here: if `claude`'s JSONL writer ever escaped non-ASCII as `ä` / `—` / surrogate pairs for `🐍` instead of preserving raw UTF-8 bytes, `bytes.Contains` would fail. The right response is to surface the finding on the ticket and stop — NOT to change the assertion to a decoded-payload check, NOT to add a fallback. The byte-strict contract is the deliverable. (No regression observed on the developer's run; the assertion passes against `claude` today.)
- **The frame-doubles-as-sanity-bound property is already encoded in the failure message, not in the assertion shape.** First instinct was to express "prompt missing vs. corrupted" as two separate assertions (`Contains(frame_open)` then `Contains(framed_payload)`). Spec correctly identified that the diagnostic difference comes from the failure message printing the framed literal and the raw event — a reader can see which substring is in the raw bytes and which isn't. Single-call assertion is structurally cleaner and matches `prompt_fidelity_test.go` 1:1.

## Out of scope (deferred)

- **Assistant-side reply fidelity.** Model output is variance-prone; asserting on the assistant's `text` block would make this test flaky. Pinned by the ticket body.
- **Unicode normalization forms (NFC vs NFD).** The literal is byte-identical, not normalized; the test asserts the bytes that were sent, not a normalized form. If a future regression involves the streamrunner introducing normalization, that's a separate test.
- **Other non-ASCII alphabets / scripts.** Finnish + em-dash + emoji canaries cover 2/3/4-byte UTF-8 widths; broader script coverage (RTL, combining marks, ZWJ sequences) is not in scope.
