# Ticket #282 — `devices`: extend `Device` with `Platform` and `PushToken`

Phase 3 Track C — schema-only slice for mobile push notifications. Adds two optional `string` fields to `internal/devices.Device` so the binary can persist per-device push registration state (`platform`, opaque APNs/FCM token) in `~/.pyry/<name>/devices.json`. Schema half of #250 (the `register_push_token` handler that writes these fields lands separately); on-disk counterpart of the wire-side `protocol.RegisterPushTokenPayload` from #275.

## Implementation

- **Two appended fields, both `string` with `omitempty`.** `Platform string json:"platform,omitempty"` + `PushToken string json:"push_token,omitempty"` after `LastSeenAt` in `internal/devices/device.go:24`. Fields are appended — never reorder — so the on-disk byte ordering for pre-existing key sets stays unchanged.
- **`omitempty` is load-bearing.** Without it, an empty string serialises as `"key": ""` and a pre-#282 `devices.json` round-tripped through load → save would sprout `"platform": ""` / `"push_token": ""` entries. With `omitempty`, the zero value (the state of every device created by the existing pairing flow) is suppressed entirely — zero-migration change.
- **`string` not `*string`.** Optional strings could be modelled as `*string` to distinguish "unset" from "empty", but the push registration path treats both as "no token registered". `nil`-vs-`""` would force every reader to nil-check before use; `string + omitempty` is the idiomatic Go shape when the empty string is not itself a meaningful value, and matches `protocol.RegisterPushTokenPayload.Platform`.
- **Doc comment mirrors the wire contract verbatim.** `Platform`'s comment names `"fcm"` (Android) and `"apns"` (iOS) and explicitly cites `protocol.RegisterPushTokenPayload.Platform` as the authoritative contract source. Keeps on-disk and wire prose verbatim-aligned so a future drift between the two is a one-grep audit.
- **Three round-trip tests in `device_test.go`.** `TestDevice_LegacyOmitsPushFields` asserts that a zero-valued `Device` marshals to bytes containing neither `"platform"` nor `"push_token"` substrings, then round-trips back to byte-equal. `TestDevice_PopulatedRoundTrip` asserts a fully-populated `Device` (both fields set) round-trips through Marshal/Unmarshal preserving every field. `TestDevice_DecodeLegacyDiskShape` decodes a literal JSON fixture representing a pre-#282 disk entry (no `platform`/`push_token` keys at all) and asserts both new fields land at `""`. All three are `t.Parallel()`, stdlib only.
- **Nothing else changed.** `registry.go`, `auth.go`, `auth_test.go`, `registry_test.go`, and the pairing flow are untouched. `encoding/json` already tolerates unknown / missing keys (no `DisallowUnknownFields`, no custom unmarshaler in this package), so the `Load` path deserialises pre-existing files without intervention. The `Save` path serialises whatever fields `Device` exposes; `omitempty` does the rest. The pairing flow constructs `Device{TokenHash, Name, PairedAt, LastSeenAt}` and Go zero-initialises the two new fields to `""` — no call-site updates required.
- **Atomic write contract (A4 / #209) untouched.** Adding optional fields doesn't affect write-temp-then-rename behaviour or the `0o600` file mode.

## Out of scope (deferred)

- **The `register_push_token` dispatch handler (#250).** Reads `protocol.RegisterPushTokenPayload`, validates `Platform ∈ {"fcm", "apns"}` at the wire boundary, calls `Registry.Update`-or-equivalent to set `(Platform, PushToken)` on the matching device, persists. Pure write path on top of this schema — no further migration.
- **Validation on the on-disk struct.** Deliberately not added. Validation belongs at the wire boundary (#250); the persistence layer accepts whatever the handler writes. If a future handler bug writes `"android"` instead of `"fcm"`, the bug surfaces in APNs/FCM dispatch, not in the JSON decoder.
- **A schema version bump or migration step.** The whole point of `omitempty` here is to make this a zero-migration change. There is no schema version field on `devices.json` and this ticket does not introduce one.
- **APNs/FCM client integration.** Out of scope for the protocol slice entirely.
