# Ticket #719 — msgqueue: queue introspection + remove-by-id + change-notification seam

Additive API on the `internal/msgqueue` engine (#704): read a conversation's
queued backlog, remove a not-yet-delivered message by id, and learn when a
conversation's backlog changes. Phase 3 (EPIC #597, ADR 025), split from #705 —
the phone-facing `queue_state` / `dequeue_message` surface (the rest of #705's
split) needs all three. Ships **additive and unwired**, the same rhythm #704
shipped the engine unwired; `Enqueue`/`Run` signatures and the stable-id
semantics are untouched.

## Implementation

Two files: `internal/msgqueue/queue.go` (+142/−14) and a new
`internal/msgqueue/queue_introspect_test.go` (+386). The existing #704
`queue_test.go` is **not edited** (AC #4 is satisfied by leaving it green with
the new API present).

- **2 new exported types, 2 new methods, 1 new `Config` field:**
  - `QueuedMessage{ID uint64; Text string; TS time.Time}` — the engine-side
    projection of ADR 025's `{queued_msg_id, text, ts}` record. The doc comment
    flags `Text` as untrusted phone transit content (consumer must never log it,
    only surface it to the authorized conversation).
  - `ChangeFunc func(convID string)` — the injected change-notification seam,
    mirroring the `DeliverFunc` seam style. Carries only `convID` (edge-triggered;
    the consumer re-reads via `Snapshot`). Documented as MUST-NOT-block and
    MUST-be-concurrency-safe (it fires from the `Enqueue` caller, each drain
    goroutine, and the `Remove` caller). `nil` ⇒ disabled.
  - `Config.OnChange ChangeFunc` (nil ⇒ disabled). `New` stores it on a new
    `Queue.onChange` field; **no validation** — unlike `Deliver`, a nil seam is
    the supported default, not a wiring error.
  - `Snapshot(convID string) []QueuedMessage` — ordered copy of the
    not-yet-confirmed-delivered FIFO under the lock; unknown conversation ⇒ `nil`
    (empty), not an error. Freshly allocated value copies, so the caller cannot
    mutate engine state through it. Includes the in-flight head (an item stays in
    `items` until `advanceLocked` drops it on a confirmed commit), and does **not**
    flag which entry is in-flight — `Remove` returning `false` is that signal.
  - `Remove(convID string, id uint64) bool` — drops a queued, not-in-flight
    message by id, preserving surviving order; `true` iff removed. Unknown conv,
    unknown/already-delivered id, or the **in-flight (draining) head** ⇒ `false`
    no-op (no panic, no reorder). This is `dequeue_message`'s engine op; the
    in-flight-head no-op is what guarantees `dequeue_message` cannot cancel an
    in-flight delivery.
- **Change seam fires after unlock, three sites:** `Enqueue` (restructured from
  `defer q.mu.Unlock()` to explicit unlock + `q.notify`), the drain's
  delivery-advance (after `advanceLocked` drops a confirmed head — **not** on the
  empty-exit or delivery-error/retry paths), and a successful `Remove` (no-ops
  fire nothing). A tiny `notify(convID)` helper does the nil-check.
- **`shrinkLocked` extraction:** the trailing backing-array-hygiene `switch`
  (release at empty, compact when `cap > 2*len`) was lifted out of
  `advanceLocked` into `convQueue.shrinkLocked()`, now shared by `advanceLocked`
  (head drop) and `Remove` (mid-FIFO drop) — extracted, not duplicated.

## Patterns established

- **Snapshot-under-lock + copy-out** — `Snapshot` mirrors the outbound sibling
  `eventring.After`'s `out := make(...); copy/project` idiom: read accessors take
  `q.mu`, allocate a fresh slice, project value copies, return after unlock. The
  read surface never hands out a pointer into engine state.
- **Caller-supplied seams are never called under the lock.** `q.mu` is now never
  held across `onChange` *or* `deliver` — both are injected, may block or
  re-enter. Firing strictly after unlock is exactly what AC #3's "without holding
  the internal lock (no re-entrancy deadlock)" means: a re-entrant `OnChange`
  that calls back into `Snapshot`/`Remove`/`Enqueue` re-acquires `q.mu` cleanly.
- **`draining`-under-the-same-lock makes the in-flight check atomic.** `Remove`'s
  "removable?" decision (`idx == 0 && c.draining`) is read under the same `q.mu`
  the drain uses to set/clear `draining` and to peek/`advanceLocked`, so the
  check-and-splice is atomic w.r.t. the drain — no new lock, no TOCTOU. This is
  the load-bearing concurrency argument the whole slice hangs on.
- **Code review NIT (non-blocking, no change made):** `Remove`'s slice-delete
  idiom `append(c.items[:idx], c.items[idx+1:]...)` leaves a stale `queued` value
  (holding the untrusted `text`) in the backing array beyond the new `len`,
  retained until `shrinkLocked` compacts or the slot is overwritten — exactly
  mirroring the existing `advanceLocked`/`items[1:]` precedent the spec blessed
  reusing. Zeroing the freed tail would be a belt-and-suspenders nicety, not a
  correctness fix, and would be inconsistent unless `advanceLocked` changed in
  lockstep. Left as-is for consistency.

## Lessons learned

- **The in-flight-head no-op is the load-bearing rule.** `advanceLocked` blindly
  drops index 0 (`items[1:]`), so `Remove` keys its "removable?" decision on
  `i == 0 && c.draining` read under the same `q.mu` the drain mutates `draining`
  under — making the check atomic w.r.t. the drain. Non-head removal is
  unconditionally safe because the drain holds a value copy of the head and only
  ever touches index 0.
- **Firing `OnChange` strictly after unlock** is what makes a re-entrant callback
  (consumer calling back into `Snapshot`/`Remove`) deadlock-free against the
  goroutine that fired it; the `FiresWithoutHoldingLock` test re-enters both from
  inside the callback to prove it.

## Out of scope / deferred (named, not built)

- **No wire types, no consumers wired.** `queue_state` / `dequeue_message` wire
  shapes + the handler that binds `convID` to the requesting phone's authorized
  conversation are #705. Cross-conversation confidentiality (`Snapshot` reading
  another conv's text) and integrity (`Remove` mutating another conv's FIFO) are
  prevented **only** by that consumer-side convID binding — the engine has no
  session context, exactly as #704 left it.
- **`Remove` reason granularity.** The collapsed `bool` (true = removed, false =
  any no-op) is the simplest contract that satisfies AC #2. A richer return
  distinguishing "in-flight, retry" from "already gone" is deferred on YAGNI.
- **Snapshot in-flight flag.** Not exposed; `Remove`'s `false` is the
  discriminator. An `InFlight bool` on `QueuedMessage` (index 0 && draining) is a
  one-field follow-on if #705 wants the phone to grey-out the head proactively.
- **Inbound bound / backpressure** (`MaxQueuedPerConversation` / `ErrQueueFull`)
  stays deferred to the wiring slice, as #704 pre-specified; this slice adds no
  new unbounded surface (`Snapshot` is read-only).
