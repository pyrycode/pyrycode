# Ticket #701 — modal/permission v2 wire types + `modal_id` nonce + `answer_token` idempotency (SSOT)

The wire **vocabulary** for a modal surfaced over the encrypted mobile wire
([EPIC #597] Phase 3, [ADR 025]). When the supervised `claude` raises a modal —
a permission prompt, a plan-approval, a tool-confirmation — the daemon describes
it to the phone, the phone answers, and the daemon drives that answer back into
`claude`. None of that vocabulary existed on the v2 wire yet. This slice adds the
four modal envelope types + their payload shapes + the documented `modal_id`
nonce / `answer_token` idempotency contract, so the runtime siblings ([#703]
control loop, [#706] two-heads ownership, [#702] per-device answer gate) all build
on one stable, security-reviewed contract instead of each inventing modal shapes
ad hoc.

**Wire shape only** — the four type constants, five payload structs, the
v2-partition classification, four round-trip tests, and the
`docs/protocol-mobile.md` SSOT. **No runtime ships here.** The minting of
`modal_id` nonces, the dedup of answers by `answer_token`, the inbound-answer
validation, and the fan-out gate all live in the producers ([#703]/[#706]/[#702]).
This is the exact split shape of [#656] (`session_transition` vocab) → [#657]
(producer), scaled from one boundary marker to four modal envelopes.

**`security-sensitive`: yes** — but **no handler ships**. The label rides on the
*shape* review, not on code: a vocab-only slice's adversarial question is not
"does this code introduce a vuln" (no code runs) but **"does the shape make a
secure runtime expressible, and does it foreclose a secure runtime anywhere?"**
The architect security pass (verdict **PASS**) blessed `modal_id` opaqueness,
cross-conversation-confusion foreclosure, and `answer_token` sufficiency, and
surfaced five producer obligations (see § Patterns established). Contrast [#700]
(the internal `PermissionRequest`/`PermissionResponse` seam, **unlabelled**):
that slice added no gate/route/trust-boundary parse, so its label rode downstream;
this slice earns the label because it defines the **new inbound (phone→daemon)
control surface** whose shape every later gate depends on.

## Implementation

Two production files, both additive. **4 new constants** (`TypeModalShown`,
`TypeModalAnswer`, `TypeModalCancel`, `TypeModalDismissed`), **5 new exported
types** (`ModalOption` + the four payloads) — at the architect's `≤5 exported
types` S ceiling, not over. **0 construction sites changed** — nothing fans out,
imports, or dispatches it yet.

The five payload structs (all in `messaging.go`):

```go
type ModalOption struct { // a single ordered choice
    ID    string `json:"id"`
    Label string `json:"label"`
}
type ModalShownPayload struct { // binary → phone
    ModalID         string        `json:"modal_id"`
    Class           string        `json:"class"`
    Title           string        `json:"title"`
    Prompt          string        `json:"prompt"`
    Options         []ModalOption `json:"options"`           // ORDERED — array order is display/selection order
    DefaultOptionID string        `json:"default_option_id"` // MUST equal one of Options[].ID (documented invariant)
}
type ModalAnswerPayload struct { // phone → binary, inbound control
    ModalID     string `json:"modal_id"`
    OptionID    string `json:"option_id"`
    AnswerToken string `json:"answer_token"` // client-minted idempotency key
}
type ModalCancelPayload struct { // phone → binary, inbound control
    ModalID string `json:"modal_id"`
}
type ModalDismissedPayload struct { // binary → phone
    ModalID string `json:"modal_id"`
    Outcome string `json:"outcome"` // selected option id, or producer-defined cancel/timeout sentinel
    Source  string `json:"source"`  // CLOSED set {remote, local, timeout}
}
```

Test files (excluded from the production count): `messaging_test.go` (four
**flat, one-func-per-type** round-trips — `TestModalShownPayload_RoundTrip`
asserts `len(Options)==2` and positional ids for the ordering invariant +
`DefaultOptionID`; `TestModalAnswerPayload_RoundTrip` asserts `AnswerToken`
round-trips per the AC; `TestModalDismissedPayload_RoundTrip` asserts
`Source=="remote"`), and four single-line fixtures under `testdata/` authored in
**struct-field order** (`modal_shown.json`, `modal_answer.json`,
`modal_cancel.json`, `modal_dismissed.json`).

## Patterns established

- **A vocab-only slice earns `security-sensitive` for the *shape* review, not for
  code.** No handler ships, but this slice defines the **new inbound (phone→daemon)
  control surface** (`modal_answer`/`modal_cancel`) and the contract for a
  high-consequence action — answering a modal injects a decision (e.g. "Allow
  Bash `rm -rf`") into `claude`. The security pass asks "does the shape make a
  secure runtime expressible / does it foreclose one," not "does this code
  introduce a vuln." Verdict **PASS**; the label stays so the pass happens before
  #703 builds minting/dedup on it. (Contrast [#700], unlabelled — internal value
  types with no inbound parse or gate; the label rode downstream there.)
- **`modal_id` as the *sole* correlation key forecloses cross-conversation
  confusion structurally.** The payloads carry **no `conversation_id`** — only
  `modal_id`. The daemon resolves `modal_id` against its **own** outstanding-modal
  state (the resync/screen_snapshot "daemon's own resolved id, never
  attacker-derived" discipline) and maps `option_id` against its own recorded
  option list. A phone can never assert *what* it is answering — only *which*
  modal and *which* of the daemon's offered options. A shape carrying *both*
  `conversation_id` and `modal_id` would admit a `{conversation_id: A,
  modal_id: <B's modal>}` disagreement the daemon must adjudicate; the single-key
  shape eliminates that class. (Justified daemon-side because one supervised
  `claude` has a single `CurrentConversation()` cursor and all paired devices are
  the one operator's — see [#632] § per-conversation confinement. Multi-active
  modals, if they ever arrive, revisit `conversation_id` in #703, not here.)
- **`answer_token` is an idempotency key, not a credential — validity gate vs
  dedup key are separated.** Its security-relevant properties are *uniqueness*
  (distinct logical answers → distinct tokens) and *stability* (a retry → the same
  token), **not secrecy**. It lets #703 collapse a replayed/reordered
  `modal_answer` to a no-op via `(modal_id, answer_token)`. It is **not** the
  authorization: authorization is `modal_id` validity ([#706]) + the per-device
  gate ([#702]). A guessed/forged token grants nothing — a fresh token on a valid
  `modal_id` is a (rejected, because already resolved/gated) answer; a replayed
  token is a no-op. The shape (a distinct opaque `string`, separate from
  `modal_id` and `option_id`) keeps the two concerns un-conflatable.
- **A mixed inbound+outbound cluster lives in one const block with one rationale
  comment.** The four modal types span both v2-only flavours (outbound events +
  inbound control) yet share one block + one comment, mirroring the
  `request_snapshot`/`screen_snapshot` precedent. `TestTypeConstants_V1V2Partition`
  is grouping-independent, so block membership is a pure readability choice; the
  deterministic gate is the partition test, not the grouping.
- **One round-trip func per type, not table-driven.** The four shapes differ too
  much for a single table (the `session_transition` two-fixture table was
  borderline; four distinct shapes tip it). Each is ~12 lines on the shared
  `roundTripEnvelope` helper. The byte-equal re-marshal is the regression detector
  for the no-`omitempty` discipline.
- **Closed-set discipline is applied selectively.** Only `source` is pinned to a
  closed set `{remote, local, timeout}` here — it is fully determined by the
  resolution *mechanism* (remote answer / local TTY / deny-on-timeout), not by
  what modals `claude` surfaces. `class` and `outcome` stay plain strings whose
  exhaustive vocabularies the producer (#703) owns (documented, not enforced) —
  the `MessagePayload.Role` / `SessionTransitionPayload.Reason` leaf-data
  convention; `internal/protocol` is stdlib-only with no named Go enums.

### Producer obligations surfaced (carry into [#703]/[#706]/[#702])

The security pass named five obligations the *runtime* must honour — none is a
wire-shape gap; the contract here supports all five:

1. Mint `modal_id` from `crypto/rand`, **globally unique across concurrently-
   outstanding modals** (not merely unique per conversation — else single-key
   resolution is ambiguous). The `conversations.NewID()` UUIDv4 precedent (an
   unguessable 122-bit nonce) is acceptable. Unguessability is a *second*
   independent barrier behind #702's gate: a device shown one modal still cannot
   blind-answer a *different* outstanding one. (#703)
2. Reject inbound `modal_answer`/`modal_cancel` whose `modal_id` ≠ the current
   outstanding one — **first-answer-wins**. (#706)
3. Dedup by `(modal_id, answer_token)`. (#703)
4. Per-device answer gate, **default OFF** — not a wire capability. (#702)
5. **Never log modal body text** (`prompt`/`title`/`label`) — the [#632]/[#589]
   logging discipline; log only `modal_id`, `class`, `source`, `option_id`,
   `conn_id`, `err`. (#703)

## Lessons learned

- **No lessons — a clean #656-template scale-out.** The PR flagged none: every
  addition mirrored an existing precedent in this exact package (the const-block
  shape, the no-`omitempty` rule, the partition lockstep, the struct-field-order
  fixtures). The only nuance worth recording counters the general
  [[pyrycode-gofmt-dirty-at-head-go1.26]] memory: **`internal/protocol` is
  gofmt-clean at HEAD under local Go 1.26.2** — no spurious dirtiness in this
  package, so no `git show HEAD:<f> | gofmt -l` triage was needed for the files
  touched here. (The dirtiness is real elsewhere in the tree; it does not affect
  `internal/protocol`.)
- **Code review left no comment** (PR [#711]); no MUST/SHOULD/NIT findings shaped
  the final shape. `go test -race ./...`, `go vet ./...`, `gofmt`, and
  `go build ./cmd/pyry` all green.

## Out of scope (deferred to siblings)

- **The modal control loop — [#703]** (`security-sensitive`): surface (mint
  `modal_id` + emit `modal_shown`), route inbound answer (validate vs current
  `modal_id` → tui-driver keystroke), dedup by `answer_token`, deny-on-timeout.
  Adds the `case protocol.TypeModalAnswer` / `case protocol.TypeModalCancel` arms
  to `v2session.go`'s `dispatchAppFrame`. This slice only defines the wire shape;
  nothing emits, intercepts, or gates it.
- **Two-heads ownership — [#706]**: stale `modal_id` rejected, first-answer-wins.
- **The remote-permission answer gate — [#702]** (per-device, default OFF): the
  gate is in the security model, **not** a wire capability.
- **No `conversation_id` field** (open question): `modal_id` is the sole
  correlation key today; if multi-active-conversation modals arrive, `modal_shown`
  may need `conversation_id` for client-side routing — revisit in #703.
- **No deadline/timeout field on `modal_shown`** (open question): deny-on-timeout
  is #703's runtime; the wire carries no countdown today. If the phone needs to
  render one, that's a follow-up field on `modal_shown` — flagged to #703.
- **The mobile consumer** — the phone-side render/answer UI (pyrycode-mobile).
