# Ticket #303 — net: `list_conversations` handler + dispatcher route

First per-verb handler plugged into the #307 dispatcher seam. Decodes a
`list_conversations` request, reads `internal/conversations.Registry`,
projects each row into a `protocol.ConversationSummary`, and replies via
`dispatch.Conn.Reply` so the dispatcher stamps `id` / `ts` /
`in_reply_to`. Read-only — no registry mutation, no error envelope,
~60 LOC + tests.

## Implementation

### New file `internal/relay/handlers/list_conversations.go`

- **`ConversationLister`** — single-method interface declared at the
  handler call site (`List(filter ...conversations.ListFilter)
  []conversations.Conversation`). `*conversations.Registry` satisfies
  it structurally; no new export from `internal/conversations`, no
  global state, no adapter. Variadic matches `Registry.List` exactly so
  Go's structural matching holds.
- **`ListConversations(reg ConversationLister) dispatch.Handler`** —
  returns a closure over `reg`. Steps inside the closure:
  1. `reg.List()` (no filter — every conversation is in the response
     regardless of `IsPromoted`).
  2. `sort.SliceStable` by `LastUsedAt` ascending, ties broken by `ID`
     ascending. Mirrors `Registry.Save`'s on-disk sort
     (`registry.go:72-83`) byte-for-byte so the wire order equals the
     serialized order.
  3. Project each `Conversation` into `protocol.ConversationSummary`
     via direct field copy (`ID`, `Name *string`, `IsPromoted`, `Cwd`,
     `LastUsedAt`).
  4. `json.Marshal(ConversationsPayload{...})` and return
     `c.Reply(ctx, env, protocol.TypeConversations, payloadJSON)`.

The handler does NOT unmarshal `env.Payload` —
`ListConversationsPayload` is `struct{}` by spec, so there is nothing
to validate beyond what the dispatcher's `IsV1Compatible` already did.
The handler does NOT stamp `ID` / `TS` / `InReplyTo` itself; that's
`Conn.Reply`'s job (#307).

### `LastMessageTS` collapse

`Conversation` carries no separate "last message ts" field today; the
registry only tracks `LastUsedAt`. The wire spec example at
`docs/protocol-mobile.md:338-376` shows both timestamps equal on each
row, so the projection sets `LastMessageTS = LastUsedAt`. A comment at
the projection site names the collapse so a future reader does not
"fix" it by leaving `LastMessageTS` zero. When a distinct
`LastMessageTS` lands on the registry side (deferred), the projection
updates in one place.

### Daemon wiring (`cmd/pyry/relay.go` + `cmd/pyry/main.go`)

Two surgical edits:

1. **`startRelay` signature** grows one trailing parameter:
   `convReg *conversations.Registry`. Inside, after `dispatch.New(...)`
   and **before** the `go d.Run(ctx)` goroutine launches, the new line:

   ```go
   d.Register(protocol.TypeListConversations, handlers.ListConversations(convReg))
   ```

   `Register`-before-`Run` is required (#307: the handler map is read
   lock-free by per-conn goroutines; late `Register` panics by design).
   When `relayURL == ""` the relay-disabled early return still fires
   and `convReg` is accepted but unused — uniform signature, no
   conditional plumbing.
2. **`runSupervisor`'s single call site** to `startRelay` threads the
   existing `convReg` variable through. No load reorder — `convReg` was
   already loaded earlier in `runSupervisor` for the sweep loop wiring
   (#243).

### Test seam (`internal/relay/handlers/list_conversations_test.go`)

Uses the real dispatcher as the seam, mirroring
`internal/dispatch/dispatch_test.go`'s pattern. Each test:

- Builds a `*conversations.Registry` directly (`&conversations.Registry{}`
  + `reg.Create(...)`) — no file I/O.
- Constructs `dispatch.New` with a buffered `chan
  protocol.RoutingEnvelope`, registers the handler,
  `go d.Run(ctx)`, pushes one `list_conversations` frame on the input
  channel, reads the response from `d.Outbound()`, decodes and asserts.
- Tears down by `cancel()` + a 2s timeout on `d.Run`'s return.

Coverage:

- **Empty registry** — `conversations` slice serialises as `[]`, not
  `null` (the `make([]ConversationSummary, 0, len(list))` shape is
  load-bearing).
- **Single conversation** — non-nil `Name`, non-zero `LastUsedAt`;
  projection round-trips field-for-field; timestamps compared via
  `time.Time.Equal` per the `time.Time` round-trip discipline.
- **Multiple conversations, deterministic order** — three records
  seeded in reverse-sort order with one `LastUsedAt` tie; response is
  in `LastUsedAt` ascending, tie broken by `ID` ascending.
- **`in_reply_to` correctness across two consecutive requests** on the
  same conn — second reply's `in_reply_to` matches the second
  request's `id` (not the first), and `id` increments monotonically
  from the per-conn counter (response 1 carries `id=1`, response 2
  carries `id=2`).

No time-injection seam, no `time.Sleep`, no goroutine race. All tests
pass under `go test -race ./...`.

## Patterns established / reinforced

- **Minimal-surface interface at the consumer.** The handler defines
  `ConversationLister` itself rather than exporting it from
  `internal/conversations` or accepting `*Registry` directly. Future
  per-verb handlers in `internal/relay/handlers/` declare their own
  read-only narrow interfaces against the same registry — keeps the
  handler unit-testable with hand-rolled fakes when registry
  construction is heavy, and keeps `internal/conversations`'
  public surface unchanged.
- **Wire ordering invariant lives at the handler.** The handler's
  `sort.SliceStable` repeats the exact comparator from
  `Registry.Save`. The duplication is deliberate: the registry's sort
  is its on-disk serialization contract; the handler's sort is its
  wire-order contract. If a future change splits the two (e.g. wire
  spec asks for descending order), only the handler shifts.
- **Wire-code mapping is not the handler's job.** `Registry.List`
  cannot fail; `json.Marshal` over stdlib types cannot realistically
  fail. The handler returns errors from `c.Reply` (ctx cancel /
  outbound closed) verbatim to the dispatcher, which logs WARN. No
  `protocol.malformed` / `protocol.unsupported` mapping in handler
  code — reinforces the rule pinned in `docs/PROJECT-MEMORY.md:20`.
- **Append-a-parameter, don't share scaffolding.** Sibling slice #313
  (`send_message`) also appends a trailing parameter to `startRelay`.
  Architect's spec explicitly says: do NOT lift shared wiring between
  the two slices. Whichever lands second rebases textually on one
  line in `runSupervisor` and one line in `startRelay`'s signature.
  Premature DRY between coordinating PRs is the coordination point;
  the parameter list is the API.

## Lessons learned

### `make([]T, 0, n)` not `var out []T` when round-tripping through JSON

`json.Marshal` on a `nil` `[]ConversationSummary` produces `null`, not
`[]`. The wire contract for an empty conversation list is the empty
JSON array. The fix is to pre-allocate with `make([]T, 0, len(list))`
so the slice header is non-nil even when `len == 0`. The empty-registry
test pins this: it asserts the decoded JSON contains `"conversations":[]`
rather than `"conversations":null`. Trivial to write, easy to
regress — kept the explicit assertion in the test rather than relying
on byte-equivalence to a fixture.

### `LastMessageTS` is a known v1 collapse, not a bug

The protocol's `ConversationSummary` has two timestamp fields
(`LastMessageTS`, `LastUsedAt`) but `internal/conversations.Conversation`
has only one (`LastUsedAt`). The wire-spec example shows both equal,
so the projection sets `LastMessageTS = LastUsedAt`. The risk is a
future "tidy-up" PR that notices the duplication and "fixes" it by
leaving `LastMessageTS` zero — which would emit
`"last_message_ts":"0001-01-01T00:00:00Z"` to the phone. The
in-source comment at the projection site names the collapse and
points at where to update when a real `LastMessageTS` lands. Worth
one comment because the WHY is non-obvious from the surrounding code.

### Variadic in the interface must match the registry's variadic exactly

`ConversationLister.List(filter ...conversations.ListFilter)` matches
`Registry.List(filter ...conversations.ListFilter)` parameter-for-parameter,
including the variadic. Dropping the variadic on the interface (e.g.
`List() []conversations.Conversation`) would silently break structural
satisfaction at compile time — Go requires exact signature match for
interface implementation, variadic and all. The handler passes no
filter at the call site, but the type must carry the variadic to make
`*Registry` an implicit implementation.

## Out of scope (deferred)

- **Permission filtering by `IsPromoted`.** Ticket explicitly says
  "no filter"; every conversation is in the response. If a future v2
  introduces a "promoted-only" listing for free-tier phones, it grows
  a new envelope type, not a flag on `list_conversations`.
- **Streaming / paginated responses.** Single envelope reply. Real
  registries are bounded by the auto-archive sweep (#243) — every
  conversation older than 30 days idle and unpromoted is gone.
- **Per-handler DoS cap.** The transport layer's 1 MiB read limit is
  the inbound ceiling; outbound has no per-handler size cap.
  Architect's security review flagged this as "SHOULD FIX in a
  separate ticket if real registries reach four-figure sizes" —
  consistent with v1's overall posture and explicitly deferred per
  `internal/dispatch/dispatch.go:22-26`.
- **Distinct `LastMessageTS` field on `Conversation`.** Deferred until
  a real message-tracking ticket needs it (sibling slice #313's
  `send_message` may or may not surface the need, depending on
  ordering).
- **Error envelopes for the registry path.** `Registry.List` is
  no-fail by contract; nothing to map. If a `json.Marshal` failure
  ever occurred (it cannot today — all fields are stdlib types) the
  handler returns the wrapped error to the dispatcher rather than
  emitting `protocol.malformed`, because `malformed` would mislead
  the phone into thinking its request was the problem.
