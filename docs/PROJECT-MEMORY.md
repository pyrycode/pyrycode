# Project Memory — Pyrycode

Repo-level session memory. Read this at the start of every session.

## What's Built

### Codebase (Phase 0)
- **Supervisor core** — PTY spawn via `creack/pty`, raw-mode stdin/stdout bridging in foreground mode, Bridge-mediated I/O in service mode, exponential backoff restart with stability reset, `--continue` injection on restart for session persistence
- **SIGWINCH forwarding** — terminal resizes propagate from controlling terminal to child PTY (foreground mode only; attach mode locks the size at attach time)
- **Control plane** — Unix domain socket (`~/.pyry/<name>.sock`, 0600), line-delimited JSON protocol, verbs: `status`, `stop`, `logs`, `attach`
- **CLI transparency** — unknown args forward verbatim to claude; pyry's own flags use `-pyry-*` prefix; `-pyry-name` plus `PYRY_NAME` env var for named multi-instance
- **Graceful shutdown** — SIGINT/SIGTERM cancel the supervisor context, child is killed via `exec.CommandContext`, socket removed on exit
- **Service configs** — systemd user unit (`systemd/pyry.service`), macOS launchd plist (`launchd/dev.pyrycode.pyry.plist`)
- **~1700 source + ~1100 test Go lines** as of late Apr 2026, 10+ PRs merged

### Codebase (Phase 1.0, tickets #28 + #29)
- **`internal/sessions` package** — `SessionID` (UUIDv4 via `crypto/rand`, stdlib only), `Session` (wraps one `*supervisor.Supervisor` + optional `*supervisor.Bridge`), `Pool` (single-bootstrap registry with `RWMutex`-protected map). Sentinel errors `ErrSessionNotFound`, `ErrAttachUnavailable`. `Pool.Lookup("")` resolves to the bootstrap entry — the seam Phase 1.1's `Request.SessionID` plugs into.
- **Production consumers wired (#29)** — `cmd/pyry/main.go` constructs `*sessions.Pool` (with the supervisor.Config template inside `SessionConfig`); `internal/control` consumes a single `SessionResolver` interface (replaces Phase 0's `StateProvider` + `AttachProvider` pair). A 5-line `poolResolver` adapter in `cmd/pyry` bridges `Pool` → `SessionResolver` (covariant-return workaround). Wire protocol unchanged; `pyry status`/`stop`/`logs`/`attach` byte-identical to Phase 0. Foreground-mode attach error string preserved verbatim via `errors.Is(err, sessions.ErrAttachUnavailable)` mapping in `handleAttach`.
- See [knowledge/features/sessions-package.md](knowledge/features/sessions-package.md), [knowledge/features/control-plane.md](knowledge/features/control-plane.md), and [ADR 003](knowledge/decisions/003-session-addressable-runtime.md).

### Codebase (test infra, ticket #68)
- **`internal/e2e` harness — spawn + cleanup primitive** — build-tag-isolated package (`//go:build e2e`, invoked via `go test -tags=e2e ./internal/e2e/...`) that builds pyry once per test process (`sync.Once`; `PYRY_E2E_BIN` short-circuits for CI), spawns it in a `t.TempDir()` `$HOME` with `/bin/sleep infinity` as the supervised "claude" and `-pyry-idle-timeout=0`, polls the Unix socket via `os.Stat` + `net.Dial` (5s deadline, 50ms gap, doneCh short-circuit on early exit), and tears down via `t.Cleanup`: SIGTERM → 3s grace → SIGKILL → 1s grace → `os.Remove(socket)`. Three exported names: `Harness`, `Start(t)`, plus the struct fields (`SocketPath`, `HomeDir`, `PID`, `Stdout`, `Stderr`).
- **`HOME` redirection contains every ~-relative path with one env var** — pyry resolves the socket, registry, and claude sessions dir all under `os.UserHomeDir()`, so `t.TempDir()` as `$HOME` isolates the daemon completely. Belt-and-suspenders `-pyry-socket=<HomeDir>/pyry.sock` flag for the load-bearing path; no new `-pyry-registry` flag needed. `PYRY_NAME` is stripped from the child env so the operator's shell alias can't leak in.
- **Failure-injection verification via test-binary re-exec** — `t.Fatal` in a same-process `t.Run` subtest propagates to the parent and ends the test before it can inspect leak state. `TestHarness_NoLeakOnFatal` re-execs `os.Args[0] -test.run=^TestInnerFatalChild$` with `PYRY_E2E_INNER_FATAL_OUT=<state-file>` so the failing path runs in a fresh process; the child writes (pid, socket) before `t.Fatal`; the parent reads the state file and asserts `processAlive(pid)` is false (`os.FindProcess` + `Signal(syscall.Signal(0))`, the POSIX zero-signal probe) and the socket is `fs.ErrNotExist`.
- **No `Option`s yet, no CLI-driver wrappers** — `Start(t)` is the only constructor; AC's "or equivalent" parenthetical covers the fail-fast `t.Fatalf` shape. `Harness.Status()`/`Stop()`/`Attach()`/`Run(args...)`, options, the first feature e2e, and CI wiring are deferred to the #51 follow-up. Build-tag isolation means existing `go test ./...` CI keeps passing untouched.
- See [knowledge/features/e2e-harness.md](knowledge/features/e2e-harness.md).

### Codebase (test infra, ticket #81)
- **`internal/e2e/install_darwin_test.go` — `pyry install-service` × launchd `gui/<uid>` round-trip on macOS.** Three tests under `//go:build darwin && e2e_install`, mirroring #80's Linux shape: `TestE2EInstall_RoundTrip_macOS` writes the plist via `install.Install` → `launchctl bootstrap gui/<uid>` → `waitForRunning` (polls `launchctl print` for `state = running`, 10s deadline, 100ms gap) → `pyry status -pyry-name=<name>` (asserts exit 0 + `Phase:` in stdout) → `launchctl bootout` → `waitForUnloaded` → cleanup; `TestE2EInstall_PathInheritance_macOS` extracts `EnvironmentVariables.PATH` via `plutil -extract ... raw` and asserts each non-empty `$PATH` entry appears verbatim — note **no `$HOME/` → `%h/` substitution** (`derivePathEnv` gates the rewrite on `PlatformSystemd`); `TestE2EInstall_CleanupOnFatal_macOS` re-execs `TestInstallFatalChild` (same name as the Linux file, non-overlapping build tags) which installs + bootstraps + `t.Fatal`s, parent verifies `stat(plistPath)` is `ErrNotExist` and `launchctl print gui/<uid>/<label>` exits non-zero.
- **Same `e2e_install` build tag as the Linux sibling.** Single `go test -tags=e2e_install ./internal/e2e/...` invocation covers both platforms in CI when intended. Default `go test ./...` compiles neither file.
- **Plist parsing via Apple's `plutil`, not Go's `encoding/xml`.** The plist's key/value alternation pattern is awkward to walk with the stdlib XML decoder. `plutil -extract EnvironmentVariables.PATH raw -o - <plist>` returns just the PATH string — one shell-out, fails loudly on missing keys, no XML parser. `plutil` is part of `/usr/bin` on macOS; absence is a broken host, not a skip condition. ~30 lines of Go XML walking rejected as anti-simplicity.
- **`launchctl bootstrap` is async — poll `launchctl print` for `state = running`.** Bootstrap returns once launchd has *accepted* the request; the job transitions to `state = running` asynchronously. Polling loop is 100ms-gap, 10s-deadline; failure dumps the last `print` output for diagnosis. `launchctl print` is a debug command whose format Apple has reserved the right to reformat — `state = running` has been stable since macOS 10.10; if it ever breaks, the test fails loudly with a clear diagnostic.
- **Cannot isolate `$HOME` for the round-trip — same shape as Linux.** `launchctl bootstrap gui/<uid>` runs services in the user's GUI domain with `HOME` inherited from the GUI session, not from the test process. Round-trip and cleanup-on-fatal use the operator's real `~/Library/LaunchAgents/`; `PathInheritance` (no launchd dependency) keeps `t.TempDir()` isolation. Cleanup helper is idempotent best-effort (`bootout` + `os.Remove(plist)`) and additionally wipes runtime artefacts: `~/.pyry/<name>` registry dir, `<name>.sock` socket file, and the launchd template's hardcoded `/tmp/pyry.<name>.{out,err}.log` log files (unlike systemd which streams to journald).
- **Skip discipline.** `skipIfNoLaunchctl` (defensive — should never fire on macOS), `skipIfRoot` (`gui/<uid>` is non-root only; system-domain bootstrap requires root and is out of scope), `os.UserHomeDir()` failure → `t.Skip` (CI without HOME set).
- **Helper identifiers (`uniqueName`, `fatalNameEnv`, `fatalOutEnv`, `TestInstallFatalChild`) duplicated across the Linux and Darwin files with identical bodies.** The two files have non-overlapping build tags (`linux && e2e_install` vs `darwin && e2e_install`) so they never compile together — duplicate identifiers are legal and the simplest path. Names mirror exactly so a reader switching between files relies on muscle memory. Extracting into a third platform-neutral file deferred until a third platform appears.
- **Open question — headless macOS CI.** `gui/<uid>` requires a logged-in GUI session for that uid. GitHub `macos-latest` runners do have one; if not, the fallback is `launchctl bootstrap user/<uid>` (background-only domain). Defer until observed on real CI — AC specifies `gui/<uid>`.

### Codebase (test infra, ticket #69)
- **`Harness.Run(t, verb, args...) RunResult` CLI driver** — invokes the cached pyry binary with `<verb> -pyry-socket=<h.SocketPath> <args...>`, waits for it to exit (10s `context.WithTimeout`), returns `RunResult{ExitCode int, Stdout, Stderr []byte}`. Auto-injects the socket flag after the verb so callers don't thread it through; reuses `binPath` (set by `ensurePyryBuilt` under `sync.Once`) and `childEnv(h.HomeDir)`. Verb is positional because pyry dispatches subcommands on `os.Args[1]` — encoding that into the signature prevents the `h.Run(t, "-pyry-socket=other", "status")` footgun. Caller-side override is naturally available via Go `flag`'s "last value wins" rule, no special case in the harness.
- **Failure asymmetry: non-zero exit returned, exec/timeout fatal** — `*exec.ExitError` populates `RunResult.ExitCode` for the caller to assert on; non-`ExitError` exec failure (binary missing, fork error) and 10s deadline expiry both `t.Fatalf` because there's no useful test recovery. The 10s wrapper budget is comfortably above `pyry status`'s own 5s socket-dial timeout (`runStatus` → `control.Status`). No regression test for the deadline branch — constructing a daemon that hangs `pyry status` for >10s requires real-claude unresponsiveness or test-only socket injection, both more invasive than the safety net buys; per evidence-based fix selection, defer until observed.
- **Struct over tuple for the result type** — `RunResult` future-proofs for `Duration`/`Combined`/`OOMed` additions without call-site churn; named fields prevent the `[]byte` mix-up between stdout and stderr that a positional tuple invites. Cost is one type declaration; worth it.
- **`TestStatus_E2E` — first feature-flavoured e2e** — spawns the daemon, runs `pyry status`, asserts exit 0 + `bytes.Contains(r.Stdout, []byte("Phase:"))`. `"Phase:"` is the leading literal in `runStatus`'s output (`fmt.Printf("Phase:         %s\n", resp.Phase)`) and is stable across phase values, restart counts, and future field additions; asserting on the *value* would couple the test to claude-child startup timing, which is exactly what `/bin/sleep infinity` was chosen to avoid. `pyry version` was rejected as the proof-of-life verb because it short-circuits in `main.go` before parsing flags — wouldn't exercise the socket plumbing the harness sells.
- **Package doc updated with copy-pasteable CLI-driver example** — replaces the bare smoke pattern from #68; godoc now surfaces a `Start(t) → Run(t, "status") → assert` recipe that downstream session-verb tickets (#52, #54, #55, #56) can copy.
- See [knowledge/features/e2e-harness.md § CLI Driver](knowledge/features/e2e-harness.md).

### Codebase (test infra, ticket #52)
- **`internal/e2e/cli_verbs_test.go` — e2e coverage for the remaining shipped non-interactive verbs.** Four tests under `//go:build e2e`: `TestStop_E2E` spawns the daemon, runs `pyry stop`, asserts exit 0 + `bytes.Contains(r.Stdout, []byte("stop requested"))`, then bounded-polls (3s deadline, 50ms gap) until both `!processAlive(pid)` AND `os.Stat(sock)` returns `fs.ErrNotExist` — both conditions in the same iteration so the test doesn't race the supervisor's deferred socket cleanup. `TestStatus_E2E_Stopped` uses `RunBare("status", "-pyry-socket="+bogusSock)` against a fresh `t.TempDir()` non-existent path: exit != 0, non-empty stderr, no `panic` / `goroutine ` / `runtime/` substrings (clean error, not a crash). `TestLogs_E2E` runs `pyry logs` against a healthy daemon and asserts exit 0 + non-empty `bytes.TrimSpace(r.Stdout)` — the supervisor's in-memory ring captures startup lines, no specific wording asserted. `TestVersion_E2E` uses `RunBare("version")`: exit 0, output starts with literal `"pyry "` prefix, remaining token non-empty (`dev` in test builds, real version under `-ldflags`).
- **`RunBare(t, args...) RunResult` — daemon-free sibling of `Harness.Run`.** Same binary cache (`ensurePyryBuilt`), same `runTimeout` (10s), same exit-code mapping — but no daemon spawn, no auto-injected `-pyry-socket`, no `childEnv(h.HomeDir)` HOME redirection. Two motivated cases: verbs that don't touch the socket (`version` short-circuits in `main.go` before flag parsing), and negative tests against a known-bad socket path (avoids the spawn-then-stop-then-race-the-teardown ordering glue). The *only* harness API added in #52 — `Harness.Stop()` mid-test, typed `Status()`/`Logs()` wrappers, etc. were explicitly declined to keep the harness surface minimal.
- **Bogus-socket vs. spawn-then-stop for stopped-status.** Spawn-then-stop-then-status needs the test to wait for the daemon to actually shut down (otherwise status hits a still-listening socket and succeeds, defeating the test). That's the same poll loop as `TestStop_E2E`, plus ordering glue, plus a second `Run` call. Bogus-socket exercises the same code path (`net.Dial` fails → error surfaces clean to stderr → non-zero exit) without any timing dependency. Strictly simpler, strictly more deterministic.
- **Negative-assertion vocabulary for "clean error, not a crash".** Three conservative substrings — `panic`, `goroutine ` (Go's stack-trace header `goroutine N [state]:`), `runtime/` (Go runtime frames in tracebacks) — catch panics and stack traces without coupling to the exact dial-failure wording (today: `pyry: status: ... connect: no such file or directory`; allowed to evolve). Same pattern fits any "verify the failure is clean" assertion across future verbs.
- **File split: `cli_verbs_test.go` vs. appending to `harness_test.go`.** New file. `harness_test.go` is about *harness behaviour* (smoke, no-leak-on-fatal, the canonical `TestStatus_E2E` proof-of-life); `cli_verbs_test.go` is about *CLI surface coverage*. `processAlive` stays in `harness_test.go` and is reused via package scope (no duplication, no move). Build tag identical: `//go:build e2e` (not `e2e_install`).
- **Production diff is zero. Test diff ~155 LOC across 2 files** (`harness.go`: +42 for `RunBare`; `cli_verbs_test.go`: +112 new file). Default `go test ./...` is unchanged — both files are gated by build tags.

### Codebase (supervisor fix, ticket #78)
- **Foreground-mode stdin reader leak fixed** — `runOnce` now opens `/dev/tty` (`os.O_RDONLY|syscall.O_NONBLOCK`) as a side fd for the input bridge instead of reading `os.Stdin`. On child exit, the per-cycle `input.Close()` wakes the in-flight `Read` via the Go runtime poller; the goroutine drains; nothing accumulates on `os.Stdin`'s `fdMutex`. Drain semantics flip from "drain one of two with timeout" to "drain both with timeout"; total worst-case `runOnce` return latency is `2 × goroutineDrainTimeout` (~200ms). Bridge-mode branch (`Config.Bridge != nil`) is unchanged.
- **`openTTYInput` + `stdinFallback`** — unexported, internal to `internal/supervisor`. Fallback path (`stdinFallback{}` — `Read` from `os.Stdin`, no-op `Close`) is taken when `/dev/tty` cannot be opened (headless containers, etc.); preserves the legacy leak in that environment by design rather than breaking the process by closing `os.Stdin`. Foreground mode is TTY-only by construction (`term.MakeRaw(stdinFd)` already early-returns on a non-TTY), so the fallback is a defensive seam, not the production path.
- **Why O_NONBLOCK is load-bearing.** Plain `os.OpenFile("/dev/tty", os.O_RDONLY, 0)` produces a blocking fd; `internal/poll.(*FD).Read` calls `syscall.Read` directly and parks in the kernel, where POSIX `close(2)` on another goroutine is a no-op for the in-flight read. With `O_NONBLOCK`, `Read` returns `EAGAIN`, the runtime parks the goroutine on the poller, and `Close` calls `runtime_pollUnblock` to wake it with `os.ErrClosed`. Drop the flag and the leak comes back.
- **Targeted test** — `TestSupervisor_Foreground_NoStdinReaderLeak` runs ~3 child-restart cycles and asserts `runtime.NumGoroutine()` does not grow by more than 2; skips with `t.Skipf("no controlling tty: %v", err)` when `/dev/tty` cannot be opened (CI without a TTY). Verified clean under `go test -race -count=20`.
- **Unblocks the flake stack.** The leak was the root cause of `TestPool_ActiveCap_RaceConcurrentActivate` deadlock (#41), `TestPool_Run_StartsWatcher` flake (#39), and `TestPool_New_Reconciles_ColdStart_PicksNewestImmediately` flake (#73). Bridge-mode-fixture migration in session tests is now belt-and-suspenders rather than required.

### Codebase (Phase 1.1e-D, ticket #102)
- **`pyry attach [<id>]` CLI positional + help text** — the operator-facing CLI surface for Phase 1.1e multi-session attach. `pyry attach` (no args) is byte-identical to v0.5.x: empty `SessionID` flows through `omitempty` → server's `Pool.ResolveID` resolves to bootstrap. `pyry attach <full-uuid>` and `pyry attach <prefix>` pass the selector verbatim through `AttachPayload.SessionID`; full-UUID and unique-prefix both attach. ### Codebase (Phase 1.1e-C, ticket #101)
- **`AttachPayload.SessionID string` (`json:"sessionID,omitempty"`) + `handleAttach` routes via `Pool.ResolveID`** — wire + server half of Phase 1.1e multi-session attach. `omitempty` is load-bearing: empty-`SessionID` payloads marshal byte-identically to v0.5.x output (`{"cols":80,"rows":24}`), so a v0.5.x client keeps working against a v0.7.x server during the rollover. `handleAttach` resolves `req.Attach.SessionID` through `Pool.ResolveID` BEFORE any bridge work, then re-fetches the session via `Pool.Lookup`. `nil req.Attach` is treated identically to empty `SessionID` — both resolve to bootstrap. CLI surface (`pyry attach <id>`) intentionally deferred to 1.1e-D; the new field is reachable only via hand-crafted JSON or the test harness this slice.
- **`SessionResolver` interface gained `ResolveID(arg string) (sessions.SessionID, error)`.** Three implementations updated with 1-line passthroughs: `cmd/pyry/main.go` `poolResolver`, plus `fakeResolver` and `recordingResolver` in `internal/control/server_test.go`. No other call sites consume `SessionResolver` (verified by grep across `internal/control` and `cmd/pyry`).
- **Two-step ResolveID-then-Lookup, two sequential RLocks.** Honours #66's choice to return `SessionID` (not `*Session`) — the second `Lookup` is the lock-clean guard against a session being removed between resolve and use. Race window is microseconds; `Lookup` returns `ErrSessionNotFound`, which encodes to the same `"attach: sessions: session not found"` wire string the resolver itself produces — identical operator diagnostic. No new locking; concurrent attaches against different sessions remain fully parallel.
- **Errors encode as `"attach: <err>"` verbatim via `fmt.Sprintf("%v", err)`.** `ErrSessionNotFound` → `"attach: sessions: session not found"`; `ErrAmbiguousSessionID` flows through with its full multi-line `<uuid> (<label>)` match list. Bridge state is **untouched** on the error path — `ResolveID` and `Lookup` both return before `conn.SetDeadline(time.Time{})`, before `Activate`, before `Attach`. Asserted in tests via `fakeSession.attachCalls == 0`, not just response-string match. Existing `sessions.ErrAttachUnavailable` mapping (foreground-mode wire string) untouched — runs after `Attach`, no collision with the new error encoding.
- **Production diff ~10 LOC across 3 files** (`protocol.go`: one field; `server.go`: interface method + dispatcher pass-through + handler swap; `cmd/pyry/main.go`: 3-line resolver passthrough). Tests carry the bulk: `attach_resolve_test.go` (~400 LOC) covers all five AC cases plus the v0.5.x byte-identical baseline.
- See [knowledge/features/control-plane.md § Attach: ResolveID-then-Lookup](knowledge/features/control-plane.md).

### Codebase (Phase 1.1e-A, ticket #66)
- **`Pool.ResolveID(arg string) (SessionID, error)` prefix resolver + `ErrAmbiguousSessionID` sentinel** — typed loose-input resolver pairing with `Pool.Lookup` (full id only). Empty arg → bootstrap (same seam as `Lookup("")`); exact full-UUID match short-circuits via single map lookup (no extra scan cost beyond the existing lookup); otherwise `strings.HasPrefix` scan over `p.sessions` collects candidates. Zero matches → `ErrSessionNotFound` (reused); unique → that id; ≥2 → `ambiguousError(matches)` returning `fmt.Errorf("%w:\n%s", ErrAmbiguousSessionID, lines)` with each `<uuid> (<label>)` pair on its own line, sorted by `SessionID` asc. The internal seam Phase 1.1e-B's `pyry attach <id>` wire + CLI surface (and any future verb taking a session selector) consumes; 47-B / 48-B may opportunistically refactor onto it when next touched.
- **Sentinel + `fmt.Errorf("%w: …")` over a struct error type.** AC required "no new exported types beyond the new typed error" + "errors.Is matching cheap." Struct error with `Matches []SessionRef` would expose a second exported type and force CLI consumers to type-assert or duplicate the formatting. Sentinel-wrap gives one pointer compare via the wrapped chain and a verbatim-printable message in one syscall — strictly simpler under the AC's exported-surface budget.
- **Synthetic `"bootstrap"` substitution mirrored in the formatter.** When the bootstrap entry's on-disk label is empty, the formatter writes `<uuid> (bootstrap)` rather than `<uuid> ()`. Mirrors `Pool.List`'s rule one-for-one — divergence would surface as operator confusion (`pyry sessions ls` saying one name, the disambiguation prompt saying another) before it surfaced as a bug report. Same precedent as the `List` substitution: substitute at the read primitive, not at the renderer.
- **Pool.mu (RLock) only; no `Session.lcMu`.** Reads `id` (mutated by `RotateID` under `Pool.mu` write — invariant documented at that site) and `label` (mutated by `Pool.Rename` under `Pool.mu` write). Both already in the `Pool.mu`-reader set. `lcState` and `lastActiveAt` are NOT consulted — `ResolveID` does not filter by lifecycle state (an evicted session must still be resolvable for `pyry sessions rename <prefix>` / `pyry sessions rm <prefix>`). Lock-order graph (`Pool.capMu → Pool.mu → Session.lcMu`) unchanged.
- **Returns `SessionID`, not `*Session` / `SessionInfo`.** Smallest possible surface, symmetric with the wire/CLI flow: 1.1e-B unmarshals the id, calls `ResolveID`, then routes to `Lookup`/`Activate`/`Remove` with the resolved id. Returning `*Session` would tempt callers to skip the second lookup — but the second lookup is the lock-clean way to guard against a session being removed between resolve and use. No CLI policies enforced at this layer (no minimum prefix length, no whitespace trimming) — same posture as `Pool.Rename`/`Pool.Create` declining to validate `label`.
- See [knowledge/features/sessions-package.md § Pool.ResolveID](knowledge/features/sessions-package.md).

### Codebase (Phase 1.1d-A1, ticket #94)
- **`Pool.Remove(ctx context.Context, id SessionID) error` write primitive + `ErrCannotRemoveBootstrap` sentinel** — terminates a session's claude process and drops its registry entry. The on-disk JSONL is **not** touched (disposition lands in 64-A2 / #95). Bootstrap is rejected via the new sentinel; unknown ids reuse `ErrSessionNotFound`. The internal seam the future `pyry sessions rm` CLI verb (#65) consumes instead of touching processes or `sessions.json` directly.
- **Delete-then-evict ordering is the load-bearing invariant.** `Pool.mu` covers the in-memory `delete(p.sessions, id)` + `saveLocked()`, then is released **before** calling `sess.Evict(ctx)`. The alternative (holding `Pool.mu` across `Evict`) deadlocks: the lifecycle goroutine's `transitionTo` reacquires `Pool.mu` via `Pool.persist`. Releasing `Pool.mu` after the in-memory delete commits is safe — concurrent `Lookup`/`Activate`/`Rename`/`List` callers see the session as gone from that moment on, so no observer can witness a half-removed state and no re-spawn race exists. Same outer-mutex pattern as the cap-policy eviction (#41), here resolved by holding `Pool.mu` only during the cheap mutate phase rather than introducing a new outer lock.
- **Save-failure rollback mirrors `Pool.Rename`.** If `saveLocked` fails after the in-memory delete, restore `p.sessions[id] = sess` before unlocking. Disk and memory remain consistent at every observable point. `saveRegistryLocked`'s temp+rename discipline makes partial writes unreachable; the rollback is belt-and-suspenders for "destination dir vanished" / `ENOSPC`.
- **Termination reuses `Session.Evict`.** `Pool.Remove` does not re-implement SIGTERM/SIGKILL/grace logic — `Session.Evict` already drives the supervisor's child via `exec.CommandContext` cancel ⇒ SIGKILL ⇒ `cmd.Wait` returns. The supervisor today does not have a SIGTERM grace window; SIGKILL is uncatchable, so no fallback path is needed (the AC's "SIGKILL fallback when child ignores SIGTERM" maps to "the existing kill-on-cancel terminates uncooperative children deterministically", verified in `TestPool_Remove_TerminatesUncooperativeChild` with a `trap "" TERM INT HUP; sleep 86400` shell child).
- **Lifecycle goroutine survives until pool shutdown.** After `Pool.Remove` returns, `sess.Run`'s loop transitions to `runEvicted` and parks on `activateCh` / `ctx.Done()`. The session is no longer reachable via `Pool.sessions`, so no caller can signal `activateCh`; the goroutine survives until `Pool.Run`'s `runCtx` cancels at pool shutdown. Bounded resource cost (one orphan goroutine + ~1KB `*Session` per `Remove` per pool lifetime) — operationally invisible for pyrycode's expected workload. Per evidence-based fix selection, no per-session terminate signal until observed.
- **No `Session.lcMu` taken; lock order unchanged.** `Pool.Remove` does not read `lcState` / `lastActiveAt` / `attached`. Lock-order graph (`Pool.capMu → Pool.mu → Session.lcMu`) is unchanged. Empty-id is *not* the bootstrap shorthand it is in `Pool.Lookup` — empty falls through to "not in map" and returns `ErrSessionNotFound`; destructive operations require an explicit id.
- See [knowledge/features/sessions-package.md § Pool.Remove](knowledge/features/sessions-package.md).

### Codebase (Phase 1.1d-A2, ticket #95)
- **`Pool.Remove` signature evolves to `(ctx, id, opts RemoveOptions) error` + `JSONLPolicy` enum** (`JSONLLeave` / `JSONLArchive` / `JSONLPurge`). Zero-value `RemoveOptions{}` preserves 1.1d-A1 behaviour byte-for-byte (JSONL untouched). `JSONLArchive` moves the live JSONL into `<pyry-data-dir>/archived-sessions/<uuid>.jsonl` (data-dir = parent of `registryPath`; subdir auto-created; errors wrapping `fs.ErrExist` if destination exists; source-absent is a no-op). `JSONLPurge` deletes the JSONL (source-absent is a no-op). The internal seam the future `pyry sessions rm --archive` / `--purge` CLI surface (#65) consumes.

### Codebase (Phase 1.1a-A2, ticket #73)
- **`Pool.Create(ctx, label) (SessionID, error)`** — user-facing primitive that mints a fresh non-bootstrap session: `NewID` → build `*Session` in `stateEvicted` (label verbatim, `bootstrap=false`, `ResumeLast=false`, `ClaudeArgs` cloned + `--session-id <uuid>` appended) → register-and-persist under `Pool.mu` (rollback on save failure) → `RegisterAllocatedUUID` (rotation skip-set) → `supervise` (#72's seam) → `Pool.Activate` (cap-aware spawn from #41). No duplicated cap logic and no transient inconsistent view from `Snapshot`/`Lookup` (the alternative — pre-flight before registering — would create one).

- **No new public types or sentinels.** `Pool.Create` is the only new exported name; `ErrPoolNotRunning` (from #72) is the only sentinel propagated.

### Codebase (Phase 1.1a-A1, ticket #72)
- **Pool supervisor fan-out seam** — unexported `Pool.supervise(sess *Session) error` schedules `sess.Run(gctx)` on `Pool.Run`'s live errgroup. Returns the new sentinel `ErrPoolNotRunning` when called before `Run` has wired the handle or after `Run` has cleared it (matchable with `errors.Is`). The helper is the join point Phase 1.1a-A2's `Pool.Create(ctx, label)` consumes; the bootstrap fan-out inside `Pool.Run` is itself rewritten to call `supervise` so the helper is exercised in production from day one. Watcher fan-out is unchanged (not a `*Session`).
- **Supervisor handle on `*Pool`** — `runGroup *errgroup.Group` + `runCtx context.Context`, both guarded by `Pool.mu` (RWMutex) and read together so a caller never sees a half-initialised handle. Set in `Pool.Run` under `Pool.mu` (write) right after `errgroup.WithContext`; cleared in a `defer` so a panicking goroutine still resets the handle. `supervise` reads under `Pool.mu` (RLock) and calls `g.Go` *off-lock*. No new lock-order edges (`Pool.mu → Session.lcMu` and `Pool.capMu → Pool.mu → Session.lcMu` unchanged) — the helper does not touch `Session.lcMu`.
- **Why fields-on-Pool, not a separate guard struct** — `Pool.mu` already protects every other "what's in this pool right now" read. A second mutex would double the lock count without buying anything; RWMutex matches the access pattern (N concurrent supervise readers, one-shot writes by `Run` setup/teardown). Concurrent `supervise` callers don't contend with each other.
- **Race-with-teardown is benign** — a `supervise` call interleaved with the deferred clear either (a) acquires RLock first, schedules onto a group whose ctx is about to cancel, and `Session.Run` exits via the existing shutdown path, or (b) acquires after, observes `nil`, returns the sentinel. No silent leak, no deadlock.
- **Defensive wrap on the bootstrap call** — `Pool.Run` does `if err := p.supervise(bootstrap); err != nil { return fmt.Errorf("sessions: supervise bootstrap: %w", err) }` even though the precondition is established two lines above. Cheap insurance against a future reordering bug.

### Codebase (Phase 1.2c-B, ticket #41)
- **Concurrent active cap (LRU eviction)** — `Config.ActiveCap` bounds the number of concurrently running claudes. `<= 0` is unset (uncapped, byte-identical to 1.2c-A); `>= 1` enforces. When activating one more session would exceed the cap, the LRU peer (oldest `lastActiveAt` among currently-active sessions, excluding the target) is evicted via the new public `Session.Evict` primitive before the new spawn proceeds. **No CLI flag yet** — Phase 2.0 will wire `-pyry-active-cap` when the auto-mint workload makes it load-bearing.
- **`Session.Evict(ctx)` public primitive** — symmetric to `Activate`. Force-eviction: unlike the idle timer, it does **not** defer for `attached > 0`. The cap is a hard limit; an attached caller will see EOF on its bridge. Phase 2.0 may add an attach-aware filter to the LRU candidate set; not in scope for #41. Pairs `evictCh` (buffered 1 signal) with `evictedCh` (closed-on-evicted broadcast), mirroring `activateCh`/`activeCh`.
- **`Pool.capMu` outermost lock** — taken only on the cap path (`activeCap > 0`). Lock order: `Pool.capMu → Pool.mu → Session.lcMu`. Serialises cap-check + victim eviction + new spawn so concurrent `Activate`s can't both observe `active < cap` and both proceed. Never re-acquired by callees, so `Session.Evict`'s callback into `Pool.persist` (which takes `Pool.mu`) is deadlock-free. The uncapped path doesn't take `capMu` at all — "byte-identical when unset" includes "doesn't even take the cap mutex".
- **`Pool.pickLRUVictim`** — O(n) iteration over `pool.sessions` under `Pool.mu.RLock`, reads each `Session.lcState`/`lastActiveAt` under `lcMu`, returns the oldest-`lastActiveAt` peer (excluding target). Premature optimisation rejected: a heap/linked-list would defeat the zero-cost-when-unset property and add a counter-vs-truth drift class of bug for an O(n) cost that doesn't matter at pyry's scale (≤ 100s of sessions).
- **`Session.touchLastActive`** — bumps `lastActiveAt` in-memory only on a cap-path `Activate` against an already-active session. **Not persisted** (the registry's `lastActiveAt` only flushes on transitions). Keeps in-memory LRU ordering reflective of the most recent touch.
- **`Pool.Activate` is now the spawn-path entry, not just a thin Lookup wrapper.** Phase 2.0's router calls this. Cap-unset path: one early-return branch and forward to `Session.Activate`. Cap-set path: cap check + victim eviction + new spawn under `capMu`.
- See [knowledge/features/idle-eviction.md](knowledge/features/idle-eviction.md), [ADR 006](knowledge/decisions/006-concurrent-active-cap-lru.md).

### Codebase (Phase 1.2c-A, ticket #40)
- **Idle eviction + lazy respawn** — per-`*Session` lifecycle goroutine drives a two-state machine (`active` ↔ `evicted`). When `attached == 0` and the idle timer fires, the supervisor's inner ctx is cancelled and claude exits cleanly; the JSONL on disk is preserved. `Session.Activate(ctx)` respawns the supervisor pointing at the same JSONL on disk so the conversation continues. `Session.Run` rewritten from one-line supervisor delegate into a `runActive` / `runEvicted` loop; `Pool.Run` shape **unchanged** (multi-session fan-out is Phase 1.1's job).
- **Activity = attach state** — while `attached > 0` the timer re-arms on fire (poll-with-grace; eviction may overshoot the configured timeout by up to one window). `last_active_at` is bumped on every state transition for the upcoming LRU follow-up's victim selection.
- **`-pyry-idle-timeout` flag** — production default `15m`, `0` disables eviction entirely (test default + operator escape hatch). Wired through `Config.IdleTimeout` → `SessionConfig.IdleTimeout` (per-session override falls back to pool default).
- **Activate-before-Attach contract** — `bridge.Attach` on an evicted session would block on the pipe forever. `internal/control.Session` interface gained `Activate(ctx)`; `handleAttach` calls it with a 30s timeout cap before binding the bridge. `handleStatus` does **not** activate (faithful `PhaseStopped` for evicted; no spurious wakeups from a poll). Wire response on respawn failure: `attach: activate: <err>`.
- **Registry `lifecycle_state` field** — `"active"` / `"evicted"`, `omitempty` so the dominant active case stays off disk (preserves #34's idempotent-reload byte-stability). Old pyry binaries ignore the field; new pyry reading old files defaults to `"active"`. Bootstrap warm-starts in whatever state the registry says — including starting in `evicted` if pyry was stopped while evicted.
- **Lock order: `Pool.mu → Session.lcMu`** — `Session.lcMu` protects `lcState` / `attached` / `activeCh` / `lastActiveAt` (when read for the registry snapshot). `Session.transitionTo` releases `lcMu` before calling `Pool.persist` so `saveLocked`'s per-session re-acquire doesn't deadlock. `RotateID` mutates `session.id` without `lcMu` — documented invariant (today's only callers run before any lifecycle goroutine observes the id).
- **Concurrency primitives** — `activeCh` (closed-on-active broadcast) + `activateCh` (buffered 1 signal). `Activate` snapshots `activeCh` under `lcMu` *before* waiting so a concurrent evict-replace doesn't drop the wakeup. Concurrent `Activate`s collapse to one signal via the buffered send.

Malformed JSON is fatal at startup — operator must fix or remove.
- **Atomic write seam** — `saveRegistryLocked` does `os.CreateTemp` → `Chmod 0600` → encode → fsync → close → `os.Rename`. Rename is the commit point; partial JSON is unreachable in the target. `defer os.Remove(tmp)` cleans up orphaned temps best-effort. `Pool.saveLocked` is the package-internal hook Phase 1.1's `Pool.Add` / `Rename` / `Remove` will call before returning success — caller holds `Pool.mu` (write) across the disk I/O.
- **Bootstrap marker on disk** — `bootstrap: true` is persisted explicitly so `Pool.Lookup("")` doesn't depend on file ordering. Phase 1.1's `pyry sessions rm <bootstrap-uuid>` thus has a clean question to answer (refuse, or promote another entry) instead of relying on array position. `omitempty` keeps non-bootstrap entries clean on disk.
- **Stable disk ordering** — `sortEntriesByCreatedAt` (then by `ID`) before write makes the file's byte content a deterministic function of the in-memory set, defeating Go's randomized map iteration and giving the AC's idempotent-reload guarantee. Degenerate for 1.2a's single entry; the property is paid for upfront ahead of 1.1.

## Patterns Established

- **Config struct pattern** — all configuration in a single `Config` struct, defaults applied in `New()`
- **Context-based cancellation** — `context.Context` flows through `Run()` → `runOnce()`, checked at every wait point
- **Structured logging** — `log/slog` with injected logger, not a global
- **Exponential backoff with stability reset** — backoff doubles on restart, resets to initial if child stayed up longer than `BackoffReset`
- **Deferred cleanup** — `defer` for terminal restore, PTY close, signal stop
- **Empty ID resolves to default** — `Pool.Lookup("")` returns the bootstrap session, so future `req.SessionID` fields can be added with no handler-side branching (old clients send empty, get the bootstrap; new clients send a real ID, get the right entry)
- **Introduce-then-rewire slicing** — split #27 into #28 (new package + tests, no consumers) and #29 (mechanical consumer rewiring) to keep each PR focused
- **Consumer-side interface definition** — `internal/control` defines the interfaces it consumes (`SessionResolver`, `Session`) rather than importing them from the producer package. Keeps `internal/sessions` free of control-plane concerns and lets tests fake the surface without exporting test seams from the producer.
- **Wire-string preservation via `errors.Is` mapping** — when refactoring an error path that crosses package boundaries, map the new sentinel back to the old wire string explicitly (`if errors.Is(err, sessions.ErrAttachUnavailable) { … Phase 0 string … }`) rather than letting `fmt.Sprintf("%v", err)` change client output. Required when an AC says "byte-identical."
- **Atomic on-disk writes via temp + rename** — for operator-recoverable JSON state files, `os.CreateTemp` in the same dir → encode → `f.Sync()` → `f.Close()` → `os.Rename(tmp, path)` makes the rename the commit point and partial files unreachable. `defer os.Remove(tmp)` cleans up orphans best-effort (no-op after successful rename). Don't bother with directory fsync unless real-world corruption is observed — pyry's registry is recoverable, not a database.
- **Stable disk ordering for idempotent reload** — sort serialized collections by a stable key (e.g. `created_at` then `id`) before writing. Defeats Go's randomized map iteration so save→load→save round-trips are byte-stable, and makes load-twice-equals-load-once into a real property rather than a probabilistic one.
- **Persistence as a Config field, empty disables** — `Config.RegistryPath` (or any "where to persist" string) defaulting to empty = no I/O lets unit tests construct a Pool with `t.TempDir()` or with persistence off entirely, with no test-only branches inside `New`. Production callers always pass a real path; the empty case is genuinely test-only. Same shape reused for `Config.ClaudeSessionsDir` in 1.2b-A.
- **One mutation seam, multiple drivers** — `Pool.RotateID` is the single place that swaps a session's in-memory id and persists. Startup reconciliation drives it from a directory scan; the live-detection watcher drives it from an fsnotify event. Defining the seam once with the lock-and-save contract baked in meant the second driver landed as a goroutine that calls a function, not a redesign.
- **Closures over primitives across package boundaries** — when a downstream package (`internal/sessions/rotation`) needs to call back into its host (`internal/sessions`), express the contract as a struct of closures over primitive types (`string`, `int`) rather than an interface that references the host's domain types (`SessionID`). The downstream package then has zero knowledge of the host, and the type-safety conversion (`SessionID ↔ string`) happens exactly once, in the wiring site. Avoids the import cycle the obvious "watcher takes a `*Pool`" design would create.
- **Test seam via package var, not interface injection** — `var newProbe = rotation.DefaultProbe` in `internal/sessions/pool.go`. Tests overwrite to inject a fake; production callers see no extra plumbing. Same shape as `allocatedTTL` and `probeRetryDelays` (test-tunable knobs that aren't config). Cheaper than threading a factory through `Config` for a single test surface.
- **Errgroup as the fan-out seam** — `Pool.Run`'s `errgroup.WithContext` ties the bootstrap supervisor and the rotation watcher; Phase 1.1's N-session fan-out reuses the same cancellation+error semantics. The 1.1a-A1 (#72) seam is the natural shape: stash `*errgroup.Group` + `context.Context` on the host struct under its existing RWMutex, expose an unexported helper that RLock-snapshots both and calls `g.Go` off-lock, and pair it with a sentinel error (`ErrPoolNotRunning`) for before/after-`Run` calls. The helper is the join point post-`Run` consumers (`Pool.Create`) plug into; the existing bootstrap fan-out is refactored onto the helper so the helper is exercised in production from day one rather than living dormant.
- **Two-state machine with explicit-verb wake-up** — when a runtime entity needs an `idle ↔ live` lifecycle, prefer (a) a per-entity goroutine that owns both the state machine and the underlying resource lifecycle, (b) an explicit `Activate(ctx)` call rather than implicit wake on resolve/lookup, and (c) a closed-on-state broadcast channel + buffered(1) signal channel for the wake-up handshake. Implicit wake-on-lookup tangles read-side concerns (status reads paying respawn latency); explicit Activate keeps every caller's contract sharp. Used for `Session.Activate` (1.2c-A); will be reused by future per-session lifecycles (router, bind/unbind).
- **Lock order `Pool.mu → Session.lcMu` with release-before-callback** — when a callee needs to call back into the host's lock-protected persistence (`Pool.persist` from `Session.transitionTo`), release the inner lock *before* the callback so the host's re-acquire of the inner lock inside its serialization path doesn't deadlock. Document the order at every site that takes both. Same shape would apply to any "child mutates state then asks parent to persist" coupling.
- **Dedicated outer mutex for "cap-shaped" critical sections** — when a sequence (cap-check + victim eviction + new spawn) must be atomic but its inner steps already take the host's main mutex (and may re-take it via persistence callbacks), introduce a *separate* outer mutex (`Pool.capMu` for #41) rather than holding the main mutex (`Pool.mu` write) across the whole sequence. The outer mutex serialises the policy decision; the inner work continues to take/release the main mutex normally. Avoids both the deadlock (re-entrancy on `Pool.mu` via `persist`) and the contention (blocking unrelated `Pool.mu` reads for the eviction's bounded window). Lock order documented; outer mutex is never re-acquired by callees.
- **"Zero cost when unset" as a hot-path discipline** — when a feature is configurable and most operators leave it unset (`Config.ActiveCap == 0`), gate ALL of its bookkeeping behind the early-return branch — including the new mutex it needs. The uncapped `Pool.Activate` doesn't even take `capMu`, doesn't iterate `sessions`, doesn't read `lastActiveAt`. The branch is one `if cap <= 0 { return … }` at the top of the cap-only path. The property "the previous phase's behaviour is byte-identical with the new feature unset" is paid for by being *strict* about what happens before the early return.
- **Id-or-empty as the failure-mode discriminator for "mint, persist, activate" primitives** — when a constructor-shaped method returns `(ID, error)` and has multiple failure points (rng, validation, persist, activate, supervise), make the returned id the caller's discriminator: empty id ⇒ "nothing was persisted, nothing to clean up"; non-empty id ⇒ "entry is on disk, decide what to do (retry the rest, accept eventual lifecycle, leave for next process start)." Callers distinguish further with `errors.Is` against named sentinels. Cleaner than a multi-valued status struct, and the empty/non-empty rule is checkable at every call site without deconstructing the error chain. Used for `Pool.Create` (1.1a-A2). The same shape suits any future "create-then-activate" seam where partial completion is recoverable.
- **Synthetic substitution at the read boundary, not at the renderer** — when an on-disk record's empty field has a well-known operator-facing default (e.g. `"bootstrap"` for an unlabelled bootstrap session), do the substitution in the *read primitive* that returns the typed snapshot, not in the CLI/wire renderer that consumes it. The on-disk record stays unmutated; every future consumer (a JSON renderer, a TUI, an HTTP handler) gets the same UX guarantee for free without re-implementing the rule. Pair with a discriminator field (e.g. `Bootstrap bool`) so the substitution is a UX nicety rather than the only signal — an operator who sets `Label="bootstrap"` on a non-bootstrap session is still distinguishable. Used for `Pool.List`'s bootstrap label (1.1b-A); same shape suits any "default for empty field" rule that crosses multiple consumers.
- **Per-consumer value type, not a shared "kitchen-sink snapshot"** — when multiple consumers each need a typed snapshot of part of a domain object's state, prefer one small value type per consumer over one shared type that unions every field everyone might want. `SnapshotEntry{ID, PID}` for the rotation watcher (closure-over-primitives boundary, cannot import the host's domain types); `SessionInfo{ID, Label, LifecycleState, LastActiveAt, Bootstrap}` for the operator-visible list. Adding lifecycle state to `SnapshotEntry` would either bloat every rotation snapshot or leak the host's enum into the rotation package's import set. Two ~5-field types are clearer than one ~10-field type with most fields irrelevant per call site.
- **Per-session template fields snapshotted at construction, read-only after** — when a Pool/Registry needs a "shape of new entries" template (claude bin, work dir, backoff timings, bridge presence), capture it as an unexported field on the host (`Pool.sessionTpl`) at `New()` time and treat it as immutable thereafter. Per-create call clones the slice fields (`slices.Clone(tpl.ClaudeArgs)`) and mints fresh resources where sharing would be wrong (`supervisor.NewBridge(p.log)` per session in service mode). No lock needed for reads; mutation is impossible by construction. Two ~15-line struct constructions (bootstrap in `New`, fresh entry in `Create`) with different lifecycle-state defaults is clearer than a `lcState lifecycleState` parameter on a shared helper — accepted intentionally.
- **Side-fd + Close to drain a blocked Read goroutine — only works with `O_NONBLOCK` on character devices.** When a goroutine is blocked in `io.Copy`/`Read` on a process-wide fd (e.g. `os.Stdin`) with no way to wake it, the standard fix is to open a parallel fd to the same underlying device, copy from *that* fd, and `Close` it on cleanup. For sockets and pipes the runtime poller mediates Reads automatically; for character devices like `/dev/tty` you must pass `os.O_RDONLY|syscall.O_NONBLOCK` so `Read` returns `EAGAIN` and the goroutine parks on the poller — only then does `Close` wake it via `runtime_pollUnblock`. Plain blocking `open` produces an fd whose in-flight `syscall.Read` cannot be interrupted by `Close` on another goroutine. Used for the foreground-mode input bridge in `internal/supervisor` (#78); applies to any "wake a stuck reader by closing a side fd" pattern.
- **Symmetric channel pairs for two-way state-machine signalling** — when a two-state machine (`active` / `evicted`) needs both *enter-state* and *exit-state* external triggers, mirror the channel pair: `activateCh` (buffered 1 signal) + `activeCh` (closed-on-active broadcast) gets a sibling `evictCh` (buffered 1 signal) + `evictedCh` (closed-on-evicted broadcast). Both pairs follow the same snapshot-under-lock-then-wait discipline. The state-transition code closes one channel and re-opens the other in lockstep. One pattern, two directions.
- **`HOME=t.TempDir()` as the e2e isolation envelope** — when the binary under test resolves all of its persistent paths via `os.UserHomeDir()` (which honors `$HOME` on Unix), redirecting the child's `HOME` to `t.TempDir()` contains every ~-relative path with a single env var — no per-resource flag plumbing required. Belt-and-suspenders explicit flags (e.g. `-pyry-socket=<tmp>/pyry.sock`) cover the load-bearing path without inventing new flags for ancillary state. The corollary: keep ALL persistent paths under `~/...` (don't sprinkle `/var/`, `/etc/`, `/tmp` defaults) so this trick keeps working.
- **Subprocess re-exec for "test the test framework's failure path"** — when verifying that a `t.Fatal`/`t.FailNow` path doesn't leak resources, you can't observe it from a same-process subtest (Go's testing framework propagates inner failures to the parent and ends the outer test before assertions run). Re-exec `os.Args[0]` with `-test.run=^TestInnerChild$` and an env-var flag (`PYRY_E2E_INNER_FATAL_OUT=<state-file>`) so the failing path runs in a fresh process; the child writes observed state (pid, socket path) to the file before failing; the parent reads it and asserts externally. The inner test is gated on the env var (`os.Getenv(...) == "" → t.Skip`) so normal runs are no-ops. Pairs naturally with the POSIX zero-signal probe (`os.FindProcess` + `Signal(syscall.Signal(0))`) for liveness checks.
- **Sentinel + `fmt.Errorf("%w: …")` over a struct error type when the only data the caller needs is a printable list** — when an error needs to carry "and here's what matched" data alongside a typed-discriminator (e.g. ambiguous prefix → list of candidate ids), prefer `var ErrFoo = errors.New(...)` plus `fmt.Errorf("%w:\n%s", ErrFoo, formattedLines)` over a struct error type with an exposed `Matches []Ref` field. Pros: zero new exported types; `errors.Is` is one pointer compare via the wrapped chain; CLI consumers do one `fmt.Fprintln(os.Stderr, err)` and get the human-readable list verbatim. Cons: callers cannot programmatically iterate matches — accept this whenever the data is fundamentally for human consumption (CLI disambiguation prompts, debug logs) and downstream code only needs the typed branch. Used for `Pool.ResolveID`'s `ErrAmbiguousSessionID` (1.1e-A); the same shape suits any "typed error + descriptive payload for printing" surface.
- **Loose-input resolver pairs with strict-input lookup** — when a domain object is keyed by a canonical id (e.g. `SessionID`) and external callers (CLIs, wire APIs) hand in loose strings (UUIDs, prefixes, empty), expose two methods: `Lookup(canonical)` for code-internal callers that already have the canonical form, and `ResolveID(loose) (canonical, error)` for input-handling callers. Resolution returns the canonical id, not `*Session` — callers then call `Lookup`/`Activate`/`Remove` with the resolved id, which guards against the entity being removed between resolve and use without saving any meaningful work. Resolver runs under the host's read lock; consumers each pay one extra hashmap probe but never observe a half-removed state. Same shape would suit any "user-input → canonical-id" transformation that today's consumers inline against the host's lock.
- **Exit-1 for runtime errors, exit-2 for "user typed it wrong"** — in CLI subcommands, return errors up the call stack for runtime failures (RPC failed, daemon-not-running, resolver error) so `main` prints them and exits 1. Use `os.Exit(2)` directly for argument-shape errors (extra positionals, missing required arg) so the two failure classes stay visually distinct in shell exit codes. POSIX convention; matches the existing `os.Exit(2)` shape at the top of `cmd/pyry/main.go`. The split is checkable: error return ⇒ runtime, `os.Exit(2)` inline ⇒ user typed wrong. Pair the inline exit with a unit-testable helper that returns a sentinel (`attachSelectorFromArgs` returning `errTooManyAttachArgs`) so the argument-shape rule is covered in a table test without intercepting `os.Exit`.
- **Extend an exported function's signature in place when there is exactly one caller.** Adding a sibling (e.g. `AttachWithID`) for a single new parameter splits the call surface for no benefit. The grep-for-callers test is the discriminator — if extending churns >1 production caller, consider a sibling; if it's one, extend. Used for `control.Attach` in 1.1e-D (one call site, fifth arg added). Same shape suits any "one new parameter, one caller" growth.
- **Build-tag-gated test packages for opt-in expensive tests** — when a test category is heavy (spawns binaries, requires `go build`, takes seconds) and not part of every-PR CI, gate the whole package with `//go:build e2e` (or similar). Default `go test ./...` doesn't compile or run it; opt-in via `go test -tags=e2e ./<path>/...`. Document the invocation in the package doc comment so godoc surfaces it. Lighter than a separate `make` target for the gating itself; the make target is the operator-facing convenience, not the mechanism.
