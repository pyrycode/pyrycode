# Project Memory ‚Äî Pyrycode

Repo-level session memory. Read this at the start of every session.

## What's Built

### Codebase (Phase 0)
- **Supervisor core** ‚Äî PTY spawn via `creack/pty`, raw-mode stdin/stdout bridging in foreground mode, Bridge-mediated I/O in service mode, exponential backoff restart with stability reset, `--continue` injection on restart for session persistence
- **SIGWINCH forwarding** ‚Äî terminal resizes propagate from controlling terminal to child PTY (foreground mode only; attach mode locks the size at attach time)
- **Control plane** ‚Äî Unix domain socket (`~/.pyry/<name>.sock`, 0600), line-delimited JSON protocol, verbs: `status`, `stop`, `logs`, `attach`
- **CLI transparency** ‚Äî unknown args forward verbatim to claude; pyry's own flags use `-pyry-*` prefix; `-pyry-name` plus `PYRY_NAME` env var for named multi-instance
- **Graceful shutdown** ‚Äî SIGINT/SIGTERM cancel the supervisor context, child is killed via `exec.CommandContext`, socket removed on exit
- **Service configs** ‚Äî systemd user unit (`systemd/pyry.service`), macOS launchd plist (`launchd/dev.pyrycode.pyry.plist`)
- **~1700 source + ~1100 test Go lines** as of late Apr 2026, 10+ PRs merged

### Codebase (Phase 1.0, tickets #28 + #29)
- **`internal/sessions` package** ‚Äî `SessionID` (UUIDv4 via `crypto/rand`, stdlib only), `Session` (wraps one `*supervisor.Supervisor` + optional `*supervisor.Bridge`), `Pool` (single-bootstrap registry with `RWMutex`-protected map). Sentinel errors `ErrSessionNotFound`, `ErrAttachUnavailable`. `Pool.Lookup("")` resolves to the bootstrap entry ‚Äî the seam Phase 1.1's `Request.SessionID` plugs into.
- **Production consumers wired (#29)** ‚Äî `cmd/pyry/main.go` constructs `*sessions.Pool` (with the supervisor.Config template inside `SessionConfig`); `internal/control` consumes a single `SessionResolver` interface (replaces Phase 0's `StateProvider` + `AttachProvider` pair). A 5-line `poolResolver` adapter in `cmd/pyry` bridges `Pool` ‚Üí `SessionResolver` (covariant-return workaround). Wire protocol unchanged; `pyry status`/`stop`/`logs`/`attach` byte-identical to Phase 0. Foreground-mode attach error string preserved verbatim via `errors.Is(err, sessions.ErrAttachUnavailable)` mapping in `handleAttach`.
- See [knowledge/features/sessions-package.md](knowledge/features/sessions-package.md), [knowledge/features/control-plane.md](knowledge/features/control-plane.md), and [ADR 003](knowledge/decisions/003-session-addressable-runtime.md).

### Codebase (supervisor fix, ticket #78)
- **Foreground-mode stdin reader leak fixed** ‚Äî `runOnce` now opens `/dev/tty` (`os.O_RDONLY|syscall.O_NONBLOCK`) as a side fd for the input bridge instead of reading `os.Stdin`. On child exit, the per-cycle `input.Close()` wakes the in-flight `Read` via the Go runtime poller; the goroutine drains; nothing accumulates on `os.Stdin`'s `fdMutex`. Drain semantics flip from "drain one of two with timeout" to "drain both with timeout"; total worst-case `runOnce` return latency is `2 √ó goroutineDrainTimeout` (~200ms). Bridge-mode branch (`Config.Bridge != nil`) is unchanged.
- **`openTTYInput` + `stdinFallback`** ‚Äî unexported, internal to `internal/supervisor`. Fallback path (`stdinFallback{}` ‚Äî `Read` from `os.Stdin`, no-op `Close`) is taken when `/dev/tty` cannot be opened (headless containers, etc.); preserves the legacy leak in that environment by design rather than breaking the process by closing `os.Stdin`. Foreground mode is TTY-only by construction (`term.MakeRaw(stdinFd)` already early-returns on a non-TTY), so the fallback is a defensive seam, not the production path.
- **Why O_NONBLOCK is load-bearing.** Plain `os.OpenFile("/dev/tty", os.O_RDONLY, 0)` produces a blocking fd; `internal/poll.(*FD).Read` calls `syscall.Read` directly and parks in the kernel, where POSIX `close(2)` on another goroutine is a no-op for the in-flight read. With `O_NONBLOCK`, `Read` returns `EAGAIN`, the runtime parks the goroutine on the poller, and `Close` calls `runtime_pollUnblock` to wake it with `os.ErrClosed`. Drop the flag and the leak comes back.
- **Targeted test** ‚Äî `TestSupervisor_Foreground_NoStdinReaderLeak` runs ~3 child-restart cycles and asserts `runtime.NumGoroutine()` does not grow by more than 2; skips with `t.Skipf("no controlling tty: %v", err)` when `/dev/tty` cannot be opened (CI without a TTY). Verified clean under `go test -race -count=20`.
- **Unblocks the flake stack.** The leak was the root cause of `TestPool_ActiveCap_RaceConcurrentActivate` deadlock (#41), `TestPool_Run_StartsWatcher` flake (#39), and `TestPool_New_Reconciles_ColdStart_PicksNewestImmediately` flake (#73). Bridge-mode-fixture migration in session tests is now belt-and-suspenders rather than required.

### Codebase (Phase 1.1a-A2, ticket #73)
- **`Pool.Create(ctx, label) (SessionID, error)`** ‚Äî user-facing primitive that mints a fresh non-bootstrap session: `NewID` ‚Üí build `*Session` in `stateEvicted` (label verbatim, `bootstrap=false`, `ResumeLast=false`, `ClaudeArgs` cloned + `--session-id <uuid>` appended) ‚Üí register-and-persist under `Pool.mu` (rollback on save failure) ‚Üí `RegisterAllocatedUUID` (rotation skip-set) ‚Üí `supervise` (#72's seam) ‚Üí `Pool.Activate` (cap-aware spawn from #41). One well-tested entry point for Phase 1.1a-B's `sessions.new` verb (#71) and Phase 2.0's channel-driven auto-mint. Bootstrap unchanged (`Pool.Lookup("")` returns the same `*Session` pre/post `Create`).
- **Per-session template fields on `*Pool`** ‚Äî `sessionTpl SessionConfig` (shallow copy of `cfg.Bootstrap` captured at `New()`) + `idleTimeoutDefault time.Duration` (mirror of `Config.IdleTimeout`), both read-only after `New`. `Create` clones `tpl.ClaudeArgs` via `slices.Clone` (appending to the template directly would mutate it across calls), and mints a fresh `supervisor.NewBridge(p.log)` per session in service mode (`tpl.Bridge != nil`) ‚Äî sharing one bridge across N sessions would multiplex their I/O into a single client view.
- **Persist-then-activate ordering, with id-or-empty discriminator.** Save failure with claude already running would leave an unsupervised orphan whose JSONL has no on-disk record; a registry-only entry that didn't activate is benign (recoverable on next `Activate`). Returned `SessionID == ""` iff nothing persisted; non-empty id + non-nil err means "entry on disk, decide what to do." Caller uses `errors.Is(err, ErrPoolNotRunning)` to distinguish the not-running case from an Activate failure.
- **Cap pre-flight via `Pool.Activate` directly (choice (a))** ‚Äî the new session is in `stateEvicted` at cap-check time, so it doesn't increment `active`; `pickLRUVictim` excludes the target. No duplicated cap logic and no transient inconsistent view from `Snapshot`/`Lookup` (the alternative ‚Äî pre-flight before registering ‚Äî would create one).
- **Lock order unchanged.** `Pool.mu ‚Üí Session.lcMu` for the persist couple; `capMu ‚Üí Pool.mu ‚Üí Session.lcMu` on the cap path. `Create` does NOT hold `p.mu` across `supervise`, `Activate`, or `RegisterAllocatedUUID` ‚Äî the lock is taken only for register+persist, then released. ctx-cancellation race documented: caller may see `(id, ctx.Err)` while the lifecycle goroutine spins up anyway (it respects the pool's run-ctx, not the caller's).
- **No new public types or sentinels.** `Pool.Create` is the only new exported name; `ErrPoolNotRunning` (from #72) is the only sentinel propagated.
- See [knowledge/features/sessions-package.md ¬ß Pool.Create](knowledge/features/sessions-package.md).

### Codebase (Phase 1.1a-A1, ticket #72)
- **Pool supervisor fan-out seam** ‚Äî unexported `Pool.supervise(sess *Session) error` schedules `sess.Run(gctx)` on `Pool.Run`'s live errgroup. Returns the new sentinel `ErrPoolNotRunning` when called before `Run` has wired the handle or after `Run` has cleared it (matchable with `errors.Is`). The helper is the join point Phase 1.1a-A2's `Pool.Create(ctx, label)` consumes; the bootstrap fan-out inside `Pool.Run` is itself rewritten to call `supervise` so the helper is exercised in production from day one. Watcher fan-out is unchanged (not a `*Session`).
- **Supervisor handle on `*Pool`** ‚Äî `runGroup *errgroup.Group` + `runCtx context.Context`, both guarded by `Pool.mu` (RWMutex) and read together so a caller never sees a half-initialised handle. Set in `Pool.Run` under `Pool.mu` (write) right after `errgroup.WithContext`; cleared in a `defer` so a panicking goroutine still resets the handle. `supervise` reads under `Pool.mu` (RLock) and calls `g.Go` *off-lock*. No new lock-order edges (`Pool.mu ‚Üí Session.lcMu` and `Pool.capMu ‚Üí Pool.mu ‚Üí Session.lcMu` unchanged) ‚Äî the helper does not touch `Session.lcMu`.
- **Why fields-on-Pool, not a separate guard struct** ‚Äî `Pool.mu` already protects every other "what's in this pool right now" read. A second mutex would double the lock count without buying anything; RWMutex matches the access pattern (N concurrent supervise readers, one-shot writes by `Run` setup/teardown). Concurrent `supervise` callers don't contend with each other.
- **Race-with-teardown is benign** ‚Äî a `supervise` call interleaved with the deferred clear either (a) acquires RLock first, schedules onto a group whose ctx is about to cancel, and `Session.Run` exits via the existing shutdown path, or (b) acquires after, observes `nil`, returns the sentinel. No silent leak, no deadlock.
- **Defensive wrap on the bootstrap call** ‚Äî `Pool.Run` does `if err := p.supervise(bootstrap); err != nil { return fmt.Errorf("sessions: supervise bootstrap: %w", err) }` even though the precondition is established two lines above. Cheap insurance against a future reordering bug.
- See [knowledge/features/sessions-package.md ¬ß Supervisor handle](knowledge/features/sessions-package.md).

### Codebase (Phase 1.2c-B, ticket #41)
- **Concurrent active cap (LRU eviction)** ‚Äî `Config.ActiveCap` bounds the number of concurrently running claudes. `<= 0` is unset (uncapped, byte-identical to 1.2c-A); `>= 1` enforces. When activating one more session would exceed the cap, the LRU peer (oldest `lastActiveAt` among currently-active sessions, excluding the target) is evicted via the new public `Session.Evict` primitive before the new spawn proceeds. **No CLI flag yet** ‚Äî Phase 2.0 will wire `-pyry-active-cap` when the auto-mint workload makes it load-bearing.
- **`Session.Evict(ctx)` public primitive** ‚Äî symmetric to `Activate`. Force-eviction: unlike the idle timer, it does **not** defer for `attached > 0`. The cap is a hard limit; an attached caller will see EOF on its bridge. Phase 2.0 may add an attach-aware filter to the LRU candidate set; not in scope for #41. Pairs `evictCh` (buffered 1 signal) with `evictedCh` (closed-on-evicted broadcast), mirroring `activateCh`/`activeCh`.
- **`Pool.capMu` outermost lock** ‚Äî taken only on the cap path (`activeCap > 0`). Lock order: `Pool.capMu ‚Üí Pool.mu ‚Üí Session.lcMu`. Serialises cap-check + victim eviction + new spawn so concurrent `Activate`s can't both observe `active < cap` and both proceed. Never re-acquired by callees, so `Session.Evict`'s callback into `Pool.persist` (which takes `Pool.mu`) is deadlock-free. The uncapped path doesn't take `capMu` at all ‚Äî "byte-identical when unset" includes "doesn't even take the cap mutex".
- **`Pool.pickLRUVictim`** ‚Äî O(n) iteration over `pool.sessions` under `Pool.mu.RLock`, reads each `Session.lcState`/`lastActiveAt` under `lcMu`, returns the oldest-`lastActiveAt` peer (excluding target). Premature optimisation rejected: a heap/linked-list would defeat the zero-cost-when-unset property and add a counter-vs-truth drift class of bug for an O(n) cost that doesn't matter at pyry's scale (‚â§ 100s of sessions).
- **`Session.touchLastActive`** ‚Äî bumps `lastActiveAt` in-memory only on a cap-path `Activate` against an already-active session. **Not persisted** (the registry's `lastActiveAt` only flushes on transitions). Keeps in-memory LRU ordering reflective of the most recent touch.
- **`Pool.Activate` is now the spawn-path entry, not just a thin Lookup wrapper.** Phase 2.0's router calls this. Cap-unset path: one early-return branch and forward to `Session.Activate`. Cap-set path: cap check + victim eviction + new spawn under `capMu`.
- See [knowledge/features/idle-eviction.md](knowledge/features/idle-eviction.md), [ADR 006](knowledge/decisions/006-concurrent-active-cap-lru.md).

### Codebase (Phase 1.2c-A, ticket #40)
- **Idle eviction + lazy respawn** ‚Äî per-`*Session` lifecycle goroutine drives a two-state machine (`active` ‚Üî `evicted`). When `attached == 0` and the idle timer fires, the supervisor's inner ctx is cancelled and claude exits cleanly; the JSONL on disk is preserved. `Session.Activate(ctx)` respawns the supervisor pointing at the same JSONL on disk so the conversation continues. `Session.Run` rewritten from one-line supervisor delegate into a `runActive` / `runEvicted` loop; `Pool.Run` shape **unchanged** (multi-session fan-out is Phase 1.1's job).
- **Activity = attach state** ‚Äî while `attached > 0` the timer re-arms on fire (poll-with-grace; eviction may overshoot the configured timeout by up to one window). `last_active_at` is bumped on every state transition for the upcoming LRU follow-up's victim selection.
- **`-pyry-idle-timeout` flag** ‚Äî production default `15m`, `0` disables eviction entirely (test default + operator escape hatch). Wired through `Config.IdleTimeout` ‚Üí `SessionConfig.IdleTimeout` (per-session override falls back to pool default).
- **Activate-before-Attach contract** ‚Äî `bridge.Attach` on an evicted session would block on the pipe forever. `internal/control.Session` interface gained `Activate(ctx)`; `handleAttach` calls it with a 30s timeout cap before binding the bridge. `handleStatus` does **not** activate (faithful `PhaseStopped` for evicted; no spurious wakeups from a poll). Wire response on respawn failure: `attach: activate: <err>`.
- **Registry `lifecycle_state` field** ‚Äî `"active"` / `"evicted"`, `omitempty` so the dominant active case stays off disk (preserves #34's idempotent-reload byte-stability). Old pyry binaries ignore the field; new pyry reading old files defaults to `"active"`. Bootstrap warm-starts in whatever state the registry says ‚Äî including starting in `evicted` if pyry was stopped while evicted.
- **Lock order: `Pool.mu ‚Üí Session.lcMu`** ‚Äî `Session.lcMu` protects `lcState` / `attached` / `activeCh` / `lastActiveAt` (when read for the registry snapshot). `Session.transitionTo` releases `lcMu` before calling `Pool.persist` so `saveLocked`'s per-session re-acquire doesn't deadlock. `RotateID` mutates `session.id` without `lcMu` ‚Äî documented invariant (today's only callers run before any lifecycle goroutine observes the id).
- **Concurrency primitives** ‚Äî `activeCh` (closed-on-active broadcast) + `activateCh` (buffered 1 signal). `Activate` snapshots `activeCh` under `lcMu` *before* waiting so a concurrent evict-replace doesn't drop the wakeup. Concurrent `Activate`s collapse to one signal via the buffered send.
- See [knowledge/features/idle-eviction.md](knowledge/features/idle-eviction.md), [ADR 005](knowledge/decisions/005-idle-eviction-state-machine.md).

### Codebase (Phase 1.2b-B, ticket #39)
- **Live `/clear` rotation watcher** ‚Äî `internal/sessions/rotation` package: fsnotify on `cfg.ClaudeSessionsDir` + per-PID FD probe (Linux walks `/proc/<pid>/fd`, macOS shells out to `lsof -nP -p <pid> -F fn`). On a CREATE for `<new>.jsonl` whose UUID a tracked PID has open, calls `Pool.RotateID(old, new)` ‚Äî same seam #38 established. Detection latency ~250ms worst-case (probeWithRetry walks 0/50/200ms attempts to handle CREATE-before-open races).
- **`Pool.Run` errgroup wrap** ‚Äî `golang.org/x/sync/errgroup` ties the bootstrap supervisor and the watcher; cancellation propagates both ways. Phase 1.1's N-session fan-out reuses the same wrapper ‚Äî the extension point is one `g.Go` per pool entry.
- **Pyry-allocated UUID skip set on `Pool`** ‚Äî `RegisterAllocatedUUID` / `IsAllocated`. Consume-on-first-hit, 30s TTL pruned opportunistically, all under `p.mu` (write). No live caller in 1.2b-B (pyry still uses `--continue`); scaffolding lands now so Phase 1.1's `claude --session-id` is a one-liner.
- **`rotation.Config` closures over primitives** ‚Äî the rotation package never imports `internal/sessions`. The `SessionID ‚Üî string` conversion happens once, inside `Pool.Run`'s closure wiring. `Pool.Snapshot` returns a small `SnapshotEntry` struct; `snapshotForRotation` translates to `rotation.SessionRef`.
- **Failure posture** ‚Äî rotation detection is never load-bearing. fsnotify init failure, missing `lsof`, missing `/proc`, save failure during `OnRotate` ‚Äî all logged warn/debug; pyry continues. `MkdirAll(dir, 0700)` for a not-yet-existing claude dir.
- **First new external runtime dep since `creack/pty`** ‚Äî `github.com/fsnotify/fsnotify` v1.10.0, justified in [ADR 004](knowledge/decisions/004-fsnotify-for-rotation-detection.md). `golang.org/x/sync` lands as a sibling for errgroup.
- See [knowledge/features/rotation-watcher.md](knowledge/features/rotation-watcher.md), [ADR 004](knowledge/decisions/004-fsnotify-for-rotation-detection.md).

### Codebase (Phase 1.2b-A, ticket #38)
- **Startup JSONL reconciliation** ‚Äî `Pool.New` scans `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl`, picks the most-recently-modified one, and rotates the registry's bootstrap entry to that UUID if it disagrees. Self-heals after `/clear` (claude rotates session UUIDs on `/clear`; without reconciliation, post-`pyry stop` the registry would still point at the pre-`/clear` UUID).
- **`Pool.RotateID(oldID, newID)`** ‚Äî atomic swap of in-memory map key + bootstrap pointer + `saveLocked`, all under `p.mu` (write). `RotateID(x, x)` is a no-op; unknown old id returns `ErrSessionNotFound`. Bumps `last_active_at`; preserves `created_at`. The load-bearing seam reused by Phase 1.2b-B (live-detection via fsnotify).
- **`encodeWorkdir`** ‚Äî claude maps both `/` AND `.` to `-` in its `~/.claude/projects/` path component (verified empirically 2026-05-02). `/foo/.bar` ‚Üí `-foo--bar` (note the doubled dash). Unexported; encoding lives inside `internal/sessions` and `cmd/pyry` only sees the resolved path via `DefaultClaudeSessionsDir`.
- **JSONL location** ‚Äî files live directly in `<encoded-cwd>/<uuid>.jsonl`. There is **no** `sessions/` subdirectory (the ticket body claimed otherwise; the spec followed reality).
- **Failure posture** ‚Äî missing/unreadable claude dir is logged warn and ignored (startup proceeds). Save failure during rotation is fatal-at-startup (matches 1.2a cold-start). The pre-rotation JSONL is never deleted or modified ‚Äî only the registry pointer moves.
- **Expected one-time rotation in 1.2a** ‚Äî on the second `pyry` restart, claude's UUID ‚âÝ pyry's mint, so reconcile fires once and logs an Info line. Becomes rare after Phase 1.1+ wires `claude --session-id <uuid>`.
- See [knowledge/features/jsonl-reconciliation.md](knowledge/features/jsonl-reconciliation.md).

### Codebase (Phase 1.2a, ticket #34)
- **Session registry on disk** ‚Äî `~/.pyry/<sanitized-name>/sessions.json` (file 0600, dir 0700), sibling to the per-name socket. Schema: `version` (forward-marker), `sessions[]` with `id` / `label` / `created_at` / `last_active_at` / `bootstrap`. Default `encoding/json` decoder tolerates unknown fields for forward compat.
- **`Pool.New` load-or-mint** ‚Äî Cold start (missing or empty file) mints a fresh UUID and writes the registry before returning. Warm start reads the bootstrap-marked entry, reuses its UUID + metadata, and does **not** rewrite the file (warm reload is not a state change). Malformed JSON is fatal at startup ‚Äî operator must fix or remove.
- **Atomic write seam** ‚Äî `saveRegistryLocked` does `os.CreateTemp` ‚Üí `Chmod 0600` ‚Üí encode ‚Üí fsync ‚Üí close ‚Üí `os.Rename`. Rename is the commit point; partial JSON is unreachable in the target. `defer os.Remove(tmp)` cleans up orphaned temps best-effort. `Pool.saveLocked` is the package-internal hook Phase 1.1's `Pool.Add` / `Rename` / `Remove` will call before returning success ‚Äî caller holds `Pool.mu` (write) across the disk I/O.
- **Bootstrap marker on disk** ‚Äî `bootstrap: true` is persisted explicitly so `Pool.Lookup("")` doesn't depend on file ordering. Phase 1.1's `pyry sessions rm <bootstrap-uuid>` thus has a clean question to answer (refuse, or promote another entry) instead of relying on array position. `omitempty` keeps non-bootstrap entries clean on disk.
- **Stable disk ordering** ‚Äî `sortEntriesByCreatedAt` (then by `ID`) before write makes the file's byte content a deterministic function of the in-memory set, defeating Go's randomized map iteration and giving the AC's idempotent-reload guarantee. Degenerate for 1.2a's single entry; the property is paid for upfront ahead of 1.1.
- See [knowledge/features/sessions-registry.md](knowledge/features/sessions-registry.md).

## Patterns Established

- **Config struct pattern** ‚Äî all configuration in a single `Config` struct, defaults applied in `New()`
- **Context-based cancellation** ‚Äî `context.Context` flows through `Run()` ‚Üí `runOnce()`, checked at every wait point
- **Structured logging** ‚Äî `log/slog` with injected logger, not a global
- **Exponential backoff with stability reset** ‚Äî backoff doubles on restart, resets to initial if child stayed up longer than `BackoffReset`
- **Deferred cleanup** ‚Äî `defer` for terminal restore, PTY close, signal stop
- **Empty ID resolves to default** ‚Äî `Pool.Lookup("")` returns the bootstrap session, so future `req.SessionID` fields can be added with no handler-side branching (old clients send empty, get the bootstrap; new clients send a real ID, get the right entry)
- **Introduce-then-rewire slicing** ‚Äî split #27 into #28 (new package + tests, no consumers) and #29 (mechanical consumer rewiring) to keep each PR focused
- **Consumer-side interface definition** ‚Äî `internal/control` defines the interfaces it consumes (`SessionResolver`, `Session`) rather than importing them from the producer package. Keeps `internal/sessions` free of control-plane concerns and lets tests fake the surface without exporting test seams from the producer.
- **Wire-string preservation via `errors.Is` mapping** ‚Äî when refactoring an error path that crosses package boundaries, map the new sentinel back to the old wire string explicitly (`if errors.Is(err, sessions.ErrAttachUnavailable) { ‚Ä¶ Phase 0 string ‚Ä¶ }`) rather than letting `fmt.Sprintf("%v", err)` change client output. Required when an AC says "byte-identical."
- **Atomic on-disk writes via temp + rename** ‚Äî for operator-recoverable JSON state files, `os.CreateTemp` in the same dir ‚Üí encode ‚Üí `f.Sync()` ‚Üí `f.Close()` ‚Üí `os.Rename(tmp, path)` makes the rename the commit point and partial files unreachable. `defer os.Remove(tmp)` cleans up orphans best-effort (no-op after successful rename). Don't bother with directory fsync unless real-world corruption is observed ‚Äî pyry's registry is recoverable, not a database.
- **Stable disk ordering for idempotent reload** ‚Äî sort serialized collections by a stable key (e.g. `created_at` then `id`) before writing. Defeats Go's randomized map iteration so save‚Üíload‚Üísave round-trips are byte-stable, and makes load-twice-equals-load-once into a real property rather than a probabilistic one.
- **Persistence as a Config field, empty disables** ‚Äî `Config.RegistryPath` (or any "where to persist" string) defaulting to empty = no I/O lets unit tests construct a Pool with `t.TempDir()` or with persistence off entirely, with no test-only branches inside `New`. Production callers always pass a real path; the empty case is genuinely test-only. Same shape reused for `Config.ClaudeSessionsDir` in 1.2b-A.
- **One mutation seam, multiple drivers** ‚Äî `Pool.RotateID` is the single place that swaps a session's in-memory id and persists. Startup reconciliation drives it from a directory scan; the live-detection watcher drives it from an fsnotify event. Defining the seam once with the lock-and-save contract baked in meant the second driver landed as a goroutine that calls a function, not a redesign.
- **Closures over primitives across package boundaries** ‚Äî when a downstream package (`internal/sessions/rotation`) needs to call back into its host (`internal/sessions`), express the contract as a struct of closures over primitive types (`string`, `int`) rather than an interface that references the host's domain types (`SessionID`). The downstream package then has zero knowledge of the host, and the type-safety conversion (`SessionID ‚Üî string`) happens exactly once, in the wiring site. Avoids the import cycle the obvious "watcher takes a `*Pool`" design would create.
- **Test seam via package var, not interface injection** ‚Äî `var newProbe = rotation.DefaultProbe` in `internal/sessions/pool.go`. Tests overwrite to inject a fake; production callers see no extra plumbing. Same shape as `allocatedTTL` and `probeRetryDelays` (test-tunable knobs that aren't config). Cheaper than threading a factory through `Config` for a single test surface.
- **Errgroup as the fan-out seam** ‚Äî `Pool.Run`'s `errgroup.WithContext` ties the bootstrap supervisor and the rotation watcher; Phase 1.1's N-session fan-out reuses the same cancellation+error semantics. The 1.1a-A1 (#72) seam is the natural shape: stash `*errgroup.Group` + `context.Context` on the host struct under its existing RWMutex, expose an unexported helper that RLock-snapshots both and calls `g.Go` off-lock, and pair it with a sentinel error (`ErrPoolNotRunning`) for before/after-`Run` calls. The helper is the join point post-`Run` consumers (`Pool.Create`) plug into; the existing bootstrap fan-out is refactored onto the helper so the helper is exercised in production from day one rather than living dormant.
- **Two-state machine with explicit-verb wake-up** ‚Äî when a runtime entity needs an `idle ‚Üî live` lifecycle, prefer (a) a per-entity goroutine that owns both the state machine and the underlying resource lifecycle, (b) an explicit `Activate(ctx)` call rather than implicit wake on resolve/lookup, and (c) a closed-on-state broadcast channel + buffered(1) signal channel for the wake-up handshake. Implicit wake-on-lookup tangles read-side concerns (status reads paying respawn latency); explicit Activate keeps every caller's contract sharp. Used for `Session.Activate` (1.2c-A); will be reused by future per-session lifecycles (router, bind/unbind).
- **Lock order `Pool.mu ‚Üí Session.lcMu` with release-before-callback** ‚Äî when a callee needs to call back into the host's lock-protected persistence (`Pool.persist` from `Session.transitionTo`), release the inner lock *before* the callback so the host's re-acquire of the inner lock inside its serialization path doesn't deadlock. Document the order at every site that takes both. Same shape would apply to any "child mutates state then asks parent to persist" coupling.
- **Dedicated outer mutex for "cap-shaped" critical sections** ‚Äî when a sequence (cap-check + victim eviction + new spawn) must be atomic but its inner steps already take the host's main mutex (and may re-take it via persistence callbacks), introduce a *separate* outer mutex (`Pool.capMu` for #41) rather than holding the main mutex (`Pool.mu` write) across the whole sequence. The outer mutex serialises the policy decision; the inner work continues to take/release the main mutex normally. Avoids both the deadlock (re-entrancy on `Pool.mu` via `persist`) and the contention (blocking unrelated `Pool.mu` reads for the eviction's bounded window). Lock order documented; outer mutex is never re-acquired by callees.
- **"Zero cost when unset" as a hot-path discipline** ‚Äî when a feature is configurable and most operators leave it unset (`Config.ActiveCap == 0`), gate ALL of its bookkeeping behind the early-return branch ‚Äî including the new mutex it needs. The uncapped `Pool.Activate` doesn't even take `capMu`, doesn't iterate `sessions`, doesn't read `lastActiveAt`. The branch is one `if cap <= 0 { return ‚Ä¶ }` at the top of the cap-only path. The property "the previous phase's behaviour is byte-identical with the new feature unset" is paid for by being *strict* about what happens before the early return.
- **Id-or-empty as the failure-mode discriminator for "mint, persist, activate" primitives** ‚Äî when a constructor-shaped method returns `(ID, error)` and has multiple failure points (rng, validation, persist, activate, supervise), make the returned id the caller's discriminator: empty id ‚áí "nothing was persisted, nothing to clean up"; non-empty id ‚áí "entry is on disk, decide what to do (retry the rest, accept eventual lifecycle, leave for next process start)." Callers distinguish further with `errors.Is` against named sentinels. Cleaner than a multi-valued status struct, and the empty/non-empty rule is checkable at every call site without deconstructing the error chain. Used for `Pool.Create` (1.1a-A2). The same shape suits any future "create-then-activate" seam where partial completion is recoverable.
- **Per-session template fields snapshotted at construction, read-only after** ‚Äî when a Pool/Registry needs a "shape of new entries" template (claude bin, work dir, backoff timings, bridge presence), capture it as an unexported field on the host (`Pool.sessionTpl`) at `New()` time and treat it as immutable thereafter. Per-create call clones the slice fields (`slices.Clone(tpl.ClaudeArgs)`) and mints fresh resources where sharing would be wrong (`supervisor.NewBridge(p.log)` per session in service mode). No lock needed for reads; mutation is impossible by construction. Two ~15-line struct constructions (bootstrap in `New`, fresh entry in `Create`) with different lifecycle-state defaults is clearer than a `lcState lifecycleState` parameter on a shared helper ‚Äî accepted intentionally.
- **Side-fd + Close to drain a blocked Read goroutine ‚Äî only works with `O_NONBLOCK` on character devices.** When a goroutine is blocked in `io.Copy`/`Read` on a process-wide fd (e.g. `os.Stdin`) with no way to wake it, the standard fix is to open a parallel fd to the same underlying device, copy from *that* fd, and `Close` it on cleanup. For sockets and pipes the runtime poller mediates Reads automatically; for character devices like `/dev/tty` you must pass `os.O_RDONLY|syscall.O_NONBLOCK` so `Read` returns `EAGAIN` and the goroutine parks on the poller ‚Äî only then does `Close` wake it via `runtime_pollUnblock`. Plain blocking `open` produces an fd whose in-flight `syscall.Read` cannot be interrupted by `Close` on another goroutine. Used for the foreground-mode input bridge in `internal/supervisor` (#78); applies to any "wake a stuck reader by closing a side fd" pattern.
- **Symmetric channel pairs for two-way state-machine signalling** ‚Äî when a two-state machine (`active` / `evicted`) needs both *enter-state* and *exit-state* external triggers, mirror the channel pair: `activateCh` (buffered 1 signal) + `activeCh` (closed-on-active broadcast) gets a sibling `evictCh` (buffered 1 signal) + `evictedCh` (closed-on-evicted broadcast). Both pairs follow the same snapshot-under-lock-then-wait discipline. The state-transition code closes one channel and re-opens the other in lockstep. One pattern, two directions.
