# Project Memory ŌĆö Pyrycode

Repo-level session memory. Read this at the start of every session.

## What's Built

### Codebase (Phase 0)
- **Supervisor core** ŌĆö PTY spawn via `creack/pty`, raw-mode stdin/stdout bridging in foreground mode, Bridge-mediated I/O in service mode, exponential backoff restart with stability reset, `--continue` injection on restart for session persistence
- **SIGWINCH forwarding** ŌĆö terminal resizes propagate from controlling terminal to child PTY in foreground mode (`internal/supervisor/winsize.go`) and in service mode via the live-resize loop landed across #136 (`Bridge.Resize` seam) + #137 (`VerbResize` wire + server applier) + #133 (client-side `startWinsizeWatcher` in `pyry attach`)
- **Control plane** ŌĆö Unix domain socket (`~/.pyry/<name>.sock`, 0600), line-delimited JSON protocol, verbs: `status`, `stop`, `logs`, `attach`
- **CLI transparency** ŌĆö unknown args forward verbatim to claude; pyry's own flags use `-pyry-*` prefix; `-pyry-name` plus `PYRY_NAME` env var for named multi-instance
- **Graceful shutdown** ŌĆö SIGINT/SIGTERM cancel the supervisor context, child is killed via `exec.CommandContext`, socket removed on exit
- **Service configs** ŌĆö systemd user unit (`systemd/pyry.service`), macOS launchd plist (`launchd/dev.pyrycode.pyry.plist`)
- **~1700 source + ~1100 test Go lines** as of late Apr 2026, 10+ PRs merged

### Codebase (Phase 1.0, tickets #28 + #29)
- **`internal/sessions` package** ŌĆö `SessionID` (UUIDv4 via `crypto/rand`, stdlib only), `Session` (wraps one `*supervisor.Supervisor` + optional `*supervisor.Bridge`), `Pool` (single-bootstrap registry with `RWMutex`-protected map). Sentinel errors `ErrSessionNotFound`, `ErrAttachUnavailable`. `Pool.Lookup("")` resolves to the bootstrap entry ŌĆö the seam Phase 1.1's `Request.SessionID` plugs into.
- **Production consumers wired (#29)** ŌĆö `cmd/pyry/main.go` constructs `*sessions.Pool` (with the supervisor.Config template inside `SessionConfig`); `internal/control` consumes a single `SessionResolver` interface (replaces Phase 0's `StateProvider` + `AttachProvider` pair). A 5-line `poolResolver` adapter in `cmd/pyry` bridges `Pool` ŌåÆ `SessionResolver` (covariant-return workaround). Wire protocol unchanged; `pyry status`/`stop`/`logs`/`attach` byte-identical to Phase 0. Foreground-mode attach error string preserved verbatim via `errors.Is(err, sessions.ErrAttachUnavailable)` mapping in `handleAttach`.
- See [knowledge/features/sessions-package.md](knowledge/features/sessions-package.md), [knowledge/features/control-plane.md](knowledge/features/control-plane.md), and [ADR 003](knowledge/decisions/003-session-addressable-runtime.md).

### Codebase (test infra, ticket #68)
- **`internal/e2e` harness ŌĆö spawn + cleanup primitive** ŌĆö build-tag-isolated package (`//go:build e2e`, invoked via `go test -tags=e2e ./internal/e2e/...`) that builds pyry once per test process (`sync.Once`; `PYRY_E2E_BIN` short-circuits for CI), spawns it in a `t.TempDir()` `$HOME` with `/bin/sleep infinity` as the supervised "claude" and `-pyry-idle-timeout=0`, polls the Unix socket via `os.Stat` + `net.Dial` (5s deadline, 50ms gap, doneCh short-circuit on early exit), and tears down via `t.Cleanup`: SIGTERM ŌåÆ 3s grace ŌåÆ SIGKILL ŌåÆ 1s grace ŌåÆ `os.Remove(socket)`. Three exported names: `Harness`, `Start(t)`, plus the struct fields (`SocketPath`, `HomeDir`, `PID`, `Stdout`, `Stderr`).
- **`HOME` redirection contains every ~-relative path with one env var** ŌĆö pyry resolves the socket, registry, and claude sessions dir all under `os.UserHomeDir()`, so `t.TempDir()` as `$HOME` isolates the daemon completely. Belt-and-suspenders `-pyry-socket=<HomeDir>/pyry.sock` flag for the load-bearing path; no new `-pyry-registry` flag needed. `PYRY_NAME` is stripped from the child env so the operator's shell alias can't leak in.
- **Failure-injection verification via test-binary re-exec** ŌĆö `t.Fatal` in a same-process `t.Run` subtest propagates to the parent and ends the test before it can inspect leak state. `TestHarness_NoLeakOnFatal` re-execs `os.Args[0] -test.run=^TestInnerFatalChild$` with `PYRY_E2E_INNER_FATAL_OUT=<state-file>` so the failing path runs in a fresh process; the child writes (pid, socket) before `t.Fatal`; the parent reads the state file and asserts `processAlive(pid)` is false (`os.FindProcess` + `Signal(syscall.Signal(0))`, the POSIX zero-signal probe) and the socket is `fs.ErrNotExist`.
- **No `Option`s yet, no CLI-driver wrappers** ŌĆö `Start(t)` is the only constructor; AC's "or equivalent" parenthetical covers the fail-fast `t.Fatalf` shape. `Harness.Status()`/`Stop()`/`Attach()`/`Run(args...)`, options, the first feature e2e, and CI wiring are deferred to the #51 follow-up. Build-tag isolation means existing `go test ./...` CI keeps passing untouched.
- See [knowledge/features/e2e-harness.md](knowledge/features/e2e-harness.md).

- **Same `e2e_install` build tag as the Linux sibling.** Single `go test -tags=e2e_install ./internal/e2e/...` invocation covers both platforms in CI when intended. Default `go test ./...` compiles neither file.
- **Plist parsing via Apple's `plutil`, not Go's `encoding/xml`.** The plist's key/value alternation pattern is awkward to walk with the stdlib XML decoder. `plutil -extract EnvironmentVariables.PATH raw -o - <plist>` returns just the PATH string ŌĆö one shell-out, fails loudly on missing keys, no XML parser. `plutil` is part of `/usr/bin` on macOS; absence is a broken host, not a skip condition. ~30 lines of Go XML walking rejected as anti-simplicity.
- **`launchctl bootstrap` is async ŌĆö poll `launchctl print` for `state = running`.** Bootstrap returns once launchd has *accepted* the request; the job transitions to `state = running` asynchronously. Polling loop is 100ms-gap, 10s-deadline; failure dumps the last `print` output for diagnosis. `launchctl print` is a debug command whose format Apple has reserved the right to reformat ŌĆö `state = running` has been stable since macOS 10.10; if it ever breaks, the test fails loudly with a clear diagnostic.
- **Cannot isolate `$HOME` for the round-trip ŌĆö same shape as Linux.** `launchctl bootstrap gui/<uid>` runs services in the user's GUI domain with `HOME` inherited from the GUI session, not from the test process. Round-trip and cleanup-on-fatal use the operator's real `~/Library/LaunchAgents/`; `PathInheritance` (no launchd dependency) keeps `t.TempDir()` isolation. Cleanup helper is idempotent best-effort (`bootout` + `os.Remove(plist)`) and additionally wipes runtime artefacts: `~/.pyry/<name>` registry dir, `<name>.sock` socket file, and the launchd template's hardcoded `/tmp/pyry.<name>.{out,err}.log` log files (unlike systemd which streams to journald).
- **Skip discipline.** `skipIfNoLaunchctl` (defensive ŌĆö should never fire on macOS), `skipIfRoot` (`gui/<uid>` is non-root only; system-domain bootstrap requires root and is out of scope), `os.UserHomeDir()` failure ŌåÆ `t.Skip` (CI without HOME set).
- **Helper identifiers (`uniqueName`, `fatalNameEnv`, `fatalOutEnv`, `TestInstallFatalChild`) duplicated across the Linux and Darwin files with identical bodies.** The two files have non-overlapping build tags (`linux && e2e_install` vs `darwin && e2e_install`) so they never compile together ŌĆö duplicate identifiers are legal and the simplest path. Names mirror exactly so a reader switching between files relies on muscle memory. Extracting into a third platform-neutral file deferred until a third platform appears.
- **Open question ŌĆö headless macOS CI.** `gui/<uid>` requires a logged-in GUI session for that uid. GitHub `macos-latest` runners do have one; if not, the fallback is `launchctl bootstrap user/<uid>` (background-only domain). Defer until observed on real CI ŌĆö AC specifies `gui/<uid>`.

### Codebase (test infra, ticket #69)
- **`Harness.Run(t, verb, args...) RunResult` CLI driver** ŌĆö invokes the cached pyry binary with `<verb> -pyry-socket=<h.SocketPath> <args...>`, waits for it to exit (10s `context.WithTimeout`), returns `RunResult{ExitCode int, Stdout, Stderr []byte}`. Auto-injects the socket flag after the verb so callers don't thread it through; reuses `binPath` (set by `ensurePyryBuilt` under `sync.Once`) and `childEnv(h.HomeDir)`. Verb is positional because pyry dispatches subcommands on `os.Args[1]` ŌĆö encoding that into the signature prevents the `h.Run(t, "-pyry-socket=other", "status")` footgun. Caller-side override is naturally available via Go `flag`'s "last value wins" rule, no special case in the harness.
- **Failure asymmetry: non-zero exit returned, exec/timeout fatal** ŌĆö `*exec.ExitError` populates `RunResult.ExitCode` for the caller to assert on; non-`ExitError` exec failure (binary missing, fork error) and 10s deadline expiry both `t.Fatalf` because there's no useful test recovery. The 10s wrapper budget is comfortably above `pyry status`'s own 5s socket-dial timeout (`runStatus` ŌåÆ `control.Status`). No regression test for the deadline branch ŌĆö constructing a daemon that hangs `pyry status` for >10s requires real-claude unresponsiveness or test-only socket injection, both more invasive than the safety net buys; per evidence-based fix selection, defer until observed.
- **Struct over tuple for the result type** ŌĆö `RunResult` future-proofs for `Duration`/`Combined`/`OOMed` additions without call-site churn; named fields prevent the `[]byte` mix-up between stdout and stderr that a positional tuple invites. Cost is one type declaration; worth it.
- **`TestStatus_E2E` ŌĆö first feature-flavoured e2e** ŌĆö spawns the daemon, runs `pyry status`, asserts exit 0 + `bytes.Contains(r.Stdout, []byte("Phase:"))`. `"Phase:"` is the leading literal in `runStatus`'s output (`fmt.Printf("Phase:         %s\n", resp.Phase)`) and is stable across phase values, restart counts, and future field additions; asserting on the *value* would couple the test to claude-child startup timing, which is exactly what `/bin/sleep infinity` was chosen to avoid. `pyry version` was rejected as the proof-of-life verb because it short-circuits in `main.go` before parsing flags ŌĆö wouldn't exercise the socket plumbing the harness sells.
- **Package doc updated with copy-pasteable CLI-driver example** ŌĆö replaces the bare smoke pattern from #68; godoc now surfaces a `Start(t) ŌåÆ Run(t, "status") ŌåÆ assert` recipe that downstream session-verb tickets (#52, #54, #55, #56) can copy.
- See [knowledge/features/e2e-harness.md ┬¦ CLI Driver](knowledge/features/e2e-harness.md).

### Codebase (test infra, ticket #52)
- **`internal/e2e/cli_verbs_test.go` ŌĆö e2e coverage for the remaining shipped non-interactive verbs.** Four tests under `//go:build e2e`: `TestStop_E2E` spawns the daemon, runs `pyry stop`, asserts exit 0 + `bytes.Contains(r.Stdout, []byte("stop requested"))`, then bounded-polls (3s deadline, 50ms gap) until both `!processAlive(pid)` AND `os.Stat(sock)` returns `fs.ErrNotExist` ŌĆö both conditions in the same iteration so the test doesn't race the supervisor's deferred socket cleanup. `TestStatus_E2E_Stopped` uses `RunBare("status", "-pyry-socket="+bogusSock)` against a fresh `t.TempDir()` non-existent path: exit != 0, non-empty stderr, no `panic` / `goroutine ` / `runtime/` substrings (clean error, not a crash). `TestLogs_E2E` runs `pyry logs` against a healthy daemon and asserts exit 0 + non-empty `bytes.TrimSpace(r.Stdout)` ŌĆö the supervisor's in-memory ring captures startup lines, no specific wording asserted. `TestVersion_E2E` uses `RunBare("version")`: exit 0, output starts with literal `"pyry "` prefix, remaining token non-empty (`dev` in test builds, real version under `-ldflags`).
- **`RunBare(t, args...) RunResult` ŌĆö daemon-free sibling of `Harness.Run`.** Same binary cache (`ensurePyryBuilt`), same `runTimeout` (10s), same exit-code mapping ŌĆö but no daemon spawn, no auto-injected `-pyry-socket`, no `childEnv(h.HomeDir)` HOME redirection. Two motivated cases: verbs that don't touch the socket (`version` short-circuits in `main.go` before flag parsing), and negative tests against a known-bad socket path (avoids the spawn-then-stop-then-race-the-teardown ordering glue). The *only* harness API added in #52 ŌĆö `Harness.Stop()` mid-test, typed `Status()`/`Logs()` wrappers, etc. were explicitly declined to keep the harness surface minimal.
- **Bogus-socket vs. spawn-then-stop for stopped-status.** Spawn-then-stop-then-status needs the test to wait for the daemon to actually shut down (otherwise status hits a still-listening socket and succeeds, defeating the test). That's the same poll loop as `TestStop_E2E`, plus ordering glue, plus a second `Run` call. Bogus-socket exercises the same code path (`net.Dial` fails ŌåÆ error surfaces clean to stderr ŌåÆ non-zero exit) without any timing dependency. Strictly simpler, strictly more deterministic.
- **Negative-assertion vocabulary for "clean error, not a crash".** Three conservative substrings ŌĆö `panic`, `goroutine ` (Go's stack-trace header `goroutine N [state]:`), `runtime/` (Go runtime frames in tracebacks) ŌĆö catch panics and stack traces without coupling to the exact dial-failure wording (today: `pyry: status: ... connect: no such file or directory`; allowed to evolve). Same pattern fits any "verify the failure is clean" assertion across future verbs.
- **File split: `cli_verbs_test.go` vs. appending to `harness_test.go`.** New file. `harness_test.go` is about *harness behaviour* (smoke, no-leak-on-fatal, the canonical `TestStatus_E2E` proof-of-life); `cli_verbs_test.go` is about *CLI surface coverage*. `processAlive` stays in `harness_test.go` and is reused via package scope (no duplication, no move). Build tag identical: `//go:build e2e` (not `e2e_install`).
- **Production diff is zero. Test diff ~155 LOC across 2 files** (`harness.go`: +42 for `RunBare`; `cli_verbs_test.go`: +112 new file). Default `go test ./...` is unchanged ŌĆö both files are gated by build tags.

### Codebase (test infra, ticket #106)
- **`internal/e2e` restart primitive ŌĆö `StartIn` + `Stop`.** `StartIn(t, home)` is the new workhorse: same body as the old `Start`, but uses the caller-supplied directory instead of allocating `t.TempDir()`. `Start(t)` becomes a one-line `return StartIn(t, t.TempDir())` ŌĆö every existing call site unchanged. `Stop(t)` is a public wrapper around the internal `teardown` (name kept private to make the public/private split obvious); `cleanupOnce` (existing `sync.Once`) makes it idempotent with `t.Cleanup` ŌĆö whichever fires first wins, the other is a no-op. Two ~10-line additions in `harness.go`; net public surface grew by exactly two names. No `Options` struct, no "respawn-in-place" method on `Harness` ŌĆö adding either today would be speculative; struct migration is mechanical and non-breaking when a second knob lands.
- **`TestE2E_Restart_PreservesActiveSessions` ŌĆö Phase 1.2's restart-survival proof at the binary boundary.** Pre-writes a registry with two `lifecycle_state: active` entries (one bootstrap, one non-bootstrap, hand-written canonical UUIDv4-shaped strings) ŌåÆ `StartIn` ŌåÆ `Stop` ŌåÆ `StartIn` against the same HOME ŌåÆ asserts `version` field, session count, per-session `lifecycle_state` and `bootstrap` flag preserved. Deliberately *not* asserted: byte-identity of the file (`MarshalIndent` output is allowed to evolve), `LastActiveAt` equality (would couple to today's no-save invariant in a way that breaks the moment a benign state-change ships).
- **Why this works against today's pyry without code changes.** Restart-time path against a pre-populated registry: `loadRegistry` reads ŌåÆ `pickBootstrap` selects the lone `bootstrap: true` entry; non-bootstrap entries are *not* materialised into `Pool.sessions` ŌåÆ `reg != nil` skips the cold-start save ŌåÆ `reconcileBootstrapOnNew` no-ops because `~/.claude/projects/<encoded-cwd>` doesn't exist under the test HOME ŌåÆ bootstrap enters `runActive`, idle timer disabled (`-pyry-idle-timeout=0`) ŌåÆ SIGTERM cancels ctx ŌåÆ `runActive` returns `ctx.Err` *before* `transitionTo(stateEvicted)`, no terminal save fires. Net: nothing in pyry calls `saveLocked` between pre-write and the second `loadRegistry` ŌĆö the on-disk file is byte-stable across the restart. The non-bootstrap entries persist *because pyry doesn't touch them*, not because pyry materialises them. The test locks in the no-save-without-state-change invariant; future tickets that materialise non-bootstrap entries will need to preserve their lifecycle state across restart explicitly, and this test will catch any regression in either direction.
- **`os.MkdirTemp` over `t.TempDir()` for the HOME ŌĆö Unix-socket `sun_path` budget.** `sun_path` caps at 104 bytes on macOS (108 on Linux). `t.TempDir()` embeds the test name into its path; `TestE2E_Restart_PreservesActiveSessions` + `pyry.sock` overflows the macOS budget. `os.MkdirTemp("", "pyry-rs-*")` + `t.Cleanup(os.RemoveAll)` keeps the prefix tiny. Tests with short names are unaffected; only path-budget-tight cases need to switch. See `lessons.md ┬¦ Unix-socket sun_path limits and t.TempDir()`.
- **Same socket path across both spawns is the realistic case.** Both `StartIn` calls derive `socket := filepath.Join(home, "pyry.sock")`. The second daemon's `Server.Listen` handles a stale socket file via dial-probe ŌåÆ ECONNREFUSED ŌåÆ `os.Remove` ŌåÆ `net.Listen`; no test-level coordination needed. By the time `Stop` returns, `cmd.Wait` has reaped the first process and the listener fd is closed; ECONNREFUSED is deterministic. The defensive `os.Remove(h.SocketPath)` in teardown belt-and-suspenders the SIGKILL path. Two harnesses (`h1`, `h2`) own independent `cleanupOnce`/`doneCh`/`cmd`; `t.Cleanup`'s LIFO order means `h2.teardown` fires first against the live second daemon, then `h1.teardown` no-ops.
- **File split: `restart_test.go`, not extending `cli_verbs_test.go`.** The latter is *CLI surface coverage* (one test per shipped verb); this is *daemon-level disk-state survival* and doesn't drive a CLI verb. Mirrors the `harness_test.go` (mechanics) vs. `cli_verbs_test.go` (verb surface) split #52 established. Local `registryFile`/`registryEntry` mirror types are duplicated intentionally ŌĆö `internal/sessions`'s on-disk types are unexported, and exporting them solely for one test would invert the dependency direction. Schema is small and stable; if a field is added, the mirror grows it too.
- **Production diff is zero. Test diff ~184 LOC across 2 files** (`harness.go`: +44 for `StartIn` + `Stop` + package doc; `restart_test.go`: +140 new file). Default `go test ./...` unaffected.

### Codebase (test infra, ticket #107)
- **Two more restart-survival e2e tests on top of #106's `StartIn` + `Stop` primitive ŌĆö no new harness API.** `TestE2E_Restart_PreservesEvictedSessions` pre-writes a registry with a bootstrap-active anchor + a non-bootstrap `lifecycle_state: "evicted"` entry + a non-bootstrap-active control, cycles `StartIn ŌåÆ Stop ŌåÆ StartIn`, and asserts the evicted entry is *still* `"evicted"` on disk after the second daemon is ready (catches silent warm-promotion to active during bootstrap ŌĆö the property `parseLifecycleState` and `pool.New` rely on for #38's warm-start semantics). `TestE2E_Restart_LastActiveAtSurvives` pre-writes three sessions whose `lastActiveAt` values are spread by 10 min and 1 hour, restarts, asserts every value roundtrips via `time.Time.Equal` (catches a re-stamp to `time.Now()` during bootstrap that would silently break the cap-policy LRU order from #41). Both tests scoped to non-bootstrap survival; bootstrap-evicted (warm-starting the bootstrap *itself* in `stateEvicted`) deferred to its own ticket ŌĆö functionally distinct path (`runEvicted` instead of spawning the child).
- **`newRegistryHome(t)` file-local helper ŌĆö rule of three.** With three restart-survival tests sharing the same four-line HOME bootstrap (`os.MkdirTemp` for sun_path safety, `t.Cleanup(RemoveAll)`, `mkdir -p <home>/.pyry/test`), #107 extracted the prefix into a file-local helper and refactored `TestE2E_Restart_PreservesActiveSessions` to call it. Intentionally not promoted to `harness.go`'s public surface ŌĆö three callers ŌēĀ a public API. `registryEntry` / `registryFile` mirror types and `writeRegistry` / `readRegistry` / `mustReadFile` from #106 stay file-local and unchanged.
- **Equality, not byte-identity, for `LastActiveAt`.** `time.Time.Equal` per entry, not byte-equal on the file. `Equal` accepts today's exact roundtrip *and* tolerates a future re-encode through `time.Now().UTC()` (strips monotonic, preserves wall-clock ŌĆö the AC's "tight tolerance"); rejects loudly when the value is re-stamped to `time.Now()` because the 10-min / 1-hour spread dwarfs any plausible JSON-roundtrip drift. Asserting on byte-identity of the file would couple to `MarshalIndent` formatting (brittle); per-field `Equal` is durable.
- **Monotonic-clock trap: capture "want" via the same JSON trip the daemon takes.** `time.Time` written via `MarshalIndent` retains monotonic-clock state in the original Go value but strips it after the JSON unmarshal trip. ### Codebase (Phase 1.1+ resize wire + applier, ticket #137)
- **`VerbResize` + `ResizePayload` + `handleResize` ŌĆö live SIGWINCH propagation rides a side-channel one-shot conn, not a framed escape in the attach byte stream.** New verb on the JSON protocol; client dials a *fresh* control connection per resize, sends one `Request{Verb:"resize", Resize:&ResizePayload{SessionID, Cols, Rows}}`, reads one `Response{OK:true}` ack, closes ŌĆö same lifecycle as `VerbStatus`/`VerbStop`/`VerbLogs`. The client has nothing useful to do with a "resize failed" response ŌĆö the next SIGWINCH will re-emit; the user's terminal already shows the new size locally.

- **No new mutexes, channels, or goroutines.** The accept loop's existing one-goroutine-per-conn pattern absorbs resize handlers; each completes in ~1ms (one resolver lookup + one `pty.Setsize`). `Bridge.ptyMu` (#136) serialises concurrent resizes; last-write-wins matches the kernel's `ioctl(TIOCSWINSZ)`.

### Codebase (Phase 1.1+ live-resize client trigger, ticket #133)
- **`startWinsizeWatcher` + `readTerminalSize` inside `internal/control/attach_client.go` ŌĆö closes the live-resize loop end-to-end.** `Attach` installs a SIGWINCH watcher *after* the handshake ack succeeds; the watcher converts each `SIGWINCH` into a `SendResize(ctx, socketPath, sessionID, cols, rows)` on a fresh control conn (independent of the long-lived attach conn). With #136 (`Bridge.Resize` seam) and #137 (`VerbResize` wire + server applier) already shipped, the full path is now: terminal resize ŌåÆ kernel SIGWINCH ŌåÆ watcher reads `pty.GetsizeFull(os.Stdin)` ŌåÆ `SendResize` ŌåÆ `handleResize` ŌåÆ `Session.Resize` ŌåÆ `Bridge.Resize` ŌåÆ `pty.Setsize` ŌåÆ child redraws. No more "detach and reattach to update."
- **No new exported type, no new file in `internal/control` for the watcher itself.** `startWinsizeWatcher` is a package-internal helper next to `Attach`; the watcher's lifetime is exactly `Attach`'s stack frame, so a struct receiver would inflate the change for no benefit. The helper takes its dependencies as function parameters (`read terminalSizeReader`, `send func(ctx, cols, rows) error`) ŌĆö keeps `os` and the network out of the watcher body, and lets unit tests inject stubs without a package-level `var` indirection (which would couple test substitution to global state shared with sibling tests).
- **Synchronous teardown is the load-bearing guarantee for AC#3.** `stop` does (1) `signal.Stop(sigCh)` to unsubscribe, (2) `close(done)` to wake the goroutine's `select`, then (3) blocks on `<-gone` until the goroutine has actually exited. No goroutine or signal subscription outlives the `defer stopWinsize()` call site ŌĆö leak-freeness is *structural*, not *eventual*. If `SendResize` is in flight when `stop` fires, `<-gone` waits out the in-flight call: detach cannot race a half-completed resize.
- **No initial-size priming after the handshake.** AC#2 prohibits regressing the handshake `AttachPayload.Cols`/`Rows` path; the first SIGWINCH (or, in the absent-resize case, the next one) covers any change between handshake and first event. Window is microseconds; priming would push a duplicate resize in the steady-state-no-change case for no benefit.
- **`os.Stdin` directly, not a fresh `*os.File` wrapper around `os.Stdin.Fd()`.** Same convention `internal/supervisor/winsize.go` follows for the same reason: a wrapper `*os.File` is finalizable, and finalizer-driven `close(2)` racing a subsequent `open(2)` is the classic fd-reuse race that makes `pty.Setsize` target the wrong file. Lessons.md ┬¦ Closing a fd to interrupt a goroutine's Read requires O_NONBLOCK has the adjacent shape; this case is its read-side cousin.
- **Best-effort posture, no logging.** `SendResize` errors silently dropped via `_ = send(ctx, cols, rows)`; the next SIGWINCH retries. Mirrors `SendResize`'s godoc and `handleResize`'s OK-on-seam-error posture (#137). The watcher logs nothing ŌĆö the package is library-level, and SendResize errors are uninteresting in practice (either "daemon went away" ŌĆö already surfaced by the attach conn dropping ŌĆö or transient hiccup ŌĆö corrected by the next resize).
- **Burst coalescing falls out of the buffered-cap-1 sigCh.** SIGWINCH arrivals during a `SendResize` collapse to one queued event; after `SendResize` returns, the goroutine reads the queued one and emits a *fresh* resize with the current size (re-read at signal time, not stale). Same coalescing shape as `supervisor/winsize.go`. Server-side `Bridge.ptyMu` serialises and last-write-wins matches `ioctl(TIOCSWINSZ)`'s native semantics.
- **Concurrency: three goroutines inside `Attach`, none share state.** Output copier (existing), input pump (calling goroutine, in `copyWithEscape`), and the new watcher. Watcher does NOT touch the attach `conn` ŌĆö each SIGWINCH is its own short-lived dial via `SendResize`. A malformed resize, daemon-side seam error, or transient dial failure lives entirely on the resize conn and never disturbs the byte stream on the attach conn (the property is structural, mirroring #137's design).
- **`defer stopWinsize()` ordering vs. `defer conn.Close()`.** `defer conn.Close()` is registered first (top of `Attach`), `defer stopWinsize()` second (after handshake ack). LIFO: stop the watcher first, then close the conn. Operationally irrelevant since resize uses its own conn anyway, but documents intent.
- **Caveat rewrite.** `internal/control/attach_client.go:25-27` Phase-0 sentence ("Subsequent live resize events are not propagated in Phase 0 ŌĆö detach and reattach to update.") **rewritten** to identify the SIGWINCH watcher as the live producer and `Bridge.Resize` as the seam. `internal/control/protocol.go` line 62 parenthetical updated from "deferred to #133" to point at `startWinsizeWatcher`.

### Codebase (supervisor fix, ticket #78)
- **Foreground-mode stdin reader leak fixed** ŌĆö `runOnce` now opens `/dev/tty` (`os.O_RDONLY|syscall.O_NONBLOCK`) as a side fd for the input bridge instead of reading `os.Stdin`. On child exit, the per-cycle `input.Close()` wakes the in-flight `Read` via the Go runtime poller; the goroutine drains; nothing accumulates on `os.Stdin`'s `fdMutex`. Drain semantics flip from "drain one of two with timeout" to "drain both with timeout"; total worst-case `runOnce` return latency is `2 ├Ś goroutineDrainTimeout` (~200ms). Bridge-mode branch (`Config.Bridge != nil`) is unchanged.
- **`openTTYInput` + `stdinFallback`** ŌĆö unexported, internal to `internal/supervisor`. Fallback path (`stdinFallback{}` ŌĆö `Read` from `os.Stdin`, no-op `Close`) is taken when `/dev/tty` cannot be opened (headless containers, etc.); preserves the legacy leak in that environment by design rather than breaking the process by closing `os.Stdin`. Foreground mode is TTY-only by construction (`term.MakeRaw(stdinFd)` already early-returns on a non-TTY), so the fallback is a defensive seam, not the production path.
- **Why O_NONBLOCK is load-bearing.** Plain `os.OpenFile("/dev/tty", os.O_RDONLY, 0)` produces a blocking fd; `internal/poll.(*FD).Read` calls `syscall.Read` directly and parks in the kernel, where POSIX `close(2)` on another goroutine is a no-op for the in-flight read. With `O_NONBLOCK`, `Read` returns `EAGAIN`, the runtime parks the goroutine on the poller, and `Close` calls `runtime_pollUnblock` to wake it with `os.ErrClosed`. Drop the flag and the leak comes back.
- **Targeted test** ŌĆö `TestSupervisor_Foreground_NoStdinReaderLeak` runs ~3 child-restart cycles and asserts `runtime.NumGoroutine()` does not grow by more than 2; skips with `t.Skipf("no controlling tty: %v", err)` when `/dev/tty` cannot be opened (CI without a TTY). Verified clean under `go test -race -count=20`.
- **Unblocks the flake stack.** The leak was the root cause of `TestPool_ActiveCap_RaceConcurrentActivate` deadlock (#41), `TestPool_Run_StartsWatcher` flake (#39), and `TestPool_New_Reconciles_ColdStart_PicksNewestImmediately` flake (#73). Bridge-mode-fixture migration in session tests is now belt-and-suspenders rather than required.

### Codebase (Phase 1.1d-A1, ticket #94)
- **`Pool.Remove(ctx context.Context, id SessionID) error` write primitive + `ErrCannotRemoveBootstrap` sentinel** ŌĆö terminates a session's claude process and drops its registry entry. The on-disk JSONL is **not** touched (disposition lands in 64-A2 / #95). Bootstrap is rejected via the new sentinel; unknown ids reuse `ErrSessionNotFound`. The internal seam the future `pyry sessions rm` CLI verb (#65) consumes instead of touching processes or `sessions.json` directly.
- **Delete-then-evict ordering is the load-bearing invariant.** `Pool.mu` covers the in-memory `delete(p.sessions, id)` + `saveLocked()`, then is released **before** calling `sess.Evict(ctx)`. The alternative (holding `Pool.mu` across `Evict`) deadlocks: the lifecycle goroutine's `transitionTo` reacquires `Pool.mu` via `Pool.persist`. Releasing `Pool.mu` after the in-memory delete commits is safe ŌĆö concurrent `Lookup`/`Activate`/`Rename`/`List` callers see the session as gone from that moment on, so no observer can witness a half-removed state and no re-spawn race exists. Same outer-mutex pattern as the cap-policy eviction (#41), here resolved by holding `Pool.mu` only during the cheap mutate phase rather than introducing a new outer lock.
- **Save-failure rollback mirrors `Pool.Rename`.** If `saveLocked` fails after the in-memory delete, restore `p.sessions[id] = sess` before unlocking. Disk and memory remain consistent at every observable point. `saveRegistryLocked`'s temp+rename discipline makes partial writes unreachable; the rollback is belt-and-suspenders for "destination dir vanished" / `ENOSPC`.
- **Termination reuses `Session.Evict`.** `Pool.Remove` does not re-implement SIGTERM/SIGKILL/grace logic ŌĆö `Session.Evict` already drives the supervisor's child via `exec.CommandContext` cancel ŌćÆ SIGKILL ŌćÆ `cmd.Wait` returns. The supervisor today does not have a SIGTERM grace window; SIGKILL is uncatchable, so no fallback path is needed (the AC's "SIGKILL fallback when child ignores SIGTERM" maps to "the existing kill-on-cancel terminates uncooperative children deterministically", verified in `TestPool_Remove_TerminatesUncooperativeChild` with a `trap "" TERM INT HUP; sleep 86400` shell child).
- **Lifecycle goroutine survives until pool shutdown.** After `Pool.Remove` returns, `sess.Run`'s loop transitions to `runEvicted` and parks on `activateCh` / `ctx.Done()`. The session is no longer reachable via `Pool.sessions`, so no caller can signal `activateCh`; the goroutine survives until `Pool.Run`'s `runCtx` cancels at pool shutdown. Bounded resource cost (one orphan goroutine + ~1KB `*Session` per `Remove` per pool lifetime) ŌĆö operationally invisible for pyrycode's expected workload. Per evidence-based fix selection, no per-session terminate signal until observed.
- **No `Session.lcMu` taken; lock order unchanged.** `Pool.Remove` does not read `lcState` / `lastActiveAt` / `attached`. Lock-order graph (`Pool.capMu ŌåÆ Pool.mu ŌåÆ Session.lcMu`) is unchanged. Empty-id is *not* the bootstrap shorthand it is in `Pool.Lookup` ŌĆö empty falls through to "not in map" and returns `ErrSessionNotFound`; destructive operations require an explicit id.

No new lock-order edges (`Pool.mu ŌåÆ Session.lcMu` and `Pool.capMu ŌåÆ Pool.mu ŌåÆ Session.lcMu` unchanged) ŌĆö the helper does not touch `Session.lcMu`.
- **Why fields-on-Pool, not a separate guard struct** ŌĆö `Pool.mu` already protects every other "what's in this pool right now" read. A second mutex would double the lock count without buying anything; RWMutex matches the access pattern (N concurrent supervise readers, one-shot writes by `Run` setup/teardown). Concurrent `supervise` callers don't contend with each other.
- **Race-with-teardown is benign** ŌĆö a `supervise` call interleaved with the deferred clear either (a) acquires RLock first, schedules onto a group whose ctx is about to cancel, and `Session.Run` exits via the existing shutdown path, or (b) acquires after, observes `nil`, returns the sentinel. No silent leak, no deadlock.
- **Defensive wrap on the bootstrap call** ŌĆö `Pool.Run` does `if err := p.supervise(bootstrap); err != nil { return fmt.Errorf("sessions: supervise bootstrap: %w", err) }` even though the precondition is established two lines above. Cheap insurance against a future reordering bug.
- See [knowledge/features/sessions-package.md ┬¦ Supervisor handle](knowledge/features/sessions-package.md).

### Codebase (Phase 1.2c-B, ticket #41)
- **Concurrent active cap (LRU eviction)** ŌĆö `Config.ActiveCap` bounds the number of concurrently running claudes. `<= 0` is unset (uncapped, byte-identical to 1.2c-A); `>= 1` enforces. When activating one more session would exceed the cap, the LRU peer (oldest `lastActiveAt` among currently-active sessions, excluding the target) is evicted via the new public `Session.Evict` primitive before the new spawn proceeds. **No CLI flag yet** ŌĆö Phase 2.0 will wire `-pyry-active-cap` when the auto-mint workload makes it load-bearing.
- **`Session.Evict(ctx)` public primitive** ŌĆö symmetric to `Activate`. Force-eviction: unlike the idle timer, it does **not** defer for `attached > 0`. The cap is a hard limit; an attached caller will see EOF on its bridge. Phase 2.0 may add an attach-aware filter to the LRU candidate set; not in scope for #41. Pairs `evictCh` (buffered 1 signal) with `evictedCh` (closed-on-evicted broadcast), mirroring `activateCh`/`activeCh`.
- **`Pool.capMu` outermost lock** ŌĆö taken only on the cap path (`activeCap > 0`). Lock order: `Pool.capMu ŌåÆ Pool.mu ŌåÆ Session.lcMu`. Serialises cap-check + victim eviction + new spawn so concurrent `Activate`s can't both observe `active < cap` and both proceed. Never re-acquired by callees, so `Session.Evict`'s callback into `Pool.persist` (which takes `Pool.mu`) is deadlock-free. The uncapped path doesn't take `capMu` at all ŌĆö "byte-identical when unset" includes "doesn't even take the cap mutex".
- **`Pool.pickLRUVictim`** ŌĆö O(n) iteration over `pool.sessions` under `Pool.mu.RLock`, reads each `Session.lcState`/`lastActiveAt` under `lcMu`, returns the oldest-`lastActiveAt` peer (excluding target). Premature optimisation rejected: a heap/linked-list would defeat the zero-cost-when-unset property and add a counter-vs-truth drift class of bug for an O(n) cost that doesn't matter at pyry's scale (Ōēż 100s of sessions).
- **`Session.touchLastActive`** ŌĆö bumps `lastActiveAt` in-memory only on a cap-path `Activate` against an already-active session. **Not persisted** (the registry's `lastActiveAt` only flushes on transitions). Keeps in-memory LRU ordering reflective of the most recent touch.
- **`Pool.Activate` is now the spawn-path entry, not just a thin Lookup wrapper.** Phase 2.0's router calls this. Cap-unset path: one early-return branch and forward to `Session.Activate`. Cap-set path: cap check + victim eviction + new spawn under `capMu`.
- See [knowledge/features/idle-eviction.md](knowledge/features/idle-eviction.md), [ADR 006](knowledge/decisions/006-concurrent-active-cap-lru.md).

### Codebase (Phase 1.2c-A, ticket #40)
- **Idle eviction + lazy respawn** ŌĆö per-`*Session` lifecycle goroutine drives a two-state machine (`active` Ōåö `evicted`). When `attached == 0` and the idle timer fires, the supervisor's inner ctx is cancelled and claude exits cleanly; the JSONL on disk is preserved. `Session.Activate(ctx)` respawns the supervisor pointing at the same JSONL on disk so the conversation continues. `Session.Run` rewritten from one-line supervisor delegate into a `runActive` / `runEvicted` loop; `Pool.Run` shape **unchanged** (multi-session fan-out is Phase 1.1's job).
- **Activity = attach state** ŌĆö while `attached > 0` the timer re-arms on fire (poll-with-grace; eviction may overshoot the configured timeout by up to one window). `last_active_at` is bumped on every state transition for the upcoming LRU follow-up's victim selection.
- **`-pyry-idle-timeout` flag** ŌĆö production default `15m`, `0` disables eviction entirely (test default + operator escape hatch). Wired through `Config.IdleTimeout` ŌåÆ `SessionConfig.IdleTimeout` (per-session override falls back to pool default).
- **Activate-before-Attach contract** ŌĆö `bridge.Attach` on an evicted session would block on the pipe forever. `internal/control.Session` interface gained `Activate(ctx)`; `handleAttach` calls it with a 30s timeout cap before binding the bridge. `handleStatus` does **not** activate (faithful `PhaseStopped` for evicted; no spurious wakeups from a poll). Wire response on respawn failure: `attach: activate: <err>`.
- **Registry `lifecycle_state` field** ŌĆö `"active"` / `"evicted"`, `omitempty` so the dominant active case stays off disk (preserves #34's idempotent-reload byte-stability). Old pyry binaries ignore the field; new pyry reading old files defaults to `"active"`. Bootstrap warm-starts in whatever state the registry says ŌĆö including starting in `evicted` if pyry was stopped while evicted.
- **Lock order: `Pool.mu ŌåÆ Session.lcMu`** ŌĆö `Session.lcMu` protects `lcState` / `attached` / `activeCh` / `lastActiveAt` (when read for the registry snapshot). `Session.transitionTo` releases `lcMu` before calling `Pool.persist` so `saveLocked`'s per-session re-acquire doesn't deadlock. `RotateID` mutates `session.id` without `lcMu` ŌĆö documented invariant (today's only callers run before any lifecycle goroutine observes the id).
- **Concurrency primitives** ŌĆö `activeCh` (closed-on-active broadcast) + `activateCh` (buffered 1 signal). `Activate` snapshots `activeCh` under `lcMu` *before* waiting so a concurrent evict-replace doesn't drop the wakeup. Concurrent `Activate`s collapse to one signal via the buffered send.
- See [knowledge/features/idle-eviction.md](knowledge/features/idle-eviction.md), [ADR 005](knowledge/decisions/005-idle-eviction-state-machine.md).

### Codebase (rotation fix, ticket #118)
- **Rotation-watcher path comparison resolves symlinks at construction.** `Watcher` gained an unexported `resolvedDir` (`filepath.EvalSymlinks(cfg.Dir)`, captured once in `New`). `handleCreate`'s match gate became `filepath.Clean(open) == filepath.Join(w.resolvedDir, base)`. Pre-#118 the gate compared `filepath.Clean(open)` to `filepath.Clean(fullPath)` from the fsnotify event ŌĆö `filepath.Clean` is lexical and never resolves symlinks, so on macOS where `/var ŌåÆ /private/var`, fsnotify's `/var/...` ev.Name and lsof's `/private/var/...` probe output never matched and rotation was silently dropped (symptom: session UUID stops updating after `/clear`). Affects every macOS user whose `~/.claude/projects/` lives behind a symlink (custom HOME, external-drive home) and every `t.TempDir`-based test on macOS ŌĆö surfaced by #55's e2e harness landing `t.TempDir()` under `/var/folders/...`.
- **Resolve once, not per event ŌĆö three properties.** (1) Zero per-event syscalls; (2) no race window between event and resolution (the directory's lifetime spans the watcher's, ducking the AC's "file unlinked between event and resolution" failure mode); (3) no API change ŌĆö `Config.Dir` stays as-is, `pool.go`'s call site untouched. `EvalSymlinks` failure at startup logs `Warn` and falls back to `cfg.Dir` unmodified ŌĆö the macOS-symlink case continues to fail in that specific run (no worse than today), startup proceeds.
- **`TestWatcher_DetectsRotationThroughSymlink` regression test.** Explicit `os.Symlink` from a side dir to the real sessions dir; watcher set up against the link; probe reports the *resolved* path (mimicking what `lsof` / `/proc/<pid>/fd` actually return). Fails on the pre-#118 code (gate rejects `Clean(real/...)` vs `Clean(link/...)` because they're different absolute paths); passes after the fix. Portable across Linux and macOS via the explicit `os.Symlink` ŌĆö Linux CI exercises the symlink branch even though Linux defaults don't put `t.TempDir` behind a symlink. Existing six tests that build a `dir` from `t.TempDir()` were updated to wrap with `filepath.EvalSymlinks(t.TempDir())` so probe-returned paths and the watcher's resolved view stay in the same canonical form.
- **Production diff: ~8 lines in `internal/sessions/rotation/watcher.go`** (one struct field, ~5 lines in `New`, one rewritten line in `handleCreate`). Test diff: ~30 LOC for the new test plus mechanical 2-line additions across the existing tests. Two files total. Zero consumer call sites touched.
- See [knowledge/features/rotation-watcher.md ┬¦ Symlink-resolved match path](knowledge/features/rotation-watcher.md), `lessons.md ┬¦ fsnotify reports as-watched, kernel probes report canonicalised`.

### Codebase (Phase 1.2a, ticket #34)
- **Session registry on disk** ŌĆö `~/.pyry/<sanitized-name>/sessions.json` (file 0600, dir 0700), sibling to the per-name socket. Schema: `version` (forward-marker), `sessions[]` with `id` / `label` / `created_at` / `last_active_at` / `bootstrap`. Default `encoding/json` decoder tolerates unknown fields for forward compat.

Warm start reads the bootstrap-marked entry, reuses its UUID + metadata, and does **not** rewrite the file (warm reload is not a state change). Malformed JSON is fatal at startup ŌĆö operator must fix or remove.

- **Stable disk ordering** ŌĆö `sortEntriesByCreatedAt` (then by `ID`) before write makes the file's byte content a deterministic function of the in-memory set, defeating Go's randomized map iteration and giving the AC's idempotent-reload guarantee. Degenerate for 1.2a's single entry; the property is paid for upfront ahead of 1.1.

## Patterns Established

- **Config struct pattern** ŌĆö all configuration in a single `Config` struct, defaults applied in `New()`
- **Context-based cancellation** ŌĆö `context.Context` flows through `Run()` ŌåÆ `runOnce()`, checked at every wait point
- **Structured logging** ŌĆö `log/slog` with injected logger, not a global
- **Exponential backoff with stability reset** ŌĆö backoff doubles on restart, resets to initial if child stayed up longer than `BackoffReset`
- **Deferred cleanup** ŌĆö `defer` for terminal restore, PTY close, signal stop
- **Empty ID resolves to default** ŌĆö `Pool.Lookup("")` returns the bootstrap session, so future `req.SessionID` fields can be added with no handler-side branching (old clients send empty, get the bootstrap; new clients send a real ID, get the right entry)
- **Introduce-then-rewire slicing** ŌĆö split #27 into #28 (new package + tests, no consumers) and #29 (mechanical consumer rewiring) to keep each PR focused
- **Consumer-side interface definition** ŌĆö `internal/control` defines the interfaces it consumes (`SessionResolver`, `Session`) rather than importing them from the producer package. Keeps `internal/sessions` free of control-plane concerns and lets tests fake the surface without exporting test seams from the producer.
- **Wire-string preservation via `errors.Is` mapping** ŌĆö when refactoring an error path that crosses package boundaries, map the new sentinel back to the old wire string explicitly (`if errors.Is(err, sessions.ErrAttachUnavailable) { ŌĆ” Phase 0 string ŌĆ” }`) rather than letting `fmt.Sprintf("%v", err)` change client output. Required when an AC says "byte-identical."
- **Atomic on-disk writes via temp + rename** ŌĆö for operator-recoverable JSON state files, `os.CreateTemp` in the same dir ŌåÆ encode ŌåÆ `f.Sync()` ŌåÆ `f.Close()` ŌåÆ `os.Rename(tmp, path)` makes the rename the commit point and partial files unreachable. `defer os.Remove(tmp)` cleans up orphans best-effort (no-op after successful rename). Don't bother with directory fsync unless real-world corruption is observed ŌĆö pyry's registry is recoverable, not a database.
- **Stable disk ordering for idempotent reload** ŌĆö sort serialized collections by a stable key (e.g. `created_at` then `id`) before writing. Defeats Go's randomized map iteration so saveŌåÆloadŌåÆsave round-trips are byte-stable, and makes load-twice-equals-load-once into a real property rather than a probabilistic one.
- **Persistence as a Config field, empty disables** ŌĆö `Config.RegistryPath` (or any "where to persist" string) defaulting to empty = no I/O lets unit tests construct a Pool with `t.TempDir()` or with persistence off entirely, with no test-only branches inside `New`. Production callers always pass a real path; the empty case is genuinely test-only. Same shape reused for `Config.ClaudeSessionsDir` in 1.2b-A.
- **One mutation seam, multiple drivers** ŌĆö `Pool.RotateID` is the single place that swaps a session's in-memory id and persists. Startup reconciliation drives it from a directory scan; the live-detection watcher drives it from an fsnotify event. Defining the seam once with the lock-and-save contract baked in meant the second driver landed as a goroutine that calls a function, not a redesign.
- **Closures over primitives across package boundaries** ŌĆö when a downstream package (`internal/sessions/rotation`) needs to call back into its host (`internal/sessions`), express the contract as a struct of closures over primitive types (`string`, `int`) rather than an interface that references the host's domain types (`SessionID`). The downstream package then has zero knowledge of the host, and the type-safety conversion (`SessionID Ōåö string`) happens exactly once, in the wiring site. Avoids the import cycle the obvious "watcher takes a `*Pool`" design would create.
- **Test seam via package var, not interface injection** ŌĆö `var newProbe = rotation.DefaultProbe` in `internal/sessions/pool.go`. Tests overwrite to inject a fake; production callers see no extra plumbing. Same shape as `allocatedTTL` and `probeRetryDelays` (test-tunable knobs that aren't config). Cheaper than threading a factory through `Config` for a single test surface.
- **Errgroup as the fan-out seam** ŌĆö `Pool.Run`'s `errgroup.WithContext` ties the bootstrap supervisor and the rotation watcher; Phase 1.1's N-session fan-out reuses the same cancellation+error semantics. The 1.1a-A1 (#72) seam is the natural shape: stash `*errgroup.Group` + `context.Context` on the host struct under its existing RWMutex, expose an unexported helper that RLock-snapshots both and calls `g.Go` off-lock, and pair it with a sentinel error (`ErrPoolNotRunning`) for before/after-`Run` calls. The helper is the join point post-`Run` consumers (`Pool.Create`) plug into; the existing bootstrap fan-out is refactored onto the helper so the helper is exercised in production from day one rather than living dormant.
- **Two-state machine with explicit-verb wake-up** ŌĆö when a runtime entity needs an `idle Ōåö live` lifecycle, prefer (a) a per-entity goroutine that owns both the state machine and the underlying resource lifecycle, (b) an explicit `Activate(ctx)` call rather than implicit wake on resolve/lookup, and (c) a closed-on-state broadcast channel + buffered(1) signal channel for the wake-up handshake. Implicit wake-on-lookup tangles read-side concerns (status reads paying respawn latency); explicit Activate keeps every caller's contract sharp. Used for `Session.Activate` (1.2c-A); will be reused by future per-session lifecycles (router, bind/unbind).
- **Lock order `Pool.mu ŌåÆ Session.lcMu` with release-before-callback** ŌĆö when a callee needs to call back into the host's lock-protected persistence (`Pool.persist` from `Session.transitionTo`), release the inner lock *before* the callback so the host's re-acquire of the inner lock inside its serialization path doesn't deadlock. Document the order at every site that takes both. Same shape would apply to any "child mutates state then asks parent to persist" coupling.
- **Dedicated outer mutex for "cap-shaped" critical sections** ŌĆö when a sequence (cap-check + victim eviction + new spawn) must be atomic but its inner steps already take the host's main mutex (and may re-take it via persistence callbacks), introduce a *separate* outer mutex (`Pool.capMu` for #41) rather than holding the main mutex (`Pool.mu` write) across the whole sequence. The outer mutex serialises the policy decision; the inner work continues to take/release the main mutex normally. Avoids both the deadlock (re-entrancy on `Pool.mu` via `persist`) and the contention (blocking unrelated `Pool.mu` reads for the eviction's bounded window). Lock order documented; outer mutex is never re-acquired by callees.
- **"Zero cost when unset" as a hot-path discipline** ŌĆö when a feature is configurable and most operators leave it unset (`Config.ActiveCap == 0`), gate ALL of its bookkeeping behind the early-return branch ŌĆö including the new mutex it needs. The uncapped `Pool.Activate` doesn't even take `capMu`, doesn't iterate `sessions`, doesn't read `lastActiveAt`. The branch is one `if cap <= 0 { return ŌĆ” }` at the top of the cap-only path. The property "the previous phase's behaviour is byte-identical with the new feature unset" is paid for by being *strict* about what happens before the early return.
- **Id-or-empty as the failure-mode discriminator for "mint, persist, activate" primitives** ŌĆö when a constructor-shaped method returns `(ID, error)` and has multiple failure points (rng, validation, persist, activate, supervise), make the returned id the caller's discriminator: empty id ŌćÆ "nothing was persisted, nothing to clean up"; non-empty id ŌćÆ "entry is on disk, decide what to do (retry the rest, accept eventual lifecycle, leave for next process start)." Callers distinguish further with `errors.Is` against named sentinels. Cleaner than a multi-valued status struct, and the empty/non-empty rule is checkable at every call site without deconstructing the error chain. Used for `Pool.Create` (1.1a-A2). The same shape suits any future "create-then-activate" seam where partial completion is recoverable.
- **Synthetic substitution at the read boundary, not at the renderer** ŌĆö when an on-disk record's empty field has a well-known operator-facing default (e.g. `"bootstrap"` for an unlabelled bootstrap session), do the substitution in the *read primitive* that returns the typed snapshot, not in the CLI/wire renderer that consumes it. The on-disk record stays unmutated; every future consumer (a JSON renderer, a TUI, an HTTP handler) gets the same UX guarantee for free without re-implementing the rule. Pair with a discriminator field (e.g. `Bootstrap bool`) so the substitution is a UX nicety rather than the only signal ŌĆö an operator who sets `Label="bootstrap"` on a non-bootstrap session is still distinguishable. Used for `Pool.List`'s bootstrap label (1.1b-A); same shape suits any "default for empty field" rule that crosses multiple consumers.
- **Per-consumer value type, not a shared "kitchen-sink snapshot"** ŌĆö when multiple consumers each need a typed snapshot of part of a domain object's state, prefer one small value type per consumer over one shared type that unions every field everyone might want. `SnapshotEntry{ID, PID}` for the rotation watcher (closure-over-primitives boundary, cannot import the host's domain types); `SessionInfo{ID, Label, LifecycleState, LastActiveAt, Bootstrap}` for the operator-visible list. Adding lifecycle state to `SnapshotEntry` would either bloat every rotation snapshot or leak the host's enum into the rotation package's import set. Two ~5-field types are clearer than one ~10-field type with most fields irrelevant per call site.
- **Per-session template fields snapshotted at construction, read-only after** ŌĆö when a Pool/Registry needs a "shape of new entries" template (claude bin, work dir, backoff timings, bridge presence), capture it as an unexported field on the host (`Pool.sessionTpl`) at `New()` time and treat it as immutable thereafter. Per-create call clones the slice fields (`slices.Clone(tpl.ClaudeArgs)`) and mints fresh resources where sharing would be wrong (`supervisor.NewBridge(p.log)` per session in service mode). No lock needed for reads; mutation is impossible by construction. Two ~15-line struct constructions (bootstrap in `New`, fresh entry in `Create`) with different lifecycle-state defaults is clearer than a `lcState lifecycleState` parameter on a shared helper ŌĆö accepted intentionally.
- **Side-fd + Close to drain a blocked Read goroutine ŌĆö only works with `O_NONBLOCK` on character devices.** When a goroutine is blocked in `io.Copy`/`Read` on a process-wide fd (e.g. `os.Stdin`) with no way to wake it, the standard fix is to open a parallel fd to the same underlying device, copy from *that* fd, and `Close` it on cleanup. For sockets and pipes the runtime poller mediates Reads automatically; for character devices like `/dev/tty` you must pass `os.O_RDONLY|syscall.O_NONBLOCK` so `Read` returns `EAGAIN` and the goroutine parks on the poller ŌĆö only then does `Close` wake it via `runtime_pollUnblock`. Plain blocking `open` produces an fd whose in-flight `syscall.Read` cannot be interrupted by `Close` on another goroutine. Used for the foreground-mode input bridge in `internal/supervisor` (#78); applies to any "wake a stuck reader by closing a side fd" pattern.
- **Symmetric channel pairs for two-way state-machine signalling** ŌĆö when a two-state machine (`active` / `evicted`) needs both *enter-state* and *exit-state* external triggers, mirror the channel pair: `activateCh` (buffered 1 signal) + `activeCh` (closed-on-active broadcast) gets a sibling `evictCh` (buffered 1 signal) + `evictedCh` (closed-on-evicted broadcast). Both pairs follow the same snapshot-under-lock-then-wait discipline. The state-transition code closes one channel and re-opens the other in lockstep. One pattern, two directions.
- **`HOME=t.TempDir()` as the e2e isolation envelope** ŌĆö when the binary under test resolves all of its persistent paths via `os.UserHomeDir()` (which honors `$HOME` on Unix), redirecting the child's `HOME` to `t.TempDir()` contains every ~-relative path with a single env var ŌĆö no per-resource flag plumbing required. Belt-and-suspenders explicit flags (e.g. `-pyry-socket=<tmp>/pyry.sock`) cover the load-bearing path without inventing new flags for ancillary state. The corollary: keep ALL persistent paths under `~/...` (don't sprinkle `/var/`, `/etc/`, `/tmp` defaults) so this trick keeps working.
- **Subprocess re-exec for "test the test framework's failure path"** ŌĆö when verifying that a `t.Fatal`/`t.FailNow` path doesn't leak resources, you can't observe it from a same-process subtest (Go's testing framework propagates inner failures to the parent and ends the outer test before assertions run). Re-exec `os.Args[0]` with `-test.run=^TestInnerChild$` and an env-var flag (`PYRY_E2E_INNER_FATAL_OUT=<state-file>`) so the failing path runs in a fresh process; the child writes observed state (pid, socket path) to the file before failing; the parent reads it and asserts externally. The inner test is gated on the env var (`os.Getenv(...) == "" ŌåÆ t.Skip`) so normal runs are no-ops. Pairs naturally with the POSIX zero-signal probe (`os.FindProcess` + `Signal(syscall.Signal(0))`) for liveness checks.
- **Sentinel + `fmt.Errorf("%w: ŌĆ”")` over a struct error type when the only data the caller needs is a printable list** ŌĆö when an error needs to carry "and here's what matched" data alongside a typed-discriminator (e.g. ambiguous prefix ŌåÆ list of candidate ids), prefer `var ErrFoo = errors.New(...)` plus `fmt.Errorf("%w:\n%s", ErrFoo, formattedLines)` over a struct error type with an exposed `Matches []Ref` field. Pros: zero new exported types; `errors.Is` is one pointer compare via the wrapped chain; CLI consumers do one `fmt.Fprintln(os.Stderr, err)` and get the human-readable list verbatim. Cons: callers cannot programmatically iterate matches ŌĆö accept this whenever the data is fundamentally for human consumption (CLI disambiguation prompts, debug logs) and downstream code only needs the typed branch. Used for `Pool.ResolveID`'s `ErrAmbiguousSessionID` (1.1e-A); the same shape suits any "typed error + descriptive payload for printing" surface.
- **Loose-input resolver pairs with strict-input lookup** ŌĆö when a domain object is keyed by a canonical id (e.g. `SessionID`) and external callers (CLIs, wire APIs) hand in loose strings (UUIDs, prefixes, empty), expose two methods: `Lookup(canonical)` for code-internal callers that already have the canonical form, and `ResolveID(loose) (canonical, error)` for input-handling callers. Resolution returns the canonical id, not `*Session` ŌĆö callers then call `Lookup`/`Activate`/`Remove` with the resolved id, which guards against the entity being removed between resolve and use without saving any meaningful work. Resolver runs under the host's read lock; consumers each pay one extra hashmap probe but never observe a half-removed state. Same shape would suit any "user-input ŌåÆ canonical-id" transformation that today's consumers inline against the host's lock.
- **Exit-1 for runtime errors, exit-2 for "user typed it wrong"** ŌĆö in CLI subcommands, return errors up the call stack for runtime failures (RPC failed, daemon-not-running, resolver error) so `main` prints them and exits 1. Use `os.Exit(2)` directly for argument-shape errors (extra positionals, missing required arg) so the two failure classes stay visually distinct in shell exit codes. POSIX convention; matches the existing `os.Exit(2)` shape at the top of `cmd/pyry/main.go`. The split is checkable: error return ŌćÆ runtime, `os.Exit(2)` inline ŌćÆ user typed wrong. Pair the inline exit with a unit-testable helper that returns a sentinel (`attachSelectorFromArgs` returning `errTooManyAttachArgs`) so the argument-shape rule is covered in a table test without intercepting `os.Exit`.
- **Extend an exported function's signature in place when there is exactly one caller.** Adding a sibling (e.g. `AttachWithID`) for a single new parameter splits the call surface for no benefit. The grep-for-callers test is the discriminator ŌĆö if extending churns >1 production caller, consider a sibling; if it's one, extend. Used for `control.Attach` in 1.1e-D (one call site, fifth arg added). Same shape suits any "one new parameter, one caller" growth.
- **Build-tag-gated test packages for opt-in expensive tests** ŌĆö when a test category is heavy (spawns binaries, requires `go build`, takes seconds) and not part of every-PR CI, gate the whole package with `//go:build e2e` (or similar). Default `go test ./...` doesn't compile or run it; opt-in via `go test -tags=e2e ./<path>/...`. Document the invocation in the package doc comment so godoc surfaces it. Lighter than a separate `make` target for the gating itself; the make target is the operator-facing convenience, not the mechanism.
- **Caller-supplied path + caller-owned cleanup, with a sugar wrapper for the common case** ŌĆö when a test-harness primitive that originally allocated its own resource (e.g. `Start(t)` calling `t.TempDir()`) needs a caller-controlled variant for one specific scenario (here: pre-populating HOME for a restart-shaped test), refactor the body into a "primitive that takes the resource" (`StartIn(t, home)`) and redefine the original as a one-line wrapper (`return StartIn(t, t.TempDir())`). Caller-owned cleanup means the primitive does not register the resource with `t.Cleanup` ŌĆö the caller decides whether to reuse it across calls. Existing callers see no change; new use cases compose. `Options` struct migration is non-breaking later when a second knob shows up. Used for `e2e.StartIn` (#106).
- **Public wrapper around a private cleanup primitive, idempotent via `sync.Once`** ŌĆö when a `t.Cleanup`-registered teardown needs to also be callable mid-test (here: `Stop()` to enable a restart pattern), expose the public verb as a one-line wrapper around the internal `teardown` rather than renaming. The `sync.Once` already guarding teardown makes both call paths safely idempotent without new state. The naming asymmetry (`Stop` public, `teardown` private) signals the public/private split to readers; renaming the private helper to "Stop_internal" or similar is anti-simplicity. Used for `Harness.Stop` (#106).
- **Sibling harness for an interactive surface, not retrofitted onto the non-interactive one** ŌĆö when a binary has both a non-interactive surface (verbs over a socket) and an interactive one (PTY-bound `attach`), give each its own harness (`Harness` + `Start(t)` for daemon-only verbs; `AttachHarness` + `StartAttach(t, sessionID)` for PTY round-trips) rather than parameterising one harness with an "interactive?" mode. The two have different lifecycle invariants (3 OS resources vs. 1), different skip conditions (`pty.Open` failure vs. only `cmd.Start`), different teardown ordering (masterŌåÆslaveŌåÆattachŌåÆdaemon vs. just daemon), different env-injection needs (helper vars on daemon's env so `runOnce` propagates them vs. plain `childEnv`). Sharing the binary cache (`ensurePyryBuilt`), `childEnv`, `killSpawned`, and the readiness timing constants is the right reuse boundary; sharing `Start`'s entry point would force an "options polymorphism" hairball. Used for #125's `AttachHarness`. Same shape suits any future harness that wraps a third interface (e.g. an HTTP API for Phase 2.0).
- **`SysProcAttr{Setsid: true, Setctty: true}` to attach a child to a specific PTY slave** ŌĆö when `exec.Cmd`'s stdio is a PTY slave fd and the child needs to treat it as its controlling terminal (so `IsTerminal(0)` reflects the slave and `term.MakeRaw` puts the right tty into raw mode), set both fields on `SysProcAttr`. `Setsid` puts the child in a fresh session (a controlling terminal can only be acquired by a session leader); `Setctty` then attaches the slave fd as the new session's controlling tty. Without this, the child inherits the parent's controlling terminal and writes to the slave still happen, but `IsTerminal` and `MakeRaw` operate on the wrong device. Used in `StartAttach` (#125) so the attach client's `term.MakeRaw(slave fd)` reaches the slave PTY rather than the test process's terminal.
