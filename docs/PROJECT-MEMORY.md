# Project Memory ŌĆö Pyrycode

Repo-level session memory. Read this at the start of every session.

## What's Built

### Codebase (Phase 0)
- **Supervisor core** ŌĆö PTY spawn via `creack/pty`, raw-mode stdin/stdout bridging in foreground mode, Bridge-mediated I/O in service mode, exponential backoff restart with stability reset, `--continue` injection on restart for session persistence
- **SIGWINCH forwarding** ŌĆö terminal resizes propagate from controlling terminal to child PTY (foreground mode only; attach mode locks the size at attach time)
- **Control plane** ŌĆö Unix domain socket (`~/.pyry/<name>.sock`, 0600), line-delimited JSON protocol, verbs: `status`, `stop`, `logs`, `attach`
- **CLI transparency** ŌĆö unknown args forward verbatim to claude; pyry's own flags use `-pyry-*` prefix; `-pyry-name` plus `PYRY_NAME` env var for named multi-instance
- **Graceful shutdown** ŌĆö SIGINT/SIGTERM cancel the supervisor context, child is killed via `exec.CommandContext`, socket removed on exit
- **Service configs** ŌĆö systemd user unit (`systemd/pyry.service`), macOS launchd plist (`launchd/dev.pyrycode.pyry.plist`)
- **~1700 source + ~1100 test Go lines** as of late Apr 2026, 10+ PRs merged

### Codebase (Phase 1.0, tickets #28 + #29)
- **`internal/sessions` package** ŌĆö `SessionID` (UUIDv4 via `crypto/rand`, stdlib only), `Session` (wraps one `*supervisor.Supervisor` + optional `*supervisor.Bridge`), `Pool` (single-bootstrap registry with `RWMutex`-protected map). Sentinel errors `ErrSessionNotFound`, `ErrAttachUnavailable`. `Pool.Lookup("")` resolves to the bootstrap entry ŌĆö the seam Phase 1.1's `Request.SessionID` plugs into.
- **Production consumers wired (#29)** ŌĆö `cmd/pyry/main.go` constructs `*sessions.Pool` (with the supervisor.Config template inside `SessionConfig`); `internal/control` consumes a single `SessionResolver` interface (replaces Phase 0's `StateProvider` + `AttachProvider` pair). A 5-line `poolResolver` adapter in `cmd/pyry` bridges `Pool` ŌåÆ `SessionResolver` (covariant-return workaround). Wire protocol unchanged; `pyry status`/`stop`/`logs`/`attach` byte-identical to Phase 0. Foreground-mode attach error string preserved verbatim via `errors.Is(err, sessions.ErrAttachUnavailable)` mapping in `handleAttach`.
- See [knowledge/features/sessions-package.md](knowledge/features/sessions-package.md), [knowledge/features/control-plane.md](knowledge/features/control-plane.md), and [ADR 003](knowledge/decisions/003-session-addressable-runtime.md).

### Codebase (Phase 1.1a-A1, ticket #72)
- **Pool supervisor fan-out seam** ŌĆö unexported `Pool.supervise(sess *Session) error` schedules `sess.Run(gctx)` on `Pool.Run`'s live errgroup. Returns the new sentinel `ErrPoolNotRunning` when called before `Run` has wired the handle or after `Run` has cleared it (matchable with `errors.Is`). The helper is the join point Phase 1.1a-A2's `Pool.Create(ctx, label)` consumes; the bootstrap fan-out inside `Pool.Run` is itself rewritten to call `supervise` so the helper is exercised in production from day one. Watcher fan-out is unchanged (not a `*Session`).
- **Supervisor handle on `*Pool`** ŌĆö `runGroup *errgroup.Group` + `runCtx context.Context`, both guarded by `Pool.mu` (RWMutex) and read together so a caller never sees a half-initialised handle. Set in `Pool.Run` under `Pool.mu` (write) right after `errgroup.WithContext`; cleared in a `defer` so a panicking goroutine still resets the handle. `supervise` reads under `Pool.mu` (RLock) and calls `g.Go` *off-lock*. No new lock-order edges (`Pool.mu ŌåÆ Session.lcMu` and `Pool.capMu ŌåÆ Pool.mu ŌåÆ Session.lcMu` unchanged) ŌĆö the helper does not touch `Session.lcMu`.
- **Why fields-on-Pool, not a separate guard struct** ŌĆö `Pool.mu` already protects every other "what's in this pool right now" read. A second mutex would double the lock count without buying anything; RWMutex matches the access pattern (N concurrent supervise readers, one-shot writes by `Run` setup/teardown). Concurrent `supervise` callers don't contend with each other.
- **Race-with-teardown is benign** ŌĆö a `supervise` call interleaved with the deferred clear either (a) acquires RLock first, schedules onto a group whose ctx is about to cancel, and `Session.Run` exits via the existing shutdown path, or (b) acquires after, observes `nil`, returns the sentinel. No silent leak, no deadlock.
- **Defensive wrap on the bootstrap call** ŌĆö `Pool.Run` does `if err := p.supervise(bootstrap); err != nil { return fmt.Errorf("sessions: supervise bootstrap: %w", err) }` even though the precondition is established two lines above. Cheap insurance against a future reordering bug.
- See [knowledge/features/sessions-package.md ┬¦ Supervisor handle](knowledge/features/sessions-package.md).

### Codebase (Phase 1.2c-B, ticket #41)
- **Concurrent active cap (LRU eviction)** ŌĆö `Config.ActiveCap` bounds the number of concurrently running claudes. `<= 0` is unset (uncapped, byte-identical to 1.2c-A); `>= 1` enforces. When activating one more session would exceed the cap, the LRU peer (oldest `lastActiveAt` among currently-active sessions, excluding the target) is evicted via the new public `Session.Evict` primitive before the new spawn proceeds. **No CLI flag yet** ŌĆö Phase 2.0 will wire `-pyry-active-cap` when the auto-mint workload makes it load-bearing.
- **`Session.Evict(ctx)` public primitive** ŌĆö symmetric to `Activate`. Force-eviction: unlike the idle timer, it does **not** defer for `attached > 0`. The cap is a hard limit; an attached caller will see EOF on its bridge. Phase 2.0 may add an attach-aware filter to the LRU candidate set; not in scope for #41. Pairs `evictCh` (buffered 1 signal) with `evictedCh` (closed-on-evicted broadcast), mirroring `activateCh`/`activeCh`.
- **`Pool.capMu` outermost lock** ŌĆö taken only on the cap path (`activeCap > 0`). Lock order: `Pool.capMu ŌåÆ Pool.mu ŌåÆ Session.lcMu`. Serialises cap-check + victim eviction + new spawn so concurrent `Activate`s can't both observe `active < cap` and both proceed. Never re-acquired by callees, so `Session.Evict`'s callback into `Pool.persist` (which takes `Pool.mu`) is deadlock-free. The uncapped path doesn't take `capMu` at all ŌĆö "byte-identical when unset" includes "doesn't even take the cap mutex".
- **`Pool.pickLRUVictim`** ŌĆö O(n) iteration over `pool.sessions` under `Pool.mu.RLock`, reads each `Session.lcState`/`lastActiveAt` under `lcMu`, returns the oldest-`lastActiveAt` peer (excluding target). Premature optimisation rejected: a heap/linked-list would defeat the zero-cost-when-unset property and add a counter-vs-truth drift class of bug for an O(n) cost that doesn't matter at pyry's scale (Ōēż 100s of sessions).
- **`Session.touchLastActive`** ŌĆö bumps `lastActiveAt` in-memory only on a cap-path `Activate` against an already-active session. **Not persisted** (the registry's `lastActiveAt` only flushes on transitions). Keeps in-memory LRU ordering reflective of the most recent touch.
- **`Pool.Activate` is now the spawn-path entry, not just a thin Lookup wrapper.** Phase 2.0's router calls this. Cap-unset path: one early-return branch and forward to `Session.Activate`. Cap-set path: cap check + victim eviction + new spawn under `capMu`.
- See [knowledge/features/idle-eviction.md](knowledge/features/idle-eviction.md), [ADR 006](knowledge/decisions/006-concurrent-active-cap-lru.md).

### Codebase (Phase 1.2c-A, ticket #40)
- **Idle eviction + lazy respawn** ŌĆö per-`*Session` lifecycle goroutine drives a two-state machine (`active` Ōåö `evicted`). When `attached == 0` and the idle timer fires, the supervisor's inner ctx is cancelled and claude exits cleanly; the JSONL on disk is preserved. `Session.Activate(ctx)` respawns the supervisor pointing at the same JSONL on disk so the conversation continues. `Session.Run` rewritten from one-line supervisor delegate into a `runActive` / `runEvicted` loop; `Pool.Run` shape **unchanged** (multi-session fan-out is Phase 1.1's job).
- **Activity = attach state** ŌĆö while `attached > 0` the timer re-arms on fire (poll-with-grace; eviction may overshoot the configured timeout by up to one window). `last_active_at` is bumped on every state transition for the upcoming LRU follow-up's victim selection.
- **`-pyry-idle-timeout` flag** ŌĆö production default `15m`, `0` disables eviction entirely (test default + operator escape hatch). Wired through `Config.IdleTimeout` ŌåÆ `SessionConfig.IdleTimeout` (per-session override falls back to pool default).
- **Activate-before-Attach contract** ŌĆö `bridge.Attach` on an evicted session would block on the pipe forever. `internal/control.Session` interface gained `Activate(ctx)`; `handleAttach` calls it with a 30s timeout cap before binding the bridge. `handleStatus` does **not** activate (faithful `PhaseStopped` for evicted; no spurious wakeups from a poll). Wire response on respawn failure: `attach: activate: <err>`.
- **Registry `lifecycle_state` field** ŌĆö `"active"` / `"evicted"`, `omitempty` so the dominant active case stays off disk (preserves #34's idempotent-reload byte-stability). Old pyry binaries ignore the field; new pyry reading old files defaults to `"active"`. Bootstrap warm-starts in whatever state the registry says ŌĆö including starting in `evicted` if pyry was stopped while evicted.
- **Lock order: `Pool.mu ŌåÆ Session.lcMu`** ŌĆö `Session.lcMu` protects `lcState` / `attached` / `activeCh` / `lastActiveAt` (when read for the registry snapshot). `Session.transitionTo` releases `lcMu` before calling `Pool.persist` so `saveLocked`'s per-session re-acquire doesn't deadlock. `RotateID` mutates `session.id` without `lcMu` ŌĆö documented invariant (today's only callers run before any lifecycle goroutine observes the id).
- **Concurrency primitives** ŌĆö `activeCh` (closed-on-active broadcast) + `activateCh` (buffered 1 signal). `Activate` snapshots `activeCh` under `lcMu` *before* waiting so a concurrent evict-replace doesn't drop the wakeup. Concurrent `Activate`s collapse to one signal via the buffered send.
- See [knowledge/features/idle-eviction.md](knowledge/features/idle-eviction.md), [ADR 005](knowledge/decisions/005-idle-eviction-state-machine.md).

### Codebase (Phase 1.2b-B, ticket #39)
- **Live `/clear` rotation watcher** ŌĆö `internal/sessions/rotation` package: fsnotify on `cfg.ClaudeSessionsDir` + per-PID FD probe (Linux walks `/proc/<pid>/fd`, macOS shells out to `lsof -nP -p <pid> -F fn`). On a CREATE for `<new>.jsonl` whose UUID a tracked PID has open, calls `Pool.RotateID(old, new)` ŌĆö same seam #38 established. Detection latency ~250ms worst-case (probeWithRetry walks 0/50/200ms attempts to handle CREATE-before-open races).
- **`Pool.Run` errgroup wrap** ŌĆö `golang.org/x/sync/errgroup` ties the bootstrap supervisor and the watcher; cancellation propagates both ways. Phase 1.1's N-session fan-out reuses the same wrapper ŌĆö the extension point is one `g.Go` per pool entry.
- **Pyry-allocated UUID skip set on `Pool`** ŌĆö `RegisterAllocatedUUID` / `IsAllocated`. Consume-on-first-hit, 30s TTL pruned opportunistically, all under `p.mu` (write). No live caller in 1.2b-B (pyry still uses `--continue`); scaffolding lands now so Phase 1.1's `claude --session-id` is a one-liner.
- **`rotation.Config` closures over primitives** ŌĆö the rotation package never imports `internal/sessions`. The `SessionID Ōåö string` conversion happens once, inside `Pool.Run`'s closure wiring. `Pool.Snapshot` returns a small `SnapshotEntry` struct; `snapshotForRotation` translates to `rotation.SessionRef`.
- **Failure posture** ŌĆö rotation detection is never load-bearing. fsnotify init failure, missing `lsof`, missing `/proc`, save failure during `OnRotate` ŌĆö all logged warn/debug; pyry continues. `MkdirAll(dir, 0700)` for a not-yet-existing claude dir.
- **First new external runtime dep since `creack/pty`** ŌĆö `github.com/fsnotify/fsnotify` v1.10.0, justified in [ADR 004](knowledge/decisions/004-fsnotify-for-rotation-detection.md). `golang.org/x/sync` lands as a sibling for errgroup.
- See [knowledge/features/rotation-watcher.md](knowledge/features/rotation-watcher.md), [ADR 004](knowledge/decisions/004-fsnotify-for-rotation-detection.md).

### Codebase (Phase 1.2b-A, ticket #38)
- **Startup JSONL reconciliation** ŌĆö `Pool.New` scans `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl`, picks the most-recently-modified one, and rotates the registry's bootstrap entry to that UUID if it disagrees. Self-heals after `/clear` (claude rotates session UUIDs on `/clear`; without reconciliation, post-`pyry stop` the registry would still point at the pre-`/clear` UUID).
- **`Pool.RotateID(oldID, newID)`** ŌĆö atomic swap of in-memory map key + bootstrap pointer + `saveLocked`, all under `p.mu` (write). `RotateID(x, x)` is a no-op; unknown old id returns `ErrSessionNotFound`. Bumps `last_active_at`; preserves `created_at`. The load-bearing seam reused by Phase 1.2b-B (live-detection via fsnotify).
- **`encodeWorkdir`** ŌĆö claude maps both `/` AND `.` to `-` in its `~/.claude/projects/` path component (verified empirically 2026-05-02). `/foo/.bar` ŌåÆ `-foo--bar` (note the doubled dash). Unexported; encoding lives inside `internal/sessions` and `cmd/pyry` only sees the resolved path via `DefaultClaudeSessionsDir`.
- **JSONL location** ŌĆö files live directly in `<encoded-cwd>/<uuid>.jsonl`. There is **no** `sessions/` subdirectory (the ticket body claimed otherwise; the spec followed reality).
- **Failure posture** ŌĆö missing/unreadable claude dir is logged warn and ignored (startup proceeds). Save failure during rotation is fatal-at-startup (matches 1.2a cold-start). The pre-rotation JSONL is never deleted or modified ŌĆö only the registry pointer moves.
- **Expected one-time rotation in 1.2a** ŌĆö on the second `pyry` restart, claude's UUID ŌēĀ pyry's mint, so reconcile fires once and logs an Info line. Becomes rare after Phase 1.1+ wires `claude --session-id <uuid>`.
- See [knowledge/features/jsonl-reconciliation.md](knowledge/features/jsonl-reconciliation.md).

### Codebase (Phase 1.2a, ticket #34)
- **Session registry on disk** ŌĆö `~/.pyry/<sanitized-name>/sessions.json` (file 0600, dir 0700), sibling to the per-name socket. Schema: `version` (forward-marker), `sessions[]` with `id` / `label` / `created_at` / `last_active_at` / `bootstrap`. Default `encoding/json` decoder tolerates unknown fields for forward compat.
- **`Pool.New` load-or-mint** ŌĆö Cold start (missing or empty file) mints a fresh UUID and writes the registry before returning. Warm start reads the bootstrap-marked entry, reuses its UUID + metadata, and does **not** rewrite the file (warm reload is not a state change). Malformed JSON is fatal at startup ŌĆö operator must fix or remove.
- **Atomic write seam** ŌĆö `saveRegistryLocked` does `os.CreateTemp` ŌåÆ `Chmod 0600` ŌåÆ encode ŌåÆ fsync ŌåÆ close ŌåÆ `os.Rename`. Rename is the commit point; partial JSON is unreachable in the target. `defer os.Remove(tmp)` cleans up orphaned temps best-effort. `Pool.saveLocked` is the package-internal hook Phase 1.1's `Pool.Add` / `Rename` / `Remove` will call before returning success ŌĆö caller holds `Pool.mu` (write) across the disk I/O.
- **Bootstrap marker on disk** ŌĆö `bootstrap: true` is persisted explicitly so `Pool.Lookup("")` doesn't depend on file ordering. Phase 1.1's `pyry sessions rm <bootstrap-uuid>` thus has a clean question to answer (refuse, or promote another entry) instead of relying on array position. `omitempty` keeps non-bootstrap entries clean on disk.
- **Stable disk ordering** ŌĆö `sortEntriesByCreatedAt` (then by `ID`) before write makes the file's byte content a deterministic function of the in-memory set, defeating Go's randomized map iteration and giving the AC's idempotent-reload guarantee. Degenerate for 1.2a's single entry; the property is paid for upfront ahead of 1.1.
- See [knowledge/features/sessions-registry.md](knowledge/features/sessions-registry.md).

## Patterns Established

- **Config struct pattern** ŌĆö all configuration in a single `Config` struct, defaults applied in `New()`
- **Context-based cancellation** ŌĆö `context.Context` flows through `Run()` ŌåÆ `runOnce()`, checked at every wait point
- **Structured logging** ŌĆö `log/slog` with injected logger, not a global
- **Exponential backoff with stability reset** ŌĆö backoff doubles on restart, resets to initial if child stayed up longer than `BackoffReset`
- **Deferred cleanup** ŌĆö `defer` for terminal restore, PTY close, signal stop
- **Empty ID resolves to default** ŌĆö `Pool.Lookup("")` returns the bootstrap session, so future `req.SessionID` fields can be added with no handler-side branching (old clients send empty, get the bootstrap; new clients send a real ID, get the right entry)
- **Introduce-then-rewire slicing** ŌĆö split #27 into #28 (new package + tests, no consumers) and #29 (mechanical consumer rewiring) to keep each PR focused
- **Consumer-side interface definition** ŌĆö `internal/control` defines the interfaces it consumes (`SessionResolver`, `Session`) rather than importing them from the producer package. Keeps `internal/sessions` free of control-plane concerns and lets tests fake the surface without exporting test seams from the producer.
- **Wire-string preservation via `errors.Is` mapping** ŌĆö when refactoring an error path that crosses package boundaries, map the new sentinel back to the old wire string explicitly (`if errors.Is(err, sessions.ErrAttachUnavailable) { ŌĆ” Phase 0 string ŌĆ” }`) rather than letting `fmt.Sprintf("%v", err)` change client output. Required when an AC says "byte-identical."
- **Atomic on-disk writes via temp + rename** ŌĆö for operator-recoverable JSON state files, `os.CreateTemp` in the same dir ŌåÆ encode ŌåÆ `f.Sync()` ŌåÆ `f.Close()` ŌåÆ `os.Rename(tmp, path)` makes the rename the commit point and partial files unreachable. `defer os.Remove(tmp)` cleans up orphans best-effort (no-op after successful rename). Don't bother with directory fsync unless real-world corruption is observed ŌĆö pyry's registry is recoverable, not a database.
- **Stable disk ordering for idempotent reload** ŌĆö sort serialized collections by a stable key (e.g. `created_at` then `id`) before writing. Defeats Go's randomized map iteration so saveŌåÆloadŌåÆsave round-trips are byte-stable, and makes load-twice-equals-load-once into a real property rather than a probabilistic one.
- **Persistence as a Config field, empty disables** ŌĆö `Config.RegistryPath` (or any "where to persist" string) defaulting to empty = no I/O lets unit tests construct a Pool with `t.TempDir()` or with persistence off entirely, with no test-only branches inside `New`. Production callers always pass a real path; the empty case is genuinely test-only. Same shape reused for `Config.ClaudeSessionsDir` in 1.2b-A.
- **One mutation seam, multiple drivers** ŌĆö `Pool.RotateID` is the single place that swaps a session's in-memory id and persists. Startup reconciliation drives it from a directory scan; the live-detection watcher drives it from an fsnotify event. Defining the seam once with the lock-and-save contract baked in meant the second driver landed as a goroutine that calls a function, not a redesign.
- **Closures over primitives across package boundaries** ŌĆö when a downstream package (`internal/sessions/rotation`) needs to call back into its host (`internal/sessions`), express the contract as a struct of closures over primitive types (`string`, `int`) rather than an interface that references the host's domain types (`SessionID`). The downstream package then has zero knowledge of the host, and the type-safety conversion (`SessionID Ōåö string`) happens exactly once, in the wiring site. Avoids the import cycle the obvious "watcher takes a `*Pool`" design would create.
- **Test seam via package var, not interface injection** ŌĆö `var newProbe = rotation.DefaultProbe` in `internal/sessions/pool.go`. Tests overwrite to inject a fake; production callers see no extra plumbing. Same shape as `allocatedTTL` and `probeRetryDelays` (test-tunable knobs that aren't config). Cheaper than threading a factory through `Config` for a single test surface.
- **Errgroup as the fan-out seam** ŌĆö `Pool.Run`'s `errgroup.WithContext` ties the bootstrap supervisor and the rotation watcher; Phase 1.1's N-session fan-out reuses the same cancellation+error semantics. The 1.1a-A1 (#72) seam is the natural shape: stash `*errgroup.Group` + `context.Context` on the host struct under its existing RWMutex, expose an unexported helper that RLock-snapshots both and calls `g.Go` off-lock, and pair it with a sentinel error (`ErrPoolNotRunning`) for before/after-`Run` calls. The helper is the join point post-`Run` consumers (`Pool.Create`) plug into; the existing bootstrap fan-out is refactored onto the helper so the helper is exercised in production from day one rather than living dormant.
- **Two-state machine with explicit-verb wake-up** ŌĆö when a runtime entity needs an `idle Ōåö live` lifecycle, prefer (a) a per-entity goroutine that owns both the state machine and the underlying resource lifecycle, (b) an explicit `Activate(ctx)` call rather than implicit wake on resolve/lookup, and (c) a closed-on-state broadcast channel + buffered(1) signal channel for the wake-up handshake. Implicit wake-on-lookup tangles read-side concerns (status reads paying respawn latency); explicit Activate keeps every caller's contract sharp. Used for `Session.Activate` (1.2c-A); will be reused by future per-session lifecycles (router, bind/unbind).
- **Lock order `Pool.mu ŌåÆ Session.lcMu` with release-before-callback** ŌĆö when a callee needs to call back into the host's lock-protected persistence (`Pool.persist` from `Session.transitionTo`), release the inner lock *before* the callback so the host's re-acquire of the inner lock inside its serialization path doesn't deadlock. Document the order at every site that takes both. Same shape would apply to any "child mutates state then asks parent to persist" coupling.
- **Dedicated outer mutex for "cap-shaped" critical sections** ŌĆö when a sequence (cap-check + victim eviction + new spawn) must be atomic but its inner steps already take the host's main mutex (and may re-take it via persistence callbacks), introduce a *separate* outer mutex (`Pool.capMu` for #41) rather than holding the main mutex (`Pool.mu` write) across the whole sequence. The outer mutex serialises the policy decision; the inner work continues to take/release the main mutex normally. Avoids both the deadlock (re-entrancy on `Pool.mu` via `persist`) and the contention (blocking unrelated `Pool.mu` reads for the eviction's bounded window). Lock order documented; outer mutex is never re-acquired by callees.
- **"Zero cost when unset" as a hot-path discipline** ŌĆö when a feature is configurable and most operators leave it unset (`Config.ActiveCap == 0`), gate ALL of its bookkeeping behind the early-return branch ŌĆö including the new mutex it needs. The uncapped `Pool.Activate` doesn't even take `capMu`, doesn't iterate `sessions`, doesn't read `lastActiveAt`. The branch is one `if cap <= 0 { return ŌĆ” }` at the top of the cap-only path. The property "the previous phase's behaviour is byte-identical with the new feature unset" is paid for by being *strict* about what happens before the early return.
- **Symmetric channel pairs for two-way state-machine signalling** ŌĆö when a two-state machine (`active` / `evicted`) needs both *enter-state* and *exit-state* external triggers, mirror the channel pair: `activateCh` (buffered 1 signal) + `activeCh` (closed-on-active broadcast) gets a sibling `evictCh` (buffered 1 signal) + `evictedCh` (closed-on-evicted broadcast). Both pairs follow the same snapshot-under-lock-then-wait discipline. The state-transition code closes one channel and re-opens the other in lockstep. One pattern, two directions.

## Open Questions

- **Backoff cooldown/bail-out** ŌĆö if crashes happen N times in T seconds, should the supervisor give up? Currently retries forever, which is the right default for a service supervisor (a supervised child that never starts is the operator's problem to investigate, not for pyry to give up on).
- **Phase 0.5 ŌĆö Real production test** ŌĆö supervisor hasn't been tested with a real `claude` child on pyrybox running as a launchd/systemd service. The tmux setup is still running. This is the only Phase 0 item left after PRs #1-#10.

(Earlier "Session ID tracking" and "Control socket design" questions were resolved by the PR series that landed Phase 0.2ŌĆō0.4: `--continue` for session continuity, line-delimited JSON over a Unix socket for control.)
